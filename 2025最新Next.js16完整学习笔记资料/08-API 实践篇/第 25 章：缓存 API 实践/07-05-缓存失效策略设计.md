# 缓存失效策略设计

## 概述

缓存失效策略是缓存系统的核心部分,决定了何时以及如何使缓存数据失效。合理的缓存失效策略可以确保用户看到最新的数据,同时最大化缓存的性能优势。

### 缓存失效的重要性

1. **数据一致性**: 确保缓存数据与源数据保持同步
2. **用户体验**: 用户能够及时看到最新内容
3. **性能平衡**: 在性能和新鲜度之间找到平衡
4. **资源优化**: 避免不必要的缓存更新
5. **系统稳定性**: 防止缓存雪崩等问题

### 缓存失效的挑战

1. **时机选择**: 何时使缓存失效
2. **范围控制**: 哪些缓存需要失效
3. **级联影响**: 一个缓存失效可能影响其他缓存
4. **性能影响**: 大量缓存失效可能导致性能问题
5. **一致性保证**: 多个缓存之间的一致性

## 缓存失效策略类型

### 1. 基于时间的失效(TTL)

最简单的失效策略,设置固定的过期时间:

```typescript
// app/posts/[slug]/page.tsx
```

```typescript
    revalidatePath(`/products/${slug}`);
```

```typescript
// 更新产品分类
export async function updateProductCategory(slug: string, categoryId: string) {

    where: { slug },
    select: { categoryId: true },
  });

  await db.product.update({
    where: { slug },
    data: { categoryId },
  });

  // 失效旧分类和新分类的列表页
  if (oldProduct) {
    revalidatePath(`/products?category=${oldProduct.categoryId}`);
  }
  revalidatePath(`/products?category=${categoryId}`);

  // 失效产品详情
  revalidatePath(`/products/${slug}`);

  return { success: true };
}
```

```typescript
  slug: string,
  data: {
    title?: string;
    content?: string;
    excerpt?: string;
    featured?: boolean;
  }
) {
  const oldPost = await db.post.findUnique({
    where: { slug },
    select: { featured: true },
  });

  const post = await db.post.update({
    where: { slug },
    data,
  });

  // 失效文章详情页
  revalidatePath(`/posts/${slug}`);

  // 如果featured状态变化,失效首页
  if (data.featured !== undefined && data.featured !== oldPost?.featured) {
    revalidatePath("/");
  }

  // 如果是featured文章,失效文章列表
  if (post.featured || oldPost?.featured) {
    revalidatePath("/posts");
  }

  revalidateTag(`post-${slug}`);

  return { success: true, post };
}

    where: { slug },
    include: {
      author: true,
      category: true,
      tags: true,
    },

    throw new Error("Post not found");

    where: { slug },

  revalidatePath("/posts");
  revalidatePath(`/category/${post.category.slug}`);
  revalidatePath(`/author/${post.author.slug}`);

    revalidatePath(`/tag/${tag.slug}`);

  revalidateTag(`category-${post.category.id}`);
```

## 实战场景四: 分层失效策略

### 三层缓存失效架构

```typescript
// lib/cache/invalidation.ts
import { revalidatePath, revalidateTag } from "next/cache";

// 缓存失效级别
enum InvalidationLevel {
  SPECIFIC = "specific", // 特定资源
  CATEGORY = "category", // 分类级别
  GLOBAL = "global", // 全局级别
}

// 缓存失效策略
interface InvalidationStrategy {
  level: InvalidationLevel;
  paths?: string[];
  tags?: string[];
  cascade?: boolean; // 是否级联失效
}

// 执行缓存失效
export async function invalidateCache(strategy: InvalidationStrategy) {
  const { level, paths = [], tags = [], cascade = false } = strategy;

  // 失效路径
  for (const path of paths) {
    revalidatePath(path);
  }

  // 失效标签
  for (const tag of tags) {
    revalidateTag(tag);
  }

  // 级联失效
  if (cascade) {
    switch (level) {
      case InvalidationLevel.SPECIFIC:
        // 特定资源失效,不级联
        break;

      case InvalidationLevel.CATEGORY:
        // 分类级别失效,失效相关列表
        revalidateTag("lists");
        break;

      case InvalidationLevel.GLOBAL:
        // 全局失效,失效所有相关缓存
        revalidateTag("global");
        revalidatePath("/");
        break;
    }
  }
}

// 产品更新的失效策略
export async function invalidateProduct(
  slug: string,
  changes: {
    info?: boolean; // 基本信息变化
    stock?: boolean; // 库存变化
    price?: boolean; // 价格变化
    category?: boolean; // 分类变化
  }
) {
  const strategies: InvalidationStrategy[] = [];

  // 基本信息变化 - 特定级别
  if (changes.info) {
    strategies.push({
      level: InvalidationLevel.SPECIFIC,
      paths: [`/products/${slug}`],
      tags: [`product-${slug}`],
    });
  }

  // 库存变化 - 特定级别
  if (changes.stock) {
    strategies.push({
      level: InvalidationLevel.SPECIFIC,
      tags: [`product-stock-${slug}`],
    });
  }

  // 价格变化 - 分类级别
  if (changes.price) {
    strategies.push({
      level: InvalidationLevel.CATEGORY,
      paths: [`/products/${slug}`, "/products"],
      tags: [`product-${slug}`, "products"],
      cascade: true,
    });
  }

  // 分类变化 - 全局级别
  if (changes.category) {
    strategies.push({
      level: InvalidationLevel.GLOBAL,
      paths: [`/products/${slug}`, "/products", "/"],
      tags: [`product-${slug}`, "products", "categories"],
      cascade: true,
    });
  }

  // 执行所有失效策略
  for (const strategy of strategies) {
    await invalidateCache(strategy);
  }
}
```

### 1. 失效粒度控制

## 注意事项

### 1. 避免缓存雪崩

缓存雪崩是指大量缓存同时失效,导致请求直接打到数据库:

```typescript
// ❌ 所有缓存同时失效
export const revalidate = 3600; // 所有页面1小时后同时失效

// ✅ 添加随机偏移
export const revalidate = 3600 + Math.floor(Math.random() * 600); // 3600-4200秒
```

### 2. 控制失效频率

避免过于频繁的失效:

```typescript
// lib/cache/throttle.ts
const lastInvalidation = new Map<string, number>();

export function throttleInvalidation(key: string, minInterval: number = 60000) {
  const now = Date.now();
  const last = lastInvalidation.get(key) || 0;

  if (now - last < minInterval) {
    return false; // 跳过失效
  }

  lastInvalidation.set(key, now);
  return true; // 允许失效
}

// 使用
export async function updateProduct(slug: string) {
  await db.product.update({ where: { slug }, data: {} });

  if (throttleInvalidation(`product-${slug}`, 60000)) {

}
```

## 常见问题

### 1. 为什么失效后还是看到旧数据?

**问题**: 调用了 revalidatePath 但页面还是显示旧数据

**解答**:

可能的原因:

1. 浏览器缓存: 清除浏览器缓存
2. CDN 缓存: CDN 可能还有缓存,需要等待 CDN 刷新
3. 失效未生效: 检查失效代码是否执行
4. 路径不匹配: 确保路径完全匹配

```typescript
// 确保路径匹配
revalidatePath("/posts/hello-world"); // 精确匹配
revalidatePath("/posts", "layout"); // 失效布局
revalidatePath("/posts", "page"); // 失效页面(默认)
```

### 2. 如何批量失效多个页面?

**问题**: 需要失效大量相关页面

**解答**:

使用标签批量失效:

```typescript
// 为相关页面添加相同标签
export const getPost = unstable_cache(
  async (slug: string) => {
    /* ... */
  },
  ["post"],
  {
    tags: ["posts", `post-${slug}`, "content"],
  }
);

// 批量失效
revalidateTag("posts"); // 失效所有文章
revalidateTag("content"); // 失效所有内容
```

### 3. 失效会影响性能吗?

**问题**: 频繁失效会不会影响网站性能?

**解答**:

失效本身很快,但重新生成页面需要时间:

```typescript
// 使用后台重新生成
export const revalidate = 3600; // 后台重新生成,不影响用户

// 避免同步失效大量页面
export async function batchUpdate(items: any[]) {
  for (const item of items) {
    await db.update(item);
  }
  // 批量失效而不是逐个失效
  revalidateTag("items");
}
```

### 4. 如何处理依赖关系?

**问题**: A 更新时需要失效 B 和 C

**解答**:

建立依赖关系映射:

```typescript
const dependencies = {
  author: ["posts", "comments"],
  category: ["posts", "products"],
  product: ["cart", "orders"],
};

export async function invalidateWithDependencies(resource: string, id: string) {
  // 失效资源本身
  revalidateTag(`${resource}-${id}`);

  // 失效依赖资源
  const deps = dependencies[resource] || [];
  for (const dep of deps) {
    revalidateTag(dep);
  }
}
```

## 总结

缓存失效策略是缓存系统的核心,直接影响数据一致性和用户体验。通过本文的学习,我们掌握了:

### 核心概念

1. **失效策略类型**: 基于时间、基于事件、基于依赖、基于条件
2. **失效粒度**: 精确失效、分类失效、批量失效、全局失效
3. **失效时机**: 立即失效、延迟失效、条件失效
4. **级联失效**: 处理资源之间的依赖关系

### 实战应用

我们通过四个实战场景学习了失效策略:

1. **电商系统**: 产品、库存、分类的多层失效

3. **用户系统**: 用户资料、设置、关注的失效处理
4. **分层架构**: 特定、分类、全局三层失效策略

### 最佳实践

1. **控制失效粒度**: 精确失效,避免过度失效
2. **选择失效时机**: 立即失效 vs 延迟失效
3. **最小化失效范围**: 只失效必要的缓存

通过合理设计缓存失效策略,可以确保数据一致性,同时最大化缓存的性能优势。

## 附录: 缓存失效决策树

### 何时使用哪种失效策略

```
数据更新
  ├─ 是否需要立即反映?
  │   ├─ 是 → 使用 revalidatePath/revalidateTag
  │   └─ 否 → 使用 revalidate (TTL)
  │
  ├─ 影响范围有多大?
  │   ├─ 单个资源 → 精确失效
  │   ├─ 一组资源 → 标签失效
  │   └─ 全局影响 → 全局失效
  │
  └─ 是否有依赖关系?
      ├─ 是 → 级联失效
      └─ 否 → 独立失效
```

| -------- | -------- | ------------- | ------- |
| 低       | 小       | 基于事件      | 24 小时 |
| 低       | 大       | 基于时间      | 12 小时 |
| 中       | 小       | 基于事件      | 1 小时  |
| 中       | 大       | 基于事件+时间 | 30 分钟 |
| 高       | 小       | 基于事件      | 5 分钟  |
| 高       | 大       | 基于时间      | 1 分钟  |

### 性能优化建议

```typescript
// 1. 使用批量失效而不是逐个失效
// ❌ 不好的做法
for (const post of posts) {
  revalidatePath(`/posts/${post.slug}`);
}

// ✅ 好的做法
revalidateTag("posts");

// 2. 避免重复失效
const invalidated = new Set<string>();

export function smartInvalidate(path: string) {
  if (invalidated.has(path)) {
    return; // 已经失效过了
  }

  revalidatePath(path);
  invalidated.add(path);

  // 清理记录
  setTimeout(() => invalidated.delete(path), 60000);
}

// 3. 使用防抖
import { debounce } from "lodash";

const debouncedInvalidate = debounce((path: string) => {
  revalidatePath(path);
}, 1000);

export function updateData(path: string) {
  // 多次调用只会执行一次失效
  debouncedInvalidate(path);
}

// 4. 分批失效
export async function batchInvalidate(paths: string[]) {
  const batchSize = 10;

  for (let i = 0; i < paths.length; i += batchSize) {
    const batch = paths.slice(i, i + batchSize);

    for (const path of batch) {
      revalidatePath(path);
    }

    // 避免一次性失效太多
    if (i + batchSize < paths.length) {
      await new Promise((resolve) => setTimeout(resolve, 100));
    }
  }
}
```
