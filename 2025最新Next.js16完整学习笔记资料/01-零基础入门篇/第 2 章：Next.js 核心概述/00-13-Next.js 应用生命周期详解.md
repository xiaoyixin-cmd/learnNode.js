# Next.js 应用生命周期详解

## 1. 概述 (Overview)

### 1.1 生命周期总览

#### 1.1.1 完整流程图

```
用户请求
  ↓
DNS 解析
  ↓
CDN/边缘缓存 (如有)
  ↓
Next.js 服务器
  ↓
中间件 (Middleware)
  ↓
路由匹配
  ↓
数据获取
  ↓
组件渲染
  ↓
HTML 生成
  ↓
流式传输
  ↓
客户端水合
  ↓
交互就绪
```

#### 1.1.2 关键阶段

Next.js 应用生命周期分为以下关键阶段:

**1. 请求阶段 (Request Phase)**

- 请求到达服务器
- 中间件执行
- 路由匹配
- 权限验证

**2. 数据获取阶段 (Data Fetching Phase)**

- Server Component 数据获取
- 并行/串行请求
- 缓存查询
- 数据预处理

**3. 渲染阶段 (Rendering Phase)**

- 组件树构建
- RSC Payload 生成
- HTML 生成
- 流式传输

**4. 客户端阶段 (Client Phase)**

- JavaScript 加载
- 水合 (Hydration)
- 交互就绪
- 客户端导航

**5. 缓存阶段 (Caching Phase)**

- 请求缓存
- 数据缓存
- 完整路由缓存
- 路由器缓存

### 1.2 渲染模式对比

#### 1.2.1 不同渲染模式的生命周期

**静态生成 (SSG)**:

```
构建时:
  数据获取 → 组件渲染 → HTML 生成 → 保存到磁盘

运行时:
  请求 → 读取缓存 HTML → 返回给客户端 → 水合

特点:
  - 构建时执行一次
  - 运行时极快
  - 适合静态内容
```

**服务端渲染 (SSR)**:

```
每次请求:
  请求 → 数据获取 → 组件渲染 → HTML 生成 → 返回 → 水合

特点:
  - 每次请求都渲染
  - 总是最新数据
  - 响应时间取决于数据获取
```

**增量静态再生 (ISR)**:

```
首次请求:
  请求 → 读取缓存 HTML → 返回 → 后台重新验证

后台:
  数据获取 → 组件渲染 → HTML 生成 → 更新缓存

特点:
  - 结合 SSG 和 SSR 优势
  - 定时更新
  - 高性能 + 新鲜数据
```

**部分预渲染 (PPR - Next.js 16)**:

```
静态部分 (构建时):
  组件渲染 → HTML 生成 → 缓存

动态部分 (请求时):
  数据获取 → 组件渲染 → 流式传输

特点:
  - 静态 shell 瞬间加载
  - 动态内容逐步显示
  - 最佳用户体验
```

#### 2.2.1 中间件概念

#### 2.2.2 中间件用例

**1. 身份验证**

```tsx
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const token = request.cookies.get("auth-token")?.value;

  // 保护的路径
  if (request.nextUrl.pathname.startsWith("/dashboard")) {
    if (!token) {
      return NextResponse.redirect(new URL("/login", request.url));
    }

    // 验证 token (简化版)
    if (!isValidToken(token)) {
      return NextResponse.redirect(new URL("/login", request.url));
    }
  }

  return NextResponse.next();
}

function isValidToken(token: string): boolean {
  // 实际应用中应该验证 JWT
  return token.length > 0;
}

export const config = {
  matcher: ["/dashboard/:path*"],
};
```

**2. 国际化重定向**

```tsx
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const locales = ["en", "zh", "ja"];
const defaultLocale = "en";

export function middleware(request: NextRequest) {
  const pathname = request.nextUrl.pathname;

  // 检查路径是否包含语言前缀
  const hasLocale = locales.some(
    (locale) => pathname.startsWith(`/${locale}/`) || pathname === `/${locale}`
  );

  if (!hasLocale) {
    // 从 Accept-Language 头获取语言
    const locale = getLocale(request);

    // 重定向到带语言前缀的 URL
    return NextResponse.redirect(new URL(`/${locale}${pathname}`, request.url));
  }
}

function getLocale(request: NextRequest): string {
  const acceptLanguage = request.headers.get("accept-language");

  if (acceptLanguage) {
    const preferredLocale = acceptLanguage.split(",")[0].split("-")[0];
    if (locales.includes(preferredLocale)) {
      return preferredLocale;
    }
  }

  return defaultLocale;
}
```

**3. A/B 测试**

```tsx
export function middleware(request: NextRequest) {

  }

  return NextResponse.next();
}
```

**4. 速率限制**

```tsx
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const rateLimitMap = new Map<string, number[]>();

export function middleware(request: NextRequest) {
  const ip = request.ip || "unknown";
  const now = Date.now();
  const windowMs = 60 * 1000; // 1分钟
  const maxRequests = 100;

  // 获取该 IP 的请求记录
  const requests = rateLimitMap.get(ip) || [];

  // 过滤掉时间窗口外的请求
  const recentRequests = requests.filter((time) => now - time < windowMs);

  if (recentRequests.length >= maxRequests) {
    return NextResponse.json({ error: "Too many requests" }, { status: 429 });
  }

  // 添加当前请求
  recentRequests.push(now);
  rateLimitMap.set(ip, recentRequests);

  return NextResponse.next();
}
```

**5. 请求/响应修改**

```tsx
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const response = NextResponse.next();

  // 添加自定义响应头

  return response;
}
```

#### 2.2.3 中间件执行顺序

多个中间件的执行流程:

```tsx
// middleware.ts (根目录)
export function middleware(request: NextRequest) {
  console.log("1. Root middleware");

  const response = NextResponse.next();

  console.log("4. Root middleware response");

  return response;
}

// 实际执行顺序:
// 1. 根中间件 (请求阶段)
// 2. 路由匹配
// 3. 页面/API 处理
// 4. 根中间件 (响应阶段)
```

链式中间件模式:

```tsx
// lib/middleware-chain.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

type Middleware = (
  request: NextRequest
) => NextResponse | Promise<NextResponse>;

export function chain(middlewares: Middleware[]) {
  return async (request: NextRequest) => {
    for (const middleware of middlewares) {
      const response = await middleware(request);

      // 如果中间件返回响应 (非 next),停止链
      if (
        response.status !== 200 ||
        response.headers.get("x-middleware-next") !== "true"
      ) {
        return response;
      }
    }

    return NextResponse.next();
  };
}

// middleware.ts
import { chain } from "./lib/middleware-chain";
import { authMiddleware } from "./middlewares/auth";
import { loggingMiddleware } from "./middlewares/logging";
import { rateLimitMiddleware } from "./middlewares/rate-limit";

export default chain([loggingMiddleware, rateLimitMiddleware, authMiddleware]);
```

### 2.3 路由匹配

#### 2.3.1 路由匹配流程

```
请求: /blog/123

匹配过程:
  1. 检查完全匹配: /blog/123
  2. 检查动态路由: /blog/[id]
  3. 检查捕获所有: /blog/[...slug]
  4. 检查可选捕获: /blog/[[...slug]]
  5. 未匹配 → 404
```

示例:

```
app/
├── blog/
│   ├── page.tsx                → /blog
│   ├── [id]/
│   │   └── page.tsx            → /blog/:id
│   └── [...slug]/
│       └── page.tsx            → /blog/*

请求匹配:
  /blog → blog/page.tsx
  /blog/123 → blog/[id]/page.tsx
  /blog/2024/01/post → blog/[...slug]/page.tsx
```

#### 2.3.2 路由优先级

```
优先级 (从高到低):

1. 静态路由
   app/dashboard/page.tsx → /dashboard

2. 动态路由
   app/blog/[id]/page.tsx → /blog/:id

3. 捕获所有路由
   app/blog/[...slug]/page.tsx → /blog/*

4. 可选捕获所有路由
   app/blog/[[...slug]]/page.tsx → /blog 或 /blog/*

5. 根捕获
   app/[...not-found]/page.tsx → /*
```

实际示例:

```
app/
├── shop/
│   ├── page.tsx              # 优先级 1: /shop
│   ├── new/
│   │   └── page.tsx          # 优先级 1: /shop/new
│   ├── [id]/
│   │   └── page.tsx          # 优先级 2: /shop/:id
│   └── [...slug]/
│       └── page.tsx          # 优先级 3: /shop/*

请求处理:
  /shop → shop/page.tsx
  /shop/new → shop/new/page.tsx
  /shop/123 → shop/[id]/page.tsx
  /shop/category/electronics → shop/[...slug]/page.tsx
```

#### 3.2.2 缓存层级

Next.js 的多层缓存:

```
请求 → Request Memoization (请求去重)
       ↓
      → Data Cache (数据缓存)
       ↓
      → Full Route Cache (完整路由缓存)
       ↓
      → Router Cache (路由器缓存)
       ↓
      → CDN/Browser Cache (CDN/浏览器缓存)
```

详细说明:

**1. Request Memoization**

```tsx
// 在单次渲染中自动去重
async function getUser(id: string) {
  return fetch(`https://api.example.com/users/${id}`);
}

export default async function Page() {
  const user1 = await getUser("123");
  const user2 = await getUser("123"); // 复用
  const user3 = await getUser("123"); // 复用

  // 实际只发送一次请求
}
```

**2. Data Cache**

```tsx
// 跨请求的数据缓存
const products = await fetch("https://api.example.com/products", {
  next: { revalidate: 3600 }, // 缓存1小时
});

// 首次请求: 获取数据并缓存
// 后续请求: 返回缓存数据
// 1小时后: 下次请求触发重新验证
```

**3. Full Route Cache**


## 4. 渲染阶段 (Rendering Phase)

### 4.1 服务端渲染流程

#### 4.1.1 React Server Components 渲染

```
Server Component 执行流程:

1. 组件函数执行
   ↓
2. 数据获取 (await fetch/db)
   ↓
3. 生成 React Element Tree
   ↓
4. 序列化为 RSC Payload
   ↓
5. 发送到客户端
```

## 5. 缓存策略详解 (Caching Strategies)

### 5.1 完整缓存架构

#### 5.1.1 四层缓存

```
Layer 1: Request Memoization (请求去重)
  - 作用域: 单次服务器请求
  - 生命周期: 请求结束
  - 用途: 去重相同数据获取

Layer 2: Data Cache (数据缓存)
  - 作用域: 跨请求
  - 生命周期: 持久化 (直到重新验证)
  - 用途: 缓存数据获取结果

Layer 3: Full Route Cache (完整路由缓存)
  - 作用域: 服务器
  - 生命周期: 持久化 (直到重新部署/验证)
  - 用途: 缓存渲染结果 (HTML + RSC)

Layer 4: Router Cache (路由器缓存)
  - 作用域: 客户端
  - 生命周期: 会话或时间
  - 用途: 缓存访问过的路由
```

#### 5.1.2 缓存交互

```tsx
// 示例: 产品列表页面

// 1. Request Memoization
async function getProducts() {
  return fetch("https://api.example.com/products");
}

export default async function Page() {
  const products1 = await getProducts(); // 发送请求
  const products2 = await getProducts(); // 复用请求

  // 2. Data Cache
  const cachedProducts = await fetch("https://api.example.com/products", {
    next: { revalidate: 3600 }, // 缓存1小时
  });

  // 3. Full Route Cache
  // 整个页面被缓存 (如果是静态生成)

  return <ProductList products={products1} />;
}

// 4. Router Cache
// 用户访问后,页面在客户端缓存
```

## 8. 常见问题 (FAQ)

**Q1: Next.js 应用的请求流程是怎样的?**

A: 完整流程:

```
用户请求 → DNS → CDN → 服务器
  ↓
中间件执行
  ↓
路由匹配
  ↓
数据获取
  ↓
组件渲染
  ↓
HTML 生成
  ↓
流式传输
  ↓
客户端水合
  ↓
交互就绪
```

**Q2: 如何优化首次加载时间?**

A: 多种策略:

1. 使用 Streaming SSR
2. 启用部分预渲染 (PPR)
3. 优化数据获取 (并行)
4. 合理使用缓存
5. 代码分割
6. 图片优化

**Q3: 什么时候使用 SSG vs SSR?**

A: 决策树:

```
内容更新频率?
  ├─ 很少变化 → SSG
  ├─ 定期变化 → ISR
  ├─ 实时变化 → SSR
  └─ 混合场景 → PPR
```

**Q4: 如何调试 Server Components?**

A: 调试方法:

1. 服务端 console.log
2. React DevTools
3. Next.js DevTools MCP
4. 性能分析工具

**Q5: 缓存如何工作?**

A: 四层缓存:

1. Request Memoization - 请求去重
2. Data Cache - 数据缓存
3. Full Route Cache - 路由缓存
4. Router Cache - 客户端缓存
