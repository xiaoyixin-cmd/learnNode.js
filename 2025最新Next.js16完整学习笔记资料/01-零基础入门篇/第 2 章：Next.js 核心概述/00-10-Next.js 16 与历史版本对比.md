# Next.js 16 与历史版本对比

## 1. 概述 (Overview)

### 1.1 Next.js 版本演进概览

Next.js 自 2016 年 10 月由 Vercel（前 ZEIT）发布以来，已经走过了近 9 年的发展历程。从最初的简单 SSR 框架，逐步演进为功能完备的全栈 React 框架。其版本演进呈现以下特点：

#### 1.1.1 演进阶段划分

**第一阶段：奠基期（v1.0 - v9.x，2016-2020）**

- **核心目标**：解决 React 应用的 SSR 需求
- **技术特征**：
  - 基于文件系统的路由
  - 自动代码分割
  - 内置 CSS 支持
  - API Routes（v9 引入）
- **典型版本**：
  - v1.0（2016.10）：首个正式版本，提供基础 SSR 能力
  - v7.0（2018.09）：引入 Dynamic Import，优化代码分割
  - v9.0（2019.07）：引入 API Routes，开启全栈能力
  - v9.3（2020.03）：引入 getStaticProps/getServerSideProps

**第二阶段：性能优化期（v10.x - v12.x，2020-2022）**

- **核心目标**：提升性能与开发体验
- **技术特征**：
  - Image/Script 组件优化
  - Webpack 5 支持
  - Middleware 引入
  - SWC 编译器替代 Babel
  - React 18 支持
- **典型版本**：
  - v10.0（2020.10）：内置 Image Optimization
  - v11.0（2021.06）：改进 Script Loading
  - v12.0（2021.10）：引入 SWC、Middleware、ESM 支持

**第三阶段：架构革命期（v13.x - v14.x，2022-2024）**

- **核心目标**：引入 App Router，重构应用架构
- **技术特征**：
  - App Router（基于 React Server Components）
  - Turbopack（Rust 驱动的打包工具）
  - Server Actions
  - Streaming SSR
  - Layouts/Templates
- **典型版本**：
  - v13.0（2022.10）：引入 App Router（beta）
  - v13.4（2023.05）：App Router 稳定版
  - v14.0（2023.10）：Turbopack Dev 模式稳定

**第四阶段：成熟优化期（v15.x - v16.x，2024-2025）**

- **核心目标**：性能极致优化、开发体验升级
- **技术特征**：
  - Turbopack 全面稳定
  - Partial Prerendering（PPR）
  - React 19 深度集成
  - 新缓存策略（Cache Components）
  - 异步请求 API
  - DevTools MCP（AI 辅助调试）
- **典型版本**：
  - v15.0（2024.10）：React 19 支持、异步请求 API
  - v16.0（2025.10 预计）：Turbopack 稳定、Cache Components

#### 1.1.2 版本号语义化规则

Next.js 遵循 **语义化版本控制 2.0.0**（Semantic Versioning）规范：

```
主版本号.次版本号.修订号 (MAJOR.MINOR.PATCH)
```

- **主版本号（MAJOR）**：包含不兼容的 API 变更
  - 示例：v12 → v13（引入 App Router）
- **次版本号（MINOR）**：向下兼容的功能新增
  - 示例：v13.0 → v13.4（App Router 稳定）
- **修订号（PATCH）**：向下兼容的问题修复
  - 示例：v13.4.0 → v13.4.1

**Canary 版本**：

```bash
npm install next@canary
```

- 包含即将发布的最新特性
- 用于测试和早期采用
- 更新频率：每周多次

### 1.2 版本支持策略

#### 1.2.1 官方维护周期

Next.js 采用 **滚动维护模式**，而非传统 LTS（长期支持）模式：

| 版本类型         | 支持周期 | 更新内容                       |
| ---------------- | -------- | ------------------------------ |
| **最新稳定版**   | 持续支持 | 功能更新 + 安全修复 + Bug 修复 |
| **前一个主版本** | 6 个月   | 安全修复 + 关键 Bug 修复       |
| **更早版本**     | 不再支持 | 建议升级                       |

**实际案例**：

- Next.js 15 发布后（2024.10）
  - v15.x：全面支持（功能 + 安全 + Bug）
  - v14.x：维护至 2025.04（安全 + 关键 Bug）
  - v13.x 及更早：不再维护

#### 1.2.2 依赖版本兼容性

| Next.js 版本 | React 版本   | Node.js 版本 | TypeScript 版本 |
| ------------ | ------------ | ------------ | --------------- |
| v16.x        | 19.x         | ≥ 18.17      | ≥ 4.5           |
| v15.x        | 18.2+ / 19.x | ≥ 18.17      | ≥ 4.5           |
| v14.x        | 18.2+        | ≥ 18.17      | ≥ 4.5           |
| v13.x        | 18.2+        | ≥ 16.8       | ≥ 4.5           |

**Node.js 版本选择建议**：

- **生产环境**：使用 LTS 版本（当前为 v20.x）
- **开发环境**：可使用最新 Current 版本（v21.x）
- **最低要求**：Next.js 15/16 需要 Node.js 18.17+

## 2. 关键版本深度对比 (Key Version Comparison)

### 2.1 Next.js 13：App Router 架构革命

#### 2.1.1 版本概览

- **发布时间**：2022 年 10 月 25 日
- **重大突破**：引入 App Router（基于 React Server Components）
- **架构变革**：从 Pages Router 向 App Router 过渡
- **核心理念**：服务端优先、流式渲染、细粒度缓存

#### 2.1.2 核心特性详解

**特性一：App Router（Beta）**

App Router 是 Next.js 13 最重要的创新，基于 React Server Components（RSC）架构：

```
项目结构对比：

Pages Router (v12)          App Router (v13)
pages/                      app/
├── index.js               ├── page.js
├── about.js               ├── about/
├── blog/                  │   └── page.js
│   ├── [slug].js         └── blog/
│   └── index.js              ├── page.js
└── _app.js                   ├── layout.js
                              └── [slug]/
                                  └── page.js
```

**核心概念**：

1. **Server Components（服务端组件）**：

```tsx
// app/blog/page.tsx - 默认为 Server Component
export default async function BlogPage() {
  // 直接在组件内获取数据，无需 getServerSideProps
  const posts = await fetch("https://api.example.com/posts").then((res) =>
    res.json()
  );

  return (
    <div>
      <h1>博客文章</h1>
      {posts.map((post) => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  );
}
```

**优势**：

- 零客户端 JS 负担（服务端渲染 HTML）
- 直接访问后端资源（数据库、文件系统）
- 自动代码分割
- 更好的 SEO

2. **Client Components（客户端组件）**：

```tsx
// app/components/Counter.tsx
"use client"; // 显式声明为客户端组件

import { useState } from "react";

export default function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>点击次数: {count}</button>;
}
```

**使用场景**：

- 需要交互性（useState、useEffect）
- 需要浏览器 API（localStorage、window）
- 需要事件监听器

3. **Layouts（布局系统）**：

```tsx
// app/layout.tsx - 根布局
export default function RootLayout({ children }) {
  return (
    <html lang="zh-CN">
      <body>
        <Header />
        <main>{children}</main>
        <Footer />
      </body>
    </html>
  );
}

// app/dashboard/layout.tsx - 嵌套布局
export default function DashboardLayout({ children }) {
  return (
    <div className="dashboard">
      <Sidebar />
      <div className="content">{children}</div>
    </div>
  );
}
```

**特点**：

- 布局在路由切换时保持状态
- 支持嵌套布局
- 自动优化重渲染

**特性二：Turbopack（Alpha）**

Turbopack 是用 Rust 编写的下一代打包工具，旨在替代 Webpack：

**性能对比**（官方数据，v13.0 时期）：

| 指标           | Webpack | Turbopack | 提升倍数  |
| -------------- | ------- | --------- | --------- |
| 冷启动         | 16.5s   | 1.8s      | **9.2x**  |
| 热更新         | 2.3s    | 0.025s    | **92x**   |
| 大型应用冷启动 | 49s     | 3.6s      | **13.6x** |

**使用方式**：

```json
// package.json
{
  "scripts": {
    "dev": "next dev --turbo"
  }
}
```

**注意事项（v13 时期）**：

- 仅支持 Dev 模式（生产构建仍用 Webpack）
- 部分 Webpack 插件不兼容
- 处于 Alpha 阶段，可能存在稳定性问题

**特性三：流式渲染与 Suspense**

Next.js 13 深度集成 React 18 的 Suspense 特性，实现流式 SSR：

```tsx
// app/dashboard/page.tsx
import { Suspense } from "react";

export default function DashboardPage() {
  return (
    <div>
      <h1>仪表盘</h1>

      {/* 快速渲染静态内容 */}
      <QuickStats />

      {/* 慢速数据加载时显示骨架屏 */}
      <Suspense fallback={<ChartSkeleton />}>
        <Chart />
      </Suspense>

      <Suspense fallback={<TableSkeleton />}>
        <DataTable />
      </Suspense>
    </div>
  );
}

// Chart 组件会在数据就绪后流式传输到客户端
async function Chart() {
  const data = await fetchChartData(); // 可能需要 2-3 秒
  return <ChartComponent data={data} />;
}
```

**工作原理**：

1. 服务器立即发送 HTML Shell（包含 QuickStats）
2. 客户端看到页面框架和骨架屏
3. Chart 数据就绪后，服务器流式传输 HTML
4. 客户端自动替换骨架屏

**优势**：

- **更快的首字节时间（TTFB）**：不等待所有数据
- **更好的用户体验**：渐进式内容加载
- **更高的服务器吞吐量**：减少阻塞时间

**特性四：新的数据获取方式**

App Router 引入新的数据获取模式，取代 getServerSideProps/getStaticProps：

**1. 扩展的 fetch API**：

```tsx
// 自动缓存（类似 getStaticProps）
const staticData = await fetch("https://api.example.com/data", {
  cache: "force-cache", // 默认行为
});

// 每次请求重新获取（类似 getServerSideProps）
const dynamicData = await fetch("https://api.example.com/user", {
  cache: "no-store",
});

// 定时重新验证（类似 ISR）
const revalidatedData = await fetch("https://api.example.com/posts", {
  next: { revalidate: 60 }, // 每 60 秒重新验证
});
```

**2. 数据库直接查询**：

```tsx
// app/products/page.tsx
import { db } from "@/lib/database";

export default async function ProductsPage() {
  // 直接在服务端组件中查询数据库
  const products = await db.product.findMany({
    where: { published: true },
    orderBy: { createdAt: "desc" },
  });

  return (
    <div>
      {products.map((product) => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  );
}
```

**3. 并行数据获取**：

```tsx
export default async function Page() {
  // 并行发起多个请求
  const [user, posts, comments] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchComments(),
  ]);

  return (
    <div>
      <UserProfile user={user} />
      <PostList posts={posts} />
      <CommentList comments={comments} />
    </div>
  );
}
```

#### 2.1.3 迁移注意事项

**1. App Router 与 Pages Router 共存**：

Next.js 13 支持两种路由模式并存：

```
项目结构：
app/                  # App Router（新）
├── page.js
└── dashboard/
    └── page.js

pages/                # Pages Router（旧）
├── index.js
└── api/
    └── hello.js
```

**路由优先级**：

- App Router 优先级高于 Pages Router
- `app/about/page.js` 会覆盖 `pages/about.js`
- API Routes 仍需保留在 `pages/api/`（v13.4 后可用 Route Handlers）

**2. 渐进式迁移策略**：

```
阶段一：新功能使用 App Router
- 保留现有 pages/ 目录
- 新页面在 app/ 中开发
- 逐步熟悉新架构

阶段二：迁移核心页面
- 从简单页面开始（如 About、Contact）
- 迁移布局组件
- 测试数据获取逻辑

阶段三：迁移复杂功能
- 迁移动态路由
- 迁移数据密集型页面
- 迁移 API Routes → Route Handlers

阶段四：清理旧代码
- 删除 pages/ 目录（保留 pages/api 可选）
- 统一代码风格
- 优化性能
```

**3. 常见迁移陷阱**：

| 问题            | 原因                    | 解决方案              |
| --------------- | ----------------------- | --------------------- |
| `useState` 报错 | Server Component 中使用 | 添加 `'use client'`   |
| 布局不保持状态  | 未使用 layout.js        | 创建 layout.js 文件   |
| 数据获取报错    | 使用旧 API              | 改用 fetch 或直接查询 |
| CSS 不生效      | 样式导入方式变化        | 检查 CSS Modules 导入 |

**特性二：Server Actions 稳定版**

Server Actions 是 Next.js 14 的核心特性，提供类型安全的服务端函数调用：

**1. 基础用法**：

```tsx
// app/actions.ts
"use server";

export async function createPost(formData: FormData) {
  const title = formData.get("title") as string;
  const content = formData.get("content") as string;

  // 直接操作数据库
  const post = await db.post.create({
    data: { title, content },
  });

  // 重新验证缓存
  revalidatePath("/blog");

  return { success: true, postId: post.id };
}

// app/new-post/page.tsx
import { createPost } from "../actions";

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" placeholder="标题" required />
      <textarea name="content" placeholder="内容" required />
      <button type="submit">发布</button>
    </form>
  );
}
```

**2. 渐进增强**：

```tsx
// 无 JavaScript 时仍可提交表单
// JavaScript 加载后自动升级为 AJAX 提交
export default function ContactForm() {
  return (
    <form action={sendEmail}>
      <input name="email" type="email" required />
      <textarea name="message" required />
      <button type="submit">发送</button>
    </form>
  );
}

// 服务器端处理
("use server");
export async function sendEmail(formData: FormData) {
  const email = formData.get("email");
  const message = formData.get("message");

  await emailService.send({ to: email, message });
  redirect("/thank-you");
}
```

**3. 客户端组件中调用**：

```tsx
"use client";

import { createPost } from "./actions";
import { useState } from "react";

export default function PostForm() {
  const [isLoading, setIsLoading] = useState(false);

  async function handleSubmit(formData: FormData) {
    setIsLoading(true);
    try {
      const result = await createPost(formData);
      if (result.success) {
        alert("发布成功！");
      }
    } finally {
      setIsLoading(false);
    }
  }

  return (
    <form action={handleSubmit}>
      <input name="title" />
      <button disabled={isLoading}>{isLoading ? "发布中..." : "发布"}</button>
    </form>
  );
}
```

**4. 数据重新验证**：

```tsx
"use server";

import { revalidatePath, revalidateTag } from "next/cache";

export async function updateProduct(productId: string, data: any) {
  await db.product.update({
    where: { id: productId },
    data,
  });

  // 重新验证特定路径
  revalidatePath("/products");
  revalidatePath(`/products/${productId}`);

  // 重新验证特定标签的所有请求
  revalidateTag("products");
}
```

**安全性**：

- Server Actions 只能在服务器端执行
- 自动生成的 Action ID 防止伪造
- 支持 CSRF 保护
- 可集成身份验证中间件

**特性三：Partial Prerendering（预览）**

Partial Prerendering（PPR）是 Next.js 14 引入的实验性功能，结合静态与动态渲染：

**工作原理**：

```tsx
// app/product/[id]/page.tsx
export const experimental_ppr = true;

export default function ProductPage({ params }) {
  return (
    <div>
      {/* 静态部分：构建时生成 */}
      <ProductImages productId={params.id} />
      <ProductDescription productId={params.id} />

      {/* 动态部分：请求时生成 */}
      <Suspense fallback={<PriceSkeleton />}>
        <ProductPrice productId={params.id} />
      </Suspense>

      <Suspense fallback={<StockSkeleton />}>
        <StockStatus productId={params.id} />
      </Suspense>
    </div>
  );
}
```

**渲染流程**：

1. 构建时：生成静态 HTML Shell（图片、描述）
2. 请求时：
   - 立即返回静态 Shell
   - 流式传输动态内容（价格、库存）
3. 客户端：渐进式替换 Suspense fallback

**优势**：

- **极快的 TTFB**：静态内容秒级返回
- **动态数据实时性**：价格、库存等实时更新
- **更低的服务器成本**：减少动态渲染压力

**启用方式**：

```js
// next.config.js
module.exports = {
  experimental: {
    ppr: true,
  },
};
```

```tsx

export default async function sitemap() {
  const posts = await getAllPosts();

  const postUrls = posts.map((post) => ({
```

```tsx
  async function ProductList() {
```

```tsx
        <UserProfile user={user} />

        <NotificationList notifications={notifications} />

export async function updateUserProfile(data) {
```

## 4. 升级路径与策略 (Upgrade Guide)

### 4.1 升级决策树

```
开始升级评估
│
├─ 当前版本 < v12？
│  ├─ 是 → 建议跳过中间版本，直接升级至 v15/v16
│  │       理由：v13 引入的 App Router 架构变化巨大，
│  │             中间版本升级成本高，不如一步到位
│  └─ 否 → 继续评估
│
├─ 项目规模如何？
│  ├─ 小型（< 50 页面）→ 建议直接升级至最新稳定版
│  ├─ 中型（50-200 页面）→ 渐进式升级（v12→v14→v15）
│  └─ 大型（> 200 页面）→ 分阶段升级，保持两个版本共存
│
├─ 是否使用 Pages Router？
│  ├─ 是 → 可继续使用，无需强制迁移至 App Router
│  │       但新功能建议使用 App Router
│  └─ 否（已使用 App Router）→ 升级成本较低
│
├─ 依赖库兼容性如何？
│  ├─ 有不兼容库 → 等待库更新或寻找替代方案
│  └─ 全部兼容 → 可安全升级
│
└─ 团队技术储备？
   ├─ 熟悉 React 19/Server Components → 建议升级至 v16
   ├─ 熟悉 React 18 → 建议升级至 v15
   └─ 仅熟悉 React 17 → 建议先升级至 v14，学习新概念后再升级
```

| **Module not found: Can't resolve 'next/headers'**    | 在客户端组件中导入了服务端 API           | 移除导入或移至服务端组件                                                  |

## 7. 常见问题 (FAQ)

### 7.1 版本选择问题

**Q1: 我应该选择哪个版本?**

A: 取决于项目需求:

- **新项目**: Next.js 15 或 16(如果愿意承担早期采用风险)
- **现有项目**: 保持当前主版本,跟进补丁更新
- **生产环境**: Next.js 15(经过 6 个月 RC 测试,相对稳定)
- **实验项目**: Next.js 16 Canary(体验最新特性)

**Q2: v13 的 App Router 可以用于生产环境吗?**

A: **不建议**。理由:

- v13 的 App Router 处于 Beta 阶段
- 存在已知的性能和稳定性问题
- v14 已经稳定,没有理由继续使用 v13
- 如果要使用 App Router,至少选择 v14

**Q3: v12 还能用多久?**

A: 官方维护周期:

- **安全更新**: 预计到 2025 年中
- **社区支持**: 会持续更长时间
- **建议**: 2026 年前升级至 v15/v16
- **注意**: v12 缺少许多现代特性(Server Components、Server Actions 等)

### 7.2 升级相关问题

**Q4: 从 v12 升级到 v16 需要多长时间?**

A: 取决于项目规模:

| 项目规模           | 预估时间  | 说明             |
| ------------------ | --------- | ---------------- |
| 小型(< 20 页面)    | 1-2 周    | 简单重构即可     |
| 中型(20-100 页面)  | 1-2 个月  | 分阶段迁移       |
| 大型(> 100 页面)   | 3-6 个月  | 需要详细计划     |
| 超大型(> 500 页面) | 6-12 个月 | 考虑重写部分功能 |

**Q5: 可以跳过中间版本直接升级吗?**

A: **可以,但有条件**:

✅ **推荐跳版本**:

```
v12 → v15/v16 (小型项目)
理由:中间版本变化大,逐步升级反而更复杂
```

⚠️ **需要渐进升级**:

```
v9 → v12 → v15 (大型项目)
理由:跨度太大,风险高,建议分步进行
```

**Q6: 升级后性能变差了怎么办?**

A: 常见原因和解决方案:

1. **fetch 缓存失效**(v15):

```tsx
// 问题:默认不缓存
const data = await fetch("url");

// 解决:显式指定缓存
const data = await fetch("url", { cache: "force-cache" });
```

2. **Server Components 误用**:

```tsx
// 问题:阻塞渲染
export default async function Page() {
  const slowData = await fetchSlowData(); // 阻塞
  return <div>{slowData}</div>;
}

// 解决:使用 Suspense
export default function Page() {
  return (
    <Suspense fallback={<Loading />}>
      <SlowDataComponent />
    </Suspense>
  );
}
```

3. **未启用新优化特性**:

```js
// next.config.js
module.exports = {
  experimental: {
    ppr: true, // 启用 Partial Prerendering
    turbo: { production: true }, // v16: 启用 Turbopack
  },
};
```

### 7.3 特性使用问题

**Q7: App Router 和 Pages Router 有什么区别?**

A: 核心差异:

| 维度               | Pages Router         | App Router          |
| ------------------ | -------------------- | ------------------- |
| **路由文件**       | `pages/index.js`     | `app/page.js`       |
| **数据获取**       | `getServerSideProps` | Server Components   |
| **布局**           | `_app.js`(全局)      | `layout.js`(嵌套)   |
| **客户端组件**     | 默认                 | 需要 `'use client'` |
| **流式渲染**       | 不支持               | 支持(Suspense)      |
| **Server Actions** | 不支持               | 支持                |
| **学习曲线**       | 低                   | 中高                |

**推荐**: 新项目使用 App Router,旧项目可以继续使用 Pages Router。

**Q8: Server Components 和 Client Components 如何选择?**

A: 决策树:

```
组件需要以下功能吗?
  ├─ 使用 useState/useEffect/useRef? → Client Component
  ├─ 使用事件监听(onClick/onChange)? → Client Component
  ├─ 使用浏览器 API(localStorage/window)? → Client Component
  ├─ 使用 React Hooks? → Client Component
  │
  └─ 以上都不需要 → Server Component(推荐)
      优势:
        - 零客户端 JS
        - 直接访问后端资源
        - 更好的 SEO
        - 更快的首屏加载
```

**示例**:

```tsx
// Server Component(默认)
export default async function ProductPage() {
  const products = await db.product.findMany();

  return (
    <div>
      {products.map((product) => (
        // 客户端组件(需要交互)
        <AddToCartButton key={product.id} productId={product.id} />
      ))}
    </div>
  );
}

// Client Component
("use client");

export function AddToCartButton({ productId }) {
  const [isLoading, setIsLoading] = useState(false);

  const handleClick = async () => {
    setIsLoading(true);
    await addToCart(productId);
    setIsLoading(false);
  };

  return (
    <button onClick={handleClick} disabled={isLoading}>
      {isLoading ? "添加中..." : "加入购物车"}
    </button>
  );
}
```

**Q9: Server Actions 和 API Routes 有什么区别?**

A: 对比:

| 维度         | API Routes                                | Server Actions |
| ------------ | ----------------------------------------- | -------------- |
| **位置**     | `app/api/*`                               | 任意服务端文件 |
| **调用方式** | fetch('https://example.com/api/endpoint') | 直接调用函数   |
| **类型安全** | 需要手动定义                              | 自动类型推导   |
| **表单集成** | 需要手动处理                              | 原生支持       |
| **渐进增强** | 不支持                                    | 支持           |
| **适用场景** | 公开 API、Webhook                         | 内部数据操作   |

**示例对比**:

```tsx
// API Route
// app/api/posts/route.ts
export async function POST(request: Request) {
  const body = await request.json();
  const post = await db.post.create({ data: body });
  return Response.json(post);
}

// 客户端调用
const response = await fetch("/api/posts", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ title: "Hello" }),
});
const post = await response.json();

// ---

// Server Action
// app/actions.ts
("use server");

export async function createPost(formData: FormData) {
  const post = await db.post.create({
    data: {
      title: formData.get("title") as string,
    },
  });
  return post;
}

// 客户端调用(更简单)
<form action={createPost}>
  <input name="title" />
  <button>Submit</button>
</form>;
```

**建议**:

- 内部数据操作 → Server Actions
- 公开 API → API Routes
- Webhook → API Routes
