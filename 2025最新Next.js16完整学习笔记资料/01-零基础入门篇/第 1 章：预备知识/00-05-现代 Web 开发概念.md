# 00-05 现代 Web 开发概念

## 1. 概述 (Overview)

### 1.1 Web 开发的演变

Web 开发经历了从静态页面到动态应用，再到全栈框架的漫长演变。

- **Web 1.0 (静态时代)**: 网页主要由 HTML 构成，内容由服务器生成，用户只能被动浏览（如早期的门户网站）。
- **Web 2.0 (动态交互)**: AJAX 技术的出现使得网页可以局部刷新，JavaScript 赋予了前端更强的交互能力。社交网络、博客兴起，用户成为内容的创造者。
- **Web 3.0 (去中心化与智能)**: 虽然定义尚在发展，但通常涉及区块链、语义网、AI 等技术。
- **现代 Web 开发**: 当前主流的开发模式，强调组件化、工程化、前后端分离与同构渲染（SSR）。Next.js 正是这一阶段的杰出代表。

### 1.2 为什么学习现代 Web 概念

在深入 Next.js 16 之前，必须理解支撑它的底层概念。Next.js 并非凭空产生，而是为了解决传统 Web 开发中的痛点（如 SEO、首屏性能、开发体验）而构建的。如果不理解 HTTP 协议、渲染模式、状态管理等基础，就很难发挥 Next.js 的全部威力。

## 2. 渲染模式详解 (Rendering Patterns)

现代 Web 开发中最核心的决策之一就是选择合适的渲染模式。Next.js 的强大之处在于它支持多种渲染模式的混合使用。

### 2.1 CSR (Client-Side Rendering) - 客户端渲染

- **原理**: 服务器仅返回一个空的 HTML 骨架（通常只包含 `<div id="root"></div>`）和 JavaScript bundle。浏览器下载 JS 后，在客户端执行代码，通过 API 获取数据，动态生成 DOM 并挂载到页面上。
- **优点**:
  - 交互体验好，页面切换流畅（无刷新）。
  - 服务器压力小，只负责提供静态资源和 API。
- **缺点**:
  - **SEO 差**: 爬虫（尤其是旧版爬虫）可能无法执行 JS，导致抓取不到内容。
  - **FCP (First Contentful Paint) 慢**: 用户需要等待 JS 下载并执行完毕才能看到内容（白屏时间长）。
- **适用场景**: 管理后台、仪表盘、不需要 SEO 的应用。

### 2.2 SSR (Server-Side Rendering) - 服务端渲染

- **原理**: 传统的 SSR（如 JSP/PHP）是指服务器生成完整的 HTML 返回给浏览器。在 React/Next.js 语境下，SSR 指的是在 Node.js 服务端运行 React 组件，生成 HTML 字符串返回给浏览器，浏览器显示内容后，再下载 JS 进行“注水” (Hydration)，使页面变得可交互。
- **优点**:
  - **SEO 友好**: 爬虫可以直接获取完整 HTML。
  - **FCP 快**: 用户能更快看到首屏内容。
- **缺点**:
  - 服务器压力大（每次请求都要计算）。
  - TTFB (Time To First Byte) 可能较慢（需等待数据获取和 HTML 生成）。
- **Next.js 实现**: `getServerSideProps` 或动态 Server Components。

### 2.3 SSG (Static Site Generation) - 静态站点生成

- **原理**: 在**构建时 (Build Time)** 预先渲染所有页面，生成静态 HTML 文件。用户访问时，Nginx/CDN 直接返回静态文件。
- **优点**:
  - **性能极致**: 静态文件由 CDN 分发，速度最快。
  - **SEO 极佳**。
  - **服务器成本低**。
- **缺点**:
  - 内容更新滞后（需重新构建）。
  - 不适合数据频繁变化的页面。
- **Next.js 实现**: `getStaticProps` (默认行为)。

### 2.4 ISR (Incremental Static Regeneration) - 增量静态再生

- **原理**: 结合了 SSG 和 SSR 的优点。页面在构建时生成静态文件，但可以在一定时间间隔后（如 60 秒），当有新请求时在后台重新生成该页面。
- **优点**: 保持了 SSG 的高性能，同时解决了内容更新滞后的问题。
- **Next.js 实现**: 在 `getStaticProps` 中返回 `revalidate` 字段。

## 3. HTTP 协议基础 (HTTP Basics)

### 3.1 请求与响应模型 (Request/Response Cycle)

Web 通信基于 HTTP 协议的请求-响应模型。

1.  **Client**: 发起 Request（包含 Method, URL, Headers, Body）。
2.  **Server**: 处理请求，返回 Response（包含 Status Code, Headers, Body）。

### 3.2 HTTP 方法 (Methods)

- **GET**: 获取资源（幂等，无副作用）。
- **POST**: 创建资源（非幂等）。
- **PUT**: 全量更新资源（幂等）。
- **PATCH**: 增量更新资源（非幂等）。
- **DELETE**: 删除资源（幂等）。

### 3.3 常见状态码 (Status Codes)

- **2xx (成功)**:
  - 200 OK: 请求成功。
  - 201 Created: 资源创建成功。
- **3xx (重定向)**:
  - 301 Moved Permanently: 永久重定向（SEO 权重转移）。
  - 302 Found: 临时重定向。
  - 304 Not Modified: 协商缓存命中，资源未修改。
- **4xx (客户端错误)**:
  - 400 Bad Request: 请求参数错误。
  - 401 Unauthorized: 未登录/无身份。
  - 403 Forbidden: 已登录但无权限。
  - 404 Not Found: 资源不存在。
- **5xx (服务端错误)**:
  - 500 Internal Server Error: 服务器内部错误。
  - 502 Bad Gateway: 网关错误（常见于 Nginx 无法连接 Node.js 进程）。

### 3.4 缓存机制 (Caching)

HTTP 缓存是性能优化的关键。

- **强缓存**: `Cache-Control: max-age=3600`。浏览器在 1 小时内直接从本地缓存读取，不请求服务器。
- **协商缓存**: `ETag` / `If-None-Match` 或 `Last-Modified` / `If-Modified-Since`。浏览器询问服务器资源是否变更，未变更则返回 304。

### 3.5 HTTP 协议演进 (HTTP Evolution)

Web 性能优化的很大一部分工作是在克服 HTTP/1.1 的局限性。

#### 3.5.1 HTTP/1.1 的局限

- **队头阻塞 (Head-of-Line Blocking)**: 浏览器限制同域名的并发连接数（通常 6 个）。如果一个请求阻塞，后续请求都会被拖延。
- **文本协议**: 冗长，解析效率低。
- **头部冗余**: 每次请求都携带大量重复的 Headers (User-Agent, Cookie 等)。

#### 3.5.2 HTTP/2 (2015)

HTTP/2 引入了二进制分帧层，极大地提升了传输效率。

- **多路复用 (Multiplexing)**: 允许在单一 TCP 连接上同时发送多个请求和响应。解决了浏览器并发连接数限制的问题。
- **头部压缩 (HPACK)**: 使用霍夫曼编码压缩 Headers，减少传输体积。
- **二进制协议**: 也就是二进制分帧，机器解析更高效，错误更少。
- **Server Push**: 服务端可以主动推送资源给客户端（注意：由于复杂性和缓存问题，该特性在 Chrome 中已被移除，不推荐过度依赖）。

#### 3.5.3 HTTP/3 (QUIC)

基于 UDP 协议构建的 QUIC 协议，进一步解决了 TCP 的固有缺陷。

- **解决 TCP 队头阻塞**: 在 HTTP/2 中，虽然 HTTP 层多路复用，但 TCP 层丢包仍会阻塞所有流。HTTP/3 基于 UDP，流之间相互独立，一个流丢包不影响其他流。
- **0-RTT 建连**: 首次连接 1-RTT，后续连接 0-RTT，大幅降低延迟。
- **连接迁移**: 切换网络（如 WiFi 切 4G）时，连接不会中断，因为 QUIC 使用 Connection ID 而非 IP:Port 标识连接。

## 4. RESTful API 与 GraphQL

### 4.1 RESTful API

一种基于资源的架构风格。

- **特点**: 使用 URL 定位资源，用 HTTP 方法描述操作。
- **示例**:
  - GET /users/1
  - POST /users
  - DELETE /users/1
- **问题**: 容易出现 Over-fetching（获取过多不必要字段）或 Under-fetching（需要多次请求才能拼凑完整数据）。

### 4.2 GraphQL

Facebook 开发的一种查询语言。

- **特点**: 客户端按需查询，只获取需要的字段。通常只有一个 Endpoint (`/graphql`)，使用 POST 请求。
- **优点**: 灵活、类型安全、减少网络请求次数。
- **缺点**: 缓存复杂、学习曲线陡峭。
- **Next.js**: 完美支持 GraphQL（通常配合 Apollo Client 或 Relay）。

## 5. 浏览器渲染原理 (Browser Rendering)

理解浏览器如何渲染页面，有助于优化 CSS 和 JavaScript 的性能。

### 5.1 关键渲染路径 (Critical Rendering Path)

1.  **解析 HTML**: 构建 DOM 树 (Document Object Model)。
2.  **解析 CSS**: 构建 CSSOM 树 (CSS Object Model)。
3.  **构建 Render Tree**: 结合 DOM 和 CSSOM，生成渲染树（只包含可见元素）。
4.  **Layout (回流/Reflow)**: 计算每个元素在屏幕上的确切位置和大小。
5.  **Paint (重绘/Repaint)**: 填充像素（颜色、背景、边框等）。
6.  **Composite (合成)**: 将不同图层合成到屏幕上。

### 5.2 优化建议

- **避免 Reflow**: 布局改变（如修改 width, height, margin）会触发回流，成本很高。尽量使用 transform 和 opacity，它们只会触发 Composite。
- **CSS 放在头部**: 避免 FOUC (Flash of Unstyled Content)。
- **JS 放在底部或使用 defer/async**: 避免阻塞 HTML 解析。

## 6. 状态管理 (State Management)

在复杂的 Web 应用中，如何管理数据（状态）至关重要。

### 6.1 状态分类

- **Local State (局部状态)**: 组件内部状态（如 input 的 value，弹窗的开关）。使用 `useState`。
- **Global State (全局状态)**: 跨组件共享的状态（如主题、用户信息）。使用 Context API, Redux, Zustand。
- **Server State (服务端状态)**: 来自 API 的数据，具有缓存、过期、重新验证等特性。使用 React Query (TanStack Query) or SWR。
- **URL State**: 将状态保存在 URL 参数中（如搜索关键词、分页页码）。这是 Web 开发的最佳实践，利于分享和书签。

### 6.2 不可变性 (Immutability)

React 依赖数据的不可变性来判断是否需要重新渲染。修改状态时，不应直接修改原对象，而应创建新对象。

```javascript
// 错误
state.count = 1;

// 正确
setState({ ...state, count: 1 });
```

## 7. 认证与授权 (Authentication & Authorization)

### 7.1 认证 (Authentication) - "你是谁？"

- **Session/Cookie**: 传统方式。服务端存储 Session，客户端 Cookie 携带 Session ID。缺点是难以扩展（服务端有状态）。
- **JWT (JSON Web Token)**: 现代方式。服务端签发 Token，客户端存储（localStorage 或 Cookie），请求时携带。服务端无状态，易扩展。
- **NextAuth.js (Auth.js)**: Next.js 生态中最流行的认证库，支持 OAuth (GitHub, Google Login) 和 Email 登录。

### 7.2 授权 (Authorization) - "你能做什么？"

- **RBAC (Role-Based Access Control)**: 基于角色的访问控制（如 Admin, Editor, User）。
- **ABAC (Attribute-Based Access Control)**: 基于属性的访问控制（更细粒度）。

## 8. 跨域问题 (CORS)

### 8.1 什么是跨域

浏览器的同源策略 (Same-Origin Policy) 限制了从一个源加载的文档或脚本如何与来自另一个源的资源进行交互。
**同源**: 协议、域名、端口完全相同。

### 8.2 解决方案

- **CORS (Cross-Origin Resource Sharing)**: 服务端设置响应头 `Access-Control-Allow-Origin` 允许特定源访问。
- **Proxy (代理)**: 在开发环境（如 Next.js 的 `next.config.js` rewrites）或生产环境（Nginx）设置代理，将跨域请求转发为同域请求。因为服务器与服务器之间没有同源策略限制。

## 9. 核心 Web 指标 (Core Web Vitals) - 2024 版

Core Web Vitals 是 Google 提出的一组用于衡量网页用户体验的关键指标。Next.js 内置了 `useReportWebVitals` hook 来帮助开发者监控这些指标。

### 9.1 LCP (Largest Contentful Paint)

衡量**加载性能**。LCP 报告视口中最大的可见内容元素（通常是图片或大块文本）渲染完成的时间。

- **目标**: 2.5 秒以内。
- **优化**:
  - 优化服务器响应时间 (TTFB)。
  - 使用 CDN。
  - 优化图片大小（WebP/AVIF）。
  - 预加载关键资源 (`<link rel="preload">`)。
  - Next.js 的 `<Image>` 组件自动处理了许多 LCP 优化。

### 9.2 INP (Interaction to Next Paint)

衡量**交互性**。INP 观察用户与页面进行的所有交互（点击、触摸、键盘输入）的延迟，并报告最差的一次（或接近最差的一次）。**注意：INP 已于 2024 年 3 月正式取代 FID (First Input Delay)。**

- **目标**: 200 毫秒以内。
- **优化**:
  - 减少主线程阻塞任务（Long Tasks）。
  - 使用 `requestIdleCallback` 或 `setTimeout` 分解长任务。
  - 减少 JavaScript 执行时间。
  - 避免复杂的 CSS 选择器。
  - React 18+ 的 Concurrent Features (并发特性) 有助于改善 INP。

### 9.3 CLS (Cumulative Layout Shift)

衡量**视觉稳定性**。CLS 测量整个页面生命周期内发生的所有意外布局偏移的得分总和。

- **目标**: 0.1 以下。
- **优化**:
  - 为图片和视频设置明确的 `width` 和 `height` 属性。
  - 为广告、嵌入内容预留空间。
  - 避免在已有内容上方动态插入内容。
  - 使用 `font-display: optional` 或 `swap` 优化字体加载。

### 9.4 测量工具

- **Lighthouse**: 实验室环境评分。
- **Chrome DevTools**: Performance 面板。
- **PageSpeed Insights**: 结合实验室和真实用户数据 (CrUX)。
- **Web Vitals Extension**: Chrome 插件。

## 10. Web 安全深度解析 (Web Security Deep Dive)

安全性是 Web 开发中不可忽视的一环。

### 10.1 XSS (跨站脚本攻击)

攻击者在网页中注入恶意脚本。

- **类型**:
  - **存储型 (Stored)**: 恶意脚本存储在数据库中（如评论区）。
  - **反射型 (Reflected)**: 恶意脚本包含在 URL 请求中。
  - **DOM 型**: 恶意脚本修改页面 DOM 结构。
- **防范**:
  - **转义输出**: React 默认会转义所有插入 DOM 的字符串，有效防止 XSS。
  - **CSP (Content Security Policy)**: 限制资源加载源。
  - **HttpOnly Cookie**: 防止 JS 读取 Cookie。
  - **避免 `dangerouslySetInnerHTML`**: 除非绝对必要且经过清洗 (Sanitize)。

### 10.2 CSRF (跨站请求伪造)

攻击者诱导已登录用户访问恶意网站，利用用户的 Cookie 向目标网站发送请求。

- **防范**:
  - **CSRF Token**: 每次请求携带随机 Token，服务端验证。
  - **SameSite Cookie**: 设置 `SameSite=Strict` 或 `Lax`，限制第三方 Cookie 发送。
  - **验证 Referer/Origin**: 检查请求来源。

### 10.3 CSP (内容安全策略)

通过 HTTP 响应头 `Content-Security-Policy` 告诉浏览器允许加载哪些资源。

```http
Content-Security-Policy: default-src 'self'; img-src *; media-src media1.com media2.com; script-src userscripts.example.com
```

- **作用**: 极大降低 XSS 风险。

### 10.4 HSTS (HTTP Strict Transport Security)

强制浏览器只通过 HTTPS 与服务器连接。

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

- **作用**: 防止中间人攻击和 SSL 剥离。

## 11. 性能优化策略 (Performance Optimization Strategies)

除了 Core Web Vitals，还有通用的性能优化手段。

### 11.1 资源加载优化 (Resource Hints)

告诉浏览器提前处理某些资源。

- **preload**: 高优先级加载当前页面的关键资源。
  ```html
  <link rel="preload" href="critical.js" as="script" />
  ```
- **prefetch**: 低优先级加载后续页面可能用到的资源。
  ```html
  <link rel="prefetch" href="next-page.js" />
  ```
- **preconnect**: 提前建立 TCP 连接和 TLS 握手。
  ```html
  <link rel="preconnect" href="https://api.example.com" />
  ```

### 11.2 代码分割与 Tree Shaking

- **Code Splitting**: 将代码拆分成小块，按需加载。Next.js 默认对页面进行拆分。
- **Tree Shaking**: 移除未引用的代码 (Dead Code)。依赖于 ES Modules 的静态分析。

### 11.3 缓存策略

- **HTTP 缓存**:
  - `Cache-Control: max-age=3600`: 强缓存 1 小时。
  - `ETag / If-None-Match`: 协商缓存，文件没变返回 304。
- **Service Worker**: 拦截网络请求，实现离线缓存和更复杂的缓存策略（如 Stale-While-Revalidate）。

## 13. 网络基础 (Network Basics)

### 13.1 DNS 解析过程

1.  浏览器缓存检查。
2.  操作系统缓存 (hosts 文件) 检查。
3.  请求本地域名服务器 (LDNS)。
4.  根域名服务器 -> 顶级域名服务器 -> 权威域名服务器。

### 13.2 TCP 三次握手

建立可靠连接的过程。

1.  **SYN**: 客户端发送连接请求。
2.  **SYN + ACK**: 服务端确认并发送连接请求。
3.  **ACK**: 客户端确认。连接建立。

### 13.3 TLS/SSL 握手

在 TCP 之上建立加密通道。

- 验证服务器证书。
- 协商加密算法。
- 生成会话密钥 (Session Key)。
- HTTP/2 和 HTTP/3 对握手进行了优化 (TLS 1.3 支持 0-RTT)。

## 14. Web 无障碍 (Accessibility / a11y)

让残障人士也能使用 Web 应用。Next.js 内置了 eslint-plugin-jsx-a11y。

### 14.1 语义化 HTML

使用正确的标签做正确的事。

- 按钮用 `<button>` 而不是 `<div onClick="...">`。
- 链接用 `<a>`。
- 文章结构用 `<header>`, `<main>`, `<footer>`, `<nav>`, `<article>`。

### 14.2 ARIA (Accessible Rich Internet Applications)

当 HTML 语义不足以描述交互时，使用 ARIA 属性。

- `role="dialog"`: 模态框。
- `aria-label="Close"`: 为只有图标的按钮提供文本描述。
- `aria-expanded="true"`: 描述展开/折叠状态。

### 14.3 键盘导航

确保所有交互元素都能通过 Tab 键访问，并且有清晰的 Focus 样式 (`outline`)。

## 15. 浏览器多进程架构 (Browser Multi-process Architecture)

现代浏览器（如 Chrome）采用多进程架构，以提高稳定性、安全性和性能。

### 15.1 核心进程

- **Browser Process (主进程)**: 负责浏览器界面显示（地址栏、书签、前进后退）、用户交互、子进程管理。
- **Renderer Process (渲染进程)**: 核心进程。负责将 HTML/CSS/JS 转换为用户可见的网页。默认情况下，每个 Tab 一个渲染进程（Process-per-site-instance）。
  - **GUI 渲染线程**: 解析 HTML/CSS，构建 DOM/Render 树，布局和绘制。与 JS 引擎线程互斥。
  - **JS 引擎线程**: 解析和执行 JavaScript (V8 引擎)。
  - **事件触发线程**: 管理事件循环。
  - **定时器触发线程**: `setTimeout` / `setInterval`。
  - **异步 HTTP 请求线程**: 处理 AJAX 请求。
- **GPU Process**: 负责 GPU 渲染（3D CSS, WebGL）。
- **Network Process**: 负责网络资源加载。
- **Plugin Process**: 负责插件运行（如 Flash，已淘汰）。

### 15.2 为什么采用多进程？

- **稳定性**: 一个 Tab 崩溃不会导致整个浏览器崩溃。
- **安全性**: 多进程沙箱机制 (Sandbox)，隔离不同站点，防止恶意代码访问系统。
- **性能**: 充分利用多核 CPU。

## 16. JavaScript 事件循环与异步机制 (Event Loop & Async)

JavaScript 是单线程的，但通过事件循环机制实现了异步操作。

### 16.1 执行栈与任务队列

- **Call Stack (调用栈)**: 同步代码执行的地方。
- **Task Queue (任务队列)**: 异步回调等待执行的地方。

### 16.2 宏任务 (Macro Task) 与 微任务 (Micro Task)

- **Macro Task**: `setTimeout`, `setInterval`, `setImmediate` (Node), I/O, UI Rendering.
- **Micro Task**: `Promise.then`, `process.nextTick` (Node), `MutationObserver`.

### 16.3 事件循环流程

1.  执行 Call Stack 中的同步代码。
2.  Call Stack 清空后，检查 Micro Task Queue。
3.  **依次执行所有** Micro Tasks，直到队列清空（包括在执行过程中新加入的 Micro Task）。
4.  尝试进行 UI 渲染 (Render)。
5.  从 Macro Task Queue 中取出一个任务执行。
6.  重复上述步骤。

### 16.4 Node.js vs 浏览器 Event Loop 的区别

虽然都基于 V8，但 Node.js 的 Event Loop 由 `libuv` 库实现，与浏览器有所不同。

- **Node.js**: 有多个阶段 (Timers, Pending Callbacks, Idle/Prepare, Poll, Check, Close Callbacks)。
- **微任务执行时机**: 在 Node.js 11 之前，微任务在每个阶段之间执行；Node.js 11+ 改为与浏览器一致，即每个宏任务执行完后立即执行微任务队列。

## 17. 进阶：HTTP 缓存策略深度解析 (Deep Dive into HTTP Caching)

HTTP 缓存是提升 Web 性能最高效的手段之一。合理的缓存策略可以显著减少服务器负载和用户等待时间。

### 17.1 Cache-Control 指令详解

`Cache-Control` 是 HTTP/1.1 中最通用的缓存控制头部。

- **可缓存性**:

  - `public`: 响应可以被任何对象（包括发送请求的客户端、代理服务器、CDN）缓存。
  - `private`: 响应只能被单个用户（客户端）缓存，代理服务器不可缓存（常用于个人信息）。
  - `no-cache`: 强制要求缓存把请求提交给原始服务器进行验证（协商缓存）。即便缓存中有副本，也要问一下服务器“我这个过期没？”。
  - `no-store`: 禁止任何缓存，每次都从服务器获取完整数据（用于敏感数据，如银行账户信息）。

- **到期时间**:

  - `max-age=<seconds>`: 资源在多少秒内是新鲜的（相对于请求时间）。
  - `s-maxage=<seconds>`: 覆盖 `max-age`，但仅对共享缓存（如 CDN）有效。
  - `max-stale=<seconds>`: 客户端愿意接收一个已经过期的资源，只要过期的秒数不超过指定值。

- **重新验证**:
  - `must-revalidate`: 一旦资源过期（超过 `max-age`），在成功向原始服务器验证之前，缓存不能使用该资源。
  - `proxy-revalidate`: 与 `must-revalidate` 类似，但仅适用于共享缓存。

### 17.2 缓存策略最佳实践

#### 1. 静态资源 (Hash 文件名)

对于带 Hash 的静态资源（如 `main.a3b4c5.js`），内容变了文件名就会变。
**策略**: 强缓存，且时间尽可能长。

```http
Cache-Control: public, max-age=31536000, immutable
```

_注_: `immutable` 表示资源永远不会改变，浏览器甚至不需要发送协商缓存请求。

#### 2. HTML 文件 (入口文件)

对于 `index.html`，文件名通常不变，但引用的 JS/CSS Hash 会变。
**策略**: 协商缓存。

```http
Cache-Control: no-cache
```

或者设置一个很短的过期时间：

```http
Cache-Control: max-age=0, must-revalidate
```

#### 3. API 响应

对于 API 数据，取决于数据的实时性要求。

- **实时性高**: `no-store`
- **一般内容**: `max-age=60` (1 分钟缓存)

## 18. 实战：Next.js 中的混合渲染模式 (Hybrid Rendering in Next.js)

Next.js 允许在同一个应用中混合使用不同的渲染模式。下面我们通过代码示例来对比它们的实现。

### 18.1 Server-Side Rendering (SSR)

在 Next.js 12/Pages Router 中使用 `getServerSideProps`。
在 Next.js 13+/App Router 中，默认就是 Server Components，相当于 SSR。

**App Router (Next.js 16)**:

```typescript
// app/blog/[id]/page.tsx
// 这是一个 Server Component
import { notFound } from "next/navigation";

interface Post {
  id: string;
  title: string;
  content: string;
}

async function getPost(id: string): Promise<Post> {
  const res = await fetch(`https://api.example.com/posts/${id}`, {
    cache: "no-store", // 禁用缓存，确保每次请求都由服务器重新获取 (SSR)
  });
  if (!res.ok) return undefined;
  return res.json();
}

export default async function Page({ params }: { params: { id: string } }) {
  const post = await getPost(params.id);

  if (!post) {
    notFound();
  }

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </article>
  );
}
```

### 18.2 Static Site Generation (SSG)

在 App Router 中，默认的 `fetch` 行为就是缓存的，即 SSG。

```typescript
// app/blog/static/[id]/page.tsx
export async function generateStaticParams() {
  const posts = await fetch("https://api.example.com/posts").then((res) =>
    res.json()
  );
  return posts.map((post) => ({
    id: post.id.toString(),
  }));
}

async function getPost(id: string) {
  const res = await fetch(`https://api.example.com/posts/${id}`, {
    cache: "force-cache", // 默认值，构建时缓存 (SSG)
  });
  return res.json();
}

export default async function Page({ params }) {
  const post = await getPost(params.id);
  // ...渲染逻辑
}
```

### 18.3 Incremental Static Regeneration (ISR)

结合了 SSG 的速度和 SSR 的动态性。

```typescript
// app/blog/isr/[id]/page.tsx
// ... generateStaticParams 同上

async function getPost(id: string) {
  const res = await fetch(`https://api.example.com/posts/${id}`, {
    next: { revalidate: 60 }, // 每 60 秒重新验证一次 (ISR)
  });
  return res.json();
}

export default async function Page({ params }) {
  // ...
}
```

### 18.4 Client-Side Rendering (CSR)

在 Server Component 中引入 Client Component。

```typescript
// app/dashboard/page.tsx
"use client"; // 标记为客户端组件

import { useState, useEffect } from "react";

export default function Dashboard() {
  const [data, setData] = useState(null);
  const [isLoading, setLoading] = useState(true);

  useEffect(() => {
    fetch("/api/profile")
      .then((res) => res.json())
      .then((data) => {
        setData(data);
        setLoading(false);
      });
  }, []);

  if (isLoading) return <p>Loading...</p>;
  if (!data) return <p>No profile data</p>;

  return (
    <div>
      <h1>Dashboard</h1>
      <p>Name: {data.name}</p>
    </div>
  );
}
```

_优化建议_: 推荐使用 `SWR` 或 `React Query` 来替代 `useEffect` 进行数据请求，它们提供了缓存、自动重试、聚焦刷新等高级功能。

```typescript
"use client";
import useSWR from "swr";

const fetcher = (...args) => fetch(...args).then((res) => res.json());

export default function Dashboard() {
  const { data, error, isLoading } = useSWR("/api/profile", fetcher);

  if (error) return <div>failed to load</div>;
  if (isLoading) return <div>loading...</div>;
  return <div>Hello {data.name}!</div>;
}
```

## 17. PWA (Progressive Web App) 深度

PWA 利用现代 Web API 提供类似原生 App 的体验。

### 17.1 核心技术

- **Manifest (`manifest.json`)**: 定义应用名称、图标、启动方式、主题色。允许“安装”到主屏幕。
- **Service Worker**: 运行在后台的独立线程，充当网络代理。
  - **拦截请求**: `fetch` 事件。
  - **离线缓存**: Cache Storage API。
  - **后台同步**: Background Sync API。
  - **推送通知**: Push API。

### 17.2 缓存策略 (Caching Strategies)

- **Cache First (缓存优先)**: 适用于静态资源 (JS, CSS, Images)。
  ```javascript
  // Service Worker fetch handler
  e.respondWith(
    caches.match(e.request).then((response) => response || fetch(e.request))
  );
  ```
- **Network First (网络优先)**: 适用于经常变化的内容（如 API 数据）。
  ```javascript
  e.respondWith(fetch(e.request).catch(() => caches.match(e.request)));
  ```
- **Stale-While-Revalidate**: 立即返回缓存，同时发起网络请求更新缓存。最常用的策略。

## 18. Web Components 与 Shadow DOM

Web Components 是一套原生标准，用于创建可复用的自定义元素。

### 18.1 三大技术

1.  **Custom Elements**: 定义新的 HTML 标签。

    class MyElement extends HTMLElement {

### 18.2 与 React 的关系

- React 组件是库层面的封装，Web Components 是浏览器层面的标准。
- 可以在 React 中使用 Web Components。
- Web Components 适合构建跨框架的 UI 组件库。

## 20. 实时通信技术 (Real-time Communication)

在现代 Web 应用中，实时性需求越来越高（如聊天、股票行情、协同编辑）。

### 20.1 短轮询 (Short Polling) 与 长轮询 (Long Polling)

- **Short Polling**: 客户端每隔几秒发送一次请求。
  - _缺点_: 浪费带宽，延迟高。
- **Long Polling**: 客户端发送请求，服务端保持连接直到有数据更新才返回。
  - _缺点_: 占用服务端连接资源。

### 20.2 WebSocket

全双工通信协议，建立在 TCP 之上。

- **握手**: HTTP Upgrade 请求。
- **特点**: 双向通信，低延迟，头部开销小。
- **适用**: 聊天室、多人游戏。
  ```javascript
  const socket = new WebSocket("ws://example.com/socket");
  socket.onopen = () => socket.send("Hello");
  socket.onmessage = (event) => console.log(event.data);
  ```

### 20.3 Server-Sent Events (SSE)

服务端向客户端单向推送数据。基于 HTTP 协议。

- **特点**: 单向（Server -> Client），断线自动重连，简单的文本协议。
- **适用**: 股票行情、新闻推送、日志监控。
  ```javascript
  const evtSource = new EventSource("/api/stream");
  evtSource.onmessage = (event) => console.log(event.data);
  ```

### 20.4 WebRTC (Web Real-Time Communication)

浏览器之间的点对点 (P2P) 通信，无需经过服务器中转（信令服务器除外）。

- **核心**:
  - **GetUserMedia**: 获取音视频流。
  - **RTCPeerConnection**: 建立 P2P 连接。
  - **RTCDataChannel**: 传输任意数据。
- **适用**: 视频会议、文件传输。

## 21. Serverless 与边缘计算 (Serverless & Edge)

### 21.1 Serverless (无服务器架构)

开发者无需管理服务器，只需编写业务逻辑代码 (FaaS - Function as a Service)。

- **优势**: 按需付费、自动扩缩容、无需运维。
- **劣势**: 冷启动 (Cold Start) 延迟。
- **Next.js API Routes**: 部署到 Vercel 时自动转换为 Serverless Functions。

### 21.2 Edge Computing (边缘计算)

将计算能力下沉到离用户最近的 CDN 节点。

- **优势**: 极低延迟（无网络回源），减少主服务器压力。
- **Next.js Middleware**: 运行在 Edge Runtime (基于 V8 Isolate，非 Node.js)。
  - _限制_: 只能使用标准 Web API，不支持 Node.js 模块（如 `fs`, `path`）。
  - _场景_: 身份验证、A/B 测试、重定向、地理位置封禁。

## 23. GraphQL 深度解析

一种用于 API 的查询语言，也是一个满足你数据查询的运行时。

### 23.1 核心概念

- **Schema (SDL)**: 定义数据结构。
  ```graphql
  type User {
    id: ID!
    name: String!
    posts: [Post]
  }
  ```
- **Query**: 获取数据（类似于 GET）。
  ```graphql
  query {
    user(id: "1") {
      name
      posts {
        title
      }
    }
  }
  ```
- **Mutation**: 修改数据（类似于 POST/PUT/DELETE）。
- **Resolver**: 也就是具体的函数实现，负责去数据库取数据。

### 23.2 优缺点对比

| 特性         | REST                                | GraphQL                             |
| :----------- | :---------------------------------- | :---------------------------------- |
| **数据获取** | 可能 Over-fetching / Under-fetching | 精确获取所需数据                    |
| **请求次数** | 可能需要多次请求                    | 通常只需一次请求                    |
| **类型系统** | 弱（依赖文档）                      | 强类型 (Schema)                     |
| **缓存**     | 利用 HTTP 缓存很简单                | 较复杂（通常依赖客户端库如 Apollo） |
| **复杂度**   | 低                                  | 高                                  |

### 23.3 N+1 问题与 DataLoader

GraphQL 容易产生 N+1 查询问题（查询列表后，对每一项再查详情）。

- **DataLoader**: 批处理工具。将多个小请求合并为一个批量请求，并进行缓存。

## 24. Web 性能监控与调优实战 (Web Performance Monitoring & Tuning)

在理解了核心 Web 指标后，我们需要掌握如何在实际项目中监控和优化这些指标。

### 24.1 Performance API

浏览器提供了强大的 `Performance API` 来获取高精度的性能数据。

- **Navigation Timing**: 获取页面加载各个阶段的耗时（DNS、TCP、TTFB、DOM Ready 等）。
- **Resource Timing**: 获取单个静态资源（JS, CSS, Image）的加载耗时。
- **User Timing**: 开发者自定义的关键点打点 (`performance.mark`, `performance.measure`)。

```javascript
// 计算 TTFB (Time To First Byte)
const navEntry = performance.getEntriesByType("navigation")[0];
console.log(`TTFB: ${navEntry.responseStart - navEntry.requestStart}ms`);

// 自定义打点
performance.mark("my-feature-start");
// ... 执行耗时操作 ...
performance.mark("my-feature-end");
performance.measure("my-feature", "my-feature-start", "my-feature-end");

const measure = performance.getEntriesByName("my-feature")[0];
console.log(`My Feature took: ${measure.duration}ms`);
```

### 24.2 使用 `web-vitals` 库

Google 提供的 `web-vitals` 库是获取 LCP, FID, CLS 等指标的标准方式。

```javascript
import { onCLS, onFID, onLCP } from "web-vitals";

function sendToAnalytics(metric) {
  const body = JSON.stringify(metric);
  // 使用 navigator.sendBeacon 保证在页面关闭时也能发送数据
  (navigator.sendBeacon && navigator.sendBeacon("/analytics", body)) ||
    fetch("/analytics", { body, method: "POST", keepalive: true });
}

onCLS(sendToAnalytics);
onFID(sendToAnalytics);
onLCP(sendToAnalytics);
```

### 24.3 Chrome DevTools 性能分析

Chrome 开发者工具的 **Performance** 面板是调优的核武器。

1. **录制 (Record)**: 点击录制按钮，执行用户操作，停止录制。
2. **火焰图 (Flame Chart)**: 分析主线程的调用栈。横轴是时间，纵轴是调用栈深度。
3. **寻找长任务 (Long Tasks)**: 标红的任务通常是导致掉帧或输入延迟的原因。
4. **Bottom-Up**: 查看哪个函数耗时最长，针对性优化。

## 25. 浏览器渲染原理深度解析 (Browser Rendering Deep Dive)

深入理解浏览器如何将 HTML/CSS/JS 转换为像素，是性能优化的基石。

### 25.1 关键渲染路径 (Critical Rendering Path)

1. **构建 DOM 树**: 解析 HTML，构建 DOM (Document Object Model)。
2. **构建 CSSOM 树**: 解析 CSS，构建 CSSOM (CSS Object Model)。
3. **构建渲染树 (Render Tree)**: 结合 DOM 和 CSSOM，生成只包含可见节点的渲染树（`display: none` 的节点不在其中）。
4. **布局 (Layout/Reflow)**: 计算每个节点在屏幕上的确切位置和大小。
5. **绘制 (Paint)**: 填充像素（颜色、背景、边框、文本）。
6. **合成 (Composite)**: 将不同的层 (Layers) 合成到一起，由 GPU 绘制到屏幕。

### 25.2 重排 (Reflow) 与重绘 (Repaint)

- **重排 (Reflow)**: 当元素的几何属性（宽、高、位置）发生变化时，浏览器需要重新计算布局。成本极高。
  - _触发条件_: 改变窗口大小、改变字体、添加/删除 DOM 节点、读取 `offsetWidth/offsetHeight` 等布局属性。
- **重绘 (Repaint)**: 当元素的外观（颜色、背景）发生变化，但几何属性不变时。成本较低。
  - _触发条件_: `color`, `background-color`, `visibility`。

**优化策略**:

- 避免频繁读取布局属性（会导致强制同步布局）。
- 使用 `transform` 和 `opacity` 做动画，因为它们只会触发合成 (Composite)，由 GPU 处理，不触发重排和重绘。

### 25.3 浏览器层与 GPU 加速

浏览器会将某些元素提升到独立的合成层 (Compositing Layer)。

- **触发条件**: `will-change`, `video`, `iframe`, `transform: translateZ(0)`。
- **优势**: 层的变换 (Transform) 和透明度 (Opacity) 变化由 GPU 处理，极其流畅。
- **风险**: 过多的层会消耗大量内存（显存），导致崩溃或变慢。

## 26. 现代前端架构模式 (Modern Frontend Architecture Patterns)

随着应用复杂度的增加，前端架构也在不断演进。

### 26.1 BFF (Backend for Frontend)

在微服务架构下，前端直接调用多个微服务会导致网络瀑布流和数据处理复杂。

- **定义**: 专门为前端服务的后端层（通常用 Node.js 编写）。
- **作用**:
  - **聚合**: 将多个微服务的响应合并为一个。
  - **裁剪**: 去除前端不需要的字段，减少载荷。
  - **格式化**: 将数据转换为前端友好的格式。
- **Next.js 中的应用**: Next.js 的 API Routes 或 Server Components 天然充当了 BFF 的角色。

### 26.2 岛屿架构 (Islands Architecture)

- **背景**: SSR 虽然利于 SEO，但 Hydration（注水）过程成本高昂，导致 TTI (Time to Interactive) 变慢。
- **理念**: 将页面视为静态的海洋，其中漂浮着交互式的“岛屿”。
- **实现**: 只有岛屿部分的组件会被 Hydrate，其余部分保持静态 HTML。
- **代表框架**: Astro, Fresh。

### 26.3 React Server Components (RSC)

Next.js 13+ 全面采用的架构，进一步融合了服务端和客户端。

- **零 Bundle**: 服务端组件的代码不会打包到客户端 Bundle 中。
- **直接后端访问**: 服务端组件可以直接访问数据库或文件系统。
- **流式传输**: 组件渲染成特殊的序列化格式流式传输到客户端，React 在客户端解析并更新 DOM。

## 27. 前端工程化进阶 (Frontend Engineering Advanced)

前端工程化是现代 Web 开发的基石，旨在提升开发效率、代码质量和交付速度。

### 27.1 代码规范与风格统一

在团队协作中，统一的代码风格至关重要。

- **ESLint**: 静态代码分析工具，用于发现潜在的 bug 和不符合规范的代码（如未使用的变量、Hooks 依赖缺失）。
- **Prettier**: 代码格式化工具，强制统一代码风格（如缩进、引号、分号）。
- **配合使用**: 通常关闭 ESLint 的格式化规则，将格式化工作完全交给 Prettier，避免冲突。

### 27.2 Git Hooks 与提交规范

利用 Git Hooks 在代码提交前进行自动检查，防止低质量代码入库。

- **Husky**: 轻松配置 Git Hooks 的工具。
- **Lint-staged**: 只对暂存区（Staged）的文件运行 Lint 和 Test，提高检查速度。
- **Commitlint**: 检查 Commit Message 是否符合规范（如 Conventional Commits: `feat: add new feature`）。

```json
// package.json 示例
{
  "lint-staged": {
    "*.{js,ts,tsx}": ["eslint --fix", "prettier --write"]
  }
}
```

### 27.3 CI/CD (持续集成/持续部署)

自动化流程是现代开发的标配。

- **CI (Continuous Integration)**: 提交代码后自动运行测试、构建，确保新代码没有破坏现有功能。
  - _工具_: GitHub Actions, GitLab CI, Jenkins。
- **CD (Continuous Deployment)**: 通过 CI 检查后，自动部署到生产环境或预发布环境。
  - _平台_: Vercel (Next.js 首选), Netlify, AWS Amplify。

### 27.4 Docker 与容器化

虽然 Next.js 可以直接部署在 Node.js 环境中，但容器化能提供一致的运行环境。

```dockerfile
# Dockerfile 示例
FROM node:18-alpine AS base

# 安装依赖
FROM base AS deps
WORKDIR /app
COPY package.json yarn.lock ./
RUN yarn install --frozen-lockfile

# 构建应用
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN yarn build

# 生产环境运行
FROM base AS runner
WORKDIR /app
ENV NODE_ENV production
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json

EXPOSE 3000
CMD ["yarn", "start"]
```

## 28. 常见问题 (FAQ)

### 28.1 为什么我的页面刷新后 404？

**问题描述**: 在 SPA 应用部署到静态服务器（如 Nginx, S3）后，访问首页正常，但进入深层路径刷新页面报 404。
**原因**: SPA 是单页应用，路由由前端 JS 控制。服务器上并没有对应的 `/users/123.html` 文件。
**解决方案**: 配置 Nginx 将所有 404 请求重定向到 `index.html` (Fallback)，让前端路由接管。Next.js 的 SSR/SSG 模式通常不存在此问题，因为它是多页面的或服务端处理了路由。

### 28.2 为什么修改了 CSS 没生效？

**原因**: 浏览器强缓存了旧的 CSS 文件。
**解决方案**:

- 开发环境：禁用缓存 (DevTools -> Network -> Disable cache)。
- 生产环境：文件名带 Hash（如 `style.a1b2c3.css`），内容变了 Hash 也会变，强制浏览器下载新文件。Next.js 会自动处理这个问题。

### 28.3 为什么 API 请求跨域？

**原因**: 前端地址与后端 API 地址不同源。
**解决方案**: 后端开启 CORS 或前端配置 Proxy。

## 29. 适用场景 (Applicable Scenarios)

### 29.1 SPA (单页应用)

- **适用**: 类似 Trello, Gmail 等强交互、应用型极强的产品。
- **不适用**: 对 SEO 要求极高的内容型网站（新闻、博客），除非使用 SSR。

### 29.2 SSR (服务端渲染)

- **适用**: 电商商品详情页、新闻门户、博客文章。需要 SEO 且数据实时更新。
- **不适用**: 纯内部管理系统（增加了服务器成本，收益不高）。

### 29.3 SSG (静态站点生成)

- **适用**: 营销落地页、文档站、个人博客。内容更新频率低。

## 30. 注意事项 (Precautions)

### 30.1 开发原则

- **不过早优化**: 首先确保代码可读和功能正确，遇到性能瓶颈再使用 DevTools 分析优化。
- **拥抱新标准**: 优先使用标准的 Web API (如 `IntersectionObserver`, `fetch`)，而不是过时的库。

### 30.2 兼容性测试

- **跨浏览器**: 虽然 Chrome 占有率高，但不要忽视 Safari (iOS) 和 Firefox。使用 BrowserStack 或真机测试。
- **移动优先**: 现代 Web 开发应优先考虑移动端体验 (Responsive Design)。

### 30.3 依赖管理

- **定期更新**: 保持 npm 包的更新，修复已知的安全漏洞 (`npm audit`)。
- **审慎引入**: 引入一个库前，检查其体积 (BundlePhobia) 和维护状态。

## 31. 总结 (Summary)

现代 Web 开发是一个庞大的体系，涵盖了网络协议、渲染原理、工程化实践等多个方面。

- **渲染模式**: 理解 CSR, SSR, SSG, ISR 的权衡是使用 Next.js 的前提。
- **HTTP**: 掌握状态码、缓存策略对于调试和优化至关重要。
- **浏览器原理**: 了解渲染路径有助于写出高性能的 CSS 和 JS。
- **状态管理**: 合理区分 Server State 和 Client State 是现代 React 开发的关键。

Next.js 作为一个全栈框架，将这些现代 Web 开发的最佳实践（如混合渲染、自动代码分割、图片优化）封装在内，让开发者能更专注于业务逻辑。但了解这些底层概念，能让你在遇到问题时知其然，更知其所以然。
