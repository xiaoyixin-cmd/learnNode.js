# 00-04 Node.js 环境基础

## 1. 概述 (Overview)

### 1.1 什么是 Node.js

Node.js 是一个基于 Chrome V8 JavaScript 引擎的 JavaScript 运行时环境。它允许开发者使用 JavaScript 编写服务器端代码，打破了 JavaScript 仅能运行在浏览器的限制。Node.js 采用事件驱动、非阻塞 I/O 模型，使其轻量且高效，特别适合数据密集型的实时应用。

在 Next.js 16 的开发中，Node.js 扮演着至关重要的角色。Next.js 的服务端渲染 (SSR)、静态站点生成 (SSG)、API Routes 以及最新的 Server Components 都依赖于 Node.js 运行时（或兼容的 Edge Runtime）。理解 Node.js 的基础概念，是掌握 Next.js 全栈开发能力的基石。

### 1.2 Node.js 的核心特性

1.  **单线程 (Single Threaded)**: Node.js 应用程序主要运行在单个线程上，但这并不意味着它不能处理并发。相反，它通过事件循环机制处理并发操作。
2.  **非阻塞 I/O (Non-blocking I/O)**: 在 Node.js 中，I/O 操作（如读取文件、网络请求）通常是异步的。当发起一个 I/O 请求时，程序不会等待其完成，而是继续执行后续代码，待 I/O 完成后通过回调函数或 Promise 通知主线程。
3.  **事件驱动 (Event Driven)**: Node.js 广泛使用事件机制。许多内置模块（如 `http`, `net`）都继承自 `EventEmitter` 类。
4.  **跨平台 (Cross-platform)**: 基于 Libuv 库，Node.js 抹平了不同操作系统（Windows, Linux, macOS）之间的系统调用差异。

### 1.3 为什么 Next.js 开发者需要深入学习 Node.js

虽然 Next.js 抽象了许多底层细节，但在以下场景中，你必须直接与 Node.js API 交互：

- **文件系统操作**: 在 `getStaticProps` 或 API Routes 中读取本地数据文件（Markdown, JSON）。
- **路径处理**: 处理不同操作系统间的文件路径差异。
- **环境变量**: 管理 `process.env` 以区分开发、测试和生产环境配置。
- **HTTP 处理**: 在中间件 (Middleware) 或自定义 Server 中处理请求头、Cookies 和流。
- **Server Components**: 理解服务端组件的运行环境限制（如无法访问 `window` 对象）。

## 2. 核心概念与架构 (Core Concepts & Architecture)

### 2.1 运行时架构

Node.js 的架构主要由以下几部分组成：

- **V8 Engine**: Google 开发的高性能 JavaScript 引擎，负责解析和执行 JavaScript 代码。
- **Libuv**: 一个跨平台的异步 I/O 库，负责处理文件系统、DNS、网络、子进程、管道、信号处理、轮询和流。它提供了事件循环机制。
- **Built-in Modules**: Node.js 提供的一系列核心模块，如 `fs`, `http`, `path`, `crypto` 等。
- **Bindings (C/C++)**: 将 V8 和 Libuv 的功能暴露给 JavaScript。

### 2.2 事件循环 (Event Loop)

事件循环是 Node.js 处理非阻塞 I/O 操作的核心机制。理解事件循环对于排查性能问题和理解异步代码执行顺序至关重要。

Node.js 的事件循环包含以下几个主要阶段（Phase）：

1.  **Timers**: 执行 `setTimeout()` 和 `setInterval()` 的回调。
2.  **Pending Callbacks**: 执行延迟到下一个循环迭代的 I/O 回调。
3.  **Idle, Prepare**: 仅供系统内部使用。
4.  **Poll**: 检索新的 I/O 事件；执行与 I/O 相关的回调（几乎所有情况下，除了关闭的回调、定时器调度和 `setImmediate` 之外，其余情况都发生在这里）；node 会在适当的时候阻塞在这里。
5.  **Check**: 执行 `setImmediate()` 的回调。
6.  **Close Callbacks**: 执行一些关闭的回调，例如 `socket.on('close', ...)`。

**微任务 (Microtasks)**:

- `process.nextTick()`
- `Promise.then()` / `catch()` / `finally()`
  微任务队列不在上述任何一个阶段中，而是在**每个阶段完成之后、进入下一个阶段之前**执行。且 `process.nextTick` 优先级高于 `Promise`。

### 2.3 阻塞与非阻塞 (Blocking vs Non-blocking)

- **阻塞 (Blocking)**: 指 Node.js 进程中的 JavaScript 执行必须等待非 JavaScript 操作完成。这是因为事件循环无法在阻塞操作进行时继续运行 JavaScript。
  - 例如：`fs.readFileSync()`
- **非阻塞 (Non-blocking)**: 指 Node.js 进程中的 JavaScript 执行无需等待非 JavaScript 操作完成。
  - 例如：`fs.readFile()`

**代码示例：阻塞 vs 非阻塞**

```javascript
const fs = require("fs");

// 阻塞方式
try {
  const data = fs.readFileSync("/file.md"); // 程序暂停于此，直到文件读取完毕
  console.log(data);
} catch (err) {
  console.error(err);
}
console.log("读取完成（阻塞）");

// 非阻塞方式
fs.readFile("/file.md", (err, data) => {
  if (err) throw err;
  console.log(data);
});
console.log("读取发起（非阻塞）"); // 这行代码会先于 console.log(data) 执行
```

## 3. 模块系统 (Module System)

Node.js 支持两种模块系统：CommonJS (CJS) 和 ECMAScript Modules (ESM)。Next.js 项目通常混合使用这两种，但趋势是向 ESM 迁移。

### 3.1 CommonJS (CJS)

这是 Node.js 最初的模块系统，使用 `require()` 导入和 `module.exports` 导出。

- **同步加载**: 模块加载是同步的，适合服务端文件系统。
- **动态路径**: `require(path)` 中的 path 可以是动态计算的字符串。

**基本语法**:

```javascript
// math.js
const add = (a, b) => a + b;
const subtract = (a, b) => a - b;

module.exports = {
  add,
  subtract,
};

// app.js
const math = require("./math");
console.log(math.add(1, 2));
```

### 3.2 ECMAScript Modules (ESM)

这是 JavaScript 的官方标准模块系统，使用 `import` 和 `export`。

- **异步/同步**: ESM 的静态解析阶段是同步的，但模块加载过程可以优化。
- **静态分析**: 导入路径必须是静态字符串（除非使用动态 `import()`），这有利于 Tree Shaking。
- **顶级 Await**: Node.js 的 ESM 支持 Top-level await。

**基本语法**:

```javascript
// math.mjs
export const add = (a, b) => a + b;
export const subtract = (a, b) => a - b;

export default function multiply(a, b) {
  return a * b;
}

// app.mjs
import multiply, { add, subtract } from "./math.mjs";
console.log(add(1, 2));
```

### 3.3 CJS 与 ESM 的互操作性

在 Next.js 16 中，你可能会遇到需要在 ESM 文件中导入 CJS 模块，或者反过来。

- **ESM 导入 CJS**: 支持良好。
  ```javascript
  import lodash from "lodash"; // lodash 是 CJS 模块
  ```
- **CJS 导入 ESM**: 不支持直接 `require()` ESM 模块，必须使用动态 `import()`。

  ```javascript
  // 错误
  // const myModule = require('./my-esm-module.mjs');

  // 正确
  (async () => {
    const myModule = await import("./my-esm-module.mjs");
  })();
  ```

### 3.4 模块解析策略

Node.js 查找模块的顺序：

1.  **核心模块**: 如 `fs`, `http`。
2.  **文件模块**: 以 `/`, `./`, `../` 开头。
3.  **node_modules**: 如果不是核心模块也不是文件路径，Node.js 会从当前目录的 `node_modules` 开始向上递归查找。

## 4. 常用内置模块详解 (Core Built-in Modules)

### 4.1 path (路径处理)

`path` 模块用于处理文件路径，这在跨平台开发中至关重要（Windows 使用 `\`，POSIX 使用 `/`）。

**常用 API**:

- `path.join([...paths])`: 使用平台特定的分隔符连接路径片段。
- `path.resolve([...paths])`: 将路径或路径片段的序列解析为绝对路径。
- `path.basename(path[, ext])`: 返回路径的最后一部分。
- `path.extname(path)`: 返回路径的扩展名。
- `path.parse(path)`: 返回一个对象，其属性表示路径的重要元素（dir, root, base, name, ext）。

**示例**:

```javascript
const path = require("path");

// 跨平台路径拼接
const fullPath = path.join("data", "posts", "intro.md");
// Windows: 'data\posts\intro.md'
// Linux/macOS: 'data/posts/intro.md'

// 获取绝对路径
const absPath = path.resolve(__dirname, "config.json");

// 解析路径
const pathInfo = path.parse("/home/user/dir/file.txt");
/*
{
  root: '/',
  dir: '/home/user/dir',
  base: 'file.txt',
  ext: '.txt',
  name: 'file'
}
*/
```

### 4.2 fs (文件系统)

`fs` 模块提供与文件系统交互的 API。在 Next.js 的 `getStaticProps` 中常用此模块读取数据。

**常用 API (Promise 版推荐)**:
建议使用 `fs/promises` 或 `require('fs').promises`，配合 `async/await` 使用。

- `readFile(path[, options])`: 读取文件内容。
- `writeFile(path, data[, options])`: 写入文件。
- `readdir(path[, options])`: 读取目录内容。
- `stat(path)`: 获取文件状态（大小、创建时间等）。
- `access(path)`: 检查文件是否存在及权限。

**示例**:

```javascript
const fs = require("fs/promises");
const path = require("path");

async function getPostContent() {
  const filePath = path.join(process.cwd(), "posts", "hello.md");
  try {
    const fileContent = await fs.readFile(filePath, "utf-8");
    return fileContent;
  } catch (error) {
    console.error("读取文件失败:", error);
  }
}
```

### 4.3 events (事件触发器)

大多数 Node.js 核心 API 都是围绕惯用的异步事件驱动架构构建的。所有能触发事件的对象都是 `EventEmitter` 类的实例。

**常用 API**:

- `on(eventName, listener)`: 注册事件监听器。
- `emit(eventName[, ...args])`: 触发事件。
- `once(eventName, listener)`: 注册一次性监听器。
- `off(eventName, listener)`: 移除监听器。

**示例**:

```javascript
const EventEmitter = require("events");

class MyEmitter extends EventEmitter {}

const myEmitter = new MyEmitter();

myEmitter.on("event", (a, b) => {
  console.log(a, b, this); // 注意箭头函数中 this 的指向
});

myEmitter.emit("event", "a", "b");
```

### 4.4 stream (流)

流是处理流式数据的抽象接口。流可以是可读的、可写的，或者两者兼有。流在处理大文件或网络请求时非常高效，因为它们不需要一次性将所有数据加载到内存中。

**流的类型**:

1.  **Readable**: 可读流（例如 `fs.createReadStream`）。
2.  **Writable**: 可写流（例如 `fs.createWriteStream`）。
3.  **Duplex**: 双工流（例如 `net.Socket`）。
4.  **Transform**: 转换流（例如 `zlib.createGzip`）。

**Pipe (管道)**:
`pipe` 方法用于将可读流的数据导向可写流。

**示例：高效复制大文件**:

```javascript
const fs = require("fs");

const readStream = fs.createReadStream("huge-file.txt");
const writeStream = fs.createWriteStream("huge-file-copy.txt");

readStream.pipe(writeStream);

readStream.on("end", () => {
  console.log("文件复制完成");
});
```

### 4.5 http (HTTP 服务器与客户端)

虽然 Next.js 封装了 HTTP 服务器的创建，但在某些高级场景（如自定义 Server）下，了解 `http` 模块很有帮助。

**创建简单的 HTTP 服务器**:

```javascript
const http = require("http");

const server = http.createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader("Content-Type", "text/plain");
  res.end("Hello World");
});

server.listen(3000, "127.0.0.1", () => {
  console.log("Server running at http://127.0.0.1:3000/");
});
```

### 4.6 crypto (加密)

提供加密功能，包括对 OpenSSL 的哈希、HMAC、加密、解密、签名和验证功能的封装。

**常用场景**:

- 生成哈希 (MD5, SHA256)。
- 生成随机 Token。
- 加密敏感数据。

**示例**:

```javascript
const crypto = require("crypto");

// 生成 SHA256 哈希
const hash = crypto.createHash("sha256");
hash.update("some data to hash");
console.log(hash.digest("hex"));

// 生成随机 ID
const id = crypto.randomBytes(16).toString("hex");
console.log(id);
```

### 4.7 process (进程对象)

`process` 对象是一个全局变量，提供有关当前 Node.js 进程的信息并对其进行控制。

**常用属性和方法**:

- `process.env`: 包含用户环境的对象。Next.js 严重依赖此属性来管理配置。
- `process.cwd()`: 返回 Node.js 进程的当前工作目录。
- `process.argv`: 一个数组，包含启动 Node.js 进程时的命令行参数。
- `process.exit([code])`: 以退出码 `code` 终止进程。
- `process.nextTick(callback)`: 将回调添加到“next tick”队列。

## 5. 异步编程模式 (Asynchronous Programming Patterns)

### 5.1 Callbacks (回调函数)

最基础的异步模式，但容易导致“回调地狱 (Callback Hell)”。

```javascript
fs.readFile("file1.txt", (err, data1) => {
  if (err) return handleError(err);
  fs.readFile("file2.txt", (err, data2) => {
    if (err) return handleError(err);
    // ...
  });
});
```

### 5.2 Promises

Promise 对象用于表示一个异步操作的最终完成（或失败）及其结果值。它解决了回调地狱的问题，使链式调用成为可能。

```javascript
fs.promises
  .readFile("file1.txt")
  .then((data1) => fs.promises.readFile("file2.txt"))
  .then((data2) => {
    // ...
  })
  .catch((err) => handleError(err));
```

### 5.3 Async/Await

这是目前处理异步操作的首选方式，它基于 Promise，但语法上更像同步代码，可读性最强。

```javascript
async function readFiles() {
  try {
    const data1 = await fs.promises.readFile("file1.txt");
    const data2 = await fs.promises.readFile("file2.txt");
    // ...
  } catch (err) {
    handleError(err);
  }
}
```

### 5.4 util.promisify

Node.js 的 `util` 模块提供了一个 `promisify` 函数，可以将遵循常见的错误优先的回调风格（(err, value) => ...）的函数转换为返回 Promise 的版本。

```javascript
const util = require("util");
const fs = require("fs");

const readFileAsync = util.promisify(fs.readFile);

async function main() {
  const data = await readFileAsync("file.txt");
}
```

## 6. 全局对象 (Global Objects)

在 Node.js 中，有一些对象是全局可用的，无需 require 即可使用。

### 6.1 global

类似于浏览器中的 `window` 对象。所有全局变量（除了模块作用域内的）都挂载在 `global` 上。
**注意**: 尽量避免污染全局命名空间。

### 6.2 **dirname 和 **filename (CJS 专属)

- `__dirname`: 当前模块的目录名（绝对路径）。
- `__filename`: 当前模块的文件名（绝对路径）。

**在 ESM 中获取 \_\_dirname**:
ESM 中没有 `__dirname`，需要手动创建：

```javascript
import { fileURLToPath } from "url";
import { dirname } from "path";

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

### 6.3 Buffer

用于处理二进制数据流。在处理文件流、TCP 流等场景下非常重要。
Buffer 类的实例类似于整数数组，但大小固定，且在 V8 堆外分配物理内存。

```javascript
// 创建 Buffer
const buf = Buffer.from("hello world", "utf8");

// 转换为字符串
console.log(buf.toString("hex")); // 68656c6c6f20776f726c64
console.log(buf.toString("base64")); // aGVsbG8gd29ybGQ=
```

## 7. 进阶：多进程与集群 (Multi-process & Cluster)

Node.js 虽然是单线程的，但可以通过多进程模型利用多核 CPU 的优势。

### 7.1 child_process 模块

`child_process` 模块允许我们生成新的进程，这对于执行系统命令或处理 CPU 密集型任务非常有用。

- **exec**: 衍生一个 shell 并在该 shell 中运行命令，当完成时将 stdout 和 stderr 传给回调函数。适合小量输出。
- **spawn**: 衍生一个新进程来执行命令。通过流处理 I/O。适合大量数据处理。
- **fork**: 专门用于衍生新的 Node.js 进程。它会建立一个 IPC 通信通道，允许父子进程发送消息。

**示例：使用 fork 进行耗时计算**

```javascript
// parent.js
const { fork } = require("child_process");
const path = require("path");

const child = fork(path.join(__dirname, "child.js"));

child.on("message", (msg) => {
  console.log("收到子进程消息:", msg);
});

child.send({ hello: "world" });

// child.js
process.on("message", (msg) => {
  console.log("收到父进程消息:", msg);
  // 模拟耗时操作
  let sum = 0;
  for (let i = 0; i < 1e9; i++) sum += i;
  process.send({ result: sum });
});
```

### 7.2 Cluster 模块

`cluster` 模块允许简单的创建共享服务器端口的子进程。

```javascript
const cluster = require("cluster");
const http = require("http");
const numCPUs = require("os").cpus().length;

if (cluster.isMaster) {
  console.log(`主进程 ${process.pid} 正在运行`);

  // 衍生工作进程
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  cluster.on("exit", (worker, code, signal) => {
    console.log(`工作进程 ${worker.process.pid} 已退出`);
    // 可以选择重启
    // cluster.fork();
  });
} else {
  // 工作进程可以共享任何 TCP 连接
  // 在本例中，它是一个 HTTP 服务器
  http
    .createServer((req, res) => {
      res.writeHead(200);
      res.end("你好世界\n");
    })
    .listen(8000);

  console.log(`工作进程 ${process.pid} 已启动`);
}
```

### 7.3 PM2 进程管理

在生产环境中，我们通常不直接使用 `cluster` 模块，而是使用 PM2 这样的进程管理器。

- **自动重启**: 崩溃后自动重启。
- **负载均衡**: 利用 cluster 模式 (`pm2 start app.js -i max`)。
- **日志管理**: 集中管理日志。
- **监控**: 查看 CPU 和内存使用情况。

## 8. 进阶：内存管理与性能调优 (Memory & Performance)

### 8.1 V8 内存模型

Node.js 的内存管理基于 V8 引擎。V8 的内存主要分为：

- **栈内存 (Stack)**: 存储基本类型值和对象引用。
- **堆内存 (Heap)**: 存储对象、闭包等。堆内存又分为：
  - **新生代 (New Space)**: 存活时间短的对象（如临时变量）。使用 Scavenge 算法（Cheney 算法）进行 GC，速度快。
  - **老生代 (Old Space)**: 存活时间长的对象。使用 Mark-Sweep (标记清除) 和 Mark-Compact (标记整理) 算法，回收效率相对较低。

### 8.2 内存限制与调整

默认情况下，Node.js 对堆内存有限制（64 位系统下约为 1.4GB - 2GB，具体取决于版本）。对于内存密集型应用，可以通过 flag 调整：

```bash
node --max-old-space-size=4096 app.js # 设置为 4GB
```

### 8.3 常见内存泄漏场景

1.  **全局变量**: 意外创建的全局变量会常驻内存。
2.  **闭包**: 闭包会维持对外部作用域变量的引用，如果闭包本身未被释放，引用的变量也不会释放。
3.  **缓存**: 无限制增长的缓存对象（如使用对象或数组作为缓存）。建议使用 `lru-cache` 库。
4.  **事件监听器**: 注册了 `on` 但从未 `off`，特别是在单例对象或全局对象上。

### 8.4 性能分析工具

- **CPU Profiling**: 使用 `node --prof` 或 Chrome DevTools 找出 CPU 热点。
- **Heap Snapshot**: 抓取堆快照，对比不同时间点的内存差异，定位泄漏点。
- **clinic.js**: 一个强大的 Node.js 性能诊断工具集（Doctor, Bubbleprof, Flame）。

```bash
npm install -g clinic
clinic doctor -- node app.js
```

## 9. 进阶：Libuv 深度解析 (Libuv Deep Dive)

Node.js 的异步非阻塞特性主要归功于 Libuv。深入理解 Libuv 有助于编写高性能的 Node.js 应用。

### 9.1 Libuv 架构

Libuv 是一个跨平台的异步 I/O 库，它封装了不同操作系统（Linux 的 epoll, macOS 的 kqueue, Windows 的 IOCP）的底层异步 I/O 通知机制，并提供了一套统一的 API。

Libuv 主要包含两个核心部分：

1.  **Event Loop (事件循环)**: 负责编排和执行所有非阻塞操作。
2.  **Thread Pool (线程池)**: 负责处理无法在操作系统内核进行异步处理的任务（主要是文件 I/O、DNS 查找和部分 Crypto 操作）。

### 9.2 Event Loop 六个阶段详解

虽然在 2.2 节简要介绍了事件循环，这里我们深入每个阶段的细节：

1.  **Timers**: 检查是否有过期的定时器（setTimeout/setInterval）。Libuv 使用最小堆（Min-Heap）来管理定时器，查找最近过期定时器的时间复杂度为 O(1)。
2.  **Pending Callbacks**: 执行上一轮循环中被推迟的 I/O 回调（如某些 TCP 错误）。
3.  **Idle, Prepare**: 仅供 Libuv 内部使用。
4.  **Poll (轮询)**:
    - 这是最关键的阶段。Libuv 会计算需要阻塞多久（timeout）。
    - 如果有定时器即将过期，timeout 设置为到那个时间点的差值。
    - 如果没有定时器，且有 `setImmediate` 回调，timeout 为 0（不阻塞）。
    - 如果都没有，可能会阻塞等待 I/O 事件（如网络请求返回、文件读取完成）。
5.  **Check**: 执行 `setImmediate` 回调。
6.  **Close Callbacks**: 执行关闭句柄的回调（如 `socket.destroy()`）。

### 9.3 线程池 (Thread Pool)

**误区**: 很多人认为 Node.js 所有的异步操作都是由线程池处理的。这是错误的。
**事实**: 网络 I/O（HTTP, TCP, UDP）通常由操作系统内核直接处理（epoll/kqueue/IOCP），**不占用**线程池。

只有以下操作会使用线程池：

- 文件系统操作 (`fs.*`，除了 `fs.watch`)
- DNS 查找 (`dns.lookup`)
- 部分 Crypto 操作 (`crypto.pbkdf2`, `crypto.randomBytes` 等)
- Zlib 压缩

**线程池大小**:
默认大小为 4。如果你的应用有大量的文件 I/O 或加密操作，可以通过环境变量 `UV_THREADPOOL_SIZE` 调整（最大 1024）。

```javascript
// 在启动应用前设置，或者在命令行设置
process.env.UV_THREADPOOL_SIZE = 8;
```

## 10. 进阶：Stream 流式编程 (Stream Programming)

流（Stream）是 Node.js 中处理流式数据的抽象接口。

### 10.1 为什么需要流？

处理大数据（如几 GB 的视频文件或日志文件）时，一次性读取到内存（Buffer）会导致内存溢出（OOM）。流允许我们将数据分割成小块（Chunk），逐块处理。

- **时间效率**: 无需等待整个文件读取完毕即可开始处理（如边下载边播放）。
- **空间效率**: 极低的内存占用。

### 10.2 背压 (Backpressure)

当数据来源（Readable）的读取速度快于消费者（Writable）的处理速度时，缓冲区会积压，导致内存暴涨。
**背压**机制允许消费者告诉生产者：“慢点，我处理不过来了”。

- 当 `writable.write(chunk)` 返回 `false` 时，表示缓冲区已满（超过 `highWaterMark`），生产者应暂停写入。
- 当缓冲区清空后，Writable 触发 `drain` 事件，生产者可以恢复写入。

### 10.3 pipeline vs pipe

推荐使用 `stream.pipeline` 替代 `.pipe`，因为它能自动处理错误并正确清理流，避免内存泄漏。

```javascript
const { pipeline } = require("stream");
const fs = require("fs");
const zlib = require("zlib");

pipeline(
  fs.createReadStream("input.txt"),
  zlib.createGzip(),
  fs.createWriteStream("input.txt.gz"),
  (err) => {
    if (err) {
      console.error("Pipeline failed.", err);
    } else {
      console.log("Pipeline succeeded.");
    }
  }
);
```

## 11. 进阶：Buffer 缓冲区深度 (Buffer Deep Dive)

Buffer 是 Node.js 用来处理二进制数据的类。

### 11.1 Buffer 与 V8 内存

Buffer 的内存分配不在 V8 的堆内存（Heap）中，而是在 C++ 层面申请的堆外内存。这使得 Buffer 处理大文件时不会占用 V8 的垃圾回收（GC）压力。

### 11.2 alloc vs allocUnsafe

- `Buffer.alloc(size)`: 分配内存并用 0 填充。**安全**，但稍慢。
- `Buffer.allocUnsafe(size)`: 分配内存但不初始化。**快**，但内存中可能包含旧数据（敏感信息），**不安全**。

**最佳实践**: 除非对性能有极致要求且能确保立即重写所有数据，否则始终使用 `Buffer.alloc`。

### 11.3 常用编码

Buffer 与字符串转换时需要指定编码：

- `utf8`: 默认，变长编码。
- `hex`: 十六进制。
- `base64`: Base64 编码。
- `latin1` (binary): 单字节编码。

```javascript
const buf = Buffer.from("Next.js", "utf8");
console.log(buf.toString("hex")); // 4e6578742e6a73
console.log(buf.toString("base64")); // TmV4dC5qcw==
```

## 12. 进阶：网络编程基础 (Network Programming)

Node.js 在网络编程方面表现出色，内置了丰富的模块来处理各种网络协议。

### 12.1 TCP 服务 (net 模块)

TCP (Transmission Control Protocol) 是一种面向连接的、可靠的、基于字节流的传输层通信协议。Node.js 的 `net` 模块提供了异步的网络 API，用于创建基于流的 TCP 服务器和客户端。

**创建 TCP 服务器**:

```javascript
const net = require("net");

const server = net.createServer((socket) => {
  console.log("客户端已连接");

  socket.on("data", (data) => {
    console.log("收到数据:", data.toString());
    socket.write("服务端已收到: " + data);
  });

  socket.on("end", () => {
    console.log("客户端已断开");
  });
});

server.listen(8124, () => {
  console.log("服务器已启动，监听 8124 端口");
});
```

### 12.2 UDP 服务 (dgram 模块)

UDP (User Datagram Protocol) 是一种无连接的协议，不保证数据包的顺序或到达。它比 TCP 快，但不可靠。适用于视频流、DNS 查询等场景。

**创建 UDP 服务器**:

```javascript
const dgram = require("dgram");
const server = dgram.createSocket("udp4");

server.on("error", (err) => {
  console.log(`服务器异常:\n${err.stack}`);
  server.close();
});

server.on("message", (msg, rinfo) => {
  console.log(`服务器收到：${msg} 来自 ${rinfo.address}:${rinfo.port}`);
});

server.on("listening", () => {
  const address = server.address();
  console.log(`服务器正在 ${address.address}:${address.port} 监听`);
});

server.bind(41234);
```

### 12.3 HTTP Agent 与 Keep-Alive

在 Node.js 中发起 HTTP 请求时（使用 `http.request` 或 `axios` 等库），默认情况下，Node.js 会为每个请求创建一个新的 TCP 连接。这在大量请求的场景下效率很低。

**HTTP Keep-Alive** 允许在一次 TCP 连接中发送多个 HTTP 请求，大大减少了连接建立的开销。

```javascript
const http = require("http");

// 创建一个自定义的 Agent
const keepAliveAgent = new http.Agent({
  keepAlive: true,
  maxSockets: 25, // 每个主机允许的最大 socket 数
  keepAliveMsecs: 1000, // 保持连接的毫秒数
});

const options = {
  hostname: "localhost",
  port: 3000,
  path: "/",
  method: "GET",
  agent: keepAliveAgent, // 使用自定义 Agent
};

const req = http.request(options, (res) => {
  // ...
});
req.end();
```

## 14. Node.js 在 Next.js 中的应用场景

### 9.1 数据获取 (Data Fetching)

在 `getStaticProps` 和 `getServerSideProps` 中，代码完全运行在 Node.js 环境中。这意味着你可以直接连接数据库、读取文件系统，而不用担心这些代码会泄露到客户端。

Next.js 的 API Routes 本质上就是 Node.js 的 HTTP 处理函数（Serverless Functions）。

```javascript

  res.status(200).json({ name: "John Doe" });
```

### 9.3 Middleware (中间件)

Next.js 中间件运行在 Edge Runtime 上，这是一个基于 Web API 标准的轻量级运行时，它**不是**完整的 Node.js 环境。
**注意**: 在 Middleware 中不能使用所有的 Node.js API（如 `fs` 模块不可用，但 `crypto` 的部分功能可用）。如果需要完整的 Node.js 能力，应将逻辑移至 API Routes 或 `getServerSideProps`。

### 9.4 Server Components

React Server Components (RSC) 在服务端渲染。在 Next.js App Router 中，默认组件都是 Server Components。它们可以访问后端资源（如数据库），但不能使用浏览器特有的 API（如 `window`, `localStorage`, `useEffect`, `useState`）。

## 16. 常见问题 (FAQ)

### 16.1 `__dirname` is not defined

**问题描述**: 在 ESM 模块（`.mjs` 或 `package.json` 中 `type: "module"`）中使用 `__dirname` 报错。
**解决方案**: 参照 6.2 节，使用 `import.meta.url` 构造 `__dirname`。

### 16.2 `fs.readFileSync` 找不到文件

**问题描述**: 使用相对路径读取文件时报错 `ENOENT`。
**原因**: Node.js 中的相对路径是相对于 `process.cwd()`（启动脚本的目录），而不是当前文件所在的目录。
**解决方案**: 始终使用 `path.join(__dirname, 'relative/path')` 或 `path.join(process.cwd(), 'relative/path')` 来构建绝对路径。

### 16.3 内存泄漏

**问题描述**: 服务器运行一段时间后内存占用持续升高，最终崩溃。
**常见原因**:

- 全局变量缓存了过多数据。
- 事件监听器未移除（闭包引用）。
- 闭包持有大对象引用。
  **排查**: 使用 Chrome DevTools 的 Memory 面板进行 Heap Snapshot 分析。

### 16.4 Event Loop 阻塞

**问题描述**: 服务器响应变慢，不仅是当前请求慢，所有请求都变慢。
**原因**: 在主线程执行了 CPU 密集型任务（如大循环、图像处理、复杂的加密解密），阻塞了 Event Loop。
**解决方案**:

- 将计算任务放入 Worker Threads。
- 使用专门的服务处理计算密集型任务。
- 将大任务拆分为小任务，使用 `setImmediate` 分批执行。

---

### 16.5 进阶实战：使用 Worker Threads 处理 CPU 密集型任务

虽然 Node.js 是单线程的，但通过 `worker_threads` 模块，我们可以创建独立的线程来处理计算密集型任务，避免阻塞主事件循环。

**场景**: 计算斐波那契数列（CPU 密集型）。

**主线程代码 (main.js)**:

```javascript
const {
  Worker,
  isMainThread,
  parentPort,
  workerData,
} = require("worker_threads");
const path = require("path");

if (isMainThread) {
  module.exports = function parseJSAsync(script) {
    return new Promise((resolve, reject) => {
      const worker = new Worker(__filename, {
        workerData: script,
      });
      worker.on("message", resolve);
      worker.on("error", reject);
      worker.on("exit", (code) => {
        if (code !== 0)
          reject(new Error(`Worker stopped with exit code ${code}`));
      });
    });
  };
} else {
  const { parse } = require("some-js-parsing-library");
  const script = workerData;
  parentPort.postMessage(parse(script));
}
```

**Next.js 中的应用**:
在 Next.js 的 API Route 中，如果你需要处理图像压缩或复杂数据分析，建议使用 Worker Threads 或将任务卸载到专门的微服务。

### 16.6 运行时对比：Node.js vs Edge Runtime

在 Next.js 16 中，你经常会听到 "Edge Runtime"。理解它与标准 Node.js 运行时的区别至关重要。

| 特性                     | Node.js Runtime                    | Edge Runtime                                                 |
| :----------------------- | :--------------------------------- | :----------------------------------------------------------- |
| **环境**                 | 完整的 Node.js 环境                | 基于 V8 的轻量级运行时 (类似 Cloudflare Workers)             |
| **启动速度**             | 较慢 (冷启动可能有延迟)            | 极快 (毫秒级)                                                |
| **API 支持**             | 所有 Node.js API                   | 有限的 API (支持 Web Standards, 如 Fetch, Request, Response) |
| **Native Modules**       | 支持 (C++ 扩展)                    | 不支持                                                       |
| **流式传输 (Streaming)** | 支持                               | 支持 (且通常更高效)                                          |
| **适用场景**             | 复杂业务逻辑、数据库连接、文件操作 | 简单的中间件重定向、鉴权、个性化内容修改                     |

**如何在 Next.js 中指定运行时**:

```typescript
// app/api/hello/route.ts
export const runtime = "edge"; // 默认为 'nodejs'

export async function GET(request: Request) {
  return new Response("Hello from Edge!");
}
```

4.  **优化**: 针对耗时函数（如复杂的计算、同步的文件 I/O）进行优化，或将其移入 Worker Threads。

## 18. 适用场景 (Applicable Scenarios)

### 18.1 推荐场景

- **API 开发 (BFF 层)**: Node.js 非常适合作为 Backend for Frontend，聚合后端微服务的数据返回给前端。
- **实时应用**: 聊天室、即时通讯、在线游戏服务器（基于 WebSocket）。
- **SSR/SSG**: Next.js 的核心场景，利用 Node.js 在服务端生成 HTML。
- **工具链开发**: Webpack, Vite, ESLint 等前端工具链都运行在 Node.js 上。
- **I/O 密集型应用**: 文件上传服务、流媒体服务。

### 18.2 不推荐场景

- **CPU 密集型应用**: 视频编码、复杂的科学计算、机器学习模型训练。因为 Node.js 的单线程特性，长时间的 CPU 占用会阻塞整个事件循环，导致无法响应其他请求。
- **高安全性要求的金融系统**: 虽然 Node.js 安全性在提升，但对于极高安全要求的事务处理，传统的 Java/Go 等强类型、多线程语言可能更有优势（尽管 TypeScript 改善了类型安全）。

## 19. 注意事项 (Precautions)

### 19.1 版本管理

Node.js 版本更新较快。

- **LTS (Long Term Support)**: 生产环境务必使用 LTS 版本（偶数版本，如 v18, v20, v22）。
- **Current**: 奇数版本包含最新特性，但不保证长期维护，适合尝鲜。
- **nvm/fnm**: 强烈建议使用 nvm (Node Version Manager) 或 fnm 来管理 Node.js 版本，方便在不同项目间切换。

### 19.2 错误处理

在 Node.js 中，未捕获的异常可能导致整个进程退出。

- **Promise**: 务必添加 `.catch()` 或使用 `try/catch` 包裹 `await`。
- **uncaughtException**: 虽然可以监听此事件防止进程退出，但这通常不是好做法，因为进程状态可能已不稳定。建议的做法是记录日志并重启进程（使用 PM2 等工具）。
- **unhandledRejection**: 监听未处理的 Promise 拒绝。

### 19.3 安全性

- **输入验证**: 永远不要信任客户端输入，防止 SQL 注入（虽然 Node.js 常用 NoSQL，但也存在 NoSQL 注入）和 XSS。
- **依赖检查**: 定期使用 `npm audit` 检查依赖包的安全漏洞。
- **避免 eval**: 严禁使用 `eval()` 执行动态代码。
