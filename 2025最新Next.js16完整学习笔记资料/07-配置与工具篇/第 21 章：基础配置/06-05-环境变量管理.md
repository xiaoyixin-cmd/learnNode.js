# 环境变量管理

## 1. 概述

环境变量是应用程序配置的核心组成部分，用于存储不同环境下的配置信息，如 API 密钥、数据库连接字符串、第三方服务凭证等。在 Next.js 16 中，环境变量管理得到了进一步优化和增强。

### 1.1 环境变量的核心价值

环境变量解决了以下关键问题：

- **配置分离**：将配置与代码分离，避免硬编码敏感信息
- **环境隔离**：在不同环境（开发、测试、生产）使用不同的配置
- **安全性**：敏感信息不会被提交到版本控制系统
- **灵活性**：无需修改代码即可更改配置
- **团队协作**：团队成员可以使用各自的本地配置

### 1.2 Next.js 16 环境变量特性

🆕 **Next.js 16 新增**：Next.js 16 对环境变量的处理进行了优化，提供了更好的类型安全和运行时验证。

Next.js 16 环境变量的主要特性：

- **自动加载**：自动从 `.env*` 文件加载环境变量
- **环境区分**：支持不同环境的配置文件
- **浏览器暴露**：通过 `NEXT_PUBLIC_` 前缀控制变量暴露
- **内置变量**：提供多个内置环境变量
- **类型安全**：支持 TypeScript 类型定义
- **运行时访问**：在服务端和客户端都能访问相应的变量

### 1.3 环境变量的分类

| 类型       | 前缀           | 访问位置       | 用途                 | 示例                  |
| :--------- | :------------- | :------------- | :------------------- | :-------------------- |
| 服务端变量 | 无             | 仅服务端       | API 密钥、数据库连接 | `DATABASE_URL`        |
| 公开变量   | `NEXT_PUBLIC_` | 服务端和客户端 | 公开配置             | `NEXT_PUBLIC_API_URL` |
| 系统变量   | `NODE_ENV`     | 服务端和客户端 | 环境标识             | `NODE_ENV=production` |

## 2. 核心概念与原理

### 2.1 环境变量的加载机制

Next.js 按以下优先级加载环境变量：

```
1. 系统环境变量（最高优先级）
   ↓
2. .env.$(NODE_ENV).local
   ↓
3. .env.local（在 test 环境中不加载）
   ↓
4. .env.$(NODE_ENV)
   ↓
5. .env（最低优先级）
```

**加载流程示例**

假设 `NODE_ENV=development`，加载顺序为：

```
1. 系统环境变量
2. .env.development.local
3. .env.local
4. .env.development
5. .env
```

### 2.2 环境变量的作用域

**服务端变量**

只能在服务端访问的变量，不会暴露给浏览器：

```typescript
// app/api/users/route.ts
export async function GET() {
  // ✅ 可以访问
  const dbUrl = process.env.DATABASE_URL;

  // 这个变量不会被发送到客户端
  return Response.json({ message: "OK" });
}
```

**客户端变量**

使用 `NEXT_PUBLIC_` 前缀的变量会被内联到客户端代码中：

```typescript
// app/components/ApiClient.tsx
"use client";

export function ApiClient() {
  // ✅ 可以访问
  const apiUrl = process.env.NEXT_PUBLIC_API_URL;

  return <div>API URL: {apiUrl}</div>;
}
```

⚠️ **重要**：`NEXT_PUBLIC_` 变量会在构建时被内联到客户端代码中，因此：

- 构建后无法更改这些变量
- 这些变量会暴露给所有用户
- 不要在这些变量中存储敏感信息

### 2.3 环境变量的替换时机

**构建时替换**

`NEXT_PUBLIC_` 变量在构建时被替换：

```typescript
// 源代码
const apiUrl = process.env.NEXT_PUBLIC_API_URL;

// 构建后的代码
const apiUrl = "https://api.example.com";
```

**运行时访问**

服务端变量在运行时访问：

```typescript
// 源代码
const dbUrl = process.env.DATABASE_URL;

// 运行时从环境中读取
// 可以在不重新构建的情况下更改
```

### 2.4 环境变量的类型系统

Next.js 16 支持为环境变量添加 TypeScript 类型定义：

```typescript
// env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    DATABASE_URL: string;
    API_SECRET: string;
    NEXT_PUBLIC_API_URL: string;
    NEXT_PUBLIC_GA_ID?: string;
  }
}
```

这样可以获得类型检查和自动补全：

```typescript
// ✅ 类型安全
const dbUrl: string = process.env.DATABASE_URL;

// ❌ 类型错误
const port: number = process.env.PORT; // 类型不匹配
```

## 3. 适用场景

### 3.1 推荐使用环境变量的场景

**场景 1：API 密钥和凭证**

存储第三方服务的 API 密钥：

```bash
# .env.local
STRIPE_SECRET_KEY=sk_test_xxxxx
STRIPE_PUBLISHABLE_KEY=pk_test_xxxxx
SENDGRID_API_KEY=SG.xxxxx
```

```typescript
// app/api/payment/route.ts
import Stripe from "stripe";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!, {
  apiVersion: "2023-10-16",
});
```

**场景 2：数据库连接**

存储数据库连接字符串：

```bash
# .env.local
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
REDIS_URL=redis://localhost:6379
```

```typescript
// lib/db.ts
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_URL,
    },
  },
});
```

**场景 3：功能开关**

使用环境变量控制功能的启用/禁用：

```bash
# .env.development
NEXT_PUBLIC_ENABLE_ANALYTICS=false
NEXT_PUBLIC_ENABLE_CHAT=true
```

```typescript
// app/layout.tsx
export default function RootLayout({ children }) {
  const enableAnalytics = process.env.NEXT_PUBLIC_ENABLE_ANALYTICS === "true";

  return (
    <html>
      <body>
        {children}
        {enableAnalytics && <Analytics />}
      </body>
    </html>
  );
}
```

**场景 4：环境特定配置**

为不同环境配置不同的 API 端点：

```bash
# .env.development
NEXT_PUBLIC_API_URL=http://localhost:3001/api

# .env.production
NEXT_PUBLIC_API_URL=https://api.production.com
```

**场景 5：构建配置**

控制构建行为：

```bash
# .env
ANALYZE=true
BUNDLE_ANALYZE=true
```

```javascript
// next.config.js
const withBundleAnalyzer = require("@next/bundle-analyzer")({
  enabled: process.env.ANALYZE === "true",
});

module.exports = withBundleAnalyzer({
  // 其他配置
});
```

### 3.2 不推荐使用环境变量的场景

**场景 1：频繁变化的配置**

如果配置需要频繁更改，应该使用配置管理系统或数据库。

**场景 2：复杂的配置对象**

复杂的配置对象应该使用配置文件而非环境变量。

```typescript
// ❌ 不推荐
const config = JSON.parse(process.env.COMPLEX_CONFIG);

// ✅ 推荐
import config from "./config.json";
```

**场景 3：大量的配置项**

如果有大量配置项，应该考虑使用配置管理服务。

## 4. 环境变量文件

### 4.1 .env 文件类型

Next.js 支持多种 `.env` 文件：

| 文件名                   | 用途                 | 是否提交到 Git | 加载时机               |
| :----------------------- | :------------------- | :------------- | :--------------------- |
| `.env`                   | 所有环境的默认值     | ✅ 是          | 总是加载               |
| `.env.local`             | 本地覆盖（所有环境） | ❌ 否          | 总是加载（test 除外）  |
| `.env.development`       | 开发环境             | ✅ 是          | `NODE_ENV=development` |
| `.env.development.local` | 开发环境本地覆盖     | ❌ 否          | `NODE_ENV=development` |
| `.env.production`        | 生产环境             | ✅ 是          | `NODE_ENV=production`  |
| `.env.production.local`  | 生产环境本地覆盖     | ❌ 否          | `NODE_ENV=production`  |
| `.env.test`              | 测试环境             | ✅ 是          | `NODE_ENV=test`        |
| `.env.test.local`        | 测试环境本地覆盖     | ❌ 否          | `NODE_ENV=test`        |

### 4.2 创建 .env 文件

**基础 .env 文件**

```bash
# .env
# 所有环境的默认配置

# 应用配置
APP_NAME=My Next.js App
APP_VERSION=1.0.0

# API 配置
NEXT_PUBLIC_API_URL=https://api.example.com
```

**开发环境配置**

```bash
# .env.development
# 开发环境特定配置

# API 配置
NEXT_PUBLIC_API_URL=http://localhost:3001/api

# 调试配置
DEBUG=true
LOG_LEVEL=debug
```

**生产环境配置**

```bash
# .env.production
# 生产环境特定配置

# API 配置
NEXT_PUBLIC_API_URL=https://api.production.com

# 性能配置
ENABLE_CACHE=true
CACHE_TTL=3600
```

**本地覆盖配置**

```bash
# .env.local
# 本地开发者特定配置（不提交到 Git）

# 数据库配置
DATABASE_URL=postgresql://localhost:5432/mydb_dev

# API 密钥
STRIPE_SECRET_KEY=sk_test_xxxxx
OPENAI_API_KEY=sk-xxxxx
```

### 4.3 .gitignore 配置

确保敏感的 `.env` 文件不被提交到版本控制：

```
# .gitignore

# 环境变量
.env*.local
.env.local
.env.development.local
.env.test.local
.env.production.local
```

### 4.4 .env.example 模板

创建 `.env.example` 作为模板，提交到版本控制：

```bash
# .env.example
# 环境变量模板
# 复制此文件为 .env.local 并填入实际值

# 数据库
DATABASE_URL=postgresql://user:password@localhost:5432/dbname

# API 密钥
STRIPE_SECRET_KEY=sk_test_xxxxx
STRIPE_PUBLISHABLE_KEY=pk_test_xxxxx

# 第三方服务
SENDGRID_API_KEY=
OPENAI_API_KEY=

# 公开配置
NEXT_PUBLIC_API_URL=http://localhost:3001/api
NEXT_PUBLIC_GA_ID=
```

团队成员可以复制此文件并填入实际值：

```bash
cp .env.example .env.local
# 然后编辑 .env.local 填入实际值
```

## 5. 访问环境变量

### 5.1 在服务端组件中访问

服务端组件可以直接访问所有环境变量：

```typescript
// app/page.tsx
export default async function Page() {
  // ✅ 可以访问服务端变量
  const dbUrl = process.env.DATABASE_URL;
  const apiSecret = process.env.API_SECRET;

  // ✅ 也可以访问公开变量
  const publicApiUrl = process.env.NEXT_PUBLIC_API_URL;

  const data = await fetchData(dbUrl);

  return <div>{/* 渲染数据 */}</div>;
}
```

### 5.2 在客户端组件中访问

客户端组件只能访问 `NEXT_PUBLIC_` 前缀的变量：

```typescript
// app/components/ApiClient.tsx
"use client";

export function ApiClient() {
  // ✅ 可以访问公开变量
  const apiUrl = process.env.NEXT_PUBLIC_API_URL;

  // ❌ 无法访问服务端变量（值为 undefined）
  const dbUrl = process.env.DATABASE_URL; // undefined

  return <div>API URL: {apiUrl}</div>;
}
```

### 5.3 在 API 路由中访问

API 路由在服务端运行，可以访问所有环境变量：

```typescript
// app/api/users/route.ts
import { NextRequest, NextResponse } from "next/server";

export async function GET(request: NextRequest) {
  // ✅ 可以访问所有环境变量
  const dbUrl = process.env.DATABASE_URL;
  const apiSecret = process.env.API_SECRET;

  // 使用环境变量
  const users = await fetchUsers(dbUrl);

  return NextResponse.json(users);
}
```

### 5.4 在中间件中访问

中间件在 Edge Runtime 中运行，可以访问环境变量：

```typescript
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  // ✅ 可以访问环境变量
  const apiSecret = process.env.API_SECRET;

  // 验证请求
  const authHeader = request.headers.get("authorization");
  if (authHeader !== `Bearer ${apiSecret}`) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  return NextResponse.next();
}
```

### 5.5 在配置文件中访问

在 `next.config.js` 中可以访问环境变量：

```javascript
// next.config.js
module.exports = {
  env: {
    // 自定义环境变量
    CUSTOM_KEY: process.env.CUSTOM_KEY,
  },

  // 使用环境变量配置
  images: {
    domains: [process.env.IMAGE_DOMAIN || "example.com"],
  },

  // 条件配置
  ...(process.env.ANALYZE === "true" &&
    {
      // 启用分析
    }),
};
```

## 6. 内置环境变量

### 6.1 NODE_ENV

`NODE_ENV` 是最重要的内置环境变量，表示当前运行环境：

```typescript
// 可能的值
type NodeEnv = "development" | "production" | "test";

// 使用示例
if (process.env.NODE_ENV === "development") {
  console.log("开发环境");
} else if (process.env.NODE_ENV === "production") {
  console.log("生产环境");
}
```

**自动设置**

Next.js 会自动设置 `NODE_ENV`：

- `next dev`：设置为 `development`
- `next build` 和 `next start`：设置为 `production`
- `next test`：设置为 `test`

**使用场景**

```typescript
// 条件日志
if (process.env.NODE_ENV === "development") {
  console.log("Debug info:", data);
}

// 条件功能
const enableDebugTools = process.env.NODE_ENV === "development";

// 条件导入
if (process.env.NODE_ENV === "development") {
  const { setupDevTools } = await import("./devTools");
  setupDevTools();
}
```

## 7. TypeScript 类型定义

### 7.1 创建类型定义文件

创建 `env.d.ts` 文件定义环境变量类型：

```typescript
// env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    // 数据库
    DATABASE_URL: string;
    REDIS_URL: string;

    // API 密钥
    STRIPE_SECRET_KEY: string;
    STRIPE_PUBLISHABLE_KEY: string;
    SENDGRID_API_KEY: string;
    OPENAI_API_KEY: string;

    // 公开变量
    NEXT_PUBLIC_API_URL: string;
    NEXT_PUBLIC_GA_ID?: string;
    NEXT_PUBLIC_ENABLE_ANALYTICS?: string;

    // 系统变量
    NODE_ENV: "development" | "production" | "test";
  }
}

export {};
```

### 7.2 使用类型定义

有了类型定义后，可以获得类型检查和自动补全：

```typescript
// ✅ 类型安全
const dbUrl: string = process.env.DATABASE_URL;

// ✅ 自动补全
const apiKey = process.env.STRIPE_SECRET_KEY;

// ❌ 类型错误
const port: number = process.env.DATABASE_URL; // 类型不匹配
```

### 7.3 可选变量处理

对于可选的环境变量，使用 `?` 标记：

```typescript
declare namespace NodeJS {
  interface ProcessEnv {
    // 必需变量
    DATABASE_URL: string;

    // 可选变量
    NEXT_PUBLIC_GA_ID?: string;
  }
}
```

使用时需要处理 undefined 情况：

```typescript
// ✅ 正确处理可选变量
const gaId = process.env.NEXT_PUBLIC_GA_ID;
if (gaId) {
  // 使用 gaId
}

// ✅ 使用默认值
const gaId = process.env.NEXT_PUBLIC_GA_ID || "default-id";

// ✅ 使用可选链
const gaId = process.env.NEXT_PUBLIC_GA_ID?.toUpperCase();
```

### 7.4 环境变量验证

创建验证函数确保必需的环境变量存在：

```typescript
// lib/env.ts
function validateEnv() {
  const required = [
    "DATABASE_URL",
    "API_SECRET",
    "NEXT_PUBLIC_API_URL",
  ] as const;

  const missing = required.filter((key) => !process.env[key]);

  if (missing.length > 0) {
    throw new Error(
      `Missing required environment variables: ${missing.join(", ")}`
    );
  }
}

// 在应用启动时调用
validateEnv();
```

## 8. 环境变量的安全性

### 8.1 敏感信息保护

**不要提交敏感信息**

```bash
# ❌ 错误：提交了包含敏感信息的文件
git add .env.local
git commit -m "Add env file"

# ✅ 正确：确保 .gitignore 包含敏感文件
echo ".env*.local" >> .gitignore
```

**使用 .env.example 模板**

```bash
# .env.example
DATABASE_URL=postgresql://user:password@localhost:5432/dbname
API_SECRET=your-secret-here
```

**检查提交历史**

如果不小心提交了敏感信息，需要从 Git 历史中删除：

```bash
# 使用 git-filter-repo 删除敏感文件
git filter-repo --path .env.local --invert-paths

# 或使用 BFG Repo-Cleaner
bfg --delete-files .env.local
```

### 8.2 客户端变量安全

**不要在 NEXT*PUBLIC* 变量中存储敏感信息**

```bash
# ❌ 错误：敏感信息暴露给客户端
NEXT_PUBLIC_API_SECRET=secret-key-123

# ✅ 正确：敏感信息只在服务端
API_SECRET=secret-key-123
```

**验证客户端变量**

```typescript
// ❌ 错误：在客户端使用服务端变量
"use client";

export function Component() {
  const secret = process.env.API_SECRET; // undefined
  // ...
}

// ✅ 正确：只使用公开变量
("use client");

export function Component() {
  const apiUrl = process.env.NEXT_PUBLIC_API_URL;
  // ...
}
```

### 8.3 生产环境安全

**使用环境变量管理服务**

在生产环境中，使用专门的环境变量管理服务：

- **Vercel**：在项目设置中配置环境变量
- **AWS Secrets Manager**：存储和管理敏感信息
- **HashiCorp Vault**：企业级密钥管理
- **Azure Key Vault**：Azure 的密钥管理服务

**轮换密钥**

定期轮换 API 密钥和凭证：

```typescript
// lib/secrets.ts
export async function rotateApiKey() {
  // 1. 生成新密钥
  const newKey = await generateNewKey();

  // 2. 更新环境变量
  await updateEnvironmentVariable("API_SECRET", newKey);

  // 3. 通知相关服务
  await notifyServices(newKey);

  // 4. 废弃旧密钥（延迟一段时间）
  setTimeout(() => {
    revokeOldKey();
  }, 24 * 60 * 60 * 1000); // 24小时后
}
```

**监控异常访问**

```typescript
// lib/env-monitor.ts
const sensitiveKeys = ["DATABASE_URL", "API_SECRET"];

export function monitorEnvAccess(key: string) {
  if (sensitiveKeys.includes(key)) {
    // 发送警报
```

```typescript
const cache = new Map<string, string>();

export function getCachedEnv(key: string): string | undefined {
  if (cache.has(key)) return cache.get(key);
  const value = process.env[key];
  if (value) cache.set(key, value);
  return value;
}
```

## 10. 常见问题

### 10.1 环境变量不生效？

**问题**：修改了 `.env` 文件但变量没有更新

**解决方案**：

1. 重启开发服务器
2. 检查文件名是否正确
3. 检查变量名拼写
4. 确保没有语法错误

### 10.2 客户端无法访问变量？

**问题**：在客户端组件中 `process.env.VAR` 返回 undefined

**解决方案**：

1. 确保变量以 `NEXT_PUBLIC_` 开头
2. 重启开发服务器
3. 检查变量是否在正确的文件中定义

### 10.3 TypeScript 类型错误？

**问题**：TypeScript 提示环境变量类型错误

**解决方案**：

1. 创建 `env.d.ts` 类型定义文件
2. 确保文件在 `tsconfig.json` 的 include 中
3. 重启 TypeScript 服务器

### 10.4 生产环境变量不同？

**问题**：本地和生产环境的变量值不一致

**解决方案**：

1. 检查部署平台的环境变量配置
2. 确保 `.env.production` 文件正确
3. 验证构建时的环境变量

### 10.5 环境变量泄露？

**问题**：不小心将敏感信息提交到 Git

**解决方案**：

1. 立即从 Git 历史中删除
2. 轮换所有泄露的密钥
3. 更新 `.gitignore`
4. 使用 git-secrets 等工具防止未来泄露

## 12. 总结

环境变量是 Next.js 应用配置的核心。合理使用环境变量可以：

1. **提高安全性**：敏感信息不提交到代码库
2. **增强灵活性**：不同环境使用不同配置
3. **简化部署**：无需修改代码即可更改配置
4. **类型安全**：TypeScript 支持类型检查

关键要点：

- 使用 `.env.local` 存储本地配置
- 使用 `NEXT_PUBLIC_` 前缀暴露客户端变量
- 创建类型定义文件获得类型安全
- 验证必需的环境变量
- 不要提交敏感信息到 Git
- 使用环境变量管理服务

记住：环境变量是配置管理的基础，合理使用可以让应用更安全、更灵活、更易维护。
