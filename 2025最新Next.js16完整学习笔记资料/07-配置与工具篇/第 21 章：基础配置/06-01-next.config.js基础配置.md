# next.config.js 基础配置

## 概述

next.config.js 是 Next.js 应用的核心配置文件，位于项目根目录。这个文件让你可以自定义 Next.js 的各种行为，从构建优化到运行时设置，几乎所有的配置都可以在这里完成。

在 Next.js 16 中，配置系统经过了重大改进，引入了更多的类型安全选项和性能优化配置。配置文件支持 CommonJS 和 ES Modules 两种格式，并且可以导出函数来实现动态配置。

### 配置文件格式

Next.js 支持三种配置文件格式：

```javascript
// next.config.js (CommonJS)
module.exports = {
  reactStrictMode: true,
}

// next.config.mjs (ES Modules)
export default {
  reactStrictMode: true,
}

// next.config.ts (TypeScript)
import type { NextConfig } from 'next'

const config: NextConfig = {
  reactStrictMode: true,
}

export default config
```

### 配置文件的加载顺序

Next.js 会按照以下顺序查找配置文件：

1. next.config.ts
2. next.config.mjs
3. next.config.js

找到第一个存在的文件后就会停止查找。建议在 TypeScript 项目中使用 next.config.ts，这样可以获得完整的类型提示和类型检查。

## 基础配置选项

### reactStrictMode

启用 React 严格模式，帮助识别潜在问题。

```typescript
// next.config.ts
import type { NextConfig } from "next";

const config: NextConfig = {
  reactStrictMode: true,
};

export default config;
```

严格模式会：

- 识别不安全的生命周期
- 关于使用过时 API 的警告
- 检测意外的副作用
- 确保可复用的状态

在开发环境中，严格模式会导致组件渲染两次，这是正常行为，用于检测副作用。

### basePath

为应用设置基础路径，当你的应用部署在子路径下时很有用。

```typescript
const config: NextConfig = {
  basePath: "/docs",
};
```

配置后：

- 所有路由都会自动添加 `/docs` 前缀
- `<Link href="/about">` 会渲染为 `/docs/about`
- 静态资源路径也会自动添加前缀

```typescript
// pages/index.tsx
import Link from "next/link";

export default function Home() {
  return (
    <div>
      {/* 实际链接: /docs/about */}
      <Link href="/about">About</Link>

      {/* 图片路径: /docs/logo.png */}
      <img src="/logo.png" alt="Logo" />
    </div>
  );
}
```

### assetPrefix

配置 CDN 前缀，用于静态资源的 CDN 加速。

```typescript
const config: NextConfig = {
  assetPrefix: "https://cdn.example.com",
};
```

这会影响：

- JavaScript 和 CSS 文件
- 静态导入的图片
- public 目录下的文件

```typescript
// 使用示例
export default function Page() {
  return (
    <div>
      {/* 实际路径: https://cdn.example.com/_next/static/... */}
      <img src="/logo.png" alt="Logo" />
    </div>
  );
}
```

### env

定义环境变量，这些变量会在构建时被内联到 JavaScript bundle 中。

```typescript
const config: NextConfig = {
  env: {
    CUSTOM_KEY: "my-value",
    API_URL: "https://api.example.com",
  },
};
```

使用方式：

```typescript
// 在任何组件中使用
export default function Page() {
  return <div>API URL: {process.env.CUSTOM_KEY}</div>;
}
```

注意：这些变量会暴露给客户端，不要在这里存储敏感信息。对于敏感信息，应该使用 `.env.local` 文件并且不添加 `NEXT_PUBLIC_` 前缀。

### distDir

自定义构建输出目录，默认是 `.next`。

```typescript
const config: NextConfig = {
  distDir: "build",
};
```

这在某些部署环境中很有用，比如当 `.next` 目录名称与其他工具冲突时。

### generateEtags

控制是否为页面生成 ETag，默认为 true。

```typescript
const config: NextConfig = {
  generateEtags: false,
};
```

ETag 用于 HTTP 缓存验证。如果你使用了自定义的缓存策略或 CDN，可能需要禁用它。

### pageExtensions

自定义页面文件的扩展名，默认是 `['tsx', 'ts', 'jsx', 'js']`。

```typescript
const config: NextConfig = {
  pageExtensions: ["page.tsx", "page.ts", "page.jsx", "page.js"],
};
```

这个配置很有用，可以让你在同一个目录下混合放置页面文件和非页面文件：

```
app/
  dashboard/
    page.tsx          # 页面文件
    Dashboard.tsx     # 组件文件
    utils.ts          # 工具文件
```

配置后，只有 `*.page.tsx` 文件会被识别为页面。

### poweredByHeader

控制是否在响应头中添加 `X-Powered-By: Next.js`，默认为 true。

```typescript
const config: NextConfig = {
  poweredByHeader: false,
};
```

出于安全考虑，建议在生产环境中禁用此选项，避免暴露技术栈信息。

### compress

启用 gzip 压缩，默认为 true。

```typescript
const config: NextConfig = {
  compress: true,
};
```

如果你使用了反向代理（如 Nginx）或 CDN 来处理压缩，可以禁用此选项以减少服务器负担。

### devIndicators

配置开发环境的指示器。

```typescript
const config: NextConfig = {
  devIndicators: {
    buildActivity: true,
    buildActivityPosition: "bottom-right",
  },
};
```

选项说明：

- `buildActivity`: 是否显示构建活动指示器
- `buildActivityPosition`: 指示器位置，可选值：`'bottom-right' | 'bottom-left' | 'top-right' | 'top-left'`

### httpAgentOptions

配置 HTTP 代理选项，用于服务端请求。

```typescript
const config: NextConfig = {
  httpAgentOptions: {
    keepAlive: true,
    keepAliveMsecs: 1000,
  },
};
```

这些选项会传递给 Node.js 的 `http.Agent` 和 `https.Agent`。启用 `keepAlive` 可以提高服务端请求的性能。

### onDemandEntries

配置按需编译的行为。

```typescript
const config: NextConfig = {
  onDemandEntries: {
    maxInactiveAge: 60 * 1000, // 页面在内存中保留的时间（毫秒）
    pagesBufferLength: 5, // 同时保留在内存中的页面数量
  },
};
```

这个配置影响开发环境的内存使用和编译速度。

### output

配置构建输出模式。

```typescript
const config: NextConfig = {
  output: "standalone",
};
```

可选值：

- `undefined`: 默认输出模式
- `'standalone'`: 生成独立的服务器包，包含所有依赖
- `'export'`: 导出为静态 HTML

`standalone` 模式特别适合 Docker 部署：

```dockerfile
FROM node:18-alpine AS base

FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

FROM base AS runner
WORKDIR /app
ENV NODE_ENV production

COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000
ENV PORT 3000

CMD ["node", "server.js"]
```

### trailingSlash

控制 URL 是否以斜杠结尾。

```typescript
const config: NextConfig = {
  trailingSlash: true,
};
```

配置为 `true` 后：

- `/about` 会重定向到 `/about/`
- 导出的 HTML 文件会放在对应的目录中

这个配置影响 SEO 和静态托管的行为。

### typescript

配置 TypeScript 相关选项。

```typescript
const config: NextConfig = {
  typescript: {
    ignoreBuildErrors: false,
    tsconfigPath: "./tsconfig.json",
  },
};
```

选项说明：

- `ignoreBuildErrors`: 是否忽略 TypeScript 错误继续构建
- `tsconfigPath`: 自定义 tsconfig.json 路径

不建议在生产环境中设置 `ignoreBuildErrors: true`，这会隐藏潜在的类型错误。

## 高级配置选项

### webpack

自定义 Webpack 配置。Next.js 允许你扩展默认的 Webpack 配置。

```typescript
const config: NextConfig = {
  webpack: (config, { buildId, dev, isServer, defaultLoaders, webpack }) => {
    // 添加自定义规则
    config.module.rules.push({
      test: /\.svg$/,
      use: ["@svgr/webpack"],
    });

    // 添加插件
    config.plugins.push(
      new webpack.DefinePlugin({
        "process.env.BUILD_ID": JSON.stringify(buildId),
      })
    );

    // 修改别名
    config.resolve.alias = {
      ...config.resolve.alias,
      "@components": path.resolve(__dirname, "components"),
    };

    return config;
  },
};
```

参数说明：

- `config`: Webpack 配置对象
- `buildId`: 构建 ID
- `dev`: 是否为开发环境
- `isServer`: 是否为服务端构建
- `defaultLoaders`: Next.js 默认的 loaders
- `webpack`: Webpack 实例

实际应用示例：

```typescript
import path from "path";
import type { NextConfig } from "next";

const config: NextConfig = {
  webpack: (config, { isServer }) => {
    // 只在客户端构建时添加
    if (!isServer) {
      config.resolve.fallback = {
        ...config.resolve.fallback,
        fs: false,
        net: false,
        tls: false,
      };
    }

    // 添加 SVG 支持
    config.module.rules.push({
      test: /\.svg$/i,
      issuer: /\.[jt]sx?$/,
      use: ["@svgr/webpack"],
    });

    // 添加 YAML 支持
    config.module.rules.push({
      test: /\.ya?ml$/,
      use: "yaml-loader",
    });

    return config;
  },
};

export default config;
```

### headers

配置自定义 HTTP 响应头。

```typescript
const config: NextConfig = {
  async headers() {
    return [
      {
        source: "/:path*",
        headers: [
          {
            key: "X-DNS-Prefetch-Control",
            value: "on",
          },
          {
            key: "Strict-Transport-Security",
            value: "max-age=63072000; includeSubDomains; preload",
          },
          {
            key: "X-Frame-Options",
            value: "SAMEORIGIN",
          },
          {
            key: "X-Content-Type-Options",
            value: "nosniff",
          },
          {
            key: "X-XSS-Protection",
            value: "1; mode=block",
          },
          {
            key: "Referrer-Policy",
            value: "origin-when-cross-origin",
          },
        ],
      },
      {
        source: "/api/:path*",
        headers: [
          {
            key: "Cache-Control",
            value: "no-store, must-revalidate",
          },
        ],
      },
    ];
  },
};
```

支持的匹配模式：

- `:path`: 匹配单个路径段
- `:path*`: 匹配所有路径段
- `:slug`: 匹配命名参数

CORS 配置示例：

```typescript
          {
            key: "Access-Control-Allow-Methods",
            value: "GET,DELETE,PATCH,POST,PUT",
          },
          {
            key: "Access-Control-Allow-Headers",
            value:
              "X-CSRF-Token, X-Requested-With, Accept, Accept-Version, Content-Length, Content-MD5, Content-Type, Date, X-Api-Version",
          },
        ],
      },
    ];
  },
};
```

### redirects

配置重定向规则。

```typescript
const config: NextConfig = {
  async redirects() {
    return [
      {
        source: "/old-blog/:slug",
        destination: "/blog/:slug",
        permanent: true,
      },
      {
        source: "/blog/:slug",
        has: [
          {
            type: "query",
            key: "preview",
            value: "true",
          },
        ],
        destination: "/preview/blog/:slug",
        permanent: false,
      },
      {
        source: "/docs",
        destination: "/docs/getting-started",
        permanent: false,
      },
    ];
  },
};
```

重定向选项：

- `source`: 源路径模式
- `destination`: 目标路径
- `permanent`: 是否为永久重定向（301 vs 307）
- `has`: 条件匹配（query、cookie、header）
- `missing`: 反向条件匹配

条件重定向示例：

```typescript
const config: NextConfig = {
  async redirects() {
    return [
      // 基于 cookie 的重定向
      {
        source: "/dashboard",
        has: [
          {
            type: "cookie",
            key: "authorized",
            value: "false",
          },
        ],
        destination: "/login",
        permanent: false,
      },
      // 基于 header 的重定向
      {
        source: "/mobile",
        has: [
          {
            type: "header",
            key: "user-agent",
            value: "(.*Mobile.*)",
          },
        ],
        destination: "/mobile-app",
        permanent: false,
      },
    ];
  },
};
```

### rewrites

配置 URL 重写规则，与重定向不同，重写不会改变浏览器地址栏的 URL。

```typescript
const config: NextConfig = {
  async rewrites() {
    return [
      {
        source: "/api/:path*",
        destination: "https://api.example.com/:path*",
      },
      {
        source: "/blog/:slug",
        destination: "/news/:slug",
      },
    ];
  },
};
```

支持三种返回格式：

```typescript
const config: NextConfig = {
  async rewrites() {
    return {
      beforeFiles: [
        // 在 pages/public 文件之前检查
        {
          source: "/some-page",
          destination: "/somewhere-else",
          has: [{ type: "query", key: "overrideMe" }],
        },
      ],
      afterFiles: [
        // 在 pages/public 文件之后，动态路由之前检查
        {
          source: "/non-existent",
          destination: "/somewhere-else",
        },
      ],
      fallback: [
        // 在 pages/public 文件和动态路由之后检查
        {
          source: "/:path*",
          destination: "https://my-old-site.com/:path*",
        },
      ],
    };
  },
};
```

API 代理示例：

```typescript
const config: NextConfig = {
  async rewrites() {
    return [
      {
        source: "/api/v1/:path*",
        destination: "https://api.example.com/v1/:path*",
      },
      {
        source: "/api/v2/:path*",
        destination: "https://api-v2.example.com/:path*",
      },
    ];
  },
};
```

配置图片优化选项。

```typescript
const config: NextConfig = {
```

### i18n

配置国际化支持。

```typescript
const config: NextConfig = {
  i18n: {
    locales: ["en", "fr", "de", "zh"],
    defaultLocale: "en",
    localeDetection: true,
    domains: [
      {
        domain: "example.com",
        defaultLocale: "en",
      },
      {
        domain: "example.fr",
        defaultLocale: "fr",
      },
    ],
  },
};
```

配置后的路由行为：

- `/` → 默认语言（en）
- `/fr` → 法语
- `/fr/about` → 法语关于页面
- `/de/about` → 德语关于页面

在组件中使用：

```typescript
import { useRouter } from "next/router";

export default function Page() {
  const router = useRouter();
  const { locale, locales, defaultLocale } = router;

  return (
    <div>
      <p>当前语言: {locale}</p>
      <p>可用语言: {locales.join(", ")}</p>
      <p>默认语言: {defaultLocale}</p>

      <button onClick={() => router.push("/about", "/about", { locale: "fr" })}>
        切换到法语
      </button>
    </div>
  );
}
```

## 实战案例

### 案例 1：完整的生产环境配置

这是一个适合生产环境的完整配置示例：

```typescript
// next.config.ts
import type { NextConfig } from "next";
import path from "path";

const config: NextConfig = {
  // 基础配置
  reactStrictMode: true,
  poweredByHeader: false,
  compress: true,
  generateEtags: true,

  // 输出配置
  output: "standalone",
  distDir: ".next",

  // 环境变量
  env: {
    APP_VERSION: process.env.npm_package_version || "1.0.0",
  },

  // 图片优化
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "cdn.example.com",
        pathname: "/images/**",
      },
    ],
    formats: ["image/avif", "image/webp"],
    deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
    minimumCacheTTL: 86400, // 24小时
  },

  // 安全头
  async headers() {
    return [
      {
        source: "/:path*",
        headers: [
          {
            key: "X-DNS-Prefetch-Control",
            value: "on",
          },
          {
            key: "Strict-Transport-Security",
            value: "max-age=63072000; includeSubDomains; preload",
          },
          {
            key: "X-Frame-Options",
            value: "SAMEORIGIN",
          },
          {
            key: "X-Content-Type-Options",
            value: "nosniff",
          },
          {
            key: "X-XSS-Protection",
            value: "1; mode=block",
          },
          {
            key: "Referrer-Policy",
            value: "origin-when-cross-origin",
          },
          {
            key: "Permissions-Policy",
            value: "camera=(), microphone=(), geolocation=()",
          },
        ],
      },
    ];
  },

  // 重定向
  async redirects() {
    return [
      {
        source: "/old-blog/:slug",
        destination: "/blog/:slug",
        permanent: true,
      },
      {
        source: "/home",
        destination: "/",
        permanent: true,
      },
    ];
  },

  // API代理
  async rewrites() {
    return [
      {
        source: "/api/v1/:path*",
        destination: "https://api.example.com/v1/:path*",
      },
    ];
  },

  // Webpack配置
  webpack: (config, { isServer }) => {
    // 客户端特定配置
    if (!isServer) {
      config.resolve.fallback = {
        ...config.resolve.fallback,
        fs: false,
        net: false,
        tls: false,
      };
    }

    // 添加别名
    config.resolve.alias = {
      ...config.resolve.alias,
      "@": path.resolve(__dirname, "./src"),
      "@components": path.resolve(__dirname, "./src/components"),
      "@utils": path.resolve(__dirname, "./src/utils"),
    };

    // SVG支持
    config.module.rules.push({
      test: /\.svg$/i,
      issuer: /\.[jt]sx?$/,
      use: ["@svgr/webpack"],
    });

    return config;
  },

  // TypeScript配置
  typescript: {
    ignoreBuildErrors: false,
  },

  // 实验性功能
  experimental: {
    serverActions: {
      bodySizeLimit: "2mb",
    },
    typedRoutes: true,
  },
};

export default config;
```

### 案例 2：多环境配置

使用环境变量来区分不同环境的配置：

```typescript
// next.config.ts
import type { NextConfig } from "next";

const isDev = process.env.NODE_ENV === "development";
const isProd = process.env.NODE_ENV === "production";

const config: NextConfig = {
  reactStrictMode: true,

  // 开发环境特定配置
```

| compress             | 需要服务端压缩               | 中     |
| poweredByHeader      | 生产环境安全考虑             | 高     |

## 注意事项

### 配置文件修改后的处理

1. **开发环境**：修改 next.config.js 后需要重启开发服务器
2. **生产环境**：需要重新构建应用
3. **环境变量**：修改 .env 文件后也需要重启

### 常见错误

#### 错误 1：配置文件语法错误

```typescript
// ❌ 错误：使用了 ES6 import 但文件是 .js
import type { NextConfig } from "next";

// ✅ 正确：使用 .mjs 或 .ts 扩展名
// next.config.mjs 或 next.config.ts
```

#### 错误 2：异步配置函数未正确处理

```typescript
// ❌ 错误：直接返回 Promise
const config: NextConfig = {
  redirects() {
    return fetch("/api/redirects").then((res) => res.json());
  },
};

// ✅ 正确：使用 async/await
const config: NextConfig = {
  async redirects() {
    const res = await fetch("/api/redirects");
    return res.json();
  },
};
```

#### 错误 3：Webpack 配置破坏了默认配置

```typescript
// ❌ 错误：直接替换 rules
const config: NextConfig = {
  webpack: (config) => {
    config.module.rules = [
      // 自定义规则
    ];
    return config;
  },
};

// ✅ 正确：添加到现有规则
const config: NextConfig = {
  webpack: (config) => {
    config.module.rules.push({
      // 自定义规则
    });
    return config;
  },
};
```

### 性能考虑

1. **图片优化**：合理配置 `deviceSizes` 和 `imageSizes`，避免生成过多尺寸
2. **压缩**：如果使用了 CDN 或反向代理的压缩，可以禁用 Next.js 的压缩
3. **按需编译**：调整 `onDemandEntries` 以平衡内存使用和编译速度
4. **Webpack 配置**：避免添加过多的 loader 和 plugin，会影响构建速度

### 安全考虑

1. **环境变量**：不要在 `env` 配置中存储敏感信息
2. **响应头**：生产环境必须配置安全相关的响应头
3. **CORS**：谨慎配置 CORS，避免安全漏洞
4. **CSP**：配置 Content Security Policy 防止 XSS 攻击

```typescript
// 推荐的安全头配置
const securityHeaders = [
  {

  {
    key: "Strict-Transport-Security",
    value: "max-age=63072000; includeSubDomains; preload",
  },
  {
    key: "X-Frame-Options",
    value: "SAMEORIGIN",
  },
  {
    key: "X-Content-Type-Options",
    value: "nosniff",
  },
  {
    key: "X-XSS-Protection",
    value: "1; mode=block",
  },

    key: "Referrer-Policy",
    value: "origin-when-cross-origin",
  },
  {
    key: "Permissions-Policy",
    value: "camera=(), microphone=(), geolocation=()",
  },
```


## 常见问题

### 1. 修改 next.config.js 后为什么不生效？

修改配置文件后必须重启开发服务器。配置文件在服务器启动时加载，运行时不会重新读取。

```bash
# 停止服务器（Ctrl+C）然后重新启动
npm run dev
```

### 2. 如何在配置中使用环境变量？

可以直接使用 `process.env`：

```typescript
const config: NextConfig = {
  assetPrefix: process.env.CDN_URL || "",

  async rewrites() {
    return [
      {
        source: "/api/:path*",
        destination: `${process.env.API_URL}/:path*`,
      },
    ];
  },
};
```

### 3. webpack 配置会完全覆盖默认配置吗？

不会。你的 webpack 函数接收默认配置作为参数，你需要修改并返回它。不要创建全新的配置对象。

```typescript
// ✅ 正确
webpack: (config) => {
  config.module.rules.push(/* 新规则 */);
  return config;
};

// ❌ 错误
webpack: () => {
  return {
    /* 全新配置 */
  };
};
```

### 4. 如何为不同环境设置不同的配置？

使用环境变量或条件判断：

```typescript
const isDev = process.env.NODE_ENV === "development";
const isProd = process.env.NODE_ENV === "production";

const config: NextConfig = {
  compress: isProd,

  images: {
    remotePatterns: isDev
      ? [{ protocol: "http", hostname: "localhost" }]
      : [{ protocol: "https", hostname: "cdn.example.com" }],
  },
};
```

### 5. 如何配置多个域名的图片？

使用 `remotePatterns` 数组：

```typescript
const config: NextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "cdn1.example.com",
      },
      {
        protocol: "https",
        hostname: "cdn2.example.com",
      },
      {
        protocol: "https",
        hostname: "**.example.com", // 通配符匹配所有子域名
      },
    ],
  },
};
```

### 6. redirects 和 rewrites 有什么区别？

- **redirects**：改变浏览器地址栏的 URL，返回 301/307 状态码
- **rewrites**：不改变地址栏 URL，在服务端进行路由映射

```typescript
const config: NextConfig = {
  // 重定向：用户会看到新 URL
  async redirects() {
    return [
      {
        source: "/old-page",
        destination: "/new-page",
        permanent: true, // 301
      },
    ];
  },

  // 重写：用户仍然看到 /api/users
  async rewrites() {
    return [
      {
        source: "/api/users",
        destination: "https://api.example.com/users",
      },
    ];
  },
};
```

### 7. 如何禁用某些页面的 TypeScript 类型检查？

不建议这样做，但如果确实需要：

```typescript
const config: NextConfig = {
  typescript: {
    ignoreBuildErrors: true, // 不推荐
  },
};
```

更好的做法是修复类型错误，或者使用 `@ts-ignore` 注释特定行。

### 8. 如何配置自定义的构建 ID？

```typescript
const config: NextConfig = {
  generateBuildId: async () => {
    // 可以返回任何字符串
    return process.env.GIT_COMMIT_SHA || "my-build-id";
  },
};
```

这在需要跟踪部署版本时很有用。

### 9. 如何在 standalone 模式下包含额外的文件？

使用 `outputFileTracingIncludes`：

```typescript
const config: NextConfig = {
  output: "standalone",

  experimental: {
    outputFileTracingIncludes: {
      "/api/**/*": ["./data/**/*"],
    },
  },
};
```

### 10. 如何配置 CORS？

通过 headers 配置：

```typescript
const config: NextConfig = {

    return [

        source: "/api/:path*",
        headers: [
          { key: "Access-Control-Allow-Credentials", value: "true" },
          { key: "Access-Control-Allow-Origin", value: "*" }, // 生产环境应该指定具体域名
          {
            key: "Access-Control-Allow-Methods",
            value: "GET,DELETE,PATCH,POST,PUT",

            key: "Access-Control-Allow-Headers",

```

### 11. 如何优化大型应用的构建速度？

```typescript
const config: NextConfig = {
```

### 12. 如何处理 SVG 文件？

有两种常见方式：

```typescript
// 方式1：作为 React 组件
const config: NextConfig = {

      test: /\.svg$/i,

```

```typescript
// 方式2：作为 URL

  webpack: (config) => {
    config.module.rules.push({
      test: /\.svg$/,
      type: "asset",

<img src={logoUrl} alt="Logo" />;
```

### 13. 如何配置代理以避免 CORS 问题？

使用 rewrites：

```typescript
const config: NextConfig = {
  async rewrites() {
    return [
      {
        source: "/api/:path*",
        destination: "https://api.example.com/:path*",
      },
    ];
  },
};
```

这样客户端请求 `/api/users` 时，实际会请求 `https://api.example.com/users`，但不会有 CORS 问题。

### 14. 如何在配置中使用异步操作？

headers、redirects、rewrites 都支持异步：

```typescript
const config: NextConfig = {
  async redirects() {
    // 可以从数据库或 API 获取重定向规则
    const response = await fetch("https://api.example.com/redirects");
    const redirects = await response.json();

    return redirects.map((r) => ({
      source: r.from,
      destination: r.to,
      permanent: r.permanent,
    }));
  },
};
```

## 总结

next.config.js 是 Next.js 应用的核心配置文件，掌握它的使用对于构建生产级应用至关重要。

关键要点：

1. **基础配置**：reactStrictMode、output、compress 等是最常用的配置
2. **安全配置**：生产环境必须配置 headers 来加强安全性
3. **性能优化**：合理配置 images、webpack、experimental 选项
4. **环境区分**：使用环境变量来区分开发和生产环境
5. **类型安全**：使用 TypeScript 配置文件获得更好的开发体验

配置文件虽然功能强大，但也要避免过度配置。大多数情况下，Next.js 的默认配置已经足够好，只在确实需要时才添加自定义配置。

记住：修改配置文件后必须重启开发服务器，生产环境需要重新构建。合理使用配置可以让你的 Next.js 应用更加强大和灵活。
