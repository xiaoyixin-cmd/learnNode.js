# 代码质量工具集成

代码质量工具是现代前端开发的基础设施。通过集成 ESLint、Prettier、Husky、lint-staged 等工具,可以确保代码风格一致、减少错误、提高团队协作效率。本文将详细介绍如何在 Next.js 16 项目中集成这些工具。

## 概述

代码质量工具链通常包括:

- **ESLint**: 代码质量检查工具,发现潜在问题
- **Prettier**: 代码格式化工具,统一代码风格
- **Husky**: Git hooks 工具,在提交前执行检查
- **lint-staged**: 只对暂存文件运行检查,提高效率
- **Commitlint**: 提交信息规范检查
- **TypeScript**: 类型检查工具

### 工具对比

| 工具       | 主要功能     | 检查内容                     | 是否自动修复 |
| ---------- | ------------ | ---------------------------- | ------------ |
| ESLint     | 代码质量检查 | 语法错误、潜在问题、代码规范 | 部分支持     |
| Prettier   | 代码格式化   | 代码风格、格式               | 完全支持     |
| TypeScript | 类型检查     | 类型错误                     | 不支持       |
| Stylelint  | 样式检查     | CSS/SCSS 规范                | 部分支持     |

## ESLint 配置

### 安装 ESLint

Next.js 16 默认集成了 ESLint,但我们需要进行自定义配置。

```bash
# Next.js 项目已包含 ESLint
npm install --save-dev eslint eslint-config-next

# 安装额外的插件
npm install --save-dev @typescript-eslint/parser @typescript-eslint/eslint-plugin
npm install --save-dev eslint-plugin-react eslint-plugin-react-hooks
npm install --save-dev eslint-plugin-import eslint-plugin-jsx-a11y
```

### 基础配置

```javascript
// .eslintrc.js

  rules: {
```

## Husky 配置

### 安装 Husky

```bash
# 安装 Husky
npm install --save-dev husky

# 初始化 Husky
npx husky init
```

### 配置 pre-commit hook

```bash
# .husky/pre-commit
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

npx lint-staged
```

### 配置 commit-msg hook

```bash
# .husky/commit-msg
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

npx --no -- commitlint --edit $1
```

### package.json 配置

```json
{
  "scripts": {
    "prepare": "husky install"
  }
}
```

## lint-staged 配置

lint-staged 只对暂存的文件运行检查,大大提高效率。

### 安装 lint-staged

```bash
npm install --save-dev lint-staged
```

### 配置文件

```javascript
// .lintstagedrc.js
module.exports = {
  // TypeScript/JavaScript 文件
  "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],

  // JSON 文件
  "*.json": ["prettier --write"],

  // Markdown 文件
  "*.md": ["prettier --write"],

  // CSS/SCSS 文件
  "*.{css,scss}": ["stylelint --fix", "prettier --write"],
};
```

### 简化配置

```json
// package.json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,css,scss}": ["prettier --write"]
  }
}
```

## Commitlint 配置

Commitlint 确保提交信息符合规范。

### 安装 Commitlint

```bash
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

### 配置文件

```javascript
// commitlint.config.js
module.exports = {
  extends: ["@commitlint/config-conventional"],
  rules: {
    "type-enum": [
      2,
      "always",
      [
        "feat", // 新功能
        "fix", // 修复 bug
        "docs", // 文档更新
        "style", // 代码格式
        "refactor", // 重构
        "perf", // 性能优化
        "test", // 测试
        "chore", // 构建/工具
        "revert", // 回滚
        "build", // 构建系统
        "ci", // CI 配置
      ],
    ],
    "type-case": [2, "always", "lower-case"],
    "type-empty": [2, "never"],
    "subject-empty": [2, "never"],
    "subject-full-stop": [2, "never", "."],
    "header-max-length": [2, "always", 100],
  },
};
```

### 提交信息格式

```bash
# 格式
<type>(<scope>): <subject>

# 示例
feat(auth): 添加用户登录功能
fix(api): 修复数据获取错误
docs(readme): 更新安装说明
style(button): 调整按钮样式
refactor(utils): 重构工具函数
perf(image): 优化图片加载
test(auth): 添加登录测试
chore(deps): 更新依赖版本
```

```javascript
module.exports = {
  meta: {
    type: "problem",
    docs: {
      description: "禁止直接操作 DOM",
    },
  },
  create(context) {
    return {
      MemberExpression(node) {
        if (
          node.object.name === "document" &&

      },
    };
  },
```

### 自定义 ESLint 插件

```javascript
// eslint-plugin-custom/index.js
module.exports = {
  rules: {
    "no-hardcoded-api-url": {
      meta: {
        type: "problem",
        docs: {
          description: "禁止硬编码 API URL",
        },
        fixable: "code",
      },
      create(context) {
        return {
          Literal(node) {
            if (
              typeof node.value === "string" &&
              node.value.startsWith("http")
            ) {
              context.report({
                node,
                message: "请使用环境变量而不是硬编码 URL",
                fix(fixer) {
                  return fixer.replaceText(
                    node,
                    "process.env.NEXT_PUBLIC_API_URL"
                  );
                },
              });
            }
          },
        };
      },
    },
  },
};

// .eslintrc.js
module.exports = {
  plugins: ["custom"],
  rules: {
    "custom/no-hardcoded-api-url": "error",
  },
};
```

## 注意事项

### 1. 配置冲突

```javascript
// 错误: ESLint 和 Prettier 规则冲突
// .eslintrc.js
module.exports = {
  rules: {
    "max-len": ["error", { code: 80 }], // 与 Prettier 冲突
  },
};

// 正确: 使用 eslint-config-prettier 禁用冲突规则
module.exports = {
  extends: [
    "next/core-web-vitals",
    "plugin:prettier/recommended", // 必须放在最后
  ],
};
```

### 2. 性能问题

```javascript
// 错误: 检查所有文件
{
  "scripts": {
    "lint": "eslint ."
  }
}

// 正确: 只检查源代码
{
  "scripts": {
    "lint": "eslint 'app/**/*.{js,jsx,ts,tsx}' 'components/**/*.{js,jsx,ts,tsx}'"
  }
}

// 更好: 使用 lint-staged 只检查修改的文件
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["eslint --fix"]
  }
}
```

### 3. Git Hooks 失效

```bash
# 问题: Husky hooks 不执行
# 解决方案 1: 重新安装 Husky
rm -rf .husky
npx husky install
npx husky add .husky/pre-commit "npx lint-staged"

# 解决方案 2: 检查文件权限
chmod +x .husky/pre-commit

# 解决方案 3: 检查 Git 配置
git config core.hooksPath
```

### 4. 忽略文件配置

```bash
# .eslintignore
# 忽略构建产物
.next/
out/
dist/
build/

# 忽略依赖
node_modules/

# 忽略配置文件
*.config.js
next-env.d.ts

# 忽略特定文件
public/
coverage/
```

### 5. 渐进式采用

```javascript

module.exports = {
  extends: ["next/core-web-vitals"],
  rules: {
    "no-console": "warn", // 先警告

  extends: ["next/core-web-vitals"],
```

**问题**: Git hooks 没有运行

**解决方案**:


## 总结

代码质量工具是现代前端开发的基础设施。通过合理配置和使用这些工具,可以:

1. **提高代码质量**: ESLint 发现潜在问题,TypeScript 提供类型安全
2. **统一代码风格**: Prettier 自动格式化,避免团队争议
3. **自动化检查**: Husky 和 lint-staged 在提交前自动检查
4. **规范提交历史**: Commitlint 确保提交信息清晰
5. **提升开发效率**: 编辑器集成,实时反馈

关键要点:

- 使用 ESLint 检查代码质量
- 使用 Prettier 统一代码风格
- 使用 Husky 自动化 Git hooks
- 使用 lint-staged 只检查修改的文件
- 使用 Commitlint 规范提交信息
- 配置 VSCode 实现编辑器集成
- 在 CI/CD 中运行自动化检查
- 渐进式采用,避免一次性改动过大
- 注意配置冲突,使用 eslint-config-prettier
- 定期更新工具和规则

记住:工具是为了提高效率,不是增加负担。根据团队实际情况选择合适的工具和配置,逐步建立完善的代码质量体系。
