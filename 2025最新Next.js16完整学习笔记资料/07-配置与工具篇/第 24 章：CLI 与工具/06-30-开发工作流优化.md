# 开发工作流优化

开发工作流是团队协作和项目交付的核心。一个优化的工作流可以提高开发效率、减少错误、加快交付速度。本文将详细介绍如何在 Next.js 16 项目中优化开发工作流,包括 Git 工作流、CI/CD、自动化部署、代码审查等方面。

## 概述

现代开发工作流通常包括以下环节:

- **版本控制**: Git 分支管理策略
- **代码审查**: Pull Request 流程
- **持续集成**: 自动化测试和构建
- **持续部署**: 自动化部署到各个环境
- **监控反馈**: 生产环境监控和错误追踪

### 工作流对比

| 工作流类型  | 适用场景          | 优点                    | 缺点                 |
| ----------- | ----------------- | ----------------------- | -------------------- |
| Git Flow    | 大型项目,定期发布 | 流程清晰,适合多版本维护 | 分支较多,流程复杂    |
| GitHub Flow | 持续部署项目      | 简单直接,快速迭代       | 不适合多版本维护     |
| GitLab Flow | 需要多环境部署    | 环境分支清晰            | 需要额外维护环境分支 |
| Trunk Based | 高频部署团队      | 简化分支管理            | 需要强大的 CI/CD     |

## Git 工作流

### GitHub Flow

GitHub Flow 是最简单实用的工作流,适合持续部署的项目。

```bash
# 1. 从 main 分支创建功能分支
git checkout main
git pull origin main
git checkout -b feature/user-authentication

# 2. 开发功能并提交
git add .
git commit -m "feat(auth): 添加用户认证功能"

# 3. 推送到远程
git push origin feature/user-authentication

# 4. 创建 Pull Request
# 在 GitHub 上创建 PR,等待代码审查

# 5. 合并到 main
# 审查通过后,合并 PR

# 6. 删除功能分支
git branch -d feature/user-authentication
git push origin --delete feature/user-authentication
```

### Git Flow

Git Flow 适合需要维护多个版本的大型项目。

```bash
# 初始化 Git Flow
git flow init

# 开始新功能
git flow feature start user-authentication

# 完成功能
git flow feature finish user-authentication

# 开始发布
git flow release start 1.0.0

# 完成发布
git flow release finish 1.0.0

# 紧急修复
git flow hotfix start critical-bug
git flow hotfix finish critical-bug
```

## 持续集成 (CI)

### GitHub Actions 配置

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  quality:
    name: 代码质量检查
    runs-on: ubuntu-latest

    steps:
      - name: 检出代码
        uses: actions/checkout@v4

      - name: 设置 Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: 安装依赖
        run: npm ci

      - name: 类型检查
        run: npm run type-check

      - name: 代码检查
        run: npm run lint

      - name: 格式检查
        run: npm run format:check

  test:
    name: 运行测试
    runs-on: ubuntu-latest

    steps:
      - name: 检出代码
        uses: actions/checkout@v4

      - name: 设置 Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: 安装依赖
        run: npm ci

      - name: 运行单元测试
        run: npm run test:unit

      - name: 运行集成测试
        run: npm run test:integration

      - name: 上传覆盖率报告
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/coverage-final.json

  build:
    name: 构建检查
    runs-on: ubuntu-latest

    steps:
      - name: 检出代码
        uses: actions/checkout@v4

      - name: 设置 Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: 安装依赖
        run: npm ci

      - name: 构建项目
        run: npm run build
        env:
          NEXT_PUBLIC_API_URL: ${{ secrets.API_URL }}

      - name: 检查构建产物
        run: |
          if [ ! -d ".next" ]; then
            echo "构建失败: .next 目录不存在"
            exit 1
          fi
```

```dockerfile
# Dockerfile
FROM node:20-alpine AS base

# 依赖阶段
FROM base AS deps
RUN apk add --no-cache libc6-compat
WORKDIR /app

RUN npm ci

# 构建阶段

COPY --from=deps /app/node_modules ./node_modules

# 运行阶段

ENV NEXT_TELEMETRY_DISABLED 1

RUN adduser --system --uid 1001 nextjs

COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

USER nextjs

EXPOSE 3000

ENV PORT 3000
ENV HOSTNAME "0.0.0.0"

CMD ["node", "server.js"]
```

## 自动化脚本

### 发布脚本

```javascript
// scripts/release.js
const { execSync } = require("child_process");
const fs = require("fs");

const version = process.argv[2];
if (!version) {
  console.error("请指定版本号: npm run release 1.0.0");
  process.exit(1);
}

console.log(`开始发布版本 ${version}...`);

try {
  // 1. 运行测试
  console.log("1. 运行测试...");
  execSync("npm test", { stdio: "inherit" });

  // 2. 更新版本号
  console.log("2. 更新版本号...");
  execSync(`npm version ${version} --no-git-tag-version`, { stdio: "inherit" });

  // 3. 构建项目
  console.log("3. 构建项目...");
  execSync("npm run build", { stdio: "inherit" });

  // 4. 生成变更日志
  console.log("4. 生成变更日志...");
  execSync("npm run changelog", { stdio: "inherit" });

  // 5. 提交变更
  console.log("5. 提交变更...");
  execSync("git add .", { stdio: "inherit" });
  execSync(`git commit -m "chore: release v${version}"`, { stdio: "inherit" });
  execSync(`git tag v${version}`, { stdio: "inherit" });

  // 6. 推送到远程
  console.log("6. 推送到远程...");
  execSync("git push", { stdio: "inherit" });
  execSync("git push --tags", { stdio: "inherit" });

  console.log(`✅ 版本 ${version} 发布成功!`);
} catch (error) {
  console.error("❌ 发布失败:", error.message);
  process.exit(1);
}
```

### 备份脚本

```javascript
// scripts/backup.js
const { execSync } = require("child_process");
const fs = require("fs");
const path = require("path");

const timestamp = new Date().toISOString().replace(/[:.]/g, "-");
const backupDir = path.join(__dirname, "..", "backups");

if (!fs.existsSync(backupDir)) {
  fs.mkdirSync(backupDir);
}

console.log("开始备份...");

try {
  // 备份数据库
  const dbBackup = path.join(backupDir, `db-${timestamp}.sql`);
  execSync(`pg_dump ${process.env.DATABASE_URL} > ${dbBackup}`);
  console.log(`数据库备份: ${dbBackup}`);

  // 备份上传文件
  const filesBackup = path.join(backupDir, `files-${timestamp}.tar.gz`);
  execSync(`tar -czf ${filesBackup} public/uploads`);
  console.log(`文件备份: ${filesBackup}`);

  // 清理旧备份(保留最近7天)
  const files = fs.readdirSync(backupDir);
  const now = Date.now();
  const sevenDays = 7 * 24 * 60 * 60 * 1000;

  files.forEach((file) => {
    const filePath = path.join(backupDir, file);
    const stats = fs.statSync(filePath);
    if (now - stats.mtimeMs > sevenDays) {
      fs.unlinkSync(filePath);
      console.log(`删除旧备份: ${file}`);
    }
  });

  console.log("✅ 备份完成!");
} catch (error) {
  console.error("❌ 备份失败:", error.message);
  process.exit(1);
}
```

```yaml
# 适合 5-20 人团队,需要多环境
# .github/workflows/multi-env-deploy.yml
name: Multi Environment Deploy

on:
  push:
    branches: [main, develop, staging]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: 安装依赖
        run: npm ci

      - name: 运行测试
        run: npm test

      - name: 确定环境
        id: env
        run: |
          if [[ $GITHUB_REF == 'refs/heads/main' ]]; then
            echo "environment=production" >> $GITHUB_OUTPUT
          elif [[ $GITHUB_REF == 'refs/heads/staging' ]]; then
            echo "environment=staging" >> $GITHUB_OUTPUT
          else
            echo "environment=development" >> $GITHUB_OUTPUT
          fi

      - name: 构建
        run: npm run build
        env:
          NODE_ENV: ${{ steps.env.outputs.environment }}
          NEXT_PUBLIC_API_URL: ${{ secrets[format('API_URL_{0}', steps.env.outputs.environment)] }}

      - name: 部署
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: ${{ steps.env.outputs.environment == 'production' && '--prod' || '' }}
          alias-domains: ${{ steps.env.outputs.environment }}.example.com
```

### 案例 5: 自动回滚机制

```yaml
# .github/workflows/deploy-with-rollback.yml
name: Deploy with Rollback

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: 安装依赖
        run: npm ci

      - name: 运行测试
        run: npm test

      - name: 构建
        run: npm run build

      - name: 保存当前版本
        id: current
        run: |
          CURRENT_VERSION=$(curl -s https://api.example.com/version)
          echo "version=$CURRENT_VERSION" >> $GITHUB_OUTPUT

      - name: 部署新版本
        id: deploy
        run: |
          # 部署逻辑
          npm run deploy

      - name: 健康检查
        id: health
        run: |
          sleep 30
          HEALTH=$(curl -s https://api.example.com/health)
          if [[ $HEALTH != "ok" ]]; then
            echo "健康检查失败"
            exit 1
          fi

      - name: 回滚
        if: failure()
        run: |
          echo "部署失败,回滚到版本 ${{ steps.current.outputs.version }}"
          npm run rollback ${{ steps.current.outputs.version }}
```

**解决方案**:

```bash
# scripts/blue-green-deploy.sh
#!/bin/bash

BLUE_PORT=3000
GREEN_PORT=3001
NGINX_CONFIG="/etc/nginx/sites-available/app"

# 检查当前运行的版本
CURRENT_PORT=$(grep "proxy_pass" $NGINX_CONFIG | grep -oP '\d+')

if [ "$CURRENT_PORT" == "$BLUE_PORT" ]; then
  NEW_PORT=$GREEN_PORT
  OLD_PORT=$BLUE_PORT
else
  NEW_PORT=$BLUE_PORT
  OLD_PORT=$GREEN_PORT
fi

echo "当前版本运行在端口 $OLD_PORT"
echo "新版本将部署到端口 $NEW_PORT"

# 部署新版本
PORT=$NEW_PORT npm run build
PORT=$NEW_PORT pm2 start npm --name "next-app-$NEW_PORT" -- start

# 等待新版本启动
sleep 10

# 健康检查
HEALTH=$(curl -s http://localhost:$NEW_PORT/api/health)
if [[ $HEALTH != "ok" ]]; then
  echo "健康检查失败,回滚"
  pm2 delete "next-app-$NEW_PORT"
  exit 1
fi

# 切换 Nginx 配置
sed -i "s/proxy_pass http:\/\/localhost:$OLD_PORT/proxy_pass http:\/\/localhost:$NEW_PORT/" $NGINX_CONFIG
nginx -s reload

# 停止旧版本
pm2 delete "next-app-$OLD_PORT"

echo "部署完成"
```

## 总结

开发工作流优化是提高团队效率的关键。通过合理配置 Git 工作流、CI/CD、自动化部署等环节,可以:

1. **提高开发效率**: 自动化重复性工作,让开发者专注于编码
2. **保证代码质量**: 通过自动化测试和代码审查,减少 bug
3. **加快交付速度**: 持续集成和持续部署,快速迭代
4. **降低部署风险**: 自动化检查和回滚机制,确保稳定性
5. **改善团队协作**: 统一的工作流程,减少沟通成本

关键要点:

- 选择适合团队规模的 Git 工作流
- 配置完善的 CI/CD 流程
- 实现多环境部署策略
- 建立代码审查机制
- 使用自动化脚本简化操作
- 配置分支保护规则
- 妥善管理密钥和环境变量
- 实现自动化测试和性能检查
- 建立监控和告警机制
- 定期备份和更新依赖

记住:工作流优化是一个持续改进的过程。根据团队实际情况,逐步完善流程,不要一次性引入过多变化。从简单开始,逐步增加自动化程度,最终建立适合团队的高效工作流。
