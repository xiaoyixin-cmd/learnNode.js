# 路由段配置详解

## 1. 概述

路由段配置(Route Segment Config)是 Next.js 16 提供的一组特殊导出选项,用于配置页面、布局或路由处理器的行为。通过这些配置选项,你可以精确控制路由的渲染模式、缓存策略、运行时环境等关键特性。

### 1.1 概念定义

路由段配置是在页面或布局文件中通过导出特定变量来实现的配置机制。这些配置会影响该路由段及其子路由的行为。

**关键特征**:

- **文件级配置**: 在 `page.tsx`、`layout.tsx` 或 `route.ts` 中导出配置变量
- **继承性**: 子路由会继承父路由的配置(除非被覆盖)
- **类型安全**: 配置选项有明确的类型定义
- **构建时生效**: 大部分配置在构建时确定,影响生成的代码

**配置方式**:

```tsx
// app/page.tsx
export const dynamic = "force-dynamic";
export const revalidate = 3600;

export default function Page() {
  return <div>My Page</div>;
}
```

### 1.2 核心价值

**精确控制渲染行为**:

通过配置选项,你可以明确指定页面是静态生成还是动态渲染,是否使用缓存,何时重新验证数据等。这让你能够根据具体需求优化每个页面的性能。

**简化配置管理**:

相比在代码中散布各种配置逻辑,路由段配置提供了统一的配置入口,使得配置更加清晰和易于维护。

**优化构建产物**:

Next.js 会根据路由段配置优化构建产物,例如为静态页面生成 HTML 文件,为动态页面生成服务端渲染代码。

**提升开发体验**:

配置选项有明确的类型定义和文档,IDE 可以提供自动补全和类型检查,减少配置错误。

### 1.3 配置选项概览

Next.js 16 提供了以下路由段配置选项:

| 配置选项          | 类型                                                                                                                      | 默认值         | 作用                 |
| :---------------- | :------------------------------------------------------------------------------------------------------------------------ | :------------- | :------------------- |
| `dynamic`         | `'auto' \| 'force-dynamic' \| 'force-static' \| 'error'`                                                                  | `'auto'`       | 控制路由的渲染模式   |
| `dynamicParams`   | `boolean`                                                                                                                 | `true`         | 控制是否允许动态参数 |
| `revalidate`      | `false \| 0 \| number`                                                                                                    | `false`        | 设置重新验证时间(秒) |
| `fetchCache`      | `'auto' \| 'default-cache' \| 'only-cache' \| 'force-cache' \| 'force-no-store' \| 'default-no-store' \| 'only-no-store'` | `'auto'`       | 控制 fetch 缓存行为  |
| `runtime`         | `'nodejs' \| 'edge'`                                                                                                      | `'nodejs'`     | 指定运行时环境       |
| `preferredRegion` | `string \| string[]`                                                                                                      | `'auto'`       | 指定首选部署区域     |
| `maxDuration`     | `number`                                                                                                                  | 取决于部署平台 | 设置最大执行时间(秒) |

## 2. 核心概念与原理

### 2.1 dynamic 配置

`dynamic` 配置控制路由的渲染模式,是最重要的配置选项之一。

**`'auto'` (默认值)**:

Next.js 自动决定渲染模式。如果页面使用了动态函数(如 `cookies()`、`headers()`、`searchParams`)或未缓存的数据获取,则使用动态渲染;否则使用静态渲染。

```tsx
// app/page.tsx
export const dynamic = "auto"; // 可以省略,这是默认值

export default async function Page() {
  // Next.js 会检测到 fetch 使用了 no-store,自动使用动态渲染
  const data = await fetch("https://api.example.com/data", {
    cache: "no-store",
  });

  return <div>{/* ... */}</div>;
}
```

**`'force-dynamic'`**:

强制使用动态渲染,即使页面没有使用动态函数。每次请求都会重新渲染页面。

```tsx
// app/dashboard/page.tsx
export const dynamic = "force-dynamic";

export default async function DashboardPage() {
  // 即使这里使用了缓存的 fetch,页面仍然会动态渲染
  const data = await fetch("https://api.example.com/data");

  return <div>{/* ... */}</div>;
}
```

**适用场景**:

- 需要实时数据的页面
- 个性化内容页面
- 依赖请求头或 Cookie 的页面

**`'force-static'`**:

强制使用静态渲染,即使页面使用了动态函数。如果页面确实需要动态数据,会在构建时报错。

```tsx
// app/blog/[slug]/page.tsx
export const dynamic = "force-static";

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await getPost(slug);

  return <article>{/* ... */}</article>;
}
```

**适用场景**:

- 博客文章、文档等静态内容
- 不需要个性化的营销页面
- 需要最佳性能的页面

**`'error'`**:

如果页面使用了动态函数,会在构建时抛出错误。用于确保页面保持静态。

```tsx
// app/about/page.tsx
export const dynamic = "error";

export default function AboutPage() {
  // ❌ 如果这里使用 cookies() 或 headers(),构建会失败
  return <div>{/* ... */}</div>;
}
```

### 2.2 dynamicParams 配置

`dynamicParams` 控制是否允许访问未在 `generateStaticParams` 中生成的动态路由参数。

**`true` (默认值)**:

允许访问未预生成的动态参数,这些页面会在首次请求时按需生成。

```tsx
// app/products/[id]/page.tsx
export const dynamicParams = true; // 默认值

export async function generateStaticParams() {
  // 只预生成前 100 个产品
  return Array.from({ length: 100 }, (_, i) => ({
    id: String(i + 1),
  }));
}

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  // 即使 id 大于 100,也能正常访问
  const product = await getProduct(id);

  return <div>{/* ... */}</div>;
}
```

**`false`**:

只允许访问在 `generateStaticParams` 中生成的参数,访问其他参数会返回 404。

```tsx
// app/docs/[slug]/page.tsx
export const dynamicParams = false;

export async function generateStaticParams() {
  const docs = await getAllDocs();
  return docs.map((doc) => ({
    slug: doc.slug,
  }));
}

export default async function DocPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const doc = await getDoc(slug);

  return <article>{/* ... */}</article>;
}
```

**适用场景**:

- `dynamicParams: true`: 产品目录、用户资料等数量庞大的动态内容
- `dynamicParams: false`: 文档、博客等数量有限且已知的内容

### 2.3 revalidate 配置

`revalidate` 设置页面的重新验证时间,用于实现增量静态再生成(ISR)。

**`false` (默认值)**:

页面永久缓存,直到下次构建。

```tsx
// app/page.tsx
export const revalidate = false; // 默认值

export default async function HomePage() {
  const data = await fetch("https://api.example.com/data");
  return <div>{/* ... */}</div>;
}
```

**数字值(秒)**:

设置重新验证的时间间隔。

```tsx
// app/news/page.tsx
export const revalidate = 60; // 每 60 秒重新验证一次

export default async function NewsPage() {
  const news = await fetch("https://api.example.com/news");
  return <div>{/* ... */}</div>;
}
```

**`0`**:

等同于 `dynamic = 'force-dynamic'`,每次请求都重新渲染。

```tsx
// app/live/page.tsx
export const revalidate = 0;

export default async function LivePage() {
  const liveData = await fetch("https://api.example.com/live");
  return <div>{/* ... */}</div>;
}
```

**工作原理**:

1. 首次请求时,返回缓存的静态页面
2. 后续请求在 `revalidate` 时间内,继续返回缓存页面
3. 超过 `revalidate` 时间后的首次请求,触发后台重新生成
4. 重新生成完成后,后续请求返回新页面

### 2.4 fetchCache 配置

`fetchCache` 控制路由中所有 `fetch` 请求的默认缓存行为。

**配置选项**:

| 值                   | 说明                                                     |
| :------------------- | :------------------------------------------------------- |
| `'auto'`             | 默认行为,使用 fetch 自己的 cache 选项                    |
| `'default-cache'`    | 允许 fetch 使用自己的 cache 选项,默认为 `force-cache`    |
| `'only-cache'`       | 所有 fetch 都使用 `force-cache`,如果有 `no-store` 会报错 |
| `'force-cache'`      | 所有 fetch 都使用 `force-cache`                          |
| `'default-no-store'` | 允许 fetch 使用自己的 cache 选项,默认为 `no-store`       |
| `'only-no-store'`    | 所有 fetch 都使用 `no-store`,如果有 `force-cache` 会报错 |
| `'force-no-store'`   | 所有 fetch 都使用 `no-store`                             |

**示例**:

```tsx
// app/products/page.tsx
export const fetchCache = "force-cache";

export default async function ProductsPage() {
  // 这个 fetch 会使用 force-cache,即使没有指定
  const products = await fetch("https://api.example.com/products");

  return <div>{/* ... */}</div>;
}
```

### 2.5 runtime 配置

`runtime` 指定路由的运行时环境。

**`'nodejs'` (默认值)**:

在 Node.js 运行时执行,支持所有 Node.js API。

```tsx
// app/api/data/route.ts
export const runtime = "nodejs";

export async function GET() {
  // 可以使用 Node.js API
  const fs = require("fs");
  const data = fs.readFileSync("./data.json");

  return Response.json(data);
}
```

**`'edge'`**:

在 Edge 运行时执行,启动更快,但 API 受限。

```tsx
// app/api/hello/route.ts
export const runtime = "edge";

export async function GET() {
  // 只能使用 Web API
  return new Response("Hello from Edge!");
}
```

**Edge Runtime 限制**:

- 不能使用 Node.js API(如 `fs`、`path`)
- 不能使用某些 npm 包
- 代码包大小有限制(通常 1-4MB)

**适用场景**:

- `nodejs`: 需要 Node.js API 的复杂逻辑
- `edge`: 需要低延迟的简单 API 或中间件

### 2.6 preferredRegion 配置

`preferredRegion` 指定路由的首选部署区域,仅在支持的平台(如 Vercel)上生效。

```tsx
// app/api/users/route.ts
export const preferredRegion = "iad1"; // 美国东部

export async function GET() {
  return Response.json({ users: [] });
}
```

**多区域配置**:

```tsx
export const preferredRegion = ["iad1", "sfo1"]; // 美国东部和西部
```

**常用区域代码**:

- `iad1`: 美国东部(弗吉尼亚)
- `sfo1`: 美国西部(旧金山)
- `hnd1`: 日本(东京)
- `fra1`: 德国(法兰克福)

### 2.7 maxDuration 配置

`maxDuration` 设置路由的最大执行时间(秒)。

```tsx
// app/api/long-task/route.ts
export const maxDuration = 300; // 5 分钟

export async function POST() {
  // 执行耗时任务
  await processLargeFile();

  return Response.json({ success: true });
}
```

**默认值**:

- Vercel Hobby: 10 秒
- Vercel Pro: 60 秒
- Vercel Enterprise: 900 秒(15 分钟)

## 3. 适用场景

### 3.1 静态博客网站

**场景**: 博客文章内容不经常变化,需要最佳性能和 SEO。

```tsx
// app/blog/[slug]/page.tsx
export const dynamic = "force-static";
export const dynamicParams = false;
export const revalidate = false;

export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map((post) => ({
    slug: post.slug,
  }));
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await getPost(slug);

  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />
    </article>
  );
}
```

### 3.2 新闻网站(ISR)

**场景**: 新闻内容需要定期更新,但不需要实时。

```tsx
// app/news/page.tsx
export const revalidate = 300; // 5 分钟重新验证

export default async function NewsPage() {
  const news = await fetch("https://api.example.com/news");
  const data = await news.json();

  return (
    <div>
      <h1>最新新闻</h1>
      {data.map((item) => (
        <article key={item.id}>
          <h2>{item.title}</h2>
          <p>{item.summary}</p>
        </article>
      ))}
    </div>
  );
}
```

### 3.3 实时仪表板

**场景**: 需要显示实时数据,每次请求都要获取最新信息。

```tsx
export default async function Dashboard() {
```

## 6. 注意事项

### 6.1 配置继承

子路由会继承父路由的配置,但可以被覆盖:

```tsx
// app/layout.tsx
export const dynamic = "force-static";

// app/dashboard/page.tsx
export const dynamic = "force-dynamic"; // 覆盖父配置
```

### 6.2 配置冲突

某些配置组合是冲突的:

```tsx
// ❌ 错误:force-static 和 revalidate = 0 冲突
export const dynamic = "force-static";
export const revalidate = 0;

// ✅ 正确
export const dynamic = "force-dynamic";
export const revalidate = 0;
```

### 6.3 Edge Runtime 限制

使用 Edge Runtime 时,不能使用 Node.js API:

```tsx
// ❌ 错误:Edge Runtime 不支持 fs
export const runtime = "edge";

export async function GET() {
  const fs = require("fs"); // 运行时错误
  return Response.json({});
}
```

## 7. 常见问题

### 7.1 dynamic 和 revalidate 有什么区别?

`dynamic` 控制渲染模式,`revalidate` 控制缓存时间。

- `dynamic = 'force-static'` + `revalidate = 3600`: 静态生成,每小时重新验证
- `dynamic = 'force-dynamic'`: 每次请求都动态渲染,不缓存

### 7.2 如何强制页面每次都重新渲染?

```tsx
export const dynamic = "force-dynamic";
export const revalidate = 0;
```

或者:

```tsx
export const dynamic = "force-dynamic";
```

### 7.3 Edge Runtime 和 Node.js Runtime 如何选择?

- 需要 Node.js API: 使用 `nodejs`
- 需要低延迟: 使用 `edge`
- 复杂逻辑: 使用 `nodejs`
- 简单 API: 使用 `edge`

## 8. 总结

路由段配置是 Next.js 16 中精确控制路由行为的关键机制。通过合理使用这些配置选项,可以优化应用性能,提升用户体验。

**核心要点**:

1. 使用 `dynamic` 控制渲染模式
2. 使用 `revalidate` 实现 ISR
3. 使用 `runtime` 选择运行时环境
4. 注意配置继承和冲突
5. 根据场景选择合适的配置组合

掌握路由段配置,可以让你的 Next.js 应用更加灵活和高效。

### 8.1 配置决策树

**如何选择 dynamic 配置**:

```
需要实时数据?
├─ 是 → dynamic = 'force-dynamic'
└─ 否 → 内容会变化?
    ├─ 是 → 使用 revalidate
    └─ 否 → dynamic = 'force-static'
```

**如何选择 runtime 配置**:

```
需要 Node.js API?
├─ 是 → runtime = 'nodejs'
└─ 否 → 需要低延迟?
    ├─ 是 → runtime = 'edge'
    └─ 否 → runtime = 'nodejs' (默认)
```

### 8.2 性能优化建议

**静态优先**:

尽可能使用静态渲染,这是性能最好的方式:

```tsx
export const dynamic = "force-static";
export const revalidate = false;
```

**合理使用 ISR**:

对于需要定期更新的内容,使用 ISR 而不是完全动态渲染:

```tsx
export const revalidate = 3600; // 1 小时
```

**Edge Runtime 优化**:

对于简单的 API 路由,使用 Edge Runtime 可以显著降低延迟:

```tsx
export const runtime = "edge";
```

### 8.3 最佳实践

**1. 明确配置意图**:

即使使用默认值,也建议显式声明配置,使代码意图更清晰:

```tsx
// ✅ 推荐:明确声明
export const dynamic = "auto";
export const revalidate = false;

// ❌ 不推荐:依赖默认值
// (没有配置声明)
```

**2. 使用类型安全**:

TypeScript 会检查配置值的类型:

```tsx
// ✅ 正确
export const dynamic: "force-static" = "force-static";

// ❌ 错误:类型不匹配
export const dynamic: "force-static" = "invalid-value";
```

**3. 文档化配置原因**:

为重要的配置添加注释,说明选择的原因:

```tsx
// 使用动态渲染,因为需要显示用户特定的数据
export const dynamic = "force-dynamic";

// 每 5 分钟重新验证,平衡实时性和性能
```

### 8.6 迁移指南

**从 Pages Router 迁移**:

Pages Router 的配置方式与 App Router 不同:

| Pages Router                     | App Router                               |
| :------------------------------- | :--------------------------------------- |
| `getStaticProps`                 | `export const dynamic = 'force-static'`  |
| `getServerSideProps`             | `export const dynamic = 'force-dynamic'` |
| `revalidate` in `getStaticProps` | `export const revalidate = number`       |
| `runtime: 'experimental-edge'`   | `export const runtime = 'edge'`          |

**迁移示例**:

```tsx
// Pages Router (pages/blog/[slug].tsx)
export async function getStaticProps({ params }) {
  const post = await getPost(params.slug);
  return {
    props: { post },
    revalidate: 3600,
  };
}

export async function getStaticPaths() {
  const posts = await getAllPosts();
  return {
    paths: posts.map((post) => ({ params: { slug: post.slug } })),
    fallback: true,
  };
}

export default function BlogPost({ post }) {
  return <article>{post.title}</article>;
}
```

```tsx
// App Router (app/blog/[slug]/page.tsx)
export const revalidate = 3600;
export const dynamicParams = true;

export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map((post) => ({ slug: post.slug }));
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await getPost(slug);
  return <article>{post.title}</article>;
}
```

### 8.7 性能对比

不同配置的性能特征:

| 配置            | TTFB | 缓存           | 构建时间 | 适用场景   |
| :-------------- | :--- | :------------- | :------- | :--------- |
| `force-static`  | 最快 | CDN            | 长       | 静态内容   |
| `revalidate`    | 快   | CDN + 定期更新 | 中       | 半静态内容 |
| `force-dynamic` | 慢   | 无             | 短       | 动态内容   |
| `edge` runtime  | 很快 | 边缘缓存       | 短       | 简单 API   |

**TTFB (Time To First Byte)**: 首字节时间

- `force-static`: ~10-50ms (CDN)
- `revalidate`: ~10-100ms (CDN 或重新生成)
- `force-dynamic`: ~100-500ms (服务端渲染)
- `edge`: ~10-50ms (边缘计算)

### 8.8 故障排查

**问题 1: 页面没有按预期静态生成**:

检查是否使用了动态函数:

```tsx
// ❌ 使用了 cookies(),会导致动态渲染
export const dynamic = "force-static";

export default async function Page() {
  const cookieStore = await cookies(); // 这会导致错误
  return <div>Page</div>;
}
```

**解决方案**: 移除动态函数或改用 `dynamic = 'force-dynamic'`

**问题 2: ISR 没有生效**:

检查 `revalidate` 值和缓存配置:

```tsx
// ✅ 正确配置
export const revalidate = 60;

export default async function Page() {
  // 确保 fetch 使用了缓存
  const data = await fetch("https://api.example.com/data", {
    next: { revalidate: 60 },
  });
  return <div>{/* ... */}</div>;
}
```

**问题 3: Edge Runtime 报错**:

检查是否使用了 Node.js API:

```tsx
// ❌ Edge Runtime 不支持
export const runtime = "edge";

export async function GET() {
  const fs = require("fs"); // 错误!
  return Response.json({});
}
```

**解决方案**: 改用 `runtime = 'nodejs'` 或移除 Node.js API
