# 服务端数据缓存策略

## 1. 概述

服务端数据缓存是 Next.js 16 性能优化的核心机制之一。通过合理的缓存策略,可以显著减少数据库查询、API 调用次数,提升应用响应速度,降低服务器负载。Next.js 16 提供了多层次、多维度的缓存控制能力,让开发者能够根据不同的业务场景选择最合适的缓存方案。

### 1.1 核心价值

服务端数据缓存解决了以下核心问题:

- **性能提升**: 减少重复的数据获取操作,加快页面渲染速度
- **成本降低**: 减少对外部 API 的调用次数,降低费用
- **用户体验**: 更快的页面加载速度,更流畅的交互体验
- **服务器负载**: 减轻数据库和后端服务的压力

### 1.2 🆕 Next.js 16 缓存架构

🆕 **Next.js 16 新增**: 引入了更细粒度的缓存控制机制,包括 `cacheLife`、`cacheTag` 等新 API。

Next.js 16 的缓存架构分为四个层次:

| 缓存层次   | 作用范围 | 生命周期       | 控制方式              |
| :--------- | :------- | :------------- | :-------------------- |
| 请求记忆化 | 单次渲染 | 渲染结束后清除 | 自动                  |
| 数据缓存   | 跨请求   | 可配置         | fetch、unstable_cache |
| 全路由缓存 | 整个路由 | 重新部署后清除 | 路由配置              |
| 路由器缓存 | 客户端   | 会话期间       | prefetch              |

### 1.3 适用场景

服务端数据缓存适用于以下场景:

- 静态内容展示(博客、文档、产品目录)
- 定期更新的数据(新闻列表、排行榜)
- 计算密集型数据(统计报表、数据分析)
- 外部 API 数据(第三方服务集成)

## 2. 核心概念与原理

### 2.1 缓存层次详解

#### 请求记忆化(Request Memoization)

请求记忆化是最底层的缓存机制,在单次渲染周期内自动去重相同的请求。

**工作原理**:

```
渲染开始
  ├─ 组件 A 请求 /api/users
  │   └─ 发起网络请求,缓存结果
  ├─ 组件 B 请求 /api/users
  │   └─ 使用缓存结果,不发起网络请求
  └─ 组件 C 请求 /api/users
      └─ 使用缓存结果,不发起网络请求
渲染结束,清除缓存
```

**特点**:

- 自动启用,无需配置
- 只在服务端组件中生效
- 缓存键基于 URL 和请求方法
- 渲染结束后立即清除

**示例**:

```typescript
// app/page.tsx
async function UserProfile({ userId }: { userId: string }) {
  // 第一次请求,会发起网络调用
  const res = await fetch(`https://api.example.com/users/${userId}`);
  const user = await res.json();
  return <div>{user.name}</div>;
}

async function UserPosts({ userId }: { userId: string }) {
  // 相同的请求,会使用缓存,不会发起网络调用
  const res = await fetch(`https://api.example.com/users/${userId}`);
  const user = await res.json();

  const postsRes = await fetch(`https://api.example.com/users/${userId}/posts`);
  const posts = await postsRes.json();

  return <div>{posts.length} 篇文章</div>;
}

export default async function Page() {
  const userId = "123";

  return (
    <div>
      <UserProfile userId={userId} />
      <UserPosts userId={userId} />
    </div>
  );
}
```

在这个例子中,虽然两个组件都请求了用户数据,但实际只会发起一次网络请求。

#### 数据缓存(Data Cache)

数据缓存是持久化的缓存层,可以跨请求、跨部署保存数据。

**工作原理**:

```
请求 1
  └─ fetch('/api/data', { cache: 'force-cache' })
      ├─ 检查缓存: 不存在
      ├─ 发起网络请求
      └─ 存储到缓存

请求 2
  └─ fetch('/api/data', { cache: 'force-cache' })
      ├─ 检查缓存: 存在
      └─ 返回缓存数据(不发起网络请求)

请求 3 (缓存过期后)
  └─ fetch('/api/data', { cache: 'force-cache', next: { revalidate: 60 } })
      ├─ 检查缓存: 已过期
      ├─ 发起网络请求
      └─ 更新缓存
```

**特点**:

- 需要显式配置
- 支持多种缓存策略
- 可以设置过期时间
- 支持标签化管理

**示例**:

```typescript
// app/products/page.tsx
async function Products() {
  // 缓存产品列表,每小时重新验证一次
  const res = await fetch("https://api.example.com/products", {
    cache: "force-cache",
    next: {
      revalidate: 3600, // 3600秒 = 1小时
      tags: ["products"], // 添加标签,方便管理
    },
  });

  const products = await res.json();

  return (
    <div>
      {products.map((product: any) => (
        <div key={product.id}>
          <h2>{product.name}</h2>
          <p>{product.price}</p>
        </div>
      ))}
    </div>
  );
}
```

#### 全路由缓存(Full Route Cache)

全路由缓存会缓存整个路由的渲染结果,包括 HTML 和 RSC Payload。

**工作原理**:

```
构建时
  └─ 静态路由
      ├─ 渲染 HTML
      ├─ 生成 RSC Payload
      └─ 存储到文件系统

运行时
  └─ 请求静态路由
      ├─ 检查缓存: 存在
      └─ 直接返回缓存的 HTML(不重新渲染)
```

**特点**:

- 只对静态路由生效
- 构建时生成,运行时使用
- 重新部署后清除
- 可以通过 revalidate 配置重新验证

**示例**:

```typescript
// app/blog/[slug]/page.tsx

// 生成静态参数
export async function generateStaticParams() {
  const posts = await fetch("https://api.example.com/posts").then((res) =>
    res.json()
  );

  return posts.map((post: any) => ({
    slug: post.slug,
  }));
}

// 配置重新验证时间
export const revalidate = 3600; // 每小时重新验证

export default async function BlogPost({
  params,
}: {
  params: { slug: string };
}) {
  const res = await fetch(`https://api.example.com/posts/${params.slug}`, {
    cache: "force-cache",
  });

  const post = await res.json();

  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />
    </article>
  );
}
```

### 2.2 缓存策略类型

Next.js 16 提供了多种缓存策略,每种策略适用于不同的场景。

#### 策略一: 永久缓存(Static Caching)

永久缓存适用于几乎不会变化的静态内容。

**配置方式**:

```typescript
// 方式一: 使用 fetch
const res = await fetch("https://api.example.com/static-content", {
  cache: "force-cache",
  // 不设置 revalidate,默认永久缓存
});

// 方式二: 使用路由配置
export const dynamic = "force-static";
export const revalidate = false;
```

**适用场景**:

- 博客文章内容
- 产品说明文档
- 法律条款页面
- 历史数据记录

**优点**:

- 最快的响应速度
- 最低的服务器负载
- 最少的 API 调用

**缺点**:

- 内容更新需要重新部署
- 不适合频繁变化的数据

**完整示例**:

```typescript
// app/docs/[slug]/page.tsx

export const dynamic = "force-static";
export const revalidate = false;

export async function generateStaticParams() {
  const docs = await fetch("https://api.example.com/docs").then((res) =>
    res.json()
  );

  return docs.map((doc: any) => ({
    slug: doc.slug,
  }));
}

export default async function DocPage({
  params,
}: {
  params: { slug: string };
}) {
  const res = await fetch(`https://api.example.com/docs/${params.slug}`, {
    cache: "force-cache",
  });

  const doc = await res.json();

  return (
    <article>
      <h1>{doc.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: doc.content }} />
    </article>
  );
}
```

#### 策略二: 定时重新验证(Time-based Revalidation)

定时重新验证适用于定期更新的内容。

**配置方式**:

```typescript
// 方式一: 使用 fetch
const res = await fetch("https://api.example.com/news", {
  next: { revalidate: 3600 }, // 每小时重新验证
});

// 方式二: 使用路由配置
export const revalidate = 3600; // 每小时重新验证
```

**适用场景**:

- 新闻列表
- 产品目录
- 排行榜
- 天气信息

**工作流程**:

```
时间 0:00
  └─ 用户访问 → 缓存不存在 → 获取数据 → 存储缓存

时间 0:30
  └─ 用户访问 → 缓存存在且未过期 → 返回缓存数据

时间 1:00
  └─ 用户访问 → 缓存过期 → 返回旧缓存 → 后台重新验证

时间 1:01
  └─ 用户访问 → 缓存已更新 → 返回新数据
```

**完整示例**:

```typescript
// app/news/page.tsx

export const revalidate = 1800; // 30分钟

export default async function NewsPage() {
  const res = await fetch("https://api.example.com/news", {
    next: {
      revalidate: 1800,
      tags: ["news"],
    },
  });

  const news = await res.json();

  return (
    <div>
      <h1>最新新闻</h1>
      {news.map((item: any) => (
        <article key={item.id}>
          <h2>{item.title}</h2>
          <p>{item.summary}</p>
          <time>{new Date(item.publishedAt).toLocaleString()}</time>
        </article>
      ))}
    </div>
  );
}
```

#### 策略三: 按需重新验证(On-demand Revalidation)

按需重新验证允许在特定事件发生时手动清除缓存。

**配置方式**:

```typescript
// 数据获取时添加标签
const res = await fetch("https://api.example.com/posts", {
  next: {
    tags: ["posts"],
    revalidate: 3600,
  },
});

// 在需要时重新验证
import { revalidateTag, revalidatePath } from "next/cache";

// 按标签重新验证
revalidateTag("posts");

// 按路径重新验证
revalidatePath("/blog");
```

**适用场景**:

- 内容管理系统
- 电商产品更新
- 用户生成内容
- 实时协作应用

**完整示例**:

```typescript
// app/blog/page.tsx
export default async function BlogPage() {
  const res = await fetch("https://api.example.com/posts", {
    next: {
      tags: ["posts"],
      revalidate: 3600,
    },
  });

  const posts = await res.json();

  return (
    <div>
      {posts.map((post: any) => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.excerpt}</p>
        </article>
      ))}
    </div>
  );
}

// app/api/revalidate/route.ts
import { revalidateTag } from "next/cache";

export async function POST(request: Request) {
  const { tag } = await request.json();

  if (tag === "posts") {
    revalidateTag("posts");
    return Response.json({ revalidated: true, now: Date.now() });
  }

  return Response.json({ revalidated: false }, { status: 400 });
}

// app/actions.ts
("use server");

import { revalidateTag } from "next/cache";

export async function createPost(formData: FormData) {
  const res = await fetch("https://api.example.com/posts", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      title: formData.get("title"),
      content: formData.get("content"),
    }),
  });

  const post = await res.json();

  // 创建文章后,重新验证文章列表缓存
  revalidateTag("posts");

  return post;
}
```

#### 策略四: 不缓存(No Caching)

不缓存适用于实时性要求高的数据。

**配置方式**:


**适用场景**:

- 用户个人信息
- 实时统计数据
- 购物车内容
- 会话状态

**完整示例**:

### 2.3 缓存策略对比

不同缓存策略的特性对比:

| 策略         | 响应速度 | 数据新鲜度 | 服务器负载 | API 调用次数 | 适用场景     |
| :----------- | :------- | :--------- | :--------- | :----------- | :----------- |
| 永久缓存     | 最快     | 最低       | 最低       | 最少         | 静态内容     |
| 定时重新验证 | 快       | 中等       | 低         | 少           | 定期更新内容 |
| 按需重新验证 | 快       | 高         | 中等       | 中等         | 内容管理系统 |
| 不缓存       | 慢       | 最高       | 高         | 最多         | 实时数据     |

### 2.4 缓存键的生成规则

Next.js 根据以下因素生成缓存键:

1. **URL**: 完整的请求 URL
2. **请求方法**: GET、POST 等
3. **请求头**: 某些特定的请求头
4. **请求体**: POST 请求的 body

**示例**:

```typescript
// 这两个请求会使用不同的缓存
const res1 = await fetch("https://api.example.com/data?lang=en");
const res2 = await fetch("https://api.example.com/data?lang=zh");

// 这两个请求会使用不同的缓存
const res3 = await fetch("https://api.example.com/data", {
  headers: { "Accept-Language": "en" },
});
const res4 = await fetch("https://api.example.com/data", {
  headers: { "Accept-Language": "zh" },
});
```

```typescript
// app/api/webhook/route.ts
import { revalidateTag } from "next/cache";

export async function POST(request: Request) {
  const { event, data } = await request.json();

  // 验证 webhook 签名
  const signature = request.headers.get("X-Webhook-Signature");
  if (!verifySignature(signature, data)) {
    return Response.json({ error: "Invalid signature" }, { status: 401 });
  }

  switch (event) {
    case "news.published":
      // 新闻发布时,重新验证首页和分类页
      revalidateTag("headlines");
      revalidateTag("categories");
      break;

    case "news.updated":
      // 新闻更新时,重新验证该新闻页面
      revalidateTag(`news-${data.id}`);
      break;

    case "news.deleted":
      // 新闻删除时,重新验证相关页面
      revalidateTag("headlines");
      revalidateTag(`news-${data.id}`);
      break;
  }

  return Response.json({ revalidated: true });
}

function verifySignature(signature: string | null, data: any): boolean {
  // 实现签名验证逻辑
  return true;
}
```

## 7. 常见问题

### 7.1 基础问题

#### 问题一: 如何选择合适的缓存策略?

**问题**: 不知道应该使用哪种缓存策略?

**简短回答**: 根据数据的更新频率和重要性选择。

**详细解释**:

选择缓存策略的决策树:

```
数据是否包含用户特定信息?
├─ 是 → 不缓存 (cache: 'no-store')
└─ 否 → 数据多久更新一次?
    ├─ 几乎不变 → 永久缓存 (cache: 'force-cache')
    ├─ 定期更新 → 定时重新验证 (revalidate: 时间)
    ├─ 按需更新 → 按需重新验证 (tags + revalidateTag)
    └─ 实时变化 → 不缓存 (cache: 'no-store')
```

**代码示例**:

```typescript
// 用户特定数据: 不缓存
const userRes = await fetch("/api/user/me", {
  cache: "no-store",
});

// 静态内容: 永久缓存
const aboutRes = await fetch("/api/about", {
  cache: "force-cache",
});

// 定期更新: 定时重新验证
const newsRes = await fetch("/api/news", {
  next: { revalidate: 300 },
});

// 按需更新: 标签化缓存
const postsRes = await fetch("/api/posts", {
  next: {
    tags: ["posts"],
    revalidate: 3600,
  },
});
```

#### 问题二: revalidate 和 cache 的区别是什么?

**问题**: `revalidate` 和 `cache` 参数有什么区别?

**简短回答**: `cache` 控制是否缓存,`revalidate` 控制缓存多久后重新验证。

**详细解释**:

- `cache: 'no-store'`: 完全不缓存,每次都请求
- `cache: 'force-cache'`: 缓存数据
- `revalidate`: 在 `force-cache` 的基础上,设置缓存过期时间

**代码示例**:

```typescript
// 不缓存
const res1 = await fetch("/api/data", {
  cache: "no-store",
  // revalidate 会被忽略
});

// 永久缓存
const res2 = await fetch("/api/data", {
  cache: "force-cache",
  // 不设置 revalidate,永不过期
});

// 定时重新验证
const res3 = await fetch("/api/data", {
  cache: "force-cache",
  next: { revalidate: 60 }, // 60秒后重新验证
});
```

#### 问题三: 如何在 Server Action 中使用缓存?

**问题**: Server Action 中的数据获取如何缓存?

**简短回答**: Server Action 中的 fetch 默认不缓存,需要显式指定。

**详细解释**:

Server Action 通常用于变更操作,所以默认不缓存。如果需要在 Server Action 中获取数据并缓存,必须显式指定缓存策略。

**代码示例**:

```typescript
// app/actions.ts
"use server";

import { revalidateTag } from "next/cache";

// 获取数据(可以缓存)
export async function getPosts() {
  const res = await fetch("https://api.example.com/posts", {
    cache: "force-cache",
    next: {
      tags: ["posts"],
      revalidate: 3600,
    },
  });

  return await res.json();
}

// 创建数据(不缓存)
export async function createPost(data: any) {
  const res = await fetch("https://api.example.com/posts", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(data),
    cache: "no-store",
  });

  const post = await res.json();

  // 重新验证文章列表缓存
  revalidateTag("posts");

  return post;
}
```

### 7.2 进阶问题

#### 问题四: 如何实现部分缓存?

**问题**: 页面中部分数据需要缓存,部分数据需要实时获取,如何实现?

**简短回答**: 对不同的数据使用不同的缓存策略。

**详细解释**:

可以在同一个页面中对不同的数据使用不同的缓存策略,Next.js 会自动处理。

**代码示例**:

```typescript
// app/dashboard/page.tsx

export default async function Dashboard() {
  // 静态配置: 永久缓存
  const configRes = await fetch("https://api.example.com/config", {
    cache: "force-cache",
  });

  // 统计数据: 缓存5分钟
  const statsRes = await fetch("https://api.example.com/stats", {
    next: { revalidate: 300 },
  });

  // 用户信息: 不缓存
  const userRes = await fetch("https://api.example.com/user/me", {
    cache: "no-store",
  });

  const [config, stats, user] = await Promise.all([
    configRes.json(),
    statsRes.json(),
    userRes.json(),
  ]);

  return (
    <div>
      <h1>欢迎, {user.name}</h1>
      <div>今日访问: {stats.todayVisits}</div>
      <div>系统版本: {config.version}</div>
    </div>
  );
}
```

#### 问题五: 如何处理缓存穿透?

**问题**: 大量请求同时访问未缓存的数据,如何避免缓存穿透?

**简短回答**: Next.js 的请求去重机制会自动处理。

**详细解释**:

Next.js 在渲染周期内会自动对相同的请求进行去重,即使缓存不存在,也只会发起一次实际的网络请求。

**代码示例**:

```typescript
// app/page.tsx

async function Component1() {
  // 第一次请求
  const res = await fetch("https://api.example.com/data");
  return <div>{await res.text()}</div>;
}

async function Component2() {
  // 相同的请求会被去重,不会发起新的网络请求
  const res = await fetch("https://api.example.com/data");
  return <div>{await res.text()}</div>;
}

export default function Page() {
  return (
    <div>
      <Component1 />
      <Component2 />
    </div>
  );
}
```

## 8. 总结

### 8.1 核心要点回顾

**缓存层次**:

1. **请求记忆化**: 单次渲染内自动去重
2. **数据缓存**: 跨请求持久化缓存
3. **全路由缓存**: 整个路由的渲染结果缓存
4. **路由器缓存**: 客户端缓存

**缓存策略**:

| 策略         | 配置                   | 适用场景     |
| :----------- | :--------------------- | :----------- |
| 永久缓存     | `cache: 'force-cache'` | 静态内容     |
| 定时重新验证 | `revalidate: 时间`     | 定期更新内容 |
| 按需重新验证 | `tags + revalidateTag` | 内容管理系统 |
| 不缓存       | `cache: 'no-store'`    | 实时数据     |

### 8.2 关键收获

1. **理解缓存层次**: 掌握四层缓存机制的作用和生命周期
2. **选择合适策略**: 根据数据特性选择最优缓存策略
3. **标签化管理**: 使用标签实现精细化的缓存控制
4. **注意安全**: 避免缓存敏感数据和用户特定数据
5. **保持一致性**: 确保相关数据的缓存一起更新

### 8.3 最佳实践

1. **显式配置**: 不要依赖默认行为,明确指定缓存策略
2. **分层缓存**: 对不同类型的数据使用不同的缓存时间
3. **标签管理**: 为缓存添加有意义的标签,方便管理
4. **监控缓存**: 在开发环境中监控缓存命中率
5. **及时更新**: 数据变更时及时重新验证相关缓存

### 8.4 下一步学习

- **cacheLife 函数**: 更灵活的缓存生命周期控制
