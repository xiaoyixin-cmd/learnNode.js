# 首次内容绘制(FCP)优化

## 核心概念

首次内容绘制(First Contentful Paint, FCP)是衡量页面加载性能的关键指标之一,它记录了页面从开始加载到首次渲染任何文本、图像、非白色 canvas 或 SVG 的时间。FCP 是用户感知页面加载速度的重要指标,直接影响用户体验。

### FCP 时间标准

| 评分     | FCP 时间   | 用户体验                    |
| -------- | ---------- | --------------------------- |
| 优秀     | 0-1.8 秒   | 快速加载,用户几乎无感知延迟 |
| 需要改进 | 1.8-3.0 秒 | 可接受,但有明显等待感       |
| 差       | >3.0 秒    | 加载缓慢,用户可能离开       |

### 影响 FCP 的主要因素

1. **服务器响应时间(TTFB)**: 服务器处理请求并返回首字节的时间
2. **资源加载时间**: CSS、JavaScript、字体等关键资源的加载时间
3. **渲染阻塞资源**: 阻止页面渲染的 CSS 和 JavaScript
4. **网络延迟**: 网络连接速度和延迟
5. **资源大小**: 传输的数据量

```typescript
  useEffect(() => {
    const link = document.createElement("link");
    link.rel = "stylesheet";
    link.href = href;
    link.media = "print";

      link.media = "all";
```

```typescript
// public/sw.js
const CACHE_NAME = "v1";
const urlsToCache = [
  "/",
  "/styles/main.css",
  "/scripts/main.js",
  "/images/logo.png",
];

self.addEventListener("install", (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.addAll(urlsToCache);
    })
  );
});

self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((response) => {
      // 缓存命中则返回缓存,否则发起网络请求
      return response || fetch(event.request);
    })
  );
});
```

## 适用场景

### 场景对比

| 优化方案       | 适用场景                 | 优先级 | 实施难度 |
| -------------- | ------------------------ | ------ | -------- |
| 静态生成       | 内容不常变化的页面       | 高     | 低       |
| 边缘函数       | 需要动态内容但要求低延迟 | 高     | 中       |
| 代码分割       | 大型应用,组件较多        | 高     | 低       |
| 资源预加载     | 关键资源明确的页面       | 中     | 低       |
| 流式渲染       | 数据获取较慢的页面       | 中     | 中       |
| Service Worker | 需要离线支持的应用       | 低     | 高       |
| CDN 加速       | 全球用户访问的应用       | 高     | 中       |
| 关键 CSS 内联  | 首屏样式较多的页面       | 中     | 中       |

### 推荐组合方案

1. **电商网站**: 静态生成 + CDN + 代码分割 + 图片优化
2. **博客网站**: 静态生成 + 流式渲染 + 字体优化
3. **SaaS 应用**: 边缘函数 + 代码分割 + 资源预加载
4. **新闻网站**: 流式渲染 + CDN + 关键 CSS 内联

## 注意事项

### 1. 避免过度优化

不要为了优化 FCP 而牺牲其他重要指标。例如,过度的代码分割可能导致更多的网络请求,反而影响总体性能。

```typescript

// ✅ 正确:合理分割

```

```typescript
}
```

## 常见问题

### 1. FCP 和 LCP 有什么区别?

FCP(First Contentful Paint)是首次内容绘制,记录页面首次渲染任何内容的时间。LCP(Largest Contentful Paint)是最大内容绘制,记录视口内最大可见内容元素的渲染时间。

FCP 关注的是"有内容显示",而 LCP 关注的是"主要内容显示"。通常 LCP 时间会晚于 FCP。

### 2. 为什么我的 FCP 时间在本地很快,但生产环境很慢?

可能的原因包括:

- 生产环境的服务器响应时间(TTFB)较慢
- 网络延迟较高
- 未启用 CDN 或 CDN 配置不当
- 生产环境的资源未压缩或优化
- 第三方脚本在生产环境加载较慢

建议使用真实设备和网络条件测试,或使用 Chrome DevTools 的网络节流功能模拟慢速网络。

### 3. 内联 CSS 会增加 HTML 大小,这样做值得吗?

对于关键 CSS,内联是值得的。虽然会增加 HTML 大小,但可以避免额外的网络请求,加快首屏渲染。关键是只内联首屏必需的 CSS,其他 CSS 异步加载。

```typescript
// 推荐做法
<head>
  {/* 内联关键CSS(通常<14KB) */}
  <style>{criticalCSS}</style>

  {/* 异步加载其他CSS */}
  <link
    rel="preload"
    href="/styles/main.css"
    as="style"
    onload="this.onload=null;this.rel='stylesheet'"
  />
</head>
```

### 4. 使用动态导入会影响 SEO 吗?

使用`dynamic()`进行代码分割不会影响 SEO,因为 Next.js 默认会在服务端渲染这些组件。但如果设置了`ssr: false`,则该组件不会被搜索引擎爬虫看到。

```typescript
// ✅ 不影响SEO
const Component = dynamic(() => import("./Component"));

// ❌ 可能影响SEO
const Component = dynamic(() => import("./Component"), { ssr: false });
```

### 5. 字体加载会阻塞 FCP 吗?

是的,如果字体文件较大或加载较慢,可能会延迟 FCP。使用`font-display: swap`可以让浏览器先使用系统字体渲染文本,字体加载完成后再替换,避免阻塞 FCP。

```typescript
const inter = Inter({
  subsets: ["latin"],
  display: "swap", // 关键配置
});
```

### 6. 预加载太多资源会不会适得其反?

是的,过度预加载会占用带宽,延迟其他关键资源的加载。只预加载真正关键的资源,通常不超过 3-5 个。

```typescript
// ❌ 过度预加载
<link rel="preload" href="/font1.woff2" as="font" />
<link rel="preload" href="/font2.woff2" as="font" />
<link rel="preload" href="/image1.jpg" as="image" />
<link rel="preload" href="/image2.jpg" as="image" />
<link rel="preload" href="/script1.js" as="script" />

// ✅ 合理预加载
<link rel="preload" href="/font-main.woff2" as="font" />
<link rel="preload" href="/hero-image.webp" as="image" />
```

### 7. Service Worker 会影响首次访问的 FCP 吗?

首次访问时,Service Worker 需要下载和安装,可能会略微影响 FCP。但后续访问可以从缓存加载资源,显著改善 FCP。建议在 FCP 完成后再注册 Service Worker。

```typescript
useEffect(() => {
  // 等待页面加载完成
  if (document.readyState === "complete") {
    registerServiceWorker();
  } else {
    window.addEventListener("load", registerServiceWorker);
  }
}, []);
```

### 8. 如何测试 FCP 优化效果?

使用以下工具测试:

- Chrome DevTools 的 Lighthouse
- WebPageTest
- Google PageSpeed Insights
- 真实用户监控(RUM)工具

建议在多种设备和网络条件下测试,获取更全面的数据。

### 9. 图片懒加载会影响 FCP 吗?

首屏图片不应该懒加载,否则会延迟 FCP。只对首屏以下的图片使用懒加载。

```typescript
// ✅ 首屏图片
<Image src="/hero.jpg" alt="Hero" priority />

// ✅ 首屏以下图片
<Image src="/product.jpg" alt="Product" loading="lazy" />
```

### 10. 使用 SSR 还是 SSG 对 FCP 更好?

对于静态内容,SSG(静态生成)通常能提供更好的 FCP,因为页面已经预先生成,无需服务器处理时间。对于动态内容,可以使用 ISR(增量静态再生)或边缘 SSR 来平衡性能和动态性。

| 渲染方式   | FCP 表现 | 适用场景            |
| ---------- | -------- | ------------------- |
| SSG        | 最快     | 静态内容            |
| ISR        | 快       | 半静态内容          |
| 边缘 SSR   | 较快     | 动态内容,低延迟要求 |
| 客户端渲染 | 慢       | 高度交互的应用      |

### 11. 如何优化移动端的 FCP?

移动端优化重点:

- 使用响应式图片,为移动端提供更小的图片
- 减少 JavaScript 包大小
- 使用更激进的代码分割
- 优先加载移动端视口内的内容
- 使用更快的字体加载策略

```typescript
// 移动端优化示例
export function MobileOptimizedPage() {
  const isMobile = useIsMobile();

  return (
    <>
      <Image
        src={isMobile ? "/hero-mobile.webp" : "/hero-desktop.webp"}
        width={isMobile ? 375 : 1920}
        height={isMobile ? 667 : 1080}
        priority
      />

      {!isMobile && <DesktopOnlyFeatures />}
    </>
  );
}
```

### 12. 第三方脚本如何不影响 FCP?

使用 Next.js 的 Script 组件,设置合适的加载策略:

```typescript
// 延迟加载分析脚本
<Script src="analytics.js" strategy="lazyOnload" />

// 在交互后加载
<Script src="chat.js" strategy="afterInteractive" />

// 关键脚本(谨慎使用)
<Script src="critical.js" strategy="beforeInteractive" />
```

### 13. 如何监控生产环境的 FCP?

实现真实用户监控(RUM):

```typescript
// lib/rum.ts
export function initRUM() {
  if (typeof window === "undefined") return;

  const observer = new PerformanceObserver((list) => {
    for (const entry of list.getEntries()) {
      if (entry.name === "first-contentful-paint") {
        // 发送到监控服务
        sendToAnalytics({
          metric: "fcp",
          value: entry.startTime,
          url: window.location.href,
          userAgent: navigator.userAgent,
        });
      }
    }
  });

  observer.observe({ entryTypes: ["paint"] });
}
```

### 14. 预渲染和 SSR 有什么区别?

预渲染是在构建时生成 HTML,SSR 是在请求时生成 HTML。预渲染的 FCP 通常更快,因为不需要服务器处理时间。

### 15. 如何平衡 FCP 和其他性能指标?

不要只关注 FCP,要综合考虑所有 Core Web Vitals 指标(FCP、LCP、CLS、INP)。有时为了优化 FCP 而做的改动可能会影响其他指标。

建议的优先级:

1. 确保 LCP < 2.5 秒
2. 确保 CLS < 0.1
3. 确保 INP < 200ms
4. 优化 FCP < 1.8 秒

## 总结

FCP 优化是提升用户体验的关键环节。通过合理使用静态生成、边缘函数、代码分割、资源预加载、流式渲染等技术,可以显著改善 FCP 指标。

关键要点:

1. 优先使用静态生成和边缘函数减少 TTFB
2. 消除渲染阻塞资源,使用异步加载
3. 合理使用代码分割,避免过度分割
4. 预加载关键资源,但不要过度预加载
5. 使用流式渲染加速首屏显示
6. 优化字体加载,使用 font-display: swap
7. 延迟加载第三方脚本
8. 使用 CDN 加速全球访问
9. 内联关键 CSS,异步加载其他样式
10. 持续监控和测试,使用真实设备和网络条件

记住,FCP 优化是一个持续的过程,需要根据实际数据不断调整和改进。不要过度优化,要在性能和开发效率之间找到平衡。
