# 内容安全策略(CSP)

内容安全策略(CSP)是一种强大的安全机制,可以有效防止 XSS、数据注入等攻击。本文介绍如何在 Next.js 应用中配置和使用 CSP。

## 核心概念

### CSP

通过 HTTP 头部指定允许加载的资源来源,限制恶意代码执行。

### 指令

CSP 策略由多个指令组成,每个指令控制特定类型资源的加载。

### Nonce

一次性随机值,用于允许特定的内联脚本和样式。

### 报告模式

CSP 可以在报告模式下运行,只记录违规而不阻止。

## CSP 指令详解

### 常用指令对比

| 指令        | 作用        | 示例                               |
| ----------- | ----------- | ---------------------------------- |
| default-src | 默认策略    | default-src 'self'                 |
| script-src  | 脚本来源    | script-src 'self' https://cdn.com  |
| style-src   | 样式来源    | style-src 'self' 'unsafe-inline'   |
| img-src     | 图片来源    | img-src 'self' data: https:        |
| font-src    | 字体来源    | font-src 'self' https://fonts.com  |
| connect-src | 连接来源    | connect-src 'self' https://api.com |
| frame-src   | iframe 来源 | frame-src 'none'                   |
| object-src  | 对象来源    | object-src 'none'                  |
| base-uri    | base 标签   | base-uri 'self'                    |
| form-action | 表单提交    | form-action 'self'                 |

### 指令值说明

| 值              | 含义           | 安全性 |
| --------------- | -------------- | ------ |
| 'none'          | 禁止所有来源   | 最高   |
| 'self'          | 只允许同源     | 高     |
| 'unsafe-inline' | 允许内联代码   | 低     |
| 'unsafe-eval'   | 允许 eval      | 低     |
| 'nonce-xxx'     | 允许特定 nonce | 高     |
| 'sha256-xxx'    | 允许特定 hash  | 高     |
| https:          | 允许所有 HTTPS | 中     |
| data:           | 允许 data URI  | 中     |
| blob:           | 允许 blob URI  | 中     |

## 实战场景

### 场景 1: 基础 CSP 配置

**适用场景**: 新项目初始化,需要基础的安全防护

**配置方式**:

```typescript
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const response = NextResponse.next();

  // 设置基础CSP策略
  const csp = [
    "default-src 'self'",
    "script-src 'self' 'unsafe-inline' 'unsafe-eval'",
    "style-src 'self' 'unsafe-inline'",
    "img-src 'self' data: https:",
    "font-src 'self'",
    "connect-src 'self'",
    "frame-ancestors 'none'",
    "base-uri 'self'",
    "form-action 'self'",
  ].join("; ");

  response.headers.set("Content-Security-Policy", csp);

  return response;
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};
```

**配置说明**:

- `default-src 'self'`: 默认只允许同源资源
- `script-src`: 允许同源脚本和内联脚本(开发阶段)
- `img-src`: 允许同源、data URI 和所有 HTTPS 图片
- `frame-ancestors 'none'`: 防止被其他网站 iframe

**使用效果**:

- 阻止 XSS 攻击
- 防止数据注入
- 限制资源加载来源
- 提升应用安全性

### 场景 2: Nonce-based CSP

```typescript
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";
import { randomBytes } from "crypto";

export function middleware(request: NextRequest) {
  const nonce = randomBytes(16).toString("base64");
  const response = NextResponse.next();

  // 使用nonce允许特定内联脚本
  const csp = [
    "default-src 'self'",
    `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'`,
    `style-src 'self' 'nonce-${nonce}'`,
    "img-src 'self' data: https:",
    "font-src 'self'",
    "connect-src 'self'",
    "object-src 'none'",
    "base-uri 'self'",
    "form-action 'self'",
    "frame-ancestors 'none'",
    "upgrade-insecure-requests",
  ].join("; ");

  response.headers.set("Content-Security-Policy", csp);
  response.headers.set("X-Nonce", nonce);

  return response;
}

// app/layout.tsx
import { headers } from "next/headers";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const nonce = headers().get("X-Nonce") || "";

  return (
    <html>
      <head>
        {/* 使用nonce允许内联脚本 */}
        <script
          nonce={nonce}
          dangerouslySetInnerHTML={{
            __html: `
            console.log('This script is allowed by nonce');
          `,
          }}
        />
      </head>
      <body>{children}</body>
    </html>
  );
}
```

**适用场景**: 需要严格控制内联脚本,提升安全性

**配置说明**:

- 使用`randomBytes`生成随机 nonce
- 通过`X-Nonce`头部传递 nonce 到页面
- 在脚本标签上添加`nonce`属性
- `strict-dynamic`允许 nonce 脚本动态加载其他脚本

**使用效果**:

- 完全禁止未授权的内联脚本
- 每次请求 nonce 都不同,更安全
- 支持动态脚本加载
- 防止 XSS 攻击

### 场景 3: 环境特定 CSP

```typescript
// lib/csp.ts
export function getCSP(env: string) {
  const isDev = env === "development";

  const csp = {
    "default-src": ["'self'"],
    "script-src": [
      "'self'",
      ...(isDev ? ["'unsafe-eval'", "'unsafe-inline'"] : []),
    ],
    "style-src": ["'self'", "'unsafe-inline'", "https://fonts.googleapis.com"],
    "img-src": ["'self'", "data:", "https:"],
    "font-src": ["'self'", "https://fonts.gstatic.com"],
    "connect-src": [
      "'self'",
      ...(isDev ? ["ws:", "wss:"] : []),
      "https://api.example.com",
    ],
    "frame-ancestors": ["'none'"],
    "base-uri": ["'self'"],
    "form-action": ["'self'"],
    "object-src": ["'none'"],
  };

  return Object.entries(csp)
    .map(([key, values]) => `${key} ${values.join(" ")}`)
    .join("; ");
}

// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";
import { getCSP } from "@/lib/csp";

export function middleware(request: NextRequest) {
  const response = NextResponse.next();

  const csp = getCSP(process.env.NODE_ENV || "production");
  response.headers.set("Content-Security-Policy", csp);

  return response;
}
```

**适用场景**: 不同环境需要不同的 CSP 策略

**配置说明**:

- 开发环境允许`unsafe-eval`和`unsafe-inline`
- 生产环境使用严格策略
- 根据环境动态生成 CSP 字符串
- 支持环境特定的第三方域名

**使用效果**:

- 开发环境更灵活,方便调试
- 生产环境更安全,严格限制
- 统一管理 CSP 配置
- 易于维护和扩展

### 场景 4: CSP 报告

```typescript
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const response = NextResponse.next();

  const csp = [
    "default-src 'self'",
    "script-src 'self'",
    "style-src 'self' 'unsafe-inline'",
    "img-src 'self' data: https:",
    "font-src 'self'",
    "connect-src 'self'",
    "report-uri /api/csp-report",
    "report-to csp-endpoint",
  ].join("; ");

  // 设置CSP策略
  response.headers.set("Content-Security-Policy", csp);

  // 设置报告端点
  response.headers.set(
    "Report-To",
    JSON.stringify({
      group: "csp-endpoint",
      max_age: 10886400,
      endpoints: [{ url: "/api/csp-report" }],
    })
  );

  return response;
}

// app/api/csp-report/route.ts
import { NextRequest, NextResponse } from "next/server";

export async function POST(request: NextRequest) {
  try {
    const report = await request.json();

    // 记录CSP违规
    console.error("CSP Violation:", JSON.stringify(report, null, 2));

    // 可以发送到日志服务
    // await sendToLoggingService(report)

    return NextResponse.json({ received: true });
  } catch (error) {
    return NextResponse.json({ error: "处理报告失败" }, { status: 500 });
  }
}
```

**适用场景**: 需要监控 CSP 违规情况

**配置说明**:

- 使用`report-uri`指定报告端点
- 使用`Report-To`头部配置报告组
- 创建 API 路由接收违规报告
- 记录或发送到日志服务

**使用效果**:

- 实时监控 CSP 违规
- 发现潜在的安全问题
- 帮助调试 CSP 配置
- 收集违规数据分析

## 常见问题

### 1. CSP 会影响性能吗?

不会。CSP 是浏览器级别的安全策略,对性能影响微乎其微。

| 方面     | 影响               |
| -------- | ------------------ |
| 页面加载 | 几乎无影响         |
| 运行时   | 阻止不安全脚本执行 |
| 内存     | 无影响             |

### 2. 如何调试 CSP 违规?

使用浏览器开发者工具:

```typescript
// middleware.ts
export function middleware(request: NextRequest) {
  const response = NextResponse.next();

  // 开发环境:只报告不阻止
  if (process.env.NODE_ENV === "development") {
    response.headers.set("Content-Security-Policy-Report-Only", cspPolicy);
  } else {
    response.headers.set("Content-Security-Policy", cspPolicy);
  }

  return response;
}
```

### 3. CSP 和 nonce 的关系?

nonce 是 CSP 的一部分,用于允许特定的内联脚本:

```typescript
// 生成 nonce
const nonce = crypto.randomUUID();

// CSP 策略
const csp = `script-src 'self' 'nonce-${nonce}'`;

// 使用 nonce
<script nonce={nonce}>console.log('允许执行');</script>;
```

### 4. 如何处理第三方脚本?

将第三方域名添加到白名单:

```typescript
const csp = [
  "default-src 'self'",
  "script-src 'self' https://www.googletagmanager.com",
  "style-src 'self' https://fonts.googleapis.com",
].join("; ");
```

### 5. CSP 会阻止 eval() 吗?

是的。默认情况下 CSP 会阻止 `eval()` 和 `new Function()`:

```typescript
// ❌ 被 CSP 阻止
eval('console.log("blocked")');

// ✅ 安全替代方案
const safeFunction = () => console.log("allowed");
safeFunction();
```

## 注意事项

### 1. 开发环境配置

开发环境使用 Report-Only 模式:

```typescript
// middleware.ts
export function middleware(request: NextRequest) {
  const response = NextResponse.next();
  const headerName =
    process.env.NODE_ENV === "development"
      ? "Content-Security-Policy-Report-Only"
      : "Content-Security-Policy";

  response.headers.set(headerName, cspPolicy);
  return response;
}
```

### 2. nonce 必须每次请求都不同

```typescript
// ❌ 错误:固定 nonce
const nonce = "fixed-nonce-123";

// ✅ 正确:每次请求生成新 nonce
const nonce = crypto.randomUUID();
```

```typescript
// ❌ 不安全
const csp = "script-src 'self' 'unsafe-inline'";

const csp = `script-src 'self' 'nonce-${nonce}'`;
```

## 常见问题

### 1. CSP 会影响性能吗?

**问题**: 担心 CSP 配置影响页面性能

**解决方案**: CSP 对性能影响很小,主要是浏览器解析头部的开销

```typescript
// 性能监控
export function middleware(request: NextRequest) {
  const start = Date.now();
  const response = NextResponse.next();

  const csp = generateCSP();
  response.headers.set("Content-Security-Policy", csp);

  const duration = Date.now() - start;
  console.log(`CSP处理时间: ${duration}ms`);

  return response;
}
```

**性能对比**:

| 场景     | 无 CSP | 有 CSP | 差异   |
| -------- | ------ | ------ | ------ |
| 首次加载 | 1.2s   | 1.21s  | +10ms  |
| 后续加载 | 0.8s   | 0.81s  | +10ms  |
| 内存占用 | 50MB   | 50MB   | 无影响 |

### 3. nonce 和 hash 的区别?

**问题**: 不清楚何时使用 nonce,何时使用 hash

**对比表格**:

| 特性       | Nonce            | Hash               |
| ---------- | ---------------- | ------------------ |
| 生成方式   | 每次请求随机生成 | 基于脚本内容计算   |
| 适用场景   | 动态内容         | 静态内容           |
| 安全性     | 高(每次不同)     | 中(内容不变则相同) |
| 实现复杂度 | 高               | 低                 |
| 性能       | 需要服务端生成   | 可预先计算         |

**Hash 示例**:

```typescript
// 使用hash允许特定脚本
const csp = ["script-src 'self' 'sha256-abc123...'"].join("; ");

// 计算脚本hash
import crypto from "crypto";

const scriptContent = "console.log('Hello')";
const hash = crypto.createHash("sha256").update(scriptContent).digest("base64");

console.log(`sha256-${hash}`);
```

### 5. CSP 会阻止 eval 吗?

**问题**: 代码中使用了 eval,被 CSP 阻止

### 9. 如何实现 SRI(子资源完整性)?

**问题**: 需要验证 CDN 资源的完整性

**解决方案**: 使用 SRI hash

```typescript
// 生成SRI hash
import crypto from "crypto";
import https from "https";

async function generateSRI(url: string) {
  return new Promise((resolve) => {
    https.get(url, (res) => {
      const hash = crypto.createHash("sha384");
      res.on("data", (chunk) => hash.update(chunk));
      res.on("end", () => {
        const integrity = `sha384-${hash.digest("base64")}`;
        resolve(integrity);
      });
    });
  });
}

// 使用SRI
<script
  src="https://cdn.example.com/script.js"
  integrity="sha384-abc123..."
  crossOrigin="anonymous"
/>;
```

### 2. CSP 配置管理

```typescript
// csp-config.ts
export const CSP_DIRECTIVES = {
  defaultSrc: ["'self'"],
  scriptSrc: ["'self'", "https://cdn.example.com"],
  styleSrc: ["'self'", "'unsafe-inline'"],
  imgSrc: ["'self'", "data:", "https:"],
  fontSrc: ["'self'", "https://fonts.gstatic.com"],
  connectSrc: ["'self'", "https://api.example.com"],
  frameSrc: ["'none'"],
  objectSrc: ["'none'"],
  baseUri: ["'self'"],
  formAction: ["'self'"],
  frameAncestors: ["'none'"],
  upgradeInsecureRequests: [],
};

export function buildCSP(directives: typeof CSP_DIRECTIVES) {
  return Object.entries(directives)
    .map(([key, values]) => {
      const directive = key.replace(/([A-Z])/g, "-$1").toLowerCase();
      return values.length > 0 ? `${directive} ${values.join(" ")}` : directive;
    })
    .join("; ");
}
```

### 3. 环境感知 CSP

```typescript
export function getCSP() {
  const env = process.env.NODE_ENV;

  switch (env) {
    case "development":
      return buildCSP({
        ...CSP_DIRECTIVES,
        scriptSrc: [...CSP_DIRECTIVES.scriptSrc, "'unsafe-eval'"],
      });
    case "staging":
      return buildCSP(CSP_DIRECTIVES) + "-Report-Only";
    case "production":
      return buildCSP(CSP_DIRECTIVES);
    default:
      return buildCSP(CSP_DIRECTIVES);
  }
}
```
