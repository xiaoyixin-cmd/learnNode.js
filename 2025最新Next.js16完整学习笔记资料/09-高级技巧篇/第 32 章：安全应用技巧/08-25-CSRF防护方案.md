# CSRF 防护方案

跨站请求伪造(CSRF)是一种常见的 Web 安全威胁。本文介绍如何在 Next.js 应用中实现有效的 CSRF 防护,保护用户免受攻击。

## 核心概念

### CSRF 攻击

攻击者诱导用户在已登录的网站上执行非预期的操作。

### CSRF Token

服务器生成的随机令牌,用于验证请求的合法性。

### SameSite Cookie

限制 Cookie 在跨站请求中的发送,防止 CSRF 攻击。

### Double Submit Cookie

将 CSRF Token 同时存储在 Cookie 和请求参数中进行验证。

## 2. Double Submit Cookie 模式

### 2.1 实现 Double Submit Cookie

```typescript
// lib/csrf.ts
import { randomBytes } from "crypto";

export function generateCSRFToken(): string {
  return randomBytes(32).toString("hex");
}

export function verifyDoubleSubmitCookie(
  cookieToken: string | undefined,
  headerToken: string | undefined
): boolean {
  if (!cookieToken || !headerToken) {
    return false;
  }

  // 使用时间安全的比较
  return timingSafeEqual(Buffer.from(cookieToken), Buffer.from(headerToken));
}

function timingSafeEqual(a: Buffer, b: Buffer): boolean {
  if (a.length !== b.length) {
    return false;
  }

  let result = 0;
  for (let i = 0; i < a.length; i++) {
    result |= a[i] ^ b[i];
  }

  return result === 0;
}
```

### 2.2 中间件验证

```typescript
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";
import { verifyDoubleSubmitCookie } from "@/lib/csrf";

export function middleware(request: NextRequest) {
  // 跳过GET、HEAD、OPTIONS请求
  if (["GET", "HEAD", "OPTIONS"].includes(request.method)) {
    return NextResponse.next();
  }

  // 跳过公开API
  if (request.nextUrl.pathname.startsWith("/api/public")) {
    return NextResponse.next();
  }

  const cookieToken = request.cookies.get("csrf-token")?.value;
  const headerToken = request.headers.get("x-csrf-token");

  if (!verifyDoubleSubmitCookie(cookieToken, headerToken)) {
    return NextResponse.json({ error: "CSRF验证失败" }, { status: 403 });
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/api/:path*", "/actions/:path*"],
};
```

## 3. 加密 CSRF Token

### 3.1 Token 加密

```typescript
// lib/csrf-crypto.ts
import { createCipheriv, createDecipheriv, randomBytes } from "crypto";

const ALGORITHM = "aes-256-gcm";
const SECRET_KEY = process.env.CSRF_SECRET_KEY || randomBytes(32);

export function encryptCSRFToken(token: string): string {
  const iv = randomBytes(16);
  const cipher = createCipheriv(ALGORITHM, SECRET_KEY, iv);

  let encrypted = cipher.update(token, "utf8", "hex");
  encrypted += cipher.final("hex");

  const authTag = cipher.getAuthTag();

  return `${iv.toString("hex")}:${encrypted}:${authTag.toString("hex")}`;
}

export function decryptCSRFToken(encryptedToken: string): string | null {
  try {
    const [ivHex, encrypted, authTagHex] = encryptedToken.split(":");

    const iv = Buffer.from(ivHex, "hex");
    const authTag = Buffer.from(authTagHex, "hex");

    const decipher = createDecipheriv(ALGORITHM, SECRET_KEY, iv);
    decipher.setAuthTag(authTag);

    let decrypted = decipher.update(encrypted, "hex", "utf8");
    decrypted += decipher.final("utf8");

    return decrypted;
  } catch (error) {
    return null;
  }
}

export function generateSecureCSRFToken(): string {
  const token = randomBytes(32).toString("hex");
  return encryptCSRFToken(token);
}
```

```typescript
import type { NextRequest } from "next/server";
import { decryptCSRFToken } from "@/lib/csrf-crypto";

export function middleware(request: NextRequest) {
  if (["POST", "PUT", "DELETE", "PATCH"].includes(request.method)) {
    const cookieToken = request.cookies.get("csrf-token")?.value;
    const headerToken = request.headers.get("x-csrf-token");

    if (!cookieToken || !headerToken) {
      return NextResponse.json({ error: "CSRF Token缺失" }, { status: 403 });
    }

    const decryptedCookieToken = decryptCSRFToken(cookieToken);
    const decryptedHeaderToken = decryptCSRFToken(headerToken);

    if (
      !decryptedCookieToken ||
      !decryptedHeaderToken ||
      decryptedCookieToken !== decryptedHeaderToken
    ) {
      return NextResponse.json({ error: "CSRF验证失败" }, { status: 403 });
    }
  }

  return NextResponse.next();
}
```

## 常见问题

### 1. SameSite Cookie 的不同值有什么区别?

**SameSite 属性对比**:

| 值     | 行为                 | 适用场景         |
| ------ | -------------------- | ---------------- |
| Strict | 完全禁止跨站发送     | 高安全要求       |
| Lax    | 允许顶级导航跨站发送 | 平衡安全和可用性 |
| None   | 允许所有跨站发送     | 需要跨站功能     |

```typescript
// 不同场景的配置
export function setCookie(name: string, value: string, scenario: string) {
  const baseOptions = {
    httpOnly: true,
    secure: true,
    path: "/",
  };

  switch (scenario) {
    case "high-security":
      return {
        ...baseOptions,
        sameSite: "strict" as const,
      };

    case "normal":
      return {
        ...baseOptions,
        sameSite: "lax" as const,
      };

    case "cross-site":
      return {
        ...baseOptions,
        sameSite: "none" as const,
        secure: true, // None必须配合Secure
      };

    default:
      return {
        ...baseOptions,
        sameSite: "lax" as const,
      };
  }
}
```

### 2. CSRF Token 应该存储在哪里?

**存储方式对比**:

| 存储位置       | 优点          | 缺点            |
| -------------- | ------------- | --------------- |
| Cookie         | 自动发送,方便 | 可能被 XSS 读取 |
| LocalStorage   | 不会自动发送  | 容易被 XSS 读取 |
| SessionStorage | 标签页隔离    | 容易被 XSS 读取 |
| 内存           | 最安全        | 刷新页面丢失    |

**推荐方案**: 使用 HttpOnly Cookie 存储 secret,将 token 返回给客户端在请求头中发送。

```typescript
// 推荐的实现方式
export async function GET() {
  const secret = generateSecret();
  const token = generateToken(secret);

  // Secret存储在HttpOnly Cookie中
  cookies().set("csrf-secret", secret, {
    httpOnly: true, // 防止XSS读取
    secure: true,
    sameSite: "strict",
  });

  // Token返回给客户端
  return NextResponse.json({ token });
}
```

```typescript
  const { searchParams } = new URL(request.url);

  // 删除用户
  await deleteUser(id);

  return NextResponse.json({ success: true });
}
```

## 注意事项

### 1. Token 生成

- 使用加密安全的随机数生成器
- Token 长度至少 32 字节
- 每个会话使用唯一的 Token
- 定期轮换 Token

### 2. Token 验证

- 使用时间安全的比较函数
- 验证 Token 的有效期
- 验证 Token 的来源
- 记录验证失败的尝试

### 3. Cookie 配置

- 设置 HttpOnly 防止 XSS
- 设置 Secure 仅 HTTPS 传输
- 设置 SameSite 防止 CSRF
- 设置合适的过期时间

### 4. 请求方法

- 只验证修改数据的请求(POST、PUT、DELETE、PATCH)
- GET 请求不应修改数据
- OPTIONS 请求不需要验证

### 5. 错误处理

- 验证失败返回 403 状态码
- 不要泄露具体的错误信息
- 记录可疑的请求
- 实施速率限制

### 6. 性能考虑

- 缓存 CSRF Token
- 使用 Edge Runtime 加速验证
- 避免每次请求都生成新 Token
- 使用高效的加密算法

### 7. 兼容性

- 支持单页应用(SPA)
- 支持移动应用
- 支持第三方集成
- 提供降级方案

## 总结

CSRF 防护是 Web 应用安全的重要组成部分。本文介绍了:

1. **基础防护**: SameSite Cookie、CSRF Token 验证
2. **高级方案**: Double Submit Cookie、加密 Token
3. **第三方集成**: 使用 csrf 库
4. **实战场景**: 认证系统、表单提交、API 保护、支付流程

关键要点:

- 使用 SameSite Cookie 作为第一道防线
- 实施 CSRF Token 验证机制
- 使用 HttpOnly 和 Secure Cookie
- 验证所有修改数据的请求
- 使用时间安全的比较函数
- 定期轮换 CSRF Token
- 记录和监控可疑请求

通过多层防护策略,可以有效防止 CSRF 攻击,保护用户数据和应用安全。
