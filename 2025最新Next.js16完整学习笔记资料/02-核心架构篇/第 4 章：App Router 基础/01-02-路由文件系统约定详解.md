# 路由文件系统约定详解

## 1. 概述 (Overview)

> **核心定义**: Next.js App Router 采用基于文件系统的路由机制,通过文件和文件夹的组织结构自动生成应用路由。这种约定优于配置(Convention over Configuration)的设计哲学,使得路由创建变得直观、简单且强大。

Next.js 16 的 App Router 将路由定义从"文件即路由"(Pages Router)演进为"文件夹即路由"(App Router),每个文件夹代表一个路由段(route segment),通过特殊命名的文件定义该路由段的 UI 和行为。

### 1.1 概念定义

**文件系统路由(File-System Based Routing)**:

文件系统路由是一种将文件和文件夹的物理结构直接映射为应用 URL 结构的路由机制。在 Next.js App Router 中:

- **文件夹**: 定义路由段(route segment)
- **特殊文件**: 定义路由的 UI 和行为(page.tsx, layout.tsx 等)
- **命名约定**: 通过特定的文件命名模式实现高级路由功能(动态路由、并行路由等)

**核心原则**:

1. **文件夹定义路由**: 每个文件夹代表一个 URL 路径段
2. **特殊文件定义 UI**: 只有特殊命名的文件才会成为路由的公开访问点
3. **嵌套自动组合**: 文件夹的嵌套自动形成路由的嵌套结构
4. **约定优于配置**: 遵循命名约定即可实现复杂功能,无需额外配置

**基本示例**:

```
app/
├── page.tsx                  → /
├── about/
│   └── page.tsx              → /about
├── blog/
│   ├── page.tsx              → /blog
│   └── [slug]/
│       └── page.tsx          → /blog/:slug
└── dashboard/
    ├── layout.tsx            → 布局(不影响URL)
    ├── page.tsx              → /dashboard
    └── settings/
        └── page.tsx          → /dashboard/settings
```

**URL 映射关系**:

| 文件路径                        | URL                  | 说明           |
| :------------------------------ | :------------------- | :------------- |
| `app/page.tsx`                  | `/`                  | 根路由         |
| `app/about/page.tsx`            | `/about`             | 静态路由       |
| `app/blog/[slug]/page.tsx`      | `/blog/:slug`        | 动态路由       |
| `app/blog/[...slug]/page.tsx`   | `/blog/*`            | Catch-all 路由 |
| `app/shop/[[...slug]]/page.tsx` | `/shop` 或 `/shop/*` | 可选 Catch-all |

> **设计理念**: App Router 的文件系统约定旨在通过目录结构的可视化组织,让开发者能够直观理解和管理应用的路由结构,同时提供强大的扩展能力。

### 1.2 背景与演进

**传统路由 vs 文件系统路由**:

在传统的 Web 框架中,路由需要通过配置文件或代码显式定义:

```javascript
// 传统路由配置(如 Express.js)
app.get('/', (req, res) => { ... });
app.get('/about', (req, res) => { ... });
app.get('/blog/:slug', (req, res) => { ... });
// 需要手动维护大量路由配置
```

Next.js 通过文件系统路由简化了这一过程:

```
// 创建文件即创建路由,无需配置
app/page.tsx              → /
app/about/page.tsx        → /about
app/blog/[slug]/page.tsx  → /blog/:slug
```

**Pages Router vs App Router 约定对比**:

| 维度     | Pages Router    | App Router       |
| :------- | :-------------- | :--------------- |
| 路由单位 | 文件            | 文件夹           |
| 路由文件 | 任意命名.tsx    | page.tsx         |
| 布局     | \_app.tsx(全局) | layout.tsx(嵌套) |
| 加载状态 | 手动实现        | loading.tsx      |
| 错误处理 | \_error.tsx     | error.tsx        |
| 动态路由 | [param].tsx     | [param]/page.tsx |
| 并行路由 | 不支持          | @folder/page.tsx |

**演进历程**:

- **Next.js 1.0 (2016)**: 引入 Pages Router,文件即路由

  ```
  pages/index.js → /
  pages/about.js → /about
  ```

- **Next.js 9.0 (2019)**: 引入动态路由

  ```
  pages/posts/[id].js → /posts/:id
  ```

- **Next.js 13.0 (2022)**: 引入 App Router,文件夹即路由

  ```
  app/posts/[id]/page.tsx → /posts/:id
  ```

- **Next.js 16.0 (2025)**: App Router 完全成熟,成为默认推荐
  - 完整的特殊文件支持
  - 并行路由和拦截路由稳定
  - 路由组和私有文件夹
  - 更强大的类型推断

### 1.3 核心价值

**为什么使用文件系统路由?**

**1. 直观性**:

文件结构直接反映 URL 结构,一目了然:

```
app/
├── products/           → /products
│   ├── page.tsx        → 产品列表
│   └── [id]/           → /products/:id
│       ├── page.tsx    → 产品详情
│       └── reviews/    → /products/:id/reviews
│           └── page.tsx → 产品评论
```

开发者无需查看配置文件,通过目录树即可理解整个应用的路由结构。

**2. 零配置**:

无需编写路由配置代码:

```typescript
// ❌ 不需要这样的配置
const routes = [
  { path: "/", component: Home },
  { path: "/about", component: About },
  { path: "/blog/:slug", component: BlogPost },
];

// ✅ 只需要创建文件
app / page.tsx;
app / about / page.tsx;
app / blog / [slug] / page.tsx;
```

**3. 自动代码分割**:

每个路由自动进行代码分割,只加载需要的代码:

```typescript
// app/products/page.tsx - 自动分割为独立 chunk
export default function Products() {
  return <ProductList />;
}

// app/products/[id]/page.tsx - 另一个独立 chunk
export default function Product() {
  return <ProductDetail />;
}
```

**4. 类型安全**:

TypeScript 可以推断路由参数类型:

```typescript
// app/blog/[slug]/page.tsx
interface PageProps {
  params: Promise<{
    slug: string; // 自动推断类型
  }>;
}

export default async function BlogPost({ params }: PageProps) {
  const { slug } = await params;
  // slug 类型为 string,有完整的类型支持
}
```

**5. 并行开发**:

团队成员可以独立开发不同的路由,减少冲突:

```
app/
├── marketing/      → 市场团队负责
│   ├── page.tsx
│   └── campaigns/
├── dashboard/      → 产品团队负责
│   ├── page.tsx
│   └── analytics/
└── admin/          → 后台团队负责
    ├── page.tsx
    └── users/
```

**6. 扩展性**:

通过命名约定支持高级功能:

- **并行路由**: `@folder` - 同时渲染多个页面
- **拦截路由**: `(.)folder` - 拦截和覆盖路由
- **路由组**: `(folder)` - 组织代码不影响 URL
- **私有文件夹**: `_folder` - 排除在路由之外

**性能优势**:

| 特性         | 传统配置路由 | Next.js 文件系统路由 |
| :----------- | :----------- | :------------------- |
| 代码分割     | 手动配置     | 自动实现             |
| 预加载       | 需要额外配置 | 自动预加载           |
| 缓存优化     | 复杂配置     | 自动优化             |
| Tree Shaking | 有限支持     | 完整支持             |
| 构建时分析   | 困难         | 简单明了             |

## 2. 特殊文件约定详解

### 2.1 完整的特殊文件列表

Next.js App Router 定义了以下特殊文件名:

| 文件名             | 用途           | 必需  | 组件类型      |
| :----------------- | :------------- | :---- | :------------ |
| `layout.tsx`       | 共享布局       | ✓(根) | Server/Client |
| `page.tsx`         | 路由页面       | ✓     | Server/Client |
| `loading.tsx`      | 加载 UI        | ✗     | Server/Client |
| `not-found.tsx`    | 404 页面       | ✗     | Server/Client |
| `error.tsx`        | 错误边界       | ✗     | Client Only   |
| `global-error.tsx` | 全局错误       | ✗     | Client Only   |
| `route.ts`         | API 端点       | ✗     | -             |
| `template.tsx`     | 重新渲染的布局 | ✗     | Server/Client |
| `default.tsx`      | 并行路由后备   | ✗     | Server/Client |

### 2.2 layout.tsx 详解

**定义**: 布局是在多个路由之间共享的 UI,在导航时保持状态,不重新渲染。

**特性**:

1. **必需性**: 根布局(`app/layout.tsx`)是必需的
2. **嵌套**: 自动嵌套,子路由继承父路由布局
3. **状态保留**: 导航时不重新渲染
4. **数据获取**: 可以是异步函数,直接获取数据

**根布局示例**:

```typescript
// app/layout.tsx
export const metadata = {
  title: {
    default: "我的应用",
    template: "%s | 我的应用",
  },
  description: "应用描述",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="zh">
      <body>
        <header>
          <nav>全局导航</nav>
        </header>
        <main>{children}</main>
        <footer>全局页脚</footer>
      </body>
    </html>
  );
}
```

**嵌套布局示例**:

```typescript
// app/dashboard/layout.tsx
export default async function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  // 布局可以获取数据
  const user = await getUser();

  return (
    <div className="dashboard-layout">
      <aside>
        <UserInfo user={user} />
        <DashboardNav />
      </aside>
      <main>{children}</main>
    </div>
  );
}

// 最终渲染结构:
// RootLayout
//   └─ DashboardLayout
//      └─ DashboardPage
```

**并行路由插槽**:

```typescript
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children, // 默认插槽
  analytics, // @analytics 插槽
  team, // @team 插槽
}: {
  children: React.ReactNode;
  analytics: React.ReactNode;
  team: React.ReactNode;
}) {
  return (
    <div className="dashboard">
      <div className="main">{children}</div>
      <aside className="sidebar">
        <section className="analytics">{analytics}</section>
        <section className="team">{team}</section>
      </aside>
    </div>
  );
}
```

**限制**:

- 根布局必须包含 `<html>` 和 `<body>` 标签
- 布局不能访问路由段参数(`params`)
- 布局之间不能直接传递数据,需通过数据获取

### 2.3 page.tsx 详解

**定义**: 页面是路由的唯一公开 UI,只有包含 `page.tsx` 的路由才可访问。

**基础示例**:

```typescript
// app/blog/page.tsx
export default function BlogPage() {
  return (
    <div>
      <h1>博客</h1>
      <PostList />
    </div>
  );
}
```

**接收参数**:

```typescript
// app/blog/[slug]/page.tsx
interface PageProps {
  params: Promise<{
    slug: string;
  }>;
  searchParams: Promise<{
    sort?: string;
    page?: string;
  }>;
}

export default async function BlogPost({ params, searchParams }: PageProps) {
  const { slug } = await params;
  const { sort = "date", page = "1" } = await searchParams;

  return (
    <article>
      <h1>文章: {slug}</h1>
      <p>
        排序: {sort}, 页码: {page}
      </p>
    </article>
  );
}
```

**数据获取**:

```typescript
// app/products/[id]/page.tsx
async function getProduct(id: string) {
  const res = await fetch(`https://api.example.com/products/${id}`, {
    next: { revalidate: 3600 }, // ISR: 1小时
  });

  if (!res.ok) {
    throw new Error("Failed to fetch product");
  }

  return res.json();
}

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const product = await getProduct(id);

  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <p>价格: ${product.price}</p>
    </div>
  );
}

// 生成静态参数
export async function generateStaticParams() {
  const products = await fetch("https://api.example.com/products").then((res) =>
    res.json()
  );

  return products.map((product: any) => ({
    id: product.id.toString(),
  }));
}

// 生成元数据
export async function generateMetadata({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const product = await getProduct(id);

  return {
    title: product.name,
    description: product.description,
    openGraph: {
      images: [product.image],
    },
  };
}
```

**段配置**:

```typescript
// app/api/users/page.tsx
export const dynamic = "force-dynamic"; // 强制动态渲染
export const runtime = "edge"; // Edge Runtime
export const revalidate = 60; // ISR: 60秒

export default function UsersPage() {
  return <UserList />;
}
```

### 2.4 loading.tsx 详解

**定义**: Loading UI 在路由段加载时自动显示,使用 Suspense 实现流式渲染。

**工作原理**:

```typescript
// app/dashboard/loading.tsx
export default function Loading() {
  return <LoadingSpinner />;
}

// 自动转换为:
<Suspense fallback={<Loading />}>
  <DashboardPage />
</Suspense>;
```

**即时加载状态**:

```typescript
// app/dashboard/loading.tsx
export default function DashboardLoading() {
  return (
    <div className="dashboard-skeleton">
      <div className="skeleton-header" />
      <div className="skeleton-content">
        <div className="skeleton-card" />
        <div className="skeleton-card" />
        <div className="skeleton-card" />
      </div>
    </div>
  );
}
```

**嵌套加载**:

```
app/
├── loading.tsx                → 全局加载
└── dashboard/
    ├── loading.tsx            → 仪表板加载
    ├── page.tsx
    └── settings/
        ├── loading.tsx        → 设置页加载
        └── page.tsx

访问 /dashboard/settings 时的加载优先级:
1. 优先显示 settings/loading.tsx
2. 如果没有,显示 dashboard/loading.tsx
3. 如果没有,显示根 loading.tsx
```

**与 Suspense 结合**:

```typescript
// app/dashboard/page.tsx
import { Suspense } from "react";

export default function Dashboard() {
  return (
    <div>
      <h1>仪表板</h1>

      {/* loading.tsx 提供整页加载状态 */}

      {/* Suspense 提供组件级加载状态 */}
      <Suspense fallback={<AnalyticsSkeleton />}>
        <Analytics />
      </Suspense>

      <Suspense fallback={<ChartSkeleton />}>
        <RevenueChart />
      </Suspense>
    </div>
  );
}
```

### 2.5 error.tsx 详解

**定义**: 错误边界捕获路由段中的错误,提供错误 UI 和恢复机制。

**必须是 Client Component**:

```typescript
// app/dashboard/error.tsx
"use client"; // 必需!

import { useEffect } from "react";

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // 记录错误到错误报告服务
    console.error("Dashboard error:", error);
  }, [error]);

  return (
    <div className="error-container">
      <h2>仪表板出错了</h2>
      <p>{error.message}</p>
      <button onClick={() => reset()}>重试</button>
    </div>
  );
}
```

**错误类型判断**:

```typescript
// app/api/error.tsx
"use client";

export default function APIError({ error, reset }: any) {
  // 判断错误类型
  const is404 =
    error.message.includes("404") || error.message.includes("Not found");
  const is401 =
    error.message.includes("401") || error.message.includes("Unauthorized");
  const is500 =
    error.message.includes("500") || error.message.includes("Server error");

  if (is404) {
    return (
      <div>
        <h2>资源未找到</h2>
        <p>请求的API端点不存在</p>
      </div>
    );
  }

  if (is401) {
    return (
      <div>
        <h2>未授权</h2>
        <p>您需要登录才能访问此资源</p>
        <a href="/login">去登录</a>
      </div>
    );
  }

  if (is500) {
    return (
      <div>
        <h2>服务器错误</h2>
        <p>服务器遇到了问题,请稍后重试</p>
        <button onClick={reset}>重试</button>
      </div>
    );
  }

  return (
    <div>
      <h2>发生错误</h2>
      <pre>{error.message}</pre>
      <button onClick={reset}>重试</button>
    </div>
  );
}
```

**错误边界嵌套**:

```
app/
├── error.tsx              → 捕获所有未处理的错误
├── global-error.tsx       → 捕获根布局错误
└── dashboard/
    ├── error.tsx          → 捕获仪表板错误
    └── analytics/
        ├── error.tsx      → 捕获分析页错误
        └── page.tsx

错误冒泡机制:
1. analytics/page.tsx 抛出错误
2. 被 analytics/error.tsx 捕获
3. 如果 analytics/error.tsx 也错误
4. 冒泡到 dashboard/error.tsx
5. 如果仍然错误,冒泡到根 error.tsx
6. 最后由 global-error.tsx 兜底
```

**global-error.tsx**:

```typescript
// app/global-error.tsx
"use client";

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    // 必须包含 html 和 body 标签
    <html>
      <body>
        <h2>应用崩溃了</h2>
        <p>发生了严重错误</p>
        <button onClick={() => reset()}>重新加载应用</button>
      </body>
    </html>
  );
}
```

**路由段级 404**:

```typescript
// app/blog/not-found.tsx
import Link from "next/link";

export default function BlogNotFound() {
  return (
    <div className="blog-not-found">
      <h2>博客文章未找到</h2>

```

**触发 404**:

```typescript
// app/blog/[slug]/page.tsx
import { notFound } from "next/navigation";

async function getPost(slug: string) {
  const res = await fetch(`https://api.example.com/posts/${slug}`);
  if (!res.ok) return null;
  return res.json();
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await getPost(slug);

  if (!post) {
    notFound(); // 触发最近的 not-found.tsx
  }

  return <article>{post.content}</article>;
}
```

### 2.7 template.tsx 详解

**定义**: 模板类似布局,但每次导航都重新挂载,不保留状态。

**layout vs template**:

```typescript
// layout.tsx - 状态保留
"use client";
import { useState } from "react";

export default function Layout({ children }) {
  const [count, setCount] = useState(0);

  return (
    <div>
      <button onClick={() => setCount(count + 1)}>
        点击: {count} {/* 路由切换时保持状态 */}
      </button>
      {children}
    </div>
  );
}

// template.tsx - 每次重置
("use client");
import { useState } from "react";

export default function Template({ children }) {
  const [count, setCount] = useState(0);

  return (
    <div>
      <button onClick={() => setCount(count + 1)}>
        点击: {count} {/* 路由切换时重置为 0 */}
      </button>
      {children}
    </div>
  );
}
```

**适用场景**:

1. **CSS 动画**: 需要每次路由都播放进入动画

```typescript
// app/blog/template.tsx
'use client';

export default function BlogTemplate({ children }) {
  return (
    <div className="blog-wrapper fade-in">
      {children}
    </div>
  );
}
```

### 3.3 Catch-all 路由

**语法**: `[...param]`

**匹配规则**:

```
[...slug] 匹配:
/docs/introduction            → slug = ['introduction']
/docs/api/overview            → slug = ['api', 'overview']
/docs/guide/getting-started   → slug = ['guide', 'getting-started']

不匹配:
/docs                         → 404 (没有参数)
```

**实现**:

```typescript
// app/docs/[...slug]/page.tsx
interface PageProps {
  params: Promise<{
    slug: string[]; // 数组类型!
  }>;
}

export default async function DocsPage({ params }: PageProps) {
  const { slug } = await params;

  // slug 是字符串数组
  const path = slug.join("/");
  const content = await getDocsContent(path);

  return (
    <div>
      <h1>文档: {path}</h1>
      <div dangerouslySetInnerHTML={{ __html: content }} />
    </div>
  );
}

export async function generateStaticParams() {
  const allDocs = await getAllDocsPages();

  }));
}
```

### 3.4 可选 Catch-all 路由

**语法**: `[[...param]]`

**匹配规则**:

```
[[...slug]] 匹配:
/shop                         → slug = undefined 或 []
/shop/electronics             → slug = ['electronics']
/shop/electronics/laptops     → slug = ['electronics', 'laptops']
```

**实现**:

```typescript
// app/shop/[[...slug]]/page.tsx
interface PageProps {
  params: Promise<{
    slug?: string[]; // 可选数组
  }>;
}

export default async function ShopPage({ params }: PageProps) {
  const { slug = [] } = await params; // 提供默认值

  if (slug.length === 0) {
    // /shop - 显示所有分类
    return <AllCategories />;
  }

  if (slug.length === 1) {
    // /shop/electronics - 显示分类产品
    return <CategoryProducts category={slug[0]} />;
  }

  if (slug.length === 2) {
    // /shop/electronics/laptops - 显示子分类
    return <SubcategoryProducts category={slug[0]} subcategory={slug[1]} />;
  }

  // 更深层级
  return <DeepCategory path={slug} />;
}

export async function generateStaticParams() {
  const paths = [];

  // 空路径
  paths.push({ slug: undefined });

  // 一级分类
  const categories = await getCategories();
  categories.forEach((cat) => {
    paths.push({ slug: [cat.slug] });
  });

  // 二级分类
  for (const cat of categories) {
    const subcats = await getSubcategories(cat.slug);
    subcats.forEach((sub) => {
      paths.push({ slug: [cat.slug, sub.slug] });
    });
  }

  return paths;
}
```

### 3.5 路由优先级

**优先级顺序**(从高到低):

1. 静态路由(精确匹配)
2. 动态路由(单个参数)
3. Catch-all 路由
4. 可选 Catch-all 路由

**示例**:

```
app/shop/
├── page.tsx                      → /shop (优先级 1)
├── new/
│   └── page.tsx                  → /shop/new (优先级 1)
├── [id]/
│   └── page.tsx                  → /shop/:id (优先级 2)
├── [category]/[product]/
│   └── page.tsx                  → /shop/:category/:product (优先级 2)
├── [...slug]/
│   └── page.tsx                  → /shop/* (优先级 3)
└── [[...slug]]/
    └── page.tsx                  → /shop 或 /shop/* (优先级 4)

URL 匹配示例:
/shop/new         → shop/new/page.tsx (静态优先)
/shop/123         → shop/[id]/page.tsx (动态匹配)
/shop/a/b         → shop/[category]/[product]/page.tsx
/shop/a/b/c       → shop/[...slug]/page.tsx (Catch-all)
```

## 6. 常见问题

### Q1: page.tsx 和 layout.tsx 必须都存在吗?

**答**:

- **layout.tsx**: 根布局(`app/layout.tsx`)是必需的
- **page.tsx**: 只有包含 `page.tsx` 的路由才可公开访问

### Q2: 动态路由参数什么时候是 Promise?

**答**: Next.js 15+ 中,`params` 和 `searchParams` 都是 Promise,需要 await:

```typescript
// Next.js 15+
export default async function Page({ params }) {
  const { id } = await params; // 必须 await
}
```

### Q3: 如何在多个动态段之间共享数据?

**答**: 使用数据获取去重或缓存:

```typescript
import { cache } from "react";

const getProduct = cache(async (id: string) => {
  return await fetch(`/api/products/${id}`).then((r) => r.json());
});

// 多次调用只会执行一次
const product1 = await getProduct(id);
const product2 = await getProduct(id); // 使用缓存
```

### Q4: Catch-all 路由会匹配空路径吗?

**答**:

- `[...slug]` - 不匹配空路径(`/docs` 会 404)
- `[[...slug]]` - 匹配空路径(`/docs` 正常)

### Q5: 并行路由的 default.tsx 什么时候需要?

**答**: 当导航到没有定义插槽内容的路由时,需要 default.tsx 提供后备内容,否则会 404。

### Q6: 私有文件夹里可以放 page.tsx 吗?

**答**: 可以放,但不会成为路由。私有文件夹内的所有内容都被排除在路由系统之外。

### Q7: 路由组会影响数据获取吗?

**答**: 不会。路由组只影响文件组织和布局,不影响数据获取逻辑。

### Q8: error.tsx 可以捕获 layout.tsx 的错误吗?

**答**: 不能。`error.tsx` 不能捕获同级或父级 layout 的错误,需要使用 `global-error.tsx`。

### Q9: loading.tsx 和 Suspense 有什么区别?

**答**:

- `loading.tsx`: 路由级加载,整个页面
- `Suspense`: 组件级加载,更细粒度

### Q10: 如何禁用文件系统路由?

**答**: 不推荐,但可以通过配置实现:

```typescript
// next.config.ts
module.exports = {
  useFileSystemPublicRoutes: false,
};
```

## 7. 注意事项

### 7.1 命名规范

1. **特殊文件必须小写**: `page.tsx` 不能写成 `Page.tsx`
2. **动态路由用方括号**: `[id]` 不能用 `:id` 或 `{id}`
3. **路由组用圆括号**: `(group)` 不能用 `[group]` 或 `{group}`
4. **私有文件夹用下划线**: `_private` 开头

### 7.2 性能注意

1. **避免过深嵌套**: 建议不超过 4-5 层
2. **合理使用 generateStaticParams**: 不要生成数万个静态页面
3. **loading.tsx 保持轻量**: 避免复杂组件

### 7.3 SEO 注意

1. **总是提供 metadata**: 静态或动态
2. **使用 generateMetadata**: 动态页面元数据
3. **提供 opengraph-image**: 社交分享优化

### 7.4 类型安全

1. **定义 PageProps 接口**: 明确参数类型
2. **Catch-all 参数是数组**: 注意类型处理
3. **可选 Catch-all 提供默认值**: `const { slug = [] } = await params`

## 8. 总结

**核心要点**:

1. **文件夹即路由**: App Router 通过文件夹定义 URL 结构
2. **特殊文件定义行为**: page.tsx、layout.tsx 等控制路由的不同方面
3. **约定优于配置**: 遵循命名约定即可实现强大功能
4. **灵活扩展**: 支持并行路由、拦截路由等高级模式

**选择建议**:

- **简单网站**: 使用静态路由
- **内容网站**: 使用动态路由 + ISR
- **复杂应用**: 组合使用路由组、并行路由
- **多租户**: 使用可选 Catch-all 路由

**最佳实践**:

1. 保持路由结构清晰简单
2. 合理使用路由组组织代码
3. 为动态路由提供类型定义
4. 使用私有文件夹管理非路由文件
5. 充分利用特殊文件(loading、error 等)
