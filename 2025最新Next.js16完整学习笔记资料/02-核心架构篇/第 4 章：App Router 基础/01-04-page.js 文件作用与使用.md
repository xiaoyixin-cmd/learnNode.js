# 01-04-page.js 文件作用与使用

## 概述

### 什么是 page.js

`page.js` (或 `page.tsx`) 是 Next.js App Router 中定义页面的核心文件。每个路由段要成为可访问的页面,必须包含一个 `page.js` 文件。它是应用中唯一能直接响应 URL 请求的特殊文件,是用户最终看到的内容。

在 Next.js 16 中,`page.js` 默认是服务端组件 (Server Component),支持异步数据获取、流式渲染、自动代码分割等先进特性,是构建高性能 Web 应用的关键。

### page.js 的核心价值

**1. 路由可访问性**

- 只有包含 `page.js` 的路由段才能被访问
- 定义了 URL 到 UI 的映射关系
- 是路由系统的终点

**2. 服务端优先**

- 默认服务端组件,减少客户端 JavaScript
- 直接访问后端资源(数据库、文件系统)
- 提升首屏渲染速度

**3. 自动优化**

- 代码自动分割,按需加载
- 预渲染支持 (SSG/ISR)
- 流式渲染提升 TTFB

**4. 类型安全**

- TypeScript 自动类型推断
- Props 类型严格校验
- 编译时错误检查

### 版本演进

| 版本         | 页面实现方式     | 特点                  | 数据获取                               |
| ------------ | ---------------- | --------------------- | -------------------------------------- |
| Next.js 1-12 | `pages/*.js`     | 基于文件路由          | `getServerSideProps`, `getStaticProps` |
| Next.js 13   | `app/*/page.tsx` | App Router 引入       | 异步 Server Component                  |
| Next.js 14   | 优化 page        | Turbopack 支持        | 优化数据缓存                           |
| Next.js 15   | Promise params   | `params` 成为 Promise | 破坏性变更                             |
| Next.js 16   | 稳定 page API    | 完整类型推断,PPR      | 最佳实践                               |

### 1.2 page.js 的 Props

页面组件可以接收两个 props: `params` 和 `searchParams`。

#### params 属性

`params` 包含动态路由段的参数,从 Next.js 15 开始必须是 Promise。

```typescript
// app/blog/[slug]/page.tsx
interface PageProps {
  params: Promise<{ slug: string }>;
}

export default async function BlogPost({ params }: PageProps) {
  const { slug } = await params; // 必须 await

  return (
    <article>
      <h1>文章: {slug}</h1>
    </article>
  );
}
```

#### searchParams 属性

`searchParams` 包含 URL 查询参数,也是 Promise 类型。

```typescript
// app/search/page.tsx
interface SearchPageProps {
  searchParams: Promise<{ q?: string; page?: string }>;
}

export default async function SearchPage({ searchParams }: SearchPageProps) {
  const { q, page } = await searchParams;

  const query = q || "";
  const currentPage = parseInt(page || "1");

  return (
    <div>
      <h1>搜索: {query}</h1>
      <p>当前页: {currentPage}</p>
    </div>
  );
}
```

async function getDocs() {
  const docsDirectory = path.join(process.cwd(), "content/docs");
  const fileNames = await fs.readdir(docsDirectory);

  const docs = await Promise.all(
    fileNames.map(async (fileName) => {
      const filePath = path.join(docsDirectory, fileName);
      const content = await fs.readFile(filePath, "utf8");
      return { fileName, content };

  );

  return docs;
}

  return <div>快速加载的组件</div>;

      {/* 慢速组件流式加载 */}

        <SlowComponent />
      </Suspense>
    </div>
  );
}

  page: number,
  category?: string,
  sort?: string

    q: query,
    page: page.toString(),
    ...(category && { category }),
    ...(sort && { sort }),

  return res.json();

## 适用场景

### 何时使用 page.js

**1. 公开可访问的页面**

- 网站首页
- 文章列表和详情
- 产品展示页
- 关于/联系页面

**2. 需要 SEO 的页面**

- 博客文章
- 产品页面
- 落地页
- 营销页面

**3. 服务端渲染需求**

- 需要实时数据的页面
- 需要访问服务端资源
- 需要保护 API 密钥

**4. 静态生成需求**

- 内容较少变化的页面
- 需要极快加载速度
- 高流量页面

### 何时使用动态路由

**1. 内容驱动的应用**

- 博客系统
- 文档站点
- 新闻网站

**2. 电商应用**

- 产品详情页
- 分类页面
- 用户个人页

**3. 用户生成内容**

- 用户资料
- 帖子详情
- 评论页面

### 何时使用 generateStaticParams

**1. 内容数量有限**

- 少于 10,000 个页面
- 构建时间可接受
- 内容变化不频繁

**2. SEO 优先**

- 需要预渲染所有页面
- 需要最快的加载速度
- 搜索引擎友好

**3. 配合 ISR**

- 定时更新内容
- 按需重新验证
- 平衡性能和实时性

### Q1: page.js 和 layout.js 有什么区别?

**答**:

- `layout.js` 定义跨多个页面共享的 UI,在导航时保持状态

    </div>
  );
}

// app/blog/page.tsx - 列表页

  return <PostList />; // 导航时重新渲染

// app/blog/[id]/page.tsx - 详情页

## 总结

### 核心要点

1. **page.js 的本质**

   - 定义路由的唯一内容
   - 是应用中唯一可公开访问的特殊文件
   - 默认是服务端组件

2. **Props 类型**

   - `params`: 动态路由参数 (Promise)
   - `searchParams`: URL 查询参数 (Promise)
   - 从 Next.js 15+ 开始必须 await

3. **数据获取**

   - 异步服务端组件支持直接 await
   - 使用 `fetch` API 内置缓存控制
   - 支持并行数据获取提升性能

4. **动态路由**

   - `[param]`: 单段动态路由
   - `[...param]`: Catch-all 路由
   - `[[...param]]`: 可选 Catch-all 路由

5. **Metadata 管理**

   - 静态 metadata 对象
   - 动态 generateMetadata 函数
   - 支持继承和覆盖

6. **路由段配置**

   - `dynamic`: 动态/静态渲染控制
   - `revalidate`: ISR 重新验证时间
   - `runtime`: Node.js/Edge Runtime

7. **性能优化**

   - 并行数据获取
   - 流式渲染
   - 代码自动分割
   - 静态生成

8. **错误处理**
   - `error.tsx` 错误边界
   - `not-found.tsx` 404 页面
   - `redirect()` 重定向

### 最佳实践

1. **优先使用服务端组件**

   - 减少客户端 JavaScript
   - 直接访问后端资源
   - 提升首屏性能

2. **合理的数据获取**

   - 并行获取独立数据
   - 选择正确的缓存策略
   - 避免过度获取

3. **类型安全**

   - 定义完整的 Props 类型
   - 使用 Metadata 类型
   - await Promise params

4. **性能优化**

   - 使用 generateStaticParams 预渲染
   - 流式渲染提升 TTFB
   - 懒加载非关键组件

5. **用户体验**
   - 提供 loading 状态
   - 友好的错误处理
   - 平滑的导航过渡
