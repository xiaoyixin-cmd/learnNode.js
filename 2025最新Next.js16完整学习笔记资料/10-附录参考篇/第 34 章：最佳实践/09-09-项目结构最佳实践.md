# 09-09-项目结构最佳实践

## 1. 概述 (Overview)

### 1.1 为什么项目结构很重要

项目结构是应用程序的骨架,直接影响代码的可维护性、可扩展性和团队协作效率。一个好的项目结构能让新成员快速上手,让代码易于理解和修改,减少技术债务的积累。

### 1.2 Next.js 16 项目结构特点

**核心特性**:

- App Router: 基于文件系统的路由
- 约定优于配置: 特殊文件名有特定含义
- 组件分层: Server Components 和 Client Components
- 模块化设计: 功能模块独立组织
- 类型安全: TypeScript 集成

### 1.3 项目结构对比

| 结构类型 | 适用场景   | 优势         | 劣势       |
| :------- | :--------- | :----------- | :--------- |
| 扁平结构 | 小型项目   | 简单直观     | 难以扩展   |
| 功能模块 | 中大型项目 | 高内聚低耦合 | 初期复杂   |
| 领域驱动 | 企业应用   | 业务清晰     | 学习成本高 |
| 混合结构 | 通用项目   | 灵活平衡     | 需要规范   |

**示例实现**:

```typescript
// domain/entities/user.ts
export class User {
  constructor(public id: string, public email: string, public name: string) {}
}

// domain/repositories/user-repository.ts
export interface UserRepository {
  findById(id: string): Promise<User | null>;
  save(user: User): Promise<void>;
}

// infrastructure/database/prisma-user-repository.ts
export class PrismaUserRepository implements UserRepository {
  async findById(id: string): Promise<User | null> {
    const user = await prisma.user.findUnique({ where: { id } });
    if (!user) return null;
    return new User(user.id, user.email, user.name);
  }

  async save(user: User): Promise<void> {
    await prisma.user.upsert({
      where: { id: user.id },
      create: user,
      update: user,
    });
  }
}

// application/use-cases/get-user.ts
export class GetUserUseCase {
  constructor(private userRepository: UserRepository) {}

  async execute(userId: string): Promise<User | null> {
    return await this.userRepository.findById(userId);
  }
}

// presentation/app/users/[id]/page.tsx
export default async function UserPage({ params }: { params: { id: string } }) {
  const { id } = await params;
  const userRepository = new PrismaUserRepository();
  const getUserUseCase = new GetUserUseCase(userRepository);
  const user = await getUserUseCase.execute(id);

  if (!user) notFound();

  return <UserProfile user={user} />;
}
```

## 9. 注意事项

### 9.1 避免过度设计

**问题**: 项目初期就采用复杂结构

**建议**:

```typescript
// ❌ 过度设计 - 小项目使用 DDD
src/
├── domains/
│   ├── user/
│   │   ├── entities/
│   │   ├── repositories/
│   │   ├── services/
│   │   └── use-cases/
│   └── ...
└── infrastructure/
    └── ...

// ✅ 简单开始
src/
├── app/
├── components/
├── lib/
└── actions/
```

**原则**:

- 从简单开始
- 根据需求演进
- 避免过早优化
- 保持灵活性

### 9.3 避免循环依赖

**问题**: 模块间相互引用

**检测**:

```bash
# 使用 madge 检测循环依赖
npx madge --circular src/

# 输出
Circular dependencies:
  components/user-card -> lib/user-utils -> components/user-card
```

**解决方案**:

```typescript
// ❌ 循环依赖
// components/user-card.tsx
import { formatUserName } from "@/lib/user-utils";

// lib/user-utils.ts
import { UserCard } from "@/components/user-card";

// ✅ 提取共享逻辑
// lib/user-utils.ts
export function formatUserName(name: string) {
  return name.toUpperCase();
}

// components/user-card.tsx
import { formatUserName } from "@/lib/user-utils";

// components/user-list.tsx
import { formatUserName } from "@/lib/user-utils";
import { UserCard } from "@/components/user-card";
```

## 10. 常见问题

### 10.1 src 目录是否必需?

**问题**: 是否需要 src 目录包裹所有代码

**解答**:

Next.js 支持两种结构:

```
# 方案一: 无 src 目录
my-app/
├── app/
├── components/
├── lib/
└── package.json

# 方案二: 有 src 目录
my-app/
├── src/
│   ├── app/
│   ├── components/
│   └── lib/
└── package.json
```

**选择建议**:

| 场景       | 推荐   | 原因           |
| :--------- | :----- | :------------- |
| 小型项目   | 无 src | 简单直观       |
| 中大型项目 | 有 src | 代码与配置分离 |
| Monorepo   | 有 src | 统一结构       |

---

### 10.2 如何组织共享组件?

**问题**: 共享组件放在哪里

**解答**:

```
# 方案一: 按类型分类
components/
├── ui/              # 基础 UI 组件
│   ├── button.tsx
│   └── input.tsx
├── forms/           # 表单组件
│   └── login-form.tsx
└── layouts/         # 布局组件
    └── header.tsx

# 方案二: 按功能分类
features/
├── auth/
│   └── components/
│       └── login-form.tsx
└── posts/
    └── components/
        └── post-card.tsx

# 方案三: 混合方式
components/
├── ui/              # 通用 UI 组件
└── shared/          # 跨功能共享组件

features/
├── auth/
│   └── components/  # 功能特定组件
└── posts/
    └── components/
```

**推荐**: 混合方式,平衡通用性和内聚性

**问题**: Server Actions 的组织方式

**解答**:

```
# 方案一: 集中管理

├── user.ts

# 方案二: 功能模块内
```

**推荐**: 方案二(功能模块内),便于维护
