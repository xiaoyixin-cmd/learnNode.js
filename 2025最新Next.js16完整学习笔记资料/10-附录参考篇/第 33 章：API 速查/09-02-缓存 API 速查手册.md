# 缓存 API 速查手册

## 1. 概述 (Overview)

Next.js 16 引入了全新的缓存系统,包括多个强大的缓存 API。本速查手册汇总了所有与缓存相关的 API,帮助开发者快速查找和使用缓存功能。

### 1.1 缓存系统概览

Next.js 16 的缓存系统包含以下几个层次:

- **请求级缓存**: fetch API 的缓存选项
- **组件级缓存**: 🆕 `'use cache'` 指令和相关 API
- **路径级缓存**: revalidatePath 和 revalidateTag
- **函数级缓存**: unstable_cache 包装器

### 1.2 🆕 Next.js 16 缓存变更

⚠️ **重要变更**: Next.js 16 对缓存系统进行了重大改革:

- fetch 默认缓存策略从 `force-cache` 改为 `no-store`
- 新增 `'use cache'` 指令用于组件级缓存
- 新增 `cacheLife` 和 `cacheTag` API
- 引入更细粒度的缓存控制

### 1.3 手册使用说明

**快速查找**: 使用 Ctrl+F 搜索 API 名称

**分类浏览**: 按照缓存层次分类查看

**版本标识**: 🆕 表示 Next.js 16 新增,⚠️ 表示有破坏性变更

## 2. 🆕 组件缓存 API (Component Caching)

### 2.1 'use cache' 指令

**功能**: 标记组件或函数使用缓存

**类型签名**:

```typescript
"use cache";
```

**使用示例**:

```typescript
// 缓存整个组件
export async function ProductList() {
  "use cache";

  const products = await db.product.findMany();

  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  );
}

// 缓存函数
export async function getProducts() {
  "use cache";

  const products = await db.product.findMany();
  return products;
}
```

**注意事项**:

- 🆕 Next.js 16 新增功能
- 必须是函数或组件的第一行
- 只能在服务端组件或异步函数中使用
- 配合 `cacheLife` 和 `cacheTag` 使用

**适用场景**:

- 数据变化不频繁的组件
- 计算密集型操作
- 数据库查询结果

### 2.2 🆕 cacheLife 函数

**功能**: 设置缓存的生命周期策略

**类型签名**:

```typescript
function cacheLife(profile: CacheLifeProfile): void;

type CacheLifeProfile =
  | "seconds" // 秒级缓存
  | "minutes" // 分钟级缓存
  | "hours" // 小时级缓存
  | "days" // 天级缓存
  | "weeks" // 周级缓存
  | "max" // 最大缓存时间
  | {
      stale: number; // 过期时间(秒)
      revalidate: number; // 重新验证时间(秒)
      expire?: number; // 失效时间(秒)
    };
```

**预设配置文件**:

| 配置文件 | stale  | revalidate | expire | 说明       |
| :------- | :----- | :--------- | :----- | :--------- |
| seconds  | 0      | 1          | 60     | 秒级更新   |
| minutes  | 60     | 60         | 3600   | 分钟级更新 |
| hours    | 300    | 3600       | 86400  | 小时级更新 |
| days     | 3600   | 86400      | 604800 | 天级更新   |
| weeks    | 86400  | 604800     | -      | 周级更新   |
| max      | 604800 | 604800     | -      | 最大缓存   |

**使用示例**:

```typescript
import { unstable_cacheLife as cacheLife } from "next/cache";

// 使用预设配置
export async function getProducts() {
  "use cache";
  cacheLife("hours");

  const products = await db.product.findMany();
  return products;
}

// 自定义配置
export async function getUser(id: string) {
  "use cache";
  cacheLife({
    stale: 60, // 60 秒内返回缓存(即使过期)
    revalidate: 300, // 5 分钟后重新验证
    expire: 3600, // 1 小时后完全失效
  });

  return await db.user.findUnique({ where: { id } });
}

// 实时数据(最小缓存)
export async function getRealtimeData() {
  "use cache";
  cacheLife("seconds");

  const data = await fetch("https://api.example.com/realtime");
  return data.json();
}

// 静态数据(最大缓存)
export async function getStaticContent() {
  "use cache";
  cacheLife("max");

  const content = await db.content.findMany();
  return content;
}
```

**配置参数详解**:

- **stale**: 过期时间,在此时间内直接返回缓存,不进行重新验证
- **revalidate**: 重新验证时间,超过此时间后在后台重新获取数据
- **expire**: 失效时间,超过此时间后缓存完全失效,必须重新获取

**缓存行为**:

```
时间轴: 0 -------- stale -------- revalidate -------- expire -------- ∞

0-stale:        返回缓存,不重新验证
stale-revalidate: 返回缓存,后台重新验证
revalidate-expire: 重新获取数据,更新缓存
expire-∞:       缓存失效,必须重新获取
```

**注意事项**:

- 🆕 Next.js 16 新增功能
- 必须配合 `'use cache'` 指令使用
- 预设配置文件简化了常见场景
- 自定义配置提供更精细的控制

**适用场景**:

- **seconds**: 实时数据,如股票价格、在线用户数
- **minutes**: 频繁更新的数据,如新闻列表、评论
- **hours**: 一般动态数据,如产品列表、用户资料
- **days**: 较稳定的数据,如文章内容、配置信息
- **weeks**: 静态数据,如归档内容、历史记录
- **max**: 永久不变的数据,如已发布的文章、图片

**相关章节**: 03-09-cacheLife 函数详解.md, 07-02-cacheLife 缓存生命周期管理案例.md

### 2.3 🆕 cacheTag 函数

**功能**: 为缓存条目添加标签,用于按需重新验证

**类型签名**:

```typescript
function cacheTag(...tags: string[]): void;
```

**使用示例**:

```typescript
import { unstable_cacheTag as cacheTag } from "next/cache";

// 单个标签
export async function getProducts() {
  "use cache";
  cacheTag("products");

  const products = await db.product.findMany();
  return products;
}

// 多个标签
export async function getProduct(id: string) {
  "use cache";
  cacheTag("products", `product-${id}`);

  const product = await db.product.findUnique({ where: { id } });
  return product;
}

// 分层标签
export async function getCategoryProducts(categoryId: string) {
  "use cache";
  cacheTag("products", "catalog", `category-${categoryId}`);

  const products = await db.product.findMany({
    where: { categoryId },
  });
  return products;
}
```

**标签命名规范**:

```typescript
// 推荐的标签命名模式
"products"; // 资源类型
"product-123"; // 特定资源
"user-456-posts"; // 用户相关资源
"category-electronics"; // 分类资源
"global"; // 全局标签
```

**配合 revalidateTag 使用**:

```typescript
// 在 Server Action 中重新验证
import { revalidateTag } from "next/cache";

async function updateProduct(id: string, data: any) {
  "use server";

  await db.product.update({ where: { id }, data });

  // 重新验证所有产品相关的缓存
  revalidateTag("products");

  // 重新验证特定产品的缓存
  revalidateTag(`product-${id}`);
}
```

**标签策略**:

| 策略     | 标签示例                   | 使用场景 | 优点                        | 缺点                     |
| :------- | :------------------------- | :------- | :-------------------------- | :----------------------- |
| 粗粒度   | `'products'`               | 批量更新 | 简单,一次性清除所有相关缓存 | 可能清除不需要更新的缓存 |
| 细粒度   | `'product-123'`            | 精确更新 | 只更新需要的缓存            | 需要管理更多标签         |
| 分层标签 | `'products', 'category-1'` | 灵活控制 | 可以按不同维度清除缓存      | 复杂度较高               |

**注意事项**:

- 🆕 Next.js 16 新增功能
- 可以为一个缓存条目添加多个标签
- 标签名称区分大小写
- 建议使用有意义的命名规范

**适用场景**:

- 需要按需重新验证的数据
- 有关联关系的数据(如产品和分类)
- 需要批量更新的缓存

**相关章节**: 03-10-cacheTag 函数详解.md, 07-01-revalidateTag 实际应用场景.md

## 3. 请求级缓存 API (Request-Level Caching)

### 3.1 fetch 缓存选项

**功能**: Next.js 扩展的 fetch API,支持缓存和重新验证

**类型签名**:

```typescript
function fetch(
  input: RequestInfo | URL,
  init?: RequestInit & {
    cache?: RequestCache;
    next?: {
      revalidate?: number | false;
      tags?: string[];
    };
  }
): Promise<Response>;

type RequestCache =
  | "default"
  | "force-cache"
  | "no-cache"
  | "no-store"
  | "only-if-cached"
  | "reload";
```

**缓存策略对比**:

| 策略        | 行为           | Next.js 15 | Next.js 16    | 使用场景       |
| :---------- | :------------- | :--------- | :------------ | :------------- |
| force-cache | 强制缓存       | 默认       | 需显式指定 ⚠️ | 静态数据       |
| no-store    | 不缓存         | 需显式指定 | 默认 ⚠️       | 动态数据       |
| no-cache    | 验证后使用缓存 | -          | -             | 需要验证的数据 |
| reload      | 总是重新获取   | -          | -             | 强制刷新       |

**使用示例**:

```typescript
// ⚠️ Next.js 16 变更: 默认不缓存
const res = await fetch("https://api.example.com/data");
// 等同于
const res = await fetch("https://api.example.com/data", {
  cache: "no-store",
});

// 强制缓存
const res = await fetch("https://api.example.com/data", {
  cache: "force-cache",
});

// 使用 revalidate(推荐)
const res = await fetch("https://api.example.com/data", {
  next: { revalidate: 3600 }, // 1 小时后重新验证
});

// 永久缓存
const res = await fetch("https://api.example.com/data", {
  next: { revalidate: false },
});

// 使用缓存标签
const res = await fetch("https://api.example.com/products", {
  next: {
    revalidate: 3600,
    tags: ["products"],
  },
});

// 不缓存(显式)
const res = await fetch("https://api.example.com/realtime", {
  cache: "no-store",
});
```

**revalidate 时间建议**:

| 数据类型 | revalidate 值 | 说明                   |
| :------- | :------------ | :--------------------- |
| 实时数据 | 0 或 no-store | 每次请求都获取最新数据 |
| 频繁更新 | 60-300        | 1-5 分钟               |
| 一般数据 | 3600          | 1 小时                 |
| 稳定数据 | 86400         | 1 天                   |
| 静态数据 | false         | 永久缓存               |

**注意事项**:

- ⚠️ Next.js 16 变更: 默认缓存策略从 `force-cache` 改为 `no-store`
- `revalidate: false` 表示永久缓存,直到手动重新验证
- `tags` 用于按需重新验证
- 只在服务端组件中有效

**适用场景**:

- API 数据获取
- 外部资源请求
- 需要缓存控制的 HTTP 请求

**相关章节**: 03-01-fetch API 在 Next.js 16 中的扩展.md

### 3.2 next.revalidate 选项

**功能**: 设置页面或布局的重新验证时间

**类型签名**:

```typescript
// 在 page.tsx 或 layout.tsx 中导出
export const revalidate = number | false;
```

**使用示例**:

```typescript
// app/blog/page.tsx
// 每 1 小时重新验证一次
export const revalidate = 3600;

export default async function BlogPage() {
  const posts = await getPosts();

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}

// app/static/page.tsx
// 永久缓存
export const revalidate = false;

export default async function StaticPage() {
  const content = await getStaticContent();

  return <div>{content}</div>;
}

// app/realtime/page.tsx
// 不缓存
export const revalidate = 0;

export default async function RealtimePage() {
  const data = await getRealtimeData();

  return <div>{data}</div>;
}
```

**与 fetch revalidate 的关系**:

```typescript
// 页面级 revalidate
export const revalidate = 3600;

export default async function Page() {
  // 这个请求会使用页面级的 revalidate(3600)
  const data1 = await fetch("https://api.example.com/data1");

  // 这个请求会覆盖页面级的 revalidate
  const data2 = await fetch("https://api.example.com/data2", {
    next: { revalidate: 60 },
  });

  // 最终页面的 revalidate 是所有请求中最小的值(60)
  return (
    <div>
      {data1} {data2}
    </div>
  );
}
```

**注意事项**:

- 页面级 revalidate 会影响所有 fetch 请求
- 单个 fetch 可以覆盖页面级设置
- 最终的 revalidate 是所有值中最小的

**适用场景**:

- 整个页面需要统一的缓存策略
- 静态生成页面(SSG)
- 增量静态再生(ISR)

## 4. 路径级缓存 API (Path-Level Caching)

### 4.1 revalidatePath 函数

**功能**: 按需重新验证指定路径的缓存

**类型签名**:

```typescript
function revalidatePath(path: string, type?: "page" | "layout"): void;
```

**使用示例**:

```typescript
import { revalidatePath } from "next/cache";

// 在 Server Action 中使用
async function createPost(formData: FormData) {
  "use server";

  const post = await db.post.create({
    data: {
      title: formData.get("title"),
      content: formData.get("content"),
    },
  });

  // 重新验证博客列表页
  revalidatePath("/blog");

  // 重新验证特定文章页
  revalidatePath(`/blog/${post.slug}`);
}

// 重新验证布局及其所有子页面
async function updateSiteSettings(data: any) {
  "use server";

  await db.settings.update({ data });

  // 重新验证整个布局
  revalidatePath("/", "layout");
}

// 在 Route Handler 中使用
export async function POST(request: Request) {
  const data = await request.json();

  await updateData(data);

  revalidatePath("/dashboard");

  return Response.json({ success: true });
}
```

**type 参数说明**:

| type          | 行为                       | 使用场景     |
| :------------ | :------------------------- | :----------- |
| 'page' (默认) | 只重新验证指定页面         | 单个页面更新 |
| 'layout'      | 重新验证布局及其所有子页面 | 全局数据更新 |

**路径匹配规则**:

```typescript
// 精确匹配
revalidatePath("/blog"); // 只匹配 /blog
revalidatePath("/blog/post-1"); // 只匹配 /blog/post-1

// 使用 layout 类型匹配所有子路径
revalidatePath("/blog", "layout"); // 匹配 /blog 及其所有子路径
```

**注意事项**:

- 只能在 Server Actions 和 Route Handlers 中使用
- 不能在服务端组件中直接调用
- 路径必须是绝对路径(以 / 开头)
- 重新验证是异步的,不会立即生效

**适用场景**:

- 数据更新后刷新页面
- 表单提交后更新列表
- CMS 内容发布后更新页面

**相关章节**: 03-15-revalidatePath 函数详解.md

### 4.2 revalidateTag 函数

**功能**: 按需重新验证带有指定标签的所有缓存

**类型签名**:

```typescript
function revalidateTag(tag: string): void;
```

**使用示例**:

```typescript
import { revalidateTag } from "next/cache";

// 基础用法
async function updateProduct(id: string, data: any) {
  "use server";

  await db.product.update({ where: { id }, data });

  // 重新验证所有产品相关的缓存
  revalidateTag("products");
}

// 多标签重新验证
async function updateCategory(id: string, data: any) {
  "use server";

  await db.category.update({ where: { id }, data });

  // 重新验证分类和产品缓存
  revalidateTag("categories");
  revalidateTag("products");
}

// 精确重新验证
async function deleteProduct(id: string) {
  "use server";

  await db.product.delete({ where: { id } });

  // 重新验证所有产品缓存
  revalidateTag("products");

  // 重新验证特定产品缓存
  revalidateTag(`product-${id}`);
}
```

**与 revalidatePath 的对比**:

| 特性       | revalidatePath | revalidateTag |
| :--------- | :------------- | :------------ |
| 粒度       | 基于路径       | 基于标签      |
| 灵活性     | 较低           | 较高          |
| 使用复杂度 | 简单           | 需要提前标记  |
| 适用场景   | 页面级更新     | 数据级更新    |

**最佳实践**:

```typescript
// 组合使用 cacheTag 和 revalidateTag

// 1. 在数据获取时添加标签
export async function getProducts() {
  "use cache";
  cacheTag("products", "catalog");

  return await db.product.findMany();
}

export async function getProduct(id: string) {
  "use cache";
  cacheTag("products", `product-${id}`);

  return await db.product.findUnique({ where: { id } });
}

// 2. 在数据更新时重新验证
async function updateProduct(id: string, data: any) {
  "use server";

  await db.product.update({ where: { id }, data });

  // 重新验证所有产品列表
  revalidateTag("products");

  // 重新验证特定产品
  revalidateTag(`product-${id}`);
}

async function deleteProduct(id: string) {
  "use server";

  await db.product.delete({ where: { id } });

  // 重新验证整个目录
  revalidateTag("catalog");
}
```

**注意事项**:

- 只能在 Server Actions 和 Route Handlers 中使用
- 标签名称区分大小写
- 会重新验证所有带有该标签的缓存条目
- 比 `revalidatePath` 更灵活,但需要提前规划标签

**适用场景**:

- 数据驱动的缓存更新
- 跨页面的数据更新
- 复杂的缓存依赖关系

**相关章节**: 03-16-revalidateTag 函数详解.md, 07-01-revalidateTag 实际应用场景.md

## 5. 函数级缓存 API (Function-Level Caching)

### 5.1 unstable_cache 函数

**功能**: 缓存函数的返回值

**类型签名**:

```typescript
function unstable_cache<T>(
  fn: () => Promise<T>,
  keyParts?: string[],
  options?: {
    revalidate?: number | false;
    tags?: string[];
  }
): () => Promise<T>;
```

**使用示例**:

```typescript
import { unstable_cache } from "next/cache";

// 基础用法
const getCachedUser = unstable_cache(
  async (id: string) => {
    return await db.user.findUnique({ where: { id } });
  },
  ["user"],
  { revalidate: 3600 }
);

// 使用标签
const getCachedProducts = unstable_cache(
  async () => {
    return await db.product.findMany();
  },
  ["products"],
  {
    revalidate: 3600,
    tags: ["products"],
  }
);

// 动态键
const getCachedPost = unstable_cache(
  async (slug: string) => {
    return await db.post.findUnique({ where: { slug } });
  },
  ["post"],
  { revalidate: 3600 }
);

// 在组件中使用
export default async function UserProfile({ id }: { id: string }) {
  const user = await getCachedUser(id);

  return <div>{user.name}</div>;
}
```

**keyParts 说明**:

```typescript
// keyParts 用于生成缓存键
// 缓存键 = keyParts + 函数参数

// 示例 1: 固定键
unstable_cache(
  async () => getAllUsers(),
  ["users"] // 缓存键: users
);

// 示例 2: 动态键
unstable_cache(
  async (id: string) => getUser(id),
  ["user"] // 缓存键: user:${id}
);

// 示例 3: 多层键
unstable_cache(
  async (category: string, page: number) => getProducts(category, page),
  ["products", "list"] // 缓存键: products:list:${category}:${page}
);
```

**与 'use cache' 的对比**:

| 特性       | unstable_cache | 'use cache' |
| :--------- | :------------- | :---------- |
| 稳定性     | 实验性         | 🆕 稳定     |
| 使用方式   | 函数包装器     | 指令        |
| 灵活性     | 高             | 中          |
| 代码侵入性 | 低             | 低          |
| 推荐使用   | 特殊场景       | 一般场景    |

**注意事项**:

- 标记为 `unstable_`,可能在未来版本变更
- 需要手动管理缓存键
- 适用于需要精细控制的场景
- 推荐优先使用 `'use cache'` 指令

**适用场景**:

- 需要精确控制缓存键的场景
- 第三方库函数的缓存
- 复杂的缓存逻辑

## 6. 高级缓存 API (Advanced Caching)

### 6.1 🆕 after 函数

**功能**: 在响应流式传输完成后执行任务

**类型签名**:

```typescript
function after(callback: () => void | Promise<void>): void;
```

**使用示例**:

```typescript
import { after } from "next/server";

// 记录日志
export default async function Page() {
  const data = await getData();

  after(() => {
    console.log("Page rendered");
  });

  return <div>{data}</div>;
}

// 异步任务
export default async function Page() {
  const data = await getData();

  after(async () => {
    await logAnalytics({
      page: "/dashboard",
      timestamp: Date.now(),
    });
  });

  return <div>{data}</div>;
}

// 缓存预热
export default async function ProductPage({
  params,
}: {
  params: { id: string };
}) {
  const product = await getProduct(params.id);

  // 预热相关产品的缓存
  after(async () => {
    const relatedIds = product.relatedProducts;
    await Promise.all(relatedIds.map((id) => getProduct(id)));
  });

  return <ProductDetail product={product} />;
}
```

**使用场景**:

| 缓存预热 | 预加载相关数据   | 相关产品、推荐内容   |
| 清理任务 | 清理临时数据     | 过期缓存、临时文件   |

**注意事项**:

- 🆕 Next.js 16 新增功能
- 不会阻塞响应
- 回调函数中的错误不会影响响应
- 适用于非关键任务

**适用场景**:

- 不影响用户体验的后台任务

## 8. 缓存策略选择指南 (Cache Strategy Guide)

### 8.1 按数据特性选择

**静态数据** (很少变化):

- 使用 `'use cache'` + `cacheLife('max')`
- 或 `export const revalidate = false`
- 示例: 已发布的文章、产品说明

**准静态数据** (定期更新):

- 使用 `'use cache'` + `cacheLife('days')` 或 `cacheLife('hours')`
- 或 `export const revalidate = 86400`
- 示例: 产品列表、分类信息

**动态数据** (频繁更新):

- 使用 `'use cache'` + `cacheLife('minutes')`
- 或 `export const revalidate = 60`
- 示例: 新闻列表、评论

**实时数据** (每次都要最新):

- 使用 `unstable_noStore()`
- 或 `export const dynamic = 'force-dynamic'`
- 示例: 股票价格、在线用户数

### 8.2 按更新方式选择

**定时更新**:

- 使用 `revalidate` 选项
- 适用于可预测的更新周期

**按需更新**:

- 使用 `cacheTag` + `revalidateTag`
- 适用于数据驱动的更新

**路径更新**:

- 使用 `revalidatePath`
- 适用于页面级更新

### 8.3 缓存层次选择

| 层次   | API            | 粒度 | 灵活性 | 使用场景 |
| :----- | :------------- | :--- | :----- | :------- |
| 组件级 | 'use cache'    | 细   | 高     | 组件缓存 |
| 请求级 | fetch options  | 中   | 中     | API 请求 |
| 路径级 | revalidatePath | 粗   | 低     | 页面更新 |
| 标签级 | cacheTag       | 中   | 高     | 数据更新 |

## 10. 注意事项 (Precautions)

### 10.1 Next.js 16 迁移注意事项

**fetch 默认行为变更**:

```typescript
// ❌ Next.js 15: 默认缓存
const res = await fetch("https://api.example.com/data");

// ✅ Next.js 16: 默认不缓存,需要显式指定
const res = await fetch("https://api.example.com/data", {
  next: { revalidate: 3600 },
});
```

**异步 API 变更**:

```typescript
// ❌ Next.js 15
const headersList = headers();
const cookieStore = cookies();

// ✅ Next.js 16
const headersList = await headers();
const cookieStore = await cookies();
```

### 10.2 缓存失效问题

**问题**: 缓存没有按预期更新

**可能原因**:

1. 标签名称拼写错误
2. revalidate 时间设置过长
3. 缓存键冲突

**解决方案**:

```typescript
// 1. 使用常量管理标签名称
const CACHE_TAGS = {
  PRODUCTS: "products",
  PRODUCT: (id: string) => `product-${id}`,
};

// 2. 合理设置 revalidate 时间
cacheLife({
  stale: 60,
  revalidate: 300,
  expire: 3600,
});

// 3. 使用唯一的缓存键
unstable_cache(
  async (id: string) => getProduct(id),
  ["product", "detail"], // 添加更多层次
  { tags: [CACHE_TAGS.PRODUCT(id)] }
);
```

### 10.3 性能考虑

**过度缓存**:

- 问题: 缓存太多数据导致内存占用过高
- 解决: 合理设置 expire 时间,定期清理过期缓存

**缓存粒度**:

- 问题: 缓存粒度太粗导致频繁失效
- 解决: 使用细粒度标签,精确控制缓存失效

**缓存预热**:

- 问题: 首次访问慢
- 解决: 使用 `after` 函数预热相关缓存

## 11. 常见问题 (FAQ)

### 11.1 'use cache' 和 unstable_cache 如何选择?

**推荐使用 'use cache'**:

- 🆕 Next.js 16 稳定功能
- 语法简洁,易于使用
- 与 cacheLife 和 cacheTag 配合良好

**使用 unstable_cache 的场景**:

- 需要精确控制缓存键
- 包装第三方库函数
- 需要更灵活的缓存控制

### 11.3 缓存在什么时候会失效?

2. 服务器重启
3. 部署新版本

**手动失效**:

1. 调用 `revalidatePath`
2. 调用 `revalidateTag`
3. 使用 `revalidate: 0`
