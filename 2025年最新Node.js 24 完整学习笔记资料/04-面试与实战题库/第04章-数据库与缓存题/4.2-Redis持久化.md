## 📚 核心知识点

### 1. 持久化方案对比

| 特性 | RDB（快照） | AOF（日志） | 混合持久化 |
|------|-----------|-----------|----------|
| **持久化方式** | 全量快照 | 增量日志 | RDB + AOF |
| **文件大小** | 小（压缩） | 大（每条命令） | 中等 |
| **恢复速度** | 快 | 慢 | 快 |
| **数据完整性** | 低（丢失间隔数据） | 高（最多丢 1 秒） | 高 |
| **性能影响** | 低（异步） | 中（同步写盘） | 中等 |
| **适用场景** | 备份、灾难恢复 | 数据安全第一 | 推荐方案 |

## 💻 RDB 持久化详解

### 1. RDB 工作原理

**触发方式：**
```bash
# 方式 1：手动触发
127.0.0.1:6379> SAVE      # 同步（阻塞主线程）
127.0.0.1:6379> BGSAVE    # 异步（fork 子进程）

# 方式 2：自动触发（配置）
# redis.conf
save 900 1      # 900 秒内至少 1 次修改
save 300 10     # 300 秒内至少 10 次修改
save 60 10000   # 60 秒内至少 10000 次修改
```

**生成流程（BGSAVE）：**
```
1. Redis 主进程 fork 子进程（使用 COW）
2. 子进程将数据写入临时 RDB 文件
3. 完成后，替换旧的 RDB 文件
4. 主进程继续处理请求（不阻塞）

┌─────────┐
│ 主进程   │ ← 继续处理请求
└────┬────┘
     │ fork（Copy-On-Write）
     ↓
┌─────────┐
│ 子进程   │ → 写 RDB 文件（dump.rdb）
└─────────┘
```

**COW（Copy-On-Write）机制：**
```
1. fork 时，子进程共享主进程的内存页
2. 主进程修改数据时，才复制该内存页
3. 优势：fork 快，内存占用少
4. 劣势：数据修改多时，内存占用翻倍
```

---

### 2. RDB 优缺点

**优点：**
- ✅ **恢复快**：直接加载 RDB 文件（大数据集：< 1 秒）
- ✅ **文件小**：压缩存储，节省磁盘
- ✅ **性能高**：子进程执行，主进程不阻塞

**缺点：**
- ❌ **数据丢失**：间隔保存，崩溃时丢失间隔数据
  ```
  00:00 → BGSAVE（保存）
  00:05 → 崩溃（丢失 5 分钟数据）
  ```
- ❌ **fork 开销**：数据量大时，fork 耗时（几百毫秒）
- ❌ **CPU 密集**：压缩、序列化消耗 CPU

---

### 3. RDB 配置优化

```bash
# redis.conf

# 自动触发（建议关闭，手动触发）
# save 900 1
# save 300 10
# save 60 10000

# RDB 文件名
dbfilename dump.rdb

# 保存路径
dir /var/lib/redis

# 压缩（默认开启）
rdbcompression yes

# 校验和（默认开启）
rdbchecksum yes

# 失败时停止写入（默认开启）
stop-writes-on-bgsave-error yes
```

## 💻 AOF 持久化详解

### 1. AOF 工作原理

**记录方式：**
```bash
# 每次写命令都追加到 AOF 文件
SET key1 value1
SET key2 value2
DEL key1

# AOF 文件内容（Redis 协议格式）
*3\r\n$3\r\nSET\r\n$4\r\nkey1\r\n$6\r\nvalue1\r\n
*3\r\n$3\r\nSET\r\n$4\r\nkey2\r\n$6\r\nvalue2\r\n
*2\r\n$3\r\nDEL\r\n$4\r\nkey1\r\n
```

**同步策略（appendfsync）：**
```bash
# 方式 1：每次写入都同步（最安全，性能最差）
appendfsync always

# 方式 2：每秒同步一次（推荐，平衡性能和安全）
appendfsync everysec

# 方式 3：操作系统决定（性能最好，可能丢失更多数据）
appendfsync no
```

**数据丢失对比：**
```
appendfsync always   → 最多丢失 1 条命令
appendfsync everysec → 最多丢失 1 秒数据
appendfsync no       → 最多丢失几十秒数据
```

---

### 2. AOF 重写

**为什么要重写？**
```bash
# 问题：AOF 文件会越来越大
SET key value1
SET key value2
SET key value3
...
SET key value1000000  # 只有最后一条有效

# 解决：AOF 重写（合并命令）
SET key value1000000  # 只保留最终状态
```

**重写流程：**
```
1. Redis 主进程 fork 子进程
2. 子进程遍历数据库，生成新 AOF 文件
3. 主进程继续写旧 AOF，同时记录增量命令到缓冲区
4. 子进程完成后，主进程将缓冲区命令追加到新 AOF
5. 替换旧 AOF 文件

┌─────────┐
│ 主进程   │ → 旧 AOF + 缓冲区
└────┬────┘
     │ fork
     ↓
┌─────────┐
│ 子进程   │ → 新 AOF（重写）
└─────────┘
```

**触发方式：**
```bash
# 手动触发
127.0.0.1:6379> BGREWRITEAOF

# 自动触发（配置）
auto-aof-rewrite-percentage 100  # 大小翻倍时触发
auto-aof-rewrite-min-size 64mb   # 至少 64MB 才触发
```

---

### 3. AOF 优缺点

**优点：**
- ✅ **数据安全**：最多丢失 1 秒数据（everysec）
- ✅ **可读性好**：文本格式，便于修复
- ✅ **自动重写**：控制文件大小

**缺点：**
- ❌ **文件大**：记录每条命令，占用空间
- ❌ **恢复慢**：重放所有命令（大数据集：几分钟）
- ❌ **性能差**：同步写盘，影响吞吐量

---

### 4. AOF 配置优化

```bash
# redis.conf

# 开启 AOF
appendonly yes

# AOF 文件名
appendfilename "appendonly.aof"

# 同步策略（推荐 everysec）
appendfsync everysec

# 重写配置
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 重写时不执行 fsync（避免阻塞）
no-appendfsync-on-rewrite yes
```

## 💻 混合持久化（Redis 4.0+）

### 1. 混合持久化原理

**结合 RDB 和 AOF 优势：**
```
AOF 文件结构：
┌─────────────┬──────────────┐
│ RDB 格式数据 │ AOF 格式增量  │
│ （快照）     │ （命令）      │
└─────────────┴──────────────┘

优势：
- 恢复快（RDB 快速加载）
- 数据完整（AOF 增量补充）
```

**配置：**
```bash
# redis.conf
aof-use-rdb-preamble yes  # 开启混合持久化
```

**重写流程：**
```
1. 子进程生成 RDB 格式的数据（快照）
2. 将 RDB 数据写入 AOF 文件头部
3. 将主进程的增量命令追加到 AOF 文件尾部
4. 最终 AOF 文件 = RDB 快照 + AOF 增量
```

## 💻 实战案例

### 案例 1：性能对比测试

```javascript
const Redis = require('ioredis');
const redis = new Redis();

// 测试 RDB 性能
console.time('RDB Save');
await redis.bgsave();
// 等待 BGSAVE 完成
while (await redis.lastsave() === lastSaveTime) {
  await new Promise(resolve => setTimeout(resolve, 100));
}
console.timeEnd('RDB Save');  // ~500ms（10 万条数据）

// 测试 AOF 性能（everysec）
await redis.config('SET', 'appendfsync', 'everysec');
console.time('AOF Write');
for (let i = 0; i < 10000; i++) {
  await redis.set(`key${i}`, `value${i}`);
}
console.timeEnd('AOF Write');  // ~2s（10000 条命令）

// 测试恢复速度
console.time('RDB Load');
await redis.shutdown();
await redis.connect();
console.timeEnd('RDB Load');  // ~100ms

console.time('AOF Load');
await redis.shutdown();
await redis.connect();
console.timeEnd('AOF Load');  // ~3s
```

### 案例 2：高可用配置（主从 + AOF）

```bash
# 主节点配置（master.conf）
port 6379

# RDB（每小时备份一次）
save 3600 1
dbfilename dump-master.rdb

# AOF（数据安全）
appendonly yes
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 混合持久化
aof-use-rdb-preamble yes

# 从节点配置（slave.conf）
port 6380
replicaof 127.0.0.1 6379

# 从节点也开启 AOF（双重保险）
appendonly yes
appendfsync everysec
```

### 案例 3：灾难恢复演练

```bash
# 场景：主节点数据损坏，需要从备份恢复

# 步骤 1：停止 Redis
redis-cli shutdown

# 步骤 2：删除损坏的数据文件
rm /var/lib/redis/dump.rdb

# 步骤 3：从备份恢复 RDB
cp /backup/dump.rdb /var/lib/redis/

# 步骤 4：启动 Redis（加载 RDB）
redis-server /etc/redis/redis.conf

# 步骤 5：验证数据
redis-cli
127.0.0.1:6379> DBSIZE
(integer) 100000

# 步骤 6：重新开启 AOF
127.0.0.1:6379> CONFIG SET appendonly yes
127.0.0.1:6379> BGREWRITEAOF
```

## 🎓 面试标准答题

### 问题 1："RDB 和 AOF 的区别是什么？各自的优缺点？"

**标准答案：**
> **核心区别：**
> 
> | 特性 | RDB | AOF |
> |------|-----|-----|
> | **方式** | 全量快照 | 增量日志 |
> | **文件** | 小（压缩） | 大（每条命令） |
> | **恢复** | 快（直接加载） | 慢（重放命令） |
> | **安全** | 低（丢失间隔数据） | 高（最多丢 1 秒） |
> | **性能** | 高（异步） | 中（同步写盘） |
> 
> **RDB 优缺点：**
> - ✅ 恢复快、文件小、性能高
> - ❌ 数据丢失多、fork 开销大
> 
> **AOF 优缺点：**
> - ✅ 数据安全、可读性好、自动重写
> - ❌ 文件大、恢复慢、性能差
> 
> **选择建议：**
> - 数据安全第一 → AOF（everysec）
> - 性能第一 → RDB
> - **推荐 → 混合持久化（RDB + AOF）**

**加分项：**
> 在实际项目中，我们使用混合持久化：
> - 主节点：AOF（数据安全）
> - 从节点：RDB（备份快）
> - 定时任务：凌晨 3 点 BGSAVE（离线备份）

---

### 问题 2："AOF 重写的原理是什么？如何触发？"

**标准答案：**
> **AOF 重写原理：**
> 
> **1. 为什么需要重写？**
> ```bash
> # 问题：重复命令导致 AOF 文件膨胀
> SET key value1
> SET key value2
> ...
> SET key value1000  # 只有最后一条有效
> 
> # 解决：重写后只保留最终状态
> SET key value1000
> ```
> 
> **2. 重写流程：**
> ```
> ① fork 子进程
> ② 子进程遍历数据库，生成新 AOF
> ③ 主进程继续写旧 AOF + 缓冲区
> ④ 子进程完成后，追加缓冲区到新 AOF
> ⑤ 替换旧 AOF
> ```
> 
> **3. 触发方式：**
> ```bash
> # 手动触发
> BGREWRITEAOF
> 
> # 自动触发（配置）
> auto-aof-rewrite-percentage 100  # 大小翻倍
> auto-aof-rewrite-min-size 64mb   # 至少 64MB
> ```
> 
> **4. 注意事项：**
> - 重写期间，主进程仍可写入
> - 重写不阻塞主线程（异步）
> - 重写期间崩溃，旧 AOF 仍可用
