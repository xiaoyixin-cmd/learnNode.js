## 📚 核心知识点

### 1. 四种事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 | 性能 |
|---------|------|-----------|------|---------|------|
| **READ UNCOMMITTED（RU）** | ✅ 可能 | ✅ 可能 | ✅ 可能 | 无锁 | 最高 |
| **READ COMMITTED（RC）** | ❌ 不会 | ✅ 可能 | ✅ 可能 | MVCC | 高 |
| **REPEATABLE READ（RR）** | ❌ 不会 | ❌ 不会 | ⚠️ 部分 | MVCC + 间隙锁 | 中 |
| **SERIALIZABLE（S）** | ❌ 不会 | ❌ 不会 | ❌ 不会 | 锁 | 最低 |

**MySQL 默认：REPEATABLE READ（InnoDB）**

---

### 2. 并发问题定义

#### 脏读（Dirty Read）
> 读取到其他事务**未提交**的数据

```sql
-- 事务 A
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE id = 1;
-- 未提交

-- 事务 B（脏读）
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到 500（未提交的数据）
COMMIT;

-- 事务 A 回滚
ROLLBACK;  -- balance 恢复为原值

-- 问题：事务 B 读到了不存在的数据（脏读）
```

---

#### 不可重复读（Non-Repeatable Read）
> 同一事务内**两次读取**同一行数据，结果不一致

```sql
-- 事务 A
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到 100

-- 事务 B（修改并提交）
UPDATE accounts SET balance = 200 WHERE id = 1;
COMMIT;

-- 事务 A 再次读取
SELECT balance FROM accounts WHERE id = 1;  -- 读到 200

-- 问题：同一事务内，两次读取结果不一致（不可重复读）
COMMIT;
```

---

#### 幻读（Phantom Read）
> 同一事务内**两次查询**，结果集的**行数**不一致

```sql
-- 事务 A
START TRANSACTION;
SELECT COUNT(*) FROM accounts WHERE balance > 100;  -- 查到 5 条

-- 事务 B（插入新数据）
INSERT INTO accounts (id, balance) VALUES (10, 150);
COMMIT;

-- 事务 A 再次查询
SELECT COUNT(*) FROM accounts WHERE balance > 100;  -- 查到 6 条

-- 问题：同一事务内，两次查询结果集不一致（幻读）
COMMIT;
```

## 💻 隔离级别详解

### 1. READ UNCOMMITTED（读未提交）

**特点：** 无锁，允许脏读

```sql
-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 事务 A
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE id = 1;
-- 未提交

-- 事务 B（脏读）
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 500（未提交的数据）
COMMIT;

-- 事务 A 回滚
ROLLBACK;  -- balance 恢复原值

-- 问题：事务 B 读到了脏数据
```

**使用场景：** 几乎不用（数据一致性太差）

---

### 2. READ COMMITTED（读已提交）

**特点：** 使用 MVCC，每次读取最新的已提交数据

```sql
-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务 A
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 100

-- 事务 B（修改并提交）
UPDATE accounts SET balance = 200 WHERE id = 1;
COMMIT;

-- 事务 A 再次读取
SELECT balance FROM accounts WHERE id = 1;  -- 200（不可重复读）
COMMIT;
```

**解决：** 脏读  
**未解决：** 不可重复读、幻读  
**使用场景：** Oracle、PostgreSQL 默认隔离级别

---

### 3. REPEATABLE READ（可重复读）

**特点：** MVCC + 间隙锁，保证同一事务内读取一致

```sql
-- 设置隔离级别（MySQL 默认）
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务 A
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 100

-- 事务 B（修改并提交）
UPDATE accounts SET balance = 200 WHERE id = 1;
COMMIT;

-- 事务 A 再次读取
SELECT balance FROM accounts WHERE id = 1;  -- 100（可重复读）
COMMIT;
```

**解决：** 脏读、不可重复读  
**部分解决：** 幻读（通过间隙锁）  
**使用场景：** MySQL/InnoDB 默认

---

### 4. SERIALIZABLE（可串行化）

**特点：** 强制事务串行执行，最安全但性能最差

```sql
-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务 A
START TRANSACTION;
SELECT * FROM accounts WHERE balance > 100;  -- 加锁

-- 事务 B（尝试插入）
INSERT INTO accounts (id, balance) VALUES (10, 150);  -- 阻塞，等待 A 提交

-- 事务 A 提交
COMMIT;

-- 事务 B 继续执行
-- 插入成功
```

**解决：** 所有并发问题  
**使用场景：** 对数据一致性要求极高的场景

## 💻 MVCC（多版本并发控制）

### 1. MVCC 原理

**核心思想：** 为每行数据维护多个版本，读取时选择合适的版本

**隐藏字段：**
```sql
CREATE TABLE accounts (
  id INT PRIMARY KEY,
  balance INT,
  DB_TRX_ID BIGINT,    -- 事务 ID（隐藏）
  DB_ROLL_PTR BIGINT   -- 回滚指针（隐藏）
);
```

**Undo Log 版本链：**
```
当前版本：balance = 200, TRX_ID = 103
     ↓ (ROLL_PTR)
历史版本：balance = 150, TRX_ID = 102
     ↓
历史版本：balance = 100, TRX_ID = 101
```

---

### 2. Read View（读视图）

**Read View 包含：**
- `m_ids`：当前活跃事务 ID 列表
- `min_trx_id`：最小活跃事务 ID
- `max_trx_id`：下一个分配的事务 ID
- `creator_trx_id`：创建 Read View 的事务 ID

**可见性判断：**
```python
def is_visible(row_trx_id, read_view):
    # 1. 数据版本由当前事务创建 → 可见
    if row_trx_id == read_view.creator_trx_id:
        return True
    
    # 2. 数据版本在 Read View 创建前已提交 → 可见
    if row_trx_id < read_view.min_trx_id:
        return True
    
    # 3. 数据版本在 Read View 创建后创建 → 不可见
    if row_trx_id >= read_view.max_trx_id:
        return False
    
    # 4. 数据版本在活跃事务列表中 → 不可见
    if row_trx_id in read_view.m_ids:
        return False
    
    # 5. 其他情况 → 可见
    return True
```

---

### 3. RC vs RR 的 Read View 差异

**RC（读已提交）：** 每次读取都生成新的 Read View
```sql
-- 事务 A
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1;  -- 生成 Read View 1

-- 事务 B 提交
UPDATE accounts SET balance = 200 WHERE id = 1;
COMMIT;

-- 事务 A 再次读取
SELECT * FROM accounts WHERE id = 1;  -- 生成 Read View 2（读到 200）
```

**RR（可重复读）：** 事务开始时生成 Read View，后续复用
```sql
-- 事务 A
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1;  -- 生成 Read View（TRX_ID = 101）

-- 事务 B 提交
UPDATE accounts SET balance = 200 WHERE id = 1;
COMMIT;

-- 事务 A 再次读取
SELECT * FROM accounts WHERE id = 1;  -- 复用 Read View（读到 100）
```

## 💻 幻读与间隙锁

### 1. 当前读 vs 快照读

**快照读（Snapshot Read）：** 读取 MVCC 版本链中的数据
```sql
SELECT * FROM accounts WHERE id = 1;  -- 快照读（不加锁）
```

**当前读（Current Read）：** 读取最新版本，加锁
```sql
SELECT * FROM accounts WHERE id = 1 LOCK IN SHARE MODE;  -- 当前读（加共享锁）
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;           -- 当前读（加排他锁）
UPDATE / DELETE / INSERT                                  -- 当前读
```

---

### 2. 间隙锁（Gap Lock）

**问题：** RR 隔离级别下，快照读不会幻读，但当前读会

```sql
-- 事务 A（快照读）
START TRANSACTION;
SELECT * FROM accounts WHERE id BETWEEN 1 AND 10;  -- 快照读，不加锁

-- 事务 B 插入
INSERT INTO accounts (id, balance) VALUES (5, 100);
COMMIT;

-- 事务 A 再次读取
SELECT * FROM accounts WHERE id BETWEEN 1 AND 10;  -- 快照读，不会读到新数据

-- 事务 A（当前读）
SELECT * FROM accounts WHERE id BETWEEN 1 AND 10 FOR UPDATE;  -- 当前读，会读到新数据（幻读）
```

**解决：间隙锁**
```sql
-- 事务 A（当前读）
START TRANSACTION;
SELECT * FROM accounts WHERE id BETWEEN 1 AND 10 FOR UPDATE;  -- 加间隙锁

-- 事务 B 尝试插入
INSERT INTO accounts (id, balance) VALUES (5, 100);  -- 阻塞（间隙被锁定）
```

**锁定范围：**
```
索引值：... 1, 3, 7, 10, 15 ...

SELECT * FROM accounts WHERE id BETWEEN 1 AND 10 FOR UPDATE;

锁定间隙：
(−∞, 1], (1, 3], (3, 7], (7, 10], (10, 15)
```

## 💻 实战案例

### 案例 1：转账场景（避免脏读）

```javascript
const mysql = require('mysql2/promise');

const pool = mysql.createPool({
  host: 'localhost',
  user: 'root',
  password: 'password',
  database: 'bank'
});

async function transfer(fromId, toId, amount) {
  const connection = await pool.getConnection();
  
  try {
    // 设置隔离级别（至少 RC）
    await connection.query('SET TRANSACTION ISOLATION LEVEL READ COMMITTED');
    await connection.beginTransaction();
    
    // 扣款
    const [rows] = await connection.query(
      'SELECT balance FROM accounts WHERE id = ? FOR UPDATE',
      [fromId]
    );
    
    if (rows[0].balance < amount) {
      throw new Error('Insufficient balance');
    }
    
    await connection.query(
      'UPDATE accounts SET balance = balance - ? WHERE id = ?',
      [amount, fromId]
    );
    
    // 入账
    await connection.query(
      'UPDATE accounts SET balance = balance + ? WHERE id = ?',
      [amount, toId]
    );
    
    await connection.commit();
  } catch (error) {
    await connection.rollback();
    throw error;
  } finally {
    connection.release();
  }
}
```

### 案例 2：库存扣减（防止超卖）

```javascript
async function decreaseStock(productId, quantity) {
  const connection = await pool.getConnection();
  
  try {
    await connection.query('SET TRANSACTION ISOLATION LEVEL REPEATABLE READ');
    await connection.beginTransaction();
    
    // 当前读 + 排他锁
    const [rows] = await connection.query(
      'SELECT stock FROM products WHERE id = ? FOR UPDATE',
      [productId]
    );
    
    if (rows[0].stock < quantity) {
      throw new Error('Out of stock');
    }
    
    // 扣减库存
    await connection.query(
      'UPDATE products SET stock = stock - ? WHERE id = ?',
      [quantity, productId]
    );
    
    await connection.commit();
  } catch (error) {
    await connection.rollback();
    throw error;
  } finally {
    connection.release();
  }
}
```

## 🎓 面试标准答题

### 问题："MySQL 的四种事务隔离级别是什么？分别解决什么问题？"

**标准答案：**
> | 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现 | 场景 |
> |---------|------|-----------|------|------|------|
> | **RU** | ✅ | ✅ | ✅ | 无锁 | 几乎不用 |
> | **RC** | ❌ | ✅ | ✅ | MVCC | Oracle默认 |
> | **RR** | ❌ | ❌ | ⚠️ | MVCC+间隙锁 | MySQL默认 |
> | **S** | ❌ | ❌ | ❌ | 锁 | 强一致性 |
> 
> **详细说明：**
> 1. **RU（读未提交）**：可能读到脏数据，几乎不用
> 2. **RC（读已提交）**：解决脏读，每次读取生成新 Read View
> 3. **RR（可重复读）**：解决不可重复读，事务开始时生成 Read View
> 4. **S（可串行化）**：强制串行执行，性能最差
> 
> **MySQL 默认 RR 的原因：**
> - Binlog 主从复制的要求
> - 间隙锁防止幻读
> - 平衡性能和一致性
