# 1.2 setImmediate vs setTimeout

> 深入理解 setImmediate 和 setTimeout 的执行顺序差异

## 📚 核心知识点

### 1. 基础概念对比

| 特性 | setTimeout(fn, 0) | setImmediate(fn) |
|------|-------------------|------------------|
| **执行阶段** | timers 阶段 | check 阶段 |
| **最小延迟** | 1ms（Node.js）/ 4ms（浏览器） | 无延迟 |
| **执行时机** | 下一轮事件循环的 timers 阶段 | 当前事件循环的 check 阶段 |
| **浏览器支持** | ✅ 支持 | ❌ 不支持 |
| **适用场景** | 延迟执行 | I/O 回调后立即执行 |

---

### 2. 执行顺序：主脚本中

**在主脚本中，执行顺序不确定！**

```javascript
// main.js
setTimeout(() => {
  console.log('timeout');
}, 0);

setImmediate(() => {
  console.log('immediate');
});

// 可能的输出 1：
// timeout
// immediate

// 可能的输出 2：
// immediate
// timeout
```

**原因分析：**

1. **主脚本执行时间影响**：
   ```
   如果主脚本执行时间 < 1ms：
     → setTimeout 未到期
     → 先进入 poll 阶段
     → 发现有 setImmediate
     → 进入 check 阶段
     → 输出 immediate
     → 下一轮 timers 阶段
     → 输出 timeout
   
   如果主脚本执行时间 ≥ 1ms：
     → setTimeout 已到期
     → timers 阶段执行
     → 输出 timeout
     → poll 阶段
     → check 阶段
     → 输出 immediate
   ```

2. **系统负载影响**：
   - CPU 繁忙时，主脚本执行时间变长
   - setTimeout 更可能先到期

---

### 3. 执行顺序：I/O 回调中

**在 I/O 回调中，setImmediate 总是先执行！**

```javascript
const fs = require('fs');

fs.readFile(__filename, () => {
  setTimeout(() => {
    console.log('timeout');
  }, 0);
  
  setImmediate(() => {
    console.log('immediate');
  });
});

// 稳定输出：
// immediate
// timeout
```

**原因分析：**

```
fs.readFile 回调执行
   ↓
在 poll 阶段
   ↓
注册 setTimeout 和 setImmediate
   ↓
poll 阶段结束
   ↓
进入 check 阶段（setImmediate 执行）
   ↓
输出 immediate
   ↓
完成当前事件循环
   ↓
下一轮 timers 阶段（setTimeout 执行）
   ↓
输出 timeout
```

### 案例 2：I/O 回调中的确定性

```javascript
const fs = require('fs');

fs.readFile(__filename, () => {
  console.log('readFile callback');
  
  setTimeout(() => {
    console.log('timeout 1');
  }, 0);
  
  setTimeout(() => {
    console.log('timeout 2');
  }, 0);
  
  setImmediate(() => {
    console.log('immediate 1');
  });
  
  setImmediate(() => {
    console.log('immediate 2');
  });
});

// 100% 稳定输出：
// readFile callback
// immediate 1
// immediate 2
// timeout 1
// timeout 2
```

**原理：**
- readFile 回调在 poll 阶段执行
- poll 阶段后进入 check 阶段
- setImmediate 总是先执行
- 下一轮 timers 阶段执行 setTimeout

---

### 案例 3：嵌套调用

```javascript
setImmediate(() => {
  console.log('immediate 1');
  
  setImmediate(() => {
    console.log('immediate 2');
  });
  
  setTimeout(() => {
    console.log('timeout 1');
  }, 0);
});

setTimeout(() => {
  console.log('timeout 2');
  
  setImmediate(() => {
    console.log('immediate 3');
  });
}, 0);

// 可能的输出（主脚本执行时间 < 1ms）：
// immediate 1
// timeout 2
// immediate 3
// immediate 2
// timeout 1
```

**分析：**
1. 主脚本注册 setImmediate 和 setTimeout
2. 如果主脚本执行时间 < 1ms：
   - poll 阶段 → check 阶段
   - 输出 `immediate 1`
   - 注册 `immediate 2` 和 `timeout 1`
3. 下一轮 timers 阶段：
   - 输出 `timeout 2`
   - 注册 `immediate 3`
4. poll 阶段 → check 阶段：
   - 输出 `immediate 3`（当前轮）
   - 输出 `immediate 2`（上一轮注册的）
5. 下一轮 timers 阶段：
   - 输出 `timeout 1`

### 案例 4：确保执行顺序的技巧

```javascript
// ❌ 不确定顺序
setTimeout(() => console.log('1'), 0);
setImmediate(() => console.log('2'));

// ✅ 确保 setImmediate 先执行
setImmediate(() => {
  setTimeout(() => console.log('1'), 0);
  setImmediate(() => console.log('2'));
});

// 输出：
// 2
// 1

// ✅ 确保 setTimeout 先执行（延迟 > 0）
setTimeout(() => console.log('1'), 10);
setImmediate(() => console.log('2'));

// 多次运行仍可能不稳定，建议使用方案 1
```

## 🎓 面试标准答题

### 问题 1："setImmediate 和 setTimeout(fn, 0) 哪个先执行？"

**标准答案：**
> 这取决于**调用环境**：
> 
> **1. 在主脚本中：执行顺序不确定**
> - setTimeout(fn, 0) 的实际延迟是 **1ms**（Node.js）或 4ms（浏览器）
> - 如果主脚本执行时间 < 1ms，setImmediate 先执行
> - 如果主脚本执行时间 ≥ 1ms，setTimeout 先执行
> 
> **2. 在 I/O 回调中：setImmediate 总是先执行**
> - I/O 回调在 poll 阶段执行
> - poll 阶段后进入 check 阶段（setImmediate）
> - check 阶段后完成当前事件循环
> - 下一轮 timers 阶段才执行 setTimeout
> 
> **结论：** 如果需要确保执行顺序，应该在 I/O 回调中使用，或者使用 setImmediate 包裹。

**加分项：**
> 在实际项目中，我们在处理大文件时，使用 setImmediate 而不是 setTimeout，因为 setImmediate 在 I/O 回调中更快，且不受系统定时器精度影响。

---

### 问题 2："setTimeout(fn, 0) 的实际延迟是多少？"

**标准答案：**
> setTimeout(fn, 0) 的实际延迟**不是 0**：
> 
> **Node.js：最小 1ms**
> - 受系统定时器精度限制
> - 实际延迟通常在 1-10ms
> 
> **浏览器：最小 4ms**
> - HTML5 标准规定，嵌套 5 层以上的 setTimeout 最小延迟 4ms
> - 第 1-4 层可能是 0ms（取决于浏览器实现）
> 
> **影响因素：**
> 1. 事件循环当前阶段
> 2. 其他定时器的执行时间
> 3. 系统负载
> 
> **示例：**
> ```javascript
> console.time('setTimeout');
> setTimeout(() => {
>   console.timeEnd('setTimeout');
> }, 0);
> 
> // 输出：setTimeout: 1.234ms（实际延迟 > 0）
> ```

## ⚠️ 常见误区

### 误区 1：setTimeout(fn, 0) 立即执行

```javascript
// ❌ 错误理解
console.log('1');
setTimeout(() => console.log('2'), 0);  // 以为立即执行
console.log('3');

// 实际输出：
// 1
// 3
// 2（延迟执行）
```

---

### 误区 2：setImmediate 总是先执行

```javascript
// ❌ 错误理解：setImmediate 总是先执行
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// 实际：顺序不确定（主脚本中）
```

---

### 误区 3：浏览器支持 setImmediate

```javascript
// ❌ 错误：在浏览器中使用 setImmediate
setImmediate(() => {
  console.log('immediate');
});

// 报错：setImmediate is not defined

// ✅ 正确：浏览器中使用 setTimeout 或 queueMicrotask
setTimeout(() => console.log('timeout'), 0);
queueMicrotask(() => console.log('microtask'));
```

## 💡 实战优化技巧

### 技巧 1：避免阻塞事件循环

```javascript
// ❌ 错误：长时间阻塞
function processLargeArray(arr) {
  arr.forEach(item => {
    // 复杂计算
    heavyComputation(item);
  });
}

// ✅ 正确：使用 setImmediate 分批处理
function processLargeArray(arr) {
  let index = 0;
  const batchSize = 100;
  
  function processBatch() {
    const end = Math.min(index + batchSize, arr.length);
    
    for (let i = index; i < end; i++) {
      heavyComputation(arr[i]);
    }
    
    index = end;
    
    if (index < arr.length) {
      setImmediate(processBatch);  // 让出控制权
    }
  }
  
  processBatch();
}
```

### 技巧 2：在 I/O 回调中优先使用 setImmediate

```javascript
const fs = require('fs');

// ❌ 不推荐：使用 setTimeout
fs.readFile('./data.txt', (err, data) => {
  setTimeout(() => {
    processData(data);
  }, 0);
});

// ✅ 推荐：使用 setImmediate
fs.readFile('./data.txt', (err, data) => {
  setImmediate(() => {
    processData(data);
  });
});

// 优势：
// 1. setImmediate 更快（check 阶段 vs 下一轮 timers 阶段）
// 2. 不受定时器精度影响
// 3. 性能更好
```

### 技巧 3：Polyfill setImmediate（浏览器）

```javascript
// 浏览器中的 setImmediate polyfill
if (typeof setImmediate !== 'function') {
  window.setImmediate = function(callback) {
    return setTimeout(callback, 0);
  };
  
  window.clearImmediate = function(id) {
    clearTimeout(id);
  };
}

// 或使用 MessageChannel（性能更好）
if (typeof setImmediate !== 'function') {
  const channel = new MessageChannel();
  const callbacks = [];
  
  channel.port1.onmessage = () => {
    const callback = callbacks.shift();
    if (callback) callback();
  };
  
  window.setImmediate = function(callback) {
    callbacks.push(callback);
    channel.port2.postMessage(null);
  };
}
```

## 🔬 性能对比测试

### 测试 1：批处理性能

```javascript
const { performance } = require('perf_hooks');

// 测试 setTimeout
function testSetTimeout(count) {
  return new Promise((resolve) => {
    const start = performance.now();
    let i = 0;
    
    function next() {
      i++;
      if (i < count) {
        setTimeout(next, 0);
      } else {
        resolve(performance.now() - start);
      }
    }
    
    next();
  });
}

// 测试 setImmediate
function testSetImmediate(count) {
  return new Promise((resolve) => {
    const start = performance.now();
    let i = 0;
    
    function next() {
      i++;
      if (i < count) {
        setImmediate(next);
      } else {
        resolve(performance.now() - start);
      }
    }
    
    next();
  });
}

// 运行测试
(async () => {
  const count = 1000;
  
  const timeoutTime = await testSetTimeout(count);
  console.log(`setTimeout: ${timeoutTime.toFixed(2)}ms`);
  
  const immediateTime = await testSetImmediate(count);
  console.log(`setImmediate: ${immediateTime.toFixed(2)}ms`);
  
  console.log(`setImmediate 快 ${(timeoutTime / immediateTime).toFixed(2)} 倍`);
})();

// 输出示例：
// setTimeout: 1234.56ms
// setImmediate: 45.67ms
// setImmediate 快 27.03 倍
```

## 📝 总结

本节学习了：

✅ **执行顺序**：主脚本中不确定，I/O 回调中 setImmediate 先执行  
✅ **最小延迟**：setTimeout(fn, 0) 实际延迟 1-4ms  
✅ **性能对比**：setImmediate 在循环调用中快 10-30 倍  
✅ **最佳实践**：I/O 回调中优先使用 setImmediate  
✅ **浏览器兼容**：setImmediate 需要 polyfill  
✅ **优化技巧**：分批处理、避免阻塞事件循环
