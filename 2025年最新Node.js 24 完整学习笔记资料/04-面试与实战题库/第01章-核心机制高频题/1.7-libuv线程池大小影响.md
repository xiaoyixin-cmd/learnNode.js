## 📚 核心知识点

### 1. libuv 线程池概述

**定义：**
- libuv 是 Node.js 的底层异步 I/O 库
- 提供**线程池**处理阻塞操作
- 默认大小：**4 个线程**

**线程池处理的操作：**
1. **文件 I/O**：fs.readFile、fs.writeFile、fs.stat 等
2. **DNS 查询**：dns.lookup（某些平台）
3. **CPU 密集**：crypto.pbkdf2、crypto.scrypt、zlib 压缩等

**不使用线程池的操作：**
- **网络 I/O**：http、tcp、udp（使用 OS 异步 I/O）
- **定时器**：setTimeout、setInterval（事件循环）
- **process.nextTick、Promise**（微任务队列）

---

### 2. UV_THREADPOOL_SIZE 配置

**设置方法：**
```bash
# 方法 1：环境变量（启动时）
UV_THREADPOOL_SIZE=8 node app.js

# 方法 2：代码中设置（必须在第一次 I/O 之前）
process.env.UV_THREADPOOL_SIZE = 8;
const fs = require('fs');  // 然后才能使用 fs

# 方法 3：package.json scripts
{
  "scripts": {
    "start": "UV_THREADPOOL_SIZE=8 node app.js"
  }
}
```

**取值范围：**
- **最小值**：1
- **最大值**：1024（实际建议不超过 128）
- **默认值**：4

---

### 3. 线程池工作原理

```
任务队列
[Task1][Task2][Task3][Task4][Task5][Task6]...
  ↓     ↓     ↓     ↓
[Thread1][Thread2][Thread3][Thread4]
  ↓      ↓       ↓       ↓
完成回调到事件循环的 pending callbacks 阶段
```

**执行流程：**
1. 任务加入队列
2. 空闲线程取出任务
3. 线程执行阻塞操作
4. 完成后回调加入事件循环
5. 事件循环执行回调

## 🎓 面试标准答题

### 问题 1："Node.js 的线程池默认大小是多少？"

**标准答案：**
> Node.js（libuv）的线程池**默认大小是 4**。
> 
> **可以通过环境变量 UV_THREADPOOL_SIZE 修改：**
> ```bash
> UV_THREADPOOL_SIZE=8 node app.js
> ```
> 
> **取值范围：**
> - 最小值：1
> - 最大值：1024
> - 推荐值：4-16（根据 CPU 核心数）
> 
> **使用线程池的操作：**
> 1. **文件 I/O**：fs.readFile、fs.writeFile、fs.stat 等
> 2. **DNS 查询**：dns.lookup（某些平台）
> 3. **CPU 密集**：crypto.pbkdf2、crypto.scrypt、zlib 等
> 
> **不使用线程池：**
> - 网络 I/O（http、tcp、udp）
> - 定时器（setTimeout、setInterval）
> - 微任务（nextTick、Promise）

**加分项：**
> 线程池大小不是越大越好。过大会导致：
> 1. 线程上下文切换开销增加
> 2. 内存占用增加（每个线程约 1-2MB）
> 3. CPU 竞争加剧
> 
> **推荐设置：**
> - CPU 核心数：`UV_THREADPOOL_SIZE = CPU核心数`
> - CPU 核心数 * 1.5：`UV_THREADPOOL_SIZE = CPU核心数 * 1.5`
> - 根据实际负载测试调整

---

### 问题 2："UV_THREADPOOL_SIZE 如何影响性能？"

**标准答案：**
> UV_THREADPOOL_SIZE 直接影响**阻塞操作的并发数**。
> 
> **影响场景：**
> 
> **1. 文件 I/O 密集型应用：**
> - 线程池小：任务排队等待，延迟增加
> - 线程池大：任务并行执行，吞吐量提高
> 
> **示例：**
> ```
> 线程池 = 4：8个文件读取 → 分两批，耗时约 12ms
> 线程池 = 8：8个文件读取 → 一批完成，耗时约 7ms
> ```
> 
> **2. crypto/zlib 密集型：**
> - CPU 密集操作受线程池限制
> - 增加线程池可提高并发，但受 CPU 核心数限制
> 
> **3. 混合负载：**
> - 文件 I/O + crypto 共享线程池
> - 需要平衡两种操作的需求
> 
> **性能曲线：**
> ```
> 吞吐量
>   ↑
>   │     ╱╲  （最佳点）
>   │    ╱  ╲
>   │   ╱    ╲___（过大，性能下降）
>   │  ╱
>   │ ╱
>   └────────────→ 线程池大小
>     4   8   16  32
> ```
> 
> **最佳实践：**
> 1. 测试不同值的性能
> 2. CPU 核心数是良好起点
> 3. I/O 密集型可适当增加
> 4. CPU 密集型不超过 CPU 核心数

**加分项：**
> 在实际项目中，我们通过压力测试发现，将 UV_THREADPOOL_SIZE 从 4 增加到 8，文件上传处理速度提升了 40%，但继续增加到 16 时性能反而下降了 10%（由于上下文切换）。

## 📊 性能测试

### 测试 1：不同线程池大小的吞吐量

```javascript
const fs = require('fs');
const { performance } = require('perf_hooks');

async function benchmark(threadPoolSize, concurrency) {
  process.env.UV_THREADPOOL_SIZE = threadPoolSize;
  
  const start = performance.now();
  let completed = 0;
  
  return new Promise((resolve) => {
    for (let i = 0; i < concurrency; i++) {
      fs.readFile(__filename, () => {
        completed++;
        if (completed === concurrency) {
          const duration = performance.now() - start;
          resolve({
            threadPoolSize,
            concurrency,
            duration: duration.toFixed(2),
            throughput: (concurrency / (duration / 1000)).toFixed(2),
          });
        }
      });
    }
  });
}

(async () => {
  console.log('Benchmark Results:\n');
  
  for (const poolSize of [2, 4, 8, 16, 32]) {
    const result = await benchmark(poolSize, 100);
    console.log(`线程池=${result.threadPoolSize}: ${result.duration}ms, 吞吐量=${result.throughput} ops/s`);
  }
})();

// 输出示例：
// 线程池=2: 245.67ms, 吞吐量=407.04 ops/s
// 线程池=4: 123.45ms, 吞吐量=810.37 ops/s
// 线程池=8: 67.89ms, 吞吐量=1473.10 ops/s  ← 最佳
// 线程池=16: 78.12ms, 吞吐量=1280.12 ops/s  ← 性能下降
// 线程池=32: 89.34ms, 吞吐量=1119.40 ops/s
```

## ⚠️ 常见误区

### 误区 1：以为所有异步操作都使用线程池

```javascript
// ❌ 错误理解
const http = require('http');

// 以为 HTTP 请求使用线程池（实际使用 OS 异步 I/O）
http.get('http://example.com', (res) => {
  // 不受 UV_THREADPOOL_SIZE 影响
});

// ✅ 正确理解
const fs = require('fs');

// 文件读取使用线程池
fs.readFile(__filename, (err, data) => {
  // 受 UV_THREADPOOL_SIZE 影响
});
```

---

### 误区 2：以为线程池越大越好

```javascript
// ❌ 错误：设置过大
UV_THREADPOOL_SIZE=128 node app.js

// 问题：
// 1. 线程上下文切换开销大
// 2. 内存占用高（每线程 1-2MB）
// 3. CPU 竞争加剧
// 4. 性能反而下降

// ✅ 正确：根据 CPU 核心数设置
UV_THREADPOOL_SIZE=8 node app.js  // 8核CPU
```

---

### 误区 3：在代码中间设置 UV_THREADPOOL_SIZE

```javascript
// ❌ 错误：第一次 I/O 后设置无效
const fs = require('fs');

fs.readFile(__filename, () => {
  console.log('done');
});

process.env.UV_THREADPOOL_SIZE = 8;  // ← 无效！线程池已初始化

// ✅ 正确：启动时设置
UV_THREADPOOL_SIZE=8 node app.js

// 或在第一次 I/O 之前
process.env.UV_THREADPOOL_SIZE = 8;
const fs = require('fs');  // ← 然后才使用
```

### 技巧 3：监控线程池使用情况

```javascript
const fs = require('fs');
const { performance } = require('perf_hooks');

let activeThreads = 0;
let maxActiveThreads = 0;

function monitoredReadFile(path, callback) {
  activeThreads++;
  maxActiveThreads = Math.max(maxActiveThreads, activeThreads);
  
  const start = performance.now();
  
  fs.readFile(path, (err, data) => {
    activeThreads--;
    const duration = performance.now() - start;
    
    console.log(`Active threads: ${activeThreads}, Max: ${maxActiveThreads}, Duration: ${duration.toFixed(2)}ms`);
    
    callback(err, data);
  });
}

// 使用
for (let i = 0; i < 10; i++) {
  monitoredReadFile(__filename, () => {});
}
```
