## 📚 核心知识点

### 1. 事件循环 6 个阶段

```
   ┌───────────────────────────┐
┌─>│           timers          │  执行 setTimeout、setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │     pending callbacks     │  执行延迟的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │       idle, prepare       │  仅内部使用
│  └─────────────┬─────────────┘      
│  ┌─────────────▼─────────────┐
│  │           poll            │  检索新的 I/O 事件；执行 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │           check           │  执行 setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
└──│      close callbacks      │  执行 close 事件回调
   └───────────────────────────┘
```

---

### 2. 每个阶段详解

#### **Phase 1: timers（定时器阶段）**
- **执行内容**：setTimeout 和 setInterval 的回调
- **执行时机**：到达指定延迟时间后
- **注意事项**：实际延迟可能大于设定值（受事件循环其他阶段影响）

```javascript
setTimeout(() => {
  console.log('timeout 1');
}, 10);

setTimeout(() => {
  console.log('timeout 2');
}, 0);

// 输出：
// timeout 2
// timeout 1
```

---

#### **Phase 2: pending callbacks（待定回调阶段）**
- **执行内容**：执行延迟到下一个循环迭代的 I/O 回调
- **典型场景**：TCP 错误回调（如 ECONNREFUSED）

```javascript
const net = require('net');

const client = net.connect({ port: 9999 }, () => {
  console.log('connected');
});

client.on('error', (err) => {
  console.log('error:', err.code);  // 在 pending callbacks 阶段执行
});
```

---

#### **Phase 3: idle, prepare（空闲、准备阶段）**
- **执行内容**：仅供 Node.js 内部使用
- **开发者无感知**

---

#### **Phase 4: poll（轮询阶段）** ⭐⭐⭐
- **最重要的阶段**
- **执行内容**：
  1. 计算应该阻塞和轮询 I/O 的时间
  2. 处理 poll 队列中的事件

- **执行流程**：
  ```
  进入 poll 阶段
  ├── 如果 poll 队列不为空
  │   └── 同步执行队列中的回调，直到队列清空或达到系统限制
  │
  ├── 如果 poll 队列为空
  │   ├── 如果有 setImmediate 回调
  │   │   └── 结束 poll 阶段，进入 check 阶段
  │   │
  │   └── 如果没有 setImmediate
  │       └── 等待回调被添加到队列中，然后立即执行
  ```

---

#### **Phase 5: check（检查阶段）**
- **执行内容**：执行 setImmediate 回调
- **特点**：在 poll 阶段完成后立即执行

```javascript
const fs = require('fs');

fs.readFile(__filename, () => {
  console.log('readFile callback');
  
  setTimeout(() => {
    console.log('timeout');
  }, 0);
  
  setImmediate(() => {
    console.log('immediate');
  });
});

// 输出：
// readFile callback
// immediate  ← check 阶段
// timeout    ← 下一轮 timers 阶段
```

---

#### **Phase 6: close callbacks（关闭回调阶段）**
- **执行内容**：执行 close 事件回调
- **典型场景**：socket.on('close')

```javascript
const net = require('net');

const server = net.createServer();

server.on('close', () => {
  console.log('server closed');  // 在 close callbacks 阶段执行
});

server.listen(8080);
server.close();
```

---

### 3. 微任务队列（Microtask Queue）⭐⭐⭐⭐⭐

**执行时机：** 在每个阶段切换时执行

```
阶段 A 执行
   ↓
执行 process.nextTick 队列
   ↓
执行 Promise 微任务队列
   ↓
进入阶段 B
```

**优先级：**
```
process.nextTick > Promise.then > queueMicrotask
```

---

### 4. 完整执行顺序

```javascript
console.log('script start');

setTimeout(() => {
  console.log('timeout 1');
  process.nextTick(() => console.log('nextTick 2'));
}, 0);

setImmediate(() => {
  console.log('immediate');
});

process.nextTick(() => {
  console.log('nextTick 1');
  Promise.resolve().then(() => console.log('promise 1'));
});

Promise.resolve().then(() => {
  console.log('promise 2');
});

console.log('script end');

// 输出分析：
// 1. 同步代码：script start → script end
// 2. nextTick 队列：nextTick 1
// 3. Promise 队列：promise 1 → promise 2
// 4. timers 阶段：timeout 1（或 immediate 先执行，取决于主脚本执行时间）
// 5. nextTick 队列：nextTick 2
// 6. check 阶段：immediate（或 timeout 1）
```

**正确输出：**
```
script start
script end
nextTick 1
promise 1
promise 2
timeout 1（或 immediate）
nextTick 2
immediate（或 timeout 1）
```

## 🎓 面试标准答题

### 问题 1："Node.js 事件循环有哪几个阶段？"

**标准答案：**
> Node.js 事件循环有 **6 个阶段**：
> 
> 1. **timers**：执行 setTimeout 和 setInterval 的回调
> 2. **pending callbacks**：执行延迟的 I/O 回调
> 3. **idle, prepare**：仅供 Node.js 内部使用
> 4. **poll**：检索新的 I/O 事件，执行 I/O 回调
> 5. **check**：执行 setImmediate 回调
> 6. **close callbacks**：执行 close 事件回调
> 
> 每个阶段都有一个 **FIFO 队列**，执行该阶段的所有回调直到队列清空或达到回调数量上限。
> 
> 特别需要注意的是，**微任务（process.nextTick 和 Promise）在每个阶段切换时执行**，其中 **nextTick 优先级最高**。

**加分项：**
> 在实际项目中，我利用 poll 阶段的特性优化了文件读取性能。通过在 I/O 回调中使用 setImmediate 而不是 setTimeout，避免了不必要的延迟，提升了吞吐量约 15%。

---

### 问题 2："微任务在哪个时机执行？"

**标准答案：**
> 微任务在 **每个阶段结束后、进入下一个阶段前** 执行。
> 
> 执行顺序：
> 1. **process.nextTick 队列**（优先级最高）
> 2. **Promise 微任务队列**
> 3. **queueMicrotask 队列**
> 
> 需要注意的是，nextTick 队列会在当前操作完成后、Promise 队列之前执行。如果 nextTick 回调中又调用了 nextTick，会形成递归，可能导致 **I/O 饥饿**。

**示例：**
```javascript
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));

// 输出：
// nextTick
// promise
```

## 💻 实战案例

### 案例 1：复杂异步执行顺序

```javascript
console.log('1');

setTimeout(() => {
  console.log('2');
  Promise.resolve().then(() => console.log('3'));
}, 0);

Promise.resolve().then(() => {
  console.log('4');
  setTimeout(() => {
    console.log('5');
  }, 0);
});

process.nextTick(() => {
  console.log('6');
  process.nextTick(() => console.log('7'));
});

console.log('8');
```

**分析步骤：**
1. 同步代码：`1` → `8`
2. nextTick 队列：`6` → `7`（递归 nextTick）
3. Promise 队列：`4`（注册了新的 setTimeout）
4. timers 阶段：`2`（第一个 setTimeout）
5. Promise 队列：`3`（在 timers 阶段后执行）
6. timers 阶段（下一轮）：`5`（第二个 setTimeout）

**输出：**
```
1
8
6
7
4
2
3
5
```

---

### 案例 2：I/O 回调中的异步

```javascript
const fs = require('fs');

fs.readFile(__filename, () => {
  console.log('fs callback');
  
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
  process.nextTick(() => console.log('nextTick'));
  Promise.resolve().then(() => console.log('promise'));
});

console.log('main');
```

**分析：**
1. 同步代码：`main`
2. poll 阶段：执行 fs.readFile 回调
   - 打印 `fs callback`
   - 注册 setTimeout、setImmediate、nextTick、Promise
3. 微任务：
   - nextTick 队列：`nextTick`
   - Promise 队列：`promise`
4. check 阶段：`immediate`
5. 下一轮 timers 阶段：`timeout`

**输出：**
```
main
fs callback
nextTick
promise
immediate
timeout
```

### 案例 3：递归 nextTick 导致 I/O 饥饿

```javascript
let count = 0;

function recursiveNextTick() {
  if (count < 1000) {
    count++;
    process.nextTick(recursiveNextTick);
  }
}

recursiveNextTick();

setTimeout(() => {
  console.log('timeout', count);  // 会延迟执行
}, 0);

// 输出：
// timeout 1000（延迟约 1000 次 nextTick 后）
```

**风险：**
- nextTick 递归会阻塞事件循环
- 导致 I/O 回调延迟执行
- **最佳实践**：使用 setImmediate 代替递归 nextTick

## 🔍 Node.js vs 浏览器事件循环

| 特性 | Node.js | 浏览器 |
|------|---------|--------|
| **阶段** | 6 个阶段（timers → poll → check...） | 只有宏任务队列和微任务队列 |
| **微任务时机** | 每个阶段结束后 | 每个宏任务后 |
| **nextTick** | 有 process.nextTick | 无 |
| **setImmediate** | 有（check 阶段） | 无 |
| **执行环境** | libuv 驱动 | 浏览器引擎驱动 |

**浏览器事件循环：**
```javascript
// 浏览器中
console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve().then(() => console.log('3'));

console.log('4');

// 输出：1 → 4 → 3 → 2
```

**Node.js 事件循环：**
```javascript
// Node.js 中
console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve().then(() => console.log('3'));

process.nextTick(() => console.log('4'));

console.log('5');

// 输出：1 → 5 → 4 → 3 → 2
```

## ⚠️ 常见误区

### 误区 1：setTimeout(fn, 0) 立即执行

```javascript
// ❌ 错误理解：setTimeout(fn, 0) 会立即执行
setTimeout(() => console.log('timeout'), 0);
console.log('sync');

// 实际输出：
// sync
// timeout

// 原因：setTimeout 最小延迟是 1ms（浏览器中是 4ms）
```

---

### 误区 2：Promise 比 nextTick 先执行

```javascript
// ❌ 错误理解：Promise 先执行
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));

// 实际输出：
// nextTick
// promise

// 原因：nextTick 优先级高于 Promise
```

---

### 误区 3：每个阶段只执行一个回调

```javascript
// ❌ 错误理解：timers 阶段只执行一个 setTimeout
setTimeout(() => console.log('timeout 1'), 0);
setTimeout(() => console.log('timeout 2'), 0);
setTimeout(() => console.log('timeout 3'), 0);

// 实际输出：
// timeout 1
// timeout 2
// timeout 3

// 原因：每个阶段会执行所有队列中的回调（或达到系统限制）
```

## 💡 面试加分技巧

### 1. 主动提及底层实现

> "Node.js 事件循环是由 **libuv** 库实现的，libuv 是一个跨平台的异步 I/O 库。在 Linux 上使用 epoll，macOS 上使用 kqueue，Windows 上使用 IOCP。"

### 2. 结合实际项目经验

> "在我之前的项目中，有一个日志系统使用了递归 nextTick 导致 CPU 占用 100%。后来改用 setImmediate，让出事件循环控制权，问题解决了。"

### 3. 对比其他技术

> "浏览器的事件循环只有宏任务和微任务两个队列，而 Node.js 有 6 个阶段，这使得 Node.js 可以更精细地控制异步任务的执行顺序。"
