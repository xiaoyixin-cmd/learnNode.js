# 3.6 服务网格（Service Mesh）简介：Istio + Envoy 基础

> 将流量管理从业务代码中剥离，让基础设施层自动处理

---

## 📚 核心知识

### 1. 什么是服务网格？

**传统微服务的痛点：**

```javascript
// ❌ 每个服务都需要实现相同的基础设施逻辑
class UserService {
  async getUser(id) {
    // 1. 服务发现
    const instance = await consul.getHealthyInstance('user-service');
    
    // 2. 负载均衡
    const endpoint = loadBalancer.select(instance);
    
    // 3. 重试逻辑
    let retries = 3;
    while (retries > 0) {
      try {
        // 4. 超时控制
        const response = await axios.get(`${endpoint}/users/${id}`, {
          timeout: 3000,
        });
        
        // 5. 追踪埋点
        span.setAttributes({ 'http.status_code': response.status });
        
        return response.data;
      } catch (error) {
        retries--;
        if (retries === 0) throw error;
      }
    }
  }
}

// 问题：
❌ 每个服务重复实现相同逻辑
❌ 多语言技术栈难以统一（Node.js、Java、Go）
❌ 升级困难（需要修改所有服务代码）
❌ 监控数据不一致
```

**服务网格架构：**

```
┌──────────────────────────────────────────────────────────┐
│                   Service Mesh 架构                       │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ┌─────────────┐         ┌─────────────┐               │
│  │ 业务代码     │         │ 业务代码     │               │
│  │ (Node.js)   │         │ (Java)      │               │
│  └──────┬──────┘         └──────┬──────┘               │
│         │                       │                       │
│    ┌────▼────┐             ┌────▼────┐                 │
│    │ Envoy   │◄───加密────►│ Envoy   │  ← Sidecar 代理 │
│    │ Proxy   │    mTLS     │ Proxy   │                 │
│    └────┬────┘             └────┬────┘                 │
│         │                       │                       │
│  ┌──────┴───────────────────────┴──────┐               │
│  │      Istio Control Plane             │               │
│  │  ┌──────────┐  ┌──────────┐         │               │
│  │  │  Pilot   │  │ Citadel  │         │               │
│  │  │ (配置)   │  │ (证书)   │         │               │
│  │  └──────────┘  └──────────┘         │               │
│  └─────────────────────────────────────┘               │
│                                                          │
└──────────────────────────────────────────────────────────┘

核心优势：
✅ 业务代码零侵入（只需要部署 Sidecar）
✅ 统一流量管理（重试、超时、熔断）
✅ 自动 mTLS 加密
✅ 开箱即用的可观测性
✅ 多语言支持（与语言无关）
```

**核心概念：**

- **Data Plane（数据面）**：Envoy Proxy，处理实际流量
- **Control Plane（控制面）**：Istio，管理配置和策略
- **Sidecar 模式**：每个服务 Pod 附带一个 Envoy 代理容器

### 2. Istio 架构

```
┌────────────────────────────────────────────────────────────┐
│                     Istio 架构图                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Control Plane（控制面）                                    │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  Istiod（单体架构，2025）                             │ │
│  │  ┌────────────┐  ┌────────────┐  ┌────────────┐     │ │
│  │  │  Pilot     │  │  Citadel   │  │ Galley     │     │ │
│  │  │ (服务发现)  │  │  (证书)    │  │ (配置验证)  │     │ │
│  │  └────────────┘  └────────────┘  └────────────┘     │ │
│  └──────────────────────────────────────────────────────┘ │
│           ↓ xDS API (配置推送)                             │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  Data Plane（数据面）                                 │ │
│  │  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐    │ │
│  │  │ Envoy  │  │ Envoy  │  │ Envoy  │  │ Envoy  │    │ │
│  │  │ Proxy  │  │ Proxy  │  │ Proxy  │  │ Proxy  │    │ │
│  │  └────────┘  └────────┘  └────────┘  └────────┘    │ │
│  │    Service A  Service B   Service C   Service D     │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**核心组件：**

1. **Pilot**：服务发现、流量管理、配置分发
2. **Citadel**：证书管理、mTLS 加密
3. **Envoy**：高性能 L7 代理（C++ 编写）

### 3. Istio 核心功能

| 功能 | 说明 | 价值 |
|------|------|------|
| **流量管理** | 灰度发布、A/B 测试、流量镜像 | 降低发布风险 |
| **安全** | mTLS、RBAC、JWT 验证 | 零信任网络 |
| **可观测性** | 自动追踪、Metrics、日志 | 免埋点监控 |
| **故障注入** | 延迟注入、错误注入 | 混沌工程 |
| **熔断** | 连接池、重试、超时 | 提升可用性 |

### 实战1：部署第一个 Istio 应用

#### 步骤1：创建 Node.js 应用

```javascript
// app.js
import express from 'express';
import os from 'os';

const app = express();
const PORT = 3000;
const VERSION = process.env.VERSION || 'v1';

app.get('/hello', (req, res) => {
  res.json({
    version: VERSION,
    hostname: os.hostname(),
    message: 'Hello from Istio!',
  });
});

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT} (version: ${VERSION})`);
});
```

```dockerfile
# Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "app.js"]
```

**构建镜像：**
```bash
docker build -t myapp:v1 .
docker tag myapp:v1 myregistry/myapp:v1
docker push myregistry/myapp:v1
```

#### 步骤2：部署到 Kubernetes

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-v1
  labels:
    app: myapp
    version: v1
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp
      version: v1
  template:
    metadata:
      labels:
        app: myapp
        version: v1
    spec:
      containers:
      - name: myapp
        image: myregistry/myapp:v1
        ports:
        - containerPort: 3000
        env:
        - name: VERSION
          value: "v1"
---
apiVersion: v1
kind: Service
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  selector:
    app: myapp  # 注意：选择所有版本
  ports:
  - port: 80
    targetPort: 3000
    name: http
```

**部署：**
```bash
kubectl apply -f deployment.yaml

# 查看 Pod（注意每个 Pod 有 2 个容器：业务容器 + Envoy Sidecar）
kubectl get pods
# NAME                        READY   STATUS    RESTARTS   AGE
# myapp-v1-5d4c8b9f7d-abc12   2/2     Running   0          30s
#                            ↑ 2个容器
```

#### 步骤3：配置 Istio Gateway（入口网关）

```yaml
# gateway.yaml
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: myapp-gateway
spec:
  selector:
    istio: ingressgateway  # 使用 Istio 内置 Ingress Gateway
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "*"  # 接受所有域名
---
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - "*"
  gateways:
  - myapp-gateway
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
        port:
          number: 80
```

**部署：**
```bash
kubectl apply -f gateway.yaml

# 获取 Ingress Gateway 地址
kubectl get svc istio-ingressgateway -n istio-system

# 测试访问
curl http://$(minikube ip)/hello
# {"version":"v1","hostname":"myapp-v1-5d4c8b9f7d-abc12","message":"Hello from Istio!"}
```

```yaml
          value: "v2"
```

```bash
kubectl apply -f deployment-v2.yaml
```

#### 步骤2：配置流量分配（10% v2，90% v1）

```yaml
# virtual-service-canary.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - "*"
  gateways:
  - myapp-gateway
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v1  # 路由到 v1
      weight: 90    # 90% 流量
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v2  # 路由到 v2
      weight: 10    # 10% 流量
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp.default.svc.cluster.local
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

```bash
kubectl apply -f virtual-service-canary.yaml

# 测试流量分配
for i in {1..100}; do
  curl -s http://$(minikube ip)/hello | jq -r '.version'
done | sort | uniq -c

# 输出：
#   90 v1
#   10 v2
```

**逐步切换流量：**
```bash
# 观察 v2 稳定后，调整为 50:50
kubectl patch virtualservice myapp --type merge -p '
spec:
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v1
      weight: 50
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v2
      weight: 50
'

# 最终切换为 100% v2
kubectl patch virtualservice myapp --type merge -p '
spec:
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v2
      weight: 100
'

# 删除 v1 版本
kubectl delete deployment myapp-v1
```

### 实战3：A/B 测试（基于请求头）

```yaml
# virtual-service-ab-test.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - "*"
  gateways:
  - myapp-gateway
  http:
  # 规则1：VIP 用户使用 v2 版本
  - match:
    - headers:
        user-type:
          exact: vip
    route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v2
  
  # 规则2：移动端用户使用 v2 版本
  - match:
    - headers:
        user-agent:
          regex: ".*Mobile.*"
    route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v2
  
  # 规则3：其他用户使用 v1 版本
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
        subset: v1
```

**测试：**

### 实战4：故障注入（混沌工程）

**场景：测试系统在依赖服务慢响应时的表现**

```yaml
# virtual-service-fault-injection.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp.default.svc.cluster.local
  http:
  - fault:
      delay:
        percentage:
          value: 50.0  # 50% 的请求注入延迟
        fixedDelay: 5s  # 延迟 5 秒
      abort:
        percentage:
          value: 10.0  # 10% 的请求注入错误
        httpStatus: 500  # 返回 500 错误
    route:
    - destination:
        host: myapp.default.svc.cluster.local
```

**测试：**
```bash
# 50% 请求会延迟 5 秒
time curl http://$(minikube ip)/hello

# 10% 请求会返回 500 错误
curl http://$(minikube ip)/hello
# {"error":"fault filter abort"}
```

### 实战5：熔断与重试

```yaml
# destination-rule-circuit-breaker.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp.default.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100  # 最大连接数
      http:
        http1MaxPendingRequests: 50  # 最大等待请求数
        http2MaxRequests: 100
        maxRequestsPerConnection: 2
    outlierDetection:
      consecutiveErrors: 3  # 连续 3 次错误后摘除
      interval: 10s
      baseEjectionTime: 30s  # 摘除 30 秒
      maxEjectionPercent: 50  # 最多摘除 50% 实例
```

**重试策略：**
```yaml
# virtual-service-retry.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp.default.svc.cluster.local
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
    retries:
      attempts: 3  # 重试 3 次
      perTryTimeout: 2s  # 每次尝试超时 2 秒
      retryOn: 5xx,reset,connect-failure,refused-stream  # 重试条件
```

### 场景1：零停机发布

**传统发布：**
```
1. 停止服务
2. 更新代码
3. 重启服务
4. 用户体验：服务中断 1-5 分钟
```

**Istio 金丝雀发布：**
```
1. 部署 v2 版本（0% 流量）
2. 逐步切换流量：1% → 5% → 10% → 50% → 100%
3. 监控关键指标（错误率、延迟）
4. 异常时立即回滚（修改 VirtualService）
5. 用户体验：无感知
```

## ❌ 高频错误

### 错误1：未启用 Sidecar 注入

**错误表现：**
```bash
kubectl get pods
# NAME                     READY   STATUS    RESTARTS   AGE
# myapp-5d4c8b9f7d-abc12   1/1     Running   0          30s
#                         ↑ 只有 1 个容器（缺少 Envoy Sidecar）
```

**原因：命名空间未标记 istio-injection**

**解决方案：**
```bash
# 方案1：标记命名空间（自动注入）
kubectl label namespace default istio-injection=enabled

# 方案2：手动注入
kubectl apply -f <(istioctl kube-inject -f deployment.yaml)

# 重新部署
kubectl rollout restart deployment myapp-v1
```

---

### 错误2：VirtualService 不生效

**错误代码：**
```yaml
# ❌ 错误：hosts 不匹配
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - "myapp"  # ❌ 错误：应该是完整域名
  http:
  - route:
    - destination:
        host: myapp
```

**正确做法：**
```yaml
# ✅ 使用完整 FQDN
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - "myapp.default.svc.cluster.local"  # ✅ 完整域名
  http:
  - route:
    - destination:
        host: myapp.default.svc.cluster.local
```

### 错误3：DestinationRule 未定义 subset

**错误日志：**
```
no healthy upstream: myapp.default.svc.cluster.local:80
```

**原因：VirtualService 引用了不存在的 subset**

**解决方案：**
```yaml
# ✅ 必须定义 DestinationRule
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp.default.svc.cluster.local
  subsets:
  - name: v1  # ✅ 定义 subset
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

## 🤔 FAQ

### Q1：Istio 会增加多少延迟？

**答案：< 1ms（Envoy 性能极高）**

**性能测试：**
```
无 Istio：QPS 50000, P95 Latency 10ms
有 Istio：QPS 48000, P95 Latency 10.5ms

延迟增加：0.5ms（1% 影响）
吞吐量下降：4%
```

**优化建议：**
- 启用 HTTP/2（默认）
- 使用 mTLS（性能损耗 < 5%）
- 合理设置连接池大小

### Q3：Istio vs Linkerd vs Consul Connect？

| 特性 | Istio | Linkerd | Consul Connect |
|------|-------|---------|---------------|
| **复杂度** | 高 | 低 | 中 |
| **功能** | 全面 | 基础 | 中等 |
| **性能** | 高 | 极高 | 中 |
| **资源占用** | 中等（Envoy） | 低（Rust 编写） | 中等 |
| **生态** | 最成熟 | 简洁 | HashiCorp 生态 |

**选择建议：**
- **大型企业**：Istio（功能最全面）
- **追求简洁**：Linkerd（最轻量）
