# 3.5 分布式追踪：OpenTelemetry Node SDK 集成

> 全链路追踪：让微服务调用链可视化

---

## 📚 核心知识

### 1. 为什么需要分布式追踪？

**微服务调用链的复杂性：**

```
客户端请求 → API 网关 → 订单服务 → 用户服务 → 数据库
                    ↓
                库存服务 → Redis
                    ↓
                支付服务 → 第三方支付 API
                    ↓
                消息队列 → 通知服务 → 短信服务

问题：
❌ 一个请求跨越 10+ 个服务，如何定位慢查询？
❌ 某个服务报错，如何找到完整调用链？
❌ 性能瓶颈在哪个环节？
❌ 如何统计各服务的延迟分布？
```

**分布式追踪解决方案：**

```
┌────────────────────────────────────────────────────────┐
│                   完整追踪示例                          │
├────────────────────────────────────────────────────────┤
│                                                        │
│  TraceID: 1234567890abcdef (贯穿整个请求)             │
│                                                        │
│  [Span 1] API Gateway          0ms ─────────┐         │
│    └─ [Span 2] Order Service   5ms ────┐    │         │
│        ├─ [Span 3] User Service  10ms ─┼────┤         │
│        │   └─ [Span 4] MySQL      15ms─┼────┼───┐     │
│        ├─ [Span 5] Inventory Svc 20ms ─┼────┼───┤     │
│        │   └─ [Span 6] Redis      5ms ─┼────┼───┼──┐  │
│        └─ [Span 7] Payment Svc   50ms ─┴────┴───┴──┘  │
│                                                        │
│  总耗时: 50ms (Payment Service 是瓶颈)                 │
└────────────────────────────────────────────────────────┘
```

**核心概念：**

- **Trace（追踪）**：一次完整的请求链路
- **Span（跨度）**：一次服务调用或操作（如数据库查询）
- **SpanContext（上下文）**：Trace ID + Span ID，跨进程传递
- **Attributes（属性）**：键值对元数据（如 http.method、db.statement）

### 2. OpenTelemetry 简介

**OpenTelemetry** 是 CNCF（云原生计算基金会）的可观测性标准，合并了 OpenTracing 和 OpenCensus。

**核心优势：**

1. **厂商中立**：支持导出到 Jaeger、Zipkin、Grafana Tempo、DataDog、New Relic 等
2. **自动埋点**：无侵入式自动追踪（HTTP、gRPC、数据库等）
3. **统一 API**：Tracing + Metrics + Logs 三合一
4. **多语言支持**：Node.js、Java、Go、Python、.NET 等

**架构图：**

```
┌──────────────────────────────────────────────────────┐
│           OpenTelemetry Architecture                 │
├──────────────────────────────────────────────────────┤
│                                                      │
│  应用代码 (Node.js)                                  │
│    ↓                                                 │
│  OpenTelemetry API (创建 Span)                       │
│    ↓                                                 │
│  OpenTelemetry SDK (收集数据)                        │
│    ↓                                                 │
│  自动埋点插件 (express, axios, mongodb, etc.)         │
│    ↓                                                 │
│  Exporter (导出器)                                   │
│    ├─ Jaeger Exporter ──> Jaeger                    │
│    ├─ Zipkin Exporter ──> Zipkin                    │
│    ├─ OTLP Exporter   ──> Grafana Tempo             │
│    └─ Console Exporter ──> 控制台 (开发调试)         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 3. Jaeger vs Zipkin vs Grafana Tempo

| 特性 | Jaeger | Zipkin | Grafana Tempo |
|------|--------|--------|---------------|
| **开发者** | Uber | Twitter | Grafana Labs |
| **存储** | Cassandra/ES | ES/MySQL | S3/GCS |
| **查询** | UI + API | UI + API | Grafana 插件 |
| **性能** | 高 | 中等 | 极高 |
| **部署** | 复杂（多组件） | 简单 | 中等 |
| **采样** | ✅ 支持 | ✅ 支持 | ✅ 支持 |
| **社区** | CNCF 项目 | CNCF 项目 | Grafana 生态 |

**选择建议：**
- **开发/测试环境**：Jaeger（UI 友好，功能完善）
- **生产环境**：Grafana Tempo（成本低，性能高）
- **遗留系统**：Zipkin（兼容性好）

#### 步骤2：初始化 OpenTelemetry

```javascript
// tracing.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { Resource } from '@opentelemetry/resources';
import { SemanticResourceAttributes } from '@opentelemetry/semantic-conventions';

// 创建 OTLP 导出器
const traceExporter = new OTLPTraceExporter({
  url: 'http://localhost:4318/v1/traces',  // Jaeger OTLP 端点
});

// 创建 SDK
const sdk = new NodeSDK({
  resource: new Resource({
    [SemanticResourceAttributes.SERVICE_NAME]: 'order-service',  // 服务名称
    [SemanticResourceAttributes.SERVICE_VERSION]: '1.0.0',
  }),
  traceExporter,
  instrumentations: [
    getNodeAutoInstrumentations({
      // 自动埋点配置
      '@opentelemetry/instrumentation-http': {
        enabled: true,
      },
      '@opentelemetry/instrumentation-express': {
        enabled: true,
      },
    }),
  ],
});

// 启动 SDK
sdk.start();

// 优雅退出
process.on('SIGTERM', () => {
  sdk.shutdown()
    .then(() => console.log('✅ Tracing terminated'))
    .catch((error) => console.error('❌ Error terminating tracing', error))
    .finally(() => process.exit(0));
});

console.log('🔍 Tracing initialized');
```

### 实战2：手动创建 Span

```javascript
// order-service-manual.js
import './tracing.js';
import express from 'express';
import axios from 'axios';
import { trace, SpanStatusCode } from '@opentelemetry/api';

const app = express();
app.use(express.json());

// 获取 Tracer
const tracer = trace.getTracer('order-service', '1.0.0');

app.post('/orders', async (req, res) => {
  // 创建根 Span
  const span = tracer.startSpan('create_order');
  
  try {
    const { userId, productId, quantity } = req.body;
    
    // 添加属性
    span.setAttributes({
      'order.user_id': userId,
      'order.product_id': productId,
      'order.quantity': quantity,
    });
    
    // 子 Span：查询用户
    const userSpan = tracer.startSpan('get_user', {
      parent: span,
    });
    const userRes = await axios.get(`http://localhost:3001/users/${userId}`);
    userSpan.setAttributes({
      'http.method': 'GET',
      'http.url': `http://localhost:3001/users/${userId}`,
      'http.status_code': userRes.status,
    });
    userSpan.end();
    
    // 子 Span：锁定库存
    const inventorySpan = tracer.startSpan('reserve_inventory', {
      parent: span,
    });
    const inventoryRes = await axios.post(
      'http://localhost:3002/inventory/reserve',
      { productId, quantity }
    );
    inventorySpan.setAttributes({
      'http.method': 'POST',
      'http.url': 'http://localhost:3002/inventory/reserve',
      'http.status_code': inventoryRes.status,
    });
    inventorySpan.end();
    
    // 子 Span：保存订单到数据库（模拟）
    const dbSpan = tracer.startSpan('insert_order', {
      parent: span,
    });
    dbSpan.setAttributes({
      'db.system': 'mongodb',
      'db.statement': 'db.orders.insertOne({...})',
    });
    
    // 模拟数据库操作
    await new Promise(resolve => setTimeout(resolve, 10));
    
    dbSpan.end();
    
    // 标记成功
    span.setStatus({ code: SpanStatusCode.OK });
    
    const order = {
      id: `ORD${Date.now()}`,
      userId,
      productId,
      quantity,
      status: 'pending',
    };
    
    res.json(order);
  } catch (error) {
    // 标记失败
    span.recordException(error);
    span.setStatus({
      code: SpanStatusCode.ERROR,
      message: error.message,
    });
    
    res.status(500).json({ error: 'Internal server error' });
  } finally {
    // 必须结束 Span
    span.end();
  }
});

app.listen(3000, () => {
  console.log('🛒 Order Service running on port 3000');
});
```

**手动 Span 的优势：**
- ✅ 更细粒度的追踪（如数据库查询、缓存操作）
- ✅ 添加业务相关属性（订单ID、用户ID）
- ✅ 记录异常详情

### 实战3：跨进程上下文传播

**问题：如何在 HTTP 请求间传递 Trace Context？**

**答案：使用 W3C Trace Context 标准（自动实现）**

```
HTTP 请求头：
traceparent: 00-1234567890abcdef-0987654321fedcba-01
             │   TraceID          │    SpanID      │ Flags

TraceID：全局唯一，贯穿整个请求链
SpanID：当前 Span 的 ID
Flags：采样标志（01=采样，00=不采样）
```

**OpenTelemetry 自动传播 Context（无需手动实现）：**

```javascript
// 订单服务发起请求时，自动添加 traceparent 头
await axios.get('http://localhost:3001/users/1');

// HTTP 请求实际包含：
// GET /users/1 HTTP/1.1
// Host: localhost:3001
// traceparent: 00-1234567890abcdef-0987654321fedcba-01  ← 自动添加

// 用户服务接收到请求后，自动提取 TraceID 并关联
```

**手动传播 Context（gRPC 等非 HTTP 场景）：**

```javascript
import { context, propagation } from '@opentelemetry/api';

// 发送方：注入 Context
const carrier = {};
propagation.inject(context.active(), carrier);

// 通过 gRPC Metadata 传递
const metadata = new grpc.Metadata();
Object.keys(carrier).forEach(key => {
  metadata.add(key, carrier[key]);
});

client.GetUser({ id: '123' }, metadata, callback);

// 接收方：提取 Context
const carrier = {};
metadata.getMap().forEach((value, key) => {
  carrier[key] = value;
});

const extractedContext = propagation.extract(context.active(), carrier);

// 在提取的 Context 中执行操作
context.with(extractedContext, () => {
  // 这里创建的 Span 会关联到同一个 Trace
  const span = tracer.startSpan('handle_request');
  // ...
});
```

### 实战4：采样策略

**问题：生产环境每秒数万请求，全量追踪开销太大。**

**解决方案：采样（Sampling）**

```javascript
// tracing-with-sampling.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { TraceIdRatioBasedSampler, ParentBasedSampler } from '@opentelemetry/sdk-trace-base';

const sdk = new NodeSDK({
  // 采样器配置
  sampler: new ParentBasedSampler({
    // 根 Span 采样：10% 概率
    root: new TraceIdRatioBasedSampler(0.1),
  }),
  // 其他配置...
});

sdk.start();
```

**采样策略对比：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **AlwaysOn** | 100% 采样 | 开发/测试环境 |
| **AlwaysOff** | 0% 采样（关闭追踪） | 性能敏感场景 |
| **TraceIdRatioBasedSampler** | 按百分比采样 | 生产环境（10%-1%） |
| **ParentBasedSampler** | 跟随父 Span 决策 | 保证完整链路 |

**自定义采样器（基于业务规则）：**

```javascript
import { Sampler, SamplingDecision } from '@opentelemetry/sdk-trace-base';

class CustomSampler {
  shouldSample(context, traceId, spanName, spanKind, attributes) {
    // 1. 错误请求必须采样
    if (attributes['http.status_code'] >= 400) {
      return {
        decision: SamplingDecision.RECORD_AND_SAMPLED,
      };
    }
    
    // 2. 慢请求必须采样（> 1 秒）
    if (attributes['http.duration'] > 1000) {
      return {
        decision: SamplingDecision.RECORD_AND_SAMPLED,
      };
    }
    
    // 3. VIP 用户请求必须采样
    if (attributes['user.tier'] === 'vip') {
      return {
        decision: SamplingDecision.RECORD_AND_SAMPLED,
      };
    }
    
    // 4. 其他请求 1% 采样
    return {
      decision: Math.random() < 0.01
        ? SamplingDecision.RECORD_AND_SAMPLED
        : SamplingDecision.NOT_RECORD,
    };
  }
}

const sdk = new NodeSDK({
  sampler: new CustomSampler(),
  // ...
});
```

## 🎯 应用场景

### 场景1：性能瓶颈定位

**问题：用户反馈下单接口慢，平均响应时间 2 秒。**

**分析步骤：**

1. **查看 Jaeger UI**：
   ```
   Trace: abc123 (总耗时: 2.1s)
   
   [order-service] POST /orders (0ms-2100ms)
     ├─ [HTTP GET] /users/1 (5ms-15ms) ✅ 正常
     ├─ [HTTP POST] /inventory/reserve (20ms-40ms) ✅ 正常
     └─ [HTTP POST] /payment/charge (50ms-2080ms) ❌ 瓶颈
         └─ [payment-service] POST /charge (51ms-2079ms)
             └─ [HTTP POST] Third-Party API (100ms-2050ms) ❌ 根因
   ```

2. **定位根因**：第三方支付 API 响应慢（2 秒）

3. **解决方案**：
   - 优化：使用异步支付（立即返回订单，后台处理支付）
   - 降级：超时 1 秒后切换到备用支付渠道

### 场景2：分布式事务问题排查

**问题：订单创建成功，但库存未扣减。**

**分析步骤：**

```
Trace: def456 (异常)

[order-service] POST /orders (0ms-50ms)
  ├─ [HTTP POST] /inventory/reserve (10ms-30ms)
  │   └─ [inventory-service] ❌ 500 Internal Server Error
  │       └─ [MongoDB] connection timeout ❌ 数据库连接超时
  └─ [MongoDB] insertOne (40ms-48ms) ✅ 订单已插入

结论：库存服务数据库超时，但订单已创建（缺少事务回滚）
```

**解决方案：**
- 引入 Saga 模式或分布式事务
- 记录补偿日志，定时任务修复不一致数据

### 错误2：Span 未正确结束导致内存泄漏

**错误代码：**
```javascript
// ❌ 错误：异常时未结束 Span
const span = tracer.startSpan('operation');

try {
  await doSomething();
  span.end();  // ✅ 正常结束
} catch (error) {
  console.error(error);
  // ❌ 异常时忘记 span.end()，导致内存泄漏
}
```

**正确做法：**
```javascript
// ✅ 使用 finally 确保必定结束
const span = tracer.startSpan('operation');

try {
  await doSomething();
  span.setStatus({ code: SpanStatusCode.OK });
} catch (error) {
  span.recordException(error);
  span.setStatus({
    code: SpanStatusCode.ERROR,
    message: error.message,
  });
  throw error;
} finally {
  span.end();  // 必定执行
}
```

### 错误3：过度追踪导致性能下降

**错误代码：**
```javascript
// ❌ 错误：为每个循环创建 Span
for (let i = 0; i < 10000; i++) {
  const span = tracer.startSpan(`process_item_${i}`);
  processItem(i);
  span.end();
}

// 后果：创建 10000 个 Span，开销巨大
```

**正确做法：**
```javascript
// ✅ 批量处理，只创建一个 Span
const span = tracer.startSpan('process_items');
span.setAttribute('item.count', 10000);

for (let i = 0; i < 10000; i++) {
  processItem(i);
}

span.end();
```

## 🤔 FAQ

### Q1：追踪会影响性能吗？

**答案：有轻微影响，但可控。**

**性能开销测试：**
```
无追踪：QPS 50000
自动埋点（100% 采样）：QPS 45000（下降 10%）
自动埋点（10% 采样）：QPS 48000（下降 4%）
自动埋点（1% 采样）：QPS 49500（下降 1%）
```

**优化建议：**
1. **生产环境使用低采样率**（1%-10%）
2. **关闭不必要的自动埋点**
3. **批量导出**（默认每 5 秒导出一次）
4. **使用异步导出**（不阻塞主线程）

### Q2：如何与现有日志系统集成？

**方案：将 TraceID 和 SpanID 注入日志**

```javascript
import winston from 'winston';
import { trace, context } from '@opentelemetry/api';

// 创建日志格式化器
const logFormat = winston.format.printf(({ level, message, timestamp }) => {
  // 获取当前 Span
  const span = trace.getSpan(context.active());
  
  if (span) {
    const spanContext = span.spanContext();
    return `${timestamp} [${level}] [trace_id=${spanContext.traceId}] [span_id=${spanContext.spanId}] ${message}`;
  }
  
  return `${timestamp} [${level}] ${message}`;
});

const logger = winston.createLogger({
  format: winston.format.combine(
    winston.format.timestamp(),
    logFormat
  ),
  transports: [new winston.transports.Console()],
});

// 使用
logger.info('Order created');
// 输出：2025-01-15 08:30:45 [info] [trace_id=1234567890abcdef] [span_id=0987654321fedcba] Order created
```

**在 ELK Stack 中查询：**
```
# Kibana 查询：找到错误日志对应的完整追踪链路
level:error AND trace_id:"1234567890abcdef"
```

```javascript
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

instrumentations: [

    },

      enabled: true,  // 自动追踪 MySQL
    },
  }),
],
```

**方案2：手动创建 Span**

```javascript
async function findUser(userId) {
  const span = tracer.startSpan('db.query', {
    attributes: {
      'db.system': 'mongodb',
      'db.operation': 'findOne',
      'db.collection': 'users',
      'db.query': `{ _id: "${userId}" }`,
    },
  });
  
  try {
    const user = await db.collection('users').findOne({ _id: userId });
    span.setStatus({ code: SpanStatusCode.OK });
    return user;
  } catch (error) {
    span.recordException(error);
    span.setStatus({ code: SpanStatusCode.ERROR });
    throw error;
  } finally {
    span.end();
  }
}
```
