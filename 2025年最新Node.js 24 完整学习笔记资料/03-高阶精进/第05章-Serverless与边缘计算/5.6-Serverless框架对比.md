# 5.6 Serverless 框架对比

> Serverless Framework vs AWS SAM vs SST - 如何选择最适合的工具？

---

## 📚 知识导读

### 为什么需要 Serverless 框架？

**原生开发 vs 使用框架：**

```
┌────────────────────────────────────────────────────────┐
│              原生开发（手动管理）                       │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 编写 Lambda 函数代码                               │
│  2. 手动打包依赖（zip）                                │
│  3. 在 AWS Console 创建 Lambda 函数                    │
│  4. 配置环境变量、IAM 角色、触发器                     │
│  5. 创建 API Gateway                                   │
│  6. 配置路由、CORS、授权                              │
│  7. 创建 DynamoDB 表                                   │
│  8. 配置 CloudWatch Logs                              │
│  9. 部署更新时重复步骤 2-8                             │
│                                                        │
│  ❌ 繁琐、易错、难以维护                               │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│            使用框架（Infrastructure as Code）           │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 编写配置文件（serverless.yml / template.yaml）    │
│  2. 编写 Lambda 函数代码                               │
│  3. 运行 `deploy` 命令                                 │
│                                                        │
│  ✅ 框架自动处理：                                     │
│     - 打包代码                                         │
│     - 创建/更新资源                                    │
│     - 配置触发器                                       │
│     - 设置权限                                         │
│     - 部署                                             │
│                                                        │
│  ✅ 简单、可复现、易维护                               │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 🎯 三大框架对比

### 1. Serverless Framework

**官方网站：** https://www.serverless.com/

**定位：** 跨云平台的 Serverless 开发框架

**支持平台：**
- AWS Lambda
- Azure Functions
- Google Cloud Functions
- Cloudflare Workers
- Alibaba Cloud Function Compute

**优势：**
- ✅ **跨云支持**：一套配置部署多个云平台
- ✅ **丰富的插件生态**：300+ 官方和社区插件
- ✅ **简单易用**：配置文件简洁
- ✅ **本地开发**：serverless-offline 插件

**劣势：**
- ❌ 不是 AWS 官方工具
- ❌ 高级 AWS 功能支持有限
- ❌ 企业版功能需付费

---

### 2. AWS SAM（Serverless Application Model）

**官方网站：** https://aws.amazon.com/serverless/sam/

**定位：** AWS 官方 Serverless 开发框架

**支持平台：**
- 仅 AWS（Lambda、API Gateway、DynamoDB、S3 等）

**优势：**
- ✅ **AWS 官方支持**：与 AWS 生态深度集成
- ✅ **完整的 CloudFormation 兼容**：可使用所有 AWS 资源
- ✅ **本地调试强大**：`sam local` 完全模拟 Lambda 运行时
- ✅ **免费**：无额外费用

**劣势：**
- ❌ 仅支持 AWS
- ❌ 配置文件冗长（CloudFormation 语法）
- ❌ 学习曲线陡峭

---

### 3. SST（Serverless Stack）

**官方网站：** https://sst.dev/

**定位：** 现代化、类型安全的 Serverless 框架

**支持平台：**
- AWS（主要）
- 未来计划支持其他云

**优势：**
- ✅ **类型安全**：TypeScript 原生支持，编辑器自动补全
- ✅ **实时调试**：Live Lambda Development（本地代码更改立即生效）
- ✅ **现代化 DX**：配置即代码（`sst.config.ts`）
- ✅ **Web 框架集成**：Next.js、Remix、SvelteKit 原生支持

**劣势：**
- ❌ 较新（2020 年发布）
- ❌ 社区和生态相对较小
- ❌ 仅支持 AWS

## 💻 实战对比：同一个 API 的实现

### 需求：创建一个 CRUD API

- **POST /users** - 创建用户
- **GET /users/{id}** - 获取用户
- **GET /users** - 列出用户
- **PUT /users/{id}** - 更新用户
- **DELETE /users/{id}** - 删除用户

**技术栈：**
- Lambda + API Gateway + DynamoDB

---

## 1. Serverless Framework 实现

**项目结构：**

```
my-api/
├── src/
│   └── handlers/
│       ├── createUser.mjs
│       ├── getUser.mjs
│       ├── listUsers.mjs
│       ├── updateUser.mjs
│       └── deleteUser.mjs
├── serverless.yml
└── package.json
```

**配置文件（`serverless.yml`）：**

```yaml
service: users-api

frameworkVersion: '3'

provider:
  name: aws
  runtime: nodejs20.x
  region: us-east-1
  stage: ${opt:stage, 'dev'}
  environment:
    USERS_TABLE: ${self:service}-users-${self:provider.stage}
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - dynamodb:PutItem
            - dynamodb:GetItem
            - dynamodb:Scan
            - dynamodb:UpdateItem
            - dynamodb:DeleteItem
          Resource: !GetAtt UsersTable.Arn

functions:
  createUser:
    handler: src/handlers/createUser.handler
    events:
      - httpApi:
          path: /users
          method: POST

  getUser:
    handler: src/handlers/getUser.handler
    events:
      - httpApi:
          path: /users/{id}
          method: GET

  listUsers:
    handler: src/handlers/listUsers.handler
    events:
      - httpApi:
          path: /users
          method: GET

  updateUser:
    handler: src/handlers/updateUser.handler
    events:
      - httpApi:
          path: /users/{id}
          method: PUT

  deleteUser:
    handler: src/handlers/deleteUser.handler
    events:
      - httpApi:
          path: /users/{id}
          method: DELETE

resources:
  Resources:
    UsersTable:
      Type: AWS::DynamoDB::Table
      Properties:
        TableName: ${self:provider.environment.USERS_TABLE}
        AttributeDefinitions:
          - AttributeName: id
            AttributeType: S
        KeySchema:
          - AttributeName: id
            KeyType: HASH
        BillingMode: PAY_PER_REQUEST

plugins:
  - serverless-offline

custom:
  serverless-offline:
    httpPort: 3000
```

**部署：**

```bash
# 安装依赖
npm install

# 部署到 dev 环境
serverless deploy --stage dev

# 部署到 prod 环境
serverless deploy --stage prod

# 本地开发
serverless offline
```

## 2. AWS SAM 实现

**项目结构：**

```
my-api/
├── src/
│   └── handlers/
│       ├── createUser.mjs
│       ├── getUser.mjs
│       ├── listUsers.mjs
│       ├── updateUser.mjs
│       └── deleteUser.mjs
├── template.yaml
└── package.json
```

**配置文件（`template.yaml`）：**

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: Users API

Parameters:
  Stage:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - prod

Globals:
  Function:
    Timeout: 10
    MemorySize: 512
    Runtime: nodejs20.x
    Environment:
      Variables:
        USERS_TABLE: !Ref UsersTable

Resources:
  UsersApi:
    Type: AWS::Serverless::Api
    Properties:
      StageName: !Ref Stage
      Cors:
        AllowMethods: "'GET,POST,PUT,DELETE,OPTIONS'"
        AllowHeaders: "'Content-Type,Authorization'"
        AllowOrigin: "'*'"

  CreateUserFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./
      Handler: src/handlers/createUser.handler
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref UsersTable
      Events:
        CreateUserAPI:
          Type: Api
          Properties:
            RestApiId: !Ref UsersApi
            Path: /users
            Method: POST

  GetUserFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./
      Handler: src/handlers/getUser.handler
      Policies:
        - DynamoDBReadPolicy:
            TableName: !Ref UsersTable
      Events:
        GetUserAPI:
          Type: Api
          Properties:
            RestApiId: !Ref UsersApi
            Path: /users/{id}
            Method: GET

  ListUsersFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./
      Handler: src/handlers/listUsers.handler
      Policies:
        - DynamoDBReadPolicy:
            TableName: !Ref UsersTable
      Events:
        ListUsersAPI:
          Type: Api
          Properties:
            RestApiId: !Ref UsersApi
            Path: /users
            Method: GET

  UpdateUserFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./
      Handler: src/handlers/updateUser.handler
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref UsersTable
      Events:
        UpdateUserAPI:
          Type: Api
          Properties:
            RestApiId: !Ref UsersApi
            Path: /users/{id}
            Method: PUT

  DeleteUserFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./
      Handler: src/handlers/deleteUser.handler
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref UsersTable
      Events:
        DeleteUserAPI:
          Type: Api
          Properties:
            RestApiId: !Ref UsersApi
            Path: /users/{id}
            Method: DELETE

  UsersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: !Sub '${AWS::StackName}-users'
      AttributeDefinitions:
        - AttributeName: id
          AttributeType: S
      KeySchema:
        - AttributeName: id
          KeyType: HASH
      BillingMode: PAY_PER_REQUEST

Outputs:
  ApiEndpoint:
    Description: API Gateway endpoint URL
    Value: !Sub 'https://${UsersApi}.execute-api.${AWS::Region}.amazonaws.com/${Stage}/'
```

**部署：**

```bash
# 构建
sam build

# 部署到 dev 环境
sam deploy --stack-name users-api-dev --parameter-overrides Stage=dev

# 本地开发
sam local start-api --port 3000
```

## 3. SST 实现

**项目结构：**

```
my-api/
├── packages/
│   ├── functions/
│   │   └── src/
│   │       ├── createUser.ts
│   │       ├── getUser.ts
│   │       ├── listUsers.ts
│   │       ├── updateUser.ts
│   │       └── deleteUser.ts
│   └── core/
│       └── src/
│           └── types.ts
├── sst.config.ts
└── package.json
```

**配置文件（`sst.config.ts`）：**

```typescript
import { SSTConfig } from 'sst';
import { API } from './stacks/API';

export default {
  config(_input) {
    return {
      name: 'users-api',
      region: 'us-east-1',
    };
  },
  stacks(app) {
    app.stack(API);
  },
} satisfies SSTConfig;
```

**Stack 定义（`stacks/API.ts`）：**

```typescript
import { StackContext, Api, Table } from 'sst/constructs';

export function API({ stack }: StackContext) {
  // 创建 DynamoDB 表
  const table = new Table(stack, 'Users', {
    fields: {
      id: 'string',
    },
    primaryIndex: { partitionKey: 'id' },
  });

  // 创建 API
  const api = new Api(stack, 'UsersApi', {
    defaults: {
      function: {
        bind: [table],  // 自动授权和注入环境变量
      },
    },
    routes: {
      'POST /users': 'packages/functions/src/createUser.handler',
      'GET /users/{id}': 'packages/functions/src/getUser.handler',
      'GET /users': 'packages/functions/src/listUsers.handler',
      'PUT /users/{id}': 'packages/functions/src/updateUser.handler',
      'DELETE /users/{id}': 'packages/functions/src/deleteUser.handler',
    },
  });

  // 输出 API 端点
  stack.addOutputs({
    ApiEndpoint: api.url,
  });

  return { api, table };
}
```

**Handler 示例（`packages/functions/src/createUser.ts`）：**

```typescript
import { APIGatewayProxyHandlerV2 } from 'aws-lambda';
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';
import { DynamoDBDocumentClient, PutCommand } from '@aws-sdk/lib-dynamodb';
import { Table } from 'sst/node/table';
import { randomUUID } from 'crypto';

const client = DynamoDBDocumentClient.from(new DynamoDBClient({}));

export const handler: APIGatewayProxyHandlerV2 = async (event) => {
  const body = JSON.parse(event.body || '{}');

  const user = {
    id: randomUUID(),
    name: body.name,
    email: body.email,
    createdAt: new Date().toISOString(),
  };

  await client.send(
    new PutCommand({
      TableName: Table.Users.tableName,  // 类型安全！
      Item: user,
    })
  );

  return {
    statusCode: 201,
    body: JSON.stringify({ success: true, data: user }),
  };
};
```

**部署：**

```bash
# 安装依赖
npm install

# 本地开发（实时热重载）
npm run dev

# 部署到 dev 环境
npm run deploy -- --stage dev

# 部署到 prod 环境
npm run deploy -- --stage prod
```

## 🎯 选择建议

### 场景 1：初学者或小型项目

**推荐：Serverless Framework**

**理由：**
- 配置文件简单易懂
- 丰富的教程和社区资源
- 快速上手（30 分钟即可部署第一个函数）

---

### 场景 2：企业级 AWS 项目

**推荐：AWS SAM**

**理由：**
- AWS 官方支持，与 AWS 生态深度集成
- 完整的 CloudFormation 兼容性
- 强大的本地调试能力
- 免费且稳定

---

### 场景 3：现代 Web 应用（Next.js、Remix）

**推荐：SST**

**理由：**
- 类型安全（TypeScript 原生支持）
- 实时热重载（Live Lambda Development）
- 原生支持 Next.js、Remix、SvelteKit
- 现代化开发体验

---

### 场景 4：多云部署

**推荐：Serverless Framework**

**理由：**
- 唯一支持多云的框架
- 一套代码部署到 AWS、Azure、GCP

## 📚 FAQ

### 1. 可以混合使用多个框架吗？

**不建议！** 每个框架管理基础设施的方式不同，混合使用会导致：
- 资源冲突
- 难以维护
- 部署不可复现

**例外情况：**
- 使用 Serverless Framework 部署 Lambda
- 使用 Terraform/Pulumi 管理其他 AWS 资源

### 2. 从一个框架迁移到另一个框架困难吗？

**Lambda 函数代码无需修改**，只需重写配置文件：

| 迁移路径 | 难度 | 时间估算 |
|---------|------|---------|
| Serverless → SAM | 中等 | 2-4 小时 |
| Serverless → SST | 简单 | 1-2 小时 |
| SAM → Serverless | 简单 | 1-2 小时 |
| SAM → SST | 简单 | 1-2 小时 |
| SST → Serverless | 中等 | 2-3 小时 |

### 3. 哪个框架性能最好？

**部署后性能完全一致！** 三个框架最终都是部署到 AWS Lambda，运行时性能无差异。

**差异在于：**
- **开发体验**：SST > Serverless > SAM
- **本地调试**：SAM > SST > Serverless
- **部署速度**：Serverless ≈ SST > SAM
