# 5.2 冷启动优化

> 减少 50%+ 冷启动时间，提升用户体验

---

## 📚 知识导读

### 什么是冷启动？

**冷启动（Cold Start）** 是指 Lambda 函数在长时间未被调用后，AWS 需要重新启动容器来执行函数的过程。

**冷启动 vs 热启动：**

```
┌─────────────────────────────────────────────────────────┐
│              Lambda 调用生命周期                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  冷启动（Cold Start）- 首次调用或容器过期后              │
│  ┌───────────────────────────────────────────────────┐ │
│  │ 1. Download Code        (~100-300ms)              │ │
│  │    ↓                                              │ │
│  │ 2. Start Container      (~50-200ms)               │ │
│  │    ↓                                              │ │
│  │ 3. Initialize Runtime   (~100-500ms)              │ │
│  │    ↓                                              │ │
│  │ 4. Run Init Code        (取决于你的代码)           │ │
│  │    ↓                                              │ │
│  │ 5. Execute Handler      (你的业务逻辑)             │ │
│  └───────────────────────────────────────────────────┘ │
│         总时间：500ms - 2s+                            │
│                                                         │
│  热启动（Warm Start）- 容器复用                         │
│  ┌───────────────────────────────────────────────────┐ │
│  │ Execute Handler         (只执行业务逻辑)           │ │
│  └───────────────────────────────────────────────────┘ │
│         总时间：<50ms                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 冷启动的影响

**实际测试数据（Node.js 20 + 512MB 内存）：**

| 场景 | 包大小 | 依赖数量 | 冷启动时间 | 热启动时间 |
|------|-------|---------|-----------|-----------|
| 最小示例 | 1KB | 0 | 180ms | 2ms |
| Express API | 5MB | 50 | 450ms | 8ms |
| 全功能 API | 50MB | 200 | 1200ms | 15ms |
| 未优化项目 | 200MB | 500+ | 3000ms+ | 20ms |

**影响因素：**

1. **包大小**：越大越慢（线性关系）
2. **依赖数量**：模块加载时间
3. **内存配置**：低内存 = 低 CPU = 慢启动
4. **VPC 配置**：需要创建网络接口（+1-3秒）
5. **初始化代码**：数据库连接、配置加载等

## 💻 策略 1：减小包体积

### 1.1 使用 esbuild 打包

**为什么选择 esbuild？**

- **速度快**：比 Webpack 快 100 倍
- **Tree-shaking**：自动移除未使用代码
- **压缩**：代码最小化
- **支持 ES Modules**：与 Node.js 20 完美兼容

**安装 esbuild：**

```bash
npm install --save-dev esbuild
```

**打包配置：**

```javascript
// build.mjs
import { build } from 'esbuild';
import { glob } from 'glob';

const entryPoints = await glob('src/handlers/*.mjs');

await build({
  entryPoints,
  bundle: true,
  platform: 'node',
  target: 'node20',
  outdir: 'dist',
  format: 'esm',
  minify: true,  // 代码最小化
  sourcemap: true,  // 生成 source map（方便调试）
  external: [
    '@aws-sdk/*',  // 排除 AWS SDK（运行时已包含）
  ],
  metafile: true,  // 生成元数据（分析包大小）
});

console.log('✅ Build complete!');
```

**执行打包：**

```bash
node build.mjs
```

**效果对比：**

| 方案 | 包大小 | 冷启动时间 |
|------|-------|-----------|
| 未打包（node_modules） | 50MB | 1200ms |
| Webpack 打包 | 8MB | 500ms |
| esbuild 打包 | 2MB | 300ms |
| esbuild + 移除未使用代码 | 500KB | 200ms |

## 💻 策略 2：使用 Lambda Layers

### 2.1 什么是 Lambda Layers？

**Lambda Layers** 是一种将公共代码（如依赖库）打包并在多个 Lambda 函数之间共享的机制。

**优势：**

```
┌────────────────────────────────────────────────────────┐
│           不使用 Layers（每个函数独立打包）             │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Function A (10MB)    Function B (10MB)   Function C   │
│  ├── Handler (1MB)    ├── Handler (1MB)   (10MB)      │
│  └── node_modules     └── node_modules    ├── Handler │
│      ├── aws-sdk          ├── aws-sdk         (1MB)   │
│      ├── lodash           ├── lodash      └── node_   │
│      └── ...              └── ...            modules  │
│                                                        │
│  总大小：30MB                                          │
│  部署时间：慢（每个函数单独上传）                       │
│  冷启动：慢（每次下载完整包）                           │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│              使用 Layers（共享依赖）                    │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Layer (9MB)                                           │
│  ├── node_modules                                      │
│  │   ├── aws-sdk                                       │
│  │   ├── lodash                                        │
│  │   └── ...                                           │
│  └── /opt/nodejs/node_modules/ (自动挂载)              │
│                                                        │
│  Function A (1MB)   Function B (1MB)   Function C (1MB)│
│  └── Handler        └── Handler        └── Handler     │
│                                                        │
│  总大小：12MB（减少 60%）                              │
│  部署时间：快（Layer 只需上传一次）                    │
│  冷启动：快（Layer 缓存在 AWS 基础设施中）             │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 2.2 创建 Lambda Layer

**步骤 1：准备依赖**

```bash
# 创建 layers 目录
mkdir -p layers/nodejs

# 安装依赖到 layers/nodejs 目录
cd layers/nodejs
npm init -y
npm install @aws-sdk/client-dynamodb @aws-sdk/lib-dynamodb lodash

# 返回项目根目录
cd ../..
```

**目录结构：**

```
layers/
└── nodejs/
    ├── package.json
    └── node_modules/
        ├── @aws-sdk/
        ├── lodash/
        └── ...
```

**注意：** 目录名必须是 `nodejs`（Node.js 运行时会自动从 `/opt/nodejs/node_modules/` 加载模块）。

---

**步骤 2：打包 Layer**

```bash
cd layers
zip -r nodejs-dependencies.zip nodejs/
```

---

**步骤 3：发布 Layer（使用 AWS CLI）**

```bash
aws lambda publish-layer-version \
  --layer-name nodejs-dependencies \
  --description "Common dependencies for Lambda functions" \
  --zip-file fileb://nodejs-dependencies.zip \
  --compatible-runtimes nodejs20.x \
  --region us-east-1
```

**输出：**

```json
{
  "LayerArn": "arn:aws:lambda:us-east-1:123456789012:layer:nodejs-dependencies",
  "LayerVersionArn": "arn:aws:lambda:us-east-1:123456789012:layer:nodejs-dependencies:1",
  "Version": 1
}
```

---

**步骤 4：在 Lambda 函数中使用 Layer**

```yaml
# serverless.yml
service: my-service

provider:
  name: aws
  runtime: nodejs20.x

layers:
  nodejsDependencies:
    path: layers
    name: ${self:service}-nodejs-dependencies
    description: Common Node.js dependencies
    compatibleRuntimes:
      - nodejs20.x

functions:
  myFunction:
    handler: src/handlers/myFunction.handler
    layers:
      - { Ref: NodejsDependenciesLambdaLayer }  # 引用 Layer

package:
  individually: true
  patterns:
    - '!node_modules/**'  # 排除 node_modules（使用 Layer 中的依赖）
```

---

### 2.3 Layer 最佳实践

**原则：**

1. **Layer 用于稳定依赖**：不经常变化的库（如 AWS SDK、lodash）
2. **业务代码不放在 Layer**：业务逻辑应该在函数代码中
3. **控制 Layer 大小**：单个 Layer < 50MB（解压后 < 250MB）
4. **版本管理**：每次更新 Layer 都会创建新版本，旧版本不会自动删除

**Layer 分层策略：**

```
Layer 1: Runtime Dependencies (aws-sdk, core libraries)
Layer 2: Utility Libraries (lodash, date-fns, validator)
Layer 3: Business Common Code (shared models, utilities)
```

## 💻 策略 3：预留并发（Provisioned Concurrency）

### 3.1 什么是预留并发？

**预留并发** 是 AWS Lambda 的一项功能，它会提前启动并保持一定数量的执行环境处于"热"状态，从而消除冷启动。

**工作原理：**

```
┌────────────────────────────────────────────────────────┐
│              预留并发 vs 按需并发                       │
├────────────────────────────────────────────────────────┤
│                                                        │
│  按需并发（默认）：                                     │
│  请求到来 → 检查是否有空闲容器 → 如果没有则冷启动      │
│                                                        │
│  预留并发：                                             │
│  提前启动 N 个容器 → 保持热状态 → 请求直接分配         │
│                                                        │
│  ┌────────────────────────────────────────────────┐   │
│  │  预留容器池（始终保持运行）                     │   │
│  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐       │   │
│  │  │ 容器1│  │ 容器2│  │ 容器3│  │ 容器4│       │   │
│  │  └──────┘  └──────┘  └──────┘  └──────┘       │   │
│  │    热        热        热        热            │   │
│  └────────────────────────────────────────────────┘   │
│         ↑请求立即分配到热容器（0 冷启动）              │
│                                                        │
│  如果预留容器不够用，会自动创建按需容器（有冷启动）     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

---

### 3.2 配置预留并发

**方法 1：AWS CLI**

```bash
# 设置预留并发为 5
aws lambda put-provisioned-concurrency-config \
  --function-name my-function \
  --provisioned-concurrent-executions 5 \
  --qualifier '$LATEST'
```

**方法 2：Serverless Framework**

```yaml
# serverless.yml
functions:
  myFunction:
    handler: src/handlers/myFunction.handler
    provisionedConcurrency: 5  # 预留 5 个并发
```

**方法 3：AWS Console**

```
Lambda 控制台 → 选择函数 → Configuration → Concurrency
→ Provisioned concurrency → Edit → 设置数量
```

### 3.3 成本分析

**定价模型：**

```
预留并发成本 = 预留时间 × 内存大小 × 单价

单价（us-east-1）：
- $0.0000041667 per GB-second

示例：
- 函数内存：512MB (0.5GB)
- 预留并发：5
- 运行时间：24小时/天

每月成本 = 5 × 0.5GB × 2,592,000秒 × $0.0000041667
         = $27.00/月

对比按需并发：
- 如果每分钟有 1 次请求，每次执行 200ms
- 每月请求：43,200 次
- 按需成本：~$0.50/月

结论：预留并发成本是按需的 54 倍！
```

**何时使用预留并发？**

✅ **适用场景：**
- 对延迟敏感的 API（如实时交易、游戏后端）
- 高频调用的函数（每秒多次）
- 需要保证 SLA 的服务

❌ **不适用场景：**
- 低频调用的函数（每小时几次）
- 成本敏感的项目
- 内部工具、定时任务

### 3.4 实战：渐进式预留并发

**场景：** API 在工作日 9:00-18:00 流量高，其他时间流量低。

**策略：** 使用 Application Auto Scaling 自动调整预留并发。

```bash
# 1. 注册可扩展目标
aws application-autoscaling register-scalable-target \
  --service-namespace lambda \
  --resource-id function:my-function:provisioned-concurrency:alias/prod \
  --scalable-dimension lambda:function:ProvisionedConcurrencyUtilization \
  --min-capacity 1 \
  --max-capacity 10

# 2. 创建扩缩容策略（目标利用率 70%）
aws application-autoscaling put-scaling-policy \
  --service-namespace lambda \
  --resource-id function:my-function:provisioned-concurrency:alias/prod \
  --scalable-dimension lambda:function:ProvisionedConcurrencyUtilization \
  --policy-name target-tracking-policy \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration \
    '{"TargetValue": 0.7, "PredefinedMetricSpecification": {"PredefinedMetricType": "LambdaProvisionedConcurrencyUtilization"}}'
```

**效果：**
- 流量高时自动增加预留并发（最多 10）
- 流量低时自动减少预留并发（最少 1）
- 降低 80%+ 成本

## 💻 策略 4：代码优化

### 4.1 懒加载依赖

**❌ 错误做法（启动时加载所有依赖）：**

```javascript
// 在模块顶层导入（即使某些请求不需要）
import { S3Client, GetObjectCommand } from '@aws-sdk/client-s3';
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';
import { SESClient } from '@aws-sdk/client-ses';

export const handler = async (event) => {
  // 只使用了 DynamoDB，但 S3 和 SES 也被加载了
  const dynamoClient = new DynamoDBClient({});
  // ...
};
```

**✅ 正确做法（按需加载）：**

```javascript
export const handler = async (event) => {
  // 根据请求类型动态导入
  if (event.action === 's3') {
    const { S3Client, GetObjectCommand } = await import('@aws-sdk/client-s3');
    const s3Client = new S3Client({});
    // ...
  } else if (event.action === 'dynamodb') {
    const { DynamoDBClient } = await import('@aws-sdk/client-dynamodb');
    const dynamoClient = new DynamoDBClient({});
    // ...
  }
};
```

**效果：**
- 减少 30-50% 初始化时间
- 降低内存占用

### 4.2 异步初始化

**❌ 错误做法（同步初始化数据库连接）：**

```javascript
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';

// 同步创建客户端（冷启动时执行）
const client = new DynamoDBClient({});

export const handler = async (event) => {
  // 使用 client
};
```

**✅ 正确做法（异步初始化 + 连接池）：**

```javascript
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';

let client = null;

async function getClient() {
  if (!client) {
    client = new DynamoDBClient({
      maxAttempts: 3,
      requestHandler: {
        // 配置 HTTP 连接池
        connectionTimeout: 1000,
        socketTimeout: 3000,
      },
    });
  }
  return client;
}

export const handler = async (event) => {
  const dynamoClient = await getClient();
  // 使用 dynamoClient
};
```

### 5.2 避免 VPC 冷启动

**VPC 配置的影响：**

```
无 VPC 配置：
- 冷启动：200-500ms

有 VPC 配置（旧版）：
- 冷启动：10-30秒（需要创建 ENI）

有 VPC 配置（新版 Hyperplane ENI）：
- 冷启动：500-1000ms（+300-500ms）
```

**优化建议：**

1. **避免不必要的 VPC 配置**：如果不需要访问 VPC 内资源（RDS、ElastiCache），就不要配置
2. **使用 RDS Proxy**：避免直接连接 RDS，减少连接开销
3. **使用 NAT Gateway**：确保 Lambda 可以访问互联网

## 💻 实战案例：电商 API 冷启动优化

### 优化前

**配置：**
- 包大小：80MB
- 依赖：300 个 npm 包
- 内存：256MB
- 无 Layers
- 无预留并发

**性能：**
- 冷启动：3500ms
- 热启动：25ms
- P99 延迟：4000ms（不可接受）

---

### 优化步骤

#### 步骤 1：使用 esbuild 打包（减少包大小）

```javascript
// build.mjs
import { build } from 'esbuild';

await build({
  entryPoints: ['src/handlers/*.mjs'],
  bundle: true,
  platform: 'node',
  target: 'node20',
  outdir: 'dist',
  format: 'esm',
  minify: true,
  external: ['@aws-sdk/*'],  // 排除 AWS SDK
});
```

**效果：** 包大小从 80MB → 5MB（减少 94%）

---

#### 步骤 2：创建 Lambda Layer（共享依赖）

```bash
# 安装公共依赖到 Layer
mkdir -p layers/nodejs
cd layers/nodejs
npm install @aws-sdk/client-dynamodb @aws-sdk/lib-dynamodb
cd ../..
zip -r nodejs-deps.zip layers/

# 发布 Layer
aws lambda publish-layer-version \
  --layer-name common-dependencies \
  --zip-file fileb://nodejs-deps.zip \
  --compatible-runtimes nodejs20.x
```

**效果：** 函数包大小从 5MB → 2MB（再减少 60%）

---

#### 步骤 3：提升内存（提高 CPU 性能）

```yaml
# serverless.yml
provider:
  memorySize: 512  # 从 256MB 提升到 512MB
```

**效果：** 冷启动时间从 2000ms → 800ms（减少 60%）

---

#### 步骤 4：代码优化（懒加载 + 异步初始化）

```javascript
// 优化前
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';
const client = new DynamoDBClient({});

// 优化后
let client = null;
async function getClient() {
  if (!client) {
    const { DynamoDBClient } = await import('@aws-sdk/client-dynamodb');
    client = new DynamoDBClient({});
  }
  return client;
}
```

**效果：** 冷启动时间从 800ms → 600ms（再减少 25%）

---

#### 步骤 5：关键接口使用预留并发

```yaml
# 只为登录接口配置预留并发（高频调用）
functions:
  login:
    handler: src/handlers/login.handler
    provisionedConcurrency: 3  # 预留 3 个并发
```

**效果：** 登录接口冷启动 ≈ 0ms

## ⚠️ 三个高频错误

### 错误 1：过度优化低频函数

**问题：** 为每分钟只调用 1 次的定时任务配置预留并发。

```yaml
# ❌ 错误：浪费成本
functions:
  dailyReport:
    handler: src/handlers/dailyReport.handler
    events:
      - schedule: cron(0 0 * * ? *)  # 每天执行 1 次
    provisionedConcurrency: 5  # 预留并发（完全不必要）
```

**成本：** $27/月（每月只执行 30 次，按需成本 < $0.01）

**正确做法：** 低频函数接受冷启动。

---

### 错误 2：把所有依赖放在 Layer

**问题：** Layer 过大（100MB+），导致 Layer 下载时间变长。

**正确做法：**
- Layer 只放稳定依赖（如 AWS SDK、lodash）
- 经常变化的业务代码放在函数包中

---

### 错误 3：忽略 VPC 冷启动影响

**问题：** 为了访问 RDS 配置了 VPC，但没有意识到冷启动增加 500ms+。

**优化方案：**
- 使用 RDS Proxy（减少连接开销）
- 使用 RDS Data API（无需 VPC）
- 使用 DynamoDB（无需 VPC）

## 📚 FAQ

### 1. Layer 和直接打包依赖哪个更快？

**答案：** Layer 更快（但有上限）。

- **Layer 优势：** 缓存在 AWS 基础设施中，下载速度快
- **Layer 劣势：** 单个 Layer 最大 50MB（解压后 250MB）

### 2. 预留并发和自动扩缩容有什么区别？

| 特性 | 预留并发 | 自动扩缩容 |
|------|---------|-----------|
| 冷启动 | 无冷启动 | 有冷启动 |
| 成本 | 按时间计费（贵） | 按请求计费（便宜） |
| 适用场景 | 对延迟敏感 | 成本敏感 |

### 3. 如何监控冷启动？

**方法 1：CloudWatch Logs Insights**

```sql
fields @timestamp, @duration, @initDuration
| filter @type = "REPORT"
| filter @initDuration > 0
| stats avg(@initDuration), max(@initDuration), count() by bin(5m)
```

**方法 2：X-Ray**

启用 X-Ray 追踪：

```yaml
# serverless.yml
provider:
  tracing:
    lambda: true
```

**方法 3：自定义 CloudWatch Metrics**

```javascript
import { CloudWatchClient, PutMetricDataCommand } from '@aws-sdk/client-cloudwatch';

const cloudwatch = new CloudWatchClient({});

export const handler = async (event, context) => {
  // 检测是否为冷启动
  const isColdStart = !global.isWarm;
  global.isWarm = true;

  if (isColdStart) {
    await cloudwatch.send(
      new PutMetricDataCommand({
        Namespace: 'Lambda/ColdStart',
        MetricData: [
          {
            MetricName: 'ColdStartCount',
            Value: 1,
            Unit: 'Count',
          },
        ],
      })
    );
  }

  // 业务逻辑...
};
```
