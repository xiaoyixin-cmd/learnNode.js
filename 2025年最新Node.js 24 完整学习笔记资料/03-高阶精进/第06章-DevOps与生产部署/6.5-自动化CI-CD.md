# 6.5 自动化 CI/CD：GitHub Actions

> 从代码提交到生产部署的完整自动化流程

---

## 📚 知识导读

### 什么是 CI/CD？

**CI（Continuous Integration）持续集成：** 频繁地将代码集成到主分支，通过自动化测试确保代码质量。

**CD（Continuous Delivery/Deployment）持续交付/部署：** 自动化将代码部署到生产环境。

**传统部署 vs CI/CD：**

```
┌────────────────────────────────────────────────────────┐
│              传统部署流程                               │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 开发者提交代码                                     │
│  2. 手动运行测试                                       │
│  3. 手动构建 Docker 镜像                               │
│  4. 手动推送镜像到仓库                                 │
│  5. 手动 SSH 到服务器                                  │
│  6. 手动拉取镜像                                       │
│  7. 手动重启容器                                       │
│                                                        │
│  ❌ 耗时：30-60 分钟                                   │
│  ❌ 容易出错（忘记步骤、配置错误）                     │
│  ❌ 难以回滚                                           │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│              CI/CD 自动化流程                           │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. git push（开发者提交代码）                         │
│     ↓                                                  │
│  2. 自动运行测试 ✅                                    │
│     ↓                                                  │
│  3. 自动构建镜像 ✅                                    │
│     ↓                                                  │
│  4. 自动推送镜像 ✅                                    │
│     ↓                                                  │
│  5. 自动部署到 K8s ✅                                  │
│     ↓                                                  │
│  6. 自动健康检查 ✅                                    │
│     ↓                                                  │
│  7. 自动通知（成功/失败）✅                            │
│                                                        │
│  ✅ 耗时：5-10 分钟                                    │
│  ✅ 零人为错误                                         │
│  ✅ 一键回滚                                           │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 🎯 GitHub Actions 基础

### 1. 工作流文件结构

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

# 触发条件
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# 环境变量
env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

# 任务
jobs:
  # 任务1：测试
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
      - name: Install dependencies
        run: npm ci
      - name: Run tests
        run: npm test

  # 任务2：构建和部署
  build-and-deploy:
    needs: test  # 依赖 test 任务
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker image
        run: docker build -t my-app .
      - name: Push to registry
        run: docker push my-app
      - name: Deploy to Kubernetes
        run: kubectl apply -f k8s/
```

---

### 2. 触发器（Triggers）

```yaml
on:
  # 1. Push 触发（特定分支）
  push:
    branches:
      - main
      - develop
    paths:
      - 'src/**'  # 只有 src 目录变化才触发
      - 'package.json'

  # 2. Pull Request 触发
  pull_request:
    branches:
      - main
    types:
      - opened
      - synchronize
      - reopened

  # 3. 定时触发（Cron）
  schedule:
    - cron: '0 2 * * *'  # 每天凌晨 2 点

  # 4. 手动触发
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production

  # 5. 标签触发
  create:
    tags:
      - 'v*'
```

## 💻 完整 CI/CD Pipeline

### 1. 测试阶段

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        node-version: [20, 22]
    
    steps:
      # 1. 检出代码
      - name: Checkout code
        uses: actions/checkout@v4
      
      # 2. 设置 Node.js
      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      
      # 3. 安装依赖（使用缓存）
      - name: Install dependencies
        run: npm ci
      
      # 4. 代码检查
      - name: Lint
        run: npm run lint
      
      # 5. 运行测试
      - name: Run tests
        run: npm test -- --coverage
      
      # 6. 上传覆盖率报告
      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          files: ./coverage/lcov.info
      
      # 7. 安全扫描
      - name: Security audit
        run: npm audit --audit-level=high
```

### 2. 构建 Docker 镜像

```yaml
# .github/workflows/build.yml
name: Build Docker Image

on:
  push:
    branches: [main]
    tags:
      - 'v*'

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    
    steps:
      # 1. 检出代码
      - name: Checkout code
        uses: actions/checkout@v4
      
      # 2. 设置 Docker Buildx（多架构构建）
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      # 3. 登录到 GitHub Container Registry
      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      # 4. 提取 Docker 元数据（标签）
      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix={{branch}}-
      
      # 5. 构建并推送镜像
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
      
      # 6. 扫描镜像漏洞
      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ steps.meta.outputs.version }}
          format: 'sarif'
          output: 'trivy-results.sarif'
      
      # 7. 上传扫描结果
      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v2
        with:
          sarif_file: 'trivy-results.sarif'
```

### 3. 部署到 Kubernetes

```yaml
# .github/workflows/deploy.yml
name: Deploy to Kubernetes

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ github.event.inputs.environment || 'staging' }}
    
    steps:
      # 1. 检出代码
      - name: Checkout code
        uses: actions/checkout@v4
      
      # 2. 配置 kubectl
      - name: Set up kubectl
        uses: azure/setup-kubectl@v3
        with:
          version: 'v1.29.0'
      
      # 3. 配置 Kubeconfig
      - name: Configure kubectl
        run: |
          mkdir -p $HOME/.kube
          echo "${{ secrets.KUBECONFIG }}" | base64 -d > $HOME/.kube/config
      
      # 4. 替换镜像标签
      - name: Update image tag
        run: |
          export IMAGE_TAG="${{ github.sha }}"
          envsubst < k8s/deployment.yaml.template > k8s/deployment.yaml
      
      # 5. 应用 Kubernetes 配置
      - name: Deploy to Kubernetes
        run: |
          kubectl apply -f k8s/ -n ${{ github.event.inputs.environment || 'staging' }}
      
      # 6. 等待部署完成
      - name: Wait for rollout
        run: |
          kubectl rollout status deployment/my-app -n ${{ github.event.inputs.environment || 'staging' }}
      
      # 7. 健康检查
      - name: Health check
        run: |
          sleep 10
          kubectl get pods -n ${{ github.event.inputs.environment || 'staging' }}
          
          # 测试应用是否可访问
          ENDPOINT=$(kubectl get ingress my-app-ingress -n ${{ github.event.inputs.environment || 'staging' }} -o jsonpath='{.spec.rules[0].host}')
          curl -f https://${ENDPOINT}/health || exit 1
      
      # 8. 发送通知
      - name: Notify deployment success
        if: success()
        uses: 8398a7/action-slack@v3
        with:
          status: custom
          text: '✅ Deployment to ${{ github.event.inputs.environment || 'staging' }} succeeded!'
          webhook_url: ${{ secrets.SLACK_WEBHOOK }}
      
      - name: Notify deployment failure
        if: failure()
        uses: 8398a7/action-slack@v3
        with:
          status: custom
          text: '❌ Deployment to ${{ github.event.inputs.environment || 'staging' }} failed!'
          webhook_url: ${{ secrets.SLACK_WEBHOOK }}
```

### 4. 多环境部署

```yaml
# .github/workflows/multi-env-deploy.yml
name: Multi-Environment Deployment

on:
  push:
    branches:
      - develop    # 自动部署到 staging
      - main       # 自动部署到 production

jobs:
  deploy-staging:
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to Staging
        run: |
          echo "Deploying to staging..."
          kubectl apply -f k8s/ -n staging
          kubectl set image deployment/my-app my-app=my-app:${{ github.sha }} -n staging
  
  deploy-production:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://api.example.com
    steps:
      - uses: actions/checkout@v4
      
      # 生产环境需要手动批准
      - name: Wait for approval
        uses: trstringer/manual-approval@v1
        with:
          secret: ${{ secrets.GITHUB_TOKEN }}
          approvers: admin-user
      
      - name: Deploy to Production
        run: |
          echo "Deploying to production..."
          kubectl apply -f k8s/ -n production
          kubectl set image deployment/my-app my-app=my-app:${{ github.sha }} -n production
```

## 💻 缓存优化

### 1. 缓存 npm 依赖

```yaml
steps:
  - name: Setup Node.js
    uses: actions/setup-node@v4
    with:
      node-version: '22'
      cache: 'npm'  # 自动缓存 node_modules
  
  - name: Install dependencies
    run: npm ci  # 使用 ci 而不是 install（更快、更可靠）
```

**效果：**
- 首次运行：安装依赖 2-3 分钟
- 后续运行（缓存命中）：安装依赖 10-30 秒

---

### 2. 缓存 Docker Layers

```yaml
- name: Build and push
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: my-app:latest
    cache-from: type=gha  # 从 GitHub Actions 缓存读取
    cache-to: type=gha,mode=max  # 写入缓存
```

**效果：**
- 首次构建：5-10 分钟
- 后续构建（缓存命中）：1-2 分钟

---

### 3. 手动缓存目录

```yaml
- name: Cache node modules
  uses: actions/cache@v3
  with:
    path: |
      ~/.npm
      node_modules
      **/node_modules
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-
```

## 💻 Secrets 管理

### 1. 添加 Secrets

**在 GitHub 仓库中添加：**

```
Settings → Secrets and variables → Actions → New repository secret
```

**常用 Secrets：**
- `KUBECONFIG`: Kubernetes 配置文件（base64 编码）
- `DOCKER_USERNAME`: Docker 用户名
- `DOCKER_PASSWORD`: Docker 密码
- `SLACK_WEBHOOK`: Slack 通知 Webhook URL
- `AWS_ACCESS_KEY_ID`: AWS 访问密钥
- `AWS_SECRET_ACCESS_KEY`: AWS 密钥

---

### 2. 使用 Secrets

```yaml
steps:
  - name: Login to Docker Hub
    uses: docker/login-action@v3
    with:
      username: ${{ secrets.DOCKER_USERNAME }}

      AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
      AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    run: |
      aws eks update-kubeconfig --name my-cluster
```

## 💻 矩阵构建（多版本测试）

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node-version: [20, 22]
        exclude:
          - os: macos-latest
            node-version: 20
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js ${{ matrix.node-version }} on ${{ matrix.os }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      
      - name: Run tests
        run: npm test
```

**生成的任务矩阵：**

```
✅ ubuntu-latest + Node.js 20
✅ ubuntu-latest + Node.js 22
❌ macos-latest + Node.js 20 (excluded)
✅ macos-latest + Node.js 22
✅ windows-latest + Node.js 20
✅ windows-latest + Node.js 22

总计：5 个并行任务
```

## 💻 条件执行

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    
    steps:
      # 1. 只在 main 分支执行
      - name: Deploy to production
        if: github.ref == 'refs/heads/main'
        run: echo "Deploying to production"
      
      # 2. 只在 PR 执行
      - name: Comment on PR
        if: github.event_name == 'pull_request'
        run: echo "This is a PR build"
      
      # 3. 只在成功时执行
      - name: Notify success
        if: success()
        run: echo "Build succeeded"
      
      # 4. 只在失败时执行
      - name: Notify failure
        if: failure()
        run: echo "Build failed"
      
      # 5. 总是执行（清理）
      - name: Cleanup
        if: always()
        run: echo "Cleaning up..."
      
      # 6. 自定义条件
      - name: Deploy if tagged
        if: startsWith(github.ref, 'refs/tags/v')
        run: echo "Deploying version ${{ github.ref }}"
```

## 💻 复用工作流（Reusable Workflows）

### 定义可复用工作流

```yaml
# .github/workflows/reusable-deploy.yml
name: Reusable Deploy

on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
      image-tag:
        required: true
        type: string
    secrets:
      KUBECONFIG:
        required: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to ${{ inputs.environment }}
        run: |
          echo "${{ secrets.KUBECONFIG }}" | base64 -d > kubeconfig
          kubectl --kubeconfig=kubeconfig set image deployment/my-app my-app=${{ inputs.image-tag }}
```

### 调用可复用工作流

```yaml
# .github/workflows/main.yml
name: Main Pipeline

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}
    steps:
      # ... 构建步骤
      - id: meta
        run: echo "version=${{ github.sha }}" >> $GITHUB_OUTPUT
  
  deploy-staging:
    needs: build
    uses: ./.github/workflows/reusable-deploy.yml
    with:
      environment: staging
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets:
      KUBECONFIG: ${{ secrets.STAGING_KUBECONFIG }}
  
  deploy-production:
    needs: build
    uses: ./.github/workflows/reusable-deploy.yml
    with:
      environment: production
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets:
      KUBECONFIG: ${{ secrets.PRODUCTION_KUBECONFIG }}
```

## ⚠️ 三个高频错误

### 错误 1：Secrets 未正确配置

**问题：** 忘记添加 Secret 或名称拼写错误。

```yaml
# ❌ 错误
- name: Deploy
  run: kubectl apply -f k8s/
  env:
    KUBECONFIG: ${{ secrets.KUBE_CONFIG }}  # 名称错误
```

**解决方案：** 检查 Secret 名称是否正确。

```yaml
# ✅ 正确
- name: Deploy
  run: kubectl apply -f k8s/
  env:
    KUBECONFIG: ${{ secrets.KUBECONFIG }}
```

---

### 错误 2：缓存键不正确导致缓存失效

```yaml
# ❌ 错误：缓存键固定，依赖变化不会刷新
- uses: actions/cache@v3
  with:
    path: node_modules
    key: node-modules-cache

# ✅ 正确：基于 package-lock.json 哈希
- uses: actions/cache@v3
  with:
    path: node_modules
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
```

---

### 错误 3：忘记设置环境批准

**问题：** 生产环境自动部署，没有人工批准。

```yaml
# ❌ 错误：直接部署到生产
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: kubectl apply -f k8s/ -n production
```

**解决方案：** 配置环境保护规则。

```yaml
# ✅ 正确：需要批准
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: production  # 在 GitHub Settings 中配置批准规则
      url: https://api.example.com
```

**在 GitHub 中配置：**

```
Settings → Environments → production → 
  ✅ Required reviewers (添加审批人)
  ✅ Wait timer (延迟部署时间)
```

## 📚 FAQ

### 1. 如何加速 CI/CD？

**优化策略：**

1. **并行执行任务**
   ```yaml
   jobs:
     test:
       # ...
     lint:
       # ...
     build:
       # ...  三个任务并行执行
   ```

2. **使用缓存**
   ```yaml
   - uses: actions/setup-node@v4
     with:
       cache: 'npm'
   ```

3. **减少依赖安装**
   ```yaml
   - run: npm ci --production  # 只安装生产依赖
   ```

4. **使用自托管 Runner（Self-hosted）**
   ```yaml
   runs-on: self-hosted  # 使用自己的服务器（更快、更便宜）
   ```

### 2. 如何调试失败的工作流？

```yaml
- name: Debug
  run: |
    echo "Event: ${{ github.event_name }}"
    echo "Ref: ${{ github.ref }}"
    echo "SHA: ${{ github.sha }}"
    env  # 打印所有环境变量

- name: SSH Debug
  if: failure()
  uses: mxschmitt/action-tmate@v3  # 远程 SSH 调试
```

### 3. 如何实现自动回滚？

```yaml
- name: Deploy
  id: deploy
  run: kubectl apply -f k8s/

- name: Health check
  run: |
    sleep 30
    if ! curl -f https://api.example.com/health; then
      echo "Health check failed, rolling back..."
      kubectl rollout undo deployment/my-app
      exit 1
    fi
```
