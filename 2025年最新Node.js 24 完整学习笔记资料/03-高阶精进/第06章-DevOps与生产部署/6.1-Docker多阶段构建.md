## 📚 知识导读

### 什么是 Docker 多阶段构建？

**Docker 多阶段构建（Multi-stage Build）** 允许在一个 Dockerfile 中使用多个 `FROM` 指令，每个 `FROM` 指令代表一个独立的构建阶段。最终镜像只包含最后一个阶段的内容，大幅减少镜像体积。

**单阶段 vs 多阶段构建：**

```
┌────────────────────────────────────────────────────────┐
│              单阶段构建（传统方式）                     │
├────────────────────────────────────────────────────────┤
│                                                        │
│  FROM node:22                                          │
│                                                        │
│  ┌──────────────────────────────────────────────┐     │
│  │  最终镜像包含：                               │     │
│  │  - Node.js 运行时                             │     │
│  │  - npm/yarn                                   │     │
│  │  - 构建工具（gcc、make、python）              │     │
│  │  - 源代码                                     │     │
│  │  - node_modules（包含 devDependencies）       │     │
│  │  - 构建产物（dist/）                          │     │
│  └──────────────────────────────────────────────┘     │
│                                                        │
│  ❌ 镜像大小：1.2GB                                    │
│  ❌ 包含不必要的构建工具和开发依赖                     │
│  ❌ 安全风险高（攻击面大）                             │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│              多阶段构建（现代方式）                     │
├────────────────────────────────────────────────────────┤
│                                                        │
│  阶段1：构建阶段（builder）                            │
│  FROM node:22 AS builder                               │
│  - 安装所有依赖（包括 devDependencies）                │
│  - 编译 TypeScript                                     │
│  - 打包资源                                            │
│                                                        │
│  阶段2：生产阶段（production）                         │
│  FROM node:22-alpine AS production                     │
│  - 只复制构建产物（dist/）                             │
│  - 只安装生产依赖（--production）                      │
│  - 不包含构建工具                                      │
│                                                        │
│  ┌──────────────────────────────────────────────┐     │
│  │  最终镜像包含：                               │     │
│  │  - Node.js 运行时（Alpine 精简版）            │     │
│  │  - 构建产物（dist/）                          │     │
│  │  - 生产依赖（node_modules - 仅 prod）         │     │
│  └──────────────────────────────────────────────┘     │
│                                                        │
│  ✅ 镜像大小：150MB（减少 87%）                        │
│  ✅ 只包含运行时必需文件                               │
│  ✅ 安全风险低（攻击面小）                             │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 🎯 Alpine Linux 优势

### 为什么选择 Alpine？

**Alpine Linux** 是一个轻量级 Linux 发行版，专为安全、简洁和资源效率设计。

**Alpine vs Debian 对比：**

| 特性 | Debian (默认) | Alpine |
|------|--------------|--------|
| 基础镜像大小 | ~120MB | ~5MB |
| Node.js 22 镜像 | ~1.1GB | ~180MB |
| C 库 | glibc | musl libc |
| 包管理器 | apt | apk |
| 安全性 | 中等 | 高（最小化攻击面） |
| 兼容性 | 高 | 中（部分 npm 包需要编译） |

**镜像大小对比（Node.js 22）：**

```bash
# Debian (默认)
node:22                 # 1.1GB
node:22-slim            # 240MB

# Alpine
node:22-alpine          # 180MB
node:22-alpine3.20      # 180MB（推荐，明确版本）
```

## 💻 基础 Dockerfile

### 1. 单阶段构建（不推荐）

```dockerfile
FROM node:22

WORKDIR /app

# 复制 package.json
COPY package*.json ./

# 安装所有依赖（包括 devDependencies）
RUN npm ci

# 复制源代码
COPY . .

# 暴露端口
EXPOSE 3000

# 启动应用
CMD ["node", "src/index.js"]
```

**构建和运行：**

```bash
# 构建镜像
docker build -t my-app:single-stage .

# 查看镜像大小
docker images my-app:single-stage
# REPOSITORY   TAG            SIZE
# my-app       single-stage   1.2GB

# 运行容器
docker run -p 3000:3000 my-app:single-stage
```

**问题：**
- ❌ 镜像体积巨大（1.2GB）
- ❌ 包含 devDependencies（如 TypeScript、Jest）
- ❌ 包含源代码（可能泄露商业逻辑）
- ❌ 启动慢（镜像拉取时间长）

---

### 2. 多阶段构建（推荐）

```dockerfile
# ============================================
# 阶段 1: 构建阶段
# ============================================
FROM node:22 AS builder

WORKDIR /app

# 复制依赖清单
COPY package*.json ./

# 安装所有依赖（包括 devDependencies）
RUN npm ci

# 复制源代码
COPY . .

# 如果是 TypeScript 项目，编译
RUN npm run build

# ============================================
# 阶段 2: 生产阶段
# ============================================
FROM node:22-alpine AS production

# 设置 Node 环境为生产
ENV NODE_ENV=production

WORKDIR /app

# 复制 package.json
COPY package*.json ./

# 只安装生产依赖
RUN npm ci --production

# 从构建阶段复制编译后的代码
COPY --from=builder /app/dist ./dist

# 暴露端口
EXPOSE 3000

# 启动应用
CMD ["node", "dist/index.js"]
```

**构建和运行：**

```bash
# 构建镜像
docker build -t my-app:multi-stage .

# 查看镜像大小
docker images my-app:multi-stage
# REPOSITORY   TAG            SIZE
# my-app       multi-stage    150MB

# 运行容器
docker run -p 3000:3000 my-app:multi-stage
```

**优势：**
- ✅ 镜像体积减少 87%（1.2GB → 150MB）
- ✅ 只包含生产依赖
- ✅ 不包含源代码
- ✅ 启动快（镜像拉取时间短）

## 💻 生产级 Dockerfile

### 完整示例（TypeScript + Express）

```dockerfile
# ============================================
# 阶段 1: 依赖安装阶段
# ============================================
FROM node:22-alpine3.20 AS deps

# 安装 dumb-init（用于正确处理信号）
RUN apk add --no-cache dumb-init

WORKDIR /app

# 复制依赖清单
COPY package.json package-lock.json ./

# 安装所有依赖（包括 devDependencies）
# 使用 npm ci 而不是 npm install（更快、更可靠）
RUN npm ci

# ============================================
# 阶段 2: 构建阶段
# ============================================
FROM node:22-alpine3.20 AS builder

WORKDIR /app

# 从依赖阶段复制 node_modules
COPY --from=deps /app/node_modules ./node_modules

# 复制源代码
COPY . .

# 构建 TypeScript
RUN npm run build

# 删除 devDependencies（减小后续复制大小）
RUN npm prune --production

# ============================================
# 阶段 3: 生产阶段
# ============================================
FROM node:22-alpine3.20 AS production

# 安装 dumb-init
RUN apk add --no-cache dumb-init

# 设置环境变量
ENV NODE_ENV=production \
    PORT=3000

WORKDIR /app

# 从依赖阶段复制 dumb-init
COPY --from=deps /usr/bin/dumb-init /usr/bin/dumb-init

# 从构建阶段复制生产依赖
COPY --from=builder /app/node_modules ./node_modules

# 从构建阶段复制编译后的代码
COPY --from=builder /app/dist ./dist

# 复制必要的配置文件
COPY --from=builder /app/package.json ./

# 暴露端口
EXPOSE 3000

# 使用 dumb-init 启动（正确处理 SIGTERM 信号）
ENTRYPOINT ["dumb-init", "--"]

# 启动应用
CMD ["node", "dist/index.js"]
```

## 💻 高级优化技巧

### 1. 利用构建缓存

**❌ 错误做法（缓存失效）：**

```dockerfile
FROM node:22-alpine AS builder

WORKDIR /app

# 先复制所有文件（源代码变化会导致缓存失效）
COPY . .

# 安装依赖（即使 package.json 没变，也要重新安装）
RUN npm ci

RUN npm run build
```

**✅ 正确做法（最大化缓存利用）：**

```dockerfile
FROM node:22-alpine AS builder

WORKDIR /app

# 1. 先复制 package.json（依赖未变时可复用缓存）
COPY package*.json ./

# 2. 安装依赖（只有 package.json 变化时才重新安装）
RUN npm ci

# 3. 再复制源代码（源代码变化不影响依赖安装缓存）
COPY . .

# 4. 构建
RUN npm run build
```

**效果对比：**

```bash
# 首次构建
# 两种方式时间相同：5 分钟

# 修改源代码后重新构建

# 错误做法：
# 缓存失效，重新安装依赖：5 分钟

# 正确做法：
# 复用依赖缓存，只重新编译：1 分钟
```

### 2. 多架构构建（支持 ARM64）

**问题：** 在 Apple Silicon (M1/M2) Mac 上构建的镜像无法在 x86_64 服务器上运行。

**解决方案：** 使用 Docker Buildx 构建多架构镜像。

```bash
# 1. 创建新的 builder 实例
docker buildx create --name multiarch --use

# 2. 启动 builder
docker buildx inspect multiarch --bootstrap

# 3. 构建并推送多架构镜像
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t your-registry.com/my-app:latest \
  --push \
  .

# 4. 验证
docker buildx imagetools inspect your-registry.com/my-app:latest
```

**Dockerfile 无需修改**（Alpine 镜像默认支持多架构）：

```dockerfile
FROM node:22-alpine3.20 AS builder
# ... 其他指令保持不变
```

### 3. 使用 BuildKit 构建优化

**启用 BuildKit（Docker 18.09+）：**

```bash
# 临时启用
DOCKER_BUILDKIT=1 docker build -t my-app .

# 永久启用（在 ~/.docker/daemon.json 中）
{
  "features": {
    "buildkit": true
  }
}
```

**BuildKit 优势：**
- ✅ 并行构建（多个 RUN 指令同时执行）
- ✅ 增量构建缓存（更智能的缓存机制）
- ✅ 安全性（自动使用密钥管理）

**使用 BuildKit 特性：**

```dockerfile
# 挂载构建缓存（加速 npm install）
FROM node:22-alpine AS builder

WORKDIR /app

COPY package*.json ./

# 使用缓存挂载（npm 缓存目录）
RUN --mount=type=cache,target=/root/.npm \
    npm ci

COPY . .

RUN npm run build
```

### 4. 精简生产镜像

**进一步减小镜像体积：**

```dockerfile
FROM node:22-alpine AS production

WORKDIR /app

# 只复制必要的文件
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

# 删除不必要的文件
RUN rm -rf /usr/local/lib/node_modules/npm \
    && rm -rf /root/.npm

# 使用 --no-cache 安装运行时依赖（如果有）
RUN apk add --no-cache dumb-init

EXPOSE 3000

CMD ["node", "dist/index.js"]
```

| Alpine + 多阶段 + 精简 | 120MB | 90% |

## ⚠️ 三个高频错误

### 错误 1：复制 node_modules 到镜像

**问题：** 直接复制本地 node_modules（可能是 macOS/Windows 编译的）到 Linux 容器。

```dockerfile
# ❌ 错误
COPY . .  # 包含 node_modules/
```

**解决方案：** 在 `.dockerignore` 中排除 node_modules，在容器内重新安装。

```
# .dockerignore
node_modules/
```

```dockerfile
# ✅ 正确
COPY package*.json ./
RUN npm ci
COPY . .
```

---

### 错误 2：使用 latest 标签

**问题：** `node:latest` 可能在不同时间拉取到不同版本，导致构建不一致。

```dockerfile
# ❌ 错误
FROM node:latest
```

**解决方案：** 明确指定版本。

```dockerfile
# ✅ 正确
FROM node:22-alpine3.20
```

---

### 错误 3：不处理信号（SIGTERM）

**问题：** Node.js 进程不响应 Docker 的 SIGTERM 信号，导致强制杀死进程。

```dockerfile
# ❌ 错误
CMD ["node", "dist/index.js"]
```

**问题表现：**
```bash
docker stop my-app
# 等待 10 秒后强制杀死（SIGKILL）
```

**解决方案：** 使用 dumb-init 或在代码中处理 SIGTERM。

```dockerfile
# ✅ 正确
RUN apk add --no-cache dumb-init
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/index.js"]
```

**或在代码中处理：**

```typescript
process.on('SIGTERM', () => {
  console.log('SIGTERM received');
  server.close(() => {
    process.exit(0);
  });
});
```

## 📚 FAQ

### 1. Alpine 有什么兼容性问题？

**问题：** 部分 npm 包（如 `bcrypt`、`sharp`）依赖 glibc，但 Alpine 使用 musl libc。

**解决方案：**

```dockerfile
# 安装编译依赖
RUN apk add --no-cache python3 make g++

# 或使用 alpine 版本的包
RUN npm install bcrypt@alpine
```

**或使用 Debian slim：**

```dockerfile
FROM node:22-slim
```

### 2. 如何调试多阶段构建？

**方法 1：构建到特定阶段**

```bash
# 只构建到 builder 阶段
docker build --target builder -t my-app:builder .

# 运行 builder 阶段容器
docker run -it my-app:builder sh
```

**方法 2：查看构建日志**

```bash
docker build --progress=plain -t my-app .
```

### 3. 如何减少构建时间？

**优化策略：**

1. **利用缓存挂载（BuildKit）：**
