# 6.2 非 root 用户运行容器

> 容器安全最佳实践：降低 90%+ 安全风险

---

## 📚 知识导读

### 为什么不能使用 root 用户？

**默认情况下，Docker 容器内的进程以 root 用户运行**，这带来严重的安全隐患。

**安全风险：**

```
┌────────────────────────────────────────────────────────┐
│           root 用户运行容器的风险                       │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 容器逃逸风险                                       │
│     如果攻击者通过漏洞逃逸出容器，将拥有主机 root 权限 │
│                                                        │
│  2. 挂载目录风险                                       │
│     容器可以修改挂载到宿主机的目录中的任何文件         │
│                                                        │
│  3. 权限提升风险                                       │
│     容器内的恶意代码可以修改关键系统文件               │
│                                                        │
│  4. 不符合安全标准                                     │
│     违反 CIS Docker Benchmark、Kubernetes Pod         │
│     Security Standards                                │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│         非 root 用户运行容器的优势                      │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 最小权限原则（Principle of Least Privilege）       │
│     进程只拥有完成任务所需的最小权限                   │
│                                                        │
│  2. 限制攻击面                                         │
│     即使容器被攻破，攻击者也只有普通用户权限           │
│                                                        │
│  3. 符合安全标准                                       │
│     满足 CIS、PCI-DSS、SOC 2 等安全合规要求           │
│                                                        │
│  4. 多租户环境安全                                     │
│     在 Kubernetes 等共享环境中更安全                  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**真实案例：**

```
2019 年 runC 容器逃逸漏洞（CVE-2019-5736）：
- 攻击者可以通过恶意镜像覆盖宿主机的 runC 二进制文件
- 如果容器以 root 运行，攻击者获得宿主机 root 权限
- 如果容器以非 root 运行，攻击失败（无权限写入）

影响：全球数百万容器
修复：升级 Docker + 使用非 root 用户
```

## 🎯 创建非 root 用户

### 方法 1：使用 node 镜像内置的 node 用户

**官方 Node.js 镜像已经包含一个名为 `node` 的非特权用户（UID 1000）。**

```dockerfile
FROM node:22-alpine

# 切换到 node 用户
USER node

WORKDIR /home/node/app

# 复制文件（注意权限）
COPY --chown=node:node package*.json ./

RUN npm ci --production

COPY --chown=node:node . .

EXPOSE 3000

CMD ["node", "index.js"]
```

**验证用户：**

```bash
docker run -it --rm node:22-alpine sh

# 查看当前用户
whoami  # node

# 查看 UID/GID
id
# uid=1000(node) gid=1000(node) groups=1000(node)

# 查看默认 shell
echo $HOME  # /home/node
```

---

### 方法 2：创建自定义用户

**如果需要特定的 UID/GID（如与宿主机用户匹配）：**

```dockerfile
FROM node:22-alpine

# 创建用户和组（UID/GID 1001）
RUN addgroup -g 1001 -S appuser && \
    adduser -u 1001 -S appuser -G appuser

# 切换到自定义用户
USER appuser

WORKDIR /home/appuser/app

COPY --chown=appuser:appuser package*.json ./

RUN npm ci --production

COPY --chown=appuser:appuser . .

EXPOSE 3000

CMD ["node", "index.js"]
```

**参数说明：**
- `-S`：创建系统用户（无密码、无登录 shell）
- `-u 1001`：指定 UID
- `-g 1001`：指定 GID
- `-G appuser`：将用户添加到组

---

### 方法 3：多阶段构建中的用户管理

```dockerfile
# ============================================
# 阶段 1: 构建阶段（使用 root 安装依赖）
# ============================================
FROM node:22-alpine AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# ============================================
# 阶段 2: 生产阶段（使用非 root 用户）
# ============================================
FROM node:22-alpine AS production

# 创建用户
RUN addgroup -g 1001 -S nodejs && \
    adduser -u 1001 -S nodejs -G nodejs

# 设置工作目录并授权
WORKDIR /app
RUN chown -R nodejs:nodejs /app

# 切换到非 root 用户
USER nodejs

# 复制文件（自动继承 nodejs 用户权限）
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --from=builder --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --chown=nodejs:nodejs package.json ./

EXPOSE 3000

CMD ["node", "dist/index.js"]
```

## 💻 文件权限处理

### 问题：非 root 用户的权限不足

**常见错误：**

```bash
docker run my-app

# 错误：EACCES: permission denied, open '/app/logs/app.log'
```

**原因：** 容器内的 `node` 用户（UID 1000）没有写入 `/app/logs` 的权限。

---

### 解决方案 1：在 Dockerfile 中授权

```dockerfile
FROM node:22-alpine

WORKDIR /app

# 创建日志目录并授权
RUN mkdir -p /app/logs && \
    chown -R node:node /app/logs

USER node

COPY --chown=node:node . .

RUN npm ci --production

EXPOSE 3000

CMD ["node", "index.js"]
```

---

### 解决方案 2：使用数据卷（推荐）

```dockerfile
FROM node:22-alpine

USER node

WORKDIR /home/node/app

COPY --chown=node:node . .

RUN npm ci --production

# 声明数据卷
VOLUME ["/home/node/app/logs"]

EXPOSE 3000

CMD ["node", "index.js"]
```

**运行时挂载数据卷：**

```bash
# 创建宿主机目录并设置权限
mkdir -p ./logs
chmod 777 ./logs  # 或设置为容器用户的 UID

# 运行容器
docker run -d \
  --name my-app \
  -v $(pwd)/logs:/home/node/app/logs \
  my-app
```

---

### 解决方案 3：使用 tmpfs（临时文件）

**对于不需要持久化的日志：**

```bash
docker run -d \
  --name my-app \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  my-app
```

**在代码中写入 /tmp：**

```javascript
import fs from 'fs';
import path from 'path';

const logDir = process.env.NODE_ENV === 'production' 
  ? '/tmp/logs' 
  : './logs';

fs.mkdirSync(logDir, { recursive: true });

const logFile = path.join(logDir, 'app.log');
fs.appendFileSync(logFile, 'Log message\n');
```

## 💻 只读根文件系统

### 为什么需要只读文件系统？

**只读根文件系统（Read-Only Root Filesystem）** 防止容器内的进程修改文件系统，进一步提升安全性。

```dockerfile
FROM node:22-alpine

USER node

WORKDIR /home/node/app

COPY --chown=node:node . .

RUN npm ci --production

# 声明需要写入的目录
VOLUME ["/home/node/app/tmp"]

EXPOSE 3000

CMD ["node", "index.js"]
```

**运行容器（启用只读根文件系统）：**

```bash
docker run -d \
  --name my-app \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  my-app
```

**效果：**

```bash
# 进入容器
docker exec -it my-app sh

# 尝试写入根文件系统
echo "test" > /test.txt
# ash: can't create /test.txt: Read-only file system

# 可以写入 tmpfs
echo "test" > /tmp/test.txt  # 成功
```

## 💻 完整的安全加固 Dockerfile

### 生产级示例

```dockerfile
# ============================================
# 阶段 1: 依赖安装
# ============================================
FROM node:22-alpine3.20 AS deps

# 安装安全工具
RUN apk add --no-cache \
    dumb-init=1.2.5-r2 \
    curl=8.5.0-r0

WORKDIR /app

COPY package.json package-lock.json ./

# 使用缓存挂载加速安装
RUN --mount=type=cache,target=/root/.npm \
    npm ci --production

# ============================================
# 阶段 2: 构建
# ============================================
FROM node:22-alpine3.20 AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build && \
    npm prune --production

# ============================================
# 阶段 3: 生产运行（安全加固）
# ============================================
FROM node:22-alpine3.20 AS production

# 1. 安装运行时依赖（固定版本）
RUN apk add --no-cache \
    dumb-init=1.2.5-r2 \
    curl=8.5.0-r0

# 2. 创建非 root 用户
RUN addgroup -g 1001 -S nodejs && \
    adduser -u 1001 -S nodejs -G nodejs

# 3. 设置环境变量
ENV NODE_ENV=production \
    PORT=3000 \
    NPM_CONFIG_LOGLEVEL=warn

# 4. 设置工作目录
WORKDIR /app

# 5. 创建必要的目录并设置权限
RUN mkdir -p /app/tmp /app/logs && \
    chown -R nodejs:nodejs /app

# 6. 切换到非 root 用户（在复制文件之前）
USER nodejs

# 7. 复制文件（自动使用 nodejs 用户权限）
COPY --from=builder --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --chown=nodejs:nodejs package.json ./

# 8. 暴露端口（非特权端口）
EXPOSE 3000

# 9. 健康检查（使用 curl）
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# 10. 声明数据卷
VOLUME ["/app/logs"]

# 11. 使用 dumb-init 启动（处理信号）
ENTRYPOINT ["dumb-init", "--"]

# 12. 启动应用
CMD ["node", "dist/index.js"]

# ============================================
# 安全标签
# ============================================
LABEL maintainer="your-email@example.com" \
      org.opencontainers.image.title="My Secure App" \
      org.opencontainers.image.version="1.0.0" \
      org.opencontainers.image.vendor="Your Company" \
      security.scan.date="2025-10-07"
```

## 💻 Kubernetes Pod Security Standards

### Pod Security Context

**在 Kubernetes 中强制使用非 root 用户：**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  # Pod 级别的安全上下文
  securityContext:
    runAsNonRoot: true        # 强制非 root 运行
    runAsUser: 1001           # 指定 UID
    runAsGroup: 1001          # 指定 GID
    fsGroup: 1001             # 文件系统组 ID
    seccompProfile:
      type: RuntimeDefault    # 启用 seccomp

  containers:
  - name: app
    image: my-app:latest
    # 容器级别的安全上下文
    securityContext:
      allowPrivilegeEscalation: false  # 禁止权限提升
      readOnlyRootFilesystem: true     # 只读根文件系统
      capabilities:
        drop:
        - ALL                          # 删除所有 Linux capabilities
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: logs
      mountPath: /app/logs

  volumes:
  - name: tmp
    emptyDir: {}
  - name: logs
    emptyDir: {}
```

**验证：**

```bash
# 部署 Pod
kubectl apply -f pod.yaml

# 检查 Pod 安全上下文
kubectl get pod my-app -o jsonpath='{.spec.securityContext}'

# 进入容器验证用户
kubectl exec -it my-app -- whoami
# nodejs (UID 1001)

# 验证只读文件系统
kubectl exec -it my-app -- sh -c 'touch /test'
# touch: /test: Read-only file system
```

## ⚠️ 三个高频错误

### 错误 1：在切换用户后执行需要 root 权限的操作

**问题：**

```dockerfile
FROM node:22-alpine

USER node  # 切换到 node 用户

# ❌ 错误：node 用户没有权限安装系统包
RUN apk add --no-cache curl
```

**错误信息：**

```
ERROR: Unable to lock database: Permission denied
ERROR: Failed to open apk database: Permission denied
```

**解决方案：** 在切换用户之前完成所有需要 root 权限的操作。

```dockerfile
FROM node:22-alpine

# ✅ 正确：先安装系统包（使用 root）
RUN apk add --no-cache curl dumb-init

# 然后切换到 node 用户
USER node

WORKDIR /home/node/app
```

---

### 错误 2：目录权限不正确

**问题：**

```dockerfile
FROM node:22-alpine

WORKDIR /app

USER node

# ❌ 错误：/app 目录属于 root，node 用户无法写入
COPY package*.json ./
RUN npm ci
```

**错误信息：**

```
npm ERR! Error: EACCES: permission denied, mkdir '/app/node_modules'
```

**解决方案：** 使用 `--chown` 参数或在切换用户前创建并授权目录。

```dockerfile
FROM node:22-alpine

# 方案 1：使用 --chown
WORKDIR /app
USER node
COPY --chown=node:node package*.json ./
RUN npm ci

# 方案 2：提前授权
WORKDIR /app
RUN chown -R node:node /app
USER node
COPY package*.json ./
RUN npm ci
```

---

### 错误 3：暴露特权端口（< 1024）

**问题：**

```dockerfile
FROM node:22-alpine

USER node

EXPOSE 80  # ❌ 错误：非 root 用户无法绑定特权端口
CMD ["node", "index.js"]
```

**错误信息：**

```
Error: listen EACCES: permission denied 0.0.0.0:80
```

**解决方案：** 使用非特权端口（≥ 1024）。

```dockerfile
FROM node:22-alpine

USER node

EXPOSE 3000  # ✅ 正确：使用非特权端口
CMD ["node", "index.js"]
```

**如果必须使用 80 端口，在 Kubernetes/Docker 中进行端口映射：**

```bash
# Docker
docker run -p 80:3000 my-app

# Kubernetes Service
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  ports:
  - port: 80        # 外部端口
    targetPort: 3000  # 容器端口
```

## 🎯 应用场景

### 场景 1：金融行业应用

**需求：** 符合 PCI-DSS 安全标准。

**安全措施：**
1. 非 root 用户运行
2. 只读根文件系统
3. 删除所有 Linux capabilities
4. 启用 seccomp 和 AppArmor

**Dockerfile：**

```dockerfile
FROM node:22-alpine AS production

RUN addgroup -g 1001 -S finapp && \
    adduser -u 1001 -S finapp -G finapp

WORKDIR /app
RUN chown finapp:finapp /app

USER finapp

COPY --chown=finapp:finapp . .
RUN npm ci --production

EXPOSE 3000

CMD ["node", "index.js"]
```

**Kubernetes 配置：**

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1001
  readOnlyRootFilesystem: true
  capabilities:
    drop: [ALL]
  seccompProfile:
    type: RuntimeDefault
```

---

### 场景 2：多租户 SaaS 平台

**需求：** 隔离不同租户的容器。

**安全策略：**
1. 每个租户使用不同的 UID
2. Pod Security Standards (Restricted)
3. Network Policies 限制网络访问

**部署：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: tenant-app
spec:
  template:
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 2001  # 租户特定 UID
        fsGroup: 2001
      containers:
      - name: app
        image: my-app:latest
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
```

## 📚 FAQ

### 1. 如何在本地开发时使用非 root 用户？

**问题：** 本地挂载的文件权限不匹配。

**解决方案：** 使用与宿主机相同的 UID/GID。

```bash
# 查看宿主机用户 UID
id -u  # 例如：1000

# 构建时传递 UID
docker build --build-arg USER_ID=$(id -u) -t my-app .
```

```dockerfile
FROM node:22-alpine

ARG USER_ID=1000

RUN addgroup -g ${USER_ID} -S appuser && \
    adduser -u ${USER_ID} -S appuser -G appuser

USER appuser
```

### 2. 如何验证容器是否以非 root 运行？

```bash
# 方法 1：检查运行的进程
docker exec my-app ps aux
# USER       PID
# node         1  node index.js

# 方法 2：进入容器检查
docker exec my-app whoami
# node

# 方法 3：使用 docker inspect
docker inspect my-app | jq '.[0].Config.User'
# "node"
```

### 3. 如何在 CI/CD 中强制执行安全策略？

**使用 OPA（Open Policy Agent）：**

```rego
# policy.rego
package docker.security

deny[msg] {
  input.User == ""
  msg = "Container must not run as root"
}

deny[msg] {
  input.User == "root"
  msg = "Container must not run as root"
}
```

**在 CI 中检查：**

```bash
# 使用 conftest
conftest test Dockerfile --policy policy.rego
```
