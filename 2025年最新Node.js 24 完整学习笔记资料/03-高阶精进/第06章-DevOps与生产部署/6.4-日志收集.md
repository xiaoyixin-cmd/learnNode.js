# 6.4 日志收集：Fluentd / Loki + Grafana

> 构建企业级日志收集与可视化系统

---

## 📚 知识导读

### 为什么需要集中式日志收集？

**传统日志 vs 集中式日志：**

```
┌────────────────────────────────────────────────────────┐
│           传统日志（分散在各个容器）                    │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Pod 1: kubectl logs pod1                             │
│  Pod 2: kubectl logs pod2                             │
│  Pod 3: kubectl logs pod3                             │
│  ...                                                   │
│  Pod 100: kubectl logs pod100                         │
│                                                        │
│  ❌ 问题：                                             │
│  - 需要逐个查看 Pod 日志                               │
│  - Pod 重启后日志丢失                                  │
│  - 无法跨 Pod 搜索日志                                 │
│  - 难以发现分布式问题                                  │
│  - 无法长期存储                                        │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│              集中式日志收集系统                         │
├────────────────────────────────────────────────────────┤
│                                                        │
│  所有 Pod → 日志采集器 → 日志存储 → 可视化界面        │
│                                                        │
│  ✅ 优势：                                             │
│  - 统一入口查询日志                                    │
│  - 持久化存储（Pod 重启不丢失）                        │
│  - 全文搜索、正则匹配                                  │
│  - 日志聚合、统计分析                                  │
│  - 实时告警（错误率、关键字）                          │
│  - 分布式追踪（Trace ID）                             │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 日志收集方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **ELK Stack** (Elasticsearch + Logstash + Kibana) | 功能强大、全文搜索、成熟 | 资源消耗大、成本高 | 大型企业 |
| **Loki + Promtail + Grafana** | 轻量级、低成本、与 Prometheus 集成 | 功能相对简单 | 中小型项目 |
| **Fluentd + Elasticsearch + Kibana** | 灵活、插件丰富 | 配置复杂 | 需要定制化 |
| **CloudWatch Logs** (AWS) | 托管、零运维 | 厂商锁定、成本高 | AWS 环境 |

### 2. 配置 Promtail（日志采集器）

**Promtail 配置文件：**

```yaml
# promtail-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: promtail-config
  namespace: monitoring
data:
  promtail.yaml: |
    server:
      http_listen_port: 3101
    
    positions:
      filename: /tmp/positions.yaml
    
    clients:
      - url: http://loki:3100/loki/api/v1/push
    
    scrape_configs:
      # 抓取所有 Pod 日志
      - job_name: kubernetes-pods
        kubernetes_sd_configs:
          - role: pod
        
        relabel_configs:
          # 添加命名空间标签
          - source_labels: [__meta_kubernetes_pod_namespace]
            target_label: namespace
          
          # 添加 Pod 名称标签
          - source_labels: [__meta_kubernetes_pod_name]
            target_label: pod
          
          # 添加容器名称标签
          - source_labels: [__meta_kubernetes_pod_container_name]
            target_label: container
          
          # 添加应用标签
          - source_labels: [__meta_kubernetes_pod_label_app]
            target_label: app
        
        pipeline_stages:
          # 解析 JSON 日志
          - json:
              expressions:
                level: level
                message: message
                timestamp: timestamp
          
          # 提取日志级别
          - labels:
              level:
          
          # 添加时间戳
          - timestamp:
              source: timestamp
              format: RFC3339
```

### 4. 在 Grafana 中查询日志

**进入 Explore 页面，使用 LogQL 查询语言：**

```logql
# 1. 查询特定命名空间的日志
{namespace="production"}

# 2. 查询特定应用的日志
{app="my-app"}

# 3. 查询特定 Pod 的日志
{pod="my-app-5d5c8f9b4d-7x8qz"}

# 4. 查询包含错误的日志
{app="my-app"} |= "error"

# 5. 查询 JSON 日志中的 level 字段
{app="my-app"} | json | level="error"

# 6. 统计错误日志数量
sum(count_over_time({app="my-app"} |= "error" [5m]))

# 7. 查询最近 1 小时的日志
{app="my-app"} [1h]

# 8. 正则匹配
{app="my-app"} |~ "user.*login.*failed"
```

### 2. 部署 Fluentd

**Fluentd 配置：**

```yaml
# fluentd-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluentd-config
  namespace: logging
data:
  fluent.conf: |
    # 输入：从 Kubernetes 容器日志读取
    <source>
      @type tail
      @id in_tail_container_logs
      path /var/log/containers/*.log
      pos_file /var/log/fluentd-containers.log.pos
      tag kubernetes.*
      read_from_head true
      <parse>
        @type json
        time_format %Y-%m-%dT%H:%M:%S.%NZ
      </parse>
    </source>
    
    # 过滤：解析 Kubernetes 元数据
    <filter kubernetes.**>
      @type kubernetes_metadata
      @id filter_kube_metadata
      kubernetes_url "#{ENV['FLUENT_FILTER_KUBERNETES_URL'] || 'https://' + ENV.fetch('KUBERNETES_SERVICE_HOST') + ':' + ENV.fetch('KUBERNETES_SERVICE_PORT') + '/api'}"
      verify_ssl "#{ENV['KUBERNETES_VERIFY_SSL'] || true}"
      ca_file "#{ENV['KUBERNETES_CA_FILE']}"
    </filter>
    
    # 过滤：排除系统命名空间
    <filter kubernetes.**>
      @type grep
      <exclude>
        key $.kubernetes.namespace_name
        pattern /^(kube-system|kube-public|kube-node-lease)$/
      </exclude>
    </filter>
    
    # 输出：发送到 Elasticsearch
    <match kubernetes.**>
      @type elasticsearch
      @id out_es
      @log_level info
      include_tag_key true
      host elasticsearch.logging.svc.cluster.local
      port 9200
      logstash_format true
      logstash_prefix kubernetes
      <buffer>
        @type file
        path /var/log/fluentd-buffers/kubernetes.system.buffer
        flush_mode interval
        retry_type exponential_backoff
        flush_thread_count 2
        flush_interval 5s
        retry_forever
        retry_max_interval 30
        chunk_limit_size 2M
        queue_limit_length 8
        overflow_action block
      </buffer>
    </match>
```

## 💻 应用日志最佳实践

### 1. 结构化日志（JSON 格式）

**推荐使用 Pino（高性能 JSON 日志库）：**

```javascript
// logger.js
import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level: (label) => {
      return { level: label.toUpperCase() };
    },
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});

export default logger;
```

**使用示例：**

```javascript
import logger from './logger.js';

// 基础日志
logger.info('Server started');

// 带上下文的日志
logger.info({ userId: '123', action: 'login' }, 'User logged in');

// 错误日志
try {
  throw new Error('Database connection failed');
} catch (error) {
  logger.error({ err: error }, 'Database error');
}

// 输出：
// {"level":"INFO","time":"2025-10-07T12:00:00.000Z","msg":"Server started"}
// {"level":"INFO","time":"2025-10-07T12:00:01.000Z","userId":"123","action":"login","msg":"User logged in"}
// {"level":"ERROR","time":"2025-10-07T12:00:02.000Z","err":{"type":"Error","message":"Database connection failed","stack":"..."},"msg":"Database error"}
```

### 2. 添加 Trace ID（分布式追踪）

```javascript
import { v4 as uuidv4 } from 'uuid';
import logger from './logger.js';

// Express 中间件
export function traceMiddleware(req, res, next) {
  req.traceId = req.headers['x-trace-id'] || uuidv4();
  
  // 添加到响应头
  res.setHeader('X-Trace-ID', req.traceId);
  
  // 创建带 traceId 的子 logger
  req.log = logger.child({ traceId: req.traceId });
  
  next();
}

// 使用
app.use(traceMiddleware);

app.get('/users/:id', async (req, res) => {
  req.log.info({ userId: req.params.id }, 'Fetching user');
  
  const user = await db.getUser(req.params.id);
  
  req.log.info({ userId: req.params.id }, 'User fetched successfully');
  
  res.json(user);
});

// 输出：
// {"level":"INFO","traceId":"abc-123","userId":"456","msg":"Fetching user"}
// {"level":"INFO","traceId":"abc-123","userId":"456","msg":"User fetched successfully"}
```

## ⚠️ 三个高频错误

### 错误 1：日志过多导致存储爆满

**问题：** Debug 日志在生产环境输出，导致每天产生数百 GB 日志。

**解决方案：**

```javascript
// ❌ 错误
console.log('Debug:', someObject);  // 所有环境都输出

// ✅ 正确：根据环境设置日志级别
const logger = pino({
  level: process.env.NODE_ENV === 'production' ? 'warn' : 'debug',
});

logger.debug('Debug info');  // 生产环境不输出
logger.warn('Warning');      // 生产环境输出
```

**Elasticsearch 索引生命周期管理：**

```json
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50GB",
            "max_age": "7d"
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

---

### 错误 2：敏感信息泄露

**问题：** 日志中包含密码、Token 等敏感信息。

```javascript
// ❌ 错误
logger.info({ user: { name: 'Alice', password: '123456' } }, 'User created');
```

**解决方案：** 过滤敏感字段。

```javascript
// ✅ 正确
import pino from 'pino';

const logger = pino({
  redact: {
    paths: ['password', 'token', 'apiKey', '*.password', 'req.headers.authorization'],
    remove: true,
  },
});

logger.info({ user: { name: 'Alice', password: '123456' } }, 'User created');
// 输出: {"level":"INFO","user":{"name":"Alice"},"msg":"User created"}
```

---

### 错误 3：Fluentd/Promtail 资源不足

**问题：** 日志采集器 CPU/内存不足，导致日志丢失。

**解决方案：** 增加资源限制。

```yaml
# ❌ 错误：资源太少
resources:
  requests:
    memory: "50Mi"
    cpu: "50m"

# ✅ 正确：根据日志量调整
resources:
  requests:
    memory: "200Mi"
    cpu: "100m"
  limits:
    memory: "500Mi"
    cpu: "500m"
```

## 📚 FAQ

### 1. Loki vs Elasticsearch 如何选择？

| 特性 | Loki | Elasticsearch |
|------|------|--------------|
| 存储成本 | 低（只索引标签） | 高（全文索引） |
| 查询速度 | 快（标签查询） | 快（全文搜索） |
| 功能 | 简单 | 强大 |
| 资源消耗 | 低 | 高 |
| 适用场景 | 中小型项目 | 大型企业 |

```yaml
    groups:
      - name: application_alerts
        interval: 1m
        rules:
          - alert: HighErrorRate
            expr: |
              sum(rate({app="my-app"} |= "error" [5m])) > 10
            for: 5m
            labels:
              severity: critical
            annotations:
              summary: "High error rate detected"
              description: "Error rate is {{ $value }} errors/second"
```

### 3. 如何实现日志轮转？

**Loki 自动轮转（配置保留期）：**

```yaml
# loki-config.yaml
limits_config:
  retention_period: 30d  # 保留 30 天

chunk_store_config:
  max_look_back_period: 30d
```

**Elasticsearch ILM（索引生命周期管理）：**

```bash
# 创建 ILM 策略
PUT _ilm/policy/logs_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_age": "7d",
            "max_size": "50GB"
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```
