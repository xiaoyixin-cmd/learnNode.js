## 🎯 蓝绿部署实现

### 1. 使用 Kubernetes Service 切换

**部署两套环境：**

```yaml
# deployment-blue.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-blue
  labels:
    app: my-app
    version: blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: blue
  template:
    metadata:
      labels:
        app: my-app
        version: blue
    spec:
      containers:
      - name: app
        image: my-app:v1.0.0
        ports:
        - containerPort: 3000
---
# deployment-green.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-green
  labels:
    app: my-app
    version: green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: green
  template:
    metadata:
      labels:
        app: my-app
        version: green
    spec:
      containers:
      - name: app
        image: my-app:v2.0.0  # 新版本
        ports:
        - containerPort: 3000
```

**Service 配置（通过修改 selector 切换）：**

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
    version: blue  # 当前指向蓝环境
  ports:
  - port: 80
    targetPort: 3000
```

**切换流程：**

```bash
# 1. 部署蓝环境（当前版本）
kubectl apply -f deployment-blue.yaml

# 2. 部署绿环境（新版本）
kubectl apply -f deployment-green.yaml

# 3. 等待绿环境就绪
kubectl rollout status deployment/my-app-green

# 4. 测试绿环境
kubectl run test --image=curlimages/curl -i --rm --restart=Never -- \
  curl http://my-app-green-service/health

# 5. 切换流量到绿环境
kubectl patch service my-app-service -p '{"spec":{"selector":{"version":"green"}}}'

# 6. 如果有问题，立即回滚到蓝环境
kubectl patch service my-app-service -p '{"spec":{"selector":{"version":"blue"}}}'

# 7. 确认无问题后，删除蓝环境
kubectl delete deployment my-app-blue
```

### 3. 使用脚本自动化蓝绿部署

```bash
#!/bin/bash
# blue-green-deploy.sh

set -e

NEW_VERSION=$1
CURRENT_COLOR=$(kubectl get service my-app-service -o jsonpath='{.spec.selector.version}')

if [ "$CURRENT_COLOR" == "blue" ]; then
  NEW_COLOR="green"
else
  NEW_COLOR="blue"
fi

echo "Current environment: $CURRENT_COLOR"
echo "Deploying to: $NEW_COLOR"

# 1. 部署新版本到新颜色环境
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-${NEW_COLOR}
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: ${NEW_COLOR}
  template:
    metadata:
      labels:
        app: my-app
        version: ${NEW_COLOR}
    spec:
      containers:
      - name: app
        image: my-app:${NEW_VERSION}
        ports:
        - containerPort: 3000
EOF

# 2. 等待新环境就绪
kubectl rollout status deployment/my-app-${NEW_COLOR}

# 3. 运行健康检查
echo "Running health checks on ${NEW_COLOR} environment..."
TEMP_SERVICE_PORT=$(kubectl get service my-app-${NEW_COLOR}-service -o jsonpath='{.spec.ports[0].nodePort}' 2>/dev/null || echo "30080")

for i in {1..5}; do
  if curl -f http://localhost:${TEMP_SERVICE_PORT}/health; then
    echo "Health check passed ($i/5)"
  else
    echo "Health check failed, rolling back..."
    exit 1
  fi
  sleep 2
done

# 4. 切换流量
echo "Switching traffic to ${NEW_COLOR}..."
kubectl patch service my-app-service -p "{\"spec\":{\"selector\":{\"version\":\"${NEW_COLOR}\"}}}"

echo "Deployment complete!"
echo "Old environment (${CURRENT_COLOR}) is still running for rollback."
echo "To delete old environment: kubectl delete deployment my-app-${CURRENT_COLOR}"
```

**使用：**

```bash
chmod +x blue-green-deploy.sh
./blue-green-deploy.sh v2.0.0
```

### 1. 使用 Istio Service Mesh

**安装 Istio：**

```bash
# 下载 Istio
curl -L https://istio.io/downloadIstio | sh -
cd istio-1.20.0

# 安装
istioctl install --set profile=default -y

# 启用自动注入
kubectl label namespace default istio-injection=enabled
```

**部署两个版本：**

```yaml
# deployment-v1.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: v1
  template:
    metadata:
      labels:
        app: my-app
        version: v1
    spec:
      containers:
      - name: app
        image: my-app:v1.0.0
        ports:
        - containerPort: 3000
---
# deployment-v2.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v2
spec:
  replicas: 1  # 初始只部署1个副本
  selector:
    matchLabels:
      app: my-app
      version: v2
  template:
    metadata:
      labels:
        app: my-app
        version: v2
    spec:
      containers:
      - name: app
        image: my-app:v2.0.0
        ports:
        - containerPort: 3000
```

**Istio VirtualService（流量分配）：**

```yaml
# virtualservice.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app
  http:
  - match:
    - headers:
        canary:
          exact: "true"  # 带 canary header 的请求到 v2
    route:
    - destination:
        host: my-app
        subset: v2
  - route:
    - destination:
        host: my-app
        subset: v1
      weight: 90  # 90% 流量到 v1
    - destination:
        host: my-app
        subset: v2
      weight: 10  # 10% 流量到 v2
---
# destinationrule.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: my-app
spec:
  host: my-app
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

**渐进式增加流量：**

```bash
# 步骤1: 10% 流量
kubectl apply -f virtualservice.yaml  # weight: 10

# 监控指标...

# 步骤2: 50% 流量
kubectl patch virtualservice my-app --type='json' \
  -p='[{"op": "replace", "path": "/spec/http/1/route/0/weight", "value":50},
       {"op": "replace", "path": "/spec/http/1/route/1/weight", "value":50}]'

# 监控指标...

# 步骤3: 100% 流量
kubectl patch virtualservice my-app --type='json' \
  -p='[{"op": "replace", "path": "/spec/http/1/route/0/weight", "value":0},
       {"op": "replace", "path": "/spec/http/1/route/1/weight", "value":100}]'
```

### 2. 使用 Flagger 自动化金丝雀发布

**Flagger** 是一个自动化渐进式交付工具，支持 Istio、Linkerd、App Mesh 等。

**安装 Flagger：**

```bash
# 添加 Flagger Helm 仓库
helm repo add flagger https://flagger.app

# 安装 Flagger（使用 Istio）
helm upgrade -i flagger flagger/flagger \
  --namespace istio-system \
  --set meshProvider=istio \
  --set metricsServer=http://prometheus:9090
```

**创建 Canary 资源：**

```yaml
# canary.yaml
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: my-app
  namespace: default
spec:
  # 目标 Deployment
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  
  # 自动扩缩容
  autoscalerRef:
    apiVersion: autoscaling/v2
    kind: HorizontalPodAutoscaler
    name: my-app
  
  # Service 配置
  service:
    port: 80
    targetPort: 3000
  
  # 金丝雀分析
  analysis:
    # 检查间隔
    interval: 1m
    # 成功阈值（连续成功5次才推进）
    threshold: 5
    # 最大权重
    maxWeight: 50
    # 每次增加的权重
    stepWeight: 10
    
    # 监控指标
    metrics:
    - name: request-success-rate
      thresholdRange:
        min: 99  # 成功率 >= 99%
      interval: 1m
    
    - name: request-duration
      thresholdRange:
        max: 500  # P99 延迟 <= 500ms
      interval: 1m
    
    # Webhook 测试
    webhooks:
    - name: acceptance-test
      type: pre-rollout
      url: http://flagger-loadtester.default/
      timeout: 30s
      metadata:
        type: bash
        cmd: "curl -sd 'test' http://my-app-canary/api/health | grep ok"
    
    - name: load-test
      type: rollout
      url: http://flagger-loadtester.default/
      metadata:
        cmd: "hey -z 2m -q 10 -c 2 http://my-app-canary/"
```

**触发金丝雀发布：**

```bash
# 1. 部署新版本（只需修改 Deployment 镜像）
kubectl set image deployment/my-app app=my-app:v2.0.0

# 2. Flagger 自动执行金丝雀发布
# - 创建金丝雀 Deployment
# - 逐步增加流量（10% → 20% → ... → 50%）
# - 监控指标
# - 如果指标异常，自动回滚
# - 如果指标正常，完成发布

# 3. 查看金丝雀状态
kubectl get canary my-app

# 输出：
# NAME     STATUS      WEIGHT   LASTTRANSITIONTIME
# my-app   Progressing   10     2025-10-07T12:00:00Z
# my-app   Progressing   20     2025-10-07T12:01:00Z
# my-app   Succeeded     0      2025-10-07T12:05:00Z
```

**Flagger 金丝雀发布流程：**

```
1. 检测到新版本
   ↓
2. 创建 Canary Deployment (10% 副本)
   ↓
3. 运行验收测试 (Webhook)
   ↓
4. 开始流量切换
   - 10% 流量 → 监控 1 分钟 → 指标正常 ✅
   - 20% 流量 → 监控 1 分钟 → 指标正常 ✅
   - 30% 流量 → 监控 1 分钟 → 指标正常 ✅
   - 40% 流量 → 监控 1 分钟 → 指标异常 ❌
   ↓
5. 自动回滚到旧版本
   或
5. 完成发布（100% 流量到新版本）
```

### 3. 基于请求头的金丝雀（灰度发布）

**允许特定用户测试新版本：**

```yaml
# virtualservice-header-based.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app
  http:
  # 规则1：VIP 用户访问新版本
  - match:
    - headers:
        user-type:
          exact: "vip"
    route:
    - destination:
        host: my-app
        subset: v2
  
  # 规则2：内部员工访问新版本
  - match:
    - headers:
        employee:
          exact: "true"
    route:
    - destination:
        host: my-app
        subset: v2
  
  # 规则3：其他用户访问旧版本
  - route:
    - destination:
        host: my-app
        subset: v1
```

```javascript
// Express 中间件
app.use((req, res, next) => {
```

## ⚠️ 三个高频错误

### 错误 1：没有充分的监控指标

**问题：** 只检查 HTTP 状态码，忽略业务指标。

```yaml
# ❌ 错误：监控不足
metrics:
- name: request-success-rate
  thresholdRange:
    min: 99
```

**解决方案：** 添加业务指标。

```yaml
# ✅ 正确：全面监控
metrics:
- name: request-success-rate
  thresholdRange:
    min: 99
- name: request-duration
  thresholdRange:
    max: 500
- name: database-connection-errors
  thresholdRange:
    max: 0
- name: business-transaction-success-rate
  thresholdRange:
    min: 99.5
```

---

### 错误 2：流量切换过快

**问题：** 直接从 10% 跳到 100%。

```yaml
# ❌ 错误
stepWeight: 90  # 一次增加 90%
```

**解决方案：** 渐进式增加。

```yaml
# ✅ 正确
stepWeight: 10  # 每次增加 10%
maxWeight: 50   # 最大到 50%，手动确认后再到 100%
```

---

### 错误 3：忘记回滚计划

**问题：** 没有预先准备回滚步骤。

**解决方案：** 准备回滚脚本。

```bash
#!/bin/bash
# rollback.sh

# 立即切换到旧版本
kubectl patch virtualservice my-app --type='json' \
  -p='[{"op": "replace", "path": "/spec/http/0/route/0/weight", "value":100},
       {"op": "replace", "path": "/spec/http/0/route/1/weight", "value":0}]'

# 或使用 Flagger 回滚
kubectl patch canary my-app -p '{"spec":{"skipAnalysis":true}}'
kubectl delete canary my-app
```

## 📚 FAQ

### 1. 蓝绿部署 vs 金丝雀发布如何选择？

| 场景 | 推荐策略 |
|------|---------|
| 重大版本更新 | 蓝绿部署（充分测试后切换） |
| 高风险变更 | 金丝雀发布（渐进式） |
| 资源充足 | 蓝绿部署 |
| 资源受限 | 金丝雀发布 |
| 需要快速回滚 | 蓝绿部署 |

```bash
# 方法3：使用 Istio 流量镜像（不影响用户）
istioctl experimental traffic-mirror my-app --to my-app-canary
```
