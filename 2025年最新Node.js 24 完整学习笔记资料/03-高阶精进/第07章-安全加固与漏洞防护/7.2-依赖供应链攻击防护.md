## 📚 知识导读

### 什么是供应链攻击？

**供应链攻击**（Supply Chain Attack）指攻击者通过篡改第三方依赖包，将恶意代码注入到目标应用中。

**著名案例：**

```
┌─────────────────────────────────────────────────────┐
│  1. event-stream 事件（2018）                        │
├─────────────────────────────────────────────────────┤
│  - 攻击者获取了 event-stream 的维护权限             │
│  - 发布恶意版本 3.3.6，包含后门代码                 │
│  - 目标：窃取 Copay 钱包的比特币                    │
│  - 影响：200 万次/周下载量，数千个项目受影响        │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  2. ua-parser-js 事件（2021）                        │
├─────────────────────────────────────────────────────┤
│  - 攻击者入侵 npm 账户                              │
│  - 发布恶意版本 0.7.29/0.8.0/1.0.0                  │
│  - 恶意代码：挖矿木马 + 密码窃取                    │
│  - 影响：900 万次/周下载量，包括 Facebook/Amazon    │
└─────────────────────────────────────────────────────┘

│  - 影响：数千个项目构建失败                         │
│  - 教训：依赖维护者可能不稳定                       │
└─────────────────────────────────────────────────────┘
```

## 🎯 Lockfile 完整性校验

### 1. 理解 Lockfile

**package-lock.json** 锁定依赖的精确版本和完整性哈希。

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "lockfileVersion": 3,
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-5/PsL6iGPdfQ/lKM1UuielYgv3BUoJfz1aUwU9vHZ+J7gyvwdQXFEBIEIaxeGf0GIcreATNyBExtalisDbuMqQ==",
      "dependencies": {
        "accepts": "~1.3.8",
        "array-flatten": "1.1.1"
      }
    }
  }
}
```

**关键字段：**
- **integrity**：SHA-512 哈希，确保包未被篡改
- **resolved**：下载地址
- **version**：精确版本号

---

### 2. Lockfile 最佳实践

**✅ 提交 Lockfile 到版本控制**

```bash
# .gitignore（不要忽略 lockfile）
node_modules/
# ❌ package-lock.json  # 不要忽略！

# ✅ 保留
# package-lock.json  ← 提交到 Git
```

---

**✅ 使用 `npm ci` 而非 `npm install`**

```bash
# ❌ npm install
# - 更新 package-lock.json
# - 允许版本范围内的升级
# - 不适合生产环境

# ✅ npm ci（clean install）
npm ci
# - 严格按照 lockfile 安装
# - 删除 node_modules 后重新安装
# - 如果 package.json 与 lockfile 不一致会报错
# - 适合 CI/CD 和生产环境
```

---

**✅ 验证 Lockfile 完整性**

```bash
# 检查 lockfile 是否与 package.json 一致
npm audit signatures

# 输出示例
# audited 1134 packages in 2.345s
# 
# 1134 packages have verified registry signatures
# 
# 0 packages have invalid signatures
# 0 packages are missing signatures
```

---

### 3. pnpm Lockfile 优势

```yaml
# pnpm-lock.yaml（更安全、更高效）
lockfileVersion: '6.0'

dependencies:
  express:
    specifier: ^4.18.2
    version: 4.18.2

packages:
  /express@4.18.2:
    resolution: {integrity: sha512-5/PsL6iGPdfQ...}
    dependencies:
      accepts: 1.3.8
      array-flatten: 1.1.1
```

**pnpm 优势：**
- 内容寻址存储（基于哈希）
- 严格模式（禁止幽灵依赖）
- 更快的安装速度

### 2. 设置审计严重级别

```bash
# 只审计 high 和 critical

# 如果发现漏洞，退出码为 1（CI/CD 失败）
npm audit --audit-level=moderate || exit 1
```

## 🎯 Subresource Integrity (SRI)

### 1. SRI 基础

**SRI** 确保从 CDN 加载的资源未被篡改。

```html
<!-- ✅ 使用 SRI -->
<script 
  src="https://cdn.jsdelivr.net/npm/vue@3.3.4/dist/vue.global.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxp63fKZYlHNYXUGV6VrWGNv6lkZM5o"
  crossorigin="anonymous">
</script>

<!-- ❌ 没有 SRI -->
<script src="https://cdn.jsdelivr.net/npm/vue@3.3.4/dist/vue.global.js"></script>
```

---

### 2. 生成 SRI 哈希

```bash
# 使用 openssl
curl -s https://cdn.jsdelivr.net/npm/vue@3.3.4/dist/vue.global.js | \
  openssl dgst -sha384 -binary | \
  openssl base64 -A

# 使用 Node.js
import crypto from 'crypto';
import fetch from 'node-fetch';

const url = 'https://cdn.jsdelivr.net/npm/vue@3.3.4/dist/vue.global.js';
const response = await fetch(url);
const content = await response.text();

const hash = crypto.createHash('sha384').update(content).digest('base64');
console.log(`sha384-${hash}`);
```

---

### 3. 自动化 SRI 生成

```javascript
// generate-sri.js
import crypto from 'crypto';
import fs from 'fs/promises';
import path from 'path';

async function generateSRI(filePath) {
  const content = await fs.readFile(filePath);
  const hash = crypto.createHash('sha384').update(content).digest('base64');
  return `sha384-${hash}`;
}

// 生成所有 JS 文件的 SRI
const files = await fs.readdir('./dist');
const sriMap = {};

for (const file of files) {
  if (file.endsWith('.js')) {
    const sri = await generateSRI(path.join('./dist', file));
    sriMap[file] = sri;
  }
}

await fs.writeFile('./dist/sri.json', JSON.stringify(sriMap, null, 2));
console.log('SRI hashes generated:', sriMap);
```

## 📚 FAQ

### 1. 如何评估一个 npm 包的安全性？

**检查清单：**
- ✅ 下载量（>100k/周 较安全）
- ✅ GitHub stars（>1k 较可信）
- ✅ 维护者数量（>2 人更好）
- ✅ 最近更新时间（<6 个月）
- ✅ 查看 `package.json` 的 `scripts`（有无可疑命令）
- ✅ 使用 Socket.dev 扫描

### 2. Snyk vs Socket.dev 如何选择？

| 工具 | 优势 | 适用场景 |
|------|------|---------|
| **Snyk** | CVE 数据库完整、自动修复 | 已知漏洞检测 |
| **Socket.dev** | 实时检测、行为分析 | 0-day 攻击防护 |

**推荐：** 同时使用

### 3. 如何处理无法修复的漏洞？

1. **评估风险**：是否影响生产环境？
2. **隔离依赖**：使用沙箱运行
3. **寻找替代包**
4. **临时忽略**：`.nsprc` 或 `snyk ignore`
5. **上报给维护者**：提交 Issue/PR
