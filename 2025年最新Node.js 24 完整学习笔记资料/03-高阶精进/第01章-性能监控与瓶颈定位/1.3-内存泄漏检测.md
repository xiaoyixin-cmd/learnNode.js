## 📖 一、理论知识

### 1.1 什么是内存泄漏？

**内存泄漏（Memory Leak）** 是指程序分配的内存无法被垃圾回收器（GC）回收，导致内存占用持续增长，最终引发 OOM（Out Of Memory）错误。

#### 正常 vs 内存泄漏

```
正常应用：
Memory ▲│     ╱╲    ╱╲    ╱╲
       ││   ╱    ╲╱    ╲╱    ╲   ← GC 后内存释放
       ││ ╱
       │└────────────────────────→ Time

内存泄漏：
Memory ▲│                  ╱
       ││              ╱   ← 持续增长
       ││          ╱
       ││      ╱
       ││  ╱
       │└────────────────────────→ Time
```

### 1.2 常见内存泄漏原因

| 原因 | 示例代码 | 风险等级 |
|-----|---------|---------|
| **全局变量** | `global.cache = {}` | ⚠️⚠️⚠️ 严重 |
| **闭包引用** | `setInterval(() => bigData, 1000)` | ⚠️⚠️ 中等 |
| **事件监听器未移除** | `emitter.on('event', handler)` | ⚠️⚠️⚠️ 严重 |
| **缓存无限增长** | `cache.set(key, value)` 无过期机制 | ⚠️⚠️⚠️ 严重 |
| **定时器未清理** | `setInterval()` 未调用 `clearInterval()` | ⚠️⚠️ 中等 |

### 1.3 Node.js 内存结构

```
Node.js 进程内存布局：
┌─────────────────────────────────┐
│  RSS (Resident Set Size)        │  进程总内存
│  ├─ Heap (堆内存)               │  JS 对象存储
│  │  ├─ New Space (新生代)       │  短期对象
│  │  └─ Old Space (老生代)       │  长期对象
│  ├─ Code Segment (代码段)       │  编译后的代码
│  ├─ Stack (栈内存)              │  函数调用栈
│  └─ External Memory (外部内存)  │  Buffer、C++ 插件
└─────────────────────────────────┘
```

#### 查看内存使用

```javascript
const used = process.memoryUsage();

console.log({
  rss: (used.rss / 1024 / 1024).toFixed(2) + ' MB',           // 总内存
  heapTotal: (used.heapTotal / 1024 / 1024).toFixed(2) + ' MB', // 堆总大小
  heapUsed: (used.heapUsed / 1024 / 1024).toFixed(2) + ' MB',   // 堆已用
  external: (used.external / 1024 / 1024).toFixed(2) + ' MB',   // 外部内存
  arrayBuffers: (used.arrayBuffers / 1024 / 1024).toFixed(2) + ' MB',
});

// 输出示例：
// {
//   rss: '45.23 MB',
//   heapTotal: '20.00 MB',
//   heapUsed: '12.34 MB',
//   external: '1.56 MB',
//   arrayBuffers: '0.20 MB'
// }
```

## 💻 二、使用指南

### 2.1 方法1：`heapdump` + Chrome DevTools

最经典的内存泄漏分析方法，**三次快照对比法**。

#### 安装

```bash
npm install heapdump
```

#### 基础用法

```javascript
// app.js
const heapdump = require('heapdump');
const express = require('express');
const app = express();

// 暴露手动触发快照的接口
app.get('/heapdump', (req, res) => {
  const filename = `heapdump-${Date.now()}.heapsnapshot`;
  heapdump.writeSnapshot(filename, (err, filepath) => {
    if (err) {
      res.status(500).send('生成失败');
    } else {
      res.send(`快照已生成: ${filepath}`);
    }
  });
});

app.listen(3000);
```

#### 三次快照对比法

**Step 1：生成第一次快照（初始状态）**

```bash
# 启动应用
node app.js

# 访问快照接口
curl http://localhost:3000/heapdump
# 生成 heapdump-1234567890.heapsnapshot
```

**Step 2：触发可疑操作**

```bash
# 访问业务接口 1000 次
for i in {1..1000}; do
  curl http://localhost:3000/api/users
done
```

**Step 3：生成第二次快照**

```bash
curl http://localhost:3000/heapdump
# 生成 heapdump-1234567900.heapsnapshot
```

**Step 4：再次触发可疑操作**

```bash
for i in {1..1000}; do
  curl http://localhost:3000/api/users
done
```

**Step 5：生成第三次快照**

```bash
curl http://localhost:3000/heapdump
# 生成 heapdump-1234567910.heapsnapshot
```

#### Chrome DevTools 分析

**Step 1：打开 Chrome DevTools**

```
1. 打开 Chrome 浏览器
2. 按 F12 打开开发者工具
3. 切换到 "Memory" 标签
4. 点击 "Load" 按钮
5. 依次加载三个 .heapsnapshot 文件
```

**Step 2：对比快照**

```
1. 选择第二个快照
2. 顶部下拉菜单选择 "Comparison"（对比模式）
3. 选择 "Snapshot 1" 作为基准
4. 查看 "Δ" 列（增量）
5. 按 "Size Delta" 排序
```

**Step 3：查找泄漏对象**

```
关键指标：
- # New：新增对象数量
- # Deleted：删除对象数量
- # Delta：增量（New - Deleted）

如果某类型对象的 # Delta 持续增长（对比 Snapshot 2 vs 1, Snapshot 3 vs 2）
→ 这就是内存泄漏的源头！
```

### 2.2 方法2：`memwatch-next` 实时监控

```bash
npm install @airbnb/node-memwatch
```

#### 基础用法

```javascript
// monitor/memory.js
const memwatch = require('@airbnb/node-memwatch');

// 监听内存泄漏事件
memwatch.on('leak', (info) => {
  console.error('检测到内存泄漏！');
  console.error(JSON.stringify(info, null, 2));
  
  // info 包含：
  // {
  //   start: '2025-01-07T10:00:00.000Z',
  //   end: '2025-01-07T10:05:00.000Z',
  //   growth: 12345678,  // 增长字节数
  //   reason: 'heap growth over 5 consecutive GCs'
  // }
  
  // 告警处理（发送钉钉/邮件）
  sendAlert(info);
});

// 监听 GC 统计
memwatch.on('stats', (stats) => {
  console.log('GC 统计:', {
    gc_type: stats.gc_type,              // GC 类型
    used_heap: (stats.current_base / 1024 / 1024).toFixed(2) + ' MB',
    heap_growth: (stats.estimated_base / 1024 / 1024).toFixed(2) + ' MB',
  });
});
```

## 🐛 三、3个高频错误

### 错误1：事件监听器未移除

#### 问题代码

```javascript
const EventEmitter = require('events');
const emitter = new EventEmitter();

class UserService {
  constructor() {
    // ❌ 每次创建实例都添加监听器，但从不移除
    emitter.on('user:created', this.handleUserCreated.bind(this));
  }

  handleUserCreated(user) {
    console.log('新用户:', user);
  }
}

// 每次请求创建新实例
app.post('/api/users', (req, res) => {
  const service = new UserService();  // 监听器累积！
  service.createUser(req.body);
  res.json({ success: true });
});
```

**问题分析**：

```bash
# 10000 次请求后
> emitter.listenerCount('user:created')
10000  # ← 10000 个监听器！

# Node.js 默认警告（超过 10 个）
# MaxListenersExceededWarning: Possible EventEmitter memory leak detected
```

#### 正确写法

**方法1：使用 `once()` 单次监听**

```javascript
class UserService {
  constructor() {
    // ✅ 使用 once() 自动移除
    emitter.once('user:created', this.handleUserCreated.bind(this));
  }
}
```

**方法2：手动移除监听器**

```javascript
class UserService {
  constructor() {
    this.handler = this.handleUserCreated.bind(this);
    emitter.on('user:created', this.handler);
  }

  destroy() {
    // ✅ 清理监听器
    emitter.removeListener('user:created', this.handler);
  }
}

app.post('/api/users', (req, res) => {
  const service = new UserService();
  service.createUser(req.body);
  service.destroy();  // 手动清理
  res.json({ success: true });
});
```

**方法3：使用单例模式**

```javascript
// ✅ 全局只有一个实例
const userService = new UserService();

app.post('/api/users', (req, res) => {
  userService.createUser(req.body);
  res.json({ success: true });
});
```

### 错误2：闭包引用大对象

#### 问题代码

```javascript
const express = require('express');
const app = express();

// ❌ 闭包引用大对象
function createHandler() {
  // 模拟大对象（10MB）
  const largeData = Buffer.alloc(10 * 1024 * 1024);
  
  return (req, res) => {
    // 即使这里不使用 largeData，它也会被闭包持有
    res.send('Hello');
  };
}

// 每次路由注册都泄漏 10MB
app.get('/route1', createHandler());  // +10MB
app.get('/route2', createHandler());  // +10MB
app.get('/route3', createHandler());  // +10MB
```

**问题分析**：

```javascript
// 使用 heapdump 分析
// Chrome DevTools 显示：
// Buffer: 30 MB (3 个 10MB Buffer)
// Retained by: (anonymous function) ← 闭包
```

#### 正确写法

**方法1：避免闭包引用**

```javascript
// ✅ 不在闭包内创建大对象
function createHandler(largeData) {
  return (req, res) => {
    // 如果不需要 largeData，就不要引用它
    res.send('Hello');
  };
}

app.get('/route1', createHandler());  // 不传入大对象
```

**方法2：及时解除引用**

```javascript
function createHandler() {
  let largeData = Buffer.alloc(10 * 1024 * 1024);
  
  return (req, res) => {
    // 使用完后立即解除引用
    largeData = null;
    res.send('Hello');
  };
}
```

**方法3：使用 WeakMap**

```javascript
// ✅ WeakMap 允许对象被 GC 回收
const cache = new WeakMap();

function processUser(user) {
  if (!cache.has(user)) {
    const processed = expensiveOperation(user);  // 大对象
    cache.set(user, processed);
  }
  return cache.get(user);
}

// 当 user 对象没有其他引用时，cache 中的条目会自动移除
```

### 错误3：缓存无限增长

#### 问题代码

```javascript
// ❌ 简单 Map 缓存，没有过期机制
const cache = new Map();

app.get('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  
  if (cache.has(userId)) {
    return res.json(cache.get(userId));
  }
  
  const user = await User.findById(userId);
  cache.set(userId, user);  // 永久缓存，无限增长！
  res.json(user);
});
```

**问题分析**：

```javascript
// 运行 1 个月后
console.log(cache.size);  // 1000000 条记录
console.log(process.memoryUsage().heapUsed / 1024 / 1024);  // 2GB+
```

#### 正确写法

**方法1：使用 LRU 缓存**

```javascript
const LRU = require('lru-cache');

// ✅ 限制缓存大小 + 过期时间
const cache = new LRU({
  max: 1000,           // 最多 1000 条
  ttl: 1000 * 60 * 5,  // 5 分钟过期
  updateAgeOnGet: true,
});

app.get('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  let user = cache.get(userId);
  
  if (!user) {
    user = await User.findById(userId);
    cache.set(userId, user);
  }
  
  res.json(user);
});
```

**方法2：使用 Redis（分布式缓存）**

```javascript
const Redis = require('ioredis');
const redis = new Redis();

app.get('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  
  // ✅ 从 Redis 读取
  let user = await redis.get(`user:${userId}`);
  
  if (!user) {
    user = await User.findById(userId);
    // 设置 5 分钟过期
    await redis.set(`user:${userId}`, JSON.stringify(user), 'EX', 300);
  } else {
    user = JSON.parse(user);
  }
  
  res.json(user);
});
```

**方法3：定期清理**

```javascript
const cache = new Map();

// ✅ 每小时清理一次
setInterval(() => {
  const now = Date.now();
  for (const [key, value] of cache.entries()) {
    if (now - value.timestamp > 3600000) {  // 1 小时
      cache.delete(key);
    }
  }
  console.log('缓存清理完成，当前大小:', cache.size);
}, 3600000);
```

**Step 3：Chrome DevTools 分析**

```javascript
// ❌ 问题代码
const wss = new WebSocket.Server({ port: 8080 });
const clients = new Map();  // 存储所有客户端

wss.on('connection', (ws, req) => {
  const userId = req.headers['user-id'];
  clients.set(userId, ws);  // 添加到 Map
  
  ws.on('message', (msg) => {
    // 处理消息...
  });
  
  // ❌ 问题：未监听 'close' 事件，连接断开后未从 Map 移除
});
```

**Step 5：修复代码**

```javascript
// ✅ 修复后
wss.on('connection', (ws, req) => {
  const userId = req.headers['user-id'];
  clients.set(userId, ws);
  
  // ✅ 添加 close 事件监听
  ws.on('close', () => {
    clients.delete(userId);  // 移除引用
    console.log(`用户 ${userId} 断开，当前在线: ${clients.size}`);
  });
  
  ws.on('message', (msg) => {
    // 处理消息...
  });
});

// ✅ 定期清理僵尸连接
setInterval(() => {
  for (const [userId, ws] of clients.entries()) {
    if (ws.readyState === WebSocket.CLOSED) {
      clients.delete(userId);
    }
  }
}, 60000);
```

**Step 6：验证效果**

```
修复前：200MB → 4GB（24小时）
修复后：200MB → 250MB（24小时）✅
```

### 场景2：定时任务内存泄漏

#### 需求背景

- 每小时执行数据同步任务
- 每次任务执行后，内存增长 50MB
- 20 小时后服务崩溃

#### 问题代码

```javascript
// task/sync.js
const cron = require('node-cron');
const axios = require('axios');

// ❌ 问题代码
cron.schedule('0 * * * *', async () => {
  console.log('开始数据同步...');
  
  // 获取 10 万条数据
  const response = await axios.get('https://api.example.com/data', {
    params: { limit: 100000 },
  });
  
  const data = response.data;  // 大对象（50MB）
  
  // 处理数据
  for (const item of data) {
    await processItem(item);
  }
  
  console.log('数据同步完成');
  // ❌ 问题：data 对象在任务结束后仍被引用（闭包）
});

// 辅助函数
const cache = [];  // ❌ 全局缓存

async function processItem(item) {
  // 每次都添加到全局缓存
  cache.push(item);  // 泄漏！
  // 实际业务逻辑...
}
```

#### 修复方案

```javascript
// ✅ 修复后
cron.schedule('0 * * * *', async () => {
  console.log('开始数据同步...');
  
  try {
    // ✅ 使用流式处理，避免一次性加载
    const response = await axios.get('https://api.example.com/data', {
      params: { limit: 100000 },
      responseType: 'stream',
    });
    
    // ✅ 逐条处理
    for await (const item of parseStream(response.data)) {
      await processItem(item);
    }
    
    console.log('数据同步完成');
    
  } catch (error) {
    console.error('同步失败:', error);
  } finally {
    // ✅ 清理临时数据
    global.gc && global.gc();  // 手动触发 GC（开发时使用）
  }
});

// ✅ 使用局部缓存
async function processItem(item) {
  // 不使用全局变量
  const localCache = new Map();
  
  if (localCache.has(item.id)) {
    return localCache.get(item.id);
  }
  
  // 处理逻辑...
  const result = await doProcess(item);
  localCache.set(item.id, result);
  
  return result;
}

// ✅ 定期清理全局资源
cron.schedule('0 0 * * *', () => {
  // 每天凌晨清理缓存
  cache.length = 0;
  console.log('缓存已清理');
});
```

## 🔬 五、深度解析

### 5.1 垃圾回收机制

Node.js 使用 V8 引擎的分代式垃圾回收：

```
堆内存分代：
┌─────────────────────────────────┐
│ New Space (新生代, ~16MB)       │
│  ├─ From Space                  │
│  └─ To Space                    │
│  算法: Scavenge (复制算法)      │
│  特点: 快速，频繁                │
└─────────────────────────────────┘
┌─────────────────────────────────┐
│ Old Space (老生代, ~1.4GB)      │
│  算法: Mark-Sweep + Mark-Compact│
│  特点: 慢，不频繁                │
└─────────────────────────────────┘
```

#### Scavenge 算法（新生代）

```
Before GC:
┌──────────────┬──────────────┐
│ From Space   │ To Space     │
│ [Obj A][Obj B][Obj C]       │  ← 对象在 From
│              │     (empty)  │
└──────────────┴──────────────┘

After GC:
┌──────────────┬──────────────┐
│ From Space   │ To Space     │
│   (empty)    │ [Obj A][Obj C]│  ← 存活对象复制到 To
└──────────────┴──────────────┘
                 ↑ Obj B 被回收
```

#### Mark-Sweep 算法（老生代）

```
1. Mark（标记）：从根对象（global, stack）开始，标记所有可达对象
2. Sweep（清除）：清除所有未标记的对象
3. Compact（压缩）：整理内存碎片（可选）
```

### 5.2 如何判断对象可回收？

```javascript
// 可回收对象（无引用）
function createTempObject() {
  const temp = { data: 'temporary' };
  // temp 在函数结束后无引用 → 可回收
}

// 不可回收对象（有引用）
const cache = {};
function createCachedObject(key) {
  cache[key] = { data: 'persistent' };
  // cache[key] 有全局引用 → 不可回收
}
```

### 5.3 Chrome DevTools 内存分析技巧

#### 技巧1：查看对象的 Retaining Path（保留路径）

```
1. 在 Chrome DevTools Memory 标签
2. 加载 heap snapshot
3. 搜索可疑对象（如 "Array"）
4. 点击对象，查看底部的 "Retainers"
5. 追踪引用链：Object A → Closure → Global
```

**示例**：

```
  ↑

  ↑

  ↑

→ 结论：该 Array 被路由的闭包引用，无法释放
```

## 📝 六、总结

### 核心要点

✅ **内存泄漏** 是指对象无法被 GC 回收，导致内存持续增长  
✅ **三次快照对比法** 是最经典的排查方法（heapdump + Chrome DevTools）  
✅ **memwatch-next** 可自动检测泄漏，适合实时监控  
✅ **常见原因**：事件监听器、闭包、全局缓存、定时器  
✅ **解决方案**：及时移除监听器、使用 WeakMap、限制缓存大小  
✅ **Chrome DevTools** 的 Retaining Path 可追踪对象引用链  
✅ **生产环境** 定期生成堆快照，对比历史数据发现泄漏

### 最佳实践

1. **事件监听器**：优先使用 `once()`，手动监听要记得 `removeListener()`
2. **缓存管理**：使用 LRU 缓存（如 `lru-cache`）限制大小和过期时间
3. **闭包使用**：避免在闭包中引用大对象，及时 `= null` 解除引用
4. **定时器清理**：使用 `clearInterval()` / `clearTimeout()`
5. **WeakMap/WeakSet**：适合临时关联数据，允许 GC 回收
6. **定期监控**：使用 `memwatch` 或自定义脚本监控内存趋势

### 排查流程

```
1. 监控内存趋势 → 发现持续增长
2. 生成 3 次堆快照 → 对比增量
3. Chrome DevTools 分析 → 找出泄漏对象
4. 查看 Retaining Path → 定位引用链
5. 修复代码 → 再次验证
```

## ❓ 七、常见问题

### Q1: 内存增长就一定是内存泄漏吗？

**A**: 不一定，需要区分**正常增长**和**泄漏**。

**正常增长**：

```javascript
// 缓存预热（启动后加载常用数据）
const cache = new Map();

async function warmup() {
  const products = await Product.find().limit(1000);
  products.forEach(p => cache.set(p.id, p));
}

// 启动时内存从 100MB → 200MB（正常）
```

**内存泄漏**：

```javascript
// 无限增长
setInterval(() => {
  cache.set(Date.now(), { data: 'leak' });  // 每秒新增，永不删除
}, 1000);

// 内存从 100MB → 200MB → 500MB → OOM（异常）
```

**判断标准**：

| 指标 | 正常 | 泄漏 |
|-----|------|------|
| **内存趋势** | 增长后稳定 | 持续线性增长 |
| **GC 后** | 内存下降 | 内存不下降 |
| **重复操作** | 内存不变 | 内存累加 |

### Q3: WeakMap 和 Map 的区别？

**A**: WeakMap 的键是弱引用，允许 GC 回收。

**Map（强引用）**：

```javascript
let obj = { id: 1 };
const map = new Map();
map.set(obj, 'value');

obj = null;  // ❌ obj 仍被 map 引用，无法回收

console.log(map.size);  // 1（依然存在）
```

**WeakMap（弱引用）**：

```javascript
let obj = { id: 1 };
const weakMap = new WeakMap();
weakMap.set(obj, 'value');

obj = null;  // ✅ obj 可以被 GC 回收

// WeakMap 中的条目会自动移除
// 无法直接验证，因为 GC 时机不确定
```

**适用场景**：

| 类型 | 适用场景 | 示例 |
|-----|---------|------|
| **Map** | 需要遍历、持久化 | 用户会话缓存 |
| **WeakMap** | 临时关联数据 | DOM 元素的元数据 |

**实战示例**：

```javascript
// ✅ 使用 WeakMap 关联 DOM 元数据
const metadata = new WeakMap();

function attachMetadata(element, data) {
  metadata.set(element, data);
}

// 当 DOM 元素被移除后，metadata 自动清理
document.getElementById('myDiv').remove();
// → WeakMap 中的条目会被 GC 回收
```

### Q4: 如何监控生产环境的内存使用？

**A**: 使用 APM 工具或自定义监控。

**方法1：自定义监控**

```javascript
// monitor/memory.js
const { promisify } = require('util');
const heapdump = require('heapdump');
const writeSnapshot = promisify(heapdump.writeSnapshot);

class MemoryMonitor {
  constructor(options = {}) {
    this.threshold = options.threshold || 500;  // 500MB
    this.interval = options.interval || 60000;  // 1分钟
    this.snapshotDir = options.snapshotDir || './heapdumps';
  }

  start() {
    setInterval(async () => {
      const used = process.memoryUsage().heapUsed / 1024 / 1024;
      
      console.log(`内存使用: ${used.toFixed(2)} MB`);
      
      if (used > this.threshold) {
        console.warn(`⚠️ 内存超过阈值（${this.threshold}MB）`);
        
        // 生成堆快照
        const filename = `${this.snapshotDir}/heap-${Date.now()}.heapsnapshot`;
        await writeSnapshot(filename);
        
        // 发送告警
        this.sendAlert({ used, threshold: this.threshold, filename });
      }
    }, this.interval);
  }

  sendAlert(data) {
    // 发送钉钉/邮件告警
    console.error('内存告警:', data);
  }
}

module.exports = MemoryMonitor;
```

**方法2：集成 New Relic**

```javascript
const newrelic = require('newrelic');

setInterval(() => {
  const mem = process.memoryUsage();
  newrelic.recordMetric('Custom/Memory/HeapUsed', mem.heapUsed / 1024 / 1024);
  newrelic.recordMetric('Custom/Memory/RSS', mem.rss / 1024 / 1024);
}, 60000);
```

### Q5: Buffer 对象会导致内存泄漏吗？

**A**: 会，Buffer 使用**外部内存**，不受 V8 堆限制。

**问题代码**：

```javascript
const buffers = [];

setInterval(() => {
  // ❌ 每秒分配 10MB Buffer
  buffers.push(Buffer.alloc(10 * 1024 * 1024));
}, 1000);

// 60 秒后：600MB 外部内存（不受 --max-old-space-size 限制）
```

**查看 Buffer 内存**：

```javascript
const used = process.memoryUsage();
console.log({
  heapUsed: (used.heapUsed / 1024 / 1024).toFixed(2) + ' MB',
  external: (used.external / 1024 / 1024).toFixed(2) + ' MB',  // ← Buffer 内存
  arrayBuffers: (used.arrayBuffers / 1024 / 1024).toFixed(2) + ' MB',
});
```

**修复方案**：

```javascript
// ✅ 使用 Buffer Pool
const BufferPool = require('buffer-pool');
const pool = new BufferPool(10 * 1024 * 1024);  // 预分配 10MB

function processData(data) {
  const buffer = pool.get(data.length);  // 从池中获取
  // 使用 buffer...
  pool.release(buffer);  // 释放回池
}
```

### Q6: 如何定位 `(closure)` 引起的泄漏？

**A**: 使用 Chrome DevTools 的 Retaining Path。

**步骤**：

```
1. 加载堆快照
2. 搜索泄漏对象（如 "Array"）
3. 选择一个实例，查看 "Retainers" 面板
```

```
Array @123456
  size: 10 MB
  ↑
(closure) in createHandler  ← 被闭包引用
  ↑
FunctionContext
  ↑
express.routes['/api/users'].handler
  ↑
global
```

**对应代码**：

```javascript
// 定位到这个函数
function createHandler() {
  const largeData = Array(1000000).fill(0);  // 泄漏源头

    // 即使不使用 largeData，闭包也会保留它
    res.send('OK');
  };
}

app.get('/api/users', createHandler());
```

**修复**：

```javascript
// ✅ 不在闭包中创建大对象
function createHandler() {
  return (req, res) => {
    res.send('OK');
  };
}
```

### Q8: 为什么 Chrome DevTools 显示的内存比 `process.memoryUsage()` 少？

**A**: 两者统计的范围不同。

| 工具 | 统计范围 | 数值 |
|-----|---------|------|
| **Chrome DevTools** | 仅 V8 堆内存 | ~100MB |
| **process.memoryUsage().heapUsed** | V8 堆内存 | ~100MB |
| **process.memoryUsage().rss** | 进程总内存（堆+栈+代码+C++） | ~200MB |

**示例**：

```javascript
const used = process.memoryUsage();
console.log({
  'Chrome DevTools 显示': (used.heapUsed / 1024 / 1024).toFixed(2) + ' MB',
  '进程实际占用': (used.rss / 1024 / 1024).toFixed(2) + ' MB',
});

// 输出：
// {
//   'Chrome DevTools 显示': '105.23 MB',
//   '进程实际占用': '220.45 MB'  ← 包含更多内存
// }
```
