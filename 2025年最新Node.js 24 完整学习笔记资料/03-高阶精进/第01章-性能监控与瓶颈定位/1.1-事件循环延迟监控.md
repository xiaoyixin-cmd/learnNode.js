# 1.1 事件循环延迟监控

> **本节目标**：深入理解事件循环延迟的产生原因，掌握使用 `perf_hooks.monitorEventLoopDelay()` 进行实时监控，并制定合理的告警策略。

---

## 📖 一、理论知识

### 1.1 什么是事件循环延迟？

**事件循环延迟（Event Loop Delay/Lag）** 是指事件循环处理下一个任务的延迟时间，它反映了 Node.js 应用的繁忙程度。

#### 核心概念

```
正常事件循环：
┌─────────────────────────┐
│  Task 1 → Task 2 → Task 3  │  延迟 < 10ms
└─────────────────────────┘

阻塞事件循环：
┌────────────────────────────────────────┐
│  Task 1 (耗时200ms) → Task 2 (延迟200ms) │
└────────────────────────────────────────┘
```

#### 延迟产生的原因

| 原因分类 | 具体场景 | 影响程度 |
|---------|---------|---------|
| **CPU 密集计算** | JSON 序列化大对象、正则匹配、图片处理 | ⚠️⚠️⚠️ 严重 |
| **同步 I/O 操作** | `fs.readFileSync()`, `child_process.execSync()` | ⚠️⚠️⚠️ 严重 |
| **内存问题** | 频繁 GC、内存泄漏导致 GC 时间增加 | ⚠️⚠️ 中等 |
| **外部依赖慢** | 数据库查询慢、第三方 API 超时 | ⚠️ 轻微（异步情况） |

### 1.2 为什么要监控事件循环延迟？

#### 影响范围

1. **请求响应时间增加**：用户感知明显的卡顿
2. **吞吐量下降**：单位时间内处理的请求数减少
3. **连接超时**：WebSocket、长连接被迫断开
4. **健康检查失败**：K8s/Docker 健康检查超时导致容器重启

#### 监控价值

```
延迟监控 → 发现性能瓶颈 → 优化代码 → 提升用户体验
```

## 💻 二、使用指南

### 2.1 `perf_hooks.monitorEventLoopDelay()` API

Node.js 11.10.0+ 提供的原生监控 API。

#### 基础用法

```javascript
const { monitorEventLoopDelay } = require('perf_hooks');

// 创建监控实例（采样精度 10ms）
const h = monitorEventLoopDelay({ resolution: 10 });

// 启动监控
h.enable();

// 等待一段时间后查看统计
setTimeout(() => {
  console.log('最小延迟:', h.min, 'ns');      // 纳秒
  console.log('最大延迟:', h.max, 'ns');
  console.log('平均延迟:', h.mean, 'ns');
  console.log('标准差:', h.stddev, 'ns');
  console.log('P50:', h.percentile(50), 'ns'); // 中位数
  console.log('P95:', h.percentile(95), 'ns');
  console.log('P99:', h.percentile(99), 'ns');
  
  // 重置统计数据
  h.reset();
}, 5000);
```

#### 核心方法说明

| 方法/属性 | 说明 | 单位 |
|---------|------|-----|
| `h.min` | 采样期间最小延迟 | 纳秒（ns） |
| `h.max` | 采样期间最大延迟 | 纳秒（ns） |
| `h.mean` | 平均延迟 | 纳秒（ns） |
| `h.stddev` | 标准差（波动程度） | 纳秒（ns） |
| `h.percentile(p)` | 百分位数（p=50/95/99） | 纳秒（ns） |
| `h.reset()` | 重置所有统计数据 | - |
| `h.disable()` | 停止监控 | - |

### 2.2 实战示例：Express 中间件监控

```javascript
// middleware/eventLoopMonitor.js
const { monitorEventLoopDelay } = require('perf_hooks');
const express = require('express');

/**
 * 创建事件循环监控中间件
 * @param {Object} options - 配置选项
 * @param {number} options.warnThreshold - 警告阈值（毫秒）
 * @param {number} options.interval - 输出间隔（毫秒）
 */
function createEventLoopMonitor(options = {}) {
  const {
    warnThreshold = 100,  // 100ms 延迟触发警告
    interval = 10000,     // 每 10 秒输出统计
  } = options;

  // 创建监控实例（10ms 采样精度）
  const monitor = monitorEventLoopDelay({ resolution: 10 });
  monitor.enable();

  // 定时输出统计
  const timer = setInterval(() => {
    const p99 = monitor.percentile(99) / 1e6; // 转换为毫秒
    const mean = monitor.mean / 1e6;

    console.log({
      timestamp: new Date().toISOString(),
      eventLoopDelay: {
        min: (monitor.min / 1e6).toFixed(2) + 'ms',
        max: (monitor.max / 1e6).toFixed(2) + 'ms',
        mean: mean.toFixed(2) + 'ms',
        p50: (monitor.percentile(50) / 1e6).toFixed(2) + 'ms',
        p95: (monitor.percentile(95) / 1e6).toFixed(2) + 'ms',
        p99: p99.toFixed(2) + 'ms',
      },
    });

    // 触发告警
    if (p99 > warnThreshold) {
      console.warn(`⚠️ 事件循环延迟超过阈值！P99=${p99.toFixed(2)}ms`);
    }

    // 重置计数器
    monitor.reset();
  }, interval);

  // 优雅关闭
  process.on('SIGTERM', () => {
    clearInterval(timer);
    monitor.disable();
  });

  // 返回中间件（可用于添加 HTTP 响应头）
  return (req, res, next) => {
    // 将当前 P99 延迟添加到响应头（用于客户端监控）
    res.setHeader('X-Event-Loop-Lag', (monitor.percentile(99) / 1e6).toFixed(2));
    next();
  };
}

module.exports = { createEventLoopMonitor };
```

#### 使用示例

```javascript
// app.js
const express = require('express');
const { createEventLoopMonitor } = require('./middleware/eventLoopMonitor');

const app = express();

// 应用监控中间件
app.use(createEventLoopMonitor({
  warnThreshold: 100,  // 100ms
  interval: 10000,     // 10秒
}));

// 模拟 CPU 密集任务（会导致延迟）
app.get('/slow', (req, res) => {
  const start = Date.now();
  // 同步阻塞 200ms
  while (Date.now() - start < 200) {}
  res.send('Done');
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

```javascript
const client = require('prom-client');

function startEventLoopMonitoring() {
```

## 🐛 三、3个高频错误

### 错误1：单位换算错误

#### 问题代码

```javascript
const h = monitorEventLoopDelay({ resolution: 10 });
h.enable();

setTimeout(() => {
  // ❌ 错误：直接使用纳秒值判断
  if (h.mean > 100) {  
    console.log('延迟超过 100ms');
  }
}, 5000);
```

**错误原因**：`h.mean` 返回的是**纳秒（nanosecond）**，1 毫秒 = 1,000,000 纳秒。上面的代码实际判断的是 100 纳秒（0.0001ms）。

#### 正确写法

```javascript
setTimeout(() => {
  // ✅ 正确：转换为毫秒后再判断
  const meanMs = h.mean / 1e6;  // 除以 1,000,000
  if (meanMs > 100) {
    console.log(`延迟超过 100ms，当前 ${meanMs.toFixed(2)}ms`);
  }
}, 5000);

// 常用单位换算
const ns = h.mean;
const us = ns / 1e3;    // 微秒
const ms = ns / 1e6;    // 毫秒
const s = ns / 1e9;     // 秒
```

### 错误2：未重置统计数据导致误判

#### 问题代码

```javascript
const h = monitorEventLoopDelay({ resolution: 10 });
h.enable();

// 每 10 秒输出一次
setInterval(() => {
  console.log('P99 延迟:', h.percentile(99) / 1e6, 'ms');
  // ❌ 错误：未调用 reset()
}, 10000);
```

**错误原因**：未调用 `h.reset()`，导致每次输出的是**从启动开始的累积统计**，而不是最近 10 秒的数据。

#### 正确写法

```javascript
setInterval(() => {
  const p99 = h.percentile(99) / 1e6;
  console.log('最近 10s 的 P99 延迟:', p99, 'ms');
  
  // ✅ 正确：重置统计，下次只统计接下来 10 秒的数据
  h.reset();
}, 10000);
```

### 错误3：过高的采样精度导致性能开销

#### 问题代码

```javascript
// ❌ 错误：1ns 采样精度（过于精细）
const h = monitorEventLoopDelay({ resolution: 1 });
h.enable();
```

**错误原因**：采样精度过高会增加监控本身的性能开销（每次事件循环都记录数据）。

#### 性能影响对比

| 采样精度 | 性能开销 | 适用场景 |
|---------|---------|---------|
| `resolution: 1` | ~5% CPU | ❌ 不推荐 |
| `resolution: 10` | ~1% CPU | ✅ 生产环境推荐 |
| `resolution: 100` | ~0.1% CPU | ✅ 低优先级监控 |

#### 正确写法

```javascript
// ✅ 正确：10ms 采样精度（平衡精度与性能）
const h = monitorEventLoopDelay({ resolution: 10 });
h.enable();

// 如果只需要粗略监控，可以使用 100ms
const hLowOverhead = monitorEventLoopDelay({ resolution: 100 });
```

## 🎯 四、2个应用场景

### 场景1：电商促销活动性能监控

#### 需求背景

- 双11大促期间，流量是平时的 100 倍
- 需要实时监控服务器负载，及时发现性能瓶颈
- 当延迟超过阈值时，自动触发限流或扩容

#### 解决方案

```javascript
// monitor/promotionMonitor.js
const { monitorEventLoopDelay } = require('perf_hooks');
const Redis = require('ioredis');

class PromotionMonitor {
  constructor(options = {}) {
    this.monitor = monitorEventLoopDelay({ resolution: 10 });
    this.redis = new Redis(options.redis);
    this.thresholds = {
      warning: 50,   // 50ms 警告
      critical: 100, // 100ms 严重
      ...options.thresholds,
    };
  }

  start() {
    this.monitor.enable();

    // 每 5 秒检查一次
    this.timer = setInterval(async () => {
      await this.checkAndAlert();
    }, 5000);
  }

  async checkAndAlert() {
    const p99 = this.monitor.percentile(99) / 1e6; // 毫秒
    const mean = this.monitor.mean / 1e6;

    // 记录到 Redis（供仪表板展示）
    await this.redis.zadd(
      'eventloop:lag',
      Date.now(),
      JSON.stringify({ p99, mean, timestamp: Date.now() })
    );

    // 告警逻辑
    if (p99 > this.thresholds.critical) {
      await this.triggerAlert('critical', p99);
      await this.enableRateLimit();  // 启用限流
    } else if (p99 > this.thresholds.warning) {
      await this.triggerAlert('warning', p99);
    }

    this.monitor.reset();
  }

  async triggerAlert(level, lag) {
    console.error(`[${level.toUpperCase()}] 事件循环延迟: ${lag.toFixed(2)}ms`);
    
    // 发送钉钉/企业微信告警
    await this.sendDingTalkAlert({
      level,
      lag: lag.toFixed(2),
      timestamp: new Date().toISOString(),
    });
  }

  async enableRateLimit() {
    // 动态启用限流（通过 Redis 标志位）
    await this.redis.set('rate_limit:enabled', '1', 'EX', 60);
    console.log('🚦 已启用限流保护');
  }

  async sendDingTalkAlert(data) {
    // 实际项目中集成钉钉机器人 API
    console.log('📢 发送告警:', data);
  }

  stop() {
    clearInterval(this.timer);
    this.monitor.disable();
  }
}

module.exports = PromotionMonitor;
```

#### 使用示例

```javascript

  },
```

### 场景2：微服务健康检查优化

#### 需求背景

- K8s 健康检查每 10 秒调用一次 `/health` 接口
- 如果事件循环延迟过高，应该返回 503（避免新流量进入）
- 需要同时检查事件循环延迟和内存使用率

#### 解决方案

```javascript
// health/check.js
const { monitorEventLoopDelay } = require('perf_hooks');
const os = require('os');

class HealthChecker {
  constructor() {
    this.monitor = monitorEventLoopDelay({ resolution: 10 });
    this.monitor.enable();
    this.isHealthy = true;
  }

  /**
   * 综合健康检查
   * @returns {Object} { healthy: boolean, details: Object }
   */
  check() {
    const eventLoopLag = this.monitor.percentile(99) / 1e6; // ms
    const memUsage = process.memoryUsage();
    const heapUsedPercent = (memUsage.heapUsed / memUsage.heapTotal) * 100;
    
    // 检查标准
    const checks = {
      eventLoop: {
        value: eventLoopLag,
        threshold: 200,  // 200ms
        healthy: eventLoopLag < 200,
      },
      memory: {
        value: heapUsedPercent.toFixed(2),
        threshold: 90,   // 90%
        healthy: heapUsedPercent < 90,
      },
    };

    const healthy = Object.values(checks).every(c => c.healthy);

    return {
      healthy,
      timestamp: new Date().toISOString(),
      uptime: process.uptime(),
      checks,
    };
  }

  /**
   * Express 中间件
   */
  middleware() {
    return (req, res) => {
      const result = this.check();

      // 重置事件循环统计
      this.monitor.reset();

      if (result.healthy) {
        res.status(200).json(result);
      } else {
        // 返回 503，K8s 会停止将流量路由到此实例
        res.status(503).json(result);
      }
    };
  }
}

module.exports = HealthChecker;
```

#### 使用示例

```javascript

const express = require('express');
const HealthChecker = require('./health/check');

const app = express();
const healthChecker = new HealthChecker();

// K8s 健康检查端点

// 业务接口
app.get('/api/users', (req, res) => {
  res.json({ users: [] });
});

app.listen(3000, () => {
  console.log('Server ready, health check at /health');
});
```

### 5.1 事件循环延迟的底层原理

#### Node.js 事件循环阶段

```
   ┌───────────────────────────┐
┌─>│           timers          │  执行 setTimeout/setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  执行延迟到下一个循环的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  内部使用
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │           poll            │<─────┤  connections, │
│  └─────────────┬─────────────┘      │   data, etc.  │
│  ┌─────────────┴─────────────┐      └───────────────┘
│  │           check           │  执行 setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks      │  执行 close 事件回调
   └───────────────────────────┘
```

#### 延迟产生的时机

```javascript
// 示例：阻塞事件循环
const start = Date.now();

// 模拟 CPU 密集任务（阻塞 500ms）
while (Date.now() - start < 500) {
  // 密集计算...
}

// 在这 500ms 内：
// - setTimeout 回调无法触发
// - HTTP 请求无法响应
// - 事件循环被完全阻塞
```

### 5.2 `monitorEventLoopDelay()` 实现原理

#### 采样机制

```javascript
// Node.js 内部伪代码
class EventLoopDelayMonitor {
  constructor(resolution) {
    this.resolution = resolution;  // 采样间隔（ms）
    this.samples = [];             // 存储延迟样本
  }

  enable() {
    this.timer = setInterval(() => {
      const expected = this.resolution;
      const actual = getActualDelay();  // 实际延迟
      const lag = actual - expected;    // 计算延迟差
      
      this.samples.push(lag);
    }, this.resolution);
  }

  percentile(p) {
    // 根据样本计算百分位数
    const sorted = this.samples.sort((a, b) => a - b);
    const index = Math.floor((p / 100) * sorted.length);
    return sorted[index];
  }
}
```

#### 精度与开销的权衡

| 采样间隔 | 样本数/10s | 内存占用 | CPU 开销 |
|---------|-----------|---------|---------|
| 1ms | 10,000 | ~80KB | ~5% |
| 10ms | 1,000 | ~8KB | ~1% |
| 100ms | 100 | ~0.8KB | ~0.1% |

**推荐配置**：生产环境使用 `resolution: 10`，开发/调试使用 `resolution: 1`。

### 5.3 延迟阈值设置指南

#### 不同场景的阈值参考

| 应用类型 | P99 警告阈值 | P99 严重阈值 | 说明 |
|---------|------------|------------|------|
| **API 服务** | 50ms | 100ms | 影响用户体验 |
| **WebSocket** | 30ms | 50ms | 实时通信要求高 |
| **后台任务** | 200ms | 500ms | 对延迟不敏感 |
| **实时游戏** | 10ms | 20ms | 极高实时性 |

#### 动态阈值计算

```javascript
// 根据历史数据动态调整阈值
class AdaptiveThreshold {
  constructor() {
    this.history = [];  // 最近 100 个 P99 值
  }

  addSample(p99) {
    this.history.push(p99);
    if (this.history.length > 100) {
      this.history.shift();
    }
  }

  getThreshold() {
    // 计算历史均值 + 2倍标准差
    const mean = this.history.reduce((a, b) => a + b, 0) / this.history.length;
    const variance = this.history.reduce((sum, val) => sum + Math.pow(val - mean, 2), 0) / this.history.length;
    const stddev = Math.sqrt(variance);
    
    return mean + 2 * stddev;  // 95% 置信区间
  }
}
```

## 📝 六、总结

### 核心要点

✅ **事件循环延迟** 是 Node.js 性能的关键指标，反映应用的实时响应能力  
✅ **`monitorEventLoopDelay()`** 提供原生、低开销的监控能力  
✅ **单位换算** 必须注意：API 返回纳秒，需转换为毫秒/秒  
✅ **定期 reset()** 确保统计数据是最近时间段的，而不是累积值  
✅ **采样精度** 推荐 10ms，平衡监控精度与性能开销  
✅ **告警阈值** 应根据业务类型设定（API: 100ms, WebSocket: 50ms）  
✅ **集成 Prometheus** 可实现持久化监控和可视化展示

### 最佳实践

1. **生产环境必备**：所有 Node.js 服务都应监控事件循环延迟
2. **与健康检查结合**：延迟过高时返回 503，避免新流量进入
3. **多维度监控**：同时监控延迟、内存、CPU，综合判断
4. **历史数据分析**：保存监控数据到 Redis/数据库，用于性能趋势分析
5. **自动化告警**：集成钉钉、企业微信、PagerDuty 等工具

## ❓ 七、常见问题

### Q1: 事件循环延迟和响应时间的区别是什么？

**A**: 两者是不同的指标：

```
响应时间 = 网络延迟 + 排队时间 + 处理时间
事件循环延迟 = 排队时间（任务等待被执行的时间）
```

| 指标 | 含义 | 监控方法 |
|-----|------|---------|
| **响应时间** | 用户发起请求到收到响应的总时间 | APM、HTTP 中间件 |
| **事件循环延迟** | 事件循环处理下一个任务的延迟 | `monitorEventLoopDelay()` |

**示例**：

```javascript
// 监控 HTTP 响应时间
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    console.log(`响应时间: ${duration}ms`);
  });
  next();
});

// 监控事件循环延迟
const h = monitorEventLoopDelay({ resolution: 10 });
h.enable();
setInterval(() => {
  console.log(`事件循环延迟: ${(h.mean / 1e6).toFixed(2)}ms`);
  h.reset();
}, 5000);
```

### Q2: P95 和 P99 哪个更重要？

**A**: 根据业务类型选择：

| 指标 | 含义 | 适用场景 |
|-----|------|---------|
| **P50（中位数）** | 50% 的请求延迟低于此值 | 评估整体性能 |
| **P95** | 95% 的请求延迟低于此值 | 一般业务监控 |
| **P99** | 99% 的请求延迟低于此值 | 高可用服务（金融、支付） |

**推荐策略**：

```javascript
// 同时监控多个百分位数
setInterval(() => {
  const p50 = h.percentile(50) / 1e6;
  const p95 = h.percentile(95) / 1e6;
  const p99 = h.percentile(99) / 1e6;

  console.log({
    p50: p50.toFixed(2) + 'ms',  // 整体性能
    p95: p95.toFixed(2) + 'ms',  // 大部分用户体验
    p99: p99.toFixed(2) + 'ms',  // 极端情况
  });

  // 告警规则
  if (p95 > 50) {
    console.warn('⚠️ P95 延迟超标');
  }
  if (p99 > 100) {
    console.error('❌ P99 延迟严重超标');
  }

  h.reset();
}, 10000);
```

### Q3: 为什么延迟监控显示正常，但用户反馈很慢？

**A**: 可能的原因：

1. **网络问题**：事件循环监控的是服务器端延迟，不包括网络传输时间
2. **数据库慢查询**：异步 I/O 不会阻塞事件循环，但会增加总响应时间
3. **前端性能**：浏览器渲染慢、JS 执行慢等

**完整监控方案**：

```javascript
// 1. 事件循环延迟
const eventLoopMonitor = monitorEventLoopDelay({ resolution: 10 });
eventLoopMonitor.enable();

// 2. HTTP 响应时间
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    // 记录到 Prometheus
    httpDuration.observe({ method: req.method, route: req.route?.path }, duration / 1000);
  });
  next();
});

// 3. 数据库查询时间
mongoose.set('debug', (collectionName, method, query, doc, options) => {
  const duration = options.executionTime;
  dbQueryDuration.observe({ collection: collectionName, method }, duration / 1000);
});

// 4. 外部 API 调用时间
async function callExternalAPI(url) {
  const start = Date.now();
  try {
    const response = await fetch(url);
    const duration = Date.now() - start;
    externalAPIDuration.observe({ url }, duration / 1000);
    return response;
  } catch (error) {
    externalAPIErrors.inc({ url });
    throw error;
  }
}
```

### Q4: 如何在 PM2 Cluster 模式下监控所有进程？

**A**: 每个 Worker 进程独立监控，然后聚合数据：

```javascript
// monitor/cluster.js
const cluster = require('cluster');
const { monitorEventLoopDelay } = require('perf_hooks');
const Redis = require('ioredis');

if (cluster.isMaster) {
  // Master 进程：启动 4 个 Worker
  for (let i = 0; i < 4; i++) {
    cluster.fork();
  }

  // 定期聚合所有 Worker 的数据
  setInterval(async () => {
    const redis = new Redis();
    const workerData = await redis.hgetall('eventloop:workers');
    
    console.log('所有 Worker 延迟统计:');
    for (const [pid, data] of Object.entries(workerData)) {
      console.log(`  Worker ${pid}:`, JSON.parse(data));
    }
  }, 10000);

} else {
  // Worker 进程：监控自己的事件循环
  const monitor = monitorEventLoopDelay({ resolution: 10 });
  monitor.enable();
  const redis = new Redis();

  setInterval(async () => {
    const p99 = monitor.percentile(99) / 1e6;
    
    // 上报到 Redis
    await redis.hset('eventloop:workers', process.pid, JSON.stringify({
      pid: process.pid,
      p99: p99.toFixed(2),
      timestamp: Date.now(),
    }));

    monitor.reset();
  }, 5000);

  // 启动 Express 服务
  const express = require('express');
  const app = express();
  app.get('/', (req, res) => res.send('Hello from Worker ' + process.pid));
  app.listen(3000);
}
```

### Q5: 延迟监控会影响性能吗？

**A**: 有轻微影响，但完全可接受：

| 配置 | CPU 开销 | 内存开销 | 建议 |
|-----|---------|---------|------|
| `resolution: 1` | ~5% | ~10MB | ❌ 仅开发/调试 |
| `resolution: 10` | ~1% | ~1MB | ✅ 生产环境推荐 |
| `resolution: 100` | ~0.1% | ~100KB | ✅ 低优先级监控 |

**性能测试**：

```javascript

const autocannon = require('autocannon');

// 有监控（resolution: 10）

  const result = await autocannon({
    url: 'http://localhost:3000',
    duration: 10,
  });
  console.log('有监控 QPS:', result.requests.mean);
  
  h.disable();
}

// 无监控: 25000 req/s
// 有监控: 24750 req/s  （下降约 1%）
```

### Q7: Docker 容器内如何准确监控？

**A**: 容器内监控需要注意 CPU 限制：

```javascript
// monitor/docker.js
const { monitorEventLoopDelay } = require('perf_hooks');
const os = require('os');

class DockerAwareMonitor {
  constructor() {
    this.monitor = monitorEventLoopDelay({ resolution: 10 });
    this.monitor.enable();
    
    // 检测是否在容器内
    this.isDocker = require('fs').existsSync('/.dockerenv');
    
    // 获取 CPU 限制
    this.cpuQuota = this.getCPUQuota();
  }

  getCPUQuota() {
    if (!this.isDocker) return os.cpus().length;
    
    try {
      const fs = require('fs');
      const quota = parseInt(fs.readFileSync('/sys/fs/cgroup/cpu/cpu.cfs_quota_us', 'utf8'));
      const period = parseInt(fs.readFileSync('/sys/fs/cgroup/cpu/cpu.cfs_period_us', 'utf8'));
      
      // 计算可用 CPU 核心数
      return quota > 0 ? quota / period : os.cpus().length;
    } catch (error) {
      return os.cpus().length;
    }
  }

  getAdjustedThreshold() {
    // 根据 CPU 配额调整阈值
    const baseLine = 50; // 基准阈值（1核心）
    return baseLine * (1 / this.cpuQuota);
  }
}

// 使用示例
const monitor = new DockerAwareMonitor();
console.log('CPU 配额:', monitor.cpuQuota);
console.log('调整后阈值:', monitor.getAdjustedThreshold());
```

### Q8: 如何与 APM 平台（New Relic, Datadog）集成？

**A**: 通过自定义指标上报：

```javascript
// monitor/newrelic.js
const newrelic = require('newrelic');
const { monitorEventLoopDelay } = require('perf_hooks');

class NewRelicMonitor {
  constructor() {
    this.monitor = monitorEventLoopDelay({ resolution: 10 });
    this.monitor.enable();
    
    setInterval(() => {
      this.reportMetrics();
    }, 10000);
  }

  reportMetrics() {
    const p50 = this.monitor.percentile(50) / 1e6;
    const p95 = this.monitor.percentile(95) / 1e6;
    const p99 = this.monitor.percentile(99) / 1e6;
    const mean = this.monitor.mean / 1e6;

    // 上报到 New Relic
    newrelic.recordMetric('Custom/EventLoop/P50', p50);
    newrelic.recordMetric('Custom/EventLoop/P95', p95);
    newrelic.recordMetric('Custom/EventLoop/P99', p99);
    newrelic.recordMetric('Custom/EventLoop/Mean', mean);

    this.monitor.reset();
  }
}

module.exports = NewRelicMonitor;
```
