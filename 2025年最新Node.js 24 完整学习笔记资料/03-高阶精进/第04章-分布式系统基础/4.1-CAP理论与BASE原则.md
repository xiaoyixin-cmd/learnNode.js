# 4.1 CAP 理论与 BASE 原则：最终一致性设计

> 分布式系统的基础理论：理解不可能三角，拥抱最终一致性

---

## 📚 核心知识

### 1. CAP 定理

**CAP 定理**由 Eric Brewer 于 2000 年提出，指出分布式系统**不可能同时满足**以下三个特性：

```
┌─────────────────────────────────────────────────────┐
│                  CAP 定理                            │
├─────────────────────────────────────────────────────┤
│                                                     │
│        C (Consistency)                              │
│        一致性                                        │
│        ┌──────────────────┐                         │
│        │ 所有节点同一时间 │                         │
│        │ 看到相同的数据   │                         │
│        └─────────┬────────┘                         │
│                  │                                  │
│     ┌────────────┼────────────┐                    │
│     │            │            │                    │
│     ▼            ▼            ▼                    │
│  ┌─────┐      ┌─────┐      ┌─────┐               │
│  │Node1│◄────►│Node2│◄────►│Node3│               │
│  └─────┘      └─────┘      └─────┘               │
│     ▲                          ▲                   │
│     │                          │                   │
│     │    P (Partition         │    A (Availability)│
│     │     Tolerance)           │    可用性          │
│     │    分区容错性             │                    │
│     │  ┌─────────────┐         │  ┌──────────────┐│
│     │  │ 网络分区时   │         │  │ 任何时候读写 ││
│     └──│ 系统仍可运行 │         └──│ 请求都能成功 ││
│        └─────────────┘            └──────────────┘│
│                                                     │
│        ⚠️ 只能同时满足其中两个 ⚠️                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**三个特性详解：**

#### 1.1 Consistency（一致性）

**定义：**所有节点在同一时刻看到相同的数据。

**示例：**
```javascript
// 写入 Node1
await node1.set('user:123:balance', 1000);

// 立即从 Node2 读取
const balance = await node2.get('user:123:balance');

// 强一致性：balance === 1000（必定读到最新值）
// 弱一致性：balance 可能是旧值（如 500）
```

**实现方式：**
- 同步复制：写入所有节点后才返回成功
- 读最新版本：读取时验证数据版本

**代价：**
- 写入延迟高（等待所有节点确认）
- 可用性下降（部分节点故障时无法写入）

#### 1.2 Availability（可用性）

**定义：**任何非故障节点都能在合理时间内响应请求。

**示例：**
```javascript
// Node2 宕机，从 Node1 读取
const balance = await node1.get('user:123:balance');
// 必须能成功返回（即使数据可能不是最新）

// 高可用：即使部分节点故障，仍可读写
// 低可用：节点故障导致服务不可用
```

**实现方式：**
- 多副本：数据冗余存储
- 异步复制：不等待所有节点确认

**代价：**
- 数据可能不一致（读到旧值）
- 需要冲突解决机制

#### 1.3 Partition Tolerance（分区容错性）

**定义：**网络分区（部分节点无法通信）时，系统仍能继续运行。

**示例：**
```
正常情况：
Node1 ◄──────► Node2 ◄──────► Node3
  ↑                              ↑
  └──────────────────────────────┘

网络分区：
Node1   ✗ ✗ ✗   Node2 ◄──────► Node3
(孤立)           (可通信)

分区容错：
- Node1 仍可提供服务（可能数据旧）
- Node2/Node3 仍可提供服务
```

**现实：**
- 网络分区是**不可避免**的（交换机故障、光纤断裂、机房故障）
- **P 是必选项**（必须容忍分区）

**因此，CAP 的实际选择是：**
```
CP：牺牲可用性，保证一致性
AP：牺牲一致性，保证可用性
```

### 2. CP 系统 vs AP 系统

#### 2.1 CP 系统（Consistency + Partition Tolerance）

**特点：**
- 强一致性
- 网络分区时，少数派节点拒绝服务

**代表系统：**
- **ZooKeeper**：集群过半节点可用才能提供服务
- **etcd**：Raft 共识算法，牺牲可用性保证一致性
- **HBase**：强一致性，分区时部分数据不可用
- **Redis Cluster**（强一致模式）

**实现原理（ZooKeeper 为例）：**
```javascript
// 5 节点集群，网络分区导致 2|3 分裂
Cluster: [Node1, Node2] | [Node3, Node4, Node5]
                ↑                    ↑
            少数派                 多数派
         拒绝服务 ❌            继续服务 ✅

// 少数派节点返回错误
try {
  await node1.create('/lock/resource');
} catch (error) {
  // Error: Not connected (没有多数派支持)
}

// 多数派节点正常工作
await node3.create('/lock/resource');  // ✅ 成功
```

**适用场景：**
- 金融交易系统（账户余额必须准确）
- 分布式锁（必须保证互斥）
- 配置中心（配置更新必须一致）

**缺点：**
```
❌ 可用性低（过半节点故障时不可用）
❌ 性能低（需要多节点确认）
❌ 网络抖动敏感（短暂分区也会拒绝服务）
```

#### 2.2 AP 系统（Availability + Partition Tolerance）

**特点：**
- 高可用性
- 网络分区时，所有节点继续服务
- 最终一致性

**代表系统：**
- **Cassandra**：多副本异步复制，可配置一致性级别
- **DynamoDB**：Amazon 的 AP 系统
- **CouchDB**：最终一致性，支持多主复制
- **Redis**（主从复制，异步）

**实现原理（Cassandra 为例）：**
```javascript
// 5 节点集群，网络分区导致 2|3 分裂
Cluster: [Node1, Node2] | [Node3, Node4, Node5]
              ↑                    ↑
          继续服务 ✅            继续服务 ✅

// 所有节点都接受写入
await node1.set('user:123:balance', 1000);  // ✅ 成功
await node3.set('user:123:balance', 1200);  // ✅ 成功（冲突）

// 分区恢复后，使用冲突解决策略（如 LWW - Last Write Wins）
// 最终所有节点收敛到一致的值（1200）
```

**适用场景：**
- 社交网络（点赞数、粉丝数）
- 内容推荐（推荐列表）
- 缓存系统（Cache-Aside）
- 电商商品浏览（商品信息）

**缺点：**
```
❌ 数据可能不一致（读到旧值或冲突值）
❌ 需要冲突解决机制
❌ 业务逻辑复杂（需处理数据不一致场景）
```

### 3. BASE 理论

**BASE** 是对 CAP 中 AP 系统的进一步扩展，由 eBay 架构师 Dan Pritchett 提出。

```
BASE = Basically Available + Soft state + Eventually consistent

┌─────────────────────────────────────────────────────┐
│                   BASE 理论                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  BA (Basically Available)                           │
│  基本可用                                            │
│  ┌────────────────────────────────────────┐        │
│  │ 允许损失部分可用性（如响应时间增加）    │        │
│  │ 分区时仍可提供基本服务                 │        │
│  └────────────────────────────────────────┘        │
│                                                     │
│  S (Soft state)                                     │
│  软状态                                              │
│  ┌────────────────────────────────────────┐        │
│  │ 允许系统中间状态（如数据同步中）        │        │
│  │ 数据可以在不同节点间不一致              │        │
│  └────────────────────────────────────────┘        │
│                                                     │
│  E (Eventually consistent)                          │
│  最终一致性                                          │
│  ┌────────────────────────────────────────┐        │
│  │ 经过一段时间后，数据最终会一致          │        │
│  │ 不保证任意时刻一致                     │        │
│  └────────────────────────────────────────┘        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**BASE vs ACID 对比：**

| 特性 | ACID（传统数据库） | BASE（分布式系统） |
|------|-------------------|-------------------|
| **一致性** | 强一致性 | 最终一致性 |
| **可用性** | 单点故障时不可用 | 高可用 |
| **性能** | 较低（锁竞争） | 高（异步复制） |
| **复杂度** | 简单 | 复杂（冲突解决） |
| **适用场景** | 金融交易 | 互联网应用 |

### 4. 最终一致性模型

**最终一致性**是一种弱一致性，保证在没有新更新的情况下，最终所有副本会收敛到一致状态。

**时间线示例：**

```
时间轴：
T0: 用户在 Node1 写入 balance=1000
T1: Node1 异步复制到 Node2（10ms 延迟）
T2: Node1 异步复制到 Node3（15ms 延迟）

T0        T1        T2        T3
│         │         │         │
├─────────┼─────────┼─────────┤
│         │         │         │
Node1: 1000 ────────┼─────────┤ 1000（一致）
│         │         │         │
Node2: 500 ─────► 1000 ───────┤ 1000（一致）
│         │         │         │
Node3: 500 ─────────┴─────► 1000（一致）
│         │         │         │
└─────────┴─────────┴─────────┘

T0-T1：不一致窗口（Node1=1000, Node2/3=500）
T3：最终一致（所有节点=1000）
```

**一致性级别（Cassandra 为例）：**

```javascript
// ONE：只需一个节点确认（最快，一致性最弱）
await cassandra.execute(
  'INSERT INTO users (id, balance) VALUES (?, ?)',
  ['123', 1000],
  { consistency: cassandra.types.consistencies.one }
);

// QUORUM：多数节点确认（平衡）
await cassandra.execute(
  'INSERT INTO users (id, balance) VALUES (?, ?)',
  ['123', 1000],
  { consistency: cassandra.types.consistencies.quorum }
);

// ALL：所有节点确认（最慢，一致性最强）
await cassandra.execute(
  'INSERT INTO users (id, balance) VALUES (?, ?)',
  ['123', 1000],
  { consistency: cassandra.types.consistencies.all }
);
```

## 💻 实战代码

### 实战1：最终一致性实现（主从复制）

```javascript
// master-slave-replication.js
import Redis from 'ioredis';
import EventEmitter from 'events';

// 主节点
class MasterNode extends EventEmitter {
  constructor() {
    super();
    this.data = new Map();
    this.version = 0;
  }
  
  // 写入数据
  async set(key, value) {
    this.version++;
    
    this.data.set(key, {
      value,
      version: this.version,
      timestamp: Date.now(),
    });
    
    console.log(`[Master] SET ${key}=${value} (v${this.version})`);
    
    // 异步复制到从节点
    this.emit('replicate', {
      operation: 'set',
      key,
      value,
      version: this.version,
    });
    
    return true;
  }
  
  // 读取数据
  get(key) {
    const data = this.data.get(key);
    return data ? data.value : null;
  }
}

// 从节点
class SlaveNode {
  constructor(id, delayMs = 0) {
    this.id = id;
    this.data = new Map();
    this.delayMs = delayMs;  // 模拟网络延迟
  }
  
  // 应用复制日志
  async applyReplication(operation) {
    // 模拟网络延迟
    await new Promise(resolve => setTimeout(resolve, this.delayMs));
    
    if (operation.operation === 'set') {
      const { key, value, version } = operation;
      
      // 检查版本号，避免乱序
      const current = this.data.get(key);
      if (!current || current.version < version) {
        this.data.set(key, { value, version });
        console.log(`[Slave${this.id}] Applied: ${key}=${value} (v${version}) [delay: ${this.delayMs}ms]`);
      } else {
        console.log(`[Slave${this.id}] Skipped: ${key}=${value} (v${version}) [old version]`);
      }
    }
  }
  
  // 读取数据
  get(key) {
    const data = this.data.get(key);
    return data ? data.value : null;
  }
}

// 测试最终一致性
async function testEventualConsistency() {
  const master = new MasterNode();
  
  // 创建3个从节点，不同延迟
  const slave1 = new SlaveNode(1, 10);   // 10ms 延迟
  const slave2 = new SlaveNode(2, 50);   // 50ms 延迟
  const slave3 = new SlaveNode(3, 100);  // 100ms 延迟
  
  const slaves = [slave1, slave2, slave3];
  
  // 监听主节点的复制事件
  master.on('replicate', (operation) => {
    slaves.forEach(slave => slave.applyReplication(operation));
  });
  
  // 写入数据
  await master.set('balance', 1000);
  
  // 立即读取（不一致窗口）
  console.log('\n=== Immediate Read (Inconsistent) ===');
  console.log('Master:', master.get('balance'));
  console.log('Slave1:', slave1.get('balance'));  // null（还未复制）
  console.log('Slave2:', slave2.get('balance'));  // null
  console.log('Slave3:', slave3.get('balance'));  // null
  
  // 等待 20ms（Slave1 已同步）
  await new Promise(resolve => setTimeout(resolve, 20));
  console.log('\n=== After 20ms (Slave1 Synced) ===');
  console.log('Master:', master.get('balance'));
  console.log('Slave1:', slave1.get('balance'));  // 1000（已同步）
  console.log('Slave2:', slave2.get('balance'));  // null
  console.log('Slave3:', slave3.get('balance'));  // null
  
  // 等待 100ms（所有从节点已同步）
  await new Promise(resolve => setTimeout(resolve, 100));
  console.log('\n=== After 120ms (Eventually Consistent) ===');
  console.log('Master:', master.get('balance'));
  console.log('Slave1:', slave1.get('balance'));  // 1000
  console.log('Slave2:', slave2.get('balance'));  // 1000
  console.log('Slave3:', slave3.get('balance'));  // 1000
  console.log('\n✅ All nodes are eventually consistent!');
}

testEventualConsistency();
```

**输出：**
```
[Master] SET balance=1000 (v1)
[Slave1] Applied: balance=1000 (v1) [delay: 10ms]

=== Immediate Read (Inconsistent) ===
Master: 1000
Slave1: null
Slave2: null
Slave3: null

[Slave2] Applied: balance=1000 (v1) [delay: 50ms]

=== After 20ms (Slave1 Synced) ===
Master: 1000
Slave1: 1000
Slave2: null
Slave3: null

[Slave3] Applied: balance=1000 (v1) [delay: 100ms]

=== After 120ms (Eventually Consistent) ===
Master: 1000
Slave1: 1000
Slave2: 1000
Slave3: 1000

✅ All nodes are eventually consistent!
```

### 实战2：读写分离（主节点写，从节点读）

```javascript
// read-write-splitting.js
import Redis from 'ioredis';

class ReadWriteSplitting {
  constructor() {
    // 主节点（写）
    this.master = new Redis({
      host: '192.168.1.10',
      port: 6379,
    });
    
    // 从节点（读）- 负载均衡
    this.slaves = [
      new Redis({ host: '192.168.1.11', port: 6379 }),
      new Redis({ host: '192.168.1.12', port: 6379 }),
      new Redis({ host: '192.168.1.13', port: 6379 }),
    ];
    
    this.currentSlaveIndex = 0;
  }
  
  // 写入（主节点）
  async set(key, value) {
    await this.master.set(key, value);
    console.log(`✍️ Write to master: ${key}=${value}`);
  }
  
  // 读取（从节点，轮询负载均衡）
  async get(key) {
    const slave = this.slaves[this.currentSlaveIndex];
    this.currentSlaveIndex = (this.currentSlaveIndex + 1) % this.slaves.length;
    
    const value = await slave.get(key);
    console.log(`📖 Read from slave${this.currentSlaveIndex}: ${key}=${value}`);
    
    return value;
  }
  
  // 强一致性读取（从主节点读）
  async getStrong(key) {
    const value = await this.master.get(key);
    console.log(`📖 Read from master (strong consistency): ${key}=${value}`);
    return value;
  }
}

// 使用示例
async function main() {
  const db = new ReadWriteSplitting();
  
  // 写入
  await db.set('user:123:balance', 1000);
  
  // 立即读取（可能读到旧值）
  const balance1 = await db.get('user:123:balance');
  console.log('Balance (may be stale):', balance1);
  
  // 强一致性读取（从主节点读，必定是最新值）
  const balance2 = await db.getStrong('user:123:balance');
  console.log('Balance (strong consistency):', balance2);
  
  // 等待主从同步后再读取
  await new Promise(resolve => setTimeout(resolve, 100));
  const balance3 = await db.get('user:123:balance');
  console.log('Balance (eventually consistent):', balance3);
}

main();
```

### 实战3：向量时钟（Vector Clock）解决冲突

```javascript
// vector-clock.js
class VectorClock {
  constructor() {
    this.clock = {};
  }
  
  // 递增节点的时钟
  increment(nodeId) {
    this.clock[nodeId] = (this.clock[nodeId] || 0) + 1;
  }
  
  // 合并两个向量时钟
  merge(other) {
    const merged = new VectorClock();
    
    // 合并所有节点的时钟，取最大值
    const allNodes = new Set([
      ...Object.keys(this.clock),
      ...Object.keys(other.clock),
    ]);
    
    for (const nodeId of allNodes) {
      merged.clock[nodeId] = Math.max(
        this.clock[nodeId] || 0,
        other.clock[nodeId] || 0
      );
    }
    
    return merged;
  }
  
  // 比较两个向量时钟
  compare(other) {
    const allNodes = new Set([
      ...Object.keys(this.clock),
      ...Object.keys(other.clock),
    ]);
    
    let lessThan = false;
    let greaterThan = false;
    
    for (const nodeId of allNodes) {
      const thisClock = this.clock[nodeId] || 0;
      const otherClock = other.clock[nodeId] || 0;
      
      if (thisClock < otherClock) lessThan = true;
      if (thisClock > otherClock) greaterThan = true;
    }
    
    if (lessThan && !greaterThan) return -1;  // this < other
    if (greaterThan && !lessThan) return 1;   // this > other
    if (!lessThan && !greaterThan) return 0;  // this == other
    return null;  // 冲突（并发修改）
  }
  
  toString() {
    return JSON.stringify(this.clock);
  }
}

// 版本化数据
class VersionedData {
  constructor(value, nodeId) {
    this.value = value;
    this.vectorClock = new VectorClock();
    this.vectorClock.increment(nodeId);
  }
  
  // 更新数据
  update(value, nodeId) {
    this.value = value;
    this.vectorClock.increment(nodeId);
  }
}

// 测试向量时钟
function testVectorClock() {
  console.log('=== Vector Clock Conflict Resolution ===\n');
  
  // Node1 写入 balance=1000
  const data1 = new VersionedData(1000, 'node1');
  console.log(`Node1 writes: balance=${data1.value}, clock=${data1.vectorClock.toString()}`);
  
  // 网络分区：Node2 和 Node3 同时更新
  
  // Node2 更新 balance=1200
  const data2 = new VersionedData(data1.value, 'node1');
  data2.update(1200, 'node2');
  console.log(`Node2 updates: balance=${data2.value}, clock=${data2.vectorClock.toString()}`);
  
  // Node3 更新 balance=1500
  const data3 = new VersionedData(data1.value, 'node1');
  data3.update(1500, 'node3');
  console.log(`Node3 updates: balance=${data3.value}, clock=${data3.vectorClock.toString()}`);
  
  // 比较版本
  const comparison = data2.vectorClock.compare(data3.vectorClock);
  
  if (comparison === null) {
    console.log('\n⚠️ Conflict detected! (Concurrent updates)');
    console.log('Node2 value:', data2.value);
    console.log('Node3 value:', data3.value);
    
    // 冲突解决策略：Last Write Wins（取时间戳较大的）
    const resolved = data2.value > data3.value ? data2 : data3;
    console.log(`✅ Resolved value: ${resolved.value} (using LWW)`);
    
    // 合并向量时钟
    const mergedClock = data2.vectorClock.merge(data3.vectorClock);
    console.log(`Merged clock: ${mergedClock.toString()}`);
  } else if (comparison < 0) {
    console.log('\nNode2 < Node3 (Node3 is newer)');
  } else if (comparison > 0) {
    console.log('\nNode2 > Node3 (Node2 is newer)');
  } else {
    console.log('\nNode2 == Node3 (Same version)');
  }
}

testVectorClock();
```

**输出：**
```
=== Vector Clock Conflict Resolution ===

Node1 writes: balance=1000, clock={"node1":1}
Node2 updates: balance=1200, clock={"node1":1,"node2":1}
```

## 🎯 应用场景

### 场景1：电商库存扣减（AP 系统）

**需求：**
- 高并发秒杀场景
- 允许短暂超卖（事后补偿）
- 优先保证可用性

**方案：最终一致性 + 异步对账**

```javascript
// inventory-ap-system.js
import Redis from 'ioredis';

class InventoryService {
  constructor() {
    this.redis = new Redis();
    this.compensationQueue = [];
  }
  
  // 扣减库存（乐观，允许超卖）
  async deduct(productId, quantity) {
    // 读取库存
    const stock = await this.redis.get(`stock:${productId}`);
    const currentStock = parseInt(stock) || 0;
    
    // 乐观扣减（不加锁，高性能）
    if (currentStock >= quantity) {
      await this.redis.decrby(`stock:${productId}`, quantity);
      
      console.log(`✅ Deducted ${quantity} from product ${productId}`);
      return { success: true, stock: currentStock - quantity };
    } else {
      console.log(`❌ Insufficient stock for product ${productId}`);
      return { success: false, stock: currentStock };
    }
  }
  
  // 异步对账（定时任务）
  async reconcile(productId) {
    const redisStock = await this.redis.get(`stock:${productId}`);
    const dbStock = await this.getStockFromDB(productId);
    
    if (parseInt(redisStock) !== dbStock) {
      console.log(`⚠️ Stock mismatch for ${productId}: Redis=${redisStock}, DB=${dbStock}`);
      
      // 以数据库为准，修复 Redis
      await this.redis.set(`stock:${productId}`, dbStock);
      console.log(`✅ Reconciled stock for ${productId} to ${dbStock}`);
    }
  }
  
  // 模拟从数据库读取
  async getStockFromDB(productId) {
    // 实际应从 PostgreSQL/MySQL 查询
    return 100;
  }
}

// 测试
async function main() {
  const service = new InventoryService();
  
  // 初始化库存
  await service.redis.set('stock:SKU001', 100);
  
  // 并发扣减（可能超卖）
  await Promise.all([
    service.deduct('SKU001', 50),
    service.deduct('SKU001', 60),  // 可能成功（超卖 10 件）
  ]);
  
  // 异步对账（1小时后）
  setTimeout(async () => {
    await service.reconcile('SKU001');
  }, 3600000);
}

main();
```

### 场景2：金融转账（CP 系统）

**需求：**
- 强一致性（不能丢失或重复转账）
- 网络分区时宁可拒绝服务

**方案：两阶段提交（2PC）**

```javascript
// transfer-cp-system.js
import { Client as PgClient } from 'pg';

class TransferService {
  constructor() {
    this.db1 = new PgClient({
      host: 'db1.example.com',
      database: 'accounts_shard1',
    });
    
    this.db2 = new PgClient({
      host: 'db2.example.com',
      database: 'accounts_shard2',
    });
  }
  
  // 两阶段提交转账
  async transfer(fromAccountId, toAccountId, amount) {
    await this.db1.connect();
    await this.db2.connect();
    
    try {
      // === 阶段1：准备（Prepare） ===
      await this.db1.query('BEGIN');
      await this.db2.query('BEGIN');
      
      // 扣减 A 账户
      const result1 = await this.db1.query(
        'UPDATE accounts SET balance = balance - $1 WHERE id = $2 AND balance >= $1 RETURNING balance',
        [amount, fromAccountId]
      );
      
      if (result1.rows.length === 0) {
        throw new Error('Insufficient balance');
      }
      
      // 增加 B 账户
      await this.db2.query(
        'UPDATE accounts SET balance = balance + $1 WHERE id = $2',
        [amount, toAccountId]
      );
      
      // 准备提交（等待所有参与者确认）
      await this.db1.query('PREPARE TRANSACTION \'tx1\'');
      await this.db2.query('PREPARE TRANSACTION \'tx2\'');
      
      console.log('✅ Phase 1 (Prepare) completed');
      
      // === 阶段2：提交（Commit） ===
      await this.db1.query('COMMIT PREPARED \'tx1\'');
      await this.db2.query('COMMIT PREPARED \'tx2\'');
      
      console.log('✅ Phase 2 (Commit) completed');
      console.log(`✅ Transferred ${amount} from ${fromAccountId} to ${toAccountId}`);
      
      return { success: true };
    } catch (error) {
      // 任何阶段失败，回滚所有事务
      console.error('❌ Transfer failed:', error.message);
      
      try {
        await this.db1.query('ROLLBACK PREPARED \'tx1\'');
      } catch (e) {}
      
      try {
        await this.db2.query('ROLLBACK PREPARED \'tx2\'');
      } catch (e) {}
      
      return { success: false, error: error.message };
    } finally {
      await this.db1.end();
      await this.db2.end();
    }
  }
}

// 测试
async function main() {
  const service = new TransferService();
  
  // 转账 100 元
  await service.transfer('account123', 'account456', 100);
}

main();
```

## ❌ 高频错误

### 错误1：误以为 Redis 主从复制是强一致性

**错误代码：**
```javascript
// ❌ 错误：假设主从同步是瞬时的
await redisMaster.set('user:123:balance', 1000);

// 立即从从节点读取
const balance = await redisSlave.get('user:123:balance');
console.log(balance);  // 可能是 null 或旧值！
```

**原因：**
- Redis 主从复制是**异步**的
- 主节点写入成功不代表从节点已同步

**正确做法：**
```javascript
// ✅ 方案1：从主节点读（强一致性）
await redisMaster.set('user:123:balance', 1000);
const balance = await redisMaster.get('user:123:balance');  // 必定最新

// ✅ 方案2：使用 WAIT 命令（等待同步）
await redisMaster.set('user:123:balance', 1000);
await redisMaster.wait(1, 1000);  // 等待至少 1 个从节点同步，超时 1 秒

const balance = await redisSlave.get('user:123:balance');  // 已同步
```

### 错误2：未处理网络分区导致的脑裂

**错误代码：**
```javascript
// ❌ 错误：未处理脑裂场景
class Cluster {
  constructor() {
    this.isMaster = true;  // 假设自己是主节点
  }
  
  async write(key, value) {
    if (this.isMaster) {
      await this.db.set(key, value);
    } else {
      throw new Error('Not master');
    }
  }
}

// 问题：网络分区时，两个节点都认为自己是主节点
// Node1 和 Node2 同时写入，导致数据不一致
```

**正确做法：使用 Raft/Paxos 共识算法**
```javascript
// ✅ 使用 etcd（内置 Raft）
import { Etcd3 } from 'etcd3';

const etcd = new Etcd3();

// 选举主节点（自动处理脑裂）
const lease = etcd.lease(10);  // 10 秒租约
await lease.grant();

try {
  await etcd.put('/master')
    .value(process.env.NODE_ID)
    .lease(lease)
    .exec();
  
  console.log('✅ I am the master');
  
  // 保持租约
  await lease.keepalive();
} catch (error) {
  console.log('❌ Another node is master');
}
```

### 错误3：忽略最终一致性的不一致窗口

**错误代码：**
```javascript
// ❌ 错误：假设写入后立即一致
await writeToMaster('user:123:score', 100);

// 立即读取并做业务判断
const score = await readFromSlave('user:123:score');

if (score >= 100) {
  // 发放奖励
  await grantReward('user:123');  // 可能重复发放！
}
```

**正确做法：**
```javascript
// ✅ 方案1：从主节点读（强一致性）
await writeToMaster('user:123:score', 100);
const score = await readFromMaster('user:123:score');  // 强一致

if (score >= 100) {
  await grantReward('user:123');
}

// ✅ 方案2：使用版本号/时间戳
await writeToMaster('user:123:score', {
  value: 100,
  version: 2,
  timestamp: Date.now(),
});

const data = await readFromSlave('user:123:score');

if (data.version >= 2 && data.value >= 100) {
  await grantReward('user:123');
}
```

## 🤔 FAQ

### Q1：什么时候选择 CP，什么时候选择 AP？

**选择 CP（强一致性）：**
- ✅ 金融交易（账户余额、转账）
- ✅ 库存扣减（严格防止超卖）
- ✅ 分布式锁（必须互斥）
- ✅ 配置中心（配置变更必须一致）

**选择 AP（高可用性）：**
- ✅ 社交网络（点赞数、粉丝数、评论数）
- ✅ 内容推荐（推荐列表可以不一致）
- ✅ 缓存系统（Cache-Aside）
- ✅ 监控指标（可以容忍短暂不准确）

**混合方案（大多数场景）：**
```javascript
// 核心数据：CP（强一致性）
await transferMoney(fromAccount, toAccount, 100);  // 2PC

// 非核心数据：AP（最终一致性）
await updateUserStats(userId, { lastLoginAt: new Date() });  // 异步
```

### Q2：如何测量最终一致性的延迟？

**方法：在数据中嵌入时间戳**

```javascript
// 写入时记录时间戳
await master.set('key', JSON.stringify({
  value: 'data',
  writeTime: Date.now(),
}));

// 从从节点读取
const data = JSON.parse(await slave.get('key'));
const replicationLag = Date.now() - data.writeTime;

console.log(`Replication lag: ${replicationLag}ms`);

// 监控告警
if (replicationLag > 1000) {
  console.error('⚠️ High replication lag!');
}
```

### Q3：如何实现读己之写（Read Your Writes）？

**问题：用户写入数据后，立即读取却看不到自己的更新**

**解决方案：**

```javascript
// 方案1：从主节点读（简单但主节点压力大）
await master.set('user:123:profile', updatedProfile);
const profile = await master.get('user:123:profile');  // 必定读到

// 方案2：使用会话亲和（Session Affinity）
// 同一用户的读写请求路由到同一节点

// 方案3：使用版本号
await master.set('user:123:profile', {
  ...updatedProfile,
  version: 2,
});

// 从从节点读，但只接受 version >= 2 的数据
let profile;
do {
  profile = await slave.get('user:123:profile');
  if (profile.version < 2) {
    await new Promise(r => setTimeout(r, 10));  // 等待同步
  }
} while (profile.version < 2);
```
