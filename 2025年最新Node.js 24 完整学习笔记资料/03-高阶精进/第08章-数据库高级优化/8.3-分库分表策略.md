# 8.3 分库分表策略

> 使用 ShardingSphere 实现分布式数据库，突破单表亿级数据瓶颈

---

## 📚 知识导读

### 为什么需要分库分表？

**单表数据量瓶颈：**
```
订单表（orders）：1亿条数据

问题：
- 查询慢：SELECT * FROM orders WHERE user_id = 123  (5秒+)
- 写入慢：INSERT INTO orders (...) (1秒+)
- 索引大：B+Tree 索引树高度增加
- 磁盘 I/O 饱和

MySQL 单表建议：< 2000万行
MongoDB 单集合建议：< 1亿文档
```

**分库分表架构：**
```
原始（单库单表）：
┌─────────────────┐
│   Database      │
│  ┌───────────┐  │
│  │  orders   │  │  1亿条
│  └───────────┘  │
└─────────────────┘

分库分表后（4库×4表=16张表）：
┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
│  db_0   │  │  db_1   │  │  db_2   │  │  db_3   │
│ orders_0│  │ orders_0│  │ orders_0│  │ orders_0│
│ orders_1│  │ orders_1│  │ orders_1│  │ orders_1│
│ orders_2│  │ orders_2│  │ orders_2│  │ orders_2│
│ orders_3│  │ orders_3│  │ orders_3│  │ orders_3│
└─────────┘  └─────────┘  └─────────┘  └─────────┘
  625万条     625万条     625万条     625万条

性能提升：10-100倍 ✅
```

## 🎯 分片策略

### 1. 垂直分片（按业务拆分）

**垂直分库：**
```
原始（单库）：
┌─────────────────┐
│   myapp_db      │
│  ┌───────────┐  │
│  │  users    │  │
│  │  orders   │  │
│  │  products │  │
│  └───────────┘  │
└─────────────────┘

垂直分库后：
┌──────────┐  ┌──────────┐  ┌──────────┐
│ user_db  │  │ order_db │  │ prod_db  │
│  users   │  │  orders  │  │ products │
└──────────┘  └──────────┘  └──────────┘

优势：
- 业务解耦
- 分散压力
- 专库专用（不同配置）
```

**垂直分表：**
```
原始（宽表）：
users (50个字段)
- id
- name
- email
- ... (46个字段)

垂直分表后：
users_base (常用字段)        users_ext (扩展字段)
- id                         - user_id (FK)
- name                       - address
- email                      - bio
- created_at                 - ... (43个字段)

优势：
- 减少单表大小
- 提高缓存命中率
- 冷热数据分离
```

---

### 2. 水平分片（按数据拆分）

**分片键选择：**

| 分片键 | 示例 | 优势 | 劣势 |
|--------|------|------|------|
| **用户 ID** | user_id | 同一用户数据在同一分片，查询快 | 可能数据分布不均 |
| **订单 ID** | order_id | 数据分布均匀 | 按用户查询需跨分片 |
| **时间** | created_at | 方便归档旧数据 | 热点问题（新数据集中） |
| **地理位置** | region | 就近访问，延迟低 | 数据分布可能不均 |

### 3. 分片算法

**取模（Mod）算法：**
```sql
-- 分片规则：user_id % 4
user_id = 123 → 123 % 4 = 3 → db_3.orders_3
user_id = 456 → 456 % 4 = 0 → db_0.orders_0

优势：
- 简单、快速
- 数据分布均匀

劣势：
- 扩容困难（需要重新分片）
```

**范围（Range）算法：**
```sql
-- 分片规则：user_id 范围
user_id [0, 250000)   → db_0
user_id [250000, 500000)  → db_1
user_id [500000, 750000)  → db_2
user_id [750000, ∞)   → db_3

优势：
- 扩容方便（只需调整边界）
- 范围查询高效

劣势：
- 数据分布可能不均（热点问题）
```

**哈希（Hash）算法：**
```sql
-- 分片规则：hash(user_id) % 4
hash(123) % 4 = 2 → db_2.orders_2

优势：
- 数据分布均匀

劣势：
- 扩容需要重新哈希
```

**一致性哈希（Consistent Hash）：**
```
哈希环（0-2^32）

节点：
db_0: hash("db_0") = 1000
db_1: hash("db_1") = 2000
db_2: hash("db_2") = 3000
db_3: hash("db_3") = 4000

数据：
user_id = 123 → hash(123) = 1500 → 顺时针找到 db_1

优势：
- 扩容时只需迁移部分数据
- 虚拟节点解决数据倾斜

劣势：
- 实现复杂
```

### 2. 配置分片规则

```javascript
// sharding-config.js
export default {
  // 数据源配置
  dataSources: {
    db_0: {
      url: 'jdbc:mysql://localhost:3306/db_0',
      username: 'root',
      password: 'password',
    },
    db_1: {
      url: 'jdbc:mysql://localhost:3307/db_1',
      username: 'root',
      password: 'password',
    },
    db_2: {
      url: 'jdbc:mysql://localhost:3308/db_2',
      username: 'root',
      password: 'password',
    },
    db_3: {
      url: 'jdbc:mysql://localhost:3309/db_3',
      username: 'root',
      password: 'password',
    },
  },

  // 分片规则
  shardingRule: {
    tables: {
      orders: {
        // 实际节点
        actualDataNodes: 'db_${0..3}.orders_${0..3}',

        // 数据库分片策略
        databaseShardingStrategy: {
          standard: {
            shardingColumn: 'user_id',
            shardingAlgorithm: {
              type: 'INLINE',
              props: {
                'algorithm-expression': 'db_${user_id % 4}',
              },
            },
          },
        },

        // 表分片策略
        tableShardingStrategy: {
          standard: {
            shardingColumn: 'order_id',
            shardingAlgorithm: {
              type: 'INLINE',
              props: {
                'algorithm-expression': 'orders_${order_id % 4}',
              },
            },
          },
        },

        // 分布式主键生成策略
        keyGenerateStrategy: {
          column: 'order_id',
          keyGeneratorName: 'snowflake',
        },
      },
    },

    // 分布式主键生成器
    keyGenerators: {
      snowflake: {
        type: 'SNOWFLAKE',
        props: {
          'worker-id': 1,
        },
      },
    },
  },

  // SQL 输出（调试用）
  props: {
    'sql-show': true,
  },
};
```

### 4. Node.js 集成

```javascript
// database.js
import ShardingSphere from 'shardingsphere-jdbc';
import shardingConfig from './sharding-config.js';

const dataSource = new ShardingSphere.ShardingDataSource(shardingConfig);

class OrderDAO {
  // 插入订单
  static async create(order) {
    const sql = `INSERT INTO orders (user_id, amount, status) VALUES (?, ?, ?)`;
    const result = await dataSource.execute(sql, [order.userId, order.amount, order.status]);
    return result.insertId;
  }

  // 根据订单 ID 查询（单分片）
  static async findById(orderId) {
    const sql = `SELECT * FROM orders WHERE order_id = ?`;
    const [rows] = await dataSource.execute(sql, [orderId]);
    return rows[0];
  }

  // 根据用户 ID 查询（单分片）
  static async findByUserId(userId) {
    const sql = `SELECT * FROM orders WHERE user_id = ?`;
    const [rows] = await dataSource.execute(sql, [userId]);
    return rows;
  }

  // 全局查询（跨分片）
  static async findAll() {
    const sql = `SELECT * FROM orders LIMIT 100`;
    const [rows] = await dataSource.execute(sql);
    return rows;
  }
}

export default OrderDAO;
```

### 5. 分布式主键生成（Snowflake）

**Snowflake 算法：**
```
64 位 ID 结构：

|--- 1 bit ---|--- 41 bit ---|--- 10 bit ---|--- 12 bit ---|
  未使用(0)     时间戳(毫秒)    工作机器ID      序列号

示例：
- 时间戳：2025-10-08 12:00:00 → 1728374400000
- 工作机器 ID：1
- 序列号：0-4095（同一毫秒内自增）

生成：1728374400000 << 22 | 1 << 12 | 0
= 7249494917120000000

特点：
- 全局唯一
- 趋势递增（利于 MySQL B+Tree 索引）
- 每毫秒可生成 4096 个 ID
- 支持分布式（通过工作机器 ID）
```

**Node.js 实现：**
```javascript
class Snowflake {
  constructor(workerId = 1, dataCenterId = 1) {
    this.workerId = BigInt(workerId);
    this.dataCenterId = BigInt(dataCenterId);
    this.sequence = 0n;
    this.lastTimestamp = -1n;

    this.workerIdBits = 5n;
    this.dataCenterIdBits = 5n;
    this.sequenceBits = 12n;

    this.maxWorkerId = -1n ^ (-1n << this.workerIdBits);
    this.maxDataCenterId = -1n ^ (-1n << this.dataCenterIdBits);
    this.sequenceMask = -1n ^ (-1n << this.sequenceBits);

    this.workerIdShift = this.sequenceBits;
    this.dataCenterIdShift = this.sequenceBits + this.workerIdBits;
    this.timestampLeftShift = this.sequenceBits + this.workerIdBits + this.dataCenterIdBits;
    this.twepoch = 1288834974657n;
  }

  nextId() {
    let timestamp = BigInt(Date.now());

    if (timestamp < this.lastTimestamp) {
      throw new Error('Clock moved backwards');
    }

    if (timestamp === this.lastTimestamp) {
      this.sequence = (this.sequence + 1n) & this.sequenceMask;
      if (this.sequence === 0n) {
        timestamp = this.tilNextMillis(this.lastTimestamp);
      }
    } else {
      this.sequence = 0n;
    }

    this.lastTimestamp = timestamp;

    return (
      ((timestamp - this.twepoch) << this.timestampLeftShift) |
      (this.dataCenterId << this.dataCenterIdShift) |
      (this.workerId << this.workerIdShift) |
      this.sequence
    );
  }

  tilNextMillis(lastTimestamp) {
    let timestamp = BigInt(Date.now());
    while (timestamp <= lastTimestamp) {
      timestamp = BigInt(Date.now());
    }
    return timestamp;
  }
}

const snowflake = new Snowflake(1, 1);
console.log(snowflake.nextId());  // 7249494917120000001n
```

## 🎯 跨分片查询

### 1. JOIN 查询优化

```sql
-- ❌ 问题：跨分片 JOIN
SELECT o.*, u.name 
FROM orders o 
JOIN users u ON o.user_id = u.id 
WHERE o.status = 'completed';

-- 需要：
-- 1. 从所有分片查询 orders
-- 2. 从所有分片查询 users
-- 3. 在应用层 JOIN（低效）

-- ✅ 解决方案1：冗余字段（空间换时间）
ALTER TABLE orders ADD COLUMN user_name VARCHAR(50);

SELECT * FROM orders WHERE status = 'completed';  -- 单表查询

-- ✅ 解决方案2：分两次查询
const orders = await OrderDAO.findByStatus('completed');
const userIds = orders.map(o => o.user_id);
const users = await UserDAO.findByIds(userIds);
```

---

### 2. 聚合查询

```sql
-- ❌ 问题：跨分片 COUNT
SELECT COUNT(*) FROM orders WHERE status = 'completed';

-- ShardingSphere 处理流程：
-- 1. 路由到所有分片
-- 2. 每个分片执行 COUNT(*)
--    db_0: 1000, db_1: 1200, db_2: 800, db_3: 1500
-- 3. 合并结果：1000 + 1200 + 800 + 1500 = 4500

-- ✅ 优化：使用缓存
const cachedCount = await redis.get('order:completed:count');
if (cachedCount) {
  return cachedCount;
}

const count = await OrderDAO.countByStatus('completed');
await redis.setEx('order:completed:count', 300, count);  // 缓存 5 分钟
```

### 3. 分页查询

```sql
-- ❌ 问题：跨分片分页
SELECT * FROM orders ORDER BY created_at DESC LIMIT 10 OFFSET 20;

-- ShardingSphere 处理流程：
-- 1. 路由到所有分片
-- 2. 每个分片执行 LIMIT 30（20 + 10）
-- 3. 合并结果并排序
-- 4. 返回第 21-30 条

-- 问题：深分页性能差（OFFSET 10000）

-- ✅ 解决方案：游标分页
SELECT * FROM orders 
WHERE order_id > ? 
ORDER BY order_id 
LIMIT 10;
```

## 🎯 数据迁移与扩容

### 1. 双写方案

```javascript
// 场景：从单表迁移到分片表

// 步骤1：双写（写旧表+新表）
async function createOrder(order) {
  // 写旧表
  await db.execute('INSERT INTO orders_old (...)', [order]);

  // 写新表（分片）
  await OrderDAO.create(order);
}

// 步骤2：迁移历史数据
async function migrateData() {
  const batchSize = 1000;
  let offset = 0;

  while (true) {
    const [orders] = await db.execute(
      `SELECT * FROM orders_old LIMIT ? OFFSET ?`,
      [batchSize, offset]
    );

    if (orders.length === 0) break;

    for (const order of orders) {
      await OrderDAO.create(order);
    }

    offset += batchSize;
    console.log(`Migrated ${offset} orders`);
  }
}

// 步骤3：数据一致性校验
async function verifyData() {
  const [oldCount] = await db.execute('SELECT COUNT(*) as count FROM orders_old');
  const newCount = await OrderDAO.count();

  console.log(`Old: ${oldCount[0].count}, New: ${newCount}`);
}

// 步骤4：切换读流量到新表
// 步骤5：停止双写，只写新表
// 步骤6：删除旧表
```

### 2. 在线扩容

```javascript
// 场景：从 4 分片扩容到 8 分片

// 方法1：一致性哈希（推荐）
// - 只需迁移部分数据（约 50%）
// - 使用虚拟节点平衡数据

// 方法2：双倍扩容
// 原：db_{0,1,2,3}
// 新：db_{0,1,2,3,4,5,6,7}

// user_id % 4 → user_id % 8
// 迁移规则：
// db_0 → db_0 (user_id % 8 === 0) + db_4 (user_id % 8 === 4)
// db_1 → db_1 (user_id % 8 === 1) + db_5 (user_id % 8 === 5)
// ...

async function scaleOut() {
  for (let i = 0; i < 4; i++) {
    const sourceDb = `db_${i}`;
    const targetDb = `db_${i + 4}`;

    // 迁移 user_id % 8 === (i + 4) 的数据
    await migrateData(sourceDb, targetDb, i + 4);
  }
}
```

## ⚠️ 三个高频错误

### 错误 1：分片键选择不当

```sql
-- ❌ 错误：使用自增 ID 作为分片键
sharding_column: 'id'  -- 自增 ID

-- 问题：
-- 1. 所有新数据都写入最后一个分片（热点）
-- 2. 无法利用分片优势

-- ✅ 正确：使用业务键
sharding_column: 'user_id'  -- 业务键
```

---

### 错误 2：忽略分布式事务

```javascript
// ❌ 错误：跨分片事务
BEGIN;
UPDATE orders SET status = 'paid' WHERE order_id = 123;  -- db_0
UPDATE users SET balance = balance - 100 WHERE id = 456; -- db_1
COMMIT;

// 问题：跨分片事务无法保证 ACID

// ✅ 解决方案1：避免跨分片事务（业务拆分）
// ✅ 解决方案2：使用 Seata 分布式事务
// ✅ 解决方案3：最终一致性（消息队列）
```

---

### 错误 3：全局查询过多

```sql
-- ❌ 错误：频繁全局查询
SELECT * FROM orders WHERE status = 'pending';  -- 扫描所有分片

-- ✅ 正确：带分片键查询
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';  -- 单分片
```

## 📚 FAQ

### 1. 什么时候需要分库分表？

- 单表数据量 > 2000万
- 单库 QPS > 3000
- 索引文件 > 2GB
- 查询响应时间 > 500ms

### 2. 如何选择分片数量？

**公式：**
```
分片数 = (预期数据量 / 单表容量) × 安全系数

示例：
- 预期 5 年后 10 亿数据
- 单表容量 2000 万
- 安全系数 2

分片数 = (1000000000 / 20000000) × 2 = 100 分片
```

### 3. ShardingSphere vs Vitess vs Citus？

| 特性 | ShardingSphere | Vitess | Citus |
|------|---------------|--------|-------|
| **数据库** | MySQL/PostgreSQL | MySQL | PostgreSQL |
| **架构** | 客户端代理 | 服务端代理 | 扩展插件 |
| **学习曲线** | 低 | 中 | 低 |
| **生态** | 中国流行 | YouTube | Citus Data |
