# 8.1 索引优化与执行计划分析

## 📚 知识导读

### 为什么需要索引？

**没有索引的查询：**
```sql
-- 表：users (1000万条数据)
SELECT * FROM users WHERE email = 'alice@example.com';

-- 执行过程：全表扫描（Table Scan）
-- 时间：5-10 秒 ❌
```

**有索引的查询：**
```sql
-- 创建索引
CREATE INDEX idx_email ON users(email);

-- 执行过程：索引查找（Index Seek）
-- 时间：5-50 毫秒 ✅
```

**性能提升：** 100-1000 倍

## 🎯 MySQL 索引原理

### 1. B+Tree 索引结构

```
MySQL InnoDB 使用 B+Tree 索引

特点：
- 所有数据存储在叶子节点
- 叶子节点通过指针连接（有序链表）
- 非叶子节点只存储索引键
- 高度通常为 3-4 层（支持百万/千万级数据）

示例：索引 age（假设页大小为3）

              [20, 40]
             /    |    \
       [10,15]  [25,30]  [45,50]
       /  |  \   / | \    / | \
     10  15 18  20 25 30  40 45 50  ← 叶子节点（存储完整数据或指针）
      ↓   ↓  ↓   ↓  ↓  ↓   ↓  ↓  ↓
    data data data...

查询：SELECT * FROM users WHERE age = 25
1. 从根节点开始：25 > 20，走中间分支
2. 到达 [25,30]：找到 25
3. 访问叶子节点：获取数据
时间复杂度：O(log n)
```

---

### 2. 聚簇索引 vs 非聚簇索引

**聚簇索引（Clustered Index）：**
```sql
-- InnoDB 主键自动创建聚簇索引
CREATE TABLE users (
  id INT PRIMARY KEY,  -- 聚簇索引
  name VARCHAR(50),
  age INT
);

-- 特点：
-- 1. 叶子节点存储完整行数据
-- 2. 一个表只能有一个聚簇索引
-- 3. 数据按主键顺序物理存储
```

**非聚簇索引（Secondary Index）：**
```sql
-- 创建非聚簇索引
CREATE INDEX idx_age ON users(age);

-- 特点：
-- 1. 叶子节点存储索引键 + 主键值
-- 2. 需要回表（根据主键查询完整数据）
-- 3. 一个表可以有多个非聚簇索引

-- 查询过程
SELECT * FROM users WHERE age = 25;

-- 步骤：
-- 1. 在 idx_age 索引找到 age=25 的主键 id=123
-- 2. 回表：在聚簇索引找到 id=123 的完整数据
```

---

### 3. 覆盖索引（Covering Index）

```sql
-- 问题：回表影响性能
SELECT id, name FROM users WHERE age = 25;
-- age 索引 → 回表获取 name

-- 解决：创建覆盖索引
CREATE INDEX idx_age_name ON users(age, name);

-- 优化后
SELECT id, name FROM users WHERE age = 25;
-- age_name 索引已包含 id, age, name，无需回表 ✅
```

---

### 4. 索引下推（Index Condition Pushdown, ICP）

```sql
-- MySQL 5.6+ 支持 ICP

-- 复合索引
CREATE INDEX idx_age_name ON users(age, name);

-- 查询
SELECT * FROM users WHERE age > 20 AND name LIKE 'A%';

-- 没有 ICP（MySQL 5.5）：
-- 1. 使用 age 索引找到所有 age>20 的行
-- 2. 回表获取完整数据
-- 3. 在 Server 层过滤 name LIKE 'A%'

-- 有 ICP（MySQL 5.6+）：
-- 1. 使用 age 索引找到 age>20 的行
-- 2. 在索引层直接过滤 name LIKE 'A%'  ← 减少回表
-- 3. 回表获取符合条件的数据

-- EXPLAIN 输出：Extra: Using index condition
```

## 🎯 MySQL EXPLAIN 详解

### 1. EXPLAIN 基本用法

```sql
-- 分析查询
EXPLAIN SELECT * FROM users WHERE age = 25;

-- 输出
+----+-------------+-------+------+---------------+---------+---------+-------+------+-------+
| id | select_type | table | type | possible_keys | key     | key_len | ref   | rows | Extra |
+----+-------------+-------+------+---------------+---------+---------+-------+------+-------+
|  1 | SIMPLE      | users | ref  | idx_age       | idx_age | 5       | const |  100 | NULL  |
+----+-------------+-------+------+---------------+---------+---------+-------+------+-------+
```

---

### 2. type 字段详解（访问类型）

**性能从好到坏：**
```
system > const > eq_ref > ref > range > index > ALL
```

**各类型说明：**

```sql
-- 1. system：表只有一行（系统表）
EXPLAIN SELECT * FROM mysql.proxies_priv;
-- type: system

-- 2. const：主键或唯一索引等值查询
EXPLAIN SELECT * FROM users WHERE id = 123;
-- type: const（最多返回一行）

-- 3. eq_ref：连接查询，使用主键或唯一索引
EXPLAIN SELECT * FROM orders o JOIN users u ON o.user_id = u.id;
-- type: eq_ref（对于前表的每一行，后表只返回一行）

-- 4. ref：非唯一索引等值查询
EXPLAIN SELECT * FROM users WHERE age = 25;
-- type: ref（可能返回多行）

-- 5. range：范围查询（>, <, BETWEEN, IN）
EXPLAIN SELECT * FROM users WHERE age BETWEEN 20 AND 30;
-- type: range

-- 6. index：全索引扫描
EXPLAIN SELECT id FROM users;
-- type: index（扫描整个索引树）

-- 7. ALL：全表扫描（最慢）❌
EXPLAIN SELECT * FROM users WHERE name = 'Alice';
-- type: ALL（没有索引，扫描所有行）
```

---

### 3. key 和 key_len 分析

```sql
-- key：实际使用的索引
-- key_len：索引使用的字节数

-- 示例1：单列索引
CREATE INDEX idx_age ON users(age);  -- INT, 4 bytes
EXPLAIN SELECT * FROM users WHERE age = 25;
-- key: idx_age
-- key_len: 5（4 bytes + 1 NULL标志位）

-- 示例2：复合索引
CREATE INDEX idx_age_name ON users(age, name);
-- age: INT (5 bytes)
-- name: VARCHAR(50) (50*3 + 2 长度 + 1 NULL = 153 bytes, UTF8MB4)

EXPLAIN SELECT * FROM users WHERE age = 25;
-- key: idx_age_name
-- key_len: 5（只用了 age 列）

EXPLAIN SELECT * FROM users WHERE age = 25 AND name = 'Alice';
-- key: idx_age_name
-- key_len: 158（age 5 + name 153，使用了整个索引）
```

---

### 4. rows 和 filtered 评估

```sql
-- rows：预计扫描的行数
-- filtered：过滤后剩余的百分比（MySQL 5.7+）

EXPLAIN SELECT * FROM users WHERE age > 20 AND status = 'active';

-- 输出
-- rows: 500000
-- filtered: 10.00
-- 实际返回：500000 * 10% = 50000 行
```

---

### 5. Extra 信息解读

```sql
-- 常见 Extra 值

-- ✅ Using index（覆盖索引，无需回表）
EXPLAIN SELECT id, age FROM users WHERE age = 25;
-- Extra: Using index

-- ✅ Using index condition（索引下推）
EXPLAIN SELECT * FROM users WHERE age > 20 AND name LIKE 'A%';
-- Extra: Using index condition

-- ❌ Using filesort（需要排序，性能差）
EXPLAIN SELECT * FROM users ORDER BY name;
-- Extra: Using filesort（name 没有索引）

-- ❌ Using temporary（需要临时表）
EXPLAIN SELECT age, COUNT(*) FROM users GROUP BY age;
-- Extra: Using temporary; Using filesort

-- ✅ Using where（WHERE 过滤）
EXPLAIN SELECT * FROM users WHERE status = 'active';
-- Extra: Using where
```

## 🎯 MySQL 索引优化技巧

### 1. 最左前缀原则

```sql
-- 复合索引
CREATE INDEX idx_abc ON users(a, b, c);

-- ✅ 可以使用索引
SELECT * FROM users WHERE a = 1;                    -- 使用 (a)
SELECT * FROM users WHERE a = 1 AND b = 2;          -- 使用 (a, b)
SELECT * FROM users WHERE a = 1 AND b = 2 AND c = 3;-- 使用 (a, b, c)
SELECT * FROM users WHERE a = 1 AND c = 3;          -- 使用 (a)，c 过滤在 Server 层

-- ❌ 无法使用索引
SELECT * FROM users WHERE b = 2;                    -- 不从 a 开始
SELECT * FROM users WHERE c = 3;                    -- 不从 a 开始
SELECT * FROM users WHERE b = 2 AND c = 3;          -- 不从 a 开始
```

---

### 2. 索引失效场景

```sql
-- 1. 使用 OR（部分列无索引）
SELECT * FROM users WHERE age = 25 OR name = 'Alice';  -- age 有索引，name 无索引 → 全表扫描

-- 解决：都加索引，或改用 UNION
SELECT * FROM users WHERE age = 25
UNION
SELECT * FROM users WHERE name = 'Alice';

-- 2. 使用 != 或 <>
SELECT * FROM users WHERE age != 25;  -- 不使用索引

-- 解决：改为范围查询
SELECT * FROM users WHERE age < 25 OR age > 25;

-- 3. 在索引列使用函数
SELECT * FROM users WHERE YEAR(created_at) = 2024;  -- 不使用索引

-- 解决：改为范围查询
SELECT * FROM users WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';

-- 4. LIKE 以通配符开头
SELECT * FROM users WHERE name LIKE '%Alice';  -- 不使用索引

-- 解决：改为前缀匹配
SELECT * FROM users WHERE name LIKE 'Alice%';  -- 使用索引

-- 5. 隐式类型转换
SELECT * FROM users WHERE id = '123';  -- id 是 INT，'123' 是字符串 → 不使用索引

-- 解决：类型匹配
SELECT * FROM users WHERE id = 123;
```

### 3. 索引优化实战案例

**案例：优化慢查询**

```sql
-- 原始查询（5秒）
SELECT * FROM orders 
WHERE user_id = 123 
  AND status = 'pending' 
  AND created_at > '2024-01-01'
ORDER BY created_at DESC
LIMIT 10;

-- EXPLAIN 分析
-- type: ALL
-- rows: 10000000
-- Extra: Using where; Using filesort

-- 优化1：创建复合索引
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);

-- EXPLAIN 分析
-- type: ref
-- rows: 100
-- Extra: Using index condition

-- 性能：5秒 → 50毫秒 ✅
```

## 🎯 MongoDB 索引优化

### 1. MongoDB 索引类型

```javascript
// 1. 单字段索引
db.users.createIndex({ email: 1 });  // 1: 升序, -1: 降序

// 2. 复合索引
db.users.createIndex({ age: 1, name: 1 });

// 3. 多键索引（数组字段）
db.posts.createIndex({ tags: 1 });

// 4. 文本索引（全文搜索）
db.articles.createIndex({ content: 'text' });

// 5. 地理空间索引
db.places.createIndex({ location: '2dsphere' });

// 6. 哈希索引（分片键）
db.users.createIndex({ _id: 'hashed' });

// 7. TTL 索引（自动过期）
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 });
```

---

### 2. explain() 输出分析

```javascript
// 查询
db.users.find({ age: 25 }).explain('executionStats');

// 输出
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "FETCH",
      "inputStage": {
        "stage": "IXSCAN",  // 索引扫描
        "indexName": "age_1",
        "keysExamined": 100,
        "docsExamined": 100
      }
    }
  },
  "executionStats": {
    "executionTimeMillis": 5,
    "totalKeysExamined": 100,
    "totalDocsExamined": 100,
    "nReturned": 100
  }
}
```

**关键指标：**
- **stage**: 
  - `COLLSCAN`（全集合扫描）❌
  - `IXSCAN`（索引扫描）✅
  - `FETCH`（回表获取完整文档）
- **totalKeysExamined**: 扫描的索引键数量
- **totalDocsExamined**: 扫描的文档数量
- **nReturned**: 返回的文档数量
- **executionTimeMillis**: 执行时间（毫秒）

**优化目标：**
```
totalDocsExamined ≈ nReturned  ← 理想状态（无额外扫描）
```

### 3. MongoDB 索引优化案例

**案例：优化聚合查询**

```javascript
// 原始查询（慢）
db.orders.aggregate([
  { $match: { status: 'completed', createdAt: { $gte: new Date('2024-01-01') } } },
  { $group: { _id: '$userId', total: { $sum: '$amount' } } },
  { $sort: { total: -1 } },
  { $limit: 10 }
]);

// explain 分析
db.orders.explain('executionStats').aggregate([...]);
// stage: COLLSCAN ❌
// executionTimeMillis: 5000

// 优化：创建复合索引
db.orders.createIndex({ status: 1, createdAt: 1, userId: 1, amount: 1 });

// explain 分析
// stage: IXSCAN ✅
// executionTimeMillis: 50

// 性能：5秒 → 50毫秒 ✅
```

### 4. MongoDB ESR 规则

**ESR = Equality → Sort → Range**

```javascript
// 查询
db.users.find({
  gender: 'male',        // Equality
  age: { $gte: 20 }      // Range
}).sort({ createdAt: -1 });  // Sort

// ❌ 错误索引顺序
db.users.createIndex({ age: 1, gender: 1, createdAt: 1 });

// ✅ 正确索引顺序（ESR）
db.users.createIndex({ gender: 1, createdAt: -1, age: 1 });
// Equality → Sort → Range
```

## ⚠️ 三个高频错误

### 错误 1：盲目创建索引

```sql
-- ❌ 错误：为每个字段都创建索引
CREATE INDEX idx_col1 ON users(col1);
CREATE INDEX idx_col2 ON users(col2);
CREATE INDEX idx_col3 ON users(col3);
-- ... 50 个索引

-- 问题：
-- 1. 索引占用大量磁盘空间
-- 2. INSERT/UPDATE/DELETE 变慢（需要更新所有索引）
-- 3. MySQL 优化器选择索引困难

-- ✅ 正确：只为高频查询创建索引
-- 1. 分析慢查询日志
-- 2. 找到最频繁的查询
-- 3. 创建覆盖索引
```

---

### 错误 2：忽略索引基数

```sql
-- 索引基数（Cardinality）：不同值的数量

-- ❌ 错误：为低基数列创建索引
CREATE INDEX idx_gender ON users(gender);  -- 只有 male/female 两个值

SELECT * FROM users WHERE gender = 'male';
-- 即使有索引，仍需扫描 50% 的行，优化器可能选择全表扫描

-- ✅ 正确：为高基数列创建索引
CREATE INDEX idx_email ON users(email);  -- 每个值都唯一

-- 建议：基数 > 总行数的 30%
```

---

### 错误 3：复合索引顺序错误

```sql
-- 查询
SELECT * FROM orders 
WHERE user_id = 123 
  AND status = 'pending'
ORDER BY created_at DESC;

-- ❌ 错误：索引顺序不当
CREATE INDEX idx_status_user_created ON orders(status, user_id, created_at);

-- EXPLAIN
-- Extra: Using filesort（需要排序）

-- ✅ 正确：高选择性列在前 + 排序列在后
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at DESC);

-- EXPLAIN
-- Extra: Using index condition（无需排序）
```

## 📚 FAQ

### 1. 什么时候不需要索引？

- 表数据量很小（< 1000 行）
- 列的基数很低（如性别、布尔值）
- 频繁更新的列（索引维护成本高）
- 查询结果返回大部分行（> 20%）

### 2. 如何选择索引列的顺序？

**原则：**
1. **等值查询列在前**
2. **排序列其次**
3. **范围查询列最后**
4. **高选择性列在前**

### 3. 索引多了会影响写性能吗？

**是的**。每个索引在 INSERT/UPDATE/DELETE 时都需要更新。

**建议：**
- 开发环境：尽量少的索引
- 生产环境：基于实际查询创建索引
- 定期检查未使用的索引

```sql
-- 查找未使用的索引
SELECT * FROM sys.schema_unused_indexes;
```
