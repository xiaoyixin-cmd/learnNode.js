# 8.5 缓存与数据库一致性方案

> 延迟双删、Canal binlog 订阅，解决缓存一致性难题

---

## 📚 知识导读

### 缓存一致性问题

**场景：用户更新资料**

```javascript
// 1. 更新数据库
await User.updateOne({ _id: userId }, { name: 'Bob' });

// 2. 删除缓存
await redis.del(`user:${userId}`);

// 问题：并发读写导致不一致
```

**时序问题：**
```
线程A（写）          线程B（读）          数据库            缓存
   │                    │                 name: Alice    name: Alice
   │                    │
   ↓                    │
更新DB（name=Bob）      │                 name: Bob      name: Alice
   │                    ↓
   │               查询缓存（未命中）
   │                    ↓
   │               查询DB（name=Bob）
   │                    ↓
   │               写入缓存（name=Bob）    name: Bob      name: Bob
   ↓
删除缓存                │                 name: Bob      （已删除）
   │                    ↓
   │               查询缓存（未命中）
   │                    ↓
   │               查询DB（name=Bob）
   │                    ↓
   │               写入缓存（name=Bob）    name: Bob      name: Bob  ✅
```

**结果：数据最终一致 ✅**

**但另一种时序：**
```
线程A（写）          线程B（读）          数据库            缓存
   │                    │                 name: Alice    name: Alice
   ↓                    │
删除缓存                │                 name: Alice    （已删除）
   │                    ↓
   │               查询缓存（未命中）
   │                    ↓
   │               查询DB（name=Alice）   ← 此时DB还未更新
   ↓                    │
更新DB（name=Bob）      │                 name: Bob      （已删除）
   │                    ↓
   │               写入缓存（name=Alice）  name: Bob      name: Alice  ❌
```

**结果：数据不一致 ❌**

## 🎯 缓存更新策略

### 1. Cache Aside（旁路缓存）⭐ 推荐

**流程：**
```
读操作：
1. 查询缓存
2. 缓存命中 → 返回
3. 缓存未命中 → 查询数据库 → 写入缓存 → 返回

写操作：
1. 更新数据库
2. 删除缓存（不是更新缓存）
```

**实现：**
```javascript
// 读操作
async function getUser(userId) {
  // 1. 查询缓存
  const cached = await redis.get(`user:${userId}`);
  if (cached) {
    return JSON.parse(cached);
  }

  // 2. 查询数据库
  const user = await User.findById(userId);

  // 3. 写入缓存（5分钟过期）
  await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));

  return user;
}

// 写操作
async function updateUser(userId, data) {
  // 1. 更新数据库
  await User.updateOne({ _id: userId }, data);

  // 2. 删除缓存
  await redis.del(`user:${userId}`);
}
```

**为什么删除而不是更新缓存？**
1. **更新缓存复杂**：需要处理复杂查询、关联数据
2. **懒加载**：只有读取时才更新缓存，节省资源
3. **避免脏数据**：更新失败不会留下脏缓存

---

### 2. Read/Write Through

**流程：**
```
应用层 → 缓存层 → 数据库层

应用层只与缓存层交互，缓存层负责数据库同步
```

**适用场景：**
- 需要强一致性
- 缓存与数据库由统一服务管理

---

### 3. Write Behind（异步写入）

**流程：**
```
1. 写入缓存（立即返回）
2. 异步批量写入数据库

优势：写入性能高
劣势：可能丢失数据（缓存宕机）
```

**适用场景：**
- 高并发写入
- 允许短暂数据丢失（如浏览量统计）

## 🎯 延迟双删策略

### 1. 原理

**流程：**
```
1. 删除缓存（第一次）
2. 更新数据库
3. 延迟N毫秒
4. 删除缓存（第二次）← 清理并发写入的脏数据
```

**为什么有效？**
```
线程A（写）          线程B（读）          数据库            缓存
   │                    │                 name: Alice    name: Alice
   ↓                    │
删除缓存（第1次）       │                 name: Alice    （已删除）
   │                    ↓
   │               查询缓存（未命中）
   │                    ↓
   │               查询DB（name=Alice）   ← 此时DB还未更新
   ↓                    │
更新DB（name=Bob）      │                 name: Bob      （已删除）
   │                    ↓
   │               写入缓存（name=Alice）  name: Bob      name: Alice  ← 脏数据
   ↓
延迟100ms...
   ↓
删除缓存（第2次）       │                 name: Bob      （已删除）✅
```

---

### 2. 实现

```javascript
// 延迟双删工具函数
async function delayedDoubleDelete(key, delayMs = 100) {
  // 第一次删除
  await redis.del(key);

  // 异步延迟删除（不阻塞）
  setTimeout(async () => {
    await redis.del(key);
  }, delayMs);
}

// 使用
async function updateUser(userId, data) {
  const cacheKey = `user:${userId}`;

  // 延迟双删
  await delayedDoubleDelete(cacheKey, 100);

  // 更新数据库
  await User.updateOne({ _id: userId }, data);
}
```

---

### 3. 延迟时间如何确定？

**原则：延迟时间 > 读操作耗时**

```javascript
// 测试读操作耗时
async function measureReadTime() {
  const start = Date.now();

  // 模拟读操作
  const user = await User.findById('123');
  await redis.setEx('user:123', 300, JSON.stringify(user));

  const duration = Date.now() - start;
  console.log(`Read operation took ${duration}ms`);
}

// 多次测试取P99
// P99 = 50ms → 延迟时间设为 100ms（2倍安全系数）
```

### 4. 分布式场景优化

**问题：** 延迟删除依赖 `setTimeout`，服务重启会丢失

**解决方案：** 使用消息队列

```javascript
import Bull from 'bull';

const deleteCacheQueue = new Bull('delete-cache', process.env.REDIS_URL);

// 消费者：延迟删除缓存
deleteCacheQueue.process(async (job) => {
  const { key } = job.data;
  await redis.del(key);
});

// 生产者
async function updateUser(userId, data) {
  const cacheKey = `user:${userId}`;

  // 第一次删除
  await redis.del(cacheKey);

  // 更新数据库
  await User.updateOne({ _id: userId }, data);

  // 延迟队列删除（100ms后）
  await deleteCacheQueue.add(
    { key: cacheKey },
    { delay: 100, attempts: 3 }
  );
}
```

## 🎯 Canal binlog 订阅方案

### 1. Canal 原理

**Canal** 伪装成 MySQL Slave，订阅 binlog 实时同步数据

```
MySQL Master                Canal                 Redis/应用
     │                       │                       │
     │  1. 写入binlog        │                       │
     ├──────────────────────>│                       │
     │                       │  2. 解析binlog        │
     │                       │     ↓                 │
     │                       │  3. 推送变更          │
     │                       ├──────────────────────>│
     │                       │                       │  4. 更新缓存
     │                       │                       │     ✅
```

**优势：**
- 数据库与缓存最终一致
- 无需修改业务代码
- 支持多种数据源（Kafka、RocketMQ、Redis）

### 3. Node.js 订阅 Canal

```bash
npm install canal-client
```

```javascript
// canal-client.js
import CanalClient from 'canal-client';
import redis from './redis.js';

const client = new CanalClient({
  host: '127.0.0.1',
  port: 11111,
  destination: 'test',
  username: '',
  password: '',
});

// 连接 Canal
await client.connect();

// 订阅数据库
await client.subscribe({ filter: 'myapp\\..*' });  // 订阅 myapp 数据库所有表

// 消费 binlog
while (true) {
  const message = await client.get(100);  // 批量获取 100 条

  if (message.entries.length > 0) {
    for (const entry of message.entries) {
      const { tableName, eventType, rowDatas } = entry;

      console.log(`Event: ${eventType}, Table: ${tableName}`);

      // 处理不同事件
      if (eventType === 'INSERT' || eventType === 'UPDATE' || eventType === 'DELETE') {
        await handleDataChange(tableName, eventType, rowDatas);
      }
    }

    // 确认消费
    await client.ack(message.id);
  }

  await new Promise((resolve) => setTimeout(resolve, 1000));
}

// 处理数据变更
async function handleDataChange(tableName, eventType, rowDatas) {
  if (tableName === 'users') {
    for (const rowData of rowDatas) {
      const userId = rowData.afterColumns.find((col) => col.name === 'id').value;

      if (eventType === 'DELETE') {
        // 删除缓存
        await redis.del(`user:${userId}`);
      } else {
        // 更新缓存
        const user = {};
        for (const col of rowData.afterColumns) {
          user[col.name] = col.value;
        }
        await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
      }

      console.log(`Cache updated for user:${userId}`);
    }
  }
}
```

## 🎯 缓存问题与解决方案

### 1. 缓存穿透

**问题：** 查询不存在的数据，缓存和数据库都没有，每次都打到数据库

```javascript
// ❌ 问题
async function getUser(userId) {
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  const user = await User.findById(userId);  // 不存在返回 null
  if (!user) return null;  // 没有缓存 null

  await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
  return user;
}

// 攻击：大量查询不存在的 ID
for (let i = 1000000; i < 1100000; i++) {
  await getUser(i);  // 每次都打数据库 ❌
}

// ✅ 解决方案1：缓存空值
async function getUser(userId) {
  const cached = await redis.get(`user:${userId}`);
  if (cached) {
    return cached === 'null' ? null : JSON.parse(cached);
  }

  const user = await User.findById(userId);

  if (!user) {
    await redis.setEx(`user:${userId}`, 60, 'null');  // 缓存空值（短期）
    return null;
  }

  await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
  return user;
}

// ✅ 解决方案2：布隆过滤器
import { BloomFilter } from 'bloomfilter';

const bloom = new BloomFilter(1000000, 4);

// 初始化：将所有存在的 ID 加入布隆过滤器
const users = await User.find({}, '_id');
for (const user of users) {
  bloom.add(user._id.toString());
}

async function getUser(userId) {
  // 布隆过滤器判断（可能存在/一定不存在）
  if (!bloom.test(userId.toString())) {
    return null;  // 一定不存在，直接返回
  }

  // 正常查询流程
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  const user = await User.findById(userId);
  if (!user) return null;

  await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
  return user;
}
```

### 2. 缓存击穿

**问题：** 热点数据过期，大量请求同时打到数据库

```javascript
// ❌ 问题
async function getHotUser(userId) {
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  // 热点数据过期，1000个并发请求同时查询数据库 ❌
  const user = await User.findById(userId);
  await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
  return user;
}

// ✅ 解决方案1：互斥锁
import { Mutex } from 'async-mutex';

const mutexMap = new Map();

async function getHotUser(userId) {
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  // 获取互斥锁
  if (!mutexMap.has(userId)) {
    mutexMap.set(userId, new Mutex());
  }
  const mutex = mutexMap.get(userId);

  const release = await mutex.acquire();
  try {
    // 再次检查缓存（可能被其他线程更新）
    const cached = await redis.get(`user:${userId}`);
    if (cached) return JSON.parse(cached);

    // 查询数据库（只有一个线程执行）
    const user = await User.findById(userId);
    await redis.setEx(`user:${userId}`, 300, JSON.stringify(user));
    return user;
  } finally {
    release();
    mutexMap.delete(userId);
  }
}

// ✅ 解决方案2：永不过期（逻辑过期）
async function getHotUser(userId) {
  const cached = await redis.get(`user:${userId}`);

  if (cached) {
    const data = JSON.parse(cached);

    // 检查逻辑过期时间
    if (data.expireAt < Date.now()) {
      // 异步更新缓存（不阻塞）
      updateCacheAsync(userId);
    }

    return data.user;
  }

  // 首次查询
  const user = await User.findById(userId);
  await redis.set(`user:${userId}`, JSON.stringify({
    user,
    expireAt: Date.now() + 300000,  // 逻辑过期时间
  }));

  return user;
}

async function updateCacheAsync(userId) {
  // 异步更新，不影响当前请求
  setImmediate(async () => {
    const user = await User.findById(userId);
    await redis.set(`user:${userId}`, JSON.stringify({
      user,
      expireAt: Date.now() + 300000,
    }));
  });
}
```

### 3. 缓存雪崩

**问题：** 大量缓存同时过期，数据库压力激增

```javascript
// ❌ 问题：所有缓存同时过期（TTL 都是 300 秒）
await redis.setEx('user:1', 300, data1);
await redis.setEx('user:2', 300, data2);
await redis.setEx('user:3', 300, data3);

// ✅ 解决方案1：过期时间随机化
function randomExpire(base) {
  return base + Math.floor(Math.random() * 60);  // 基础时间 ± 60秒
}

await redis.setEx('user:1', randomExpire(300), data1);  // 300-360秒
await redis.setEx('user:2', randomExpire(300), data2);
await redis.setEx('user:3', randomExpire(300), data3);

// ✅ 解决方案2：Redis 集群 + 主从复制
// ✅ 解决方案3：缓存预热
async function warmupCache() {
  const hotUsers = await User.find({ isHot: true });
  for (const user of hotUsers) {
    await redis.setEx(`user:${user._id}`, randomExpire(3600), JSON.stringify(user));
  }
}
```

## ⚠️ 三个高频错误

### 错误 1：先删缓存再更新数据库

```javascript
// ❌ 错误
await redis.del(`user:${userId}`);  // 先删缓存
await User.updateOne({ _id: userId }, data);  // 再更新数据库

// 问题：并发读取会缓存旧数据

// ✅ 正确
await User.updateOne({ _id: userId }, data);  // 先更新数据库
await redis.del(`user:${userId}`);  // 再删缓存
```

---

### 错误 2：更新缓存而不是删除

```javascript
// ❌ 错误
await User.updateOne({ _id: userId }, { name: 'Bob' });
await redis.setEx(`user:${userId}`, 300, JSON.stringify({ name: 'Bob' }));

// 问题：
// - 复杂查询难以构建缓存
// - 更新失败留下脏数据

// ✅ 正确
await User.updateOne({ _id: userId }, { name: 'Bob' });
await redis.del(`user:${userId}`);  // 删除缓存，下次读取时更新
```

---

### 错误 3：忽略缓存预热

```javascript
// ❌ 错误：应用启动后缓存为空，所有请求打数据库

// ✅ 正确：应用启动时预热缓存
async function warmupCache() {
  console.log('Warming up cache...');

  // 预热热点数据
  const hotUsers = await User.find({ isHot: true }).limit(1000);
  for (const user of hotUsers) {
    await redis.setEx(`user:${user._id}`, 3600, JSON.stringify(user));
  }

  console.log(`Warmed up ${hotUsers.length} users`);
}

// 应用启动时执行
await warmupCache();
```

## 📚 FAQ

### 1. 延迟双删 vs Canal binlog 如何选择？

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **延迟双删** | 简单、无需额外组件 | 延迟时间难以确定 | 中小型项目 |
| **Canal binlog** | 强一致性、无侵入 | 部署复杂、依赖MySQL | 大型项目 |

### 2. 如何保证 Canal 消费不丢失？

**方案：**
1. 使用 Kafka 存储 binlog（持久化）
2. Canal 消费失败自动重试
3. 应用层幂等处理

### 3. 缓存一致性能达到强一致吗？

**不能**。分布式系统中，缓存与数据库是两个独立系统，无法保证强一致性（CAP 定理）。

**只能做到最终一致性。**
