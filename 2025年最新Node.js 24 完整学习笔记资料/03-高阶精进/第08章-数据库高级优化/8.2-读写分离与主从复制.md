# 8.2 读写分离与主从复制配置

> 搭建高可用数据库架构：MySQL 主从复制 + MongoDB 副本集 + Node.js 读写分离

---

## 📚 知识导读

### 为什么需要读写分离？

**单机数据库的瓶颈：**
```
场景：电商网站
- 读请求：商品浏览、搜索（95%）
- 写请求：下单、支付（5%）

单机数据库：
- 读写都在同一台服务器
- 高峰期CPU 100%、磁盘 I/O 饱和
- 响应时间变慢 ❌
```

**读写分离架构：**
```
           ┌─────────┐
           │ 应用层  │
           └────┬────┘
                │
        ┌───────┴───────┐
        │               │
    写请求（5%）     读请求（95%）
        │               │
        ↓               ↓
   ┌────────┐      ┌────────┐
   │ Master │ →→→  │ Slave1 │  (复制)
   │  主库  │      │  从库1 │
   └────────┘      └────────┘
        │               ↓
        └──────→→→ ┌────────┐
                   │ Slave2 │
                   │  从库2 │
                   └────────┘

优势：
✅ 读写分离，分担负载
✅ 主库专注写操作
✅ 从库分担读操作（可水平扩展）
✅ 高可用：主库故障，从库提升为主库
```

## 🎯 MySQL 主从复制原理

### 1. 复制流程

```
主库（Master）                       从库（Slave）
    │                                    │
    │ 1. 执行 INSERT/UPDATE/DELETE       │
    │    ↓                                │
    │ 2. 写入 binlog（二进制日志）        │
    │    ↓                                │
    │ 3. binlog dump 线程               │
    │    ↓                                │
    │ ═══════════════════════════════>   │
    │                                    ↓
    │                           4. I/O 线程读取 binlog
    │                                    ↓
    │                           5. 写入 relay log（中继日志）
    │                                    ↓
    │                           6. SQL 线程执行 relay log
    │                                    ↓
    │                           7. 数据同步完成 ✅
```

---

### 2. 复制模式

**异步复制（Asynchronous Replication）：**
```
主库：
1. 执行事务
2. 写入 binlog
3. 提交事务 ✅
4. 返回客户端（不等待从库）

从库：
1. 异步读取 binlog
2. 执行 SQL

问题：主库宕机时，可能有部分数据未同步到从库
```

**半同步复制（Semi-Synchronous Replication）：**
```
主库：
1. 执行事务
2. 写入 binlog
3. 等待至少一个从库确认接收 ← 半同步
4. 提交事务 ✅
5. 返回客户端

优势：保证至少一个从库有数据
劣势：性能略有下降
```

**组复制（Group Replication / MySQL 8.0+）：**
```
多主模式：所有节点都可写
单主模式：一个主节点 + 多个从节点

特点：
- 基于 Paxos 算法
- 强一致性
- 自动故障转移
```

## 🎯 MySQL 主从复制搭建

### 1. 主库配置

```bash
# 主库配置文件：my.cnf
[mysqld]
server-id = 1                  # 唯一 ID
log-bin = mysql-bin            # 开启 binlog
binlog-format = ROW            # ROW 模式（推荐）
sync-binlog = 1                # 每次事务都同步 binlog
expire-logs-days = 7           # binlog 保留 7 天
max-binlog-size = 100M         # binlog 最大 100MB

# 可选：半同步复制
plugin-load = "rpl_semi_sync_master=semisync_master.so"
rpl-semi-sync-master-enabled = 1
rpl-semi-sync-master-timeout = 1000  # 1秒超时
```

**创建复制用户：**
```sql
-- 主库执行
CREATE USER 'replicator'@'%' IDENTIFIED BY 'StrongPassword123!';
GRANT REPLICATION SLAVE ON *.* TO 'replicator'@'%';
FLUSH PRIVILEGES;

-- 查看主库状态
SHOW MASTER STATUS;
-- +------------------+----------+--------------+------------------+
-- | File             | Position | Binlog_Do_DB | Binlog_Ignore_DB |
-- +------------------+----------+--------------+------------------+
-- | mysql-bin.000001 |      154 |              |                  |
-- +------------------+----------+--------------+------------------+
```

---

### 2. 从库配置

```bash
# 从库配置文件：my.cnf
[mysqld]
server-id = 2                  # 唯一 ID（不能与主库相同）
relay-log = mysql-relay-bin    # 中继日志
read-only = 1                  # 只读模式（防止误写）
super-read-only = 1            # 超级只读（连 root 也不能写）

# 可选：半同步复制
plugin-load = "rpl_semi_sync_slave=semisync_slave.so"
rpl-semi-sync-slave-enabled = 1
```

**配置主从复制：**
```sql
-- 从库执行
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',        -- 主库 IP
  MASTER_PORT=3306,
  MASTER_USER='replicator',
  MASTER_PASSWORD='StrongPassword123!',
  MASTER_LOG_FILE='mysql-bin.000001', -- 主库 binlog 文件
  MASTER_LOG_POS=154;                 -- 主库 binlog 位置

-- 启动复制
START SLAVE;

-- 查看从库状态
SHOW SLAVE STATUS\G

-- 关键字段
-- Slave_IO_Running: Yes   ← I/O 线程运行中
-- Slave_SQL_Running: Yes  ← SQL 线程运行中
-- Seconds_Behind_Master: 0 ← 主从延迟（秒）
```

---

### 3. GTID 模式（推荐）

**GTID（Global Transaction ID）= 全局事务 ID**

```bash
# 主库和从库都配置
[mysqld]
gtid-mode = ON
enforce-gtid-consistency = ON
log-slave-updates = ON
```

**配置主从（GTID 模式）：**
```sql
-- 从库执行
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_PORT=3306,
  MASTER_USER='replicator',
  MASTER_PASSWORD='StrongPassword123!',
  MASTER_AUTO_POSITION=1;  -- 使用 GTID 自动定位

START SLAVE;
```

**GTID 优势：**
- 无需手动指定 binlog 文件和位置
- 故障转移更简单
- 支持多级复制

## 🎯 MongoDB 副本集

### 1. 副本集架构

```
   Primary（主节点）     Secondary1（从节点）
        │                      │
        │← oplog 复制 →←      │
        ↓                      ↓
   Secondary2（从节点）   Arbiter（仲裁节点）
```

**节点类型：**
- **Primary**：处理所有写操作
- **Secondary**：复制 Primary 数据，可处理读操作
- **Arbiter**：只参与选举，不存储数据

---

### 3. Read Preference（读偏好）

```javascript
import { MongoClient } from 'mongodb';

const client = new MongoClient('mongodb://localhost:27017,localhost:27018,localhost:27019/?replicaSet=rs0', {
  // 读偏好设置
  readPreference: 'secondaryPreferred'
});

await client.connect();
```

**读偏好模式：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **primary** | 只读主节点 | 强一致性 |
| **primaryPreferred** | 优先读主节点，主节点不可用时读从节点 | 一般场景 |
| **secondary** | 只读从节点 | 读写分离 |
| **secondaryPreferred** | 优先读从节点，从节点不可用时读主节点 | 推荐 ⭐ |
| **nearest** | 读最近的节点（延迟最低） | 地理分布 |

## 💻 Node.js 读写分离实现

### 1. MySQL 读写分离（mysql2）

```javascript
import mysql from 'mysql2/promise';

// 创建主库连接池（写）
const masterPool = mysql.createPool({
  host: '192.168.1.100',
  port: 3306,
  user: 'root',
  password: 'password',
  database: 'myapp',
  waitForConnections: true,
  connectionLimit: 10,
  queueLimit: 0,
});

// 创建从库连接池（读）
const slavePool = mysql.createPool({
  host: '192.168.1.101',
  port: 3306,
  user: 'root',
  password: 'password',
  database: 'myapp',
  waitForConnections: true,
  connectionLimit: 50,  // 从库连接数更多
  queueLimit: 0,
});

// 封装数据库操作
class Database {
  // 写操作（使用主库）
  static async write(sql, params) {
    const [result] = await masterPool.execute(sql, params);
    return result;
  }

  // 读操作（使用从库）
  static async read(sql, params) {
    const [rows] = await slavePool.execute(sql, params);
    return rows;
  }

  // 事务（使用主库）
  static async transaction(callback) {
    const connection = await masterPool.getConnection();
    await connection.beginTransaction();

    try {
      const result = await callback(connection);
      await connection.commit();
      return result;
    } catch (error) {
      await connection.rollback();
      throw error;
    } finally {
      connection.release();
    }
  }
}

// 使用示例
export default Database;
```

**使用：**
```javascript
import Database from './database.js';

// 写操作（主库）
await Database.write(
  'INSERT INTO users (name, email) VALUES (?, ?)',
  ['Alice', 'alice@example.com']
);

// 读操作（从库）
const users = await Database.read(
  'SELECT * FROM users WHERE age > ?',
  [18]
);

// 事务（主库）
await Database.transaction(async (conn) => {
  await conn.execute('UPDATE accounts SET balance = balance - 100 WHERE id = ?', [1]);
  await conn.execute('UPDATE accounts SET balance = balance + 100 WHERE id = ?', [2]);
});
```

### 2. MySQL 读写分离中间件

```javascript
import express from 'express';

const app = express();

// 读写分离中间件
function readWriteSplitter(req, res, next) {
  const method = req.method;

  if (method === 'GET' || method === 'HEAD') {
    // 读请求 → 从库
    req.db = slavePool;
  } else {
    // 写请求 → 主库
    req.db = masterPool;
  }

  next();
}

app.use(readWriteSplitter);

// 路由
app.get('/api/users', async (req, res) => {
  const [users] = await req.db.execute('SELECT * FROM users');
  res.json(users);
});

app.post('/api/users', async (req, res) => {
  const { name, email } = req.body;
  const [result] = await req.db.execute(
    'INSERT INTO users (name, email) VALUES (?, ?)',
    [name, email]
  );
  res.json({ id: result.insertId });
});
```

```javascript
import mongoose from 'mongoose';

// 连接副本集
await mongoose.connect('mongodb://localhost:27017,localhost:27018,localhost:27019/myapp?replicaSet=rs0', {
  readPreference: 'secondaryPreferred',  // 读偏好

// 定义 Schema
const UserSchema = new mongoose.Schema({
  name: String,
  email: String,
});

// 为特定查询指定读偏好
UserSchema.statics.findByIdFromPrimary = function(id) {
  return this.findById(id).read('primary');  // 强制读主节点
};

UserSchema.statics.findAllFromSecondary = function() {
  return this.find().read('secondary');  // 强制读从节点
};

const User = mongoose.model('User', UserSchema);

// 使用
const users = await User.findAllFromSecondary();  // 从从节点读取
```

### 2. 强制读主库

```javascript
// 场景：用户刚注册，立即查询自己的信息

  const profile = await User.findById(user.id).read('primary');  // 读主库 ✅
```

### 3. 会话一致性（Sticky Session）

```javascript
// 使用 Redis 存储用户最后写操作时间
import { createClient } from 'redis';

const redis = createClient();
await redis.connect();

// 中间件：会话一致性
async function sessionConsistency(req, res, next) {
  const userId = req.user?.id;

  if (!userId) {
    return next();
  }

  // 获取用户最后写操作时间
  const lastWrite = await redis.get(`last_write:${userId}`);

  if (lastWrite) {
    const elapsed = Date.now() - parseInt(lastWrite, 10);

    // 如果距离最后写操作 < 5秒，强制读主库
    if (elapsed < 5000) {
      req.forceReadMaster = true;
    }
  }

  next();
}

app.use(sessionConsistency);

// 路由
app.get('/api/profile', async (req, res) => {
  const readPref = req.forceReadMaster ? 'primary' : 'secondaryPreferred';
  
  const user = await User.findById(req.user.id).read(readPref);
  res.json(user);
});

app.put('/api/profile', async (req, res) => {
  const user = await User.findByIdAndUpdate(req.user.id, req.body);
  
  // 记录写操作时间
  await redis.setEx(`last_write:${req.user.id}`, 10, Date.now().toString());
  
  res.json(user);
});
```

### 错误 3：主从复制未监控

```javascript
// ✅ 正确：定期检查复制状态
import mysql from 'mysql2/promise';

async function checkReplicationHealth() {
  const [status] = await slavePool.execute('SHOW SLAVE STATUS');

  if (status.length === 0) {
    console.error('Replication not configured!');
    return;
  }

  const { Slave_IO_Running, Slave_SQL_Running, Seconds_Behind_Master } = status[0];

    // 发送告警
  }

    console.warn(`Replication lag: ${Seconds_Behind_Master} seconds`);
    // 发送告警
  }
}

// 每分钟检查一次
setInterval(checkReplicationHealth, 60000);
```

## 📚 FAQ

### 1. 读写分离一定能提升性能吗？

**不一定**。如果：
- 写操作占比高（> 20%）
- 主从延迟大（> 1 秒）
- 从库配置低于主库

可能效果不明显。

### 2. 主库故障如何切换？

**手动切换：**
```sql
-- 1. 提升从库为主库
STOP SLAVE;
RESET SLAVE ALL;

-- 2. 去除只读
SET GLOBAL read_only = 0;
SET GLOBAL super_read_only = 0;

-- 3. 应用切换连接到新主库
```

**自动切换：** 使用 MHA（Master High Availability）或 Orchestrator

### 3. MongoDB 如何处理写冲突？

**MongoDB 副本集只有一个 Primary，不存在写冲突。**

如果 Primary 宕机：
1. 副本集自动选举新 Primary
2. 选举时间：10-30 秒
3. 写操作失败，需要应用重试
