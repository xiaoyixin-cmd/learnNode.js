## 1️⃣ 设计哲学一：事件驱动（Event-Driven）

### 什么是事件驱动？

**核心概念：**
程序的执行流程由事件来决定，而不是按照顺序执行。通过事件循环（Event Loop）监听和响应事件。

### 传统同步模型 vs 事件驱动模型

#### 传统同步模型（阻塞式）
```javascript
// ❌ 传统同步模型（如 PHP、Python 同步代码）
console.log('开始读取文件1');
const data1 = readFileSync('file1.txt'); // 阻塞 100ms
console.log('文件1读取完成');

console.log('开始读取文件2');
const data2 = readFileSync('file2.txt'); // 阻塞 100ms
console.log('文件2读取完成');

console.log('总耗时：200ms');

// 输出顺序：
// 开始读取文件1
// (等待 100ms)
// 文件1读取完成
// 开始读取文件2
// (等待 100ms)
// 文件2读取完成
// 总耗时：200ms
```

#### 事件驱动模型（非阻塞式）
```javascript
// ✅ Node.js 事件驱动模型
console.log('开始读取文件1');
readFile('file1.txt', (err, data1) => {
  console.log('文件1读取完成');
});

console.log('开始读取文件2');
readFile('file2.txt', (err, data2) => {
  console.log('文件2读取完成');
});

console.log('继续执行其他任务');
console.log('总耗时：~100ms（并发执行）');

// 输出顺序：
// 开始读取文件1
// 开始读取文件2
// 继续执行其他任务
// 总耗时：~100ms（并发执行）
// (约 100ms 后)
// 文件1读取完成
// 文件2读取完成
```

### 实际案例：处理 1000 个请求

```javascript
// 传统阻塞模型（如 Apache + PHP）
for (let i = 0; i < 1000; i++) {
  // 每个请求创建一个线程
  createThread(() => {
    const data = queryDatabase();  // 阻塞 50ms
    sendResponse(data);
  });
}
// 结果：1000 个线程，内存占用 ~2GB，响应时间慢

// Node.js 事件驱动模型
for (let i = 0; i < 1000; i++) {
  // 所有请求共用一个线程
  queryDatabase((err, data) => {  // 非阻塞
    sendResponse(data);
  });
}
// 结果：1 个线程，内存占用 ~50MB，响应时间快
```

### 事件循环（Event Loop）详解

#### 事件循环可视化

```
   ┌───────────────────────────┐
┌─>│        timers 阶段        │  执行 setTimeout、setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │   pending callbacks 阶段  │  执行延迟到下一轮的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │    idle, prepare 阶段    │  仅内部使用
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │        poll 阶段          │<─────┤  connections, │
│  │  (轮询新的 I/O 事件)      │      │   data, etc.  │
│  └─────────────┬─────────────┘      └───────────────┘
│  ┌─────────────┴─────────────┐
│  │        check 阶段         │  执行 setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤   close callbacks 阶段    │  执行 close 事件回调（如 socket.on('close')）
   └───────────────────────────┘
```

#### 事件循环代码示例

```javascript
const fs = require('fs');

console.log('1. 同步代码开始');

// Timers 阶段
setTimeout(() => {
  console.log('2. setTimeout 回调');
}, 0);

// Check 阶段
setImmediate(() => {
  console.log('3. setImmediate 回调');
});

// Poll 阶段（I/O 操作）
fs.readFile('file.txt', () => {
  console.log('4. fs.readFile 回调');
  
  setTimeout(() => {
    console.log('5. setTimeout 嵌套');
  }, 0);
  
  setImmediate(() => {
    console.log('6. setImmediate 嵌套');
  });
});

// Microtask 队列（Promise）
Promise.resolve().then(() => {
  console.log('7. Promise.then');
});

// Next Tick 队列（优先级最高）
process.nextTick(() => {
  console.log('8. process.nextTick');
});

console.log('9. 同步代码结束');

// 输出顺序：
// 1. 同步代码开始
// 9. 同步代码结束
// 8. process.nextTick (微任务优先)
// 7. Promise.then
// 2. setTimeout 回调
// 3. setImmediate 回调
// 4. fs.readFile 回调
// 6. setImmediate 嵌套 (在 I/O 回调中，setImmediate 优先于 setTimeout)
// 5. setTimeout 嵌套
```

## 2️⃣ 设计哲学二：非阻塞 I/O（Non-Blocking I/O）

### 什么是非阻塞 I/O？

**核心概念：**
I/O 操作（文件读写、网络请求）不会阻塞主线程，而是通过回调函数异步处理结果。

### 阻塞 vs 非阻塞对比

#### 阻塞 I/O
```javascript
// ❌ 阻塞式（线程被挂起，等待 I/O 完成）
const fs = require('fs');

console.log('开始读取文件');
const data = fs.readFileSync('large-file.txt', 'utf-8'); // 阻塞 2 秒
console.log('文件读取完成');
console.log('继续执行其他任务'); // 2 秒后才能执行

// 问题：
// 1. 主线程被阻塞 2 秒
// 2. 无法处理其他请求
// 3. CPU 空闲等待 I/O
```

#### 非阻塞 I/O
```javascript
// ✅ 非阻塞式（立即返回，通过回调处理结果）
const fs = require('fs');

console.log('开始读取文件');
fs.readFile('large-file.txt', 'utf-8', (err, data) => {
  console.log('文件读取完成'); // 2 秒后执行
  if (err) throw err;
  console.log('文件内容长度：', data.length);
});
console.log('继续执行其他任务'); // 立即执行

// 输出：
// 开始读取文件
// 继续执行其他任务
// (2 秒后)
// 文件读取完成
// 文件内容长度：xxx

// 优势：
// 1. 主线程不被阻塞
// 2. 可以同时处理多个请求
// 3. CPU 可以执行其他任务
```

### Node.js 如何实现非阻塞？

#### 完整架构图

```
┌──────────────────────────────────────────┐
│     JavaScript 代码（用户代码）           │
│                                          │
│  fs.readFile('file.txt', callback);     │
└──────────────┬───────────────────────────┘
               │
               ↓ 通过 V8 和 N-API 调用
┌──────────────┴───────────────────────────┐
│          Node.js C++ Bindings           │
│         (Node.js 核心层)                 │
│                                          │
│  • 将 JavaScript 调用转为 C++ 调用       │
│  • 管理回调函数队列                      │
└──────────────┬───────────────────────────┘
               │
               ↓ 交给 libuv 处理
┌──────────────┴───────────────────────────┐
│              libuv 事件循环              │
│         (跨平台异步 I/O 库)              │
│                                          │
│  • 管理事件循环                          │
│  • 将 I/O 任务分配给线程池               │
└──────────────┬───────────────────────────┘
               │
               ↓ I/O 操作
┌──────────────┴───────────────────────────┐
│           libuv 线程池                   │
│        (默认 4 个工作线程)               │
│                                          │
│  [线程1] [线程2] [线程3] [线程4]         │
│    ↓       ↓       ↓       ↓            │
│  • 文件 I/O                              │
│  • DNS 查询                              │
│  • 压缩/解压                             │
│  • 加密/解密                             │
└──────────────┬───────────────────────────┘
               │
               ↓ I/O 完成，返回结果
┌──────────────┴───────────────────────────┐
│         事件循环（主线程）                │
│                                          │
│  检测到 I/O 完成 → 执行回调函数          │
└──────────────────────────────────────────┘
```

### 实战示例：并发读取多个文件

```javascript
const fs = require('fs').promises;

// 串行读取（阻塞式）
async function serialRead() {
  console.time('serial');
  const file1 = await fs.readFile('file1.txt');
  const file2 = await fs.readFile('file2.txt');
  const file3 = await fs.readFile('file3.txt');
  console.timeEnd('serial');
  // 总耗时：~300ms (100ms + 100ms + 100ms)
}

// 并发读取（非阻塞式）
async function parallelRead() {
  console.time('parallel');
  const [file1, file2, file3] = await Promise.all([
    fs.readFile('file1.txt'),
    fs.readFile('file2.txt'),
    fs.readFile('file3.txt')
  ]);
  console.timeEnd('parallel');
  // 总耗时：~100ms (并发执行)
}

// 性能提升 3 倍！
```

## 3️⃣ 设计哲学三：单线程 + libuv

### 为什么选择单线程？

#### 单线程的优势

```javascript
// ✅ 单线程优势

1. 避免线程同步问题
   // 多线程需要加锁
   lock.acquire();
   sharedVariable++;
   lock.release();
   
   // Node.js 单线程无需加锁
   sharedVariable++;

2. 避免上下文切换开销
   // 多线程：频繁切换上下文（耗时）
   // Node.js：单线程无切换

3. 编程模型简单
   // 无需担心线程安全
   // 无需管理锁和信号量
   // 代码更容易理解和维护
```

#### 单线程的劣势

```javascript
// ❌ 单线程劣势

1. CPU 密集型任务会阻塞
   // 错误示例：计算密集型任务
   function fibonacci(n) {
     if (n <= 1) return n;
     return fibonacci(n - 1) + fibonacci(n - 2);
   }
   
   // 计算 fibonacci(45) 会阻塞主线程 ~30秒
   const result = fibonacci(45); // ❌ 阻塞！

2. 无法充分利用多核 CPU
   // 单进程只能使用一个 CPU 核心
   // 需要手动创建多个进程（Cluster 模式）
```

### libuv 的作用

#### libuv 是什么？

```
libuv：跨平台异步 I/O 库
├─ 由 C 语言编写
├─ 为 Node.js 提供底层支持
├─ 抽象不同操作系统的 I/O 实现
│  ├─ Linux：epoll
│  ├─ macOS：kqueue
│  └─ Windows：IOCP
└─ 管理线程池和事件循环
```

#### libuv 核心功能

```c
// libuv 事件循环核心函数（来自 Node.js v22.17.0 官方文档）

int uv_run(uv_loop_t* loop, uv_run_mode mode)
  描述：运行事件循环，根据指定模式执行
  
  模式：
    UV_RUN_DEFAULT：运行直到没有活动句柄/请求
    UV_RUN_ONCE：轮询一次 I/O，如果有待处理回调则阻塞
    UV_RUN_NOWAIT：轮询一次 I/O，非阻塞
  
  返回值：
    0：事件循环正常结束
    非0：事件循环被 uv_stop() 停止

// 其他核心函数
uv_loop_init(loop)      // 初始化事件循环
uv_loop_close(loop)     // 关闭事件循环
uv_stop(loop)           // 停止事件循环
uv_loop_alive(loop)     // 检查事件循环是否活跃
```

#### libuv 线程池

```javascript
// libuv 线程池大小（默认 4 个）
process.env.UV_THREADPOOL_SIZE = 8; // 可以调整为 8 个

const fs = require('fs');

// 这些操作会使用线程池
fs.readFile('file.txt', callback);        // 文件 I/O
dns.lookup('nodejs.org', callback);       // DNS 查询
crypto.pbkdf2('password', 'salt', ...)    // 加密操作
zlib.gzip(buffer, callback);              // 压缩操作

// 这些操作不使用线程池（使用操作系统的异步 I/O）
http.get('http://example.com', callback); // HTTP 请求
net.connect({ port: 3000 }, callback);    // TCP 连接
```

## 🔄 完整工作流程示例

### 一个 HTTP 请求的完整生命周期

```javascript
const http = require('http');
const fs = require('fs');

const server = http.createServer((req, res) => {
  console.log('1. 收到 HTTP 请求');
  
  // 2. 异步读取文件
  fs.readFile('data.txt', 'utf-8', (err, data) => {
    console.log('3. 文件读取完成');
    
    // 4. 发送响应
    res.writeHead(200, { 'Content-Type': 'text/plain' });
    res.end(data);
    console.log('5. 响应发送完成');
  });
  
  console.log('2. 继续处理其他请求（非阻塞）');
});

server.listen(3000);

// 执行流程：
// 1. 收到 HTTP 请求
// 2. 继续处理其他请求（非阻塞） ← 立即执行
// (等待 I/O 完成...)
// 3. 文件读取完成
// 5. 响应发送完成
```

### 事件循环各阶段的实际应用

```javascript
const fs = require('fs');

// ===== 1. Timers 阶段 =====
setTimeout(() => {
  console.log('setTimeout: 定时任务执行');
}, 1000);

setInterval(() => {
  console.log('setInterval: 周期任务执行');
}, 2000);

// ===== 2. I/O Callbacks 阶段 =====
fs.readFile('file.txt', (err, data) => {
  console.log('I/O Callback: 文件读取完成');
});

// ===== 3. Poll 阶段 =====
// 等待新的 I/O 事件

// ===== 4. Check 阶段 =====
setImmediate(() => {
  console.log('setImmediate: I/O 后立即执行');
});

// ===== 5. Close Callbacks 阶段 =====
const server = require('http').createServer();
server.on('close', () => {
  console.log('Close Callback: 服务器关闭');
});
server.listen(3000);
setTimeout(() => server.close(), 5000);

// ===== 6. Microtask 队列（优先级最高）=====
process.nextTick(() => {
  console.log('nextTick: 最高优先级');
});

Promise.resolve().then(() => {
  console.log('Promise: 微任务');
});
```

## 💡 常见问题（FAQ）

**Q1：Node.js 单线程如何处理高并发？**  
A：JavaScript 执行是单线程，但 I/O 操作由 libuv 的线程池异步处理。主线程只负责调度回调，不会被阻塞。

**Q2：为什么 Node.js 不适合 CPU 密集型任务？**  
A：CPU 密集型任务（如大量计算）会阻塞主线程，导致无法处理其他请求。应该使用 Worker Threads 或将任务分发到其他进程。

**Q3：如何充分利用多核 CPU？**  
A：使用 Cluster 模块创建多个进程，每个进程运行在不同的 CPU 核心上。

**Q4：事件循环的优先级顺序是什么？**  
A：process.nextTick > Promise > setImmediate > setTimeout/setInterval
