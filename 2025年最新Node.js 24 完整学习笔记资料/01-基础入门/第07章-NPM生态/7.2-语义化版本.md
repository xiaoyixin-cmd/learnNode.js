# 7.2 语义化版本(SemVer)规则

> 理解版本号的含义,正确管理依赖版本

---

## 🎯 什么是语义化版本？

### 基础概念

```
语义化版本(Semantic Versioning) = 版本号有明确含义

格式: 主版本号.次版本号.修订号
      Major   .Minor   .Patch

例如: 2.3.5
     │ │ │
     │ │ └─ 修订号(Patch):  修复bug
     │ └─── 次版本号(Minor): 新增功能(向后兼容)
     └───── 主版本号(Major): 不兼容的变更

类比生活：
汽车型号: 2024款 宝马 3系 325i
         年份   品牌 系列 型号
         Major Minor Patch
```

### 版本号递增规则

```
初始版本: 1.0.0

场景1: 修复了一个bug
1.0.0 → 1.0.1 (Patch +1)

场景2: 新增了一个功能,且向后兼容
1.0.1 → 1.1.0 (Minor +1, Patch归零)

场景3: 重大更新,不向后兼容
1.1.0 → 2.0.0 (Major +1, Minor和Patch归零)

💡 理解"向后兼容":
- 1.0.0 的代码在 1.1.0 中能正常运行 = 向后兼容
- 1.0.0 的代码在 2.0.0 中可能报错 = 不兼容
```

## 📋 版本号详解

### 1. 主版本号(Major)

```
Major: 主版本号,不兼容的API变更

什么时候增加：
✅ 删除了公开的API
✅ 修改了API的行为(不兼容)
✅ 重构了核心架构

示例：
// v1.x.x
function getUser(id) {
  return { id, name: 'Alice' };
}

// v2.0.0 (不兼容变更)
async function getUser(id) {  // 改成了async,不兼容
  return { userId: id, username: 'Alice' };  // 字段名变了
}

// 用户代码可能报错：
const user = getUser(123);  // v1返回对象, v2返回Promise!
console.log(user.name);     // v1有name字段, v2没有!

💡 建议：
主版本号更新要慎重,提供迁移指南
```

### 2. 次版本号(Minor)

```
Minor: 次版本号,新增功能但向后兼容

什么时候增加：
✅ 新增了公开API
✅ 标记某些功能为废弃(但仍可用)
✅ 内部增强(性能优化、新特性)

示例：
// v1.0.0
function getUser(id) {
  return { id, name: 'Alice' };
}

// v1.1.0 (新增功能,兼容)
function getUser(id) {
  return { id, name: 'Alice' };
}

function getUserWithOrders(id) {  // ✅ 新增函数
  return {
    ...getUser(id),
    orders: []
  };
}

// 用户代码不受影响：
const user = getUser(123);  // 仍然工作
console.log(user.name);     // 仍然有效

💡 建议：
次版本号可以放心升级
```

### 3. 修订号(Patch)

```
Patch: 修订号,向后兼容的bug修复

什么时候增加：
✅ 修复了bug
✅ 性能优化(不改变API)
✅ 文档更新

示例：
// v1.0.0 (有bug)
function calculateTotal(price, quantity) {
  return price * quantity;  // ❌ 忘记处理小数
}

// v1.0.1 (修复bug)
function calculateTotal(price, quantity) {
  return parseFloat((price * quantity).toFixed(2));  // ✅ 修复
}

// 用户代码不受影响：
const total = calculateTotal(9.99, 3);
// v1.0.0: 29.970000000000002 (bug)
// v1.0.1: 29.97 (修复)

💡 建议：
修订号应该立即升级
```

## 🎨 版本号前缀

### 1. ^ (Caret) - 兼容版本

```json
{
  "dependencies": {
    "express": "^4.18.0"
  }
}

// ===== 规则 =====

^4.18.0 = >=4.18.0 且 <5.0.0
// 允许安装 4.18.0 到 4.x.x 的任意版本
// 但不允许 5.0.0

^1.2.3 → 允许 1.2.3, 1.2.4, 1.3.0, 1.9.9
         不允许 2.0.0

^0.2.3 → 允许 0.2.3, 0.2.4
         不允许 0.3.0 (0.x版本特殊处理)

^0.0.3 → 只允许 0.0.3 (0.0.x版本锁定)

// ===== 示例 =====

// package.json
"express": "^4.18.0"

// 实际安装的版本(取决于发布时间)
npm install
// 可能安装 4.18.2 (2023年发布的最新版)
// 可能安装 4.19.0 (2024年发布的新版)
// 不会安装 5.0.0 (主版本号不同)

// 💡 推荐使用 ^,平衡稳定性和新特性
```

### 2. ~ (Tilde) - 近似版本

```json
{
  "dependencies": {
    "express": "~4.18.0"
  }
}

// ===== 规则 =====

~4.18.0 = >=4.18.0 且 <4.19.0
// 只允许修订号(Patch)更新
// 不允许次版本号(Minor)更新

~1.2.3 → 允许 1.2.3, 1.2.4, 1.2.9
         不允许 1.3.0

~1.2 → 允许 1.2.0, 1.2.1, ..., 1.2.9999
       不允许 1.3.0

~1 → 允许 1.0.0, 1.1.0, ..., 1.9.9
     不允许 2.0.0

// ===== 示例 =====

"express": "~4.18.0"

npm install
// 可能安装 4.18.1, 4.18.2 (修订版)
// 不会安装 4.19.0 (次版本号不同)

// 💡 适合追求稳定的项目
```

### 3. 无前缀 - 精确版本

```json
{
  "dependencies": {
    "express": "4.18.0"
  }
}

// ===== 规则 =====

4.18.0 = 只安装 4.18.0
// 完全锁定版本

// ===== 示例 =====

"express": "4.18.0"

npm install
// 只安装 4.18.0,不管有没有新版本

// 💡 适合需要精确控制的场景
```

### 4. 其他版本范围

```json
{
  "dependencies": {
    // 最新版本
    "pkg1": "*",          // ❌ 不推荐,太不稳定
    "pkg2": "latest",     // ❌ 不推荐
    
    // 版本范围
    "pkg3": ">=1.0.0 <2.0.0",
    "pkg4": "1.0.0 - 1.5.0",
    
    // 多个版本
    "pkg5": "1.0.0 || 2.0.0",
    
    // Git 仓库
    "pkg6": "git+https://github.com/user/pkg.git",
    "pkg7": "github:user/pkg#v1.0.0",
    
    // 本地路径
    "pkg8": "file:../my-package"
  }
}
```

## 🔒 版本锁定

### 1. package-lock.json

```
package-lock.json = 锁定依赖的精确版本

作用：
✅ 确保团队成员安装相同版本
✅ 确保CI/CD安装相同版本
✅ 记录完整的依赖树(包括依赖的依赖)

生成：
npm install 时自动生成

提交：
✅ 应该提交到Git
```

```json
// package.json
{
  "dependencies": {
    "express": "^4.18.0"  // 允许 4.x.x
  }
}

// package-lock.json (自动生成)
{
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",  // 锁定精确版本
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-...",
      "dependencies": {
        // express 的依赖也被锁定
      }
    }
  }
}

// 💡 其他团队成员 npm install 时,会安装相同的 4.18.2
```

### 2. npm ci(推荐用于CI/CD)

```bash
# ===== npm install vs npm ci =====

# npm install
# - 读取 package.json
# - 根据版本范围安装最新版本
# - 更新 package-lock.json

# npm ci (Continuous Integration)
# - 只读取 package-lock.json
# - 安装精确版本
# - 不会修改 package-lock.json
# - 速度更快

# ===== 使用场景 =====

# 开发环境
npm install

# CI/CD 环境(推荐)
npm ci

# 💡 npm ci 确保每次构建使用相同依赖
```

### 案例1：依赖版本升级

```bash
# ===== 场景：express 发布了新版本 =====

# 当前安装：express@4.18.0
# 新版本：4.18.2 (修复bug), 4.19.0 (新功能), 5.0.0 (重大更新)

# package.json
"express": "^4.18.0"

# ===== 检查可用更新 =====

npm outdated
# Package  Current  Wanted  Latest
# express   4.18.0  4.19.0   5.0.0

# Current: 当前安装版本
# Wanted: 满足package.json规则的最新版本
# Latest: npm上的最新版本

# ===== 更新策略 =====

# 1. 安全更新(只更新Patch)
npm update express
# 更新到 4.18.2

# 2. 小版本更新(更新Minor)
npm update express
# 更新到 4.19.0 (如果^4.18.0)

# 3. 主版本更新(手动)
npm install express@latest
# 更新到 5.0.0
# ⚠️ 需要查看变更日志,测试兼容性!

# 4. 查看变更日志
npm view express versions  # 所有版本
npm view express@5.0.0     # 查看5.0.0信息
```

### 案例2：依赖冲突解决

```bash
# ===== 场景：依赖冲突 =====

# package.json
{
  "dependencies": {
    "pkg-a": "^1.0.0",  // 依赖 lodash@^4.17.0
    "pkg-b": "^2.0.0"   // 依赖 lodash@^3.10.0
  }
}

# ===== npm如何处理 =====

# npm v3+: 扁平化依赖树
node_modules/
  ├─ pkg-a/
  ├─ pkg-b/
  │  └─ node_modules/
  │     └─ lodash@3.10.1  # pkg-b独立安装旧版本
  └─ lodash@4.17.21       # 提升到顶层

# ===== 解决方案 =====

# 1. 查看依赖树
npm list lodash
# 查看哪些包依赖 lodash

# 2. 使用 overrides(npm 8.3+)
{
  "overrides": {
    "lodash": "^4.17.0"  // 强制所有包使用4.x
  }
}

# 3. 升级有冲突的包
npm update pkg-b  # 升级到支持lodash@4的版本
```

### 案例3：安全漏洞修复

```bash
# ===== npm audit(安全审计) =====

npm audit
# 发现 3 个漏洞 (1 低危, 1 中危, 1 高危)

# ===== 查看详情 =====

npm audit --json  # JSON格式
npm audit --parseable  # 适合CI解析

# ===== 自动修复 =====

npm audit fix
# 自动升级到安全版本(不跨主版本)

# 强制升级(可能跨主版本)
# ⚠️ 小心破坏兼容性!

# ===== 手动修复 =====

# 1. 查看漏洞详情
npm audit
# found: lodash <4.17.21
# severity: high
# recommendation: upgrade to 4.17.21

# 2. 更新依赖
npm install lodash@4.17.21

# 3. 再次审计
npm audit
# found 0 vulnerabilities
```

## 🎓 版本选择决策树

```
┌─ 需要最新特性吗？
│
├─ 是 → 使用 ^ (^4.18.0)
│       定期 npm update
│
└─ 否 → 需要绝对稳定吗？
        │
        ├─ 是 → 锁定精确版本 (4.18.0)
        │
        └─ 否 → 使用 ~ (~4.18.0)
                只接受bug修复
```

### 2. npm version(版本管理命令)

```bash
# ===== 自动更新版本号 =====

# 修复bug
npm version patch
# 1.0.0 → 1.0.1

# 新增功能
npm version minor
# 1.0.1 → 1.1.0

# 重大更新
npm version major
# 1.1.0 → 2.0.0

# 自定义版本
npm version 3.5.7

# ===== 自动创建Git标签 =====

npm version patch
# 1. 修改 package.json 版本号

# 3. git commit -m "v1.0.1"
# 4. git tag v1.0.1

# ===== 禁用Git操作 =====

# 只修改版本号,不执行Git操作

# ===== 预发布版本 =====

npm version prerelease --preid=alpha

# 1.0.1-alpha.0 → 1.0.1-alpha.1

# 1.0.0 → 1.0.1-beta.0

```

### 3. 预发布版本规范

```bash
# ===== 预发布版本格式 =====

1.0.0-alpha.0    # Alpha版本(内部测试)
1.0.0-beta.0     # Beta版本(公开测试)
1.0.0-rc.0       # RC版本(Release Candidate,发布候选)
1.0.0            # 正式版本

1.0.0-alpha.0 < 1.0.0-alpha.1 < 1.0.0-beta.0 < 1.0.0-rc.0 < 1.0.0

npm install express@next       # 安装next标签(通常是beta)
npm install express@beta       # 安装beta标签
npm install express@1.0.0-rc.0 # 安装指定预发布版本

# ===== 发布预发布版本 =====

npm publish --tag beta
# 发布为beta版本,不会成为latest

# 用户需要指定标签安装
npm install my-package@beta
```

### 问题2：package.json 和 package-lock.json 版本冲突怎么办？

**解答：**

```bash
# ===== 场景 =====

# package.json: "express": "^4.18.0"
# package-lock.json: "version": "4.17.0"(旧版本)

# ===== 解决方案1：重新生成 lock 文件 =====

rm -rf node_modules package-lock.json
npm install
# 生成新的 package-lock.json

# ===== 解决方案2：使用 npm update =====

npm update express
# 更新到满足 ^4.18.0 的最新版本

# ===== 解决方案3：强制更新(不推荐) =====

npm install express@latest --save
# 更新 package.json 和 package-lock.json

# ===== 最佳实践 =====

# 1. 团队协作时,总是先 git pull
# 2. 执行 npm install 同步依赖
# 3. 提交代码前确保 package-lock.json 是最新的
```

### 问题4：^1.2.3 和 ~1.2.3 到底有什么区别？

**解答：**

```
精确对比：

^1.2.3 (Caret - 兼容版本)
✅ 允许：1.2.3, 1.2.4, 1.3.0, 1.4.0, 1.9.9
❌ 不允许：2.0.0
规则：不改变最左边的非零数字

~1.2.3 (Tilde - 近似版本)
✅ 允许：1.2.3, 1.2.4, 1.2.99
❌ 不允许：1.3.0, 2.0.0
规则：只允许修订号(Patch)更新

特殊情况(0.x.x版本)：

^0.2.3
✅ 允许：0.2.3, 0.2.4
❌ 不允许：0.3.0
规则：0.x版本被认为不稳定,锁定次版本号

~0.2.3
✅ 允许：0.2.3, 0.2.4
❌ 不允许：0.3.0
规则：与^0.2.3相同

实际应用建议：
✅ 大多数情况：使用 ^(获得bug修复和新特性)
✅ 关键依赖：使用精确版本或~(更保守)
✅ 开发工具(prettier,eslint)：使用精确版本(保持团队一致)
```

### 问题5：如何处理"无法满足的对等依赖"警告？

**解答：**

```bash
# ===== 警告示例 =====

npm WARN my-plugin@1.0.0 requires a peer of react@^18.0.0 but none is installed.

# ===== 原因 =====

# my-plugin 需要 react@18,但你没有安装

# ===== 解决方案1：安装缺失的依赖 =====

npm install react@18

# ===== 解决方案2：检查版本冲突 =====

npm list react
# 查看当前安装的 react 版本

# 如果已安装但版本不匹配：
npm install react@18  # 升级到兼容版本

# ===== 解决方案3：忽略警告(不推荐) =====

npm install --legacy-peer-deps
# 忽略对等依赖冲突

# ===== 最佳实践 =====
```

## 🎓 常见面试题

### 问题1：什么是语义化版本(SemVer)？解释主版本号、次版本号、修订号的含义。

**回答：**

```
语义化版本(Semantic Versioning, SemVer)定义：

格式：主版本号.次版本号.修订号 (Major.Minor.Patch)
示例：2.3.5

1. 主版本号(Major) - 不兼容的API变更
   - 删除公开API
   - 修改API行为(不向后兼容)
   - 重构核心架构
   - 示例：Express 4 → Express 5(移除deprecated API)

2. 次版本号(Minor) - 向后兼容的功能新增
   - 新增公开API
   - 标记功能为废弃(但仍可用)
   - 内部大量改进
   - 示例：新增 app.route() 方法

3. 修订号(Patch) - 向后兼容的bug修复
   - 修复bug
   - 性能优化(不改变API)
   - 文档更新
   - 示例：修复内存泄漏

递增规则：
- Patch: 1.0.0 → 1.0.1
- Minor: 1.0.1 → 1.1.0 (Patch归零)
- Major: 1.1.0 → 2.0.0 (Minor和Patch归零)

实际价值：
✅ 用户可以根据版本号判断是否可以安全升级
✅ ^和~等版本范围符号才有意义
✅ 减少"依赖地狱"问题
```

### 问题2：^ 和 ~ 有什么区别？实际项目中如何选择？

**回答：**

```
核心区别：

^ (Caret) - 兼容更新
"express": "^4.18.0"
- 允许：4.18.0 → 4.99.99
- 不允许：5.0.0
- 规则：不改变主版本号
- 适用：获得新功能和bug修复

~ (Tilde) - 近似更新
"express": "~4.18.0"
- 允许：4.18.0 → 4.18.99
- 不允许：4.19.0
- 规则：只允许修订号更新
- 适用：只接受bug修复

实际项目选择策略：

1. 第三方库(lodash, axios)：
   ✅ 使用 ^ - 获得新特性和bug修复
   "lodash": "^4.17.21"

2. 框架(express, react)：
   ✅ 使用 ^ - 框架通常遵循SemVer
   "express": "^4.18.0"

3. 关键业务依赖：
   ✅ 使用 ~ 或精确版本 - 更保守
   "payment-sdk": "~2.3.0"

4. 开发工具(prettier, eslint)：
   ✅ 使用精确版本 - 确保团队一致
   "prettier": "2.8.8"

5. 不稳定的包(0.x.x)：
   ✅ 使用 ~ 或精确版本
   "experimental-lib": "~0.5.0"

推荐配置：
{
  "dependencies": {
    "express": "^4.18.0",        // 框架：^
    "lodash": "^4.17.21",        // 工具库：^
    "payment-api": "~2.3.0"      // 关键依赖：~
  },
  "devDependencies": {
    "prettier": "2.8.8",         // 开发工具：精确
    "eslint": "8.40.0"
  }
}
```

### 问题3：package-lock.json 的作用是什么？为什么需要它？

**回答：**

```
package-lock.json 核心作用：

1. 锁定精确版本
   package.json:      "express": "^4.18.0" (允许4.x.x)
   package-lock.json: "version": "4.18.2" (精确锁定)

2. 锁定完整依赖树
   - 不仅锁定直接依赖
   - 还锁定依赖的依赖(传递依赖)
   - 记录每个包的下载地址和校验和

3. 提升安装速度
   - 跳过版本解析过程
   - 直接从锁定的地址下载

4. 保证一致性
   ✅ 团队成员：相同环境
   ✅ CI/CD：可重现构建
   ✅ 生产部署：与测试环境一致

为什么需要它？

问题场景：
团队成员A在2023年1月安装：express@4.18.0
团队成员B在2023年6月安装：express@4.18.2(新版本)
→ 相同的package.json,不同的依赖版本
→ 可能导致"在我电脑上能跑"问题

解决方案：
有了package-lock.json,所有人都安装4.18.0

最佳实践：
✅ 应用项目：必须提交到Git
✅ npm包(library)：添加到.gitignore
✅ CI/CD：使用npm ci而非npm install

常见命令：
npm install      # 可能更新lock文件
npm ci           # 严格按lock文件安装(推荐CI/CD)
npm update       # 更新依赖并修改lock文件
```

### 问题4：如何安全地升级项目的依赖包？

**回答：**

```bash
安全升级依赖包的完整流程：

步骤1：检查可用更新
npm outdated
# Package   Current  Wanted  Latest  Location
# express    4.18.0  4.19.0   5.0.0  my-app
# lodash     4.17.20 4.17.21 4.17.21  my-app

步骤2：查看变更日志
# 访问项目的CHANGELOG或GitHub releases
# 了解新版本的变更内容

步骤3：安全更新(Patch和Minor)
npm update
# 只更新到Wanted版本(遵循^或~规则)
# 风险低,通常向后兼容

步骤4：运行测试
npm test
npm run build
# 确保更新后功能正常

步骤5：主版本更新(Major)
# 谨慎处理,可能有破坏性变更

# 5.1 创建测试分支
git checkout -b upgrade-express-v5

# 5.2 安装新版本
npm install express@5

# 5.3 查看迁移指南
# 访问官方文档/迁移指南

# 5.4 修改代码适配新版本
# 按照迁移指南调整代码

# 5.5 全面测试
npm test
npm run e2e-test
# 手动测试关键功能

# 5.6 代码审查
git diff
# 检查所有变更

# 5.7 合并到主分支
git checkout main
git merge upgrade-express-v5

步骤6：处理安全漏洞
npm audit
# 检查安全问题

npm audit fix
# 自动修复(不跨主版本)

npm audit fix --force
# 强制修复(⚠️可能破坏兼容性)

步骤7：更新package-lock.json
npm install
git add package.json package-lock.json
git commit -m "chore: upgrade dependencies"

工具辅助：
# npm-check-updates(批量升级工具)
npx ncu -u
npm install
npm test

最佳实践：
✅ 定期更新(每周/每月)
✅ 小步迭代(每次只升级少数包)
✅ 充分测试(自动化测试覆盖)
✅ 查看变更日志(了解breaking changes)
✅ 回滚准备(Git标签,备份)
```
