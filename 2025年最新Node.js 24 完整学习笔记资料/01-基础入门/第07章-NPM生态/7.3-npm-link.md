# 7.3 npm link 本地开发调试

> 在本地测试 npm 包,无需发布到 npm 仓库

---

## 🎯 什么是 npm link？

### 核心概念

```
npm link = 创建软链接,让本地包像已发布的包一样使用

问题场景：
你正在开发一个npm包 my-utils
另一个项目 my-app 要使用 my-utils

传统方式(❌ 麻烦)：
1. 修改 my-utils 代码
2. npm publish发布到npm
3. 在 my-app 中 npm install my-utils
4. 测试
5. 发现bug,回到步骤1

npm link 方式(✅ 方便)：
1. 在 my-utils 目录: npm link
2. 在 my-app 目录: npm link my-utils
3. 修改 my-utils 代码
4. my-app 立即看到变化!
5. 测试完成后再发布

类比：
npm link = 给文件夹创建快捷方式
修改快捷方式指向的文件夹,所有使用它的地方都能看到
```

## 🚀 基础使用

### 1. 创建link

```bash
# ===== 场景：开发 npm 包 =====

# 1. 进入包目录
cd /path/to/my-utils

# 2. 创建全局链接
npm link

# 输出：
# /usr/local/lib/node_modules/my-utils -> /path/to/my-utils

# 💡 这一步做了什么：
# 在全局 node_modules 中创建了一个软链接
# 指向你的本地开发目录

# 3. 验证
npm list -g --depth=0
# └── my-utils@1.0.0 -> /path/to/my-utils
```

### 2. 使用link

```bash
# ===== 在其他项目中使用 =====

# 1. 进入项目目录
cd /path/to/my-app

# 2. 链接本地包
npm link my-utils

# 输出：
# /path/to/my-app/node_modules/my-utils -> /usr/local/lib/node_modules/my-utils -> /path/to/my-utils

# 💡 这一步做了什么：
# 在项目的 node_modules 中创建软链接
# 指向全局的软链接,最终指向本地开发目录

# 3. 在代码中使用
# my-app/index.js
const utils = require('my-utils');
console.log(utils.sayHello());
```

### 3. 取消link

```bash
# ===== 在项目中取消链接 =====

cd /path/to/my-app
npm unlink my-utils

# 或者(删除并重新安装正常依赖)
npm unlink my-utils --save
npm install

# ===== 取消全局链接 =====

cd /path/to/my-utils
npm unlink

# 或者在任意位置
npm unlink -g my-utils
```

## 🎓 npm link vs 其他方案

```bash
# ===== 方案对比 =====

# 1. npm link(软链接)
# ✅ 实时更新
# ✅ 不占用额外空间
# ❌ 需要手动管理
npm link my-utils

# 2. npm install <folder>(硬拷贝)
# ✅ 简单
# ❌ 修改后需要重新安装
# ❌ 占用空间
npm install ../my-utils

# 3. file: 协议(package.json)
# ✅ 自动处理
# ❌ 修改后需要重新安装
{
  "dependencies": {
    "my-utils": "file:../my-utils"
  }
}

# 4. Git 依赖
# ✅ 团队协作方便
# ❌ 需要提交才能看到变化
{
  "dependencies": {
    "my-utils": "git+https://github.com/user/my-utils.git#dev"
  }
}

# 5. npm workspaces(推荐,Monorepo)
# ✅ 自动管理链接
# ✅ 适合多包项目
{
  "workspaces": [
    "packages/*"
  ]
}

# 💡 选择建议:
# - 单包开发: npm link
# - 多包项目: npm workspaces
# - 临时测试: npm install <folder>
```

### 技巧4：检查 link 状态

```javascript
// check-links.js
const fs = require('fs');
const path = require('path');

const nodeModulesPath = path.join(process.cwd(), 'node_modules');

function checkLinks(dir) {
  const items = fs.readdirSync(dir);
  const links = [];
  
  for (const item of items) {
    const itemPath = path.join(dir, item);
    const stats = fs.lstatSync(itemPath);
    
    if (stats.isSymbolicLink()) {
      const realPath = fs.realpathSync(itemPath);
      links.push({ name: item, path: realPath });
    } else if (stats.isDirectory() && item.startsWith('@')) {
      // 检查作用域包
      links.push(...checkLinks(itemPath));
    }
  }
  
  return links;
}

const linkedPackages = checkLinks(nodeModulesPath);

if (linkedPackages.length > 0) {
  console.log('🔗 检测到链接的包:');
  linkedPackages.forEach(({ name, path }) => {
    console.log(`  ${name} -> ${path}`);
  });
} else {
  console.log('✅ 没有链接的包');
}

// 使用
// node check-links.js
```

## 🎓 常见面试题

### 问题1：npm link 的工作原理是什么？

**回答：**

```
npm link 创建符号链接(symbolic link),分两步：

步骤1：npm link (在包目录)
- 在全局 node_modules 创建符号链接
- /usr/local/lib/node_modules/my-pkg -> /path/to/my-pkg
- 如果有 bin 字段,也会在全局 bin 目录创建链接

步骤2：npm link <pkg> (在项目目录)
- 在项目 node_modules 创建符号链接
- /project/node_modules/my-pkg -> /usr/local/lib/node_modules/my-pkg
- 最终指向: /project/node_modules/my-pkg -> /path/to/my-pkg

实际效果：
✅ 修改源代码立即生效(无需重新安装)
✅ 不占用额外磁盘空间
✅ 支持多个项目同时链接

底层实现：
- Unix/Mac: 使用 ln -s (软链接)
- Windows: 使用 mklink /D (目录链接)或 junction

与硬拷贝(npm install ../pkg)的区别：
- npm link: 符号链接,实时更新
- npm install: 硬拷贝,需要重新安装才能更新
```

### 问题2：npm link 有什么限制和注意事项？

**回答：**

```
主要限制：

1. 依赖冲突
   问题：链接的包和项目的依赖版本可能不同
   示例：项目用 React 17,链接的包用 React 18
   解决：使用 peerDependencies 或在包中也 link 共享依赖

2. TypeScript 类型问题
   问题：链接后类型定义可能找不到
   解决：确保编译后有 .d.ts 文件,使用 tsc --watch

3. 路径解析问题
   问题：某些构建工具可能不支持符号链接
   示例：Webpack 默认不解析符号链接
   解决：配置 resolve.symlinks: true

4. CI/CD 环境
   问题：CI 环境不应该使用 npm link
   解决：只在本地开发使用,CI 使用正常的 npm install

5. 团队协作
   问题：团队成员路径可能不同
   解决：提供清晰的开发文档,或使用 Workspaces

注意事项：
✅ 开发完成后及时 unlink
✅ 发布前确保使用正式版本测试
✅ 记录哪些包被 link 了
✅ 使用 npm list 检查 link 状态
✅ Windows 需要管理员权限或开发者模式
```

### 问题3：npm link 和 npm workspaces 有什么区别？应该选择哪个？

**回答：**

```
核心区别：

npm link:
- 手动管理符号链接
- 适合单个包开发
- 灵活但需要手动操作
- 支持 npm 5+

npm workspaces:
- 自动管理符号链接
- 适合 Monorepo (多包项目)
- 自动化但有结构要求
- 需要 npm 7+

对比示例：

npm link 方式：
# 步骤较多
cd pkg-a && npm link
cd pkg-b && npm link pkg-a
cd pkg-c && npm link pkg-a && npm link pkg-b

npm workspaces 方式：
# 一次搞定
npm install
# 自动处理所有依赖关系

选择建议：

使用 npm link:
✅ 开发单个 npm 包
✅ 包不在同一个仓库
✅ 临时调试第三方包
✅ 需要灵活控制

使用 npm workspaces:
✅ Monorepo 项目
✅ 多个相关的包
✅ 团队协作开发
✅ 需要统一管理

实际场景：
- 开发一个工具库给多个项目用 → npm link
- 开发一套组件库(多个包) → npm workspaces
- 修改第三方包调试 → npm link
- 大型前端项目(多个模块) → npm workspaces

混合使用：
也可以在 Workspaces 项目外部 link 某个包
cd external-pkg && npm link
cd my-workspace/packages/app && npm link external-pkg
```

### 问题4：如何在生产部署前确保没有使用 link 的包？

**回答：**

```bash
# ===== 方法1：检查 node_modules =====

# 查找所有符号链接
find node_modules -type l

# 如果有输出,说明有 link 的包

# ===== 方法2：npm list 检查 =====

npm list --depth=0 | grep "^├─\|^└─" | grep "->"

# 有 "->" 符号表示是链接的包
# 示例输出:
# ├── my-utils@1.0.0 -> /path/to/my-utils

# ===== 方法3：Node.js 脚本检查 =====

// check-production.js
const fs = require('fs');
const path = require('path');

function findSymlinks(dir, results = []) {
  if (!fs.existsSync(dir)) return results;
  
  const items = fs.readdirSync(dir);
  
  for (const item of items) {
    const fullPath = path.join(dir, item);
    const stats = fs.lstatSync(fullPath);
    
    if (stats.isSymbolicLink()) {
      results.push(fullPath);
    } else if (stats.isDirectory() && !item.startsWith('.')) {
      findSymlinks(fullPath, results);
    }
  }
  
  return results;
}

const symlinks = findSymlinks('./node_modules');

if (symlinks.length > 0) {
  console.error('❌ 检测到链接的包:');
  symlinks.forEach(link => console.error(`  ${link}`));
  process.exit(1);
} else {
  console.log('✅ 没有链接的包,可以部署');
  process.exit(0);
}

# ===== 方法4：CI/CD 检查 =====

# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  check-links:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Install dependencies
        run: npm ci  # 使用 ci 而不是 install
      
      - name: Check for symlinks
        run: |
          if find node_modules -type l | grep -q .; then
            echo "Error: Found symlinked packages"
            find node_modules -type l
            exit 1
          fi
      
      - name: Run tests
        run: npm test
      
      - name: Build
        run: npm run build

# ===== 方法5：package.json 脚本 =====

{
  "scripts": {
    "check:links": "node check-production.js",
    "prebuild": "npm run check:links",
    "predeploy": "npm run check:links"
  }
}

# 构建前自动检查
npm run build
# 如果有 link 的包,会报错并阻止构建

# ===== 最佳实践 =====

1. 使用 npm ci 而不是 npm install (CI/CD 环境)
   - npm ci 会删除 node_modules 重新安装
   - 不会保留 link

2. 添加 pre-commit hook
   # .husky/pre-commit
   npm run check:links

3. 在 package.json 添加注释
   {
     "dependencies": {
       "my-utils": "^1.0.0"  // ⚠️ 本地开发使用 npm link
     }
   }

4. 创建清理脚本
   npm run clean && npm install
```
