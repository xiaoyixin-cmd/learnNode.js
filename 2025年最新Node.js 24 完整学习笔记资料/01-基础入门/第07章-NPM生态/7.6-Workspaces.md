# 7.6 npm Workspaces - Monorepo 管理

> 一个仓库管理多个包,提升开发效率

---

## 🎯 什么是 Workspaces？

### 核心概念

```
Workspaces = Monorepo 的官方解决方案

Monorepo vs Multirepo：

Multirepo(传统方式)：
my-company/
  ├─ repo-A/  (独立仓库)
  ├─ repo-B/  (独立仓库)
  └─ repo-C/  (独立仓库)

❌ 问题：
- 共享代码麻烦(发布→安装)
- 版本管理混乱
- 重复的配置文件
- CI/CD 复杂

Monorepo(单一仓库)：
my-company/
  └─ monorepo/
      ├─ packages/
      │  ├─ package-A/
      │  ├─ package-B/
      │  └─ package-C/
      └─ package.json

✅ 优势：
- 代码共享简单
- 统一的版本管理
- 共享配置和工具
- 统一的CI/CD
```

### Workspaces 特性

```
npm Workspaces (npm 7+)：

✅ 自动软链接(类似 npm link)
✅ 共享 node_modules(减少空间)
✅ 统一的依赖安装
✅ 批量运行脚本
✅ 包之间自动解析依赖
```

## 🚀 快速开始

### 1. 创建 Workspaces 项目

```bash
# ===== 初始化项目 =====

mkdir my-monorepo
cd my-monorepo
npm init -y

# ===== 配置 Workspaces =====

# package.json
{
  "name": "my-monorepo",
  "version": "1.0.0",
  "private": true,  // ⚠️ 必须设为 true
  "workspaces": [
    "packages/*"    // 所有 packages 下的包都是 workspace
  ]
}

# ===== 创建包 =====

mkdir -p packages/pkg-a
mkdir -p packages/pkg-b

# packages/pkg-a/package.json
{
  "name": "@mycompany/pkg-a",
  "version": "1.0.0",
  "main": "index.js"
}

# packages/pkg-a/index.js
module.exports = {
  sayHello() {
    return 'Hello from pkg-a!';
  }
};

# packages/pkg-b/package.json
{
  "name": "@mycompany/pkg-b",
  "version": "1.0.0",
  "main": "index.js",
  "dependencies": {
    "@mycompany/pkg-a": "^1.0.0"  // 依赖 pkg-a
  }
}

# packages/pkg-b/index.js
const pkgA = require('@mycompany/pkg-a');

module.exports = {
  greet() {
    return `pkg-b says: ${pkgA.sayHello()}`;
  }
};

# ===== 安装依赖 =====

npm install

# 💡 npm 自动：
# 1. 安装所有包的依赖
# 2. 创建软链接：pkg-b → pkg-a
# 3. 共享 node_modules

# 目录结构：
# my-monorepo/
#   ├─ node_modules/
#   │  ├─ @mycompany/
#   │  │  ├─ pkg-a → ../../packages/pkg-a (软链接)
#   │  │  └─ pkg-b → ../../packages/pkg-b (软链接)
#   ├─ packages/
#   │  ├─ pkg-a/
#   │  └─ pkg-b/
#   └─ package.json
```

### 2. 测试包

```javascript
// ===== 创建测试文件 =====

// test.js (根目录)
const pkgA = require('@mycompany/pkg-a');
const pkgB = require('@mycompany/pkg-b');

console.log(pkgA.sayHello());  // Hello from pkg-a!
console.log(pkgB.greet());     // pkg-b says: Hello from pkg-a!

// 运行
node test.js

// ✅ pkg-b 能直接使用 pkg-a,无需发布到 npm！
```

## 📦 依赖管理

### 1. 安装依赖

```bash
# ===== 给根项目安装依赖 =====

npm install lodash
# 安装到根目录的 node_modules

# ===== 给特定 workspace 安装依赖 =====

npm install express --workspace=@mycompany/pkg-a
# 或
npm install express -w @mycompany/pkg-a

# 等同于：
# cd packages/pkg-a
# npm install express

# ===== 给所有 workspaces 安装 =====

npm install lodash --workspaces
# 所有包都会添加 lodash 依赖

# ===== 查看已安装的包 =====

npm list --workspaces
# 列出所有 workspace 的依赖
```

### 2. 运行脚本

```json
// ===== packages/pkg-a/package.json =====
{
  "scripts": {
    "test": "echo 'Testing pkg-a'",
    "build": "echo 'Building pkg-a'"
  }
}

// ===== packages/pkg-b/package.json =====
{
  "scripts": {
    "test": "echo 'Testing pkg-b'",
    "build": "echo 'Building pkg-b'"
  }
}
```

```bash
# ===== 运行特定 workspace 的脚本 =====

npm run test --workspace=@mycompany/pkg-a
# Testing pkg-a

# ===== 运行所有 workspaces 的脚本 =====

npm run test --workspaces
# Testing pkg-a
# Testing pkg-b

# ===== 串行运行(一个接一个) =====

npm run build --workspaces

# ===== 并行运行(同时执行) =====

npm run test --workspaces --workspaces-parallel
# 或
npm run test --ws-parallel

# 💡 适合独立的任务,提高速度
```

### 3. 查看 workspaces

```bash
# ===== 列出所有 workspaces =====

npm ls --workspaces
# 或
npm ls --ws

# 输出：
# my-monorepo@1.0.0
# ├─ @mycompany/pkg-a@1.0.0
# └─ @mycompany/pkg-b@1.0.0

# ===== 查看 workspace 信息 =====

npm query "[workspaces]"
# 返回 JSON 格式的详细信息
```

## 🎯 实战案例

### 案例1：完整的 Monorepo 项目

```bash
# ===== 项目结构 =====

my-app/
  ├─ packages/
  │  ├─ core/           # 核心包
  │  ├─ utils/          # 工具函数
  │  ├─ ui/             # UI 组件
  │  └─ app/            # 主应用
  ├─ package.json
  └─ package-lock.json

# ===== 根 package.json =====

{
  "name": "my-app",
  "private": true,
  "workspaces": [
    "packages/*"
  ],
  "scripts": {
    "dev": "npm run dev --workspaces --if-present",
    "build": "npm run build --workspaces --if-present",
    "test": "npm run test --workspaces",
    "clean": "rm -rf packages/*/dist packages/*/node_modules"
  },
  "devDependencies": {
    "typescript": "^5.0.0",
    "eslint": "^8.0.0",
    "prettier": "^2.8.0"
  }
}

# ===== packages/core/package.json =====

{
  "name": "@myapp/core",
  "version": "1.0.0",
  "main": "dist/index.js",
  "scripts": {
    "build": "tsc",
    "test": "jest"
  }
}

# ===== packages/utils/package.json =====

{
  "name": "@myapp/utils",
  "version": "1.0.0",
  "main": "dist/index.js",
  "dependencies": {
    "@myapp/core": "^1.0.0"  // 依赖 core 包
  },
  "scripts": {
    "build": "tsc",
    "test": "jest"
  }
}

# ===== packages/ui/package.json =====

{
  "name": "@myapp/ui",
  "version": "1.0.0",
  "main": "dist/index.js",
  "dependencies": {
    "@myapp/core": "^1.0.0",
    "@myapp/utils": "^1.0.0",
    "react": "^18.0.0"
  },
  "scripts": {
    "build": "tsc && webpack",
    "test": "jest"
  }
}

# ===== packages/app/package.json =====

{
  "name": "@myapp/app",
  "version": "1.0.0",
  "private": true,
  "dependencies": {
    "@myapp/core": "^1.0.0",
    "@myapp/utils": "^1.0.0",
    "@myapp/ui": "^1.0.0",
    "express": "^4.18.0"
  },
  "scripts": {
    "dev": "nodemon src/index.js",
    "build": "webpack",
    "start": "node dist/index.js"
  }
}

# ===== 开发流程 =====

# 1. 安装所有依赖
npm install

# 2. 构建所有包
npm run build

# 3. 启动开发服务器
npm run dev

# 4. 运行所有测试
npm run test

# 💡 修改任何包的代码,app 立即看到变化!
```

## 🛠️ 高级技巧

### 1. 条件执行脚本

```bash
# ===== --if-present 标志 =====

# 只运行存在的脚本,没有则跳过
npm run lint --workspaces --if-present

# 例如：
# pkg-a 有 lint 脚本 → 运行
# pkg-b 没有 lint 脚本 → 跳过(不报错)
```

### 2. 发布包

```bash
# ===== 发布单个 workspace =====

npm publish --workspace=@myapp/core

# ===== 发布所有公开包 =====

# 先确保所有 private: true 的包不会被发布
# packages/app/package.json
{
  "private": true  // 不会被发布
}

# 发布所有非私有包
npm publish --workspaces --access=public

# ===== 使用 lerna 管理版本 =====

npx lerna version
# 交互式选择版本号

npx lerna publish
# 发布所有有变更的包
```

### 3. 依赖分析

```bash
# ===== 查看依赖树 =====

npm list --workspaces --depth=0
# 只显示第一层依赖

# ===== 查看某个包被谁依赖 =====

npm list @myapp/core --workspaces

# 输出：
# @myapp/utils@1.0.0
# └─ @myapp/core@1.0.0
# @myapp/ui@1.0.0
# └─ @myapp/core@1.0.0
```

## 🆚 Workspaces vs 其他方案

| 特性 | npm Workspaces | Lerna | Yarn Workspaces | pnpm Workspaces |
|------|----------------|-------|-----------------|-----------------|
| **内置** | ✅ npm 7+ | ❌ 需要安装 | ✅ Yarn 1.0+ | ✅ pnpm |
| **学习成本** | ⭐ 低 | ⭐⭐ 中 | ⭐ 低 | ⭐ 低 |
| **功能** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **性能** | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **版本管理** | ❌ | ✅ 强大 | ✅ | ✅ |
| **适合场景** | 简单项目 | 复杂项目 | 中等项目 | 所有项目 |

### 推荐选择

```
小型项目(2-3个包) → npm Workspaces
中型项目(4-10个包) → npm Workspaces 或 Yarn/pnpm
大型项目(10+个包) → Lerna + npm/Yarn/pnpm
追求性能 → pnpm Workspaces
```

## 🚨 常见问题

### 问题1：循环依赖问题

```bash
# ===== 场景：pkg-a 依赖 pkg-b,pkg-b 也依赖 pkg-a =====

# packages/pkg-a/package.json
{
  "dependencies": {
    "@myapp/pkg-b": "^1.0.0"
  }
}

# packages/pkg-b/package.json
{
  "dependencies": {
    "@myapp/pkg-a": "^1.0.0"
  }
}

# 运行 npm install
npm install
# ❌ 错误：循环依赖!

# 解决方案1：拆分共享代码
packages/
  ├─ pkg-a/
  ├─ pkg-b/
  └─ shared/  ← 新建共享包
      # pkg-a 和 pkg-b 都依赖 shared

# 解决方案2：使用 peerDependencies
{
  "peerDependencies": {
    "@myapp/pkg-b": "^1.0.0"
  }
}

# 解决方案3：动态导入
// 延迟加载,避免启动时循环
const pkgB = require('@myapp/pkg-b');  // 改为
const pkgB = () => require('@myapp/pkg-b');
```

### 问题2：本地包版本不更新

```bash
# ===== 场景：修改了 pkg-a,但 pkg-b 使用的还是旧代码 =====

# 原因：软链接没有正确建立

# 解决方案1：清理重装
rm -rf node_modules package-lock.json
npm install

# 解决方案2：强制重建软链接
npm install --force

# 解决方案3：检查软链接
ls -la node_modules/@myapp/
# 应该看到 pkg-a -> ../../packages/pkg-a

# 解决方案4：使用 npm link 手动链接
cd packages/pkg-a
npm link
cd ../pkg-b
npm link @myapp/pkg-a
```

### 问题3：依赖重复安装

```bash
# ===== 场景：多个包安装了同一依赖的不同版本 =====

# pkg-a: lodash@4.17.21
# pkg-b: lodash@4.17.20

npm list lodash --workspaces
# 发现重复

# 解决方案1：统一版本
# 修改所有 package.json 使用相同版本
{
  "dependencies": {
    "lodash": "^4.17.21"  // 统一
  }
}

# 解决方案2：提升到根目录
# 根 package.json
{
  "dependencies": {
    "lodash": "^4.17.21"
  }
}
# 删除 workspace 中的 lodash

# 解决方案3：npm dedupe
npm dedupe
# 自动去重
```

### 问题4：无法发布包

```bash
# ===== 错误：This package has been marked as private =====

# 原因：根项目设为 private: true

# 解决方案：只发布特定 workspace
npm publish --workspace=@myapp/pkg-a

# 确保 workspace 的 package.json 没有 private: true
{
  "name": "@myapp/pkg-a",
  "private": false  // 或删除这一行
}
```

### 问题5：Windows 软链接失败

```bash
# ===== 错误：EPERM: operation not permitted =====

# 原因：Windows 需要管理员权限创建软链接

# 解决方案1：以管理员身份运行
# 右键 → 以管理员身份运行 PowerShell
npm install

# 解决方案2：启用开发者模式
# Windows 设置 → 更新和安全 → 开发者选项 → 开发人员模式

# 解决方案3：使用 pnpm
# pnpm 不使用软链接,用硬链接
npm install -g pnpm
pnpm install
```

## 🎓 常见面试题

### 问题1：什么是 Monorepo？与 Multirepo 相比有什么优缺点？

**回答：**

```
Monorepo（单一仓库）：
将多个相关项目/包放在一个Git仓库中。

与 Multirepo（多仓库）对比：

| 方面 | Monorepo | Multirepo |
|------|----------|-----------|
| 代码共享 | ✅ 简单(直接引用) | ❌ 复杂(发布→安装) |
| 依赖管理 | ✅ 统一版本 | ❌ 版本分散 |
| 重构 | ✅ 原子性提交 | ❌ 需要多次提交 |
| CI/CD | ✅ 统一流程 | ❌ 各自配置 |
| 权限控制 | ❌ 难以细分 | ✅ 仓库级别 |
| 仓库大小 | ❌ 随时间增长 | ✅ 保持小巧 |
| 构建速度 | ❌ 可能很慢 | ✅ 独立快速 |

Monorepo 优点：
✅ 代码共享方便(无需发布)
✅ 统一的工具链和配置
✅ 原子性提交(跨包重构)
✅ 依赖版本统一管理
✅ 更好的代码复用
✅ 减少重复配置

Monorepo 缺点：
❌ 仓库体积大
❌ CI/CD 复杂(需要增量构建)
❌ 权限管理困难
❌ Git 操作可能变慢
❌ IDE 性能压力

适合 Monorepo 的场景：
✅ 多个包高度关联
✅ 需要频繁跨包重构
✅ 团队规模中小(< 100人)
✅ 共享组件库
✅ 前端 + 后端 + 共享代码

适合 Multirepo 的场景：
✅ 项目完全独立
✅ 不同团队维护
✅ 需要严格权限控制
✅ 发布周期不同
✅ 大型组织(多团队)
```

### 问题2：npm Workspaces 的工作原理是什么？

**回答：**

```
npm Workspaces 工作原理：

1. 配置解析
   - 读取根 package.json 的 workspaces 字段
   - 支持 glob 模式："packages/*"
   - 扫描匹配的目录,识别为 workspace

2. 依赖解析
   步骤1：读取所有 workspace 的 package.json
   步骤2：构建依赖图
   步骤3：检测循环依赖
   步骤4：确定安装顺序

3. 软链接创建
   当 pkg-b 依赖 pkg-a 时：
   
   node_modules/
     └─ @myapp/
         └─ pkg-a → ../../packages/pkg-a (软链接)
   
   ✅ require('@myapp/pkg-a') 直接指向源代码
   ✅ 修改 pkg-a 立即影响 pkg-b
   ✅ 无需重新安装

4. 依赖提升(Hoisting)
   共享依赖提升到根 node_modules：
   
   root/
     ├─ node_modules/
     │  ├─ lodash/        ← 共享依赖
     │  └─ @myapp/
     │      ├─ pkg-a → ...
     │      └─ pkg-b → ...
     └─ packages/
         ├─ pkg-a/        (依赖 lodash)
         └─ pkg-b/        (依赖 lodash)
   
   ✅ 只安装一次 lodash
   ✅ 节省空间和时间

5. 脚本执行
   npm run build --workspaces
   
   步骤1：遍历所有 workspace
   步骤2：检查是否有 build 脚本
   步骤3：执行脚本(默认串行)
   步骤4：收集输出和错误

6. 版本匹配
   pkg-b 依赖："@myapp/pkg-a": "^1.0.0"
   pkg-a 当前版本："1.2.3"
   
   ✅ 1.2.3 匹配 ^1.0.0
   ✅ 使用本地软链接
   
   ❌ 如果 pkg-a 是 2.0.0
   ❌ 不匹配 ^1.0.0
   ❌ 从 npm registry 安装 1.x 版本

关键机制：

1. 符号链接(Symbolic Link)
   - 不是复制文件
   - 只是指向原文件的指针
   - 修改原文件,链接自动反映

2. 依赖提升(Hoisting)
   - 尽可能共享依赖
   - 减少磁盘占用
   - 避免重复安装

3. 拓扑排序(Topological Sort)
   - 按依赖顺序执行脚本
   - 确保依赖先构建

实际示例：

# 1. 配置 workspaces
{
  "workspaces": ["packages/*"]
}

# 2. npm install 时
- 扫描 packages/* 目录
- 读取每个 package.json
- 分析依赖关系
- 创建软链接
- 提升共享依赖

# 3. 目录结构
my-app/
  ├─ node_modules/
  │  ├─ express/        ← 共享依赖
  │  └─ @myapp/
  │      ├─ pkg-a → ../../packages/pkg-a
  │      └─ pkg-b → ../../packages/pkg-b
  ├─ packages/
  │  ├─ pkg-a/
  │  └─ pkg-b/
  └─ package.json
```

### 问题3：如何解决 Workspaces 中的版本冲突？

**回答：**

```bash
# ===== 场景：不同 workspace 需要不同版本的依赖 =====

packages/
  ├─ pkg-a/  (需要 lodash@4.17.21)
  └─ pkg-b/  (需要 lodash@3.10.1)

# 问题：npm 会提升哪个版本？如何处理冲突？

# ===== 解决方案1：统一版本(推荐) =====

# 1. 检查是否真的需要不同版本
npm list lodash --workspaces

# 2. 升级到兼容版本
# 修改 pkg-b 使用 lodash@4
{
  "dependencies": {
    "lodash": "^4.17.21"
  }
}

# 3. 测试确保兼容性
npm run test --workspaces

# ===== 解决方案2：使用 overrides(npm 8.3+) =====

# 根 package.json
{
  "overrides": {
    "lodash": "^4.17.21"  // 强制所有包使用此版本
  }
}

# 等同于 Yarn 的 resolutions
{
  "resolutions": {
    "lodash": "^4.17.21"
  }
}

# ===== 解决方案3：嵌套安装(不推荐) =====

# 允许 pkg-b 使用自己的版本
cd packages/pkg-b
npm install lodash@3.10.1

# 结果：
# root/node_modules/lodash@4.17.21     ← pkg-a 使用
# pkg-b/node_modules/lodash@3.10.1     ← pkg-b 使用

# ❌ 缺点：重复安装,增加体积

# ===== 解决方案4：拆分包(架构层面) =====

# 如果两个包需求差异太大,考虑：
packages/
  ├─ pkg-a/          (使用新技术栈)
  ├─ pkg-b-legacy/   (保留旧版本)
  └─ pkg-b-new/      (迁移到新版本)

# ===== 方案5：使用 pnpm(完美隔离) =====

# pnpm 天然支持不同版本
pnpm install

# 每个包有独立的 node_modules
packages/
  ├─ pkg-a/
  │  └─ node_modules/  (lodash@4.17.21)
  └─ pkg-b/
     └─ node_modules/  (lodash@3.10.1)

# 通过硬链接节省空间

# ===== 检查冲突的命令 =====

# 1. 查看依赖树
npm list lodash --workspaces --depth=999

# 2. 查看重复依赖
npm dedupe --dry-run

# 3. 审计版本差异
npm outdated --workspaces

# ===== 最佳实践 =====

1. 优先统一版本
   ✅ 在根 package.json 管理共享依赖
   {
     "dependencies": {
       "lodash": "^4.17.21",
       "express": "^4.18.0"
     }
   }

2. 使用 overrides 强制版本
   ✅ 确保传递依赖版本一致

3. 定期审计
   npm outdated --workspaces
   npm audit --workspaces

4. 考虑迁移到 pnpm
   ✅ 更好的版本隔离
   ✅ 更快的安装速度
   ✅ 更小的磁盘占用
```

### 问题4：Workspaces 如何优化 CI/CD 流程？

**回答：**

```yaml
# ===== 问题：每次 push 都要构建所有包,太慢! =====

# ===== 解决方案：增量构建(只构建变更的包) =====

# .github/workflows/ci.yml

name: CI

on: [push, pull_request]

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      packages: ${{ steps.changes.outputs.packages }}
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0  # 获取完整历史
      
      - name: Detect changed packages
        id: changes
        run: |
          # 获取变更的文件
          CHANGED_FILES=$(git diff --name-only HEAD~1 HEAD)
          
          # 提取变更的包
          CHANGED_PACKAGES=$(echo "$CHANGED_FILES" | \
            grep '^packages/' | \
            cut -d'/' -f2 | \
            sort -u | \
            jq -R -s -c 'split("\n")[:-1]')
          
          echo "packages=$CHANGED_PACKAGES" >> $GITHUB_OUTPUT
      
      - name: Show changed packages
        run: echo "Changed packages: ${{ steps.changes.outputs.packages }}"

  build:
    needs: detect-changes
    runs-on: ubuntu-latest
    if: ${{ needs.detect-changes.outputs.packages != '[]' }}
    strategy:
      matrix:
        package: ${{ fromJSON(needs.detect-changes.outputs.packages) }}
    steps:
      - uses: actions/checkout@v3
      
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Build package
        run: npm run build -w @myapp/${{ matrix.package }}
      
      - name: Test package
        run: npm run test -w @myapp/${{ matrix.package }}
      
      - name: Lint package
        run: npm run lint -w @myapp/${{ matrix.package }}

  # ===== 缓存优化 =====
  
  build-with-cache:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'  # 自动缓存 node_modules
      
      - name: Cache build outputs
        uses: actions/cache@v3
        with:
          path: |
            packages/*/dist
            packages/*/.cache
          key: build-${{ hashFiles('packages/**/package.json') }}
      
      - run: npm ci
      - run: npm run build --workspaces --if-present

# ===== 并行构建 =====

  parallel-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      
      # 并行运行所有测试
      - run: npm run test --workspaces --workspaces-parallel
      
      # 串行构建(有依赖顺序)
      - run: npm run build --workspaces

# ===== Turborepo 优化(推荐) =====

  turborepo:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - run: npm ci
      - run: npx turbo build test lint
        # Turborepo 自动:
        # ✅ 增量构建
        # ✅ 并行执行
        # ✅ 缓存结果
        # ✅ 依赖排序

# ===== 最佳实践总结 =====

1. 增量构建
   ✅ 只构建变更的包
   ✅ 减少CI时间

2. 缓存策略
   ✅ 缓存 node_modules
   ✅ 缓存构建产物
   ✅ 使用 Turborepo 远程缓存

3. 并行执行
   ✅ 测试并行运行
   ✅ lint 并行运行
   ✅ 独立任务并行

4. 依赖排序
   ✅ 构建按依赖顺序
   ✅ 使用 Turborepo 自动排序

5. 失败快速
   ✅ 先运行 lint(最快)
   ✅ 再运行 test
   ✅ 最后 build

实际效果：
- 传统方式：10分钟
- 增量构建：2-3分钟
- + 缓存：1-2分钟
- + 并行：30-60秒
```

## 📚 总结

### npm Workspaces 核心要点

```
1. 基础概念
   ✅ Monorepo 单一仓库管理
   ✅ 软链接本地包
   ✅ 共享 node_modules
   ✅ 统一依赖管理

2. 配置方式
   ✅ workspaces 字段
   ✅ 支持 glob 模式
   ✅ private: true 防止误发布

3. 命令使用
   ✅ --workspace/-w 指定包
   ✅ --workspaces 所有包
   ✅ --if-present 条件执行
   ✅ --workspaces-parallel 并行

4. 依赖管理
   ✅ 提升共享依赖
   ✅ 自动创建软链接
   ✅ npm dedupe 去重
   ✅ overrides 强制版本

5. 常见问题
   ✅ 循环依赖 → 拆分共享包
   ✅ 版本冲突 → 统一版本
   ✅ 软链接失败 → 管理员权限
   ✅ CI/CD 慢 → 增量构建
```
