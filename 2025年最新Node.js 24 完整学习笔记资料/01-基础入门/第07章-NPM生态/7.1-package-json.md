# 7.1 package.json 深度解析

> package.json 是 Node.js 项目的"身份证" —— 定义项目信息、依赖、脚本等一切

---

## 🎯 package.json 是什么？

### 核心概念

```
package.json = Node.js 项目的配置文件

类比：
package.json = 身份证
- 姓名(name)
- 年龄(version)
- 住址(repository)
- 依赖关系(dependencies)
- 技能(scripts)

作用：
✅ 描述项目信息
✅ 管理依赖包
✅ 定义脚本命令
✅ 配置发布信息
✅ 项目配置(如 ESLint、Babel等)
```

## 📝 基础字段

### 1. 必需字段

```json
{
  "name": "my-app",
  "version": "1.0.0"
}

// ===== name(项目名称) =====

// 规则：
// ✅ 小写字母、数字、连字符、下划线
// ✅ 不能以 . 或 _ 开头
// ✅ 不能包含空格
// ✅ 不能使用 Node.js 核心模块名(如 http、fs)

// 示例：
"name": "my-app"          // ✅ 正确
"name": "my-app-v2"       // ✅ 正确
"name": "my_app"          // ✅ 正确
"name": "@scope/my-app"   // ✅ 作用域包
"name": "My App"          // ❌ 包含空格
"name": "http"            // ❌ 核心模块名

// ===== version(版本号) =====

// 遵循语义化版本(SemVer)规范
// 格式：主版本号.次版本号.修订号
// 例如：1.2.3

"version": "1.0.0"        // 初始版本
"version": "1.0.1"        // 修复bug
"version": "1.1.0"        // 新增功能(向后兼容)
"version": "2.0.0"        // 不兼容的变更

// 💡 详细说明见下文 "语义化版本" 章节
```

## 📦 依赖管理

### 1. dependencies(生产依赖)

```json
{
  "dependencies": {
    "express": "^4.18.0",
    "mongoose": "~7.0.0",
    "dotenv": "16.0.3"
  }
}

// ===== 安装方式 =====

// 方式1：手动编辑 package.json,然后 npm install

// 方式2：使用命令(推荐)
npm install express
// 自动添加到 dependencies

npm install express --save
// --save 可省略,默认就是保存到 dependencies

// ===== 版本号前缀含义 =====

"express": "4.18.0"     // 精确版本,只安装 4.18.0
"express": "^4.18.0"    // 兼容版本,安装 >=4.18.0 且 <5.0.0
"express": "~4.18.0"    // 近似版本,安装 >=4.18.0 且 <4.19.0
"express": "*"          // 最新版本(不推荐,不稳定)
"express": "latest"     // 最新版本(不推荐)
```

### 2. devDependencies(开发依赖)

```json
{
  "devDependencies": {
    "nodemon": "^3.0.0",
    "jest": "^29.0.0",
    "eslint": "^8.0.0",
    "typescript": "^5.0.0"
  }
}

// ===== 特点 =====

// 只在开发时使用,生产环境不需要
// 例如：测试工具、构建工具、代码检查工具

// ===== 安装方式 =====

npm install nodemon --save-dev
// 或
npm install nodemon -D

// ===== 生产环境安装时会跳过 =====

npm install --production
// 只安装 dependencies,不安装 devDependencies
```

### 3. peerDependencies(同伴依赖)

```json
{
  "name": "my-plugin",
  "peerDependencies": {
    "react": "^18.0.0"
  }
}

// ===== 使用场景 =====

// 你开发了一个 React 插件,需要用户安装 React
// 但你不想在插件中包含 React(避免重复)
// 所以声明 peerDependencies,告诉用户需要安装 React 18

// ===== 用户安装时 =====

npm install my-plugin
// npm 会警告：需要安装 react@^18.0.0

// 用户需要手动安装：
npm install react@18
```

### 4. optionalDependencies(可选依赖)

```json
{
  "optionalDependencies": {
    "fsevents": "^2.3.0"
  }
}

// ===== 使用场景 =====

// 某些依赖只在特定平台需要
// 例如：fsevents 只在 macOS 上有用
// 安装失败不会影响整个项目

// ===== 代码中的使用 =====

try {
  const fsevents = require('fsevents');
  // 使用 fsevents
} catch (error) {
  // fsevents 不可用,使用备用方案
  console.log('fsevents 不可用,使用轮询方式');
}
```

## 🚀 脚本命令

### 1. scripts 字段

```json
{
  "scripts": {
    "start": "node server.js",
    "dev": "nodemon server.js",
    "test": "jest",
    "build": "tsc",
    "lint": "eslint ."
  }
}

// ===== 运行脚本 =====

npm run start    // 启动应用
npm run dev      // 开发模式
npm run test     // 运行测试
npm run build    // 构建项目
npm run lint     // 代码检查

// ===== 简写命令 =====

npm start        // 等同于 npm run start
npm test         // 等同于 npm run test
npm stop         // 等同于 npm run stop
npm restart      // 等同于 npm run restart
```

### 2. 生命周期钩子

```json
{
  "scripts": {
    "pretest": "echo 测试前准备",
    "test": "jest",
    "posttest": "echo 测试完成",
    
    "prebuild": "rm -rf dist",
    "build": "tsc",
    "postbuild": "cp package.json dist/"
  }
}

// ===== 执行顺序 =====

npm run test
// 1. pretest   (测试前准备)
// 2. test      (运行测试)
// 3. posttest  (测试完成)

// ===== 常用钩子 =====

// pre<script>   脚本前执行
// post<script>  脚本后执行
// prepublish    发布前执行(已废弃)
// prepare       安装后、发布前执行
// prepublishOnly 仅发布前执行
```

### 3. 传递参数

```json
{
  "scripts": {
    "start": "node server.js"
  }
}
```

```bash
# ===== 传递参数 =====

npm run start -- --port=3000
# 等同于: node server.js --port=3000

npm run start -- --port=3000 --env=production
# 等同于: node server.js --port=3000 --env=production

# 💡 注意：-- 是必需的,用于分隔 npm 参数和脚本参数
```

### 4. 实战案例

```json
{
  "scripts": {
    // 开发
    "dev": "nodemon --watch src src/index.js",
    "dev:debug": "nodemon --inspect src/index.js",
    
    // 测试
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage",
    
    // 构建
    "build": "tsc",
    "build:prod": "tsc --sourceMap false",
    
    // 代码质量
    "lint": "eslint src/**/*.js",
    "lint:fix": "eslint src/**/*.js --fix",
    "format": "prettier --write src/**/*.js",
    
    // 数据库
    "db:migrate": "knex migrate:latest",
    "db:seed": "knex seed:run",
    
    // 部署
    "deploy": "npm run build && npm run deploy:prod",
    "deploy:prod": "scp -r dist/* user@server:/var/www/",
    
    // 清理
    "clean": "rm -rf dist node_modules",
    
    // 复合命令
    "prepare": "husky install",  // Git hooks
    "prepublishOnly": "npm run test && npm run build"
  }
}
```

## 📁 文件配置

### 1. main、module、exports

```json
{
  "main": "./dist/index.js",
  "module": "./dist/index.mjs",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.js"
    },
    "./utils": {
      "import": "./dist/utils.mjs",
      "require": "./dist/utils.js"
    }
  }
}

// ===== main =====

// CommonJS 入口文件
// require('my-package') 会加载 main 指定的文件

// ===== module =====

// ES Module 入口文件
// import pkg from 'my-package' 会加载 module 指定的文件

// ===== exports(推荐,2025) =====

// 更精确的导出控制
// 支持条件导出(import/require)
// 支持子路径导出

// 用户使用：
import pkg from 'my-package';           // 加载 ./dist/index.mjs
import utils from 'my-package/utils';   // 加载 ./dist/utils.mjs
```

### 2. files、bin

```json
{
  "files": [
    "dist",
    "README.md",
    "LICENSE"
  ],
  "bin": {
    "my-cli": "./bin/cli.js"
  }
}

// ===== files =====

// 发布到 npm 时包含的文件
// 默认包含：package.json、README、LICENSE
// node_modules、.git 等会自动忽略

// ===== bin =====

// 命令行工具入口
// npm install -g my-package 后
// 用户可以直接运行: my-cli

// bin/cli.js 内容：
#!/usr/bin/env node
console.log('Hello from CLI!');
```

## ⚙️ 项目配置

### 1. engines

```json
{
  "engines": {
    "node": ">=18.0.0",
    "npm": ">=9.0.0"
  }
}

// ===== 作用 =====

// 指定项目需要的 Node.js 和 npm 版本
// 如果用户版本不匹配,会显示警告

// ===== 严格模式 =====

// .npmrc 文件中添加：
engine-strict=true
// 版本不匹配时会报错,不允许安装
```

### 2. type

```json
{
  "type": "module"
}

// ===== 默认模块系统 =====

// "type": "commonjs"  (默认)
// - .js 文件使用 CommonJS (require/module.exports)
// - .mjs 文件使用 ES Module (import/export)

// "type": "module"
// - .js 文件使用 ES Module (import/export)
// - .cjs 文件使用 CommonJS (require/module.exports)

// 💡 2025 年推荐使用 "type": "module"
```

### 3. config

```json
{
  "config": {
    "port": 3000,
    "db_host": "localhost"
  },
  "scripts": {
    "start": "node server.js"
  }
}
```

```javascript
// server.js 中访问配置

const port = process.env.npm_package_config_port;
// 3000

const dbHost = process.env.npm_package_config_db_host;
// localhost

console.log(`服务器运行在端口 ${port}`);
```

## 🔧 高级配置字段

### 1. workspaces(Monorepo)

```json
{
  "name": "my-monorepo",
  "workspaces": [
    "packages/*",
    "apps/*"
  ]
}

// ===== 目录结构 =====

my-monorepo/
├── package.json
├── packages/
│   ├── utils/
│   │   └── package.json
│   └── core/
│       └── package.json
└── apps/
    └── web/
        └── package.json

// ===== 好处 =====

// ✅ 统一管理依赖
// ✅ 共享 node_modules
// ✅ 跨包开发更方便
// ✅ 一次性安装所有依赖

npm install  // 安装所有 workspace 的依赖

// ===== 在子包中引用其他子包 =====

// packages/core/package.json
{
  "dependencies": {
    "@mycompany/utils": "*"  // 引用同 monorepo 中的 utils 包
  }
}
```

### 2. publishConfig

```json
{
  "publishConfig": {
    "access": "public",
    "registry": "https://registry.npmjs.org/",
    "tag": "latest"
  }
}

// ===== 字段说明 =====

// access: "public" | "restricted"
//   - public: 公开包(作用域包必须设置)
//   - restricted: 私有包(需要付费账号)

// registry: 发布到的仓库地址
//   - npm 官方: https://registry.npmjs.org/
//   - 私有仓库: https://registry.mycompany.com/

// tag: 发布标签
//   - latest: 默认标签(npm install 默认安装)
//   - next: 测试版本
//   - beta: 测试版本
```

### 3. browserslist

```json
{
  "browserslist": [
    "> 1%",
    "last 2 versions",
    "not dead"
  ]
}

// ===== 作用 =====

// 指定项目支持的浏览器范围
// 用于 Babel、Autoprefixer 等工具

// ===== 常用配置 =====

"> 1%"                 // 全球使用率 > 1% 的浏览器
"last 2 versions"      // 每个浏览器的最后 2 个版本
"not dead"             // 官方仍在维护的浏览器
"Chrome >= 60"         // Chrome 60 及以上
"ie 11"                // 兼容 IE 11
```

### 4. overrides(依赖覆盖)

```json
{
  "overrides": {
    "lodash": "4.17.21"
  }
}

// ===== 使用场景 =====

// 某个依赖包的子依赖有安全漏洞
// 你想强制所有依赖使用指定版本

// 例如：
// express 依赖 lodash@4.17.15
// 但 lodash@4.17.15 有安全漏洞
// 你可以用 overrides 强制升级到 4.17.21

// ===== 效果 =====

// 所有包的 lodash 依赖都会被替换为 4.17.21
```

### 5. resolutions(Yarn)

```json
{
  "resolutions": {
    "lodash": "4.17.21",
    "**/lodash": "4.17.21"
  }
}

// ===== Yarn 专用 =====

// 功能同 overrides,但只在 Yarn 中有效
// npm v8.3+ 支持 overrides
```

## 🎯 常见问题与解决

### 问题1：package-lock.json 是什么？要提交到 Git 吗？

**解答：**

```
package-lock.json 的作用：

1. 锁定依赖版本
   - package.json: "express": "^4.18.0"(允许 4.x.x)
   - package-lock.json: 精确锁定为 4.18.2

2. 加速安装
   - 记录依赖树结构
   - 记录下载地址
   - 跳过版本解析

3. 保证一致性
   - 团队成员安装相同版本
   - CI/CD 环境一致

是否提交到 Git？
✅ 应用项目(app): 必须提交
❌ npm 包(library): 建议不提交(.gitignore)

原因：
- 应用项目需要版本一致性
- npm 包的使用者会根据自己的 package.json 安装
```

### 问题2：dependencies 和 devDependencies 如何选择？

**解答：**

```javascript
// ===== dependencies =====

// 生产环境需要的依赖
{
  "dependencies": {
    "express": "^4.18.0",      // Web 框架
    "mongoose": "^7.0.0",      // 数据库
    "dotenv": "^16.0.3",       // 环境变量
    "winston": "^3.8.2"        // 日志
  }
}

// 判断标准：
// ✅ 代码中直接 require/import 的包
// ✅ 生产环境运行需要的包
// ✅ 部署到服务器需要的包

// ===== devDependencies =====

{
  "devDependencies": {
    "nodemon": "^3.0.0",       // 开发工具
    "jest": "^29.0.0",         // 测试工具
    "eslint": "^8.0.0",        // 代码检查
    "typescript": "^5.0.0",    // TypeScript 编译器
    "@types/node": "^20.0.0"   // TypeScript 类型定义
  }
}

// 判断标准：
// ✅ 只在开发时使用
// ✅ 测试、构建、代码检查工具
// ✅ 生产环境不需要

// ===== 实际案例 =====

// 错误：把 nodemon 放在 dependencies
{
  "dependencies": {
    "nodemon": "^3.0.0"  // ❌ 错误！生产不需要
  }
}

// 正确：放在 devDependencies
{
  "devDependencies": {
    "nodemon": "^3.0.0"  // ✅ 正确
  }
}
```

### 问题3：^ 和 ~ 的区别是什么？

**解答：**

```
版本号格式：主版本号.次版本号.修订号
示例：1.2.3

^ (插入符号)
"express": "^4.18.2"
✅ 允许：4.18.2, 4.18.3, 4.19.0, 4.99.99
❌ 不允许：5.0.0, 3.x.x
规则：不改变主版本号

~ (波浪号)
"express": "~4.18.2"
✅ 允许：4.18.2, 4.18.3, 4.18.99
❌ 不允许：4.19.0, 5.0.0
规则：不改变主版本号和次版本号

无前缀(精确版本)
"express": "4.18.2"
✅ 允许：4.18.2
❌ 不允许：4.18.3, 任何其他版本
规则：必须完全匹配

*(通配符)
"express": "*"
✅ 允许：任何版本
规则：总是安装最新版本(不推荐)

推荐使用：
✅ ^ (大多数情况)
✅ 无前缀(需要精确控制)
❌ ~ (不常用)
❌ * (不推荐)
```

### 问题4：如何创建 CLI 工具？

**解答：**

```json
// ===== 步骤1：package.json 配置 =====

{
  "name": "my-cli",
  "version": "1.0.0",
  "bin": {
    "mycli": "./bin/cli.js"
  },
  "files": [
    "bin"
```

### 问题5：如何处理依赖冲突？

**解答：**

```bash
# ===== 问题场景 =====

# 项目依赖 A@1.0.0 和 B@2.0.0
# A 依赖 lodash@4.17.15
# B 依赖 lodash@4.17.21
# 导致 lodash 版本冲突

# ===== 解决方案1：使用 overrides(npm 8.3+) =====

{
  "overrides": {
    "lodash": "4.17.21"
  }
}

# 强制所有依赖使用 lodash@4.17.21

# ===== 解决方案2：使用 resolutions(Yarn) =====

{
  "resolutions": {
    "lodash": "4.17.21"
  }
}

# ===== 解决方案3：npm dedupe =====

npm dedupe

# 自动去重依赖树中的重复包

# ===== 解决方案4：查看依赖树 =====

npm ls lodash

# 输出:
# my-app@1.0.0
# ├─┬ package-a@1.0.0
# │ └── lodash@4.17.15
# └─┬ package-b@2.0.0
#   └── lodash@4.17.21

# 找出哪些包依赖了 lodash
```

## 🎓 常见面试题

### 问题1：package.json 中的 ^ 和 ~ 有什么区别？

**回答：**

```
^ (插入符号) - 兼容更新
"express": "^4.18.2"
- 允许：4.18.2 → 4.99.99
- 不允许：5.0.0
- 规则：不改变主版本号

~ (波浪号) - 近似更新
"express": "~4.18.2"
- 允许：4.18.2 → 4.18.99
- 不允许：4.19.0
- 规则：不改变主版本号和次版本号

实际使用：
✅ ^ 更常用(允许向后兼容的更新)
✅ ~ 更保守(只允许 bug 修复)

推荐：
对于第三方库：使用 ^ (获得 bug 修复和新功能)
对于核心依赖：使用 ~ 或精确版本(更稳定)
```

### 问题2：dependencies 和 devDependencies 有什么区别？什么时候用哪个？

**回答：**

```
核心区别：

dependencies(生产依赖)
✅ 应用运行时需要
✅ 部署到服务器需要
✅ npm install --production 会安装
✅ 示例：express, mongoose, dotenv

devDependencies(开发依赖)
✅ 只在开发时需要
✅ 部署到服务器不需要
✅ npm install --production 不会安装
✅ 示例：jest, eslint, nodemon, typescript

判断标准：
问：代码中有 require('xxx') 或 import xxx 吗？
答：是 → dependencies
答：否 → devDependencies

问：生产环境需要这个包吗？
答：是 → dependencies
答：否 → devDependencies

特殊情况：
TypeScript 编译器 → devDependencies(因为生产运行的是 JS)
@types/* 类型定义 → devDependencies
构建工具(webpack, babel) → devDependencies
```

### 问题3：package-lock.json 的作用是什么？

**回答：**

```
核心作用：

1. 锁定精确版本
   package.json:      "express": "^4.18.0"(允许 4.x.x)
   package-lock.json: 锁定为 4.18.2(精确版本)

2. 保证一致性
   ✅ 团队成员安装相同版本
   ✅ CI/CD 环境一致
   ✅ 避免"在我电脑上能跑"问题

3. 加速安装
   ✅ 记录依赖树结构
   ✅ 记录包的下载地址
   ✅ 跳过版本解析过程

4. 安全性
   ✅ 记录包的 integrity(校验和)
   ✅ 防止包被篡改

是否提交到 Git？
✅ 应用项目(app): 必须提交
   - 需要版本一致性
   - 避免生产环境意外升级

❌ npm 包(library): 建议不提交
   - 使用者会根据自己的 package.json 安装
   - 避免强制使用者使用特定版本

常见问题：
Q: package-lock.json 和 package.json 冲突怎么办？
A: 删除 node_modules 和 package-lock.json,重新 npm install

Q: 如何更新 package-lock.json？
A: npm update 或 npm install <package>@latest
```

## 📚 总结

### package.json 核心知识点

```
1. 基础字段
   ✅ name, version(必需)
   ✅ description, keywords(SEO)
   ✅ author, license(法律)

2. 依赖管理
   ✅ dependencies(生产)
   ✅ devDependencies(开发)
   ✅ peerDependencies(插件)
   ✅ optionalDependencies(可选)

3. 脚本命令
   ✅ scripts(自定义命令)
   ✅ 生命周期钩子(pre/post)
   ✅ 参数传递(npm run <script> -- --args)

4. 高级配置
   ✅ workspaces(Monorepo)
   ✅ overrides(依赖覆盖)
   ✅ exports(精确导出)
   ✅ publishConfig(发布配置)

5. 版本符号
   ✅ ^: 兼容更新(推荐)
   ✅ ~: 近似更新
   ✅ 无前缀: 精确版本
   ❌ *: 任意版本(不推荐)
```
