# 7.5 npm 安全审计

> 保护项目免受依赖包漏洞攻击

## 🎯 为什么需要安全审计？

### 安全风险

```
依赖包的安全问题 = 你的应用的安全问题

真实案例：
2018年：event-stream 包被植入恶意代码
2021年：ua-parser-js 包被劫持
2022年：node-ipc 包恶意破坏文件

风险来源：
❌ 依赖包有已知漏洞
❌ 传递依赖(依赖的依赖)有漏洞
❌ 包被恶意篡改
❌ 供应链攻击(maintainer账号被盗)

后果：
• 数据泄露
• 服务器被控制
• 用户信息被窃取
• 勒索软件
• 挖矿木马
```

## 🔍 npm audit 内置审计

### 1. 基础使用

```bash
# ===== 运行安全审计 =====

npm audit

# 输出示例：
# found 3 vulnerabilities (1 low, 1 moderate, 1 high)
#
# High            Prototype Pollution
# Package         lodash
# Dependency of   express
# Path            express > lodash
# More info       https://npmjs.com/advisories/1065

# ===== 严重程度级别 =====

# Critical (严重)  - 立即修复
# High (高危)      - 尽快修复
# Moderate (中危)  - 计划修复
# Low (低危)       - 可选修复

# ===== 详细报告 =====

npm audit --json
# JSON格式,适合CI集成

npm audit --parseable
# 易于解析的格式
```

### 2. 自动修复

```bash
# ===== 自动修复(安全) =====

npm audit fix

# 输出：
# fixed 2 of 3 vulnerabilities in 1234 scanned packages
# 1 vulnerability required manual review

# 💡 只会升级到兼容版本(不跨主版本)
# lodash@4.17.15 → lodash@4.17.21 (修复漏洞)
# express@4.17.0 → express@4.17.3 (✅ 兼容)
# express@4.17.0 → express@5.0.0 (❌ 不自动升级)

# ===== 强制修复(可能不兼容) =====

npm audit fix --force

# ⚠️ 警告：可能会：
# - 跨主版本升级(不兼容)
# - 破坏现有功能
# - 修改 package-lock.json

# 建议：
# 1. 先备份代码
# 2. 查看将要做的修改：npm audit fix --dry-run
# 3. 运行测试确保功能正常
```

### 3. 只显示特定级别

```bash
# ===== 只显示高危及以上 =====

npm audit --audit-level=high

# 级别：
# low, moderate, high, critical

# ===== CI 中使用 =====

# 如果有高危漏洞,返回非零退出码
npm audit --audit-level=high
# 有漏洞: exit 1 (构建失败)
# 无漏洞: exit 0 (构建通过)
```

## 🛡️ snyk - 专业安全工具

### 1. 安装与使用

```bash
# ===== 安装 Snyk =====

npm install -g snyk

# ===== 认证 =====

snyk auth
# 打开浏览器登录 https://snyk.io

# ===== 扫描项目 =====

cd my-project
snyk test

# 输出：
# Testing my-project...
#
# ✗ High severity vulnerability found in lodash
#   Affected versions: <4.17.21
#   Fixed in: 4.17.21
#   Path: express > lodash
#   Recommendation: Upgrade to lodash@4.17.21

# ===== 生成报告 =====

snyk test --json > snyk-report.json

# ===== 监控项目(持续监控) =====

snyk monitor
# 将项目添加到 Snyk 仪表板
# 有新漏洞时会邮件通知
```

### 2. 自动修复

```bash
# ===== 自动修复漏洞 =====

snyk wizard
# 交互式修复向导

# 步骤：
# 1. 显示漏洞列表
# 2. 对每个漏洞选择：
#    - Upgrade: 升级到安全版本
#    - Patch: 应用补丁
#    - Ignore: 暂时忽略
#    - Review: 稍后处理

# ===== 一键修复 =====

snyk fix
# 自动应用所有推荐的修复
```


### 案例3：自定义安全策略

```javascript
// ===== scripts/security-check.js =====

const { execSync } = require('child_process');
const fs = require('fs');

// 运行 npm audit
console.log('运行 npm audit...');
const auditResult = execSync('npm audit --json', { encoding: 'utf8' });
const audit = JSON.parse(auditResult);

// 分析结果
const { vulnerabilities } = audit.metadata;
const critical = vulnerabilities.critical || 0;
const high = vulnerabilities.high || 0;
const moderate = vulnerabilities.moderate || 0;
const low = vulnerabilities.low || 0;

console.log('\n=== 安全审计结果 ===');
console.log(`严重: ${critical}`);
console.log(`高危: ${high}`);
console.log(`中危: ${moderate}`);
console.log(`低危: ${low}`);

// 自定义策略
const policy = {
  critical: 0,   // 不允许任何严重漏洞
  high: 0,       // 不允许任何高危漏洞
  moderate: 5,   // 最多5个中危漏洞
  low: 10        // 最多10个低危漏洞
};

let failed = false;

if (critical > policy.critical) {
  console.error(`\n❌ 严重漏洞数 (${critical}) 超过策略限制 (${policy.critical})`);
  failed = true;
}

if (high > policy.high) {
  console.error(`\n❌ 高危漏洞数 (${high}) 超过策略限制 (${policy.high})`);
  failed = true;
}

if (moderate > policy.moderate) {
  console.warn(`\n⚠️  中危漏洞数 (${moderate}) 超过策略限制 (${policy.moderate})`);
}

if (low > policy.low) {
  console.warn(`\n⚠️  低危漏洞数 (${low}) 超过策略限制 (${policy.low})`);
}

// 生成报告
const report = {
  timestamp: new Date().toISOString(),
  vulnerabilities,
  policy,
  passed: !failed
};

fs.writeFileSync('security-report.json', JSON.stringify(report, null, 2));

if (failed) {
  console.error('\n❌ 安全审计失败!');
  process.exit(1);
} else {
  console.log('\n✅ 安全审计通过!');
}
```

```json
// package.json
{
  "scripts": {
    "security": "node scripts/security-check.js",
    "precommit": "npm run security"
  }
}
```

## 🚨 应急响应流程

```
发现安全漏洞时的处理流程：

1. 评估严重程度
   ├─ Critical/High → 立即处理
   └─ Moderate/Low → 计划修复

2. 确认影响范围
   ├─ 生产环境是否受影响？
   ├─ 哪些功能受影响？
   └─ 用户数据是否暴露？

3. 修复漏洞
   ├─ 升级到安全版本
   ├─ 应用补丁
   └─ 或临时禁用功能

4. 测试验证
   ├─ 运行测试套件
   ├─ 手动测试关键功能
   └─ 再次运行安全扫描

5. 部署修复
   ├─ 紧急发布热修复
   ├─ 通知用户(如需要)
   └─ 监控系统

6. 事后总结
   ├─ 记录事件
   ├─ 改进流程
   └─ 更新安全策略
```

## 🎓 常见面试题

### 问题1：npm audit 的工作原理是什么？如何使用？

**回答：**

```
npm audit 工作原理：

1. 数据来源
   - npm官方维护的漏洞数据库(npm advisory database)
   - 来自GitHub Security Advisory、Snyk等安全机构
   - 社区提交的漏洞报告

2. 检测流程
   步骤1：读取package-lock.json或package.json
   步骤2：提取所有依赖包及版本信息(包括传递依赖)
   步骤3：向npm registry发送请求查询漏洞
   步骤4：匹配依赖版本与已知漏洞
   步骤5：生成报告(按严重程度分类)

3. 严重程度分级
   - Critical(严重): 立即修复,可能导致系统被攻破
   - High(高危): 尽快修复,存在重大安全风险
   - Moderate(中危): 计划修复,有一定安全风险
   - Low(低危): 可选修复,风险较小

4. 使用方法
   # 基础审计
   npm audit
   
   # 自动修复(安全)
   npm audit fix
   
   # 强制修复(可能不兼容)
   npm audit fix --force
   
   # 只显示特定级别
   npm audit --audit-level=high
   
   # JSON格式(CI集成)
   npm audit --json

5. 限制
   ❌ 只能检测已知漏洞
   ❌ 需要联网查询
   ❌ 可能有误报(false positive)
   ❌ 不能检测代码逻辑漏洞
```

### 问题2：如何在CI/CD中集成安全检查？

**回答：**

```yaml

name: Security Check

    branches: [main]

      - name: Install dependencies

      # 2. npm audit
      - name: Run npm audit
        run: npm audit --audit-level=moderate
        continue-on-error: true

      - name: Run Snyk
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

      - name: Check licenses

          npm install -g license-checker

        run: |

          npm audit --json > audit-report.json

        uses: actions/upload-artifact@v3
        if: always()

          name: security-report
          path: audit-report.json

# ===== 集成策略 =====

   ✅ 手动触发

2. 失败策略
   ✅ High/Critical → 阻止合并
   ✅ Moderate → 警告但允许合并
   ✅ Low → 仅记录

3. 通知机制
   ✅ GitHub PR评论
   ✅ 邮件通知
   ✅ 企业微信/钉钉

   ✅ 保存为artifact

   ✅ 生成趋势报告
```

### 问题3：什么是供应链攻击？如何防护？

**回答：**

```
供应链攻击定义：
攻击者通过入侵npm包的维护者账号或依赖包,
植入恶意代码,影响所有使用该包的项目。

真实案例：

1. event-stream事件(2018)
   - 流行的Node.js流处理库
   - 维护者将权限转给攻击者
   - 植入恶意代码窃取比特币钱包
   - 影响200万次下载/周

2. ua-parser-js劫持(2021)
   - 维护者账号被盗
   - 发布恶意版本(0.7.29, 0.8.0, 1.0.0)
   - 包含挖矿木马和密码窃取器
   - 每周900万次下载

3. node-ipc破坏性更新(2022)
   - 维护者故意添加恶意代码
   - 检测俄罗斯和白俄罗斯IP
   - 删除用户文件

防护措施：

1. 依赖审查
   ✅ 检查包的下载量和受欢迎程度
   ✅ 查看维护者信息和项目活跃度
   ✅ 阅读最近的issues和PR
   ✅ 检查依赖包数量(少即是多)

2. 版本锁定
   ✅ 使用package-lock.json
   ✅ 关键依赖使用精确版本
   {
     "dependencies": {
       "critical-lib": "1.0.0"  // 不用^或~
     }
   }

3. 安全扫描
   ✅ npm audit自动检测
   ✅ Snyk持续监控
   ✅ GitHub Dependabot alerts
   ✅ Socket.dev实时威胁检测

4. 代码审计
   ✅ 审查依赖包源代码(关键包)
   ✅ 使用子资源完整性(SRI)
   ✅ 监控package.json变化

5. 隔离运行
   ✅ 使用沙箱环境(Docker/VM)
   ✅ 限制包的权限(deno权限系统)
   ✅ 使用私有registry代理

6. 及时更新
   ✅ 定期更新依赖
   ✅ 订阅安全公告
   ✅ 设置自动化更新(Dependabot)

7. 多因素认证
   ✅ npm账号开启2FA
   ✅ GitHub账号开启2FA
   ✅ 使用Token而非密码

实际防护代码：

# .npmrc配置
audit=true
audit-level=high

# package.json
{
  "scripts": {
    "preinstall": "npm run security:check",
    "security:check": "npm audit && license-checker"
  }
}

# 使用Socket.dev监控
npx socket npm install
```

### 问题4：如何处理无法修复的安全漏洞？

**回答：**

```bash
# ===== 场景：依赖包有漏洞但无法升级 =====

情况1：没有安全版本
- 包的维护者已停止维护
- 安全版本尚未发布

情况2：升级会破坏兼容性
- 新版本有重大变更
- 需要大量代码修改

处理方案：

# 方案1：使用补丁(patch-package)

# 1. 安装patch-package
npm install patch-package --save-dev

# 2. 手动修复漏洞
# 直接编辑node_modules中的代码

# 3. 生成补丁
npx patch-package package-name

# 4. 应用补丁(package.json)
{
  "scripts": {
    "postinstall": "patch-package"
  }
}

# 方案2：Fork并维护

# 1. Fork有漏洞的包
git clone https://github.com/original/package
cd package

# 2. 修复漏洞
# 修改代码...
git commit -am "fix: security vulnerability"

# 3. 使用Fork版本
{
  "dependencies": {
    "package": "git+https://github.com/yourname/package.git#fixed"
  }
}

# 方案3：寻找替代包

# 1. 搜索功能相似的包
npm search alternative-package

# 2. 评估替代包
# - 下载量
# - 维护状态
# - 安全记录

# 3. 迁移代码
npm uninstall vulnerable-package
npm install alternative-package

# 方案4：临时忽略(不推荐)

# 创建.npmrc
audit=false

# 或忽略特定漏洞
npm audit --json | grep advisory-id
# 记录在案并计划修复

# 方案5：隔离风险

# 1. 不在关键路径使用
# 2. 添加输入验证
# 3. 限制包的权限
# 4. 监控异常行为

# 方案6：联系维护者

# 1. 在GitHub提issue
# 2. 提交PR修复
# 3. 联系npm security team

决策流程：

1. 评估漏洞严重性
   Critical/High → 必须处理
   Moderate/Low → 可暂时接受

2. 评估修复成本
   简单 → 立即修复
   复杂 → 制定计划

3. 选择方案
   有安全版本 → 升级
   无安全版本 → patch或fork
   无法修复 → 替换或隔离

4. 记录决策
   # security-exceptions.md
   ## 已知漏洞
   - 包名: vulnerable-package
   - 漏洞ID: CVE-2023-1234
   - 严重性: Moderate
   - 原因: 升级会破坏兼容性
   - 缓解措施: 添加输入验证
   - 计划: 2025-Q2迁移到替代包
```
