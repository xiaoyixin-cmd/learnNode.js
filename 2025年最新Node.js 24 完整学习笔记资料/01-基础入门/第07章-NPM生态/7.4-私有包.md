# 7.4 私有包管理与发布

> 企业内部包管理,保护代码不公开

---

## 🎯 什么是私有包？

### 核心概念

```
私有包 = 不公开的 npm 包,只在企业/团队内部使用

公开包 vs 私有包：

公开包(免费)：
✅ 任何人都能安装
✅ 代码开源
✅ 发布到 npmjs.com
例如：express, lodash

私有包(收费/自建)：
✅ 只有授权用户能安装
✅ 代码不公开
✅ 发布到私有仓库
例如：企业内部组件库、工具包

使用场景：
• 企业内部组件库
• 业务代码封装
• 敏感代码保护
• 付费闭源产品
```

## 🔐 方案1：npm官方私有包

### 1. 注册与登录

```bash
# ===== 注册 npm 账号 =====

# 访问 https://www.npmjs.com/signup
# 注册账号

# ===== 登录 =====

npm login
# Username: your-username
# Password: ***
# Email: your@email.com

# 验证登录
npm whoami
# your-username

# ===== 升级到付费计划 =====

# 访问 https://www.npmjs.com/settings/billing
# 选择计划并付费
```

### 2. 创建私有包

```bash
# ===== 初始化项目 =====

mkdir my-private-pkg
cd my-private-pkg
npm init --scope=@your-username
# 或
npm init --scope=@your-org

# package.json
{
  "name": "@your-username/my-private-pkg",  // 必须带作用域
  "version": "1.0.0",
  "description": "My private package",
  "main": "index.js",
  "private": false,  // ❌ 不要设为 true
  "publishConfig": {
    "access": "restricted"  // ✅ 私有包
  }
}

# ===== 编写代码 =====

// index.js
module.exports = {
  secret() {
    return '这是私有代码';
  }
};

# ===== 发布 =====

npm publish --access=restricted
# 或者(如果设置了 publishConfig)
npm publish

# 输出：
# + @your-username/my-private-pkg@1.0.0
```

### 3. 使用私有包

```bash
# ===== 安装私有包 =====

# 其他项目中
npm login  # 先登录
npm install @your-username/my-private-pkg

# 使用
const pkg = require('@your-username/my-private-pkg');
console.log(pkg.secret());  // 这是私有代码

# ===== 团队协作 =====

# 1. 创建组织(团队)
# 访问 https://www.npmjs.com/org/create

# 2. 邀请成员
# 在组织设置中添加成员

# 3. 发布到组织
{
  "name": "@mycompany/my-pkg"
}
npm publish

# 团队成员登录后就能安装
npm install @mycompany/my-pkg
```

```javascript
// ===== scripts/check-version.js =====

const { version } = require('../package.json');
const https = require('https');

// 检查版本是否已存在
https.get(`http://npm.company.com/@mycompany/my-pkg`, (res) => {
  let data = '';
  
  res.on('data', chunk => data += chunk);
  res.on('end', () => {
    const pkg = JSON.parse(data);
    
    if (pkg.versions && pkg.versions[version]) {
      console.error(`❌ 版本 ${version} 已存在!`);
      process.exit(1);
    }
    
    console.log(`✅ 版本 ${version} 可以发布`);
  });
}).on('error', () => {
  // 包不存在,可以发布
  console.log(`✅ 新包,可以发布`);
});

// ===== scripts/check-files.js =====

const fs = require('fs');
const { files } = require('../package.json');

if (!files || files.length === 0) {
  console.error('❌ package.json 缺少 files 字段!');
  process.exit(1);
}

// 检查文件是否存在
for (const file of files) {
  if (!fs.existsSync(file)) {
    console.error(`❌ 文件不存在: ${file}`);
    process.exit(1);
  }
}

console.log('✅ 所有文件检查通过');
```

### 3. 包使用统计

```javascript
// ===== Verdaccio 插件：下载统计 =====

// verdaccio-plugin-download-stats.js
class DownloadStatsPlugin {
  constructor(config, options) {
    this.config = config;
    this.stats = new Map();
  }
  
  // 拦截下载请求
  middleware() {
    return (req, res, next) => {
      if (req.method === 'GET' && req.url.includes('.tgz')) {
        const pkg = req.url.split('/')[1];
        
        // 记录下载
        const count = this.stats.get(pkg) || 0;
        this.stats.set(pkg, count + 1);
        
        console.log(`📦 ${pkg} 被下载 ${count + 1} 次`);
      }
      next();
    };
  }
}

module.exports = DownloadStatsPlugin;
```

## 🎓 常见面试题

### 问题1：私有npm包有哪些实现方案？各有什么优缺点？

**回答：**

```
三种主要方案：

1. npm官方私有包
   优点：
   ✅ 官方支持,稳定可靠
   ✅ 与公开包使用体验一致
   ✅ 无需维护服务器
   ✅ 与GitHub无缝集成
   
   缺点：
   ❌ 需要付费($7/用户/月)
   ❌ 依赖外部服务
   ❌ 在国内可能较慢
   
   适用：小团队,对价格不敏感,不想自建服务器

2. 自建Registry(Verdaccio)
   优点：
   ✅ 完全自主控制
   ✅ 一次性成本(服务器)
   ✅ 可内网部署
   ✅ 无包数量限制
   ✅ 可以代理npm官方
   
   缺点：
   ❌ 需要维护服务器
   ❌ 需要配置备份和高可用
   ❌ 需要运维能力
   
   适用：中大型企业,对安全要求高,有运维团队

3. Git仓库
   优点：
   ✅ 完全免费
   ✅ 利用现有Git基础设施
   ✅ 版本控制集成
   
   缺点：
   ❌ 不支持SemVer版本范围
   ❌ 安装速度慢
   ❌ 需要Git权限
   ❌ 没有依赖管理功能
   
   适用：小团队,临时方案,预算有限

实际推荐：
- 5人以下 → Git仓库
- 5-20人 → npm官方或Verdaccio
- 20人以上 → Verdaccio自建
- 高安全要求 → Verdaccio内网部署
```

### 问题2：如何在Verdaccio中实现细粒度的权限控制？

**回答：**

```yaml
# ===== config.yaml 权限配置 =====

# 基础配置
packages:
  # 1. 按作用域控制
  '@frontend/*':
    access: frontend-team    # 只有前端团队能访问
    publish: frontend-lead   # 只有前端负责人能发布
    
  '@backend/*':
    access: backend-team
    publish: backend-lead
    
  # 2. 按包名模式控制
  '@company/core-*':
    access: $authenticated   # 所有登录用户能访问
    publish: admin           # 只有管理员能发布
    unpublish: admin         # 只有管理员能删除
    
  '@company/experimental-*':
    access: developers       # 开发者能访问
    publish: developers      # 开发者能发布
    
  # 3. 严格控制的包
  '@company/security':
    access: security-team    # 只有安全团队
    publish: security-lead
    proxy: false             # 不代理到npm官方

# 用户组定义(需要auth插件支持)
auth:
  htpasswd:
    file: ./htpasswd

# 使用LDAP集成企业账号(需要verdaccio-ldap插件)
auth:
  ldap:
    url: "ldap://ldap.company.com"
    baseDN: "dc=company,dc=com"
    groupDN: "ou=groups,dc=company,dc=com"

实现细粒度控制的方法：

1. 使用作用域(@scope)区分包类型
2. 配置不同的访问和发布权限
3. 集成LDAP/AD实现企业账号统一认证
4. 使用自定义auth插件实现复杂权限逻辑
5. 结合CI/CD使用不同的token权限
```

### 问题4：如何实现私有包的版本管理和发布流程？

**回答：**

```bash
# ===== 标准发布流程 =====

# 1. 开发分支工作
git checkout -b feature/new-feature
# 开发代码...
git commit -am "feat: add new feature"

# 2. 创建PR,代码审查
git push origin feature/new-feature
# 在GitHub/GitLab创建PR

# 3. 合并到main
git checkout main
git merge feature/new-feature

# 4. 更新版本号
npm version patch  # 1.0.0 -> 1.0.1 (bug修复)
npm version minor  # 1.0.1 -> 1.1.0 (新功能)
npm version major  # 1.1.0 -> 2.0.0 (破坏性变更)

# 5. 推送tag
git push --tags

# 6. CI自动发布(GitLab CI)
# .gitlab-ci.yml
publish:
  stage: deploy
  script:
    - npm run build
    - npm publish --registry http://npm.company.com
  only:
    - tags  # 只在推送tag时执行

# ===== 使用Lerna管理多包版本 =====

# 1. 初始化Lerna
npx lerna init

# 2. 发布新版本
npx lerna version
# 交互式选择版本号,自动创建tag

# 3. 发布到私有registry
npx lerna publish --registry http://npm.company.com

# ===== 版本策略 =====

1. Semantic Versioning (SemVer)
   - Major: 破坏性变更
   - Minor: 新功能(向后兼容)
   - Patch: Bug修复

2. Pre-release版本
   - 1.0.0-alpha.0 (内部测试)
   - 1.0.0-beta.0 (公开测试)
   - 1.0.0-rc.0 (发布候选)
   
3. Tag命名规范
   - v1.0.0 (正式版本)
   - v1.0.0-alpha.0 (预发布)
   
4. 发布检查清单
   ✅ 所有测试通过
   ✅ 代码审查完成
   ✅ 文档已更新
   ✅ CHANGELOG已更新
   ✅ 版本号符合规范
   ✅ 没有敏感信息
```
