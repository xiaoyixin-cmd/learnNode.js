# 10.6 CI集成（GitHub Actions实战）

> GitHub Actions配置、自动化测试、覆盖率报告、多环境测试、缓存优化、并行测试

---

## 📖 知识点概述

### 核心概念

**持续集成（CI）**通过自动化测试确保每次代码提交的质量，GitHub Actions是最流行的CI/CD平台，与GitHub深度集成，配置简单、功能强大。

### CI/CD流程

```
┌──────────────────────────────────────────────────────────┐
│              GitHub Actions CI 流程                       │
├──────────────────────────────────────────────────────────┤
│                                                            │
│  1. 触发条件                                               │
│     ├─ Push到main/develop分支                            │
│     ├─ 创建Pull Request                                  │
│     ├─ 定时任务（Cron）                                   │
│     └─ 手动触发                                           │
│                                                            │
│  2. 准备环境                                               │
│     ├─ 启动Ubuntu/Windows/macOS                          │
│     ├─ 安装Node.js                                        │
│     ├─ 缓存node_modules                                   │
│     └─ 安装依赖                                           │
│                                                            │
│  3. 代码检查                                               │
│     ├─ ESLint代码规范                                     │
│     ├─ Prettier格式检查                                   │
│     └─ TypeScript类型检查                                 │
│                                                            │
│  4. 自动化测试                                             │
│     ├─ 单元测试（Jest/Vitest）                            │
│     ├─ 集成测试（Supertest）                              │
│     ├─ E2E测试（Playwright）                              │
│     └─ 覆盖率报告                                         │
│                                                            │
│  5. 构建应用                                               │
│     ├─ npm run build                                      │
│     ├─ Docker镜像                                         │
│     └─ 验证构建产物                                       │
│                                                            │
│  6. 部署（可选）                                           │
│     ├─ 发布到Vercel/Netlify                              │
│     ├─ 推送Docker镜像                                     │
│     └─ 通知团队                                           │
│                                                            │
└──────────────────────────────────────────────────────────┘
```

### GitHub Actions核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **Workflow** | 工作流文件 | `.github/workflows/test.yml` |
| **Job** | 任务（可并行） | `test`, `build`, `deploy` |
| **Step** | 步骤（顺序执行） | `checkout`, `install`, `test` |
| **Runner** | 执行环境 | `ubuntu-latest`, `windows-latest` |
| **Action** | 可复用操作 | `actions/checkout@v4` |

## 🚀 使用指南

### 实践1：基础测试工作流

```yaml
# .github/workflows/test.yml
name: Test

# 💡 触发条件
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

# 💡 任务定义
jobs:
  test:
    name: Run Tests
    runs-on: ubuntu-latest  # 运行环境
    
    # 💡 步骤
    steps:
      # 1. 检出代码
      - name: Checkout code
        uses: actions/checkout@v4
      
      # 2. 设置Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      # 3. 安装依赖
      - name: Install dependencies
        run: npm ci  # npm ci比npm install更快更稳定
      
      # 4. 运行测试
      - name: Run tests
        run: npm test
      
      # 5. 上传覆盖率报告
      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage/coverage-final.json
          flags: unittests
          name: codecov-umbrella
```

### 实践2：缓存优化（加速构建）

```yaml
# .github/workflows/test-with-cache.yml
name: Test with Cache

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          # 💡 自动缓存node_modules
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      # 💡 手动缓存（高级用法）
      - name: Cache Jest cache
        uses: actions/cache@v3
        with:
          path: .jest-cache
          key: ${{ runner.os }}-jest-${{ hashFiles('**/package-lock.json') }}
      
      - name: Run tests
        run: npm test -- --cacheDirectory=.jest-cache

# 效果：
# - 首次运行：3-5分钟
# - 缓存后：30-60秒（5-10倍提升）
```

### 实践3：多Node.js版本测试（矩阵策略）

```yaml
# .github/workflows/matrix.yml
name: Multi-Version Test

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    
    # 💡 矩阵策略：测试多个版本
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
        # 总共9个组合（3个OS × 3个Node版本）
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests
        run: npm test
      
      - name: Display environment
        run: |
          echo "OS: ${{ matrix.os }}"
          echo "Node: ${{ matrix.node-version }}"
          node --version
          npm --version

# 💡 并行运行，总时间 = 最慢的那个
```

### 实践5：E2E测试（Playwright）

```yaml
# .github/workflows/e2e.yml
name: E2E Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  e2e:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      # 💡 安装Playwright浏览器
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps
      
      # 💡 启动应用
      - name: Build application
        run: npm run build
      
      - name: Start application
        run: npm start &
        env:
          NODE_ENV: test
          PORT: 3000
      
      # 等待应用启动
      - name: Wait for app
        run: npx wait-on http://localhost:3000 --timeout 60000
      
      # 💡 运行E2E测试
      - name: Run Playwright tests
        run: npx playwright test
      
      # 💡 上传测试报告和失败截图
      - name: Upload Playwright Report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
      
      - name: Upload test results
        uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-screenshots
          path: test-results/
```

### 实践8：PR自动评论

```yaml
# .github/workflows/pr-comment.yml
name: PR Comment

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  test-and-comment:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests
        id: test
        run: |
          npm test -- --coverage --json --outputFile=test-results.json
          echo "exitCode=$?" >> $GITHUB_OUTPUT
      
      # 💡 解析测试结果
      - name: Parse test results
        if: always()
        run: |
          cat test-results.json | jq -r '
            "## 🧪 测试结果\n\n" +
            "- ✅ 通过: \(.numPassedTests)\n" +
            "- ❌ 失败: \(.numFailedTests)\n" +
            "- ⏭️  跳过: \(.numPendingTests)\n" +
            "- 📊 覆盖率: " + (.coverageMap | length | tostring) + "\n"
          ' > comment.md
      
      # 💡 评论到PR
      - name: Comment PR
        uses: thollander/actions-comment-pull-request@v2
        if: always()
        with:
          filePath: comment.md
          comment_tag: test-results

# 效果：每次提交PR自动显示测试结果
```

## 🔧 错误处理

### 高频错误1：依赖安装失败

**现象：**
```
npm ERR! code ERESOLVE
npm ERR! ERESOLVE could not resolve
```

**解决方案：**
```yaml
# ✅ 使用npm ci而不是npm install
- name: Install dependencies
  run: npm ci  # 更快、更稳定、可重现

# 💡 如果package-lock.json有问题
- name: Install dependencies
  run: npm install --legacy-peer-deps

# 💡 清理缓存
- name: Clear npm cache
  run: npm cache clean --force
```

### 高频错误2：测试超时

**现象：**
```
Error: Timeout - Async callback was not invoked within the 5000 ms timeout
```

**解决方案：**
```yaml
# ✅ 增加超时时间
- name: Run tests
  run: npm test
  timeout-minutes: 10  # 默认360分钟

# ✅ Jest/Vitest配置
# jest.config.js
export default {
  testTimeout: 30000  // 30秒
};

# ✅ 单个测试增加超时
it('slow test', async () => {
  // ...
}, 30000);
```

### 高频错误3：环境变量缺失

**现象：**
```
Error: DATABASE_URL is not defined
```

**解决方案：**
```yaml
# ✅ 在workflow中设置环境变量
- name: Run tests
  env:
    DATABASE_URL: postgresql://test:test@localhost:5432/test_db
    JWT_SECRET: test-secret
    NODE_ENV: test
  run: npm test

# ✅ 使用GitHub Secrets（敏感信息）
- name: Deploy
  env:
    API_KEY: ${{ secrets.API_KEY }}
    DATABASE_PASSWORD: ${{ secrets.DB_PASSWORD }}
  run: npm run deploy

# 💡 在GitHub仓库设置Secrets：
# Settings → Secrets and variables → Actions → New repository secret
```

## 📚 应用场景

### 场景1：完整的CI/CD流程

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # 阶段1：代码质量检查
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint        # ESLint
      - run: npm run format:check  # Prettier
      - run: npm run type-check    # TypeScript
  
  # 阶段2：测试
  test:
    needs: quality
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        # ... 配置 ...
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test -- --coverage
      - uses: codecov/codecov-action@v4
  
  # 阶段3：构建
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-artifacts
          path: dist/
  
  # 阶段4：部署（仅main分支）
  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
        with:
          name: build-artifacts
          path: dist/
      - name: Deploy to production
        env:
          VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
        run: npx vercel --prod --token=$VERCEL_TOKEN
```

## 📋 总结

### 核心要点

1. **自动化**：每次提交自动运行测试
2. **并行化**：拆分任务加速构建
3. **缓存**：缓存依赖提升速度
4. **矩阵**：多版本多平台测试

### 实现关键

1. **触发条件**：push、PR、定时、手动
2. **缓存策略**：node_modules、Jest缓存
3. **并行测试**：多Job并行运行
4. **覆盖率报告**：Codecov、Coveralls

### 应用价值

- **快速反馈**：2-5分钟完成CI
- **质量保障**：每次提交都测试
- **团队协作**：PR自动检查
- **部署信心**：测试通过才部署

### 面试重点

**Q1：为什么使用npm ci而不是npm install？**
答：npm ci更快、更稳定、可重现。它会删除node_modules后根据package-lock.json精确安装，避免版本不一致。

**Q2：如何加速GitHub Actions？**
答：1) 缓存node_modules 2) 并行运行Job 3) 使用矩阵策略 4) 只在需要时运行（路径过滤） 5) 使用自托管Runner。

**Q3：CI/CD的最佳实践是什么？**
答：1) 快速反馈（<5分钟） 2) 稳定的测试 3) 清晰的错误信息 4) 自动部署 5) 监控和告警。

### 注意事项

⚠️ **敏感信息**：使用GitHub Secrets存储
⚠️ **超时设置**：合理设置timeout-minutes
⚠️ **并发限制**：控制并发Job数量
⚠️ **成本控制**：公共仓库免费，私有仓库有限额
