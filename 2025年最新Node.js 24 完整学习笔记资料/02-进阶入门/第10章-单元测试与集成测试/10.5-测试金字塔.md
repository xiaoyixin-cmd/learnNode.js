## 📖 知识点概述

### 核心概念

**测试金字塔**是Mike Cohn提出的软件测试策略模型，指导团队如何合理分配不同类型的测试，实现最佳的投资回报比。

### 测试金字塔结构

```
┌──────────────────────────────────────────────────────────┐
│              测试金字塔（Test Pyramid）                    │
├──────────────────────────────────────────────────────────┤
│                                                            │
│                    E2E 测试                                │
│                   (5-10%)                                  │
│              ├─ Playwright/Cypress                        │
│              ├─ 真实浏览器                                 │
│              ├─ 完整用户流程                               │
│              ├─ 速度：慢 (秒级)                            │
│              ├─ 成本：高 💰💰💰                             │
│              └─ 稳定性：低 ⚠️                              │
│                                                            │
│         ─────────────────────────                         │
│                                                            │
│              集成测试 / API测试                            │
│                  (20-30%)                                  │
│           ├─ Supertest/Postman                            │
│           ├─ 组件集成                                     │
│           ├─ 数据库交互                                   │
│           ├─ 速度：中 (100ms级)                           │
│           ├─ 成本：中 💰💰                                 │
│           └─ 稳定性：中 ✅                                 │
│                                                            │
│     ───────────────────────────────────                   │
│                                                            │
│            单元测试 (60-70%)                               │
│         ├─ Jest/Vitest                                    │
│         ├─ 函数/类测试                                    │
│         ├─ Mock依赖                                       │
│         ├─ 速度：快 (ms级)                                │
│         ├─ 成本：低 💰                                    │
│         └─ 稳定性：高 ✅✅✅                               │
│                                                            │
└──────────────────────────────────────────────────────────┘

原则：越往下，数量越多，速度越快，成本越低
```

### 测试类型对比

| 维度 | 单元测试 | 集成测试 | E2E测试 |
|------|---------|---------|---------|
| **范围** | 单个函数/类 | 多个模块 | 完整系统 |
| **依赖** | Mock | 部分真实 | 全部真实 |
| **速度** | 极快 (1-10ms) | 中 (10-100ms) | 慢 (1-10s) |
| **稳定性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **调试难度** | 简单 | 中等 | 困难 |
| **维护成本** | 低 | 中 | 高 |
| **信心** | 低 | 中 | 高 |

### 为什么需要测试金字塔？

```
❌ 反模式：冰淇淋锥（Ice Cream Cone）
    大量E2E测试
    少量集成测试
    极少单元测试
    
    问题：
    - 测试慢（每次运行10分钟+）
    - 不稳定（E2E易失败）
    - 调试难（难以定位问题）
    - 成本高（维护困难）

✅ 推荐：金字塔
    大量单元测试
    适量集成测试
    少量E2E测试
    
    优点：
    - 测试快（秒级完成）
    - 稳定（单元测试不易失败）
    - 易调试（精确定位）
    - 成本低（维护简单）
```

```javascript
// utils/validators.js
export function isValidEmail(email) {
  if (typeof email !== 'string') return false;
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

export function isValidAge(age) {
  return Number.isInteger(age) && age >= 0 && age <= 150;
}

export function sanitizeInput(input) {
  if (typeof input !== 'string') return '';
  return input.trim().replace(/[<>]/g, '');
}

// utils/validators.test.js
describe('Validators (单元测试)', () => {
  describe('isValidEmail', () => {
    // 💡 单元测试：快速、独立、无依赖
    it('valid emails', () => {
      expect(isValidEmail('test@example.com')).toBe(true);
      expect(isValidEmail('user+tag@domain.co.uk')).toBe(true);
    });
    
    it('invalid emails', () => {
      expect(isValidEmail('invalid')).toBe(false);
      expect(isValidEmail('@example.com')).toBe(false);
      expect(isValidEmail('test@')).toBe(false);
    });
    
    it('edge cases', () => {
      expect(isValidEmail(null)).toBe(false);
      expect(isValidEmail(undefined)).toBe(false);
      expect(isValidEmail(123)).toBe(false);
    });
  });
  
  describe('isValidAge', () => {
    it('valid ages', () => {
      expect(isValidAge(0)).toBe(true);
      expect(isValidAge(25)).toBe(true);
      expect(isValidAge(150)).toBe(true);
    });
    
    it('invalid ages', () => {
      expect(isValidAge(-1)).toBe(false);
      expect(isValidAge(151)).toBe(false);
      expect(isValidAge(25.5)).toBe(false);
      expect(isValidAge('25')).toBe(false);
    });
  });

    it('trims whitespace', () => {
      expect(sanitizeInput('  hello  ')).toBe('hello');
    });
    
    it('removes dangerous characters', () => {
      expect(sanitizeInput('<script>alert(1)</script>')).toBe('scriptalert(1)/script');
    });
    
    it('handles non-strings', () => {
      expect(sanitizeInput(null)).toBe('');
      expect(sanitizeInput(123)).toBe('');
    });
  });
});
```

### 实践2：集成测试（中层）

```javascript
// services/user.service.js
import { isValidEmail, isValidAge } from '../utils/validators.js';

export class UserService {
  constructor(db) {
    this.db = db;
  }
  
  async createUser(data) {
    // 💡 集成多个模块：验证 + 数据库
    if (!isValidEmail(data.email)) {
      throw new Error('Invalid email');
    }
    
    if (!isValidAge(data.age)) {
      throw new Error('Invalid age');
    }
    
    const existingUser = await this.db.findUserByEmail(data.email);
    if (existingUser) {
      throw new Error('Email already exists');
    }
    
    return await this.db.createUser(data);
  }
  
  async getUserById(id) {
    const user = await this.db.findUserById(id);
    
    if (!user) {
      throw new Error('User not found');
    }
    
    return user;
  }
}

// services/user.service.test.js
import { createTestDatabase } from '../test-utils/db.js';

describe('UserService (集成测试)', () => {
  let db;
  let userService;
  
  beforeAll(async () => {
    // 💡 使用真实数据库（或内存数据库）
    db = await createTestDatabase();
  });
  
  afterEach(async () => {
    await db.clearUsers();
  });
  
  afterAll(async () => {
    await db.close();
  });
  
  beforeEach(() => {
    userService = new UserService(db);
  });
  
  it('创建用户', async () => {
    const userData = {
      username: 'john',
      email: 'john@example.com',
      age: 25
    };
    
    const user = await userService.createUser(userData);
    
    expect(user.id).toBeDefined();
    expect(user.username).toBe('john');
    
    // 💡 验证数据库中确实存在
    const dbUser = await db.findUserById(user.id);
    expect(dbUser.email).toBe('john@example.com');
  });
  
  it('重复邮箱抛出错误', async () => {
    await userService.createUser({
      username: 'alice',
      email: 'alice@example.com',
      age: 30
    });
    
    await expect(
      userService.createUser({
        username: 'alice2',
        email: 'alice@example.com',
        age: 25
      })
    ).rejects.toThrow('Email already exists');
  });
  
  it('无效邮箱抛出错误', async () => {
    await expect(
      userService.createUser({
        username: 'bob',
        email: 'invalid-email',
        age: 25
      })
    ).rejects.toThrow('Invalid email');
  });
  
  it('获取不存在的用户', async () => {
    await expect(
      userService.getUserById(999)
    ).rejects.toThrow('User not found');
  });
});

// 运行：中等速度 (每个测试 10-100ms)
```

```javascript
// TDD流程：红 → 绿 → 重构

// 第一步：写失败的测试（红）
describe('sum', () => {
  it('adds two numbers', () => {
    expect(sum(1, 2)).toBe(3);
  });
});

// 此时运行测试会失败（因为sum函数不存在）

// 第二步：写最少代码让测试通过（绿）
function sum(a, b) {
  return a + b;
}

// 测试通过

// 第三步：重构代码（保持测试通过）
function sum(...numbers) {
  return numbers.reduce((acc, n) => acc + n, 0);
}

// 第四步：添加更多测试
it('adds multiple numbers', () => {
  expect(sum(1, 2, 3, 4)).toBe(10);
});

it('handles empty input', () => {
  expect(sum()).toBe(0);
});

// TDD优点：
// 1. 设计优先（API设计先于实现）
// 2. 100%覆盖率（代码都是为了通过测试而写）
// 3. 快速反馈（立即知道是否正确）
// 4. 重构信心（测试保护代码）
```

## 🔧 错误处理

### 高频错误1：过度依赖E2E测试

**现象：**
```
项目测试结构：
- 10个单元测试
- 5个集成测试
- 50个E2E测试

问题：
- 测试运行时间：30分钟
- 失败率高（E2E不稳定）
- 调试困难（不知道哪里出错）
```

**解决方案：**
```javascript
// ✅ 金字塔结构
- 500个单元测试 (运行时间：10秒)
- 100个集成测试 (运行时间：1分钟)
- 10个E2E测试 (运行时间：2分钟)

总运行时间：3分钟

// 💡 E2E只测试关键流程
const criticalFlows = [
  '用户注册登录',
  '购买商品下单',
  '支付流程',
  '核心业务流程'
];

// 其他功能用单元测试和集成测试覆盖
```

### 高频错误2：覆盖率误区

**现象：**
```javascript
// ❌ 为了覆盖率而写无意义的测试
it('getter returns value', () => {
  const user = new User('john');
  expect(user.getName()).toBe('john');  // 毫无价值
});

// 覆盖率：100% ✅
// 实际价值：0 ❌
```

**解决方案：**
```javascript
// ✅ 测试真正的逻辑
it('validates username length', () => {
  expect(() => new User('ab')).toThrow('Username too short');
  expect(() => new User('a'.repeat(51))).toThrow('Username too long');
  expect(new User('john').isValid()).toBe(true);
});

// 💡 覆盖率目标
// - 80%是良好平衡点
// - 关注未覆盖的关键代码
// - 不要为了100%而浪费时间
```

### 高频错误3：测试不独立

**现象：**
```javascript
// ❌ 测试相互依赖
let userId;

it('creates user', async () => {
  const user = await createUser({ username: 'john' });
  userId = user.id;  // 保存给下个测试用
  expect(user.id).toBeDefined();
});

it('updates user', async () => {
  await updateUser(userId, { age: 25 });  // 依赖上个测试
  // ...
});

// 问题：
// - 第一个测试失败，第二个也失败
// - 无法单独运行测试
// - 调试困难
```

**解决方案：**
```javascript
// ✅ 每个测试独立
describe('User operations', () => {
  let testUser;
  
  beforeEach(async () => {
    // 💡 每个测试前准备数据
    testUser = await createUser({ username: 'john' });
  });
  
  afterEach(async () => {
    // 清理
    await deleteUser(testUser.id);
  });
  
  it('creates user', async () => {
    expect(testUser.id).toBeDefined();
  });
  
  it('updates user', async () => {
    await updateUser(testUser.id, { age: 25 });
    const updated = await getUser(testUser.id);
    expect(updated.age).toBe(25);
  });
  
  // 两个测试完全独立，可以单独运行
});
```

## 📋 总结

### 核心要点

1. **70/20/10原则**：单元/集成/E2E测试比例
2. **速度优先**：快速反馈是关键
3. **覆盖率平衡**：80%是最佳点
4. **测试独立**：每个测试可单独运行

### 实现关键

1. **单元测试**：快速、独立、Mock依赖
2. **集成测试**：关键流程、真实依赖
3. **E2E测试**：核心业务、完整流程
4. **TDD**：测试先行、设计优先

### 应用价值

- **开发效率**：快速反馈（秒级）
- **重构信心**：测试保护代码
- **文档作用**：测试即规格说明
- **质量保障**：自动化验证

### 面试重点

**Q1：为什么要遵循测试金字塔？**
答：因为单元测试快、稳定、易维护。大量E2E测试会导致测试慢、不稳定、难调试。金字塔结构实现最佳投资回报比。

**Q2：测试覆盖率应该达到多少？**
答：80-90%是最佳平衡点。100%覆盖率成本高且收益递减。应关注核心业务逻辑和复杂代码的覆盖率。

**Q3：什么是TDD？优缺点是什么？**
答：测试驱动开发，先写测试再写代码。优点是设计优先、100%覆盖、快速反馈。缺点是初期投入大、学习曲线陡。

### 注意事项

⚠️ **避免反模式**：不要大量E2E测试
⚠️ **覆盖率误区**：不要盲目追求100%
⚠️ **测试独立**：每个测试可单独运行
⚠️ **合理Mock**：单元测试Mock，集成测试真实
