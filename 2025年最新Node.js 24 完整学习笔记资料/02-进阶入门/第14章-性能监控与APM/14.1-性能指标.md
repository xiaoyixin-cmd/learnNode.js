# 14.1 性能指标

## 知识点概述

### 核心概念

性能指标是衡量 Node.js 应用运行效率和用户体验的量化标准。通过监控关键指标，你可以：

- **发现性能瓶颈**：识别响应慢、内存泄漏、CPU 占用高等问题
- **评估系统容量**：了解当前系统能承载多少并发用户和请求
- **优化资源配置**：根据指标数据调整服务器规格、数据库连接池等
- **保障用户体验**：确保页面加载快、接口响应及时、操作流畅

### 核心性能指标分类

#### 1. **响应时间类指标**
- **RT (Response Time)**：单次请求从发起到接收完整响应的耗时
- **平均响应时间**：所有请求响应时间的平均值
- **P50/P90/P99**：50%/90%/99% 的请求响应时间低于此值（更能反映真实用户体验）

#### 2. **吞吐量类指标**
- **QPS (Queries Per Second)**：每秒处理的查询/请求数
- **TPS (Transactions Per Second)**：每秒完成的事务数
- **并发连接数**：同时在线的 WebSocket/TCP 连接数

#### 3. **系统资源类指标**
- **CPU 使用率**：进程占用的 CPU 百分比
- **内存使用量**：堆内存、RSS、外部内存占用情况
- **事件循环延迟 (Event Loop Lag)**：事件循环处理的延迟时间

#### 4. **业务指标**
- **错误率**：HTTP 4xx/5xx 状态码占比
- **可用性 (Availability)**：服务正常运行的时间百分比（如 99.9% 可用性）
- **数据库慢查询数**：超过阈值的 SQL 查询次数

### 典型应用场景

- **电商大促**：监控 QPS、P99 响应时间，确保抢购高峰期系统稳定
- **SaaS 平台**：跟踪事件循环延迟和内存增长，预防内存泄漏
- **金融交易系统**：严格控制 P99 < 100ms，保障交易实时性

## 使用指南

### 第一步：理解

#### 为什么需要监控性能指标？

假设你的电商 API 某天突然收到用户投诉"加载很慢"。如果没有性能指标：
- ❌ 你不知道是偶发还是持续问题
- ❌ 不知道是数据库慢、第三方接口超时还是内存不足
- ❌ 只能"盲猜"问题原因，浪费大量时间

有了性能指标监控：
- ✅ 发现 P99 响应时间从 200ms 飙升到 3s
- ✅ 定位到数据库慢查询数激增
- ✅ 快速优化 SQL 索引，问题解决

#### 核心指标选择原则

| 指标         | 适用场景                     | 监控阈值示例      |
|--------------|------------------------------|-------------------|
| QPS          | 评估系统容量、扩容决策       | > 5000 时告警     |
| P99 响应时间 | 保障用户体验（99%用户满意）  | > 500ms 时告警    |
| 错误率       | 发现系统异常                 | > 1% 时告警       |
| 内存使用量   | 预防内存泄漏                 | > 1.5GB 时告警    |
| Event Loop Lag | 发现阻塞事件循环的代码     | > 50ms 时告警     |

💡 **技巧**：P99 比平均值更重要！假设平均响应时间 100ms，但 P99 达到 5s，说明有 1% 用户体验极差。

### 第三步：实践

#### 方案1：使用 `prom-client` 手动采集指标

**`src/metrics/collector.js`** - 创建指标采集器

```javascript
const client = require('prom-client');

// 💡 创建指标注册表（所有指标都会注册到这里）
const register = new client.Registry();

// 1️⃣ 收集默认系统指标（CPU、内存、事件循环延迟等）
client.collectDefaultMetrics({ register });

// 2️⃣ 创建自定义指标：HTTP 请求计数器
const httpRequestCounter = new client.Counter({
  name: 'http_requests_total',           // 指标名称（符合 Prometheus 命名规范）
  help: 'Total number of HTTP requests', // 指标说明
  labelNames: ['method', 'path', 'status'], // 标签（用于分组统计）
  registers: [register]                  // 注册到指标表
});

// 3️⃣ 创建自定义指标：HTTP 请求响应时间直方图
// 💡 直方图自动计算 P50/P90/P99
const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'path', 'status'],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 2, 5], // 时间分桶（单位：秒）
  registers: [register]
});

// 4️⃣ 创建自定义指标：数据库查询响应时间
const dbQueryDuration = new client.Histogram({
  name: 'db_query_duration_seconds',
  help: 'Duration of database queries in seconds',
  labelNames: ['operation', 'table'],
  buckets: [0.001, 0.01, 0.05, 0.1, 0.5, 1],
  registers: [register]
});

// 5️⃣ 创建自定义指标：当前活跃连接数（Gauge 可增可减）
const activeConnections = new client.Gauge({
  name: 'active_connections',
  help: 'Number of active connections',
  registers: [register]
});

module.exports = {
  register,
  httpRequestCounter,
  httpRequestDuration,
  dbQueryDuration,
  activeConnections
};
```

**`src/metrics/middleware.js`** - 创建 Express 中间件

```javascript
const responseTime = require('response-time');
const {
  httpRequestCounter,
  httpRequestDuration,
  activeConnections
} = require('./collector');

// 💡 中间件：记录每个请求的指标
function metricsMiddleware(req, res, next) {
  // ⚠️ 排除 /metrics 路径本身，避免污染数据
  if (req.path === '/metrics') {
    return next();
  }

  // 记录请求开始时间
  const start = Date.now();

  // 连接数 +1
  activeConnections.inc();

  // 监听响应结束事件
  res.on('finish', () => {
    // 计算响应时间（单位：秒）
    const duration = (Date.now() - start) / 1000;

    // 规范化路径（避免 /user/123、/user/456 被视为不同路径）
    const path = normalizePath(req.path);

    // 记录请求计数
    httpRequestCounter.inc({
      method: req.method,
      path: path,
      status: res.statusCode
    });

    // 记录响应时间
    httpRequestDuration.observe(
      {
        method: req.method,
        path: path,
        status: res.statusCode
      },
      duration
    );

    // 连接数 -1
    activeConnections.dec();
  });

  next();
}

// 💡 路径规范化：将 /user/123 转为 /user/:id
function normalizePath(path) {
  return path
    .replace(/\/\d+/g, '/:id')              // /123 -> /:id
    .replace(/\/[a-f0-9-]{36}/g, '/:uuid'); // /uuid -> /:uuid
}

module.exports = { metricsMiddleware };
```

## 3个高频错误

### 错误1：事件循环被阻塞，`nodejs_eventloop_lag` 飙升

**现象**：监控显示 `nodejs_eventloop_lag_seconds` 从 0.001s 增长到 0.5s+

**原因**：代码中存在同步阻塞操作（如大量 JSON.parse、同步文件读取）

**解决方案**：

```javascript
// ❌ 错误：同步读取大文件阻塞事件循环
const data = fs.readFileSync('/path/to/large-file.json');
const parsed = JSON.parse(data); // 如果文件很大，会阻塞数百毫秒

// ✅ 正确：使用异步操作
const data = await fs.promises.readFile('/path/to/large-file.json');
const parsed = JSON.parse(data); // 仍然会阻塞，需进一步优化

// ✅ 更好：使用 Worker Threads 处理 CPU 密集型任务
const { Worker } = require('worker_threads');

function parseInWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./parser-worker.js', { workerData: data });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

const parsed = await parseInWorker(data); // 不阻塞主线程
```

**`parser-worker.js`**：
```javascript
const { parentPort, workerData } = require('worker_threads');
const parsed = JSON.parse(workerData);
parentPort.postMessage(parsed);
```

### 错误2：内存泄漏导致 `nodejs_heap_size_used_bytes` 持续增长

**现象**：监控显示堆内存从 100MB 逐渐增长到 1.5GB，最终进程崩溃

**原因**：全局变量积累、事件监听器未移除、闭包引用未释放

**解决方案**：

```javascript
// ❌ 错误：全局缓存无限增长
const cache = {};
app.get('/api/user/:id', (req, res) => {
  cache[req.params.id] = userData; // 缓存永远不清理
  res.json(userData);
});

// ✅ 正确：使用 LRU 缓存（自动淘汰旧数据）
const LRU = require('lru-cache');
const cache = new LRU({ max: 500, maxAge: 60000 }); // 最多 500 条，60 秒过期

app.get('/api/user/:id', (req, res) => {
  cache.set(req.params.id, userData);
  res.json(userData);
});
```

```javascript
// ❌ 错误：事件监听器未移除
const eventEmitter = new EventEmitter();
app.get('/api/stream', (req, res) => {
  eventEmitter.on('data', (data) => { // 每次请求都添加监听器
    res.write(data);
  });
});

// ✅ 正确：使用 once 或手动移除
app.get('/api/stream', (req, res) => {
  const handler = (data) => res.write(data);
  eventEmitter.once('data', handler); // 触发一次后自动移除
  
  req.on('close', () => {
    eventEmitter.removeListener('data', handler); // 请求关闭时移除
  });
});
```

**监控内存泄漏**：

### 错误3：指标标签（Label）过多导致 Prometheus 查询缓慢

**现象**：Prometheus 查询超时，内存占用过高

**原因**：标签值太多（如 `user_id` 有 100 万个不同值），导致指标时间序列爆炸

**解决方案**：

```javascript
// ❌ 错误：使用高基数（high cardinality）标签
const userRequestCounter = new client.Counter({
  name: 'user_requests_total',
  labelNames: ['user_id'] // 如果有 100 万用户，会生成 100 万个时间序列
});

userRequestCounter.inc({ user_id: req.user.id });

// ✅ 正确：使用低基数标签（如用户类型、地区）
const userRequestCounter = new client.Counter({
  name: 'user_requests_total',
  labelNames: ['user_type', 'region'] // 只有几十个不同值
});

userRequestCounter.inc({ 
  user_type: req.user.isPremium ? 'premium' : 'free',
  region: req.user.region 
});
```

⚠️ **注意**：标签的唯一值数量 × 指标数量 = 总时间序列数。建议单个指标的时间序列数 < 10,000。

## 应用场景

### 场景1：电商大促实时监控

**问题背景**：
某电商平台双十一期间需要处理平时 10 倍的流量。运维团队需要实时监控系统容量，决策是否扩容。

**技术选型理由**：
- **Prometheus + Grafana**：开源免费、社区活跃、支持 PromQL 灵活查询
- **`prom-client`**：官方推荐库、支持直方图自动计算分位数

**完整实现**：

```javascript
// metrics/business-metrics.js
const client = require('prom-client');
const register = new client.Registry();

// 1️⃣ 订单创建成功率
const orderSuccessRate = new client.Counter({
  name: 'order_created_total',
  help: 'Total number of orders created',
  labelNames: ['status'], // 'success' | 'failed'
  registers: [register]
});

// 2️⃣ 支付响应时间
const paymentDuration = new client.Histogram({
  name: 'payment_duration_seconds',
  help: 'Payment gateway response time',
  labelNames: ['gateway'], // 'alipay' | 'wechat'
  buckets: [0.1, 0.5, 1, 2, 5, 10],
  registers: [register]
});

// 3️⃣ 库存剩余量（Gauge 可增可减）
const stockGauge = new client.Gauge({
  name: 'product_stock',
  help: 'Current product stock',
  labelNames: ['product_id'],
  registers: [register]
});

module.exports = { orderSuccessRate, paymentDuration, stockGauge, register };
```

```javascript
// routes/order.js
const { orderSuccessRate, paymentDuration, stockGauge } = require('../metrics/business-metrics');

app.post('/api/orders', async (req, res) => {
  try {
    // 1️⃣ 检查库存
    const stock = await getStock(req.body.productId);
    if (stock < 1) {
      orderSuccessRate.inc({ status: 'failed' });
      return res.status(400).json({ error: '库存不足' });
    }

    // 2️⃣ 调用支付接口（记录响应时间）
    const paymentEnd = paymentDuration.startTimer({ gateway: 'alipay' });
    await callPaymentGateway(req.body);
    paymentEnd();

    // 3️⃣ 扣减库存并更新指标
    await decreaseStock(req.body.productId, 1);
    stockGauge.dec({ product_id: req.body.productId }, 1);

    // 4️⃣ 记录成功
    orderSuccessRate.inc({ status: 'success' });
    res.json({ orderId: 'ORDER123' });

  } catch (error) {
    orderSuccessRate.inc({ status: 'failed' });
    res.status(500).json({ error: error.message });
  }
});
```

### 场景2：SaaS 平台内存泄漏排查

**问题背景**：
某 SaaS 平台每周需重启一次服务器，否则内存占用从 500MB 增长到 2GB，最终触发 OOM。

**技术选型理由**：
- **`prom-client` 默认指标**：自动采集堆内存、RSS、外部内存
- **自定义 Gauge**：跟踪业务对象数量（如 WebSocket 连接数、缓存大小）

**完整实现**：

```javascript
// metrics/memory-metrics.js
const client = require('prom-client');
const register = new client.Registry();

// 1️⃣ 收集默认内存指标
client.collectDefaultMetrics({ register, timeout: 5000 });

// 2️⃣ 自定义指标：WebSocket 连接数
const wsConnectionsGauge = new client.Gauge({
  name: 'websocket_connections',
  help: 'Number of active WebSocket connections',
  registers: [register]
});
```

### 性能优化：指标采集的性能影响

#### 优化前：每次请求都调用 `Date.now()`

```javascript
// ❌ 性能开销较大（每秒 1 万次请求会调用 2 万次 Date.now）
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    histogram.observe(duration);
  });
  next();
});
```

**压测结果**（wrk 测试）：
- 无监控：**15,000 QPS**
- 加监控后：**13,500 QPS**（下降 10%）

#### 优化后：使用 `process.hrtime.bigint()`

```javascript
// ✅ 使用高精度时间戳（性能损耗更小）
app.use((req, res, next) => {
  const start = process.hrtime.bigint();
  res.on('finish', () => {
    const duration = Number(process.hrtime.bigint() - start) / 1e9; // 转为秒
    histogram.observe(duration);
  });
  next();
});
```

**压测结果**：
- 加监控后：**14,700 QPS**（仅下降 2%）

#### 优化关键指标对比

| 指标                  | 优化前     | 优化后     | 提升     |
|-----------------------|------------|------------|----------|
| QPS                   | 13,500     | 14,700     | +8.9%    |
| 平均延迟              | 1.2ms      | 1.05ms     | -12.5%   |
| P99 延迟              | 5.8ms      | 4.2ms      | -27.6%   |

## 总结

### ✅ 核心要点

1. **必须监控的 4 类指标**：响应时间（P99）、吞吐量（QPS）、系统资源（内存/CPU）、错误率
2. **选对工具**：`prom-client` 是 Node.js 监控的事实标准，配合 Prometheus + Grafana 使用
3. **避免高基数标签**：标签值不要超过 1 万个，否则会拖垮 Prometheus

### 🔑 实现关键

1. 在所有路由前注册 `metricsMiddleware`
2. 使用 `histogram.startTimer()` 自动记录耗时
3. 为业务关键操作（支付、订单、库存）创建自定义指标
4. 暴露 `/metrics` 端点供 Prometheus 抓取

### 💎 应用价值

- **预防故障**：P99 超阈值时自动告警，避免用户投诉后才发现问题
- **优化性能**：通过 Event Loop Lag 发现阻塞代码，提升并发能力
- **辅助扩容**：根据 QPS 趋势提前扩容，避免流量峰值时系统崩溃

### 🎯 面试重点

- **Q1**：为什么 P99 比平均响应时间更重要？  
  **A**：平均值会被少数异常值拉高/拉低，P99 更能反映 99% 用户的真实体验。

- **Q2**：如何避免指标采集影响性能？  
  **A**：使用 `process.hrtime.bigint()`、避免高基数标签、采样监控（如只监控 10% 请求）。

- **Q3**：Event Loop Lag 是什么？如何优化？  
  **A**：事件循环处理延迟，由同步阻塞代码引起。优化方法：使用异步 API、Worker Threads 处理 CPU 密集任务。

### ⚠️ 注意事项

1. **不要监控敏感数据**：避免在标签中包含用户 ID、手机号等（可能泄露隐私）
2. **定期清理指标**：如果指标不再使用，调用 `register.removeSingleMetric()` 清理
3. **监控 `/metrics` 本身**：排除 `/metrics` 路径，避免污染数据

### 🚀 进阶思考（非必需）

- **分布式追踪**：结合 OpenTelemetry 实现跨服务的请求链路追踪
- **实时告警**：配置 Alertmanager 将告警推送到钉钉、Slack、PagerDuty
- **成本优化**：使用 Prometheus Remote Write 将数据写入低成本存储（如 Thanos、VictoriaMetrics）

## 🤔 常见问题（FAQ）

### Q1：为什么要监控 P99 而不是平均值？

**A**：平均值容易被少数极端值影响，无法反映真实用户体验。

**举例说明**：
```
100 个请求的响应时间：
- 99 个请求：50ms
- 1 个请求：5000ms

平均值 = (99×50 + 5000) / 100 = 99.5ms  ← 看起来很好
P99 = 5000ms  ← 暴露了 1% 用户体验极差的问题
```

**结论**：P99 更能反映"最差 1% 用户"的体验，对于保障 SLA 更有意义。

---

### Q2：Prometheus 的 Counter 和 Gauge 有什么区别？

**A**：
- **Counter（计数器）**：只增不减，适合累计值（如请求总数）
- **Gauge（仪表）**：可增可减，适合瞬时值（如当前内存使用量）

**使用场景**：
```javascript
// Counter：HTTP 请求总数（只增）
const httpRequestsTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests'
});
httpRequestsTotal.inc();  // 每次请求 +1

// Gauge：当前活跃连接数（可增可减）
const activeConnections = new client.Gauge({
  name: 'active_connections',
  help: 'Current active connections'
});
activeConnections.inc();  // 连接建立时 +1
activeConnections.dec();  // 连接关闭时 -1
```

### Q3：如何避免监控指标采集影响应用性能？

**A**：采用以下优化策略：

1. **使用高精度时间戳**：
```javascript
// ❌ 避免频繁调用 Date.now()
const start = Date.now();

// ✅ 使用 process.hrtime.bigint()（性能更好）
const start = process.hrtime.bigint();
```

2. **限制标签基数**（Label Cardinality）：
```javascript
// ❌ 错误：用户 ID 作为标签（可能有百万级不同值）
counter.inc({ user_id: userId });

// ✅ 正确：使用低基数标签
counter.inc({ user_type: isPremium ? 'premium' : 'free' });
```

3. **采样监控**（只监控 10% 请求）：
```javascript
app.use((req, res, next) => {
  if (Math.random() < 0.1) {  // 10% 采样率
    // 记录监控指标
  }
  next();
});
```

### Q4：Event Loop Lag 持续升高，如何排查？

**A**：Event Loop Lag 表示事件循环被阻塞，常见原因：

**排查步骤**：
1. **检查同步阻塞代码**：
```javascript
// ❌ 同步读取大文件
const data = fs.readFileSync('/large-file.json');

// ✅ 改为异步
const data = await fs.promises.readFile('/large-file.json');
```

2. **检查 CPU 密集计算**：
```javascript
// ❌ 主线程计算斐波那契（阻塞）
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}

// ✅ 移到 Worker Thread
const { Worker } = require('worker_threads');
const worker = new Worker('./fibonacci-worker.js');
```

### Q5：Prometheus 的 `rate()` 和 `irate()` 函数有什么区别？

**A**：
- **rate()**：计算一段时间内的平均速率（平滑曲线）
- **irate()**：计算瞬时速率（最近两个数据点）

**使用场景**：
```promql
# 适合告警（平滑，避免误报）
rate(http_requests_total[5m]) > 1000

# 适合实时监控（更敏感）
irate(http_requests_total[5m]) > 1000
```

**对比图示**：
```

irate():  ──▁▃▁▄▂▅▁▆▃▇▂█▁▇  （波动大）
```
