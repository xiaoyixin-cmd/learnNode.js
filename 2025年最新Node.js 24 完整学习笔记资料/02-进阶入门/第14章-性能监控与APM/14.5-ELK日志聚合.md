# 14.5 ELK日志聚合与分析

## 知识点概述

### 核心概念

ELK 是由 **Elasticsearch**、**Logstash**、**Kibana** 三个开源工具组成的日志聚合分析栈（现已加入 **Beats**，统称 Elastic Stack）。通过 ELK，你可以：

- **集中管理日志**：将分散在多台服务器的日志统一收集到 Elasticsearch
- **实时搜索分析**：毫秒级搜索 TB 级日志，支持复杂查询（全文搜索、正则、聚合）
- **可视化呈现**：通过 Kibana 创建仪表板，实时监控关键指标
- **告警通知**：设置规则，日志异常时自动发送钉钉/邮件/Slack 通知

### ELK 架构中的角色

```
┌─────────────┐
│ Node.js App │ → 生成日志（JSON 格式）
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  Filebeat   │ → 轻量级日志采集器（读取日志文件）
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  Logstash   │ → 日志处理引擎（解析、过滤、转换）
└──────┬──────┘
       │
       ↓
┌─────────────┐
│Elasticsearch│ → 分布式搜索引擎（存储+索引）
└──────┬──────┘
       │
       ↓
┌─────────────┐
│   Kibana    │ → 可视化平台（查询+仪表板+告警）
└─────────────┘
```

### 典型应用场景

- **微服务日志聚合**：10+ 个服务的日志统一查询
- **问题排查**：根据 Trace ID 关联分布式调用链路的所有日志
- **业务监控**：统计订单量、支付成功率、用户活跃度
- **安全审计**：检测异常登录、SQL 注入、接口滥用

## 使用指南

### 第一步：理解

#### 为什么需要日志聚合？

假设你的系统有 5 台服务器，某个用户反馈"下单失败"。传统方式：
- ❌ 需要 SSH 登录 5 台服务器，逐个查看日志文件
- ❌ 使用 `grep` 搜索，但日志格式不统一（有些是 JSON，有些是纯文本）
- ❌ 无法关联多个服务的日志（订单服务→库存服务→支付服务）

**有了 ELK**：
- ✅ 在 Kibana 界面搜索 `userId: "12345" AND error`，秒出结果
- ✅ 自动高亮关键词、展开 JSON 字段
- ✅ 根据 Trace ID 自动关联所有相关日志

#### ELK vs 传统日志管理

| 维度         | 传统方式（SSH + grep）    | ELK Stack                  |
|--------------|---------------------------|----------------------------|
| 日志搜索     | 慢（逐行扫描）            | 快（倒排索引，毫秒级）     |
| 跨服务器     | ❌ 需逐台登录             | ✅ 统一界面                |
| 结构化查询   | ❌ 仅支持正则             | ✅ 支持 JSON 字段查询      |
| 可视化       | ❌ 无                     | ✅ Kibana 仪表板           |
| 告警         | ❌ 需脚本轮询             | ✅ 内置告警规则            |

#### 使用 Docker Compose 启动 ELK 栈

**`docker-compose-elk.yml`**

```yaml
version: '3.8'

services:
  # 1️⃣ Elasticsearch：日志存储和搜索引擎
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.11.0
    container_name: elasticsearch
    environment:
      - discovery.type=single-node        # 单节点模式（测试用）
      - xpack.security.enabled=false      # 关闭安全认证（简化配置）
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"  # 限制内存使用
    ports:
      - "9200:9200"
      - "9300:9300"
    volumes:
      - es-data:/usr/share/elasticsearch/data
    networks:
      - elk

  # 2️⃣ Logstash：日志处理管道
  logstash:
    image: docker.elastic.co/logstash/logstash:8.11.0
    container_name: logstash
    ports:
      - "5000:5000/tcp"    # TCP 输入
      - "5000:5000/udp"    # UDP 输入
      - "9600:9600"        # API 端口
    volumes:
      - ./logstash/pipeline:/usr/share/logstash/pipeline  # 配置文件
    environment:
      - "LS_JAVA_OPTS=-Xms256m -Xmx256m"
    networks:
      - elk
    depends_on:
      - elasticsearch

  # 3️⃣ Kibana：可视化界面
  kibana:
    image: docker.elastic.co/kibana/kibana:8.11.0
    container_name: kibana
    ports:
      - "5601:5601"
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    networks:
      - elk
    depends_on:
      - elasticsearch

  # 4️⃣ Filebeat：日志采集器（可选）
  filebeat:
    image: docker.elastic.co/beats/filebeat:8.11.0
    container_name: filebeat
    user: root  # 需要读取宿主机日志文件
    volumes:
      - ./filebeat/filebeat.yml:/usr/share/filebeat/filebeat.yml:ro
      - /var/log:/var/log:ro  # 挂载宿主机日志目录
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
    networks:
      - elk
    depends_on:
      - elasticsearch

volumes:
  es-data:

networks:
  elk:
    driver: bridge
```

**`logstash/pipeline/logstash.conf`（Logstash 配置）**

```conf
input {
  # 💡 从 TCP 5000 端口接收日志（Node.js 应用通过网络发送）
  tcp {
    port => 5000
    codec => json_lines  # 解析 JSON 格式日志
  }
}

filter {
  # 💡 解析 JSON 日志
  json {
    source => "message"
    target => "log"
  }

  # 💡 添加时间戳字段
  date {
    match => [ "[log][timestamp]", "ISO8601" ]
    target => "@timestamp"
  }

  # 💡 提取 HTTP 状态码范围（用于聚合）
  if [log][statusCode] {
    mutate {
      add_field => {
        "statusCodeRange" => "%{[log][statusCode]}"
      }
    }
    
    # 分类：2xx, 3xx, 4xx, 5xx
    if [log][statusCode] >= 500 {
      mutate { update => { "statusCodeRange" => "5xx" } }
    } else if [log][statusCode] >= 400 {
      mutate { update => { "statusCodeRange" => "4xx" } }
    } else if [log][statusCode] >= 300 {
      mutate { update => { "statusCodeRange" => "3xx" } }
    } else if [log][statusCode] >= 200 {
      mutate { update => { "statusCodeRange" => "2xx" } }
    }
  }
}

output {
  # 💡 输出到 Elasticsearch
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "nodejs-logs-%{+YYYY.MM.dd}"  # 按日期创建索引
  }

  # 💡 同时打印到控制台（调试用）
  stdout {
    codec => rubydebug
  }
}
```

**启动 ELK 栈**：

```bash
mkdir -p logstash/pipeline
# 将上述 logstash.conf 保存到该目录

docker-compose -f docker-compose-elk.yml up -d

# 等待服务启动（约 1-2 分钟）
docker-compose -f docker-compose-elk.yml ps
```

**访问界面**：
- Elasticsearch: http://localhost:9200
- Kibana: http://localhost:5601

```javascript
const logger = pino({
```

```javascript

const express = require('express');
const logger = require('./utils/logger');

const app = express();

// 💡 请求日志中间件
app.use((req, res, next) => {
  const start = Date.now();
  
  res.on('finish', () => {
    const duration = Date.now() - start;
    
    logger.info({
      type: 'http',
      method: req.method,
      path: req.path,
      statusCode: res.statusCode,
      duration,
      ip: req.ip,
      userAgent: req.headers['user-agent']
    }, `${req.method} ${req.path} ${res.statusCode} ${duration}ms`);
  });
  
  next();
});

// 业务接口
app.get('/api/users', async (req, res) => {
  try {
    logger.debug({ userId: req.query.id }, 'Fetching user');
    
    const users = await getUsersFromDB();
    
    logger.info({ count: users.length }, 'Users fetched successfully');
    res.json({ users });
    
  } catch (error) {
    // 💡 记录错误日志（包含堆栈）
    logger.error({
      type: 'error',
      error: {
        message: error.message,
        stack: error.stack,
        code: error.code
      },
      context: {
        userId: req.query.id,
        path: req.path
      }
    }, 'Failed to fetch users');
    
    res.status(500).json({ error: 'Internal Server Error' });
  }
});

app.listen(3000, () => {
  logger.info({ port: 3000 }, 'Server started');
});
```

## 3个高频错误

### 错误1：Elasticsearch 启动失败（Out of Memory）

**现象**：

```bash
docker logs elasticsearch
# 输出：OpenJDK 64-Bit Server VM warning: INFO: os::commit_memory(...) failed; error='Cannot allocate memory'
```

**原因**：Elasticsearch 默认需要 2GB 内存，Docker 内存限制不足

**解决方案**：

```yaml
# docker-compose-elk.yml
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.11.0
    environment:
      # ✅ 限制 JVM 堆内存（根据实际内存调整）
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
      - discovery.type=single-node
    # ✅ 限制容器总内存
    deploy:
      resources:
        limits:
          memory: 1g
```

**验证**：


### 错误3：日志量过大导致 Elasticsearch 磁盘满

**现象**：

```bash

  "status": "red",
  "number_of_nodes": 1,
```

**原因**：日志量过大（如每天 10GB），磁盘空间不足

**解决方案**：

**1️⃣ 配置索引生命周期管理（ILM）**

```bash

  -H 'Content-Type: application/json' \
  -d '{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "1GB",
            "max_age": "1d"
          }
        }
      },
      "delete": {
        "min_age": "7d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}'
```

**2️⃣ 应用策略到索引模板**

```bash
curl -X PUT "http://localhost:9200/_index_template/nodejs-logs-template" \
  -H 'Content-Type: application/json' \
  -d '{
  "index_patterns": ["nodejs-logs-*"],
  "template": {
    "settings": {
      "index.lifecycle.name": "nodejs-logs-policy",
      "index.lifecycle.rollover_alias": "nodejs-logs"
    }
  }
}'
```

**3️⃣ 手动删除旧索引**

```bash
# 删除 7 天前的索引
curl -X DELETE "http://localhost:9200/nodejs-logs-2025.09.30"
```

**4️⃣ 在应用层过滤敏感日志**

```javascript
// ❌ 错误：记录所有请求（包括健康检查）

  logger.info({ path: req.path }, 'Request');

});

// ✅ 正确：排除高频低价值日志
```

## 应用场景

### 场景1：微服务分布式追踪

**问题背景**：
某电商平台采用微服务架构（API 网关→订单服务→库存服务→支付服务），用户投诉"下单失败"，需要快速定位是哪个服务出错。

**技术选型理由**：
- **ELK + Trace ID**：通过唯一 Trace ID 关联所有服务的日志

**完整实现**：

**1️⃣ 在 API 网关生成 Trace ID**

```javascript
// api-gateway/server.js
const express = require('express');
const { v4: uuidv4 } = require('uuid');
const logger = require('./utils/logger');

const app = express();

// 💡 生成 Trace ID 并传递给下游服务
app.use((req, res, next) => {
  req.traceId = req.headers['x-trace-id'] || uuidv4();
  res.setHeader('X-Trace-ID', req.traceId);
  
  // 💡 日志中包含 traceId
  req.log = logger.child({ traceId: req.traceId });
  next();
});

app.post('/api/orders', async (req, res) => {
  req.log.info({ body: req.body }, 'Order request received');
  
  try {
    // 调用订单服务（传递 Trace ID）
    const response = await axios.post('http://order-service:3001/orders', req.body, {
      headers: { 'X-Trace-ID': req.traceId }
    });
    
    req.log.info({ orderId: response.data.orderId }, 'Order created successfully');
    res.json(response.data);
    
  } catch (error) {
    req.log.error({ error: error.message }, 'Order creation failed');
    res.status(500).json({ error: 'Internal Server Error' });
  }
});

app.listen(3000, () => logger.info('API Gateway started'));
```

**2️⃣ 订单服务透传 Trace ID**

```javascript
// order-service/server.js
const express = require('express');
const logger = require('./utils/logger');
const axios = require('axios');

const app = express();
```

## 深度解析

### 对比分析：日志采集方案对比

| 方案                  | 优点                     | 缺点                     | 适用场景             |
|-----------------------|--------------------------|--------------------------|----------------------|
| **Filebeat**          | 轻量、稳定、缓冲能力强   | 需要写文件（磁盘 I/O）   | 传统部署、日志量大   |
| **Pino + TCP**        | 性能最高、零磁盘 I/O     | 网络故障时日志丢失       | 云原生、容器化       |
| **Winston + ES**      | 直接写 ES、无需 Logstash | ES 故障时影响应用性能    | 小规模、日志量少     |

**推荐选择**：
- 🚀 **高性能要求**：Pino + Logstash（异步发送，不阻塞应用）
- 🛡️ **可靠性优先**：Filebeat（本地缓冲，网络故障时不丢日志）
- 🎯 **简化架构**：Winston + ES（省去 Logstash）

```
  优势：
  - 日志不会丢失（先写本地）
  - Logstash 可以做复杂过滤
```

### Q3：如何在微服务中实现分布式追踪（Trace ID）？

```javascript
  res.setHeader('X-Trace-ID', req.traceId);
```

```json
        "traceId": { "type": "keyword" },  // Trace ID（常查）
        "level": { "type": "keyword" },    // 日志级别
        "message": { "type": "text" }      // 日志内容（全文搜索）
      }
    }
  }
}
```

---

**优化3：使用精确匹配而非模糊搜索**
```
❌ 慢：模糊搜索
message: *payment*  // 全文搜索（慢）

✅ 快：精确匹配
traceId: "abc-123"  // Keyword 字段（快 10 倍）
```

---

**优化4：减少返回字段**
```json
// Kibana Discover 设置
{
  "_source": ["timestamp", "level", "message", "traceId"]  // 只返回需要的字段
}
```

---

**优化5：使用聚合而非原始日志**
```
❌ 慢：查询 100 万条日志统计错误数
GET /nodejs-logs-*/_search
{
  "query": { "match": { "level": "error" } }
}

✅ 快：使用聚合（只返回统计结果）
GET /nodejs-logs-*/_search
{
  "size": 0,  // 不返回原始文档
  "aggs": {
    "error_count": {
      "value_count": { "field": "level" }
    }
  }
}
```

**性能对比**：

| 优化项               | 查询时间 | 提升   |
|----------------------|----------|--------|
| 优化前               | 12s      | -      |
| + 限制时间范围       | 3s       | +4x    |
| + 使用精确匹配       | 0.8s     | +3.8x  |
| + 使用聚合           | 0.2s     | +4x    |
