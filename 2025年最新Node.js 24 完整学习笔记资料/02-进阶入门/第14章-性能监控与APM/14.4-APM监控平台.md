# 14.4 APM监控平台

## 知识点概述

### 核心概念

APM（Application Performance Monitoring，应用性能监控）是一套完整的监控解决方案，通过自动采集应用的性能指标、错误日志、调用链路等数据，帮助你：

- **实时监控应用健康状态**：CPU、内存、响应时间、错误率一目了然
- **快速定位性能瓶颈**：自动标记慢事务、慢查询、慢接口
- **分析用户体验**：追踪真实用户访问的完整路径（从前端到后端到数据库）
- **提前预警故障**：设置告警规则，故障发生前自动通知

### APM vs 传统监控

| 维度         | 传统监控（日志+指标）       | APM平台                        |
|--------------|-----------------------------|---------------------------------|
| 数据采集     | 手动埋点、分散日志          | 自动注入代理、统一采集          |
| 调用链追踪   | ❌ 不支持                   | ✅ 分布式追踪（跨服务）         |
| 根因分析     | ❌ 需手动关联               | ✅ 自动关联事务→查询→错误       |
| 可视化       | 需自建 Grafana              | 开箱即用的仪表板                |
| 学习成本     | 高（需掌握多个工具）        | 低（统一平台）                  |

### 主流 APM 平台对比

| 平台                    | 部署方式   | 定价         | 特点                         | 适用场景         |
|-------------------------|------------|--------------|------------------------------|------------------|
| **New Relic**           | SaaS       | $99+/月      | 功能最全、易用性强           | 中大型企业       |
| **Datadog**             | SaaS       | $15+/月      | 云原生、集成度高             | DevOps 团队      |
| **AppDynamics**         | SaaS/私有化| 按需报价     | 企业级、复杂场景             | 大型企业         |
| **Prometheus + Grafana**| 私有化     | 免费         | 开源、灵活、需自建           | 技术团队         |
| **Elastic APM**         | SaaS/私有化| 免费/付费    | ELK 栈集成、日志+指标        | 已有 ELK 栈      |
| **Sentry**              | SaaS/私有化| 免费/付费    | 专注错误追踪                 | 错误监控优先     |

## 使用指南

### 第一步：理解

#### 为什么需要 APM 平台？

假设你的 API 突然响应变慢，传统监控只能告诉你：
- ❌ "某台服务器 CPU 80%"（但不知道是哪个接口慢）
- ❌ "数据库连接数增加"（但不知道是哪个查询慢）
- ❌ "错误日志有异常"（但不知道影响了哪些用户）

**有了 APM 平台**：
- ✅ 自动发现 `/api/orders` 接口 P95 从 100ms 增长到 3s
- ✅ 定位到 `SELECT * FROM orders WHERE user_id = ?` 查询耗时 2.8s
- ✅ 关联到 200 个用户受影响，自动发送告警

#### APM 的核心功能

**1️⃣ 分布式追踪（Distributed Tracing）**
```
用户请求 → API网关 → 订单服务 → 库存服务 → MySQL
    ↓         ↓          ↓          ↓          ↓
  50ms      10ms      2800ms      50ms      2750ms  ← 🔥 瓶颈在库存服务的数据库查询
```

**2️⃣ 错误追踪与聚合**
```
错误：Cannot read property 'id' of undefined
  - 发生次数：523 次
  - 影响用户：89 人
  - 首次出现：2025-10-07 10:23:45
  - 堆栈：
      at getUserOrders (/app/services/order.js:45)
      at /app/routes/api.js:23
```

**3️⃣ 性能基线与异常检测**
```
过去7天 /api/products 平均响应时间：120ms
今天突然：350ms ← 🚨 自动触发告警
```

**3️⃣ 创建配置文件 `newrelic.js`**

```javascript
// newrelic.js（放在项目根目录）
'use strict'

/**
 * New Relic agent 配置文件
 */
exports.config = {
  // 💡 应用名称（会显示在 New Relic 仪表板中）
  app_name: ['My Node.js App'],
  
  // 💡 License Key（从环境变量读取，避免泄漏）
  license_key: process.env.NEW_RELIC_LICENSE_KEY,
  
  // 日志级别
  logging: {
    level: 'info'
  },
  
  // 💡 启用分布式追踪
  distributed_tracing: {
    enabled: true
  },
  
  // 💡 慢事务阈值（超过此值会被标记）
  transaction_tracer: {
    enabled: true,
    transaction_threshold: 'apdex_f', // 或具体值如 '500ms'
    record_sql: 'obfuscated',          // 记录 SQL 但混淆参数
    explain_threshold: 500             // 超过 500ms 自动 EXPLAIN
  },
  
  // 💡 错误追踪
  error_collector: {
    enabled: true,
    ignore_status_codes: [404], // 忽略 404 错误
    capture_events: true
  },
  
  // 允许所有环境（生产/开发）
  allow_all_headers: true,
  attributes: {
    exclude: [
      'request.headers.cookie',  // 不采集敏感信息
      'request.headers.authorization'
    ]
  }
}
```

**4️⃣ 在应用入口引入（必须第一行）**

```javascript
// server.js
// ⚠️ 必须在所有 require 之前引入
require('newrelic');

const express = require('express');
const app = express();

app.get('/api/users', async (req, res) => {
  const users = await getUsersFromDB();
  res.json({ users });
});

app.get('/api/slow', async (req, res) => {
  await new Promise(resolve => setTimeout(resolve, 3000)); // 慢接口
  res.json({ message: 'This is slow' });
});

app.listen(3000, () => {
  console.log('🚀 Server running on http://localhost:3000');
});
```

**5️⃣ 启动应用**

```bash
export NEW_RELIC_LICENSE_KEY="your-license-key-here"
node server.js
```

**6️⃣ 查看 New Relic 仪表板**

```
访问 https://one.newrelic.com
导航到 APM → 选择你的应用

核心指标一览：

  - Throughput: QPS
  - Error Rate: 错误率
  - Apdex Score: 用户满意度（0-1分，越高越好）

慢事务列表：
```

**1️⃣ 使用 Docker Compose 启动 Prometheus + Grafana**

```yaml
  # Prometheus 时序数据库

    container_name: prometheus
    ports:
      - "9090:9090"

      - ./prometheus.yml:/etc/prometheus/prometheus.yml

      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--web.console.libraries=/usr/share/prometheus/console_libraries'
      - '--web.console.templates=/usr/share/prometheus/consoles'

      - monitoring

  # Grafana 可视化平台
  grafana:
    image: grafana/grafana:latest
    container_name: grafana
    ports:
      - "3001:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin
    volumes:
      - grafana-data:/var/lib/grafana
    networks:
      - monitoring

volumes:
  prometheus-data:
  grafana-data:

networks:
  monitoring:
```

```yaml
global:

  # 采集 Node.js 应用的 /metrics 端点
```

### 错误3：APM 数据量过大导致成本暴增

**现象**：New Relic/Datadog 账单从 $100/月 增长到 $800/月

**原因**：采集了所有事务/日志，包括 `/health`、`/metrics` 等高频接口

**解决方案**：

**1️⃣ 忽略健康检查接口**

```javascript
// newrelic.js
exports.config = {
  // ...
  rules: {
    ignore: [
      '^/health$',         // 忽略健康检查
      '^/metrics$',        // 忽略 Prometheus 指标
      '^/favicon.ico$'     // 忽略静态资源
    ]
  }
};
```

**2️⃣ 采样（仅采集 10% 的事务）**

```javascript
// elastic-apm-node
const apm = require('elastic-apm-node').start({
  serviceName: 'my-app',
  transactionSampleRate: 0.1  // 💡 仅采集 10% 事务
});
```

**3️⃣ 仅在响应慢时采集详细 Trace**

```javascript
// 自定义采样逻辑
const apm = require('elastic-apm-node').start({
  // ...
  transactionFilter(payload) {
    // 💡 仅采集响应时间 > 500ms 的事务
    if (payload.duration && payload.duration > 500) {
      return payload;
    }
    return false; // 忽略
  }
});
```

**成本对比**：

| 策略                | 月度成本 | 数据完整性 |
|---------------------|----------|------------|
| 采集全部事务        | $800     | 100%       |
| 忽略 /health        | $650     | 95%        |
| 采样 10%            | $120     | 10%（足够）|
| 仅采集慢事务(>500ms)| $180     | 关键数据   |

## 应用场景

### 场景1：微服务架构调用链追踪

**问题背景**：
某电商平台采用微服务架构（订单→库存→支付→物流），用户投诉"下单很慢"，但不知道是哪个服务拖累了整体响应。

**技术选型理由**：
- **New Relic** 或 **Elastic APM**：支持分布式追踪，自动关联跨服务调用

**完整实现**：

**1️⃣ 在所有微服务中集成 New Relic**

```javascript
// order-service/server.js
require('newrelic');
const express = require('express');
const axios = require('axios');

const app = express();

app.post('/api/orders', async (req, res) => {
  try {
    // 1. 检查库存
    const stock = await axios.get('http://inventory-service/api/check', {
      params: { productId: req.body.productId }
    });
    
    // 2. 调用支付
    const payment = await axios.post('http://payment-service/api/pay', {
      orderId: 'ORDER123',
      amount: 99.99
    });
    
    // 3. 创建物流单
    const shipping = await axios.post('http://shipping-service/api/create', {
      orderId: 'ORDER123'
    });
    
    res.json({ success: true });
  } catch (error) {
    res.status(500).json({ error: error.message });
  }
});

app.listen(3000);
```

```javascript
// inventory-service/server.js
require('newrelic');
const express = require('express');
const app = express();

app.get('/api/check', async (req, res) => {
  // 模拟慢查询
  await new Promise(resolve => setTimeout(resolve, 2000));
  res.json({ inStock: true });
});

app.listen(3001);
```

**2️⃣ 在 New Relic 中查看分布式追踪**

```
Distributed Tracing:
  POST /api/orders (3.5s total)
    ├─ GET inventory-service/api/check (2.1s) 🔥 瓶颈
    ├─ POST payment-service/api/pay (0.8s)
    └─ POST shipping-service/api/create (0.5s)
```

**3️⃣ 点击慢调用查看详情**

```
GET inventory-service/api/check (2.1s)
```

### 场景2：错误聚合与影响面分析

**问题背景**：
某 SaaS 平台收到用户反馈"无法登录"，需要快速确定影响范围（多少用户受影响、哪个版本引入的bug）。

**技术选型理由**：
- **Sentry**：专注错误追踪，自动聚合相同错误、关联版本和用户

**完整实现**：

**1️⃣ 安装并配置 Sentry**

```bash
npm install @sentry/node @sentry/profiling-node
```

```javascript
// server.js
const Sentry = require('@sentry/node');
const { ProfilingIntegration } = require('@sentry/profiling-node');

// 💡 在所有代码之前初始化
Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV || 'development',
  release: process.env.GIT_COMMIT || 'unknown',
  
  // 💡 启用性能监控
  tracesSampleRate: 1.0,
  profilesSampleRate: 1.0,
  
  // 💡 自动采集用户信息
  integrations: [
    new ProfilingIntegration(),
    new Sentry.Integrations.Http({ tracing: true }),
    new Sentry.Integrations.Express({ app })
  ]
});

const express = require('express');
const app = express();

// 💡 Sentry 请求处理器（必须在路由之前）
app.use(Sentry.Handlers.requestHandler());
app.use(Sentry.Handlers.tracingHandler());

app.post('/api/login', (req, res) => {
  try {
    const user = getUserById(req.body.userId);
    
    // ❌ Bug：用户不存在时抛出错误
    if (!user) {
      throw new Error('User not found');
    }
    
    // 💡 手动设置用户上下文
    Sentry.setUser({
      id: user.id,
      email: user.email,
      username: user.username
    });
    
    res.json({ token: 'fake-jwt' });
  } catch (error) {
    // 💡 手动捕获错误并添加上下文
    Sentry.captureException(error, {
      tags: {
        route: '/api/login',
        severity: 'high'
      },
      extra: {
        requestBody: req.body,
        userAgent: req.headers['user-agent']
      }
    });
    
    res.status(500).json({ error: error.message });
  }
});

// 💡 Sentry 错误处理器（必须在路由之后）
app.use(Sentry.Handlers.errorHandler());

app.listen(3000);
```

## 总结

### ✅ 核心要点

1. **选对平台**：商业平台易用但贵，开源平台免费但需运维
2. **分布式追踪**：APM 最大价值是自动关联跨服务调用链路
3. **错误聚合**：Sentry 可自动聚合相同错误，快速定位影响面

### 🔑 实现关键

1. **APM Agent 必须第一行引入**：`require('newrelic')` 必须在所有代码之前
2. **配置采样率降低成本**：生产环境建议采样 10-20% 事务
3. **忽略健康检查接口**：避免 `/health`、`/metrics` 污染数据
4. **设置告警规则**：P99 > 500ms、错误率 > 1% 时自动通知

### 💎 应用价值

- **快速定位瓶颈**：从 10 个微服务中 3 分钟定位到慢查询
- **降低 MTTR**：故障平均恢复时间从 2 小时降至 15 分钟
- **优化用户体验**：提前发现性能劣化，避免用户投诉

### 🎯 面试重点

- **Q1**：APM 和日志监控有什么区别？  
  **A**：APM 自动采集指标+追踪+错误，日志需手动埋点且无法关联调用链。

- **Q2**：如何在微服务架构中实现分布式追踪？  
  **A**：使用 Trace ID 贯穿所有服务，APM Agent 自动传递和采集。

- **Q3**：APM 性能开销如何优化？  
  **A**：降低采样率（如 10%）、禁用 Body 采集、忽略高频接口。

### ⚠️ 注意事项

1. **不要采集敏感信息**：配置 `ignore_headers` 排除 Cookie、Authorization
2. **商业平台注意成本**：按事务量计费，忽略健康检查接口
3. **开源方案需运维**：Prometheus + Grafana 需定期清理旧数据

### Q4：如何在微服务架构中实现分布式追踪？

```javascript
// Gateway Service（API 网关）
const { v4: uuidv4 } = require('uuid');

app.use((req, res, next) => {
  // 生成或传递 Trace ID

  res.setHeader('X-Trace-ID', req.traceId);
  next();
});
```

**步骤2：通过 HTTP Header 传递**

```javascript
// Order Service 调用 Payment Service

async function createOrder(req) {

    orderId: '12345',

      'X-Trace-ID': req.traceId,  // 传递 Trace ID
      'X-Parent-Span-ID': currentSpanId  // 传递父级 Span ID
    }
  });
}
```
