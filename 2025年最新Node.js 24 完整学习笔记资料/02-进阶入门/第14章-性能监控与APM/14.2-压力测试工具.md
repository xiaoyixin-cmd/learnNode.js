# 14.2 压力测试工具

## 知识点概述

### 核心概念

压力测试（Stress Testing）是通过模拟大量并发请求，评估系统在极限负载下的性能表现。通过压测，你可以：

- **确定系统容量上限**：找出单台服务器能承载的最大 QPS
- **发现性能瓶颈**：识别数据库慢查询、内存泄漏、CPU 占用高等问题
- **验证扩容效果**：对比扩容前后的性能指标，评估 ROI
- **保障大促稳定性**：提前模拟双十一流量，避免真实场景崩溃

### 主流压测工具对比

| 工具              | 语言    | 特点                             | 适用场景                   |
|-------------------|---------|----------------------------------|----------------------------|
| **wrk**           | C       | 极致性能、脚本化、轻量级         | HTTP API 压测、性能基准测试 |
| **autocannon**    | Node.js | 零配置、Node.js 生态友好         | Node.js 项目快速压测       |
| **Apache Bench**  | C       | 简单易用、老牌工具               | 简单场景、快速验证         |
| **k6**            | Go      | 云原生、脚本化、丰富的指标       | 复杂场景、CI/CD 集成       |
| **Gatling**       | Scala   | 企业级、录制回放、详细报告       | 大型项目、业务场景模拟     |
| **JMeter**        | Java    | GUI、功能全面、学习曲线陡峭      | 传统企业、复杂业务流程     |

### 核心性能指标

- **QPS (Queries Per Second)**：每秒处理的请求数
- **平均延迟 (Avg Latency)**：所有请求的平均响应时间
- **P50/P90/P99 延迟**：50%/90%/99% 请求的响应时间
- **错误率**：HTTP 4xx/5xx 状态码占比
- **吞吐量 (Throughput)**：每秒传输的字节数

### 典型应用场景

- **上线前验证**：确保新版本性能不低于旧版本
- **扩容决策**：通过压测确定需要几台服务器
- **SLA 验证**：确保 P99 < 200ms、可用性 > 99.9%

## 使用指南

### 第一步：理解

#### 为什么需要压力测试？

假设你开发了一个电商 API，本地测试一切正常。但上线后：
- ❌ 用户访问量稍大就出现 502 Bad Gateway
- ❌ 数据库连接池耗尽，请求排队超时
- ❌ 内存占用飙升，最终 OOM 崩溃

**有了压力测试**：
- ✅ 提前发现系统在 500 QPS 时数据库连接池不足
- ✅ 提前优化慢查询，将 P99 从 3s 降至 200ms
- ✅ 提前确定需要 3 台服务器才能支撑预期流量

#### 压测的核心原则

1. **从低到高逐步加压**：先测 100 QPS，再测 500、1000、5000...
2. **监控系统资源**：同时监控 CPU、内存、磁盘 IO、网络带宽
3. **模拟真实场景**：不要只测 GET 接口，要模拟真实用户行为（登录→浏览→下单）
4. **多次测试取平均值**：单次测试可能受网络抖动影响，至少测试 3 次

⚠️ **注意**：永远不要对生产环境直接压测！使用与生产环境配置相同的测试环境。

```javascript
// test/benchmark.js
const autocannon = require('autocannon');

async function runBenchmark() {
  const result = await autocannon({
    url: 'http://localhost:3000/api/fast',
    connections: 100,    // 并发连接数
    duration: 10,        // 持续时间（秒）
    pipelining: 1,       // HTTP 管道请求数
    workers: 4           // worker 进程数
  });

  console.log('🎯 压测结果：');
  console.log(`QPS: ${result.requests.average}`);
  console.log(`平均延迟: ${result.latency.mean} ms`);
  console.log(`P99 延迟: ${result.latency.p99} ms`);
  console.log(`错误数: ${result.errors}`);

  // 💡 判断性能是否达标
  if (result.requests.average < 10000) {
    console.error('❌ 性能不达标！QPS < 10000');
    process.exit(1);
  }

  if (result.latency.p99 > 100) {
    console.error('❌ P99 延迟过高！> 100ms');
    process.exit(1);
  }

  console.log('✅ 性能测试通过！');
}

runBenchmark();
```

## 3个高频错误

### 错误1：压测结果不稳定，QPS 波动 ±30%

**现象**：同一接口多次压测，QPS 从 15k 到 22k 不等

**原因**：
1. 测试时间太短（< 10 秒），受启动阶段影响
2. 压测机器与被测服务器共享资源
3. 网络抖动或其他进程占用 CPU

**解决方案**：

```bash
# ❌ 错误：测试时间太短
wrk -t10 -c100 -d5s http://localhost:3000/api/fast

# ✅ 正确：至少测试 30 秒，取后 20 秒数据
wrk -t10 -c100 -d30s http://localhost:3000/api/fast

# ✅ 更好：使用独立压测机器
# 压测机（192.168.1.100）
wrk -t10 -c100 -d30s http://192.168.1.101:3000/api/fast

# 被测服务器（192.168.1.101）
node server.js
```

**多次测试取平均值**：

```bash
#!/bin/bash
# benchmark.sh
for i in {1..5}; do
  echo "第 $i 次压测..."
  wrk -t10 -c100 -d30s http://localhost:3000/api/fast | grep "Requests/sec"
  sleep 5 # 间隔 5 秒
done
```

### 错误3：压测时 CPU 100%，但 QPS 不增反降

**现象**：增加并发数从 100 到 500，CPU 从 60% 飙升到 100%，但 QPS 从 15k 降至 8k

**原因**：遇到性能瓶颈（数据库连接池耗尽、事件循环阻塞）

**排查步骤**：

**1️⃣ 检查数据库连接池**

```javascript
// ❌ 错误：连接池太小
const pool = new Pool({
  max: 10 // 只有 10 个连接，500 并发时会排队
});

// ✅ 正确：根据并发数调整
const pool = new Pool({
  max: 50,              // 最大连接数
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000 // 超时时间
});
```

**2️⃣ 检查事件循环延迟**

```javascript
const { monitorEventLoopDelay } = require('perf_hooks');
const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();

setInterval(() => {
  console.log(`Event Loop Lag: ${h.mean / 1000000} ms`);
  h.reset();
}, 5000);
```

**3️⃣ 使用 Cluster 模式利用多核**

```javascript
// ❌ 错误：单进程无法利用多核 CPU
node server.js

// ✅ 正确：使用 PM2 启动 4 个进程
pm2 start server.js -i 4
```

**压测对比**：

| 模式           | QPS     | CPU 使用率 |
|----------------|---------|------------|
| 单进程         | 8,000   | 100% (1核) |
| Cluster 4进程  | 28,000  | 70% (4核)  |

## 应用场景

### 场景1：电商大促前容量评估

**问题背景**：
某电商平台预计双十一流量是平时的 10 倍（从 500 QPS 增长到 5000 QPS）。运维需要确定需要多少台服务器。

**技术选型理由**：
- **wrk + Lua 脚本**：模拟真实用户行为（浏览→加购→下单）
- **Prometheus + Grafana**：实时监控 QPS、P99、错误率

**完整实现**：

**1️⃣ 编写 Lua 脚本模拟用户行为**

```lua
-- user-journey.lua
local token = nil
local cartId = nil

request = function()
  -- 💡 模拟用户旅程：登录 → 浏览商品 → 加入购物车 → 下单
  if token == nil then
    -- 第1步：登录
    return wrk.format("POST", "/api/login", 
      {["Content-Type"] = "application/json"}, 
      '{"username":"user' .. math.random(10000) .. '","password":"123456"}'
    )
  elseif cartId == nil then
    -- 第2步：浏览商品（随机查看 1-1000 号商品）
    return wrk.format("GET", "/api/products/" .. math.random(1000), 
      {["Authorization"] = "Bearer " .. token}
    )
  elseif math.random() < 0.3 then
    -- 第3步：加入购物车（30% 概率）
    return wrk.format("POST", "/api/cart/add", 
      {["Authorization"] = "Bearer " .. token, ["Content-Type"] = "application/json"},
      '{"productId":' .. math.random(1000) .. ',"quantity":1}'
    )
  else
    -- 第4步：下单（10% 概率）
    if math.random() < 0.1 then
      return wrk.format("POST", "/api/orders", 
        {["Authorization"] = "Bearer " .. token, ["Content-Type"] = "application/json"},
        '{"cartId":"cart123","paymentMethod":"alipay"}'
      )
    else
      -- 继续浏览商品
      return wrk.format("GET", "/api/products/" .. math.random(1000), 
        {["Authorization"] = "Bearer " .. token}
      )
    end
  end
end

response = function(status, headers, body)
  -- 提取登录 token
  if status == 200 and not token and body:match('"token"') then
    token = body:match('"token":"([^"]+)"')
  end
end
```

**2️⃣ 逐步加压测试**

```bash
#!/bin/bash
# load-test.sh

echo "📊 开始容量评估..."

for qps in 500 1000 2000 3000 5000 8000; do
  echo "🔥 目标 QPS: $qps"
  
  # 计算需要的并发数（经验值：并发数 ≈ QPS × 平均延迟）
  # 假设平均延迟 50ms = 0.05s，则并发数 = qps * 0.05
  connections=$((qps / 20))
  
  wrk -t10 -c$connections -d60s -s user-journey.lua http://localhost:3000 \
    | tee results-$qps.txt
  
  echo "⏸️  等待 30 秒让系统恢复..."
  sleep 30
done

echo "✅ 测试完成，查看 results-*.txt"
```

**3️⃣ 分析结果确定容量**

**1️⃣ 编写性能基准测试**

```javascript
// test/performance.test.js
const autocannon = require('autocannon');
const { spawn } = require('child_process');
const fs = require('fs');

// 性能阈值（根据历史数据设定）
const BASELINE = {
  '/api/products': { qps: 8000, p99: 100 },
  '/api/users':    { qps: 12000, p99: 50 },
  '/api/orders':   { qps: 5000, p99: 200 }
};

async function runBenchmark(path, baseline) {
  const result = await autocannon({
    url: `http://localhost:3000${path}`,
    connections: 100,
    duration: 20,
    workers: 4
  });

  const actualQps = result.requests.average;
  const actualP99 = result.latency.p99;

  console.log(`\n📊 ${path}`);
  console.log(`  QPS: ${actualQps} (baseline: ${baseline.qps})`);
  console.log(`  P99: ${actualP99}ms (baseline: ${baseline.p99}ms)`);

  // 💡 判断是否回归（允许 10% 波动）
  const qpsPass = actualQps >= baseline.qps * 0.9;
  const p99Pass = actualP99 <= baseline.p99 * 1.1;

  if (!qpsPass) {
    console.error(`  ❌ QPS 回归 ${((1 - actualQps / baseline.qps) * 100).toFixed(1)}%`);
  }
  if (!p99Pass) {
    console.error(`  ❌ P99 延迟增加 ${((actualP99 / baseline.p99 - 1) * 100).toFixed(1)}%`);
  }

  return qpsPass && p99Pass;
}

async function main() {
  // 启动测试服务器
  const server = spawn('node', ['server.js']);
  await new Promise(resolve => setTimeout(resolve, 2000)); // 等待启动

  let allPassed = true;

  for (const [path, baseline] of Object.entries(BASELINE)) {
    const passed = await runBenchmark(path, baseline);
    if (!passed) allPassed = false;
  }

  server.kill();

  if (!allPassed) {
    console.error('\n❌ 性能测试未通过！');
    process.exit(1);
  }

  console.log('\n✅ 所有性能测试通过！');
}

main();
```

## 🤔 常见问题（FAQ）

### Q1：QPS 和并发数有什么关系？

**A**：它们通过"小利特尔法则"（Little's Law）关联：

**公式**：
```
并发数 = QPS × 平均响应时间（秒）
```

**举例说明**：
```
场景1：
- QPS = 1000
- 平均响应时间 = 50ms = 0.05s
- 并发数 = 1000 × 0.05 = 50

场景2：
- QPS = 1000（不变）
- 优化后响应时间 = 10ms = 0.01s
- 并发数 = 1000 × 0.01 = 10（降低了 80%）
```

**结论**：响应时间越低，达到相同 QPS 所需的并发数越少，服务器资源利用率越高。

---

### Q2：为什么压测结果每次都不一样？

**A**：受以下因素影响：

**主要原因**：
1. **系统资源波动**：CPU、内存、磁盘 I/O 被其他进程占用
2. **网络波动**：Wi-Fi 不稳定、交换机带宽抢占
3. **数据库缓存**：首次查询慢（冷缓存），后续查询快（热缓存）
4. **JIT 预热**：V8 引擎需要时间优化热点代码

**解决方案**：
```bash
# 1. 多次测试取平均值
for i in {1..5}; do
  autocannon -c 10 -d 30 http://localhost:3000 >> results.txt
done

# 2. 测试前预热
autocannon -c 5 -d 10 http://localhost:3000  # 预热
sleep 5
autocannon -c 10 -d 30 http://localhost:3000  # 正式测试

# 3. 关闭其他进程
pkill Chrome  # 关闭浏览器
sudo service mysql stop  # 临时关闭 MySQL（如果不测数据库）
```

**判断结果是否可信**：
- 标准差 < 10%：结果可信
- 标准差 > 20%：环境不稳定，需排查

---

### Q3：生产环境 QPS 远低于压测结果，为什么？

**A**：常见原因有 5 个：

**原因1：压测场景过于简化**
```javascript
// ❌ 压测：无鉴权、空数据库
app.get('/api/users', (req, res) => {
  res.json({ users: [] });  // 直接返回空数组
});

// ✅ 生产：有鉴权、复杂查询、数据库 I/O
app.get('/api/users', authenticateToken, async (req, res) => {
  const users = await db.query('SELECT * FROM users WHERE status = ?', ['active']);
  res.json({ users });
});
```

**原因2：网络延迟**
```
压测环境：localhost（延迟 < 1ms）
生产环境：跨地域（延迟 50-200ms）

影响：
- 压测 QPS = 10000（1ms 延迟）
- 生产 QPS = 500（200ms 延迟）
```

**原因3：第三方服务依赖**
```javascript
// 生产环境调用微信支付（响应时间 300ms）
await wechatPay.createOrder(orderId);  

// 压测环境 Mock（响应时间 1ms）
return { status: 'success' };
```

**原因4：数据量差异**
```sql
-- 压测：10 条数据
SELECT * FROM products LIMIT 10;  -- 1ms

-- 生产：100 万条数据（无索引）
SELECT * FROM products WHERE category = 'electronics';  -- 500ms
```

**原因5：资源竞争**
```
压测环境：独立服务器（8 核 16GB）
生产环境：共享容器（限制 2 核 4GB）+ 其他微服务竞争 CPU
```

### Q4：如何判断压测结果是否真实可信？

**A**：检查以下 6 个维度：

**检查清单**：
```
✅ 1. P99 延迟是否 < 500ms？（如果 > 1s 说明系统已过载）
✅ 2. 错误率是否 = 0？（如果 > 1% 说明测试结果无效）
✅ 3. CPU 使用率是否 < 80%？（如果 100% 说明已达上限）
✅ 4. 多次测试标准差 < 10%？（如果波动大说明环境不稳定）
✅ 5. 监控日志无异常？（数据库连接超时、内存溢出）
✅ 6. 压测机 CPU 未满？（如果压测机 CPU 100% 说明瓶颈在客户端）
```

**选择建议**：
```bash
# 场景1：快速验证接口能否正常访问
ab -n 100 -c 10 http://localhost:3000/

# 场景2：Node.js 项目日常性能测试
autocannon -c 50 -d 30 http://localhost:3000/

# 场景3：复杂业务流程（登录→购物→支付）
wrk -t 4 -c 100 -d 60s -s complex-flow.lua http://localhost:3000/

# 场景4：分布式压测（模拟百万 QPS）
k6 run --vus 1000 --duration 30s script.js
```

### Q6：如何处理压测时出现的大量 `ECONNREFUSED` 错误？

**A**：原因是服务器拒绝新连接，解决方案：

**原因1：端口耗尽**
```bash
# 查看当前连接数
netstat -an | grep ESTABLISHED | wc -l

# 增加可用端口范围
sudo sysctl -w net.ipv4.ip_local_port_range="10000 65535"

# 减少 TIME_WAIT 时间
sudo sysctl -w net.ipv4.tcp_fin_timeout=30
```

**原因2：文件描述符限制**
```bash
# 查看限制
ulimit -n  # 通常是 1024

# 临时增加（当前 Shell 有效）
ulimit -n 65536

# 永久增加
echo "* soft nofile 65536" | sudo tee -a /etc/security/limits.conf
echo "* hard nofile 65536" | sudo tee -a /etc/security/limits.conf
```

**原因3：连接池不足**
```javascript
// ❌ 连接池太小
const pool = mysql.createPool({
  connectionLimit: 10  // 压测 100 并发时会耗尽
});

// ✅ 增加连接池
const pool = mysql.createPool({
  connectionLimit: 100,  // 至少等于并发数
  queueLimit: 0  // 无限队列
});
```

### Q7：如何在 CI/CD 中集成性能测试？

**A**：使用 autocannon + GitHub Actions 自动化：

**步骤1：编写测试脚本**
```javascript
// perf-test.js
const autocannon = require('autocannon');

const run = async () => {
  const result = await autocannon({
    url: 'http://localhost:3000/api/users',
    connections: 10,
    duration: 30
  });

  const qps = result.requests.average;
  const p99 = result.latency.p99;

  console.log(`QPS: ${qps}, P99: ${p99}ms`);

  // 性能回归检测
  if (qps < 1000) {
    console.error('❌ 性能回归：QPS 低于 1000');
    process.exit(1);  // CI 失败
  }

  if (p99 > 500) {
    console.error('❌ 性能回归：P99 超过 500ms');
    process.exit(1);
  }

  console.log('✅ 性能测试通过');
};

run();
```

**步骤2：GitHub Actions 配置**
```yaml
# .github/workflows/perf.yml
name: Performance Test

on: [push, pull_request]

jobs:
  perf-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Start server
        run: |
          npm start &
          sleep 5  # 等待服务启动
      
      - name: Run performance test
        run: node perf-test.js
      
      - name: Upload report
        uses: actions/upload-artifact@v3
        with:
          name: perf-report
          path: report.json
```

### Q8：压测时 CPU 使用率很低，QPS 却上不去？

**A**：可能遇到了 I/O 瓶颈而非 CPU 瓶颈：

**场景1：数据库 I/O 慢**
```javascript
// 问题：每个请求都查询数据库（50ms）
app.get('/api/products/:id', async (req, res) => {
  const product = await db.query('SELECT * FROM products WHERE id = ?', [req.params.id]);
  res.json(product);
});

// 解决：添加 Redis 缓存（1ms）
app.get('/api/products/:id', async (req, res) => {
  let product = await redis.get(`product:${req.params.id}`);
  if (!product) {
    product = await db.query('SELECT * FROM products WHERE id = ?', [req.params.id]);
    await redis.setex(`product:${req.params.id}`, 3600, JSON.stringify(product));
  }
  res.json(JSON.parse(product));
});
```

**场景2：网络 I/O 慢（调用第三方 API）**
```javascript
// 问题：串行调用 3 个接口（300ms）
const user = await fetch('http://api.example.com/user/1');
const orders = await fetch('http://api.example.com/orders');
const profile = await fetch('http://api.example.com/profile');

// 解决：并行调用（100ms）
const [user, orders, profile] = await Promise.all([
  fetch('http://api.example.com/user/1'),
  fetch('http://api.example.com/orders'),
  fetch('http://api.example.com/profile')
]);
```

**场景3：Event Loop 阻塞**
```javascript
// 问题：同步 JSON 解析（阻塞 Event Loop）
const data = JSON.parse(largeJsonString);  // 10MB JSON，解析 200ms

// 解决：移到 Worker Thread
const { Worker } = require('worker_threads');
const worker = new Worker(`
  const { parentPort } = require('worker_threads');
  parentPort.on('message', (json) => {
    parentPort.postMessage(JSON.parse(json));
  });
`, { eval: true });

worker.postMessage(largeJsonString);
worker.on('message', (data) => {
  res.json(data);
});
```
