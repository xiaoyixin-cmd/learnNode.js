# 13.5 Git提交规范

> Conventional Commits、Commitlint、Husky、lint-staged、自动化版本发布

---

## 📖 知识点概述

### 核心概念

**Git提交规范**是团队协作的基础，统一的提交信息格式便于追踪变更历史、自动生成changelog、触发CI/CD流程。

### 为什么需要提交规范？

```
❌ 不规范的提交：

git commit -m "fix bug"
git commit -m "update"
git commit -m "修改了一些东西"
git commit -m "aaaaa"

问题：
- 无法从提交信息了解改动内容
- 难以追踪bug来源
- 无法自动生成changelog
- 团队协作混乱

✅ 规范的提交（Conventional Commits）：

git commit -m "feat(auth): add JWT authentication"
git commit -m "fix(api): resolve user creation bug"
git commit -m "docs(readme): update installation guide"

优点：
- 清晰表达改动内容和范围
- 便于代码审查
- 自动生成changelog
- 触发自动化流程（CI/CD）
```

### Conventional Commits格式

```
<type>(<scope>): <subject>

<body>

<footer>

示例：
feat(auth): add JWT authentication

Implement JWT-based authentication system with refresh tokens.
This replaces the previous session-based authentication.

BREAKING CHANGE: Session-based authentication is no longer supported.
Closes #123
```

### Type类型

```
┌─────────────────────────────────────────┐
│        Commit Type 说明                 │
├─────────────────────────────────────────┤
│                                          │
│  feat      新功能                        │
│  fix       Bug修复                       │
│  docs      文档更新                      │
│  style     代码格式（不影响逻辑）        │
│  refactor  重构（不是新功能或bug修复）   │
│  perf      性能优化                      │
│  test      测试相关                      │
│  build     构建系统或外部依赖            │
│  ci        CI配置文件和脚本              │
│  chore     其他不修改src或test的变更     │
│  revert    回滚之前的commit              │
│                                          │
└─────────────────────────────────────────┘
```

## 🚀 使用指南

### 实践1：配置Commitlint

```bash
# 安装依赖
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

```javascript
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  
  rules: {
    // 💡 Type必须是以下之一
    'type-enum': [
      2,
      'always',
      [
        'feat',     // 新功能
        'fix',      // Bug修复
        'docs',     // 文档
        'style',    // 格式
        'refactor', // 重构
        'perf',     // 性能优化
        'test',     // 测试
        'build',    // 构建
        'ci',       // CI
        'chore',    // 其他
        'revert'    // 回滚
      ]
    ],
    
    // 💡 Type不能为空
    'type-empty': [2, 'never'],
    
    // 💡 Subject不能为空
    'subject-empty': [2, 'never'],
    
    // 💡 Subject不能以.结尾
    'subject-full-stop': [2, 'never', '.'],
    
    // 💡 Subject大小写（关闭，允许任意大小写）
    'subject-case': [0],
    
    // 💡 Header最大长度
    'header-max-length': [2, 'always', 100],
    
    // 💡 Scope可以为空
    'scope-empty': [0],
    
    // 💡 自定义Scope范围
    'scope-enum': [
      2,
      'always',
      [
        'auth',      // 认证
        'user',      // 用户
        'api',       // API
        'database',  // 数据库
        'config',    // 配置
        'deps',      // 依赖
        'core',      // 核心
        'ui',        // 界面
        'test'       // 测试
      ]
    ]
  }
};

// 测试配置
// echo "feat(auth): add login" | npx commitlint
// ✅ 通过

// echo "add login" | npx commitlint
// ❌ 失败：type不能为空
```

### 实践3：配置lint-staged

```bash
# 安装lint-staged
npm install --save-dev lint-staged
```

```json
// package.json
{
  "lint-staged": {
    // 💡 TypeScript/JavaScript文件
    "*.{js,ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    
    // 💡 JSON/Markdown文件
    "*.{json,md}": [
      "prettier --write"
    ],
    
    // 💡 CSS文件
    "*.{css,scss}": [
      "prettier --write"
    ]
  },
  
  "scripts": {
    "prepare": "husky",
    "lint": "eslint src --ext .ts,.tsx",
    "format": "prettier --write \"src/**/*.{ts,tsx,json,md}\""
  }
}

// 💡 工作流程
// 1. git add .
// 2. git commit -m "feat: add feature"
// 3. pre-commit hook触发lint-staged
// 4. 自动执行eslint --fix和prettier --write
// 5. commit-msg hook触发commitlint验证
// 6. 提交成功
```

### 实践6：自动生成Changelog

```bash
# 安装standard-version
npm install --save-dev standard-version
```

```json
// package.json
{
  "scripts": {
    "release": "standard-version",
    "release:minor": "standard-version --release-as minor",
    "release:major": "standard-version --release-as major",
    "release:patch": "standard-version --release-as patch"
  }
}

// .versionrc.json
{
  "types": [
    {"type": "feat", "section": "✨ Features"},
    {"type": "fix", "section": "🐛 Bug Fixes"},
    {"type": "perf", "section": "⚡ Performance"},
    {"type": "refactor", "section": "♻️ Refactoring"},
    {"type": "docs", "section": "📝 Documentation"},
    {"type": "style", "hidden": true},
    {"type": "test", "hidden": true},
    {"type": "build", "hidden": true},
    {"type": "ci", "hidden": true},
    {"type": "chore", "hidden": true}
  ]
}

// 使用
// npm run release
// 
// 自动执行：
// 1. 读取所有commit
// 2. 根据commit类型确定版本号（feat=minor, fix=patch）
// 3. 更新package.json版本号
// 4. 生成CHANGELOG.md
// 5. 创建git tag
// 6. 提交变更
```

生成的CHANGELOG.md示例：
```markdown
# Changelog

## [1.2.0](https://github.com/user/repo/compare/v1.1.0...v1.2.0) (2025-01-01)

### ✨ Features

* **auth:** add JWT authentication ([abc1234](https://github.com/user/repo/commit/abc1234))
* **user:** add user profile page ([def5678](https://github.com/user/repo/commit/def5678))

### 🐛 Bug Fixes

* **api:** resolve CORS issue ([ghi9012](https://github.com/user/repo/commit/ghi9012))
* **user:** fix email validation ([jkl3456](https://github.com/user/repo/commit/jkl3456))
```

## 🔧 错误处理

### 高频错误1：提交被拒绝

**现象：**
```bash
$ git commit -m "add feature"
⧗   input: add feature
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]
```

**解决方案：**
```bash
# ✅ 正确格式
git commit -m "feat: add user profile feature"
git commit -m "fix: resolve login bug"

# 💡 使用commitizen（交互式）
npm run commit

# 💡 临时跳过验证（不推荐）
git commit -m "add feature" --no-verify
```

### 高频错误2：Husky hooks不生效

**问题：**
```bash
# commit没有触发commitlint
git commit -m "wrong format"
# 提交成功（不应该）
```

**排查步骤：**
```bash
# 1. 检查.husky目录是否存在
ls -la .husky

# 2. 检查hook文件权限
ls -la .husky/commit-msg
# 应该有执行权限（-rwxr-xr-x）

# 3. 重新初始化Husky
rm -rf .husky
npx husky init

# 4. 重新创建hook
echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg
chmod +x .husky/commit-msg

# 5. 测试
git commit -m "test" --allow-empty
```

### 高频错误3：lint-staged卡住

**现象：**
```bash
$ git commit
⚠ Some of your tasks use `git add` command. Please remove it...
```

**解决方案：**
```json
// ❌ 错误：不要在lint-staged命令中使用git add
{
  "lint-staged": {
    "*.js": [
      "eslint --fix",
      "git add"  // ❌ 移除这行
    ]
  }
}

// ✅ 正确：lint-staged会自动add修改后的文件
{
  "lint-staged": {
    "*.js": [
      "eslint --fix",
      "prettier --write"
    ]
  }
}
```

## 📚 应用场景

### 场景：完整的Git工作流

```bash
# 项目结构
.
├── .husky/
│   ├── commit-msg
│   └── pre-commit
├── .versionrc.json
├── commitlint.config.js
├── .gitmessage
└── package.json

# commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [
      2,
      'always',
      ['feat', 'fix', 'docs', 'style', 'refactor', 'perf', 'test', 'build', 'ci', 'chore', 'revert']
    ],
    'scope-enum': [
      2,
      'always',
      ['auth', 'user', 'api', 'database', 'config', 'deps', 'core', 'ui', 'test']
    ],
    'subject-case': [0]
  }
};

# package.json
{
  "scripts": {
    "prepare": "husky",
    "commit": "cz",
    "lint": "eslint src --ext .ts,.tsx",
    "format": "prettier --write \"src/**/*.{ts,tsx,json,md}\"",
    "release": "standard-version",
    "release:minor": "standard-version --release-as minor",
    "release:major": "standard-version --release-as major"
  },
  
  "lint-staged": {
    "*.{js,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  },
  
  "config": {
    "commitizen": {
      "path": "cz-conventional-changelog"
    }
  }
}

# 工作流程
# 1. 开发功能
git checkout -b feature/user-profile

# 2. 提交代码（交互式）
git add .
npm run commit
# 选择type: feat
# 输入scope: user
# 输入subject: add user profile page

# 3. pre-commit自动执行
# - lint-staged运行eslint --fix
# - lint-staged运行prettier --write
# - 自动git add修改后的文件

# 4. commit-msg验证
# - commitlint验证提交信息格式
# - 格式正确，提交成功

# 5. 推送并创建PR
git push origin feature/user-profile

# 6. 合并到main后，发布新版本
git checkout main
git pull
npm run release  # 自动生成changelog和tag

# 7. 推送tag触发CI/CD
git push --follow-tags origin main
```

## 📋 总结

### 核心要点

1. **Conventional Commits**：统一提交格式
2. **Commitlint**：验证提交信息
3. **Husky**：Git Hooks管理
4. **lint-staged**：提交前代码检查

### 实现关键

1. **commitlint.config.js**: 配置提交规则
2. **.husky/commit-msg**: 触发commitlint
3. **.husky/pre-commit**: 触发lint-staged
4. **standard-version**: 自动生成changelog

### 应用价值

- **规范化**：统一团队提交风格
- **可追溯**：清晰的变更历史
- **自动化**：自动生成changelog和版本号
- **质量保证**：提交前自动检查代码

### 面试重点

**Q1：什么是Conventional Commits？**
答：一种提交信息规范，格式为`<type>(<scope>): <subject>`。type表示变更类型（feat/fix等），scope表示影响范围，subject是简短描述。

**Q2：如何强制执行提交规范？**
答：1) Commitlint验证提交信息 2) Husky在commit-msg hook触发 3) 格式不正确时拒绝提交。

**Q3：lint-staged的作用是什么？**
答：只对暂存区（staged）的文件执行lint和format，提高效率，确保提交的代码符合规范。

**Q4：如何自动生成changelog？**
答：使用standard-version，根据Conventional Commits自动生成changelog，更新版本号，创建git tag。

### 注意事项

⚠️ **Type必须小写**：feat不是Feat
⚠️ **Subject简洁**：不超过50字符
⚠️ **Breaking Change**：用BREAKING CHANGE标注
⚠️ **团队统一**：全员遵守规范
