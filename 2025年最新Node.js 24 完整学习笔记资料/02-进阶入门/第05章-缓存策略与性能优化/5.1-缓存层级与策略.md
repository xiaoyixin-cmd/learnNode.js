# 5.1 缓存层级与策略

> 理解缓存的层级结构和常见缓存策略

---

## 📖 知识点概述

### 核心概念

**缓存（Cache）**：将数据存储在高速存储介质中，减少对慢速存储（如数据库）的访问，提升系统性能。

### 为什么需要缓存？

- **减少延迟**：内存访问（ns级）比磁盘（ms级）快1000倍
- **降低负载**：减少数据库查询次数
- **提升并发**：支持更多并发请求
- **节省成本**：减少服务器资源消耗

### 缓存的代价

- **一致性问题**：缓存和数据库可能不一致
- **内存占用**：需要额外内存空间
- **复杂度增加**：需要考虑缓存更新、失效等
- **冷启动慢**：缓存为空时性能差

---

## 🏗️ 缓存层级

### 完整缓存架构

```
用户请求
    ↓
[1] 浏览器缓存（Browser Cache）
    ↓ 未命中
[2] CDN缓存（边缘节点）
    ↓ 未命中
[3] 负载均衡器
    ↓
[4] 应用服务器本地缓存（Memory Cache）
    ↓ 未命中
[5] 分布式缓存（Redis）
    ↓ 未命中
[6] 数据库查询缓存（MySQL Query Cache）
    ↓ 未命中
[7] 数据库磁盘存储
```

### 各层级特点

| 层级 | 存储位置 | 速度 | 容量 | 作用域 | 适用场景 |
|------|---------|------|------|--------|---------|
| **浏览器缓存** | 客户端磁盘 | 快 | 小（MB级） | 单用户 | 静态资源 |
| **CDN缓存** | 边缘节点 | 很快 | 中（GB级） | 区域用户 | 静态资源、API |
| **内存缓存** | 应用服务器内存 | 极快 | 小（GB级） | 单机 | 热点数据 |
| **Redis缓存** | 独立服务器内存 | 快 | 大（GB-TB级） | 集群共享 | 会话、热点数据 |
| **数据库缓存** | 数据库内存 | 中 | 中 | 数据库级 | 查询结果 |

## 📋 缓存策略

### 1. Cache-Aside（旁路缓存）

**模式**：应用程序直接管理缓存

**流程**：
```javascript
// 读取流程
async function getData(key) {
  // 1. 先查缓存
  let data = await cache.get(key);
  
  if (data) {
    return data; // 缓存命中
  }
  
  // 2. 缓存未命中，查询数据库
  data = await db.query(key);
  
  // 3. 写入缓存
  await cache.set(key, data, 3600); // 1小时过期
  
  return data;
}

// 写入流程
async function updateData(key, value) {
  // 1. 更新数据库
  await db.update(key, value);
  
  // 2. 删除缓存（而非更新）
  await cache.del(key);
  
  // 💡 为什么删除而不是更新？
  // - 避免并发更新导致缓存不一致
  // - 下次读取时自动更新缓存
}
```

**优点**：
- ✅ 简单直观，应用完全控制
- ✅ 缓存失败不影响数据库操作

**缺点**：
- ❌ 首次访问需要查询数据库（冷启动慢）
- ❌ 缓存和数据库可能短暂不一致

**适用场景**：
- 读多写少的场景
- 大部分Web应用

### 2. Read-Through（读穿透）

**模式**：缓存层自动加载数据

```javascript
// 💡 缓存层封装了数据加载逻辑
class ReadThroughCache {
  constructor(cache, dataLoader) {
    this.cache = cache;
    this.dataLoader = dataLoader; // 数据加载函数
  }
  
  async get(key) {
    // 缓存层自动处理缓存未命中
    let data = await this.cache.get(key);
    
    if (!data) {
      // 自动加载数据
      data = await this.dataLoader(key);
      await this.cache.set(key, data);
    }
    
    return data;
  }
}

// 使用
const cache = new ReadThroughCache(redisCache, (key) => db.query(key));
const user = await cache.get('user:123');
```

**优点**：
- ✅ 应用代码更简洁
- ✅ 缓存逻辑集中管理

**缺点**：
- ❌ 缓存层需要知道如何加载数据

**适用场景**：
- 缓存系统统一管理
- 多个应用共享缓存逻辑

### 3. Write-Through（写穿透）

**模式**：写操作同时更新缓存和数据库

```javascript
async function updateData(key, value) {
  // 1. 同时更新缓存和数据库
  await Promise.all([
    cache.set(key, value),
    db.update(key, value)
  ]);
  
  // 💡 保证缓存和数据库一致性
}
```

**优点**：
- ✅ 缓存始终是最新的
- ✅ 读操作一定命中缓存

**缺点**：
- ❌ 写操作延迟增加（两次写入）
- ❌ 浪费缓存空间（即使数据不被读取）

**适用场景**：
- 强一致性要求
- 写入后立即读取

### 4. Write-Behind（写回）

**模式**：先更新缓存，异步批量写入数据库

```javascript
class WriteBehindCache {
  constructor(cache, db) {
    this.cache = cache;
    this.db = db;
    this.dirtyKeys = new Set(); // 待同步的key
    
    // 💡 定期批量同步
    setInterval(() => this.flush(), 5000);
  }
  
  async set(key, value) {
    // 1. 立即更新缓存
    await this.cache.set(key, value);
    
    // 2. 标记为脏数据
    this.dirtyKeys.add(key);
  }
  
  async flush() {
    if (this.dirtyKeys.size === 0) return;
    
    // 3. 批量写入数据库
    const keys = Array.from(this.dirtyKeys);
    const values = await this.cache.mget(keys);
    
    await this.db.batchUpdate(
      keys.map((key, i) => ({ key, value: values[i] }))
    );
    
    this.dirtyKeys.clear();
  }
}
```

**优点**：
- ✅ 写操作极快（只写缓存）
- ✅ 批量写入，减少数据库压力
- ✅ 支持更高的写并发

**缺点**：
- ❌ 数据可能丢失（缓存宕机）
- ❌ 缓存和数据库短暂不一致

**适用场景**：
- 高并发写入
- 允许短暂数据丢失（如点赞数、浏览量）

## 🎯 实战：完整缓存示例

### 场景：用户信息缓存

```javascript
// user-cache.js
import { createClient } from 'redis';
import pool from './db.js'; // MySQL连接池

const redis = createClient({ url: 'redis://localhost:6379' });
await redis.connect();

// 💡 缓存管理类
class UserCache {
  constructor() {
    this.prefix = 'user:';
    this.ttl = 3600; // 1小时
  }
  
  // 获取用户（Cache-Aside策略）
  async getUser(userId) {
    const key = `${this.prefix}${userId}`;
    
    // 1. 查询缓存
    const cached = await redis.get(key);
    if (cached) {
      console.log('缓存命中');
      return JSON.parse(cached);
    }
    
    console.log('缓存未命中，查询数据库');
    
    // 2. 查询数据库
    const [rows] = await pool.query(
      'SELECT id, username, email, avatar FROM users WHERE id = ?',
      [userId]
    );
    
    const user = rows[0];
    if (!user) return null;
    
    // 3. 写入缓存
    await redis.setEx(key, this.ttl, JSON.stringify(user));
    
    return user;
  }
  
  // 更新用户
  async updateUser(userId, updates) {
    // 1. 更新数据库
    await pool.query(
      'UPDATE users SET ? WHERE id = ?',
      [updates, userId]
    );
    
    // 2. 删除缓存
    const key = `${this.prefix}${userId}`;
    await redis.del(key);
    
    console.log('用户已更新，缓存已失效');
  }
  
  // 批量获取用户（优化N+1问题）
  async getUsers(userIds) {
    const keys = userIds.map(id => `${this.prefix}${id}`);
    
    // 1. 批量查询缓存
    const cached = await redis.mGet(keys);
    
    const users = [];
    const missIds = [];
    
    cached.forEach((data, index) => {
      if (data) {
        users[index] = JSON.parse(data);
      } else {
        missIds.push(userIds[index]);
      }
    });
    
    // 2. 查询未命中的用户
    if (missIds.length > 0) {
      const [rows] = await pool.query(
        'SELECT id, username, email, avatar FROM users WHERE id IN (?)',
        [missIds]
      );
      
      // 3. 写入缓存
      const pipeline = redis.multi();
      rows.forEach(user => {
        const key = `${this.prefix}${user.id}`;
        pipeline.setEx(key, this.ttl, JSON.stringify(user));
        
        const index = userIds.indexOf(user.id);
        users[index] = user;
      });
      
      await pipeline.exec();
    }
    
    return users;
  }
}

export default new UserCache();
```

### 使用示例

```javascript
// app.js
import userCache from './user-cache.js';

// 获取单个用户
const user = await userCache.getUser(123);
console.log('用户:', user);

// 更新用户
await userCache.updateUser(123, { username: 'newname' });

// 批量获取用户
const users = await userCache.getUsers([1, 2, 3, 4, 5]);
console.log('用户列表:', users);
```

### 性能对比

```javascript
// benchmark.js
import userCache from './user-cache.js';

async function benchmark() {
  const userId = 123;
  
  // 💡 无缓存
  console.time('无缓存-100次查询');
  for (let i = 0; i < 100; i++) {
    await pool.query('SELECT * FROM users WHERE id = ?', [userId]);
  }
  console.timeEnd('无缓存-100次查询');
  // 输出：~500ms
  
  // 💡 有缓存

}
```

## 📊 缓存决策树

```
是否需要缓存？
├─ 数据很少变化？ → 是 → 长TTL缓存（24小时+）
├─ 数据频繁访问？ → 是 → 短TTL缓存（5-30分钟）
├─ 数据实时性要求高？ → 是 → 不缓存或极短TTL（秒级）
└─ 数据访问量小？ → 是 → 不需要缓存

选择哪种缓存？
├─ 静态资源？ → CDN + 浏览器缓存
├─ 用户会话？ → Redis（分布式）
├─ 热点数据？ → 内存缓存 + Redis
├─ 计算结果？ → 内存缓存
└─ API响应？ → HTTP缓存头

选择哪种策略？
├─ 读多写少？ → Cache-Aside（推荐）
├─ 强一致性？ → Write-Through
├─ 高并发写入？ → Write-Behind
└─ 统一缓存逻辑？ → Read-Through
```

## 📋 总结

### 核心要点

1. **缓存层级**：浏览器→CDN→应用→Redis→数据库
2. **Cache-Aside**：最常用策略，应用管理缓存
3. **Write-Through**：强一致性，同时更新
4. **Write-Behind**：高性能，异步写入

### 实现关键

1. **Cache-Aside**：先查缓存，未命中查数据库，再写缓存
2. **更新时删除缓存**：避免并发问题
3. **批量操作**：使用mget/mset优化性能
4. **设置TTL**：避免缓存永久占用内存

### 应用价值

- **性能提升**：10-100倍性能提升
- **成本降低**：减少数据库负载
- **用户体验**：响应更快

### 面试重点

1. **什么是Cache-Aside？**
   - 应用管理缓存，先查缓存，未命中查数据库

2. **为什么更新时删除缓存而非更新？**
   - 避免并发更新导致不一致

3. **如何选择缓存策略？**
   - 读多写少：Cache-Aside
   - 强一致性：Write-Through
   - 高并发写：Write-Behind

4. **缓存层级有哪些？**
   - 浏览器、CDN、应用内存、Redis、数据库

### 注意事项

⚠️ **设置合理的TTL**：
```javascript
// 根据数据特点设置TTL
await redis.setEx('hot:data', 60, value);    // 热点数据：1分钟
await redis.setEx('user:info', 3600, value); // 用户信息：1小时
await redis.setEx('static:data', 86400, value); // 静态数据：1天
```

⚠️ **更新时删除缓存**：
```javascript
// ✅ 正确
await db.update(key, value);
await cache.del(key);

// ❌ 错误
await db.update(key, value);
await cache.set(key, value); // 可能导致不一致
```
