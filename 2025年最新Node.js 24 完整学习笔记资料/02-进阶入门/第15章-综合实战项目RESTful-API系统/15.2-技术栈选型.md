# 15.2 技术栈选型：Express + MongoDB + Redis + JWT

## 知识点概述

### 核心概念

技术栈选型是项目启动前最重要的决策之一。合适的技术栈能让开发事半功倍，不合适的选择会导致后期重构成本高昂。本节将从实战角度讲解：

- **如何选择 Web 框架**：Express vs Fastify vs Koa
- **数据库选型依据**：MongoDB vs MySQL vs PostgreSQL
- **缓存方案对比**：Redis vs Memcached vs Node-cache
- **认证方案**：JWT vs Session vs OAuth

### 为什么技术选型如此重要？

假设选错了技术栈：
- ❌ 选择了不支持 TypeScript 的库，后期迁移成本高
- ❌ 选择了性能差的框架，上线后发现无法支撑流量
- ❌ 选择了社区不活跃的库，遇到 Bug 无人维护

**正确的技术选型**：
- ✅ 团队熟悉度高，开发效率快
- ✅ 社区活跃，遇到问题容易找到解决方案
- ✅ 性能满足需求，成本可控

### 第二步：准备 - 技术栈对比分析

#### 1️⃣ Web 框架选型

##### Express vs Fastify vs Koa

| 框架       | 性能（QPS） | 学习曲线 | 社区活跃度 | 推荐场景             |
|------------|-------------|----------|------------|----------------------|
| **Express**| 15k         | ⭐       | ⭐⭐⭐⭐⭐    | 快速开发、中小型项目 |
| **Fastify**| 48k         | ⭐⭐⭐    | ⭐⭐⭐⭐     | 高性能、微服务       |
| **Koa**    | 42k         | ⭐⭐     | ⭐⭐⭐⭐     | 定制化需求高的项目   |

**性能对比**（简单 JSON 响应）：

```bash
# Express
wrk -t10 -c100 -d10s http://localhost:3000/api/hello
# QPS: 15,234

# Fastify
wrk -t10 -c100 -d10s http://localhost:3000/api/hello
# QPS: 48,521

# Koa
wrk -t10 -c100 -d10s http://localhost:3000/api/hello
# QPS: 42,189
```

**选型结论**：
- 🚀 **本项目选择 Express**（理由如下）
  - 社区最活跃（GitHub 60k+ Star，NPM 周下载 2000 万+）
  - 中间件生态完善（body-parser、cors、helmet 等）
  - 学习曲线低，适合教学
  - 性能满足需求（博客系统 QPS 预计 < 5000）

#### 2️⃣ 数据库选型

##### MongoDB vs MySQL vs PostgreSQL

| 数据库     | 类型     | 适用场景                 | 优势                 | 劣势                 |
|------------|----------|--------------------------|----------------------|----------------------|
| **MongoDB**| 文档型   | CMS、日志、社交动态      | Schema 灵活、水平扩展 | 事务支持弱（4.0前）  |
| **MySQL**  | 关系型   | 电商、金融、ERP          | 成熟稳定、ACID 强    | Schema 变更成本高    |
| **PostgreSQL** | 关系型 | 地理数据、全文搜索      | 功能最全、支持 JSON  | 学习曲线陡           |

**本项目选择 MongoDB**（理由）：

```javascript
// 文章内容是非结构化的 Markdown
{
  title: "如何学习 Node.js",
  content: "# 第一章\n\n这是一篇很长的 Markdown 内容...",  // 长度不固定
  tags: ["Node.js", "教程"],  // 数组字段
  metadata: {  // 嵌套对象
    readingTime: 10,
    difficulty: "beginner"
  }
}

// 💡 MongoDB 天然支持这种结构，MySQL 需要额外的 JSON 字段或多表关联
```

**Schema 灵活性对比**：

```javascript
// MongoDB：新增字段无需迁移数据
await Post.updateOne(
  { _id: postId },
  { $set: { newField: 'value' } }
);

// MySQL：需要 ALTER TABLE（锁表，大表耗时数小时）
ALTER TABLE posts ADD COLUMN new_field VARCHAR(255);
```

#### 3️⃣ 缓存选型

##### Redis vs Memcached vs Node-cache

| 方案         | 数据结构 | 持久化 | 性能   | 适用场景             |
|--------------|----------|--------|--------|----------------------|
| **Redis**    | 丰富     | 支持   | 高     | Session、排行榜、消息队列 |
| **Memcached**| 简单     | 不支持 | 极高   | 纯缓存（如页面缓存） |
| **Node-cache**| 简单    | 不支持 | 最高   | 单机内存缓存         |

**本项目选择 Redis**（理由）：

```javascript
// 使用场景1：缓存热点文章（String）
await redis.setex(`post:${postId}`, 3600, JSON.stringify(post));

// 使用场景2：文章点赞数（Hash）
await redis.hincrby(`post:${postId}:stats`, 'likes', 1);

// 使用场景3：热门文章排行榜（Sorted Set）
await redis.zadd('posts:trending', Date.now(), postId);

// 使用场景4：Session 存储（String）
await redis.setex(`session:${sessionId}`, 86400, JSON.stringify(user));
```

**性能对比**（10万次 GET 操作）：

```bash
# Redis
Time: 0.8s (125,000 ops/s)

# Memcached
Time: 0.6s (166,667 ops/s)

# Node-cache
Time: 0.1s (1,000,000 ops/s)
```

💡 **选型理由**：虽然 Redis 性能不如 Memcached，但数据结构丰富，支持持久化，更适合复杂业务。

#### 4️⃣ 认证方案选型

##### JWT vs Session vs OAuth

| 方案      | 状态存储 | 扩展性 | 安全性 | 适用场景             |
|-----------|----------|--------|--------|----------------------|
| **JWT**   | 无状态   | ⭐⭐⭐⭐⭐ | ⭐⭐⭐   | 微服务、移动端 API   |
| **Session**| 有状态  | ⭐⭐   | ⭐⭐⭐⭐⭐ | 传统 Web 应用        |
| **OAuth** | 无状态   | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 第三方登录           |

**本项目选择 JWT**（理由）：

```javascript
// 优势1：无状态（不依赖 Redis/数据库）
const token = jwt.sign({ userId: user._id, role: user.role }, SECRET, { expiresIn: '7d' });

// 优势2：跨域友好（适合前后端分离）
res.json({ token }); // 前端存储到 localStorage，每次请求携带

// 优势3：水平扩展（任意服务器都能验证）
const payload = jwt.verify(token, SECRET); // 无需查询数据库
```

**JWT 的问题与解决方案**：

```javascript
// 问题1：Token 无法主动失效（如用户退出登录）
// 解决：引入 Token 黑名单（Redis）
await redis.setex(`blacklist:${token}`, 3600, '1');

// 验证时检查黑名单
if (await redis.exists(`blacklist:${token}`)) {
  throw new Error('Token 已失效');
}

// 问题2：Token 被盗用
// 解决：刷新 Token 机制 + HTTPS
// 短期 Access Token（15分钟）+ 长期 Refresh Token（7天）
```

**`src/config/database.js`** - MongoDB 连接

```javascript
const mongoose = require('mongoose');
const logger = require('../utils/logger');

const connectDB = async () => {
  try {
    const conn = await mongoose.connect(process.env.MONGODB_URI, {
      // Mongoose 8.x 不再需要这些选项（已默认启用）
      // useNewUrlParser: true,
      // useUnifiedTopology: true
    });

    logger.info(`MongoDB Connected: ${conn.connection.host}`);
    
    // 监听连接事件
    mongoose.connection.on('disconnected', () => {
      logger.warn('MongoDB disconnected');
    });

    mongoose.connection.on('error', (err) => {
      logger.error('MongoDB connection error:', err);
    });

  } catch (error) {
    logger.error('MongoDB connection failed:', error);
    process.exit(1);
  }
};

module.exports = connectDB;
```

## 3个高频错误

### 错误1：依赖版本不兼容导致项目无法启动

**现象**：`npm install` 后启动报错

```bash
Error: Cannot find module 'mongoose'
  at Function.Module._resolveFilename (internal/modules/cjs/loader.js:880:15)
```

**原因**：`package.json` 中版本号使用了 `^` 或 `~`，导致安装了不兼容版本

**解决方案**：

```json
// ❌ 错误：使用宽松版本范围
{
  "dependencies": {
    "mongoose": "^7.0.0"  // 可能安装 7.x 的任意版本
  }
}

// ✅ 正确：锁定主版本
{
  "dependencies": {
    "mongoose": "~8.0.3"  // 只允许 8.0.x 的补丁版本
  }
}

// ✅ 更好：使用 package-lock.json
npm ci  # 严格按照 package-lock.json 安装
```

### 错误2：MongoDB 连接字符串配置错误

**现象**：应用启动时报错 `MongoServerError: Authentication failed`

**原因**：连接字符串格式错误或密码包含特殊字符未编码

**解决方案**：

```bash
# ❌ 错误：密码包含特殊字符未编码
MONGODB_URI=mongodb://user:p@ssw0rd@localhost:27017/blog-api
# @ 符号会被误解析为 host 分隔符

# ✅ 正确：使用 encodeURIComponent 编码密码
MONGODB_URI=mongodb://user:p%40ssw0rd@localhost:27017/blog-api
# %40 是 @ 的编码

# ✅ 或使用 SRV 连接字符串（MongoDB Atlas）
MONGODB_URI=mongodb+srv://user:password@cluster.mongodb.net/blog-api?retryWrites=true&w=majority
```

**在代码中生成正确的连接字符串**：

```javascript
function buildMongoURI(config) {
  const { user, password, host, port, database } = config;
  const encodedPassword = encodeURIComponent(password);
  return `mongodb://${user}:${encodedPassword}@${host}:${port}/${database}`;
}

const uri = buildMongoURI({
  user: 'admin',
  password: 'p@ssw0rd!',
  host: 'localhost',
  port: 27017,
  database: 'blog-api'
});
// mongodb://admin:p%40ssw0rd%21@localhost:27017/blog-api
```

### 错误3：CORS 配置错误导致前端无法调用接口

**现象**：浏览器控制台报错

```
Access to XMLHttpRequest at 'http://localhost:3000/api/posts' from origin 'http://localhost:5173'
has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

**原因**：未配置 CORS 或配置不正确

**解决方案**：

```javascript
// ❌ 错误：CORS 配置过于宽松（生产环境不安全）
app.use(cors());  // 允许所有来源

// ✅ 开发环境：允许特定来源
app.use(cors({
  origin: ['http://localhost:5173', 'http://localhost:3000'],
  credentials: true  // 允许携带 Cookie
}));

// ✅ 生产环境：从环境变量读取
app.use(cors({
  origin: (origin, callback) => {
    const allowedOrigins = process.env.ALLOWED_ORIGINS?.split(',') || [];
    
    // 允许同源请求（如 Postman）
    if (!origin) return callback(null, true);
    
    if (allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  },
  credentials: true
}));
```

## 总结

### ✅ 核心要点

1. **性能优先**：高 QPS 场景选 Fastify，中小型项目 Express 足够
2. **Schema 灵活性**：文档型数据选 MongoDB，关系型数据选 MySQL
3. **缓存分层**：热点数据用 Redis，极热数据用 Node-cache

### 🔑 实现关键

1. **版本锁定**：使用 `package-lock.json` 避免依赖版本不一致
2. **环境变量**：敏感信息（如数据库密码）使用 `.env` 文件管理
3. **优雅关闭**：监听 `SIGTERM` 信号，关闭数据库连接后再退出
4. **健康检查**：暴露 `/health` 端点，方便运维监控

### 💎 应用价值

- **减少试错成本**：提前对比技术方案，避免选型失误
- **提升开发效率**：选择团队熟悉的技术栈，减少学习成本
- **保障系统稳定**：选择社区活跃的库，遇到问题容易找到解决方案

### 🎯 面试重点

- **Q1**：为什么选择 MongoDB 而不是 MySQL？  
  **A**：文章内容是非结构化的 Markdown，MongoDB 的 Schema 灵活性更适合快速迭代。

- **Q2**：JWT 和 Session 有什么区别？  
  **A**：JWT 无状态（不依赖服务器存储），适合微服务；Session 有状态（需 Redis 存储），更安全。

- **Q3**：如何选择 Web 框架？  
  **A**：性能要求高选 Fastify，快速开发选 Express，定制化需求高选 Koa。

### ⚠️ 注意事项

1. **不要盲目追求新技术**：稳定性比新特性更重要
2. **避免过度工程化**：MVP 阶段先用简单方案，后续再优化
3. **关注社区活跃度**：GitHub Star 少、长期未更新的库慎用

### 🚀 进阶思考

- **微服务架构**：当单体应用无法支撑流量时，拆分为多个服务
- **Serverless**：AWS Lambda + MongoDB Atlas，零运维成本
- **GraphQL**：替代 RESTful API，减少接口数量

## 🤔 常见问题（FAQ）

### Q1：为什么选择 Express 而不是 Fastify？

**A**：Express 成熟度更高，适合学习：

| 框架     | 优势                     | 劣势                 | 适用场景         |
|----------|--------------------------|----------------------|------------------|
| Express  | 生态最强、资料最多       | 性能较低             | 学习、中小项目   |
| Fastify  | 性能最强、TypeScript友好 | 生态较小、学习曲线陡 | 高性能项目       |
| Koa      | 中间件优雅、轻量         | 生态少、需手动组合   | 定制化需求高     |

**性能对比**：Fastify QPS 比 Express 高 30-50%，但本项目重点是学习完整开发流程，性能差异不明显。

---

### Q2：MongoDB 和 MySQL 该怎么选？

**A**：根据数据特点选择：

**选 MongoDB**：
- 数据结构灵活（如文章内容、评论嵌套）
- 快速迭代（Schema 可随时调整）
- 水平扩展需求

**选 MySQL**：
- 事务要求严格（如电商订单、支付）
- 复杂关联查询（多表 JOIN）
- 数据一致性优先

**本项目**：博客系统无复杂事务，MongoDB 更合适。

---

### Q3：为什么用 bcrypt 而不是 MD5 哈希密码？

**A**：MD5 不安全，bcrypt 抗暴力破解：

```javascript
// ❌ MD5（已破解，仅需几秒）
const crypto = require('crypto');
const hash = crypto.createHash('md5').update('123456').digest('hex');

// ✅ bcrypt（加盐 + 慢哈希，破解需数年）
const bcrypt = require('bcryptjs');
const hash = await bcrypt.hash('123456', 10);  // 10 rounds
```

**暴力破解时间对比**：
- MD5：数秒
- bcrypt (10 rounds)：数千年

---

### Q4：JWT 的 secret 应该多长？

**A**：至少 256 位（32 字节）：

```javascript
// ❌ 弱密钥（易破解）
const secret = '123456';

// ✅ 强密钥（256位）
const secret = crypto.randomBytes(32).toString('hex');
// 生成：a1b2c3d4e5f6... (64个字符)
```

**生成方法**：
```bash
# 在终端生成随机密钥
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

---

### Q5：为什么需要 dotenv 而不是直接写在代码里？

**A**：避免敏感信息泄露到 Git：

```javascript
// ❌ 硬编码（危险）
const mongoUri = 'mongodb://admin:password123@localhost:27017';

// ✅ 环境变量（安全）
require('dotenv').config();
const mongoUri = process.env.MONGO_URI;
```

**最佳实践**：
1. 创建 `.env` 文件存储密钥
2. 添加 `.env` 到 `.gitignore`
3. 提供 `.env.example` 作为模板
