# 15.1 项目需求分析与数据库设计

## 知识点概述

### 核心概念

需求分析与数据库设计是项目开发的第一步，也是最关键的一步。好的设计能让后续开发事半功倍，差的设计会导致频繁重构。本节将带你：

- **需求分析**：从产品需求到技术方案的转化
- **功能拆分**：如何拆解复杂功能为可实现的子模块
- **数据库设计**：表结构设计、关系建模、索引规划
- **技术评审**：设计方案的可行性验证

### 为什么需求分析和数据库设计如此重要？

假设直接开始写代码（跳过设计阶段）：
- ❌ 写到一半发现数据表关系设计错误，需要大量迁移数据
- ❌ 功能遗漏，上线后频繁添加字段，导致表结构混乱
- ❌ 性能问题（如N+1查询），后期优化成本高

**有了需求分析和数据库设计**：
- ✅ 提前发现技术难点，规避风险
- ✅ 数据表关系清晰，减少联表查询
- ✅ 索引提前规划，保障查询性能

## 使用指南

### 第一步：理解项目需求

#### 项目背景

我们要构建一个**博客系统 API**，支持以下核心功能：

**1️⃣ 用户系统**
- 用户注册（邮箱+密码）
- 用户登录（返回 JWT Token）
- 个人资料管理（昵称、头像、简介）
- 角色权限（普通用户、作者、管理员）

**2️⃣ 文章系统**
- 发布文章（标题、内容、分类、标签）
- 编辑/删除文章（仅作者本人或管理员）
- 文章列表（分页、排序、搜索）
- 文章详情（阅读量自动+1）
- 草稿功能（发布前保存草稿）

**3️⃣ 评论系统**
- 发表评论
- 回复评论（嵌套评论，最多3层）
- 点赞评论
- 删除评论（仅评论者本人或管理员）

**4️⃣ 互动功能**
- 文章点赞/收藏
- 关注作者
- 个人主页（展示用户的所有文章）

#### 功能优先级

| 优先级 | 功能模块           | MVP版本 | 完整版本 |

### 第三步：实践 - 数据库设计

#### 方案1：MongoDB 数据库设计（推荐）

**优势**：
- 适合文档型数据（文章内容、评论嵌套）
- Schema 灵活，易于迭代
- 水平扩展能力强

##### 1️⃣ 用户表（users）

```javascript
// User Model (Mongoose Schema)
const UserSchema = new Schema({
  // 基础信息
  email: {
    type: String,
    required: true,
    unique: true,
    lowercase: true,
    trim: true,
    match: /^[\w-]+(\.[\w-]+)*@([\w-]+\.)+[a-zA-Z]{2,7}$/
  },
  password: {
    type: String,
    required: true,
    minlength: 6,
    select: false  // 💡 查询时默认不返回密码字段
  },
  username: {
    type: String,
    required: true,
    unique: true,
    minlength: 3,
    maxlength: 30
  },
  
  // 个人资料
  profile: {
    avatar: {
      type: String,
      default: 'https://cdn.example.com/default-avatar.png'
    },
    bio: {
      type: String,
      maxlength: 200,
      default: ''
    },
    website: String,
    location: String
  },
  
  // 角色权限
  role: {
    type: String,
    enum: ['user', 'author', 'admin'],
    default: 'user'
  },
  
  // 统计信息
  stats: {
    postsCount: { type: Number, default: 0 },
    followersCount: { type: Number, default: 0 },
    followingCount: { type: Number, default: 0 }
  },
  
  // 账号状态
  isActive: {
    type: Boolean,
    default: true
  },
  emailVerified: {
    type: Boolean,
    default: false
  },
  
  // 时间戳
  lastLoginAt: Date,
  createdAt: {
    type: Date,
    default: Date.now
  },
  updatedAt: {
    type: Date,
    default: Date.now
  }
}, {
  timestamps: true  // 自动管理 createdAt 和 updatedAt
});

// 💡 索引设计
UserSchema.index({ email: 1 }, { unique: true });
UserSchema.index({ username: 1 }, { unique: true });
UserSchema.index({ createdAt: -1 });  // 用于"最新注册用户"查询
```

**字段说明**：
- `select: false`：查询用户时默认不返回密码，避免泄露
- `enum`：限制角色字段只能是指定值
- `timestamps: true`：Mongoose 自动添加 createdAt/updatedAt

##### 2️⃣ 文章表（posts）

```javascript
const PostSchema = new Schema({
  // 基础信息
  title: {
    type: String,
    required: true,
    trim: true,
    minlength: 5,
    maxlength: 100
  },
  slug: {
    type: String,
    unique: true,
    lowercase: true
    // 💡 如："how-to-build-restful-api"（用于 SEO 友好的 URL）
  },
  content: {
    type: String,
    required: true,
    minlength: 10
  },
  excerpt: {
    type: String,  // 摘要（用于列表页展示）
    maxlength: 300
  },
  
  // 作者信息
  author: {
    type: Schema.Types.ObjectId,
    ref: 'User',  // 💡 引用 User 表
    required: true,
    index: true
  },
  
  // 分类与标签
  category: {
    type: String,
    enum: ['技术', '生活', '随笔', '教程'],
    default: '技术'
  },
  tags: [{
    type: String,
    trim: true
  }],
  
  // 封面图
  coverImage: {
    type: String,
    default: ''
  },
  
  // 状态
  status: {
    type: String,
    enum: ['draft', 'published', 'archived'],
    default: 'draft'
  },
  
  // 统计信息
  stats: {
    viewsCount: { type: Number, default: 0 },
    likesCount: { type: Number, default: 0 },
    commentsCount: { type: Number, default: 0 },
    favoritesCount: { type: Number, default: 0 }
  },
  
  // SEO 相关
  seo: {
    metaTitle: String,
    metaDescription: String,
    keywords: [String]
  },
  
  // 发布时间
  publishedAt: Date,
  
  // 时间戳
  createdAt: {
    type: Date,
    default: Date.now
  },
  updatedAt: Date
}, {
  timestamps: true
});

// 💡 索引设计
PostSchema.index({ author: 1, createdAt: -1 });  // 复合索引：查询某作者的文章列表
PostSchema.index({ status: 1, publishedAt: -1 }); // 查询已发布文章（按时间倒序）
PostSchema.index({ tags: 1 });                    // 按标签查询
PostSchema.index({ 
  title: 'text',   // 全文索引：搜索标题
  content: 'text'  // 全文索引：搜索内容
});

// 💡 虚拟字段（不存储在数据库，动态计算）
PostSchema.virtual('url').get(function() {
  return `/posts/${this.slug}`;
});
```

**设计要点**：
- **slug 字段**：用于 SEO 友好的 URL（如 `/posts/how-to-build-api` 而非 `/posts/507f1f77bcf86cd799439011`）
- **author 引用**：使用 ObjectId 引用 User 表，支持 `.populate('author')` 联表查询
- **全文索引**：支持 `$text` 搜索（中文需特殊配置）
- **复合索引**：`{ author: 1, createdAt: -1 }` 可快速查询"某作者的最新文章"

##### 3️⃣ 评论表（comments）

```javascript
const CommentSchema = new Schema({
  // 评论内容
  content: {
    type: String,
    required: true,
    trim: true,
    minlength: 1,
    maxlength: 500

  },
  
  // 回复层级（限制最多3层）
  level: {
    type: Number,
    default: 1,
    min: 1,
    max: 3
  },
  
  // 统计信息
  likesCount: {
    type: Number,
    default: 0
  },
  
  // 状态
  isDeleted: {
    type: Boolean,
    default: false  // 软删除（不真正删除，仅标记）
  },
  
  // 时间戳
  createdAt: {
    type: Date,
    default: Date.now
  }
}, {
  timestamps: true
});

// 💡 索引设计
CommentSchema.index({ post: 1, createdAt: -1 });     // 查询某文章的所有评论
CommentSchema.index({ author: 1 });                   // 查询某用户的所有评论
CommentSchema.index({ parentComment: 1 });            // 查询某评论的所有回复
```

**嵌套评论设计说明**：

```
评论1（ID: c1, level: 1, parentComment: null）
  └─ 评论2（ID: c2, level: 2, parentComment: c1）
      └─ 评论3（ID: c3, level: 3, parentComment: c2）
          └─ 评论4（❌ 不允许，level 超过 3）
```

**查询逻辑**：
```javascript
// 查询文章的所有顶级评论
db.comments.find({ post: postId, parentComment: null });

// 查询某条评论的所有回复
db.comments.find({ parentComment: commentId });
```

##### 4️⃣ 点赞表（likes）

```javascript
const LikeSchema = new Schema({
  user: {
    type: Schema.Types.ObjectId,
    ref: 'User',
    required: true
  },
  
  // 点赞的目标（文章或评论）
  targetType: {
    type: String,
    enum: ['Post', 'Comment'],
    required: true
  },
  targetId: {
    type: Schema.Types.ObjectId,
    required: true,
    // 💡 根据 targetType 动态引用不同的表
    refPath: 'targetType'
  },
  
  createdAt: {
    type: Date,
    default: Date.now
  }
});

// 💡 唯一索引：同一用户不能重复点赞同一目标
LikeSchema.index({ user: 1, targetType: 1, targetId: 1 }, { unique: true });

// 💡 复合索引：查询某文章/评论的所有点赞用户
LikeSchema.index({ targetType: 1, targetId: 1, createdAt: -1 });
```

**设计要点**：
- **refPath 动态引用**：`targetId` 可能指向 Post 或 Comment
- **唯一索引防止重复点赞**：同一用户点赞同一目标只能有一条记录

**查询2：获取文章及其所有评论（包含嵌套回复）**

```javascript
// MongoDB
const comments = await Comment.find({ post: postId })
  .populate('author', 'username avatar')
  .sort({ createdAt: 1 });

// 前端处理嵌套结构
const tree = buildCommentTree(comments);

function buildCommentTree(comments) {
  const map = {};
  const roots = [];
  
  comments.forEach(comment => {
    map[comment._id] = { ...comment.toObject(), replies: [] };
  });
  
  comments.forEach(comment => {
    if (comment.parentComment) {
      map[comment.parentComment].replies.push(map[comment._id]);
    } else {
      roots.push(map[comment._id]);
    }
  });
  
  return roots;
}
```

```sql
-- MySQL（需要递归查询，MySQL 8.0+）
WITH RECURSIVE comment_tree AS (
  SELECT * FROM comments WHERE post_id = ? AND parent_comment_id IS NULL
  UNION ALL
  SELECT c.* FROM comments c
  INNER JOIN comment_tree ct ON c.parent_comment_id = ct.id
)
SELECT * FROM comment_tree;
```

✅ **验证通过**：MongoDB 更简单，MySQL 需要递归 CTE

## 3个高频错误

### 错误1：字段类型选择不当导致性能问题

**现象**：文章内容字段使用 `String` 类型，数据库体积暴增

**原因**：MongoDB 默认限制单个文档 16MB，大量长文章会导致碎片化

**解决方案**：

```javascript
// ❌ 错误：将长文本直接存储
const PostSchema = new Schema({
  content: String  // 可能有 100KB+ 的 Markdown 内容
});

// ✅ 方案1：使用 GridFS 存储大文件
const PostSchema = new Schema({
  contentFileId: {
    type: Schema.Types.ObjectId,
    ref: 'fs.files'  // 引用 GridFS
  }
});

// ✅ 方案2：内容存储到对象存储（如 AWS S3）
const PostSchema = new Schema({
  contentUrl: String  // 存储 S3 链接
});

// ✅ 方案3：分离热数据和冷数据
const PostSchema = new Schema({
  excerpt: String,  // 列表页显示（热数据）
  content: String   // 详情页显示（冷数据，可考虑单独存储）
});
```

### 错误2：缺少唯一索引导致数据重复

**现象**：用户注册时，同一邮箱被注册多次

**原因**：并发请求导致重复插入

**解决方案**：

```javascript
// ❌ 错误：仅在应用层检查
async function register(email, password) {
  const exists = await User.findOne({ email });
  if (exists) {
    throw new Error('邮箱已存在');
  }
  // ⚠️ 如果两个请求同时到达这里，都会通过检查
  await User.create({ email, password });
}

// ✅ 正确：数据库层面添加唯一索引
UserSchema.index({ email: 1 }, { unique: true });

// 应用层捕获重复错误
async function register(email, password) {
  try {
    await User.create({ email, password });
  } catch (error) {
    if (error.code === 11000) {  // MongoDB 重复键错误码
      throw new Error('邮箱已存在');
    }
    throw error;
  }
}
```

### 错误3：外键级联删除导致数据丢失

**现象**：删除用户后，该用户的所有文章也被删除（用户反悔找不回）

**原因**：设置了 `ON DELETE CASCADE`

**解决方案**：

```sql
-- ❌ 错误：级联删除
ALTER TABLE posts
ADD FOREIGN KEY (author_id) REFERENCES users(id) ON DELETE CASCADE;

-- ✅ 方案1：软删除用户（推荐）
ALTER TABLE users ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE;

-- 删除用户时仅标记
UPDATE users SET is_deleted = TRUE WHERE id = 123;

-- 查询时排除已删除用户
SELECT * FROM posts p
INNER JOIN users u ON p.author_id = u.id
WHERE u.is_deleted = FALSE;
```

```javascript
// ✅ 方案2：MongoDB 不删除文章，仅标记作者为"已注销"
async function deleteUser(userId) {
  // 1. 软删除用户
  await User.updateOne({ _id: userId }, { isActive: false });
  
  // 2. 更新文章作者信息
  await Post.updateMany(
    { author: userId },
    { $set: { 'authorInfo.isDeleted': true } }
  );
}
```

## 深度解析

### 对比分析：MongoDB vs MySQL 如何选择？

| 维度         | MongoDB               | MySQL                 | 推荐场景             |
|--------------|-----------------------|-----------------------|----------------------|
| 数据结构     | 文档型（JSON）        | 关系型（表）          | 非结构化数据用 Mongo |
| Schema       | 灵活（无模式）        | 严格（预定义）        | 快速迭代用 Mongo     |
| 查询能力     | 聚合管道强大          | JOIN 成熟             | 复杂关联用 MySQL     |
| 事务支持     | 4.0+ 支持多文档事务   | 原生 ACID             | 金融场景用 MySQL     |
| 扩展性       | 水平扩展容易          | 垂直扩展为主          | 海量数据用 Mongo     |
| 学习曲线     | 中等                  | 低                    | 新手推荐 MySQL       |

**推荐选择**：
- 🚀 **内容管理系统（CMS）**：MongoDB（文章、评论、标签）
- 💰 **电商系统**：MySQL（订单、库存、支付）
- 🎯 **社交网络**：MongoDB（动态、评论）+ MySQL（用户关系）

## 总结

### ✅ 核心要点

1. **需求分析先行**：从用户故事到接口设计，清晰定义功能边界
2. **数据库选型**：根据数据特点选择 MongoDB（文档型）或 MySQL（关系型）
3. **索引提前规划**：根据高频查询设计索引，避免全表扫描

### 🔑 实现关键

1. **字段设计**：选择合适的数据类型（如 `enum` 限制角色值）
2. **引用关系**：MongoDB 使用 `ObjectId` 引用，支持 `.populate()` 联表
3. **唯一索引**：防止并发插入重复数据（如邮箱、用户名）
4. **软删除**：重要数据使用 `isDeleted` 标记，而非物理删除

### 💎 应用价值

- **减少返工**：提前设计好数据表，避免后期频繁迁移数据
- **提升性能**：合理的索引设计，查询速度提升 10-100 倍
- **保障质量**：唯一索引、外键约束等，从数据库层面防止脏数据

### 🎯 面试重点

- **Q1**：如何设计评论的嵌套回复？  
  **A**：使用 `parentComment` 字段指向父评论，限制 `level ≤ 3`。

- **Q2**：为什么给 email 字段加唯一索引？  
  **A**：防止并发请求时重复注册，数据库层面保证唯一性。

- **Q3**：MongoDB 和 MySQL 如何选择？  
  **A**：非结构化数据（文章、评论）用 MongoDB，强一致性数据（订单、支付）用 MySQL。

### ⚠️ 注意事项

1. **避免过度设计**：MVP 阶段只设计核心表，后续再迭代
2. **索引不是越多越好**：每个索引会增加写入开销，权衡读写比例
3. **数据迁移成本高**：表结构设计要充分讨论，避免频繁修改

### 🚀 进阶思考

- **分库分表**：当单表数据量超过 1000 万，考虑分表策略
- **读写分离**：主库写入，从库读取，提升并发能力
- **NoSQL + SQL 混合架构**：根据数据特点选择不同数据库

## 🤔 常见问题（FAQ）

### Q1：为什么用 UUID 而不是自增 ID？

**A**：UUID 适合分布式系统，避免 ID 冲突：

| 方案        | 优点                          | 缺点                     | 适用场景           |
|-------------|-------------------------------|--------------------------|-------------------|
| 自增 ID     | 简单、有序、占用空间小        | 分库分表时易冲突、暴露数据量 | 单库单表         |
| UUID        | 全局唯一、无冲突              | 无序、占用空间大（36字节） | 分布式系统       |
| Snowflake   | 有序、占用小（8字节）         | 需要独立服务             | 高并发分布式     |

**实战建议**：本项目使用 UUID v4，确保多服务器并发创建用户时无冲突。

### Q2：为什么密码字段长度是 255？

**A**：考虑哈希后的长度：

```javascript
// bcrypt 哈希后的长度是 60 字符
const hashedPassword = await bcrypt.hash('123456', 10);
console.log(hashedPassword.length); // 60

// 但为了兼容其他哈希算法（如 Argon2），设置 255
```

**最佳实践**：永远不要存储明文密码，使用 bcrypt/Argon2 哈希。

---

### Q3：为什么 createdAt 和 updatedAt 需要索引？

**A**：支持常见的时间范围查询：

```javascript
// 查询最近 7 天创建的文章
db.articles.find({
  createdAt: { $gte: new Date(Date.now() - 7 * 24 * 3600 * 1000) }
});

// 没有索引：全表扫描（慢）
// 有索引：索引扫描（快）
```

**性能对比**：
- 无索引：100万条数据，查询耗时 2-5秒
- 有索引：100万条数据，查询耗时 < 50ms

---

### Q4：如何处理删除数据（硬删除 vs 软删除）？

**A**：推荐软删除（逻辑删除）：

```javascript
// 方案1：硬删除（物理删除）❌
await Article.findByIdAndDelete(id);  // 数据永久丢失

// 方案2：软删除（推荐）✅
await Article.findByIdAndUpdate(id, { 
  isDeleted: true, 
  deletedAt: new Date() 
});

// 查询时排除已删除数据
Article.find({ isDeleted: false });
```

**优势**：
- 可恢复数据
- 审计需求（谁删除了什么）
- 数据分析（了解删除原因）

### Q5：外键约束在 MongoDB 中如何实现？

**A**：MongoDB 不支持外键约束，需应用层实现：

```javascript
// 创建文章时验证作者是否存在
async createArticle(data) {
  // 1. 检查作者是否存在
  const author = await User.findById(data.authorId);
  if (!author) {
    throw new Error('作者不存在');
  }
  
  // 2. 创建文章
  return await Article.create(data);
}

// 删除用户时检查是否有文章
async deleteUser(userId) {
  const articleCount = await Article.countDocuments({ authorId: userId });
  if (articleCount > 0) {
    throw new Error('用户有文章，无法删除');
  }
  
  await User.findByIdAndDelete(userId);
}
```

**最佳实践**：使用 Mongoose 的 `ref` + `populate()` 简化关联查询。
