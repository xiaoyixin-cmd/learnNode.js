# 15.5 Docker 部署与环境配置

## 知识点概述

### 核心概念

Docker 容器化部署是现代应用部署的标准方式。通过 Docker，你可以：

- **环境一致性**：开发、测试、生产环境完全一致
- **快速部署**：一条命令启动所有服务
- **资源隔离**：每个服务独立运行，互不影响
- **易于扩展**：通过编排工具快速扩容

### Docker 三件套

```
Dockerfile → 构建镜像的配方
Docker镜像 → 打包好的应用程序
Docker容器 → 运行中的应用实例
```

## 使用指南

### 第一步：理解 - Docker 化的优势

#### 传统部署方式的痛点

```bash
# ❌ 传统部署
1. SSH 登录服务器
2. 安装 Node.js（版本可能不一致）
3. 安装 MongoDB、Redis（配置复杂）
4. 拉取代码、安装依赖
5. 配置环境变量
6. 启动应用

# 问题：
- 环境差异导致"本地能跑，服务器不行"
- 依赖冲突（Node.js 版本、系统库）
- 部署步骤多，容易出错
```

#### Docker 部署的优势

```bash
# ✅ Docker 部署
docker-compose up -d

# 优势：
- 一条命令启动所有服务（App + MongoDB + Redis）
- 环境完全一致（镜像固化环境）
- 快速回滚（切换镜像版本）
```

### 第二步：准备 - Dockerfile 编写

#### 多阶段构建 Dockerfile

**`Dockerfile`**

```dockerfile
# ============================================
# 第一阶段：构建依赖
# ============================================
FROM node:18-alpine AS deps

# 设置工作目录
WORKDIR /app

# 💡 先复制 package.json（利用 Docker 缓存）
COPY package*.json ./

# 安装生产依赖
RUN npm ci --only=production && \
    npm cache clean --force

# ============================================
# 第二阶段：开发环境（可选）
# ============================================
FROM node:18-alpine AS dev

WORKDIR /app

# 复制依赖
COPY package*.json ./
RUN npm install

# 复制源码
COPY . .

# 暴露端口
EXPOSE 3000

# 启动开发服务器
CMD ["npm", "run", "dev"]

# ============================================
# 第三阶段：生产环境
# ============================================
FROM node:18-alpine AS production

# 💡 创建非 root 用户（安全）
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001

WORKDIR /app

# 从第一阶段复制 node_modules
COPY --from=deps --chown=nodejs:nodejs /app/node_modules ./node_modules

# 复制源码
COPY --chown=nodejs:nodejs . .

# 切换到非 root 用户
USER nodejs

# 暴露端口
EXPOSE 3000

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD node -e "require('http').get('http://localhost:3000/health', (r) => {process.exit(r.statusCode === 200 ? 0 : 1)})"

# 启动应用
CMD ["node", "src/server.js"]
```

**关键优化点**：

1. **多阶段构建**：最终镜像只包含生产依赖，减小体积
2. **Alpine 基础镜像**：体积更小（~5MB vs ~900MB）
3. **依赖缓存**：先复制 `package.json`，依赖未变时跳过安装
4. **非 root 用户**：提升安全性
5. **健康检查**：容器自动检测应用状态

### 第三步：实践 - Docker Compose 编排

#### docker-compose.yml

**`docker-compose.yml`**

```yaml
version: '3.8'

# 💡 定义网络
networks:
  blog-network:
    driver: bridge

# 💡 定义数据卷（持久化）
volumes:
  mongo-data:
  redis-data:
  uploads-data:

services:
  # ============================================
  # MongoDB 数据库
  # ============================================
  mongodb:
    image: mongo:7.0
    container_name: blog-mongodb
    restart: unless-stopped
    environment:
      MONGO_INITDB_ROOT_USERNAME: ${MONGO_ROOT_USER}
      MONGO_INITDB_ROOT_PASSWORD: ${MONGO_ROOT_PASSWORD}
      MONGO_INITDB_DATABASE: ${MONGO_DATABASE}
    volumes:
      - mongo-data:/data/db
      - ./docker/mongo-init.js:/docker-entrypoint-initdb.d/mongo-init.js:ro
    ports:
      - "27017:27017"
    networks:
      - blog-network
    healthcheck:
      test: echo 'db.runCommand("ping").ok' | mongosh localhost:27017/test --quiet
      interval: 10s
      timeout: 5s
      retries: 5

  # ============================================
  # Redis 缓存
  # ============================================
  redis:
    image: redis:7-alpine
    container_name: blog-redis
    restart: unless-stopped
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis-data:/data
    ports:
      - "6379:6379"
    networks:
      - blog-network
    healthcheck:
      test: ["CMD", "redis-cli", "--raw", "incr", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

  # ============================================
  # Node.js 应用
  # ============================================
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: production
    container_name: blog-api
    restart: unless-stopped
    env_file:
      - .env
    environment:
      NODE_ENV: production
      MONGODB_URI: mongodb://${MONGO_ROOT_USER}:${MONGO_ROOT_PASSWORD}@mongodb:27017/${MONGO_DATABASE}?authSource=admin
      REDIS_HOST: redis
      REDIS_PORT: 6379
      REDIS_PASSWORD: ${REDIS_PASSWORD}
    volumes:
      - uploads-data:/app/uploads
      - ./logs:/app/logs
    ports:
      - "3000:3000"
    networks:
      - blog-network
    depends_on:
      mongodb:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "node", "-e", "require('http').get('http://localhost:3000/health')"]
      interval: 30s
      timeout: 3s
      retries: 3

  # ============================================
  # Nginx 反向代理（可选）
  # ============================================
  nginx:
    image: nginx:alpine
    container_name: blog-nginx
    restart: unless-stopped
    volumes:
      - ./docker/nginx.conf:/etc/nginx/nginx.conf:ro
      - uploads-data:/usr/share/nginx/html/uploads:ro
    ports:
      - "80:80"
      - "443:443"
    networks:
      - blog-network
    depends_on:
      - app
```

#### MongoDB 初始化脚本

**`docker/mongo-init.js`**

```javascript
// 💡 创建应用用户（非 root）
db = db.getSiblingDB('blog-api');

db.createUser({
  user: 'blog_user',
  pwd: 'blog_password',
  roles: [
    {
      role: 'readWrite',
      db: 'blog-api'
    }
  ]
});

// 创建集合并添加索引
db.createCollection('users');
db.users.createIndex({ email: 1 }, { unique: true });
db.users.createIndex({ username: 1 }, { unique: true });

db.createCollection('posts');
db.posts.createIndex({ author: 1, createdAt: -1 });
db.posts.createIndex({ status: 1, publishedAt: -1 });
db.posts.createIndex({ title: 'text', content: 'text' });

print('✅ MongoDB initialized successfully');
```

#### Nginx 配置

**`docker/nginx.conf`**

```nginx
events {
    worker_connections 1024;
}

http {
    # 💡 上游服务器（Node.js App）
    upstream blog_api {
        server app:3000;
    }

    # 💡 限流配置
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    server {
        listen 80;
        server_name localhost;

        # 💡 访问日志
        access_log /var/log/nginx/access.log;
        error_log /var/log/nginx/error.log;

        # 💡 客户端请求体大小限制
        client_max_body_size 10M;

        # 💡 API 接口反向代理
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;

            proxy_pass http://blog_api;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_cache_bypass $http_upgrade;
        }

        # 💡 静态文件服务
        location /uploads/ {
            alias /usr/share/nginx/html/uploads/;
            expires 30d;
            add_header Cache-Control "public, immutable";
        }

        # 💡 健康检查
        location /health {
            proxy_pass http://blog_api/health;
            access_log off;
        }
    }
}
```

## 3个高频错误

### 错误1：容器无法连接到 MongoDB

**现象**：

```
Error: connect ECONNREFUSED 127.0.0.1:27017
```

**原因**：使用了错误的主机名（`localhost` 而非容器名）

**解决方案**：

```yaml
# ❌ 错误
environment:
  MONGODB_URI: mongodb://localhost:27017/blog-api

# ✅ 正确（使用容器名）
environment:
  MONGODB_URI: mongodb://mongodb:27017/blog-api
```

---

### 错误2：数据卷挂载后数据丢失

**现象**：重启容器后数据库数据消失

**原因**：未正确配置数据卷

**解决方案**：

```yaml
# ❌ 错误：使用绑定挂载（相对路径可能有问题）
volumes:
  - ./data:/data/db

# ✅ 正确：使用命名卷
volumes:
  - mongo-data:/data/db

# 在顶层定义卷
volumes:
  mongo-data:
```

---

### 错误3：环境变量未生效

**现象**：应用读取不到 `.env` 文件的变量

**原因**：Docker Compose 的 `env_file` 和 `environment` 混淆

**解决方案**：

```yaml
# ❌ 错误：environment 不会自动读取 .env
services:
  app:
    environment:
      NODE_ENV: production

# ✅ 方案1：使用 env_file
services:
  app:
    env_file:
      - .env

# ✅ 方案2：在 environment 中引用 .env 变量
services:
  app:
    environment:
      NODE_ENV: production
      MONGODB_URI: mongodb://${MONGO_USER}:${MONGO_PASSWORD}@mongodb:27017/${MONGO_DATABASE}
```

## 总结

### ✅ 核心要点

1. **多阶段构建**：deps → dev → production，最终镜像只包含生产依赖
2. **Docker Compose**：一键启动 App + MongoDB + Redis + Nginx
3. **数据持久化**：使用命名卷（volumes）保存数据
4. **健康检查**：容器自动检测应用状态，异常时重启

### 🔑 实现关键

1. **Dockerfile**：Alpine 基础镜像 + 非 root 用户 + 健康检查
2. **docker-compose.yml**：定义网络、数据卷、依赖关系
3. **环境变量**：使用 `.env` 文件管理配置，避免硬编码
4. **Nginx**：反向代理 + 限流 + 静态文件服务

### 💎 应用价值

- **环境一致性**：开发、测试、生产环境完全一致
- **快速部署**：一条命令启动所有服务
- **易于扩展**：通过 `docker-compose scale` 快速扩容

### 🎯 面试重点

- **Q1**：Dockerfile 的 COPY 和 ADD 有什么区别？  
  **A**：COPY 只复制文件，ADD 支持自动解压 tar 和从 URL 下载。推荐用 COPY。

- **Q2**：如何减小 Docker 镜像体积？  
  **A**：使用 Alpine 基础镜像、多阶段构建、合并 RUN 命令、清理缓存。

- **Q3**：Docker 容器内的应用如何访问宿主机服务？  
  **A**：使用 `host.docker.internal`（Mac/Windows）或宿主机 IP。

### ⚠️ 注意事项

1. **安全**：不要在镜像中包含 `.env` 文件，使用环境变量注入
2. **数据备份**：定期备份数据卷（MongoDB、Redis、上传文件）
3. **日志管理**：使用 Docker 日志驱动或挂载日志目录

## 🤔 常见问题（FAQ）

### Q1：为什么要用 Docker而不是直接部署？

**A**：环境一致性和快速部署：

**传统部署的问题**：
```bash
# 开发环境
Node.js 18.x + MongoDB 6.0 + Redis 7.0

# 生产环境
Node.js 16.x + MongoDB 5.0 + Redis 6.0

结果：在开发环境正常，生产环境报错 😭
```

**Docker 解决方案**：
```dockerfile
FROM node:18-alpine  # 锁定 Node.js 版本
# 镜像包含所有依赖，开发环境 = 生产环境
```

**优势对比**：
| 维度           | 传统部署         | Docker 部署     |
|----------------|-----------------|-----------------|
| 环境一致性     | ❌ 难保证        | ✅ 完全一致      |
| 部署速度       | ❌ 慢（需配置环境）| ✅ 快（30秒启动）|
| 回滚能力       | ❌ 困难          | ✅ 秒级回滚      |
| 资源隔离       | ❌ 无隔离        | ✅ 进程隔离      |

---

### Q2：多阶段构建有什么好处？

**A**：减小镜像体积：

**单阶段构建**（体积大）：
```dockerfile
FROM node:18
WORKDIR /app
COPY . .
RUN npm install  # 包含 devDependencies（如 Jest、ESLint）
CMD ["node", "server.js"]
```

镜像体积：**1.2 GB**（包含开发依赖）

**多阶段构建**（体积小）：
```dockerfile
# 阶段1：构建（包含所有依赖）
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install  # 完整依赖

# 阶段2：生产（只包含运行时依赖）
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
CMD ["node", "server.js"]
```

镜像体积：**200 MB**（减少 83%）

### Q3：如何在 Docker 中使用环境变量？

**A**：通过 `.env` 文件或 `-e` 参数传递：

**方法1：docker-compose.yml**
```yaml
services:
  app:
    environment:
      - NODE_ENV=production
      - MONGO_URI=${MONGO_URI}  # 从宿主机读取
    env_file:
      - .env.production  # 从文件加载
```

**方法2：docker run**
```bash
docker run -e MONGO_URI=mongodb://mongo:27017/blog \
           -e JWT_SECRET=mysecret \
           my-app
```

**方法3：.env 文件 + docker-compose**
```bash
# .env.production
MONGO_URI=mongodb://mongo:27017/blog
REDIS_URL=redis://redis:6379

# docker-compose.yml 自动加载 .env
```

### Q4：Docker 容器如何访问宿主机的 MongoDB？

**A**：使用 `host.docker.internal`（Mac/Windows）或宿主机 IP：

**Mac/Windows**：
```yaml
services:
  app:
    environment:
      - MONGO_URI=mongodb://host.docker.internal:27017/blog
```

**Linux**：
```yaml
services:
  app:
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      - MONGO_URI=mongodb://host.docker.internal:27017/blog
```

**最佳实践**：直接在 docker-compose 中启动 MongoDB：
```yaml
services:
  mongo:
    image: mongo:6
    ports:
      - "27017:27017"
  
  app:
    environment:
      - MONGO_URI=mongodb://mongo:27017/blog  # 使用服务名
```

### Q5：如何查看容器日志排查问题？

**A**：使用 `docker logs` 命令：

```bash
# 查看最近 100 行日志
docker logs --tail 100 blog-api

# 实时查看日志（类似 tail -f）
docker logs -f blog-api

# 查看最近 10 分钟的日志
docker logs --since 10m blog-api

# 查看包含 "error" 的日志
docker logs blog-api 2>&1 | grep -i error
```

**docker-compose 日志**：
```bash
# 查看所有服务日志
docker-compose logs

# 只看 app 服务的日志
docker-compose logs app

# 实时查看
docker-compose logs -f app
```
