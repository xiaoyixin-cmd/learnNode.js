# 12.2 编写Dockerfile

> Dockerfile指令详解、多阶段构建、镜像优化、基础镜像选择、安全最佳实践

---

## 📖 知识点概述

### 核心概念

**Dockerfile**是一个文本文件，包含构建Docker镜像的所有指令。Docker按照Dockerfile中的指令逐层构建镜像，每条指令创建一个新的镜像层。

### Dockerfile工作流程

```
Dockerfile (文本文件)
    ↓ docker build
镜像层1: FROM node:20-alpine
镜像层2: WORKDIR /app
镜像层3: COPY package*.json ./
镜像层4: RUN npm ci
镜像层5: COPY . .
镜像层6: CMD ["node", "app.js"]
    ↓ docker run
运行中的容器
```

### 镜像层原理

```
每条指令创建一个新层，层是只读的，可以被复用：

┌────────────────────────────────┐
│ 镜像A                          │
│ ├─ Layer 1: FROM node:20       │ ← 100MB
│ ├─ Layer 2: RUN npm install    │ ← 50MB
│ └─ Layer 3: COPY app.js        │ ← 1KB
└────────────────────────────────┘

┌────────────────────────────────┐
│ 镜像B                          │
│ ├─ Layer 1: FROM node:20       │ ← 复用，不占额外空间
│ ├─ Layer 2: RUN npm install    │ ← 复用
│ └─ Layer 3: COPY index.js      │ ← 新层，1KB
└────────────────────────────────┘

优点：
- 节省磁盘空间
- 加速构建（缓存）
- 加速镜像传输（只传输变化的层）
```

## 🚀 使用指南

### 理解：Dockerfile指令

#### FROM - 指定基础镜像

```dockerfile
# 💡 基本语法
FROM <image>[:<tag>] [AS <name>]

# 示例1：使用官方Node.js镜像
FROM node:20-alpine

# 示例2：指定精确版本
FROM node:20.11.0-alpine3.19

# 示例3：多阶段构建（命名阶段）
FROM node:20-alpine AS builder

# 示例4：从scratch（空镜像）构建
FROM scratch

# 💡 选择合适的基础镜像：

# ❌ 不推荐：标准镜像（体积大）
FROM node:20
# Size: ~950MB

# ✅ 推荐：Alpine镜像（体积小）
FROM node:20-alpine
# Size: ~120MB

# ✅ 推荐：Slim镜像（兼容性好）
FROM node:20-slim
# Size: ~200MB

# 对比：
# - alpine: 最小（120MB），但可能有兼容性问题
# - slim: 折中（200MB），兼容性好
# - standard: 最大（950MB），包含完整工具链
```

#### WORKDIR - 设置工作目录

```dockerfile
# 💡 基本语法
WORKDIR /path/to/workdir

# 示例
FROM node:20-alpine

# 💡 设置工作目录（自动创建）
WORKDIR /app

# 后续的COPY、RUN、CMD等指令都在/app目录下执行
COPY package.json ./  # 复制到/app/package.json
RUN npm install        # 在/app目录下执行

# 💡 可以多次使用WORKDIR
WORKDIR /app
WORKDIR src            # 等同于 /app/src
WORKDIR utils          # 等同于 /app/src/utils
```

#### COPY vs ADD - 复制文件

```dockerfile
# 💡 COPY（推荐）：复制文件/目录
COPY <src> <dest>

# 示例
COPY package.json ./
COPY src/ ./src/
COPY . .

# 💡 ADD：类似COPY，但支持URL和自动解压
ADD <src> <dest>

# 示例
ADD https://example.com/file.tar.gz /tmp/
ADD archive.tar.gz /app/  # 自动解压

# ⚠️ 最佳实践：优先使用COPY，除非需要ADD的特殊功能

# 💡 使用.dockerignore排除文件
# .dockerignore:
# node_modules
# .git
# *.log
# .env

# 💡 利用缓存优化（先复制依赖文件）
# ✅ 推荐：分层复制
COPY package*.json ./
RUN npm ci
COPY . .

# ❌ 不推荐：一次复制所有文件
COPY . .
RUN npm ci
# 问题：代码变化会导致依赖重新安装
```

#### RUN - 执行命令

```dockerfile
# 💡 两种形式

# 1. Shell形式（在/bin/sh -c中执行）
RUN npm install
RUN echo "Hello World"

# 2. Exec形式（直接执行，不启动shell）
RUN ["npm", "install"]
RUN ["/bin/bash", "-c", "echo Hello"]

# 💡 合并多个RUN减少镜像层
# ❌ 不推荐：多个RUN
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git
RUN apt-get clean
# 问题：创建4个镜像层

# ✅ 推荐：合并为一个RUN
RUN apt-get update && \
    apt-get install -y \
        curl \
        git \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
# 优点：只创建1个镜像层，体积更小

# 💡 使用换行符提高可读性
RUN npm ci \
    && npm run build \
    && npm prune --production

# 💡 清理缓存减小镜像体积
RUN apk add --no-cache python3  # Alpine
RUN apt-get update && apt-get install -y python3 && rm -rf /var/lib/apt/lists/*  # Debian/Ubuntu
```

#### ENV - 设置环境变量

```dockerfile
# 💡 基本语法
ENV <key>=<value>

# 示例
ENV NODE_ENV=production
ENV PORT=3000

# 多个变量
ENV NODE_ENV=production \
    PORT=3000 \
    LOG_LEVEL=info

# 💡 环境变量可以在后续指令中使用
ENV APP_HOME=/app
WORKDIR $APP_HOME
COPY package.json $APP_HOME/

# 💡 ARG vs ENV
# ARG: 构建时变量（docker build时传入）
# ENV: 运行时变量（容器运行时可用）

ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine

ENV NODE_ENV=production
```

#### EXPOSE - 声明端口

```dockerfile
# 💡 基本语法
EXPOSE <port> [<port>/<protocol>...]

# 示例
EXPOSE 3000
EXPOSE 3000/tcp
EXPOSE 3000/udp

# 多个端口
EXPOSE 3000 4000 5000

# ⚠️ EXPOSE只是声明，实际映射需要docker run -p
# Dockerfile:
EXPOSE 3000

# 运行容器：
# docker run -p 3000:3000 my-app
```

#### CMD vs ENTRYPOINT - 容器启动命令

```dockerfile
# 💡 CMD：容器启动时执行的默认命令（可被覆盖）
CMD ["node", "app.js"]

# 运行容器（使用CMD）
# docker run my-app
# 执行：node app.js

# 运行容器（覆盖CMD）
# docker run my-app node debug.js
# 执行：node debug.js

# 💡 ENTRYPOINT：容器启动时执行的命令（不易被覆盖）
ENTRYPOINT ["node"]
CMD ["app.js"]

# 运行容器
# docker run my-app
# 执行：node app.js

# docker run my-app debug.js
# 执行：node debug.js（CMD被替换）

# 💡 最佳实践：ENTRYPOINT + CMD组合
# 示例：灵活的启动脚本
FROM node:20-alpine
WORKDIR /app
COPY . .

# ENTRYPOINT定义执行器
ENTRYPOINT ["node"]

# CMD提供默认参数
CMD ["server.js"]

# 使用：
# docker run my-app → node server.js
# docker run my-app worker.js → node worker.js
# docker run my-app --inspect server.js → node --inspect server.js
```

#### USER - 指定运行用户

```dockerfile
# 💡 基本语法
USER <user>[:<group>]

# 示例
FROM node:20-alpine

# 创建非root用户
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001 -G nodejs

# 设置文件权限
WORKDIR /app
COPY --chown=nodejs:nodejs . .

# 切换到非root用户
USER nodejs

# 后续指令以nodejs用户运行
CMD ["node", "app.js"]

# ⚠️ 安全最佳实践：不要以root用户运行应用
```

### 实践2：多阶段构建

#### 为什么需要多阶段构建？

```
问题：构建工具增加镜像体积

单阶段构建：
┌────────────────────────────┐
│ FROM node:20               │
│ RUN npm install            │ ← 包含devDependencies
│ RUN npm run build          │ ← 包含构建工具
│ CMD ["node", "dist/app.js"]│
└────────────────────────────┘
镜像大小：950MB

多阶段构建：
┌────────────────────────────┐
│ 阶段1: Builder             │
│ FROM node:20               │
│ RUN npm install            │ ← devDependencies
│ RUN npm run build          │ ← 构建工具
└────────────────────────────┘
         ↓ 只复制构建产物
┌────────────────────────────┐
│ 阶段2: Production          │
│ FROM node:20-alpine        │
│ COPY --from=builder dist/  │ ← 只复制必需文件
│ CMD ["node", "dist/app.js"]│
└────────────────────────────┘
镜像大小：120MB（减少87%）
```

#### TypeScript应用多阶段构建

```dockerfile
# 💡 阶段1：构建阶段
FROM node:20-alpine AS builder

WORKDIR /app

# 复制依赖文件
COPY package*.json ./
COPY tsconfig.json ./

# 安装所有依赖（包括devDependencies）
RUN npm ci

# 复制源代码
COPY src/ ./src/

# 构建TypeScript
RUN npm run build

# 💡 阶段2：生产阶段
FROM node:20-alpine AS production

WORKDIR /app

# 只复制package.json和package-lock.json
COPY package*.json ./

# 只安装生产依赖
RUN npm ci --only=production

# 从builder阶段复制构建产物
COPY --from=builder /app/dist ./dist

# 非root用户
RUN addgroup -g 1001 nodejs && \
    adduser -S nodejs -u 1001 -G nodejs && \
    chown -R nodejs:nodejs /app

USER nodejs

EXPOSE 3000

CMD ["node", "dist/server.js"]

# 💡 构建镜像
# docker build -t my-ts-app:1.0.0 .

# 查看镜像大小
# docker images my-ts-app
# REPOSITORY   TAG     SIZE
# my-ts-app    1.0.0   120MB  ← 只包含运行时文件
```

#### Next.js应用多阶段构建

```dockerfile
# 多阶段构建 - Next.js应用
FROM node:20-alpine AS base

# 💡 阶段1：安装依赖
FROM base AS deps
RUN apk add --no-cache libc6-compat
WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci

# 💡 阶段2：构建应用
FROM base AS builder
WORKDIR /app

COPY --from=deps /app/node_modules ./node_modules
COPY . .

ENV NEXT_TELEMETRY_DISABLED 1

RUN npm run build

# 💡 阶段3：生产镜像
FROM base AS runner
WORKDIR /app

ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED 1

RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

# 复制public目录
COPY --from=builder /app/public ./public

# 复制.next目录（构建产物）
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

USER nextjs

EXPOSE 3000

ENV PORT 3000
ENV HOSTNAME "0.0.0.0"

CMD ["node", "server.js"]

# 💡 优化效果
# 构建阶段镜像: ~1.5GB（包含所有依赖和构建工具）
# 最终镜像: ~150MB（只包含运行时文件）
# 体积减少: 90%
```

### 实践3：镜像优化

#### 优化1：选择合适的基础镜像

```dockerfile
# ❌ 不推荐：完整镜像
FROM node:20
# Size: 950MB
# 包含：完整的Debian系统、构建工具、调试工具

# ✅ 推荐：Alpine镜像
FROM node:20-alpine
# Size: 120MB
# 包含：最小的Alpine Linux、Node.js运行时

# ✅ 推荐：Slim镜像
FROM node:20-slim
# Size: 200MB
# 包含：精简的Debian系统、Node.js运行时

# 💡 选择建议：
# - Alpine：体积最小，但可能有兼容性问题（使用musl libc）
# - Slim：兼容性好，体积适中
# - Standard：完整工具链，仅用于开发/调试

# 💡 解决Alpine兼容性问题
FROM node:20-alpine

# 安装glibc兼容层（如果需要）
RUN apk add --no-cache libc6-compat

# 或者使用Slim
FROM node:20-slim
```

#### 优化2：利用构建缓存

```dockerfile
# ❌ 不推荐：缓存失效
FROM node:20-alpine
WORKDIR /app

# 先复制所有文件
COPY . .

# 再安装依赖
RUN npm ci

# 问题：代码变化会导致依赖重新安装

# ✅ 推荐：优化缓存
FROM node:20-alpine
WORKDIR /app

# 💡 第1层：复制依赖文件（很少变化）
COPY package*.json ./

# 💡 第2层：安装依赖（缓存）
RUN npm ci --only=production

# 💡 第3层：复制代码（经常变化）
COPY . .

CMD ["node", "app.js"]

# 优点：
# - package.json未变化时，npm ci使用缓存
# - 只有代码变化时，才会执行COPY . .
# - 构建速度提升90%
```

#### 优化3：减小镜像层数

```dockerfile
# ❌ 不推荐：多个RUN（多个层）
FROM node:20-alpine
RUN apk add --no-cache curl
RUN apk add --no-cache git
RUN apk add --no-cache python3
RUN apk clean
# 问题：创建4个镜像层，体积增加

# ✅ 推荐：合并RUN（单个层）
FROM node:20-alpine
RUN apk add --no-cache \
    curl \
    git \
    python3 \
    && rm -rf /var/cache/apk/*
# 优点：只创建1个镜像层，体积减小
```

#### 优化4：清理临时文件

```dockerfile
# ❌ 不推荐：临时文件未清理
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
# 问题：npm缓存保留在镜像中

# ✅ 推荐：清理缓存
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production \
    && npm cache clean --force
# 优点：删除npm缓存，减小镜像体积

# 💡 多阶段构建（最佳实践）
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/app.js"]
# 优点：构建工具、缓存全部丢弃
```

## 🔧 错误处理

### 高频错误1：COPY失败

**现象：**
```
COPY failed: file not found in build context
```

**解决方案：**
```dockerfile
# ❌ 错误：路径不正确
COPY ./src/app.js /app/

# ✅ 正确：相对于构建上下文的路径
COPY src/app.js /app/

# 💡 构建上下文是docker build命令的路径
# docker build -t my-app .
# "." 是构建上下文，所有COPY路径相对于这个目录

# 💡 检查.dockerignore
# 确保文件没有被排除
```

### 高频错误2：RUN命令失败

**现象：**
```
RUN npm ci
The command '/bin/sh -c npm ci' returned a non-zero code: 1
```

**排查步骤：**
```dockerfile
# 1. 查看详细错误
docker build --no-cache -t my-app .

# 2. 使用docker run调试
docker run -it node:20-alpine sh
/ # cd /app
/ # npm ci
# 查看具体错误

# 3. 常见原因：
# - package-lock.json缺失
# - package.json语法错误
# - 网络问题（无法下载依赖）
# - 权限问题

# 4. 解决方案：
# - 确保package-lock.json已提交
# - 使用国内镜像
RUN npm config set registry https://registry.npmmirror.com
RUN npm ci
```

### 高频错误3：镜像体积过大

**问题：**
```
REPOSITORY   TAG     SIZE
my-app       1.0.0   1.2GB  ← 太大了！
```

**优化方案：**
```dockerfile
# 1. 使用Alpine镜像
FROM node:20-alpine  # 而非 node:20

# 2. 多阶段构建
FROM node:20 AS builder
RUN npm ci && npm run build

FROM node:20-alpine
COPY --from=builder /app/dist ./dist

# 3. 清理缓存
RUN npm ci --only=production \
    && npm cache clean --force

# 4. 使用.dockerignore
# 排除不必要的文件

# 5. 合并RUN指令
RUN apk add curl && apk clean  # 而非两个RUN

# 💡 查看镜像各层大小
docker history my-app:1.0.0
```

## 📋 总结

### 核心要点

1. **多阶段构建**：分离构建和运行，减小镜像体积
2. **Alpine镜像**：使用Alpine Linux减少基础镜像大小
3. **构建缓存**：先复制依赖文件，再复制代码
4. **安全加固**：非root用户、镜像扫描

### 实现关键

1. **FROM**: 选择合适的基础镜像（alpine/slim）
2. **COPY**: 利用缓存，分层复制
3. **RUN**: 合并指令，清理缓存
4. **USER**: 使用非root用户

### 应用价值

- **体积优化**：从950MB降到120MB（87%减少）
- **构建加速**：利用缓存，构建速度提升90%
- **安全加固**：非root用户、漏洞扫描
- **标准化**：统一的构建流程

### 面试重点

**Q1：如何优化Docker镜像体积？**
答：1) 使用Alpine基础镜像 2) 多阶段构建 3) 合并RUN指令 4) 清理缓存 5) .dockerignore排除无用文件。

**Q2：多阶段构建的优势？**
答：将构建和运行分离，构建阶段包含所有工具和依赖，运行阶段只包含必需文件，大幅减小镜像体积。

**Q3：COPY和ADD的区别？**
答：COPY只复制文件，ADD额外支持URL下载和自动解压。推荐使用COPY，除非需要ADD的特殊功能。

**Q4：为什么要使用非root用户？**
答：安全原则。容器逃逸时，攻击者只有普通用户权限，无法修改系统文件。

### 注意事项

⚠️ **先复制依赖文件**：package.json在前，代码在后
⚠️ **合并RUN指令**：减少镜像层数
⚠️ **使用.dockerignore**：排除无用文件
⚠️ **非root用户**：生产环境必须使用
