## 📖 知识点概述

### 核心概念

**Docker**是一个开源的容器化平台，使用操作系统级虚拟化技术，将应用及其依赖打包成轻量级、可移植的容器，实现"一次构建，到处运行"。

### Docker解决的问题

```
传统部署 vs Docker容器化

❌ 传统部署问题：
┌─────────────────────────────────────┐
│ 开发环境：macOS + Node.js 20.1       │
│ ├─ 依赖：npm install                  │
│ └─ 运行：npm start ✅                 │
└─────────────────────────────────────┘
              ↓ 部署到生产
┌─────────────────────────────────────┐
│ 生产环境：Ubuntu + Node.js 18.x      │
│ ├─ 依赖：npm install（版本不一致）    │
│ └─ 运行：npm start ❌ 启动失败       │
└─────────────────────────────────────┘
问题："在我机器上能运行"

✅ Docker容器化：
┌─────────────────────────────────────┐
│ Dockerfile                           │
│ ├─ FROM node:20-alpine               │
│ ├─ COPY package*.json ./             │
│ ├─ RUN npm ci                        │
│ └─ CMD ["node", "app.js"]            │
└─────────────────────────────────────┘
              ↓ 构建镜像
┌─────────────────────────────────────┐
│ Docker Image (myapp:1.0.0)           │
│ ├─ Node.js 20 运行时                 │
│ ├─ 应用代码                          │
│ └─ 所有依赖                          │
└─────────────────────────────────────┘
              ↓ 运行容器
┌──────────────┬──────────────┬────────┐
│ 开发环境 ✅   │ 测试环境 ✅   │ 生产 ✅ │
└──────────────┴──────────────┴────────┘
优点：环境一致、隔离性强、易于扩展
```

### Docker vs 虚拟机

| 特性 | Docker容器 | 虚拟机 |
|------|-----------|--------|
| **启动时间** | 秒级 | 分钟级 |
| **磁盘占用** | MB级 | GB级 |
| **性能** | 接近原生 | 有性能损耗 |
| **资源占用** | 低（共享宿主机内核） | 高（独立OS） |
| **隔离性** | 进程级隔离 | OS级隔离 |
| **可移植性** | 高 | 较低 |

### 理解1：镜像（Image）

#### 什么是镜像？

```
镜像是一个只读模板，包含运行应用所需的一切：
- 操作系统文件
- 运行时环境（Node.js、Python等）
- 应用代码
- 依赖包
- 配置文件

镜像由多个层（Layer）组成，采用Union FS技术：
┌────────────────────────┐
│ Layer 4: 应用代码       │  ← 可写层（容器运行时）
├────────────────────────┤
│ Layer 3: npm依赖        │  ← 只读层
├────────────────────────┤
│ Layer 2: Node.js运行时  │  ← 只读层
├────────────────────────┤
│ Layer 1: Alpine Linux   │  ← 只读层（基础镜像）
└────────────────────────┘

优点：
- 层可以复用（多个镜像共享基础层）
- 增量更新（只传输变化的层）
- 节省磁盘空间
```

#### 镜像操作命令

```bash
# 💡 搜索镜像
docker search node
# 输出：
# NAME                     STARS     OFFICIAL
# node                     12345     [OK]
# node-red                 567       

# 💡 拉取镜像
docker pull node:20-alpine
# 20-alpine: Pulling from library/node
# Digest: sha256:abc123...
# Status: Downloaded newer image for node:20-alpine

# 💡 列出本地镜像
docker images

# 输出：
# REPOSITORY   TAG        IMAGE ID       CREATED        SIZE
# node         20-alpine  abc123def456   2 weeks ago    120MB
# node         20         def456ghi789   2 weeks ago    950MB

# 💡 查看镜像详细信息
docker inspect node:20-alpine

# 💡 查看镜像历史（各层信息）
docker history node:20-alpine

# 💡 删除镜像
docker rmi node:20-alpine
docker image rm node:20-alpine

# 强制删除（即使有容器使用）
docker rmi -f node:20-alpine

# 删除所有未使用的镜像

# 💡 给镜像打标签
docker tag node:20-alpine my-node:latest

# 💡 保存镜像为tar文件
docker save -o node-20-alpine.tar node:20-alpine

# 💡 从tar文件加载镜像
docker load -i node-20-alpine.tar
```

### 理解2：容器（Container）

#### 什么是容器？

```
容器是镜像的运行实例，是一个隔离的进程。

镜像 vs 容器：
┌─────────────────┐
│  镜像（Image）   │  ← 类（Class）
│  - 只读模板      │
│  - 静态文件      │
└─────────────────┘
        ↓ docker run
┌─────────────────┐
│  容器（Container）│  ← 对象（Instance）
│  - 运行实例      │
│  - 动态进程      │
│  - 可写层        │
└─────────────────┘

容器隔离：
- 文件系统隔离（独立的根文件系统）
- 进程隔离（PID namespace）
- 网络隔离（独立的网络栈）
- 用户隔离（UID namespace）
- IPC隔离（进程间通信）
```

#### 容器操作命令

```bash
# 💡 运行容器（最简单）
docker run node:20-alpine node -v
# 输出：v20.11.0
# 容器执行完命令后自动停止

# 💡 后台运行容器（-d detach）
docker run -d --name my-app node:20-alpine sleep 3600
# 输出：容器ID（abc123def456...）

# 💡 运行并进入容器（-it interactive + tty）
docker run -it node:20-alpine sh
# 进入容器的shell
/ # node -v
v20.11.0
/ # exit

# 💡 运行容器并映射端口（-p）
docker run -d -p 3000:3000 --name api node:20-alpine node app.js
# -p 宿主机端口:容器端口

# 💡 列出运行中的容器
docker ps

# 输出：
# CONTAINER ID   IMAGE            STATUS         PORTS                   NAMES
# abc123def456   node:20-alpine   Up 5 minutes   0.0.0.0:3000->3000/tcp  api

# 💡 列出所有容器（包括已停止）
docker ps -a

# 💡 查看容器日志
docker logs my-app           # 查看所有日志
docker logs -f my-app        # 实时跟踪日志（类似tail -f）
docker logs --tail 100 my-app  # 查看最后100行
docker logs --since 10m my-app # 查看最近10分钟的日志

# 💡 进入运行中的容器
docker exec -it my-app sh
docker exec -it my-app bash  # 如果容器有bash

# 在容器中执行单个命令
docker exec my-app node -v

# 💡 查看容器详细信息
docker inspect my-app

# 💡 查看容器资源使用情况
docker stats my-app

# 输出：
# CONTAINER   CPU %   MEM USAGE / LIMIT   MEM %   NET I/O
# my-app      0.50%   50MB / 2GB          2.5%    1.2kB / 648B

# 💡 停止容器
docker stop my-app          # 优雅停止（发送SIGTERM）
docker stop -t 30 my-app    # 等待30秒后强制停止

# 💡 强制停止容器
docker kill my-app          # 立即停止（发送SIGKILL）

# 💡 启动已停止的容器
docker start my-app

# 💡 重启容器
docker restart my-app

# 💡 删除容器
docker rm my-app            # 删除已停止的容器
docker rm -f my-app         # 强制删除（即使正在运行）

# 💡 删除所有已停止的容器
docker container prune

# 💡 一次性容器（运行后自动删除）
docker run --rm node:20-alpine node -v
```

### 实践1：运行Node.js应用

#### 创建简单的Express应用

```javascript
// app.js
import express from 'express';

const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.json({
    message: 'Hello from Docker!',
    container: process.env.HOSTNAME,
    nodeVersion: process.version
  });
});

app.get('/health', (req, res) => {
  res.json({ status: 'healthy' });
});

app.listen(PORT, '0.0.0.0', () => {
  console.log(`Server running on port ${PORT}`);
});

// package.json
{
  "name": "docker-node-app",
  "version": "1.0.0",
  "type": "module",
  "dependencies": {
    "express": "^4.18.2"
  }
}
```

#### 直接运行容器（无Dockerfile）

```bash
# 💡 方式1：交互式运行
docker run -it --rm \
  -v $(pwd):/app \
  -w /app \
  -p 3000:3000 \
  node:20-alpine \
  sh -c "npm install && node app.js"

# 解释：
# -it          交互式
# --rm         容器停止后自动删除
# -v $(pwd):/app  挂载当前目录到容器的/app
# -w /app      设置工作目录
# -p 3000:3000 端口映射
# node:20-alpine 使用的镜像

# 💡 方式2：后台运行
docker run -d \
  --name my-node-app \
  -v $(pwd):/app \
  -w /app \
  -p 3000:3000 \
  node:20-alpine \
  sh -c "npm install && node app.js"

# 查看日志

# 停止并删除
```

### 实践2：数据卷（Volume）

#### 为什么需要数据卷？

```
问题：容器删除后，数据会丢失

┌─────────────────────────┐
│ 容器（Container）        │
│ ├─ 应用代码             │
│ ├─ 日志文件 ← 写入      │
│ └─ 数据库文件 ← 写入    │
└─────────────────────────┘
     ↓ docker rm
❌ 所有数据丢失

解决方案：数据卷（Volume）

┌─────────────────────────┐
│ 宿主机                  │
│ ├─ /var/lib/docker/     │
│ │  volumes/my-data/     │ ← 持久化存储
│ │  └─ data.db           │
└─────────────────────────┘
        ↑ 挂载
┌─────────────────────────┐
│ 容器                    │
│ └─ /app/data/ → Volume  │
└─────────────────────────┘

优点：
- 数据持久化
- 多个容器共享数据
- 性能更好（比bind mount）
```

#### 数据卷操作

```bash
# 💡 创建命名卷
docker volume create my-data

# 💡 列出所有卷
docker volume ls

# 💡 查看卷详情
docker volume inspect my-data
# 输出：
# [
#   {
#     "Name": "my-data",
#     "Mountpoint": "/var/lib/docker/volumes/my-data/_data"
#   }
# ]

# 💡 使用命名卷运行容器
docker run -d \
  --name mongodb \
  -v my-data:/data/db \
  -p 27017:27017 \
  mongo:7-jammy

# 数据存储在宿主机的/var/lib/docker/volumes/my-data/_data
# 即使删除容器，数据也不会丢失

# 💡 Bind Mount（挂载宿主机目录）
docker run -d \
  --name my-app \
  -v $(pwd)/logs:/app/logs \
  -p 3000:3000 \
  my-app:latest

# 宿主机的./logs目录映射到容器的/app/logs

# 💡 只读挂载
docker run -d \
  -v $(pwd)/config:/app/config:ro \
  my-app:latest
# :ro 表示只读

# 💡 删除卷
docker volume rm my-data

# 删除所有未使用的卷
docker volume prune
```

#### 数据卷实战示例

```bash
# 场景：MongoDB数据持久化

# 1. 创建数据卷
docker volume create mongo-data

# 2. 运行MongoDB容器
docker run -d \
  --name mongodb \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_ROOT_USERNAME=admin \

# 3. 插入测试数据

# > use testdb
# > db.users.insertOne({ name: 'Alice', email: 'alice@example.com' })
# > exit

# 4. 删除容器

# 5. 创建新容器（使用相同的卷）

  --name mongodb-new \
  -v mongo-data:/data/db \
  -e MONGO_INITDB_ROOT_USERNAME=admin \
  -e MONGO_INITDB_ROOT_PASSWORD=secret \
  -p 27017:27017 \
  mongo:7-jammy

# 6. 验证数据仍然存在
docker exec -it mongodb-new mongosh -u admin -p secret
# > use testdb
# > db.users.find()
# [ { _id: ..., name: 'Alice', email: 'alice@example.com' } ]
# ✅ 数据未丢失！
```

### 实践3：网络（Network）

#### Docker网络模式

```
1. bridge（默认）
   - 容器之间可以通过容器名互相访问
   - 需要端口映射才能从宿主机访问

2. host
   - 容器使用宿主机网络
   - 性能最好，但隔离性差

3. none
   - 容器没有网络

4. container
   - 共享另一个容器的网络
```

#### 网络操作

```bash
# 💡 列出网络
docker network ls

# 输出：
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123def456   bridge    bridge    local
# def456ghi789   host      host      local
# ghi789jkl012   none      null      local

# 💡 创建自定义网络
docker network create my-network

# 💡 查看网络详情
docker network inspect my-network

# 💡 运行容器并连接到自定义网络
docker run -d \
  --name api \
  --network my-network \
  -p 3000:3000 \
  my-api:latest

docker run -d \
  --name mongodb \
  --network my-network \
  mongo:7-jammy

# 💡 在api容器中可以通过容器名访问mongodb
docker exec api ping mongodb
# PING mongodb (172.18.0.2): 56 data bytes
# ✅ 可以访问

# 💡 将运行中的容器连接到网络
docker network connect my-network my-app

# 💡 断开容器与网络的连接
docker network disconnect my-network my-app

# 💡 删除网络
docker network rm my-network

# 删除所有未使用的网络
docker network prune
```

## 🔧 错误处理

### 高频错误1：端口冲突

**现象：**
```
Error: bind: address already in use
```

**解决方案：**
```bash
# 💡 检查端口占用
lsof -i :3000
netstat -ano | findstr :3000  # Windows

# 💡 停止占用端口的容器
docker ps | grep 3000
docker stop <container_id>

# 💡 使用不同的宿主机端口
docker run -d -p 3001:3000 my-app:latest
```

### 高频错误2：容器启动后立即退出

**现象：**
```bash
docker ps  # 容器不在列表中
docker ps -a  # 容器状态为Exited
```

**排查步骤：**
```bash
# 1. 查看容器日志
docker logs <container_id>

# 2. 查看退出代码
docker ps -a
# STATUS: Exited (1) 10 seconds ago

# 3. 常见原因：
# - 应用启动失败（语法错误、端口被占用）
# - 环境变量缺失
# - 依赖未安装
# - CMD命令错误

# 4. 调试：进入容器手动运行
docker run -it --rm my-app:latest sh
/ # node app.js
# 查看具体错误
```

### 高频错误3：找不到镜像

**现象：**
```
Unable to find image 'my-app:latest' locally
Error response from daemon: pull access denied
```

**解决方案：**
```bash
# 💡 确认镜像名称正确
docker images | grep my-app

# 💡 如果镜像不存在，需要先构建
docker build -t my-app:latest .

# 💡 如果是私有镜像，需要先登录
docker login registry.example.com
docker pull registry.example.com/my-app:latest
```

## 📚 应用场景

### 场景：完整的开发环境

```bash
# 项目结构
project/
├── docker-compose.yml
├── api/
│   ├── Dockerfile
│   └── src/
└── docker-scripts/
    └── init-db.sh

# docker-compose.yml
version: '3.8'

services:
  # Node.js API
  api:
    build: ./api
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: development
      MONGODB_URI: mongodb://mongodb:27017/devdb
      REDIS_URL: redis://redis:6379
    volumes:
      - ./api/src:/app/src  # 代码热重载
      - ./api/logs:/app/logs
    depends_on:
      - mongodb
      - redis
    restart: unless-stopped

  # MongoDB
  mongodb:
    image: mongo:7-jammy
    ports:
      - "27017:27017"
    volumes:
      - mongo-data:/data/db
      - ./docker-scripts/init-db.sh:/docker-entrypoint-initdb.d/init.sh
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: secret
    restart: unless-stopped

  # Redis
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    restart: unless-stopped

  # Redis Commander（可视化工具）
  redis-commander:
    image: rediscommander/redis-commander:latest
    ports:
      - "8081:8081"
    environment:
      REDIS_HOSTS: local:redis:6379
    depends_on:
      - redis

  # Mongo Express（可视化工具）
  mongo-express:
    image: mongo-express:latest
    ports:
      - "8082:8081"
    environment:
      ME_CONFIG_MONGODB_ADMINUSERNAME: admin
```

## 📋 总结

### 核心要点

1. **镜像（Image）**：只读模板，包含应用运行所需的一切
2. **容器（Container）**：镜像的运行实例，隔离的进程
3. **仓库（Registry）**：存储和分发镜像
4. **数据卷（Volume）**：持久化数据
5. **网络（Network）**：容器间通信

### 实现关键

1. **docker pull**: 拉取镜像
2. **docker run**: 运行容器
3. **docker ps**: 查看容器
4. **docker logs**: 查看日志
5. **docker exec**: 进入容器

### 应用价值

- **环境一致**：开发、测试、生产环境完全一致
- **快速部署**：秒级启动，随时扩缩容
- **资源高效**：轻量级，共享内核
- **易于管理**：统一的管理方式

### 面试重点

**Q1：Docker容器和虚拟机的区别？**
答：容器共享宿主机内核，启动快、资源占用少；虚拟机有独立OS，隔离性更强但开销大。

**Q2：如何实现容器数据持久化？**
答：使用Volume或Bind Mount。Volume由Docker管理，性能好；Bind Mount挂载宿主机目录，灵活。

**Q3：多个容器如何通信？**
答：创建自定义网络，容器连接到同一网络后可通过容器名互相访问。

**Q4：容器启动后立即退出怎么办？**
答：1) docker logs查看日志 2) docker run -it进入容器调试 3) 检查CMD命令和应用启动逻辑。

### 注意事项

⚠️ **容器不是虚拟机**：不要在容器内运行多个进程
⚠️ **数据持久化**：重要数据必须使用Volume
⚠️ **网络安全**：生产环境不要暴露所有端口
⚠️ **资源限制**：使用`--memory`、`--cpus`限制容器资源
